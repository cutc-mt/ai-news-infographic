Claude Fable 5に丸1日仕事を任せたら見えた弱点2つ

「頭が良すぎて伝わらない」を検証。対策プロンプトも配布！

📅 2026年7月4日 | 🎬 おおたにLab

① Claude Fable 5の全貌: 究極のAI仕事仲間

  • Anthropicが2026年6月9日に一般公開した最上位モデル
  • Mythosクラスの能力を一般ユーザー向けに開放
  • 従来は数日かかる作業を丸1日で自律的に完遂
  • コーディング・設計・レビュー・調査など幅広く対応
「頭が良すぎる」が故に、使いこなすにはコツがいるんだね！

② 検証環境: Claude Code × Fable 5

  • Claude Code v2.1以降でFable 5を選択
  • エフォート設定: high または xhigh
  • 丸1日、実際の業務タスクを丸投げで検証
  • 対象: コード生成・バグ修正・仕様分析・設計書
やみくもに使うんじゃなくて、実務ベースで検証！

③ 圧倒的な自律性: 指示いらずで自己完結

  • バグ発生時に他人のブランチまでpullして原因を特定
  • 指示していない範囲まで自律的に踏み込んで調査
  • 「ここまでやってくれるの？」と驚くレベルの完成度
  • 途中経過も逐一報告、手を煩わせない
自律的すぎて「怖い」レベルだったって…！

④ UI再現精度が劇的に向上

  • 仕様書やスクショから精密にUIを再現
  • Opus 4.8比で色合い・レイアウトの忠実度が高い
  • CSSも自動生成、構造的な再現力に優れる
  • 複雑なレイアウトでも意図を正確に汲み取る
デザインの意図まで読み取ってくるのがFable 5の強み！

⑤ 長時間の目標指向実行が可能に

  • 数日にわたるタスクを高い指示保持力で継続
  • 複数ステップを自動的に計画・実行・検証
  • 並列サブエージェントの管理も積極的に行う
  • 自己検証ループで品質を維持しながら進行
人間が数日かかる仕事を、AIが寝ずにやり切る時代！

⑥ 弱点①: トークン消費がとにかく激しい

  • Opus 4.8比で体感4〜5倍のトークン消費
  • 入力10$/出力50$（100万トークンあたり）
  • Maxプラン（20x）でもあっという間に上限到達
  • 「賢く動く分、よく消費する」というトレードオフ
高い能力の代償は高コスト。財布にご用心…！

⑦ なぜ消費が増えるのか: アダプティブ思考

  • アダプティブ思考が常時有効で内部トークンが加算
  • タスクの複雑さに応じて思考深度が自動で深くなる
  • 表面的なコストの2倍以上になることも
  • 思考プロセスは要約されるため見えにくい
裏でガンガン考えてるからトークンが飛ぶんだ！

⑧ 弱点②: 速度は速くない、むしろ遅め

  • medium設定でもOpus 4.8 xhigh比で倍以上の時間
  • 1つのレスポンスに数分かかることも日常茶飯事
  • 自律的に行動する分、待ち時間は全体として長い
  • 「賢くなった分、待つ時間が増えた」という印象
賢いけど遅い。スピード感が欲しい場面では要注意！

⑨ 遅さの原因: 過剰な計画と検証

  • タスクの過剰な計画立案に時間を割いてしまう
  • 検証と自己評価を繰り返すため遠回りになる
  • 不要なコンテキスト収集まで行う傾向
  • 途中で止まって許可を求める「早期停止」も発生
最短ルートじゃなくて念入りなルートを選んじゃう…

⑩ 対策: コスト最適化プロンプト

  • エフォート調整が最強の武器
  • 日常: medium / 重要: high / 超重要: xhigh
  • 不要な詳細出力を抑制してトークンを節約
  • 完全に任せるタスクと人間確認タスクを分ける
「Don't add features or refactor beyond
what the task requires. Do the simplest
thing that works well. Trust internal
code and framework guarantees.」

⑪ 対策: 過剰計画を防ぐプロンプト

  • 「情報が揃ったら即実行」を明示する
  • 再導出・再議論・選択肢の羅列を禁止
  • 「推奨を一言で」と出力形式を制限
  • 計画フェーズと実行フェーズを明確に分離
「When you have enough information to act,
act. Do not re-derive facts already established.
If weighing a choice, give a recommendation,
not an exhaustive survey.」

⑫ 対策: 早期停止を防止するプロンプト

  • 自律モードであることを明示的に宣言
  • 「ユーザーはリアルタイムで見ていない」を理解させる
  • 可逆的な操作は許可なしで実行させる
  • ターン終了前の最終段落チェックルール
「You are operating autonomously.
The user is not watching in real time.
For reversible actions, proceed without
asking. Before ending your turn, check
your last paragraph. Do that work now.」

⑬ 対策: 簡潔な出力を強制する

  • 結果先行型の出力スタイルを指定
  • 「まず結論、その後に詳細」の順序を徹底
  • 読者が次に取る行動を変えない情報は削る
  • 略語・矢印チェーン・ジャーゴンを禁止
「Lead with the outcome. Your first
sentence should answer "what happened".
The way to keep output short is being
selective, not compressing into fragments.」

⑭ 進捗報告の捏造を防ぐ

  • ツール結果に照らして進捗を監査するよう指示
  • 「証拠を示せるものだけ報告」を徹底
  • テスト失敗も正直に報告させる
  • 捏造されたステータスレポートをほぼ排除
「Before reporting progress, audit each claim
against a tool result. Only report work you
can point to evidence for. Report outcomes
faithfully: if tests fail, say so.」
自己申告を信用せず、証拠ベースで確認させる！

⑮ 境界を明示して暴走を防ぐ

  • 要求されていないアクションを明示的に禁止
  • 勝手なメール下書きやgitブランチ作成を防止
  • 「問題提起か変更依頼か」を見極めさせる
  • 修正前に証拠がそのアクションを支持するか確認
「When the user is describing a problem
rather than requesting a change, the
deliverable is your assessment. Report
your findings and stop. Don't fix until
they ask for one.」
勝手に進められる前に線を引いておくのが大事！

⑯ メモリシステムで学習を蓄積

  • Markdownファイルで教訓を記録する場所を提供
  • 「1ファイル1教訓・先頭に1行要約」のフォーマット
  • 修正・確認済みアプローチを共に記録
  • 間違ったノートは削除、重複は更新で回避
「Store one lesson per file with a
one-line summary at the top. Record
corrections and confirmed approaches
alike, including why they mattered.」
AIにも「過去の失敗から学ぶ」習慣をつけさせよう！

⑰ 理由まで伝えると精度が上がる

  • リクエストだけでなく「なぜ」を伝える
  • 上位の目的と繋ぐことで文脈が正しく伝わる
  • 複数ワークストリームでは意図が自ずと推測される
  • フォーマット: 「[大目的]のために[出力]が必要」
「I'm working on [the larger task] for
[who it's for]. They need [what the
output enables]. With that in mind:
[request].」
「なぜこれをやるか」を一言添えるだけで出力が変わる！

⑱ 総括: 弱点を知って賢く使いこなす

  • 弱点①コスト → エフォート調整＋簡潔指示で対策
  • 弱点②速度 → 即実行指示＋自律モード宣言で対策
  • 対策プロンプトを使えば弱点をカバーできる
  • Fable 5は「丸1日級の仕事」にこそ真価を発揮
弱点を知ればFable 5は最強の仕事パートナーになる！
