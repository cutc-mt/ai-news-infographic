③ループエンジニアリングとは: AIが自律的に回す設計のこと

  • ループエンジニアリングとは、AIが調査・実行・検証・修正を繰り返せるようにする設計
  • 単純に「うまくやって」と丸投げしても良い結果にはならない
  • ループの設計が仕事のスピードと品質に直結する
  • AIを使いこなせる人の差は「ループの設計力」で決まる
プロンプトの上手下手じゃなくて
ループの設計ができるかどうか！
これが2026年の新たなスキル

④プロンプトエンジニアリングからの移行: プロンプトだけじゃ足りない

  • 昔はプロンプトエンジニアリングが最重要スキルだった
  • プロンプトがAIの出力に直結していたため、非常に重視されていた
  • 今はプロンプトよりもループをどう設計するかが重要
  • プロンプト自体はもちろん大事だが、それだけでは不十分になった
プロンプト = 一回きりの命令
ループ設計 = AIを自律運転させる全体設計
次元が違う

⑤ループの基本サイクル: 調査→実行→検証→修正を回す

  • 調査: 問題の原因を特定する
  • 実行: 修正や実装を行う
  • 検証: テストを実行して結果を確認する
  • 修正: 問題があれば再修正して再ループ
  • このサイクルをAIが自律的に回し続ける
ループがうまく回れば、
数日間AIに自律作業させ続けることも可能！

⑥ループ設計のコツ1: 目的を明確に定義する

  • 目的が明確でないとループを回せない
  • 「何を達成したいのか」をAIにはっきり伝える必要がある
  • 目標が不明確だとAIが無駄な動きを繰り返してしまう
  • ゴールを明確にすることでループの終了条件も定めやすくなる
「うまくやって」は目的じゃない！
「〇〇が完成したら終了」のように
具体的な終了条件を設定しよう

⑦ループ設計のコツ2: 終了条件と検証方法を決める

  • 終了条件: 「このテストが通ったらOK」など明確に設定
  • 検証方法: どうやって結果が正しいかを判定するのか
  • 終了条件のミスで一生エラーを起こし続けることがある
  • 人間が全部決める必要はなく、AIと壁打ちしながら決めてもOK
終了条件がないとAIが無限ループに陥って
無駄にAPIコストやトークンを消費しちゃうよ！

⑧ループ設計のコツ3: 役割を持たせたサブエージェントを活用

  • 同じAIでも役割を持たせることで品質が上がる
  • 「あなたは〇〇の専門家です」と役割を与えて使い分ける
  • レビュー用テスト用エージェントを別々に分ける
  • 自分に自分をレビューさせるとバイアスがかかるので分けるのが正解
Claude CodeでもCodexでも
サブエージェント機能が使える！
並列で複数のエージェントを立ち上げよう

⑨並列処理で効率化: サブエージェントで同時に作業

  • 1つのエージェントだと効率が悪い
  • サブエージェントを立ち上げて並列処理させると速い
  • 時間のかかる作業でも並列化で大幅に短縮できる
  • 適切に並列処理を指示に入れることで数倍の速度アップが可能
1つのエージェントでじっくりやるより
3つに分けて同時進行した方が
圧倒的に早い！

⑩作業場所の分離: Gitのブランチで衝突を防ぐ

  • 複数エージェントが同じファイルを触ると衝突（コンフリクト）が起きる
  • Gitのブランチを分けて作業場所を分離する
  • 各ブランチで作業後にマージで統合する
  • 「ブランチを切って衝突しないように進めてください」と指示を入れる
先祖帰り現象に注意！
後から別エージェントが上書きして
せっかくの作業が消えることがある

⑪人間の役割: AIの「ディレクター」になること

  • 目的と方向性を決める — これが一番大事な人間の仕事
  • 制約を設定する — 「これはやってはいけない」を明確に
  • 優先順位を決める — 何を優先して進めるべきか
  • 承認と判断 — 危険な操作や最終的な結果のジャッジ
人間は作業をする人じゃない。
「オーケストレーターの上のディレクター」
がこれからの人間の役割！

⑫オーケストレーターとサブエージェントの構造

  • オーケストレーター: 各エージェントに指示を飛ばす中間管理役のAI
  • サブエージェント: 実際の作業を担当するAI
  • その上に人間がいて全体の方向を決める
  • 会社で言えば、ディレクター→マネージャー→ライターの構造に似ている
オーケストレーターが計画を立てて
仕事を割り振り、結果を集めて
次のアクションを考える。これがループ！

⑬実例: バグ修正のループ

  • ①原因調査: バグの原因を特定する
  • ②修正実行: 見つかった原因を修正する
  • ③テスト実行: 修正が正しいかテストする
  • ④レビュー: 結果がOKか確認する
  • 問題があれば再修正してループする
このサイクルをAIが自律的に回す！
人間は最終結果を確認するだけ。

⑭実例: 資料作成のループ

  • ①情報リサーチ: 必要な情報を調査・収集する
  • ②構成案作成: 資料の構成を決める
  • ③ドラフト作成: 本文を執筆する
  • ④レビュー: 品質や構成を確認する
  • 画像生成→品質チェック→ダメなら再生成も自動で
スライド資料ならスキルを使えば
100ページでも200ページでも
自動生成できる！

⑮AIエージェントの落とし穴: コストと無限ループ

  • APIの従量課金だとループ設計ミスでコスト暴騰するリスクがある
  • サブスクでもレート制限に引っかかって使えなくなることがある
  • エラー時に停止条件がなく無限に動き続ける危険性
  • 終わらなくなって時間もコストも無駄に消費してしまう
止めるための仕組みを必ず入れよう！
エラー時の停止条件は
ループ設計の最重要項目

⑯セキュリティ: 作業環境の分離が重要

  • ローカルで動かすとAIが機密ファイルを読み取るリスクがある
  • プロンプトインジェクション攻撃で機密情報が漏洩する可能性
  • クラウドの作業環境を使うかDockerコンテナで分離する
  • 開発コンテナの中でAIを動かすことでリスクを大幅に低減できる
AIは自律的に動くからこそ
触っていいファイルの範囲を
しっかり制限しておこう

⑰環境と権限の設計: 何を許して何を禁じるか

  • 環境: どこで、どういう資料で作業するか
  • 権限: 何を許可し、何を許さないか
  • CodexのFull AccessやClaude CodeのBypass Permissionは危険
  • 人間の確認ポイントをどこに挟むかを設計する
  • 使用するツールを明確に定義する
全部自動でやらせるのはリスクが高い。
最低限の危険操作だけは
人間がチェックしよう
