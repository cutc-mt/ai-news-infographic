Human in the Loopとは？

  • エージェントの実行を一時停止し、人間の承認を待つための組み込み機能。
  • Pythonの関数をデコレーターで修飾するだけで簡単に導入可能。
  • マルチターン対話をサポートし、継続的なやり取りが可能。
  • 条件付きロジックもサポートし、承認を求める状況を制御できる。
エージェントの自律的な実行に「待った」をかける、重要な安全機能なんです！

なぜ必要か？

  • 人間の判断が必要な、100%自動化できないワークフローのため。
  • 安全性と説明責任を確保するため。誰が承認したかを明確にする。
  • 人間の入力が価値を持つ、機密性の高い、または創造的なタスクに最適。
すべてをAI任せにせず、重要な局面で人間が介入することで、リスクを管理します。

主なユースケース

  • 承認ワークフロー: 大量メール送信やレポート配布前の最終確認。
  • コンテンツ生成: 人間の感性が必要な文章の要約やリフレーズ。
  • 意思決定: マルチステップのワークフローでのトリアージやエスカレーション。
  • 機密性の高い操作: データ削除やアラート発報など、影響の大きい操作の前のダブルチェック。
  • 創造的な作業: ユーザーがガイドしながら、文章のトーンを調整したり、アイデアを練り上げる。

実装の概要

  • Azure OpenAIを利用。まずリソースグループを作成。
  • 次にAzure AI Foundryリソースを作成。
  • Azure AI Foundryポータルで、GPT-4oなどのモデルをデプロイ。
  • 生成されたAPIキーとエンドポイントを取得してコードに設定。
Azureのガイド付きプロセスに従うだけで、簡単に環境をセットアップできます！

コードのポイント

承認が必要な関数にデコレーターを追加します。

@ai_function(approval_mode=ApprovalMode.ALWAYS_REQUIRED)
def generate_childish_text(topic: str) -> str:
    # この関数は実行前に承認を求める
    return f"This is a childish text about {topic}."
  • デコレーターを付けた関数を呼び出す際、エージェントはユーザーに実行許可を求めます。
  • 重要: ユーザーの応答（Yes/No）を含めて、次のエージェント呼び出しのコンテキストを更新する必要があります。これにより対話の継続性が保たれます。

設定のポイント

approval_modeにはいくつかの設定があります。

  • ALWAYS_REQUIRED: 常に関数の実行前に承認を要求します。
  • NEVER_REQUIRED: 承認を要求しません（デフォルト）。
ワークフローの要件に応じて、この設定を使い分けることが重要です。
