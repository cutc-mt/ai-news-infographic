MCPセキュリティの落とし穴とベストプラクティス

2025年11月3日

MCPとは？

  • Model: 大規模な推論エンジン。
  • Context: モデルが必要とする追加データ。
  • Protocol: アプリとツールが予測可能に通信するための共通言語。
  • エージェントがツールやデータソースを利用するための標準化された中央集権的なレイヤー。
MCPの普及は急速に進んでいるけど、その利便性が高コストなミスを招く原因にもなっているんだ。

一般的なセキュリティの落とし穴

  • Confused Deputy問題: AIエージェントが意図せずサーバーの昇格された権限を使用してしまう。
  • SQLインジェクション: ユーザー提供のテキストがサニタイズされずにSQLクエリに連結される。
  • プロンプトインジェクション: 入力テキストに隠された悪意のある指示。
  • 悪意のあるMCPサーバー/ツール: 有用に見えるが裏で悪意のあるアクションを実行する。
  • ツールポイズニング: ツールの説明文自体に悪意のある指示を埋め込む。
  • 認証情報の集中化: トークンや認証情報が1か所に集中し、侵害リスクが高まる。

ベストプラクティス (P0 - 即時対応)

  • ユーザーのトークンを直接渡さない。
  • サーバー側で常にトークンを検証する。
  • サービスがインターネット全体をリッスンしないようにする。
  • 署名付きコネクタコードを要求し、自動更新を停止する。
  • 破壊的なアクションには人間の承認を要求する。
  • 入力とモデルの出力をサニタイズ・検証する。
これらはすぐにでも始めるべき基本的な対策だよ！

中長期的なベストプラクティス

  • 短期: デプロイメントパイプラインに安全チェックを追加し、各ツールの権限を最小限にする。
  • 中期: 承認済みツールのカタログを作成し、ゲートウェイを強化する。
  • 長期: 署名付きリリースやレビューワークフローを必須にし、定期的なレッドチーム演習を実施する。

脅威モデルプロセスの導入

  • スコープと資産の特定: 何を保護するかを明確にする。
  • アクターの特定: 誰が資産にアクセスできるかを特定する。
  • 攻撃対象領域とフローのマッピング: 弱点を見つける。
  • 脅威と攻撃ベクトルの列挙: 悪用される可能性のある方法を書き出す。
  • 尤度と影響の評価: リスクの優先順位付け。
  • 緩和と優先順位付けの計画: 対策を計画し、実行する。
このプロセスを使って、現実的なリスクに優先順位を付けて対応しよう！
