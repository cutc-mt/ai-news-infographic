エージェントデベロップメントキット(ADK)徹底解説

マルチエージェントによるカスタムAIエージェント開発

AIエージェントとReAct

  • AIエージェントは、人間の代わりにタスクを思考とアクションに分解し、自律的に遂行するアプリ。
  • 従来のLLM活用が「作業」単位の指示だったのに対し、エージェントはより抽象的な「タスク」単位での指示が可能。
  • ReAct (Reason & Act) は、推論→行動→観察のサイクルを繰り返すことで、回答精度と信頼性を向上させるアプローチ。

シングル vs マルチエージェント

  • シングルエージェントは単純なタスクには十分だが、複雑なタスクではプロンプトが肥大化し、保守性や性能が低下する課題がある。
  • マルチエージェントは、各エージェントに明確な責務を持たせることで、プロンプトをモジュール化し、保守性と再利用性を高める。
  • 複雑なタスクを複数の単純なエージェントの協調作業として解決するアプローチ。

ADKのコアコンポーネント

  • Runner: AIエージェントの実行エンジン。ユーザー、LLM、サービス間のやり取りを担う中心的な役割。
  • LLM Agent: LLMとツールが接続されたオブジェクト。エージェントの核となる部分。
  • Session Service: ユーザーとの会話履歴（短期記憶）を保持する。本番環境では永続的なストレージが必要。
  • Memory Service: 複数のセッションをまたいだ長期記憶を担う。`Memory Bank`が推奨。
  • Artifact Service: 画像などのバイナリデータを格納するための領域。
  • Callback: エージェント実行の各ステップ（モデル呼び出し前後など）で任意のロジックを挟むためのフック。

ADKの強力なツール機能

  • FunctionTool: Gemini APIより強力。型ヒントとdocstringから関数を自動で認識し、フレームワーク側で実行まで行う。
  • ビルトインツール: Google SearchやBigQueryなど、すぐに使える便利なツール群。
  • MCPツール: 外部のMCPサーバーで提供されているツールをADK内から利用可能。
# OpenAPIスペック不要！
@adk.tool
def get_weather(city: str) -> str:
    """指定された都市の天気を取得します。"""
    ...
# ADKが型ヒントとdocstringを読み取り、自動でツール化

ワークフローエージェント

  • SequentialAgent: 複数のサブエージェントを直列に逐次実行する。
  • LoopAgent: サブエージェントを反復実行する。終了条件の定義が必要。
  • ParallelAgent: 複数のサブエージェントを並列に実行する。リサーチタスクなどで強力。
  • これらを組み合わせることで、ルールベースの確実なフロー制御とLLMの柔軟な思考を両立できる。

実行環境とエコシステム

  • Vertex AI Agent Engine: ADKなどで開発したエージェントをデプロイ・実行するためのマネージドサービス。Runnerや各種サービス（Session, Memory）を管理してくれる。
  • Google Agent Space: 開発したエージェントを登録し、利用するための統合プラットフォーム。カスタムエージェントの登録は現在プレビュー段階。
  • A2A (Agent-to-Agent): フレームワークに依存しないエージェント間通信プロトコル。ADKもA2Aとの相互運用性を強化中。
