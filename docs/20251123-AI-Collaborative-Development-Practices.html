  • BDD（ビジネス要件駆動）
  • Design by Contract

バイブコーディングの限界

プロンプトだけでの開発の問題点

  • 300文字程度では意図が伝わらない
  • 考える余地があるほど失敗する
  • 一貫性のないコードが生成される

スムーズ開発

AIがはちゃめちゃコードを出す原因

  • 型定義を再定義してしまう
  • 型定義が曖昧でハンドリング崩壊
  • セッションが長くなると忘れる
  • 過去の実装との一貫性がなくなる
解決策：AIが頑張るのではなく、環境でどうにかする

モノレポ構成の重要性

1つのリポジトリにフロント・バック・ツールをまとめる

  • 参照が簡単になる
  • コンテキスト供給が楽
  • 自動生成パイプラインが組みやすい
  • 2回git pushしなくて済む
別リポジトリだとコンテキストの供給が大変！

Single Source of Truth

バックエンドのコードを唯一の真実とする

Backend → OpenAPI → Frontend
バックエンドが更新されたら自動でフロントエンドの型も更新される

パーツ化の概念

良質なパーツを作り、それを組み立てる

木から削り出し
後戻り不可能
組み立てキット
同品質で作れる
パーツの質が生活の質を決める！

OpenAPI Spec自動生成

バックエンドのDTOから自動生成

  • 型定義が自動で作られる
  • 立ち上げるたびに更新される
  • 手動での型の二重管理が不要
ステップ1：DTOを唯一の型定義源とする

エラー型設計の必要性

エラー定義がないとフロントが破綻

  • エラーレスポンスがanyになる
  • useEffectで無限ループ発生
  • リンターエラーの真っ赤な画面
出た時マジで恐怖した...

人間が注力すべきこと

  • 良質なパーツ（型定義）を作る
  • 仕様書の作成・設計
  • ツール選定
  • どこを自動生成するか決める
  • プロジェクト全体の指針決定
パーツの質は人間次第！

AIに任せること

  • パーツを組み立てる（使う）
  • 自動生成された型を使う
  • API関数の呼び出し
  • UIコンポーネントの構築
AIは確定で無視をする可能性がある（10-20%）。ぶち切れたらダメ！

docs vs アプリケーション

ディレクトリ体系でフェーズを分離

  • docs：仕様書・計画書（人間が読む）
  • apps：実装コード
  • 参照するコンテキストを節約
フロントのコードを書く時はフロントの規約だけ参照

CLAUDE.mdの活用
