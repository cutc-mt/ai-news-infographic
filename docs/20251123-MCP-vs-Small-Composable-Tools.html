MCP vs 小規模コンポーザブルツール：トークン効率の真実

核心問題：トークン効率

日常のエージェント作業にMCPサーバーは本当に必要？Bashと小規模スクリプトで十分では？

エージェントを高速・低コスト・スマートに保つための選択

MCPの問題点

トークンを大量消費。巨大なマニフェストをコンテキストに読み込む必要あり

人気サーバーは全機能カバーを目指し、冗長な説明を持つ大規模ツールに

Playwright MCP

13.7K トークン（21ツール）

Claudeのコンテキストウィンドウの約6.8%を消費

Chrome DevTools MCP

18K トークン（26ツール）

コンテキストウィンドウの約9%を消費

軽量ツールキット

225 READMEのトークン数

Puppeteer coreを使ったNodeスクリプト群。エージェントは既にBash/JSを理解

start.js

Chromeをリモートデバッグモードで起動（ポート9222）
  • -profile オプションでログイン状態を維持
  • puppeteer.connect成功まで待機

nav.js

URLナビゲーション。-new で新規タブ作成

クリーンな出力：「opened」「navigated」。20種類のclick/hover/wait不要

eval.js

コンポーザビリティの要！ページ内でJS実行

document.title、querySelectorAllなど。結果をディスクに保存可能でプロンプト肥大化を回避

screenshot.js

スクリーンショット撮影→tempディレクトリに保存→パス出力

画像はファイルとして保存。送信を選択するまでコンテキストを消費しない

pick.js

視覚的に要素を選択！カーソルに追従するハイライト
  • クリックで選択
  • Cmd/Ctrl+クリックで複数選択
  • Enter確定 / Escキャンセル

pick.jsの出力情報

  • タグ、ID、クラス
  • トリミングされたテキスト
  • 親要素チェーン
  • outer HTMLスニペット

スクレイピング用のCSSセレクター/XPath生成に活用

セットアップ手順

  • ~/agent-tools/を作成
  • ツールリポジトリをクローン
  • PATHにエイリアス設定
  • READMEを作業ディレクトリに追加

名前衝突を回避

プレフィックスで整理：browser-tools-start.js、browser-tools-nav.js

READMEに「これらのスクリプトはグローバルに利用可能」と記載

コンポーザビリティ

成果物をチェーン、ディスクに保存、後で処理

TypeScript、Python、Bun、Go、ローカルVLMパイプラインなど何でも

Unix哲学

小さなツール、パイプ、ファイル。エージェントと相性抜群

RPCスキーマ不要、スキルカタログのパース不要。決定論的で誠実

MCPが有効な場面

企業環境でのガバナンス・アクセス制御が必要な場合
  • ロールベース権限
  • 監査可能性
  • キュレートされたルート

MCP利用時の注意

13-18Kトークンの説明 = ウィンドウの6.8-9%がメタデータで消費
  • 説明をトリム
  • 機能をオンデマンドでロード
  • 結果をファイルに出力

実践的推奨

デフォルトは小規模ツールキット。MCPは制約がある時のみ導入
出力はディスクへ、READMEはコンパクトに、エージェントの得意技（Bash&コード）を活かす
