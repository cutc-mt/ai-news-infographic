現行AIコーディングの限界

Cursor・Windsurf・Claude Code等はバイブコーディング向け。チャット→コード記述→修正→次へ。小タスク・クイックスクリプト・Reactコンポーネントには最適。しかし複雑アプリ(バックエンド・DB・複雑ロジック)では破綻。コンテキスト喪失・怠惰・幻覚・偽修正報告。AIベビーシッター必須。

幻覚インターン問題

高速・熱心だが幻覚見るインターン管理者に。10分前の指示忘れる。完了バイアス→未完成でも「完了」主張。コード辛うじて動作レベル。人間の継続監視・指導必須。スケールしない開発体験。複雑プロジェクトで生産性崩壊。

G3・敵対的協力パラダイム

研究論文「Adversarial Cooperation in Code Synthesis」ベース。Goose研究者開発オープンソースツール。AIエージェント同士が互いに戦ってより良いコード記述。実際のソフトウェアチーム模倣。衝撃的結果。新パラダイムでAIコーディング限界突破。

Goose・Jack Dorsey・Linux Foundation

GooseはTwitter創設者Jack Dorsey主導の優れたAIエージェント。現在Linux Foundation AI基金下で運営。全てオープンソース維持方針。G3もこの流れ。コミュニティ駆動開発。企業色なし。透明性・アクセス性重視。次世代AIインフラ構築。

弁証法的オートコーディング

Dialectical Autocoding概念導入。通常AIコーディング:1エージェントがユーザーを喜ばせようと推測。完了バイアス存在。「完了」と言いたい→コード辛うじて動作でもOK。G3は逆転発想。2つの対立エージェント・明確な対立役割。実際のチーム開発模倣。

Player・ビルダーエージェント

要件読む→コード記述→ファイル作成→コマンド実行。創造性・問題解決最適化。ただ構築したい。実装担当。コードを生み出す役割。前向き・積極的・行動派。仕様からアプリケーション具現化。テスト作成も担当。証明責任持つ。

Coach・批評家エージェント

実装コード1行も書かない。Player作業レビュー→テスト実行→コンパイル確認→エラー読む→不備指摘。敵対的ループ作成。Player提出→Coach欠陥発見。往復議論・修正→要件実際満たすまで継続。品質保証・QA役割。厳格なレビュアー。

フレッシュコンテキスト・秘密のソース

最大革新:コンテキストウィンドウ処理方法。チャット長くなる→モデル愚かに。古エラー・削除済み悪コード・混乱会話履歴で汚染。過去に引きずられる。G3は毎ターンメモリ完全消去。これが秘密のソース。注意限界バイパス。スケール可能。

ワークフロー・記憶リセット

Coach現状レビュー→具体的フィードバック生成(40行目ビルド失敗等)→新しいPlayerインスタンス起動。重要:新Player過去50ターンの失敗試行見ない。元要件文書+Coach最新フィードバックのみ。毎回フレッシュな脳でコード記述。前の悪アイデアに固執せず。

ケーススタディ・Git TUIエクスプローラー

ターミナルUIアプリ構築:コメント閲覧・差分表示・ブランチナビゲーション。非自明タスク。外部プロセス呼び出し・複雑テキスト出力解析・UI状態管理必要。Goose・Open Hands・Cursor(Claude 3.5 Sonnet)と比較テスト。目を見張る結果。

圧倒的成果・100%要件達成

Open Hands・Goose:部分実装のみ。エッジケース失敗・起動クラッシュ。Cursor:素晴らしいが手動プロンプト修正必要→人間ガイド必須。G3:3時間自律実行。完全機能アプリ提供。100%要件準拠。ゼロクラッシュ。初回プロンプト後人間入力ゼロ。1800行コード+完全テストスイート記述。

テスト駆動・堅牢性強制

Coach敵対的→Playerテスト記述強制。コード動作証明必須。テスト不合格→Coach拒否。標準バイブコーディングより遥かに堅牢なコードベース。品質保証組み込み。TDD自然実現。リグレッション防止。本番環境対応コード。

使用方法・要件文書必須

GitHubオープンソース。Rust実装(最新AIインフラトレンド)。マインドセット変更必要。「ゲーム作って」チャット不可。要件文書(Markdown)提供:技術スタック・機能・制約・設計ガイドライン明記。プロダクトマネージャーがエンジニアへ仕様書作成する感覚。

実行プロセス・完全自律

要件ファイル+APIキー→G3へ投入。Claude 4.5 Sonnet等ハイエンドモデル推奨(高度推論能力必要)。マシンで実行放置。エージェント起動→互いに議論→ターミナルコマンド実行→ファイル作成→数時間後動作アプリ完成。人間介入なし。

欠点1・速度とコスト

バイブコーディング向けではない。ボタン色変更・ナビバー追加には不向き。「寝てる間にバックエンドサービス構築」タイプタスク用。重イテレーションで時間かかる。毎ターンフレッシュコンテキスト+複数ループ→トークン大量消費。Claude 3.5 Sonnet APIで複雑タスク5-10ドル。

欠点2・ループリスク

人間介入したい場合あり。Coach些細スタイル問題に過度にこだわる→Player1時間空回り可能性。ターン制限(デフォルト10-20)でクレジットカード燃焼防止。しかしリスク存在。重要でない問題で停滞。マイナー問題vs主要機能のバランス課題。

アブレーション研究・Coach重要性証明

Coach除去テスト実施。結果:フィードバックループなし→Player動いているように見えるが実際完全破損の解決策を幻覚。敵対的フィードバックが重要コンポーネントと証明。批評家の存在が品質決定的。チェック機構なしではAI暴走。
