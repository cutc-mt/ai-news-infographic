MCP Apps 解説

MCP UIからMCP Appsへ - アーキテクチャの進化を2分で理解
Goose OSS 2分 2026年1月25日

①見た目は同じ

MCP UIとMCP Appsは視覚的には全く同じ。同じUI、同じユーザー体験だが、内部の仕組みが完全に異なる。

②MCP UI（従来）

ツール実行時にUIをレスポンスペイロード内に直接返す。createUIResourceを使ってUIをインラインで返却する方式。

③MCP Apps（新方式）

UIそのものではなくUIの場所へのポインタを返す。ホスト（Gooseなど）が別途UIをフェッチする。

④インライン vs 参照型

アーキテクチャの大きな転換。インラインUI → 参照型UIへ。ユーザー体験は同じだが、ポータビリティが向上。

⑤Webリソースとして扱う

UIを実際のWebリソースのようにフェッチ。他のホストでも同じUIを再利用できる移植性を実現。

⑥サンドボックスiframe

MCP Appsはサンドボックス化されたiframe内で動作。実際のセキュリティ制約が適用される。

⑦CSPによる制限

外部リソース（画像など）はデフォルトでブロックされる。Content Security Policyが適用される。

⑧画像が壊れる問題

MCP Apps移行時、Cloudinaryの画像がCSPにより表示されない問題が発生。URLは正常でもブラウザ内では動作しない。

⑨許可リストで解決

ホストに対してCloudinaryからのリソース読み込みを明示的に許可する設定を追加。これで正常に動作。

⑩本質的な違いに気づく瞬間

CSP問題を解決した時、これは単なるUIではなく本格的なインフラだと理解。セキュリティ境界がある。

⑪ポータブルなUI

MCP Appsにより同じUIを複数のホストで利用可能に。エージェント間での再利用性が向上。

⑫セキュアな設計

単なるUI強化ではなく、インタラクティブUIをセキュアに標準化する仕組み。エージェント間の互換性を確保。

⑬ホストが責任を持つ

ホスト側がUIをフェッチして表示する責任を負う。サーバーはポインタを返すだけ。

⑭分離されたアーキテクチャ

UIとツールのレスポンスが分離される。それぞれ独立してフェッチ・管理可能。

⑮既存サーバーからの移行

MCP UIサーバーからMCP Appsへの切り替えは比較的簡単。レンダリングは完璧に動作。

⑯セキュリティ設定が必要

外部リソースを使う場合は明示的な許可設定が必要。これがMCP Appsの特徴。

⑰エージェント間標準化

MCP Appsは複数エージェント間でのUI標準化を目指す。ポータビリティとセキュリティの両立。

⑱次世代のMCPへ

UIをよりポータブル、セキュア、標準化する進化。見た目は同じでも内部は大きく進化。
