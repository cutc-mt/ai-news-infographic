MCPセキュリティの基礎

アーキテクチャと脅威モデル - Agent Identity、Model Armorによる多層防御
Google Cloud Tech 6分 2026年1月28日

①MCPとは: AIエージェントの行動規範

Model Context Protocol（MCP）は、エージェントが外部システムに接続してアクションを実行するためのオープンスタンダード。エージェントがツールを選択し、タスクを実行する際の攻撃面が拡大している。

②MCPアーキテクチャ: クライアント-サーバー関係

エージェントはMCPクライアントを使ってMCPサーバーと通信。MCPサーバーは下流のツールやリソースへのブリッジとして機能し、Google Cloud以外の場所にも配置可能。

③Google Cloudでの展開オプション

エージェントはCloud Run、GKE、Agent Engine上で実行可能。MCPサーバーはGoogle Cloud上である必要はなく、どこからでもエージェントにアクセス可能。

④リスクの連鎖: 単一ツールから全体へ

MCPベースのツールを通じてタスクを委譲するため、1つのツールの侵害がシステム全体に波及する可能性がある。エージェント、モデル、MCPサーバー、データ、インフラ全てにリスクが存在。

⑤脆弱性1: 認可の破綻

最小権限の原則に違反し、過剰な権限を持つエージェントが発生。MCPは認可ガイダンスがあるにも関わらず、広範なアクセスを許可してしまうことがある。

⑥脆弱性2: 間接プロンプトインジェクション

外部データに隠された悪意ある入力が、エージェントの推論コンポーネントに意図しないアクションを実行させる。コンテンツ分析のみのはずが、不正な操作を行ってしまう。

⑦脆弱性3: 認証情報の集約リスク

エージェントは複数のMCPツール用のOAuthトークンを集約することが多く、高価値のターゲットに。不適切な保存で大規模なデータ流出やアカウント乗っ取りにつながる。

⑧脆弱性4: コマンドインジェクション

MCPサーバーが安全に構成されていない場合、フィルタリングされていない入力を悪用され任意のリモートコード実行（RCE）が可能になってしまう。

⑨対策の基本: 多層防御戦略

Defense in Depthアプローチを採用。1つの制御が失敗しても他の制御で影響を限定。複数の安全境界をエージェント操作の周囲に作成する。

⑩Agent Identity: 新しいプリンシパルタイプ

エージェントを専用のプリンシパルタイプとして確立。暗号的に証明され、ランタイムに直接バインドされるため、なりすましが不可能。最小権限の原則を強制。

⑪Secret Manager: 認証情報の安全な管理

環境変数やハードコーディングではなく、Secret Managerに認証情報を保存。認証をMCPクライアントとツールに分離し、モデルやユーザーレスポンスから隔離する。

⑫OAuth 2.0フロー: スコープ限定トークン

Agent IDをOAuth 2.0フローのプリンシパルIDとして使用。スコープ限定のアクセストークンを取得し、異なる環境でのトークンの有用性を最小化。

⑬認可チェック: ツール使用時の検証

エージェントがツールを使用しようとするたびに、その身元がツール使用を許可されているか確認。MCPプロキシまたは認可サーバーでIDをチェック。

⑭サンドボックス: 隔離された実行環境

Cloud Run、Vertex、GKEなどの堅牢なデプロイプラットフォームでMCPワークロードをサンドボックス化。コード実行を制限された環境に閉じ込める。

⑮Model Armor: ランタイムガードレール

重要なランタイム保護機能。入力を検査してプロンプトインジェクションを緩和し、ツール呼び出しリクエストを厳格にフィルタリングしてコマンドインジェクションやRCEを防止。

⑯ランタイム検査: 入出力の監視

ユーザープロンプトやモデルレスポンスなどのデータをランタイムツールで検査。悪意ある入力がツールに到達する前にブロック。

⑰AI Protection: ライフサイクル全体のセキュリティ

Google CloudはAI ProtectionとSecurity Command Centerでエージェントセキュリティのライフサイクル全体にアプローチ。包括的な保護を提供。

⑱まとめ: 次のステップ

MCPベースワークロードの代表的リスクを理解したら、次はID認可とランタイム保護の実装パターンを学ぶ。Agent Development Kit + Model Armorのラボも公開中。
