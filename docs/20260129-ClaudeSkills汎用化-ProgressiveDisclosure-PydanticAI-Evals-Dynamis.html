Claude Skillsを任意のエージェントに実装

Progressive Disclosure、Pydantic AI、Evals＆Observabilityで本格運用
Dynamis 20分 2026年1月29日

①Skillsの本質: Claude専用ではない

AnthropicがSkillsを広めたが、Claudeエコシステムに限定される必要はない。Progressive Disclosureの概念はどんなAIエージェントにも適用可能な普遍的アイデア。

②MCPサーバーの問題点

MCPサーバーは全てのツールを最初から一括でエージェントに渡してしまう。使わないツールまで大量にロードされ、LLMのコンテキストウィンドウを圧迫。

③Progressive Disclosure: 3層構造

第1層: 説明文（50〜100語）をシステムプロンプトに配置。第2層: skill.md本体（300〜500行）。第3層: 参照ドキュメント。必要に応じて段階的に読み込む。

④第1層: YAMLフロントマター

各スキルの説明文とパスをシステムプロンプトに配置。全体コンテキストの約5%。エージェントがいつスキルを使うべきか判断するための最小情報。

⑤第2層: skill.md本体

エージェントがスキルを使うと判断した時にskill.mdを読み込む。能力の完全な手順書。全体の約30%のコンテキスト。load_skillツールで取得。

⑥第3層: 参照ドキュメント

skill.mdが参照するスクリプトやMarkdown。さらに詳細な情報が必要な時のみ読み込む。read_referenceツールで取得。シンプルなスキルでは不要。

⑦実装: Pydantic AIエージェント

Dynamic System Promptでスキル説明を動的注入。skillsフォルダのskill.mdからYAMLフロントマターを自動抽出してシステムプロンプトに追加。

⑧3つのツール定義

load_skill（スキル本体を読み込み）、read_reference（参照文書を読み込み）、list_references（参照文書一覧）の3ツールで完結。

⑨スキルの追加は超簡単

skillsフォルダに新しいフォルダとskill.mdを追加するだけ。次回エージェント起動時に自動的に利用可能。Claude Codeと同じ感覚で運用。

⑩任意のLLM/フレームワーク対応

OpenRouter、Ollama、OpenAIに対応。LangChain、CrewAI、フレームワークなしでも同じ概念が適用可能。ローカルAIでも動作。

⑪デモ: 複数スキルの使い分け

レシピ検索、天気API、コードレビュー、ワールドクロックなど多数のスキルを1つのエージェントに搭載。会話内容に応じて必要なスキルだけを自動選択。

⑫システムプロンプトの設計

LLM自体はスキルの使い方を知らない。「スキルとは何か、どう使うか」を明示的に指示する必要がある。段階的な手順をプロンプトに記述。

⑬スキル作成のヒント

Claude DesktopのSettings→Capabilities→Skills→Skill Creatorを有効化。スキルを作るためのスキルが使える。skill.mdと参照文書を自動生成。

⑭Evals: スキル使用の自動テスト

YAMLで質問と期待するスキル呼び出しを定義。自動テストで全スキルが適切に使われるか検証。Pydantic AIの評価フレームワークを活用。

⑮Golden Data Set

25件のテストケースで全スキルの動作を検証。システムプロンプトやスキル説明を変更するたびにEvalを実行。Haikuモデルで高速かつ低コスト。

⑯Logfire: 本番環境の可観測性

Pydanticチーム製のLogfireで本番トレースを監視。ツール呼び出し、パラメータ、LLMの判断を全て追跡。ユーザー報告時の問題調査に活用。

⑰トークン使用量とコスト管理

Logfireでトークン使用量とコストを本番監視。スキルの段階的読み込みにより、全ツール一括ロードと比べてコンテキスト使用を大幅に削減。

⑱まとめ: スキルは無限に拡張可能

数十のスキルを追加してもエージェントは正常動作。Progressive Disclosureで最小限のコンテキストで最大限の能力を発揮。GitHubテンプレートを公開。
