Google ADKでマルチエージェントAIアプリ構築

Poet One: 複数のAIエージェントが協力して詩を書くデモアプリ
Google Cloud Tech 13分 2026年1月30日

①Poet One: AIと人間が協力して詩を書く

GoogleのAgent Development Kit（ADK）を使って構築されたデモアプリ。ユーザーとLLMが一緒に詩を書くマルチエージェントアプリケーション。複数の専門エージェントが段階的に詩を完成させる。

②Museエージェント: テーマのブレスト

最初に呼び出されるMuseエージェントが詩のテーマとコンセプトをブレインストーミング。核となる感情、視覚的アンカー、音、テクスチャー、メタファーを提案する。

③Architectエージェント: 形式の決定

Architectエージェントが詩の形式を提案。例えばカプレット（対句）形式を、「同期されたハーモニー」というコアコンセプトに合うからという理由で選択する。

④Scribeエージェント: 初稿の執筆

MuseとArchitectの出力を入力として受け取り、実際に詩の初稿を執筆するエージェント。ユーザーからのフィードバックを反映した修正版も作成できる。

⑤Criticエージェント: レビュー担当

エージェントが生成した出力を別のエージェントがレビューするという設計。Criticエージェントが弱い動詞や予測可能な表現を見つけ出して改善点を指摘する。

⑥Wordsmithエージェント: 最終仕上げ

Criticのレビューを受けて最終的な推敲を行う。例えば弱い動詞「make」を「marshall」に、予測可能な「symphony」をより鋭い表現に置き換えてアクション感を強調する。

⑦エージェントアーキテクチャの全体像

Poet Coordinatorがルートエージェントとして全体を統括し、Muse → Architect → Scribe → Critic → Wordsmithの順にサブエージェントが連携。複雑なタスクを分担して処理する。

⑧コードはシンプル: LLMAgentオブジェクト

ルートエージェントのコードは非常に簡潔。LLMAgent型のオブジェクトにモデル・説明・プロンプト・出力変数・サブエージェントを定義するだけ。

⑨プロンプトが「魔法」の部分

プロンプトにはエージェントの役割、ユーザーやサブエージェントとのやり取り方法、ステップバイステップの手順を記述。ここが実際のロジックの核となる。

⑩サブエージェントも同じ構造

サブエージェントのコードはルートエージェントと同様の構造。使用モデル、プロンプト、出力変数を定義。プロンプトには全体目標、他エージェントからの入力、処理フロー、出力形式を記載する。

⑪エージェント間の通信方法

各エージェントはoutput_keyを定義し、他のエージェントがそのキーをプロンプト内で参照する仕組み。例えばMuseの出力をScribeのプロンプトで使用できる。

⑫グルーコードが不要

ADKがエージェント間の接続を自動で行うため、接続用のグルーコードを書く必要がない。開発者はエージェント自体の構築に集中できる。ルートエージェントがオーケストレーションを担当する。

⑬なぜ複数エージェントが必要か

1つの長いプロンプトではなく複数エージェントを使う理由は、専門化したエージェントの方が賢いから。特にサブタスクが大きく異なる場合（例: サイバーセキュリティで原因分析・修正・ユーザー通知を混同させない）。

⑭実用例: サイバーセキュリティ

インシデント対応では原因分析、修正、ユーザーへの連絡という全く異なるスキルとツールが必要。1つのAIに全部やらせると混同する危険があるため、エージェントを分けることで安全性が向上する。

⑮ADKを選ぶ3つの理由

①ワークフローを平易な英語で定義でき、コードが少なくて済む。②セッションデータを管理し、各エージェントに必要なデータを配信。③デプロイ前にテスト・デバッグ用のWebツールを提供。

⑯プレーンイングリッシュで定義可能

ADKではマルチエージェントアプリのワークフローをコードではなく自然言語で定義できる。プロンプトエンジニアリングが中心で、複雑なプログラミングは不要。

⑰Cloud Runで即デプロイ

完成したマルチエージェントアプリはCloud Runにコマンド1つでデプロイでき、他の人もすぐに利用可能になる。開発からデプロイまでのワークフローがシームレス。

⑱まとめ: 3つの学び

①ADKでマルチエージェントアプリを簡単に作れる。②AIに一度に1つのタスクに集中させると最良の結果が得られる。③Cloud Runで即デプロイして共有できる。
