Claude公式スキルガイド徹底解説

SKILL.mdの書き方・設計原則・MCPとの連携・推奨パターンまで完全網羅
まさおAIじっくり解説ch 32分 2026年2月1日

①スキルとは何か

  • シンプルなフォルダーとしてパッケージ化された一連の指示
  • Claudeに特定のタスクやワークフローの処理方法を教える仕組み
  • 1回教えたら毎回同じ説明が不要になる
  • SKILL.mdとスクリプト・リファレンスなどで構成
  • Gitで管理してチームで共有も可能
エージェントに「仕事のやり方」と「知識」を追加するフレームワーク

②スキルが威力を発揮する場面

  • 仕様書からUI生成
  • 一貫した方法でリサーチを実行
  • コミットメッセージやドキュメントの一貫性を確保
  • マルチプロセスオーケストレーション — 複数ステップの自動化
  • MCP・ドキュメント生成・コード実行とも連携可能

③スキルの構造と設計原則

  • 最小構成: スキルディレクトリ直下にSKILL.md（大文字必須）
  • 追加: assets/、references/、templates/、scripts/ を配置可能
  • 3段階の段階的開示: YAMLフロントマター → SKILL.md本文 → リファレンス
  • 複数スキルを同時にロード可能、他のスキルと共存すべき
my-skill/
├── SKILL.md        # 必須
├── references/     # 詳細ドキュメント
├── templates/      # テンプレート
└── scripts/        # スクリプト

④SKILL.mdのフロントマター

  • 最低限必要なのはnameとdescriptionの2つだけ
  • nameはケバブケース（例: my-cool-skill）。スペース・大文字・アンダースコア禁止
  • フォルダー名とnameを一致させる
  • descriptionは1024文字以内
  • 追加でlicense、compatibility（環境要件）、metadata（著者・MCP情報）も記載可能
---
name: my-skill
description: |
  ○○を行うスキル。
  ユーザーが「○○して」と言った時に発動。
---

⑤descriptionの書き方が最重要

  • YAMLフロントマターはシステムプロンプトに常にロードされる
  • 何をするかといつ使うかの2つを明記
  • ユーザーが言いそうな具体的なタスク名を含める
  • NG: 曖昧すぎる / トリガー欠落 / 技術的すぎてユーザートリガーがない
descriptionの質がスキルの発動精度を左右する！

⑥MCPとスキルの関係

  • MCP = 実際のツール（道具そのもの）
  • スキル = レシピ（道具の使い方・ワークフロー・ベストプラクティス）
  • MCPツールをどう使うかをスキルで教えることで信頼性の高いツール利用を実現
  • スキルは知識を埋め込める点がMCPとの大きな違い
MCPが「何ができるか」、スキルが「どうやるか」を担当

⑦計画と設計: ユースケース定義

  • コードを書く前に2〜3個の具体的ユースケースを特定
  • トリガー（いつ）→ 問題解決の流れ（どう）を言語化する
  • インプットとアウトプットを明確にする
  • ドメイン知識やベストプラクティスを組み込むか判断
  • 汎用にするなら知識部分を剥がしてラッパー構造にする手もある

⑧3つのスキルパターン

  • パターン1: ドキュメント/アセット生成 — デザインルールやテンプレートを定義
  • パターン2: ワークフロー自動化 — スクリプト込みでコード自動生成やバリデーション
  • パターン3: MCP連携 — MCPツールの使い方をガイドするスキル（例: Sentryコードレビュー）
公式が3パターンを共有。自分のユースケースに当てはめよう

⑨指示の書き方ベストプラクティス

  • ステップごとに具体的な手順を記載
  • 成功事例（Example）セクションで期待する出力を示す
  • 問題が起こった時のフォールバック処理も記載
  • 段階的開示: SKILL.mdから詳細はreferences/内ドキュメントへ誘導

⑩テストとイテレーション

  • 10〜20個のテストクエリでトリガー精度を確認
  • 定量評価: 関連トリガー率90%目標、ツールコール数、失敗APIコール、トークン消費
  • 定性評価: 細かいプロンプト不要で完了するか、修正なしで目標達成するか
  • 言い換えリクエスト（記事生成→記事執筆→article generate）でもトリガーされるか
プロのコツ: まず1タスクでイテレーションしてから拡張！

⑪セキュリティ制限

  • nameにXMLアングルブラケットは禁止
  • 名前に「claude」「anthropic」を含めることは禁止
  • 悪意のあるコンテンツの指示を挿入する可能性があるため
  • READMEは含めない — すべてのドキュメントはreferences/に配置

⑫配布と共有

  • スキルフォルダーをZIPで共有、Claude.aiにアップロード
  • Claude Codeの場合は.claude/skills/スキル名/に配置
  • Agent Skillとしてオープンスタンダードで公開されており、他のAIエージェントでも使用可能
  • APIでプログラマティックにスキルを使用することも可能
compatibilityフィールドをちゃんと書いて環境を明示しよう

⑬マルチMCPオーケストレーション

  • 複数のMCPがある場合、スキルで使用順序をオーケストレーション
  • フェーズを分けてMCP間でデータを受け渡し
  • 各段階でバリデーションとエラーハンドリングを実施
  • ステップ間の依存関係とロールバック処理も明記

⑭出力品質の改良ループ

  • まず難しいタスクで実行してみる
  • 成功したパターンをスキルに抽出していく
  • 品質チェックの基準を明確に分けて定義
  • フィードバックに基づいてスキルを継続的に更新し続ける
画像生成スキルなど — 思った結果が出ない時は分析させてスキル自体を更新

⑮ドメイン特有の専門知識を埋め込む

  • 特定の業務をやっていないと知り得ない知識をスキルに記載
  • APIの仕組み、UIデザインルール、業務固有の手順など
  • スマートファイルストレージの例: ファイルタイプに応じた配置ルールを定義
  • ユーザーへの報告方法まで含めて書くとより完成度が高い

⑯トラブルシューティング

  • 認識されない: 大文字小文字の区別を確認。ls -laでSKILL.mdが表示されるか
  • トリガーされない: descriptionが曖昧すぎないか、トリガーフレーズを見直す
  • トリガーされすぎる: ネガティブトリガー追加 or より具体的にする（抽象度のバランスが重要）
  • 指示が守られない: 重要な指示が埋もれている、曖昧な言葉を使っている

⑰コンテキストウィンドウとスキル設計

  • Claude Opusのコンテキストウィンドウは約200Kトークン
  • システムプロンプトで30K程度消費。残りで作業する
  • ロングコンテキストになるほど動作が悪化する傾向
  • SKILL.mdはスマートに — 使っていないスキルは削除すべき
AIエージェントに気持ちよく働いてもらうには注意力のコントロールが鍵

⑱スキルクリエイターの活用

  • スキルクリエイターを使えばスキル作成自体を自動化できる
  • グローバル領域に置いておけば常時使える
  • 「スキル作って」と言うだけで自動的に発火する
  • スキルは更新し続けるもの — Opus 4.5に直接アップデートさせている
  • 抽象度とコントロールのバランスがユーザー体験を作る
公式ドキュメント＋実践イテレーションが最強の学習法
