Agent2Agent (A2A) プロトコル入門

AIエージェント間通信のオープン標準 - Googleが提唱するマルチエージェント連携の共通言語

2026年2月6日 | Google Cloud Tech - Holt Skinner

①A2Aプロトコルとは: AIエージェントの共通言語

  • Googleが作ったオープン標準で、AIエージェント同士が通信・協調するための共通プロトコル
  • 各エージェントの実装方法に関係なく、統一された方法でやりとりができる
  • LangChainがモデル切り替えを簡単にしたように、A2Aはエージェント間通信を標準化する
A2Aは「エージェント版HTTP」のような存在！

②なぜA2Aが必要か: エージェント乱立の課題

  • さまざまな企業が異なるフレームワークでAIエージェントを開発している
  • 他社のエージェントは完全にブラックボックスで、実装の詳細がわからない
  • 統一規格がないと、エージェント同士の連携が非常に困難になる
  • 複雑な問題を解くには複数のエージェントの協力が不可欠

③旅行計画の例え: マルチエージェントの理想形

  • フライトエージェントが航空券を検索・予約する
  • ホテルエージェントが宿泊先を手配する
  • アクティビティエージェントが現地の体験を提案する
  • すべてを自分で作る必要はなく、他者のエージェントを活用できるのが理想
各エージェントが専門分野を担当して
協力すれば最高の旅行プランが作れる！

④レゴブロックの設計思想: 標準化された接続

  • A2Aエージェントはレゴブロックのように自由に接続できる設計
  • 各エージェントが標準化された情報を公開し、同じパブリックメソッドをサポートする
  • どのエージェントからでも呼び出せるため、柔軟なオーケストレーションが可能になる
レゴのようにパチッとはめるだけで
エージェント同士がつながる！

⑤クライアント vs リモート: 2つの役割

  • クライアントエージェントはリクエスト作成とエンドユーザーとのやりとりを担当
  • リモートエージェントはリクエストを受けて実際の処理を行う
  • 1つのエージェントが文脈に応じて両方の役割を果たせる
クライアントにもサーバーにもなれるのがA2Aの柔軟性！

⑥業界の幅広い採用: パートナー企業の支持

  • ソフトウェア業界ですでに広く普及が進んでいる
  • 多数のパートナー企業がA2Aのサポートに合意している
  • 最新のパートナーリストはA2Aの公式ドキュメントで確認できる
  • 標準はその採用率によってのみ価値が決まるため、広範な支持は大きな強み

⑦コア機能: A2Aプロトコルの5つの能力

  • 動的ディスカバリーエージェントが互いを自動で発見できる
  • 標準化タスク統一された形式でタスクを共有し協調する
  • マルチモーダルテキスト・ファイル・構造化データなど多様なコンテンツを共有
  • 長時間プロセス実行に時間がかかるタスクも管理できる
  • エンタープライズセキュリティ企業レベルの安全性を確保
重要: エージェントの実装は常に
不透明（opaque）でOK！

⑧エージェントカード: デジタル名刺の仕組み

  • エージェントが公開する標準JSON形式のメタデータ
  • ドメイン上のWell-Known URIで提供される 
{
  "name": "Hotel Agent",
  "description": "ホテル検索・予約エージェント",
  "endpoint": "https://hotel.example/a2a",
  "skills": ["search", "booking"],
  "capabilities": { "streaming": true },
  "authentication": { "type": "oauth2" }
}
robots.txtやマイクロサービスの
サービスレジストリに似た概念！

⑨通信プロトコル: HTTPS + JSON-RPC 2.0

  • エージェント間の通信には標準HTTPSを使用してセキュアに通信する
  • メッセージの「封筒」としてJSON-RPC 2.0を採用
  • JSON-RPC はリモートサーバー上の関数を呼び出すためのシンプルな仕組み 
// JSON-RPC 2.0 リクエスト例
{
  "jsonrpc": "2.0",
  "method": "message/send",
  "id": "req-001",
  "params": { ... }
}

⑩メッセージとパート: コンテンツの構造

  • Messageは会話の1ターンを表す（質問や回答）
  • ロールは「user」または「agent」を指定する
  • Partは実際のコンテンツで、以下の種類がある:
  • プレーンテキスト・ファイル・マルチモーダル・構造化JSONデータ
テキストだけでなく、ファイルや
構造化データも送れるのが強み！

⑪エージェントエグゼキューター: SDKとロジックの橋渡し

  • A2A SDKのプロトコル処理とエージェント固有のロジックをつなぐクラス
  • 開発者が書くコードの中核部分で、エージェントの振る舞いを定義する
  • SDKがHTTP、JSON-RPC、イベント管理を自動で処理してくれる
  • エグゼキューターのおかげで、エージェントが他のエージェントと接続可能な「レゴ」になる

⑫タスクオブジェクト: ジョブのライフサイクル管理

  • Taskはエージェントが実行するジョブを表すオブジェクト
  • 固有のIDとステータスを持ち、ライフサイクルを管理する

Submitted Working Input Required Completed / Failed

長時間かかるタスクも「Working」状態で
待たせずに管理できる！

⑬ポーリングによる進捗確認: tasks/getメソッド

  • クライアントがリクエストを送ると、リモートが「タスクID: 123、現在Working」と応答
  • tasks/getメソッドで定期的にステータスを問い合わせる
  • 最終的にCompletedが返され、結果がタスクオブジェクト内に格納される 
// ポーリングの流れ
Client: "タスク123の状況は？"
Remote: { status: "working" }
Client: "タスク123の状況は？"
Remote: { status: "completed",
          result: "要約テキスト..." }

⑭SSEストリーミング: リアルタイム更新

  • Server-Sent Events (SSE) によるリアルタイムストリーミングに対応
  • message/streamメソッドでHTTP接続を維持したまま更新を受信
  • タスクステータス更新イベント: 「特定の部分を処理中」「完了」など
  • タスクアーティファクト更新: 長い結果をチャンク単位で段階的に配信できる
ドキュメントが生成される様子が
リアルタイムで見えるUX！

⑮エージェントスタック: Google推奨アーキテクチャ

  • A2Aだけでは本番環境のエージェントは完成しない
  • Googleが推奨するエージェントスタックの一部としてA2Aが位置づけられている
  • 推奨スタックの各コンポーネントは好みに応じて自由に入れ替え可能
A2Aは「エージェント間通信」のレイヤーだけを
担当するプロトコル！

⑯A2A vs MCP: 補完的な関係

  • A2AとMCPは競合ではなく補完関係にある
  • MCPはエージェントが自身のツール・API・リソースに接続する標準 = ファンクションコーリング
  • A2Aは独立したエージェント同士の動的な通信・協調・タスク委任の標準
  • 高度なエージェントシステムでは両方が同時に使われる
A2A = エージェント間の会話
MCP = エージェントとツールの接続
