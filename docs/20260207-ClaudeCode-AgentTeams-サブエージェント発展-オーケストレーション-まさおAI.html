Claude Code Agent Teams 徹底解説

サブエージェントの発展形 - チームで並列に作業するオーケストレーション機能

2026年2月7日 | まさおAI最新情報ch

①エージェントチームとは: Claude Codeの新機能

  • Claude Codeに新しく導入された実験的なチーム機能
  • Opus 4.6と同時に公開された注目の新機能
  • 複数のAIエージェントがチームとして協調して作業を進める仕組み
  • 1つのリーダーが複数のメンバーに仕事を割り振って並列で処理する
AIエージェントが「チーム」として
協力する時代が来た！

②破壊的な変化ではない: 既存の仕組みの発展系

  • AIエージェントのオーケストレーションの1つのステップとして位置づけられる
  • 既存のサブエージェント機能を拡張した発展形であり、全く新しいものではない
  • 今までの仕組みを理解していれば、スムーズに移行できる設計になっている
  • 段階的な機能追加であり、急に複雑になるわけではない

③チームの構成要素: リーダー・メンバー・管理ツール

  • リーダーチーム全体を統括し、タスクを割り振る司令塔
  • メンバーリーダーから割り振られたタスクを実行するエージェント
  • タスクリストチーム全体で共有される作業一覧
  • メールボックスメンバー間のメッセージ通信を管理する仕組み
人間のチーム開発と似た構造で
わかりやすい設計になっている！

④適したユースケース: 並列性の高いタスクに最適

  • 調査・レビュー複数の観点から同時に調査やコードレビューを実施
  • 新モジュール開発独立したモジュールを並行して開発する
  • 並行して実行できる独立性の高いタスクが最も効果を発揮する
  • お互いに依存しないタスクを同時に処理することで大幅に時間を短縮できる
独立したタスクを並列に回すのが
Agent Teamsの真骨頂！

⑤競合仮説デバッグ: 複数の仮説を同時に検証

  • バグの原因について複数の仮説を同時に検証できる
  • フロントエンドとバックエンドなどクロスレイヤーの同時開発が可能
  • 仮説Aと仮説Bを別々のメンバーに割り当て、同時並行で調査を進める
  • 順番に試すよりもはるかに高速にデバッグが完了する

⑥サブエージェントとの比較: 同期と非同期の違い

  • サブエージェントは親が一方的に呼び出す同期的な仕組み
  • サブエージェントはトークンコストが比較的低いのがメリット
  • 親エージェントが完了を待ってから次の処理に進むため、順次実行になる
  • シンプルなタスクならサブエージェントのほうが効率的な場合もある
サブエージェントは「1対1の依頼」
チームは「チーム全体での協力」！

⑦Agent Teamsの仕組み: タスクリストとメールボックスで管理

  • 複数のサブエージェントを一気に呼び出して並列で実行する
  • タスクリストでチーム全体の作業を一元管理する
  • メールボックスでメンバー間のコミュニケーションを実現する
  • リーダーがオーケストレーターとして全体の進捗を管理する
タスクリスト + メールボックスで
チームワークを実現！

⑧コスト面の違い: トークン消費量に注意

  • Agent Teamsは多数のサブエージェントを呼び出すためトークンコストが高い
  • 並列で動くメンバーの数に比例してコストが増加する
  • タスクの複雑さと並列性のメリットを天秤にかけて使い分ける必要がある
  • シンプルなタスクは通常のサブエージェントで十分な場合が多い
速さを取るかコストを取るか
タスクに応じて使い分けよう！

⑨関心ごとの分離: より細かい出力が得られるメリット

  • 各メンバーが特定の関心ごとに集中できるため、品質が向上する
  • より細かく関心ごとを分離できるので、より精緻な出力が得られる
  • 1つのエージェントが全てを考えるよりも、分担したほうが精度が上がる
  • 各メンバーのコンテキストウィンドウが独立しているため、複雑なタスクでも混乱しにくい

⑩環境変数での有効化: 実験版のため手動設定が必要

  • まだ実験版の機能なので、デフォルトでは無効になっている
  • settings.jsonで環境変数を設定して有効化する必要がある
  • 設定後にClaude Codeを再起動すると、チーム機能が使えるようになる 
// settings.json に以下を追加
{
  "env": {
    "CLAUDE_CODE_ENABLE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
この設定を入れないと
チーム機能は使えないので注意！

⑪表示モード: インプロセスとスプリットペインズ

  • インプロセス1つのターミナル内でチームの作業を順番に表示する
  • スプリットペインズTMuxやiTermで複数のペインに分割して並列で見渡せる
  • スプリットペインズならチーム全体の作業をリアルタイムで監視できる
  • TMuxやiTermが利用可能な環境ではスプリットペインズがおすすめ

⑫デリゲートモード: チームメンバーへの自動権限委任

  • チームメンバーに自動で権限を委任するモード
  • リーダーの許可なしにメンバーが自律的に作業を進められる
  • 毎回の承認を待たなくてよいため、作業スピードが大幅に向上する
  • 権限が広い分、意図しない操作が行われるリスクもある
「任せたよ！」と言えば
メンバーが自律的に動く！

⑬@メンションでの指示: 特定のメンバーに直接指示

  • チームモード中に@メンバー名で特定のエージェントに指示を出せる
  • リーダーを経由せずに、直接メンバーに追加の指示や修正依頼ができる
  • SlackやTeamsのメンションと似た直感的な操作感 
// チームモード中の使い方
@frontend-dev CSSのレイアウトを修正して
@backend-dev APIのレスポンス形式を変更して
@reviewer 変更内容をレビューして
人間のチーム開発と同じように
メンションで指示を飛ばせる！

⑭内部の仕組み（TaskTool）: サブエージェントの進化

  • サブエージェントは内部的にTaskToolで呼び出される仕組み
  • Agent Teamsはその上にチーム管理レイヤーを追加したもの
  • TaskToolの基盤はそのまま活用し、管理機能を拡張している
  • 既存のTaskToolの実装を理解していれば、チームの仕組みも理解しやすい

⑮チーム作成の流れ: 4つのステップで構成

  • Team Createチームを作成する
  • Task Createタスクリストを作成する
  • メンバーSpawnTaskToolでサブエージェントを起動する
  • タスク割り当て各メンバーにタスクを割り振る 
// チーム作成フローの概念図
Team Create
  └─> Task Create (タスクリスト)
       └─> Member Spawn (TaskTool)
            └─> Task Assign (割り当て)
                 └─> 並列で実行開始
この4ステップでチームが
立ち上がって作業を開始する！

⑯メッセージ通信: SendMessageツールで連携

  • SendMessageツールで各メンバー間の通信を実現する
  • メッセージにはサマリー・コンテンツ・タイプを指定できる
  • リーダーからメンバーだけでなく、メンバー同士でもメッセージを送り合える
  • メールボックスを介して非同期にメッセージを受け取る仕組み 
// SendMessageの概念
SendMessage({
  to: "frontend-dev",
  summary: "API仕様変更の通知",
  content: "レスポンス形式がJSONに変更",
  type: "notification"
})

⑰タスクの依存関係: DAGによる順序管理

  • DAG（有向非巡回グラフ）をサポートしている
  • 依存するタスクが完了するまで後続のタスクは自動的にブロックされる
  • 例: 「API設計が完了 → フロントエンド実装を開始」のような順序を定義できる
  • 依存関係のないタスクは自動的に並列で実行される

タスクA タスクB タスクC

（AとBに依存関係がなければ並列、Bが完了してからCを開始）

DAGで依存関係を管理するので
順番を間違えることがない！

⑱まとめと今後の展望: サブエージェントの正統進化

  • Agent Teamsはサブエージェントをより効率的にした発展形の機能
  • 今後の機能拡張の伏線となる重要なステップと位置づけられている
  • 現時点では実験的な機能だが、今後正式な機能として統合されていく見込み
  • AIエージェントのオーケストレーションが今後さらに進化していくことを示唆している
チーム機能はまだ始まったばかり！
