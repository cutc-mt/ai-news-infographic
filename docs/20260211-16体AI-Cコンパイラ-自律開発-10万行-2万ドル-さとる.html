16体のAIが2週間でCコンパイラを自律開発！10万行・2万ドルの舞台裏

2026年2月11日 | さとるチャンネル@毎日AIニュース

①衝撃の実験概要: 16体のAIが自律的にCコンパイラを開発

  • 16体のClaude AIが中央司令なしで協力し、2週間でCコンパイラを開発した。
  • 完成したコンパイラは約10万行のRustコードで構成されている。
  • API費用は合計で約2万ドル。人間の開発に比べて圧倒的に低コスト。
  • 最終的にLinuxカーネルをコンパイルできるレベルの実用的なCコンパイラが完成した。
人間の管理者なしでAIだけで協力し合うという前代未聞の実験
自律協調 Claude AI

②実験の背景: Anthropic安全性チームによるストレステスト

  • Anthropicの安全性チームが主導した実験で、ニコラス・カーリーニ氏がリーダーを務めた。
  • LLMが自律的にどこまで複雑なソフトウェア開発を達成できるかを検証する目的。
  • AI能力の限界と安全性のバランスを探るストレステストとして設計された。
  • 研究論文として公開され、AI開発コミュニティに大きな反響を呼んでいる。
安全性チームが「AIにどこまで任せられるか」を本気で試した結果
Anthropic 安全性研究

③なぜCコンパイラか: 分担しやすく自動テストが可能

  • Cコンパイラは工程が6つの明確なステージに分かれており、複数のAIが分担しやすい。
  • GCC tortureテストスイートにより、正解・不正解の判定を完全に自動化できる。
  • 字句解析、構文解析、意味解析、IR生成、最適化、コード生成の6段階で構成される。
  • 成功基準が明確なため、AIが自律的に進捗を判断して作業を進めやすい。
「正解がはっきりしているタスク」だからこそAIに適していた
コンパイラ6段階 GCC torture

④コスト比較: 人間の10分の1の時間と費用

  • 人間がCコンパイラを開発すると6ヶ月から1年かかり、費用は15万〜30万ドルと見積もられる。
  • AIは2週間で完成させ、費用はわずか2万ドル。時間も費用も約10分の1。
  • 単純なコスト比較だけでなく、24時間稼働できるAIの効率性が大きく貢献している。
  • ただし品質面では人間の開発者に及ばない部分もあり、単純比較には注意が必要。
コスパだけ見れば圧倒的だが、品質との兼ね合いも重要
2週間 2万ドル

⑤驚異の成果: GCC tortureテスト99%通過

  • GCC tortureテストの99%を通過し、実用レベルのコンパイラであることが証明された。
  • Linux 6.9カーネルのコンパイルに成功。OSレベルのソフトウェアを処理できる能力を持つ。
  • SQLite、FFmpeg、QEMUなどの大規模オープンソースプロジェクトもコンパイル可能。
  • 単なるサンプルレベルではなく、実際のプロダクションコードを処理できる完成度に到達した。
SQLiteやLinuxカーネルまでコンパイルできるのは驚き
99%通過 Linux 6.9

⑥3アーキテクチャ対応: x86-64、AArch64、RISC-V

  • x86-64はPC・サーバーで最も一般的なアーキテクチャで、メインターゲットとして対応。
  • AArch64はARMベースのアーキテクチャで、スマホやMacでも広く使われている。
  • RISC-Vはオープンソースの命令セットで、今後の普及が期待される新興アーキテクチャ。
  • 3つのアーキテクチャすべてで動作確認が取れており、クロスプラットフォーム対応を実現。
メジャーなCPUアーキテクチャをすべてカバーしている点が実用的
クロスプラットフォーム

⑦16体の協力方法: 無限ループで自律的に動作

  • 各AIは無限ループ構造で動作し、タスクリストを確認して自律的に作業を続ける。
  • タスクを見つけたらロックを取得し、他のAIとの競合を防止する仕組み。
  • 実行が完了したらGitにプッシュし、次のタスクを探すサイクルを繰り返す。
  • 中央の管理者やオーケストレーターは存在せず、完全に分散型の協調作業。
人間の管理者なし。AIが自分でタスクを取って自分で完了させる
無限ループ 分散型

⑧Gitベースのタスクロック: カンバンボード方式の管理

  • pending / current / completedの3フォルダでタスクの状態を管理する。
  • タスクファイルをフォルダ間で移動させることで、カンバンボードのように状態を遷移させる。
  • Gitのコミットとプッシュにより排他制御を実現し、同じタスクの重複実行を防止する。
  • すべての状態変更がGit履歴に残るため、後から作業の追跡と分析が可能になる。
シンプルだけど確実。Gitの仕組みをタスク管理に転用するアイデア
タスク管理 Git

⑨Agent Teamsメッセージング: AI同士のコミュニケーション

  • JSONファイルを使ってAI同士がメッセージを交換する仕組みを構築した。
  • バグ報告、仮説の提示、検証結果の共有など、人間のチームと同様のコミュニケーションを実現。
  • あるAIが発見した問題を別のAIが検証し、さらに別のAIが修正するという連携が可能。
  • メッセージの履歴も記録されるため、プロジェクト全体の意思決定プロセスが追跡できる。
Slackのような非同期コミュニケーションをAI同士がJSONで実現
JSON通信 非同期協調

⑩Docker環境: 16個のコンテナで統一環境を確保

  • 1台のサーバー上に16個のDockerコンテナを立ち上げて、各AIに独立した環境を提供。
  • 全コンテナが同一のイメージから起動するため、環境差異による問題を排除できる。
  • 統一された環境により、再現性のあるビルドとテストが保証される。
  • コンテナの分離により、あるAIの暴走が他のAIに影響を与えない安全設計になっている。
Dockerコンテナで環境を揃えるのはソフトウェア開発の基本中の基本
Docker 再現性

⑪設計原則その1: テストが羅針盤となる完璧性

  • テストが唯一の羅針盤であり、すべての進捗はテスト結果で判断される。
  • 偽陰性（本当は通るのに失敗と判定）が発生すると、AIが無限ループに陥るリスクがある。
  • 偽陽性（本当は失敗なのに成功と判定）が発生すると、バグを見逃してしまう。
  • テストの信頼性がプロジェクト全体の成否を左右するため、テスト設計が最も重要。
テストが壊れると全体が崩壊する。テストの品質がすべてを決める
テスト駆動 品質保証

⑫設計原則その2: コンテキストの外部化で引き継ぎ

  • README / ARCHITECTURE / PROGRESSの3ドキュメントでプロジェクト知識を外部化。
  • AIのコンテキストウィンドウには限りがあるため、知識を外部ファイルに保存して共有する。
  • 新しいAIがタスクに着手する際、これらのドキュメントを読むことで迅速にキャッチアップできる。
  • 進捗が更新されるたびにドキュメントも更新され、常に最新の状態が維持される。
人間の開発チームでもドキュメント整備は必須。AIも同じだった
外部記憶 ドキュメント

⑬設計原則その3: 並列化による効率的なデバッグ

  • テストに失敗した箇所を特定し、10体のAIが各1ファイルずつ同時にデバッグを行う。
  • 並列デバッグにより、直列で処理するよりも大幅に修正時間を短縮できる。
  • それぞれのAIが独立したファイルを担当するため、コンフリクトが発生しにくい。
  • この並列化戦略が、2週間という短期間での完成を可能にした重要な要因の1つ。
10体が同時にバグを潰す光景は圧巻。AIならではの力技
並列デバッグ 高速修正

⑭設計原則その4・5: 役割分担と漸進的な開発

  • AIには実装・QA・最適化・統合の4つの役割が割り当てられた。
  • 最小限の動作するMVPからスタートし、段階的に複雑度を上げていく漸進的開発を採用。
  • 実装担当が新機能を作り、QA担当がテストし、最適化担当がパフォーマンスを改善する流れ。
  • 統合担当が各部品を組み合わせ、全体として正しく動作するかを確認する役割を担う。
人間のアジャイル開発と同じ手法をAIチームにも適用した
役割分担 MVP開発

⑮限界その1: コード効率でGCCに及ばない

  • 生成される実行ファイルはGCCと比べて約2倍遅いという性能差がある。
  • コンパイル速度自体もGCCの1.5倍ほど遅く、大規模プロジェクトでは差が大きくなる。
  • 最適化パスの実装が不十分であり、GCCが数十年かけて磨いた最適化には遠く及ばない。
  • 動作の正確性は高いが、効率面ではまだまだ改善の余地が大きい状態。
「動く」と「速い」は別問題。GCCは40年の歴史がある
性能差 最適化不足

⑯限界その2: 大規模化するとリグレッションが発生

  • 10万行付近で新機能の追加がリグレッション（既存機能の破壊）を引き起こすようになった。
  • コードベースが大きくなるほど、変更の影響範囲を正確に予測することが困難になる。
  • AIのコンテキストウィンドウの限界により、プロジェクト全体を把握しきれなくなる問題。
  • この壁を突破するには、より高度なアーキテクチャ理解やモジュール分割が必要になる。
人間の開発者も同じ問題を抱える。規模が大きくなると管理が難しい
リグレッション スケーリング

⑰著者の警告: テスト通過は品質保証を意味しない

  • 著者は「自分で検証していないソフトウェアをデプロイするのは本当に怖い」と警告している。
  • テストスイートを99%通過しても、残り1%に致命的なバグが潜んでいる可能性がある。
  • AIが生成したコードを人間がレビューせずに本番環境に投入するのは危険だと強調。
  • 自律開発の成果は印象的だが、安全性の観点からは慎重なアプローチが求められる。
「テストが通った＝安全」ではない。人間の目によるレビューは不可欠
安全性 人間レビュー

⑱持ち帰りポイント: 人間の役割は設計とテスト設計へ

  • 人間の役割は「設計」と「テスト設計」にシフトしていく。実装はAIが担う時代が来る。
  • テストがすべてを決める。良いテストを設計できるかが、プロジェクト成功の鍵になる。
  • AIの能力と安全性のバランスをどう取るかが、今後の重要な戦略的課題となる。
  • AIに任せる範囲を広げつつ、品質と安全性を担保する新しい開発プロセスが必要になる。
  • この実験は、AI時代のソフトウェア開発の未来像を示す重要なマイルストーンとなった。
コードを書く能力より「何をテストすべきか」を考える能力が重要に
未来の開発 テスト設計
