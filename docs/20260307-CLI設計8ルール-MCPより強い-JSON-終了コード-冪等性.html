AIエージェントが本当に使えるCLI設計

MCPより強い8ルール・JSON・終了コード・冪等性

2026-03-07

①CLIの敵対的設計

世の中のCLIはエージェントにとって敵対的。次の一手を決める材料を出さない。

②何が問題か

構造化データ・終了コード・再実行可能性が曖昧だとエージェントが詰まる。

③MCPよりCLIが良い理由1

トークン効率。MCPはツール定義がコンテキストに常駐。CLIは必要な時だけヘルプを読む。

④MCPよりCLIが良い理由2

シンプルさ。ヘルプを読んで実行して検証。この流れが自然。

⑤MCPよりCLIが良い理由3

信頼性。CLIはステートレス。失敗しても再実行しやすい。

⑥ルール1: JSON必須

stdoutはJSONだけ。進捗はstderrへ。パース可能な契約に固定。

⑦ルール2: 終了コード

0=成功・1=失敗・2=使い方エラー・3=未検出・4=権限・5=競合。意味のあるコードを。

⑧ルール3: 冪等性

CreateよりEnsureを優先。同じコマンドを2回叩いても同じ状態に。

⑨ルール4: 自己完結ヘルプ

ヘルプだけで完結。Examples・よくある失敗・復旧手順を含める。

⑩ルール5: 組み合わせ可能性

--quietでパイプ向きに。stdin入力も対応。小さいツールを繋げる。

⑪ルール6: ドライラン

--dry-runで何を変えるか表示。実際の変更はしない。安全性向上。

⑫ルール7: アクション可能なエラー

エラーに失敗した入力・推奨コマンド・リトライ可否を含める。

⑬ルール8: 一貫した名詞動詞

リソースと操作を分離。user-list・user-create・user-deleteなど。

⑭3本の柱

機械が読める契約・安全な運用・発見可能性。8ルールはこの3つに集約。

⑮CLI vs MCP判断基準

コマンド数少ない・ステートレスならCLI優先。状態共有必要ならMCP。

⑯最小実装3つ

--json・終了コード整理・--dry-run。これだけで8割の効果。

⑰実践的な進め方

まずMCPで始めてパターンが固まったらCLIに落とす。2段階アプローチ。

⑱まとめ

エージェントが信じられるのは機械が読める契約だけ。JSONと終了コードが鍵。
