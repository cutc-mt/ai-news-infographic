エージェント設計パターンとは

①この動画で学ぶこと

  • マルチエージェントシステムを設計する方法
  • エージェントに反復・自己修正能力を与える
  • 適切なツールを動的に選択する仕組み
  • ADK（Agent Development Kit）での実装
前回: 単一エージェント → 順次・並列ワークフロー

②学ぶ3つのパターン

  • Loop Pattern: Review & Critique（反復改善）
  • Coordinator Pattern: Dynamic Routing（動的ルーティング）
  • Agent-as-Tool Pattern: サブエージェントをツールとして扱う

ADK実装 コード例付き

パターン1: Loop Pattern（Review & Critique）

③Loop Patternの概要

  • 特定の条件を満たす必要がある場合に使用
  • 例: ホテルと会場の移動時間30分以内
  • 条件を満たさない場合、フィードバックと共に再生成
  • 条件を満たすか最大反復回数に達するまでループ
自己修正能力を持つ反復改善フロー！

④Generator Agent（生成担当）

  • 初期の旅行プランを作成
  • Criticからのフィードバックを受け取る
  • フィードバックを元にプランを修正
  • 条件を満たすまで再生成を繰り返す
Generator Agent:
1. 旅行プランを作成
2. フィードバックを受け取る
3. 修正版を作成
4. 条件満たすまで繰り返し

⑤Critic Agent（評価担当）

  • 生成されたプランを条件に対して評価
  • 例: 移動時間が30分以内かチェック
  • 条件を満たさない場合却下
  • 改善のためのフィードバックを提供
条件未達 → 却下して再生成！
条件達成 → 承認！

⑥Loop Patternの実装

  • Critic AgentをLoopAgentで実装
  • Generator + CriticをSequentialAgentで結合
  • ADK Web UIでテスト可能（adk webコマンド）
  • Tracingタブでループの様子を確認
LoopAgent(critic_agent)
    ↓
SequentialAgent(generator, critic)
    ↓
条件満たすまでループ

⑦Loop Patternのメリット・デメリット

  • メリット: 品質基準・制約を確実に満たせる
  • 特定の基準を優先する場合に有効
  • デメリット: レイテンシーとコストが増加
  • 終了条件の設計に注意が必要

注意: 無限ループ防止

パターン2: Coordinator Pattern（Dynamic Routing）

⑧Coordinator Patternの概要

  • スマートなプロジェクトマネージャーのような役割
  • ユーザーリクエストを分析
  • 正しい専門エージェントに委任
  • これまでのワークフローは事前定義（順次・並列）
動的ルーティングで柔軟に対応！

⑨階層的タスク分解（Hierarchical Task Decomposition）

  • 大きなタスクを小さなサブタスクに分解
  • 各サブタスクを専門エージェントに委任
  • トップレベル: Coordinator Agent
  • 下層: 専門特化したサブエージェント
Coordinator Agent（トップ）
    ├── Food & Transportation Agent
    ├── Nearby Places Agent
    └── Trip Cost Agent

⑩サブエージェントの例

  • Food & Transportation Agent: 順次エージェント（前回構築）
  • Nearby Places Agent: 並列エージェント（前回構築）
  • Trip Cost Agent: 旅行費用計算
  • 各エージェントは独自のワークフローを実行
前回のエージェントを再利用！

⑪Coordinator Patternの動作例

  • 「サンフランシスコで寿司を見つけて行き方を知りたい」
  • Coordinator → Food & Transportation Agentに委任
  • 「博物館とコンサートも見つけたい」
  • Coordinator → Nearby Places Agentに委任（並列検索）
リクエスト → 分析 → 適切なエージェントへ
                ↓
         専門エージェントが実行

⑫Coordinator Patternのメリット・デメリット

  • メリット: 非常に柔軟、複雑な問題を分解可能
  • デメリット: ルーティング用のモデルコスト増
  • 多層構造は設計・トラブルシューティングが複雑
  • レイテンシーが増加

高柔軟性 高コスト

パターン3: Agent-as-Tool Pattern

⑬Agent-as-Tool Patternの概要

  • Coordinatorに似ているが重要な違いがある
  • 制御と状態管理のアプローチが異なる
  • サブエージェントを単純なステートレスツールとして扱う
  • メインエージェントが完全な制御を維持
違いは制御と状態管理！

⑭Coordinator vs Agent-as-Tool の違い

  • Coordinator: タスクを委任、サブエージェントが制御を持つ
  • Agent-as-Tool: ツールとして呼び出し、結果を受け取る
  • メインエージェントが全体の状態を管理
  • サブエージェントはステートレス
Coordinator = マネージャーが社員にプロジェクトを委任
Agent-as-Tool = 職人が道具を使って作業

⑮Agent-as-Toolの実装

  • Coordinatorの例をAgent-as-Toolとして書き換え可能
  • サブエージェントをツールとして定義
  • メインエージェントがツールを呼び出す
  • 結果を取得して次の処理へ
Main Agent
    ├── tool: food_agent()
    ├── tool: nearby_agent()
    └── tool: cost_agent()

ツール呼び出し → 結果取得 → 次の処理
まとめ: パターン選択ガイド

⑯パターン比較サマリー

  • Single Agent: シンプルなプロトタイプ
  • Sequential/Parallel: 信頼性ある構造化ワークフロー
  • Loop: 特定の基準を満たす必要がある
  • Coordinator/Agent-as-Tool: 複雑な問題を動的に解決

制御 柔軟性 コスト 複雑さ

⑰いつどのパターンを使う？

  • 品質基準が必須 → Loop Pattern
  • 複雑な問題を分解 → Coordinator Pattern
  • 制御を維持したい → Agent-as-Tool Pattern
  • シンプルな処理 → Sequential/Parallel Agent
それぞれトレードオフがある！
