GSD Claude Code Workflow AICodeKing

①GSDとは何か

コンテキストエンジニアリング層として機能。Claude Code、Codex、Gemini CLI、Cursor、Antigravityの上にインストールするワークフローシステム。

②コンテキストロット問題を解決

長いセッションでコンテキストが劣化し、モデルが忘れる、ランダムに変更する等问题を解決。spec-driven workflowで対応。

③簡単インストール

npx get-shit-done-cc@latestを実行。Mac、Windows、Linux対応。複数のランタイムを一括でサポート。

④map-codebaseコマンド

並列エージェントでアーキテクチャ、規約、スタックを分析。新機能を頼む前にプロジェクト構造を理解させる。

⑤new-projectフロー

質問、リサーチ、要件抽出、ロードマップ作成。project.md、requirements.md、roadmap.md、state.mdを生成。

⑥discuss-phase: グレーエリアを表面化

UIならレイアウト、密度、インタラクション。APIならレスポンス形式、フラグ、エラーハンドリングを事前に確認。

⑦plan-phase: 原子的タスク計画

新しいコンテキストウィンドウに収まるサイズでタスクを分割。大きな会話を覚えながら実装する問題を解消。

⑧execute-phase: 並列実行

依存関係に基づいてタスクをウェーブに分けて並列実行。独立したプランは同時に、依存するものは待機。

⑨アトミックGitコミット

各タスクで独立したGitコミットを作成。謎の変更の山ではなく、クリーンな履歴とロールバックポイントを提供。

⑩verify-work: 実際の成果物を検証

テストが通るだけでなく、ユーザーが実際に使えるかを確認。ログインできるか？オンボーディングは動くか？を検証。

⑪nextコマンド

次の論理ステップを自動判断。毎回どのコマンドを使うか考えなくても、ワークフローを維持可能。

⑫誰向けか

ソロ開発者、インディーハッカー、パワーユーザー。50人の企業を装うのではなく、構造を求める個人向け。

⑬小さなタスクには過剰

関数をリネーム、小さなバグ修正ならGSDは不要。タスクが大きくてコンテキスト管理が問題になる時に有効。

⑭権限スキップの注意

--dangerously-skip-permissionsの推奨は慎重に。信頼できるマシンと理解しているプロジェクトでのみ検討。

⑮コストの考慮

GSD自体はMITライセンスのオープンソース。ただし高価なモデルと並列エージェントを使うとトークンコストは増加。

⑯ハーネス設計の重要性

モデルの違いだけでなく、ワークフローの設計が重要。検証ループ、タスク分解の方法が結果を左右。

⑰ターミナルファースト

CLIベースでワークフロー重視。派手なGUIやオンボーディングダッシュボードはなし。学習曲線はある。

⑱結論: 大規模プロジェクトに最適

文脈管理がボトルネックになる中〜大規模機能に最適。カオスなエージェントと信頼できるエージェントの差はモデルではなくワークフロー。
