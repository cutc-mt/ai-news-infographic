概要: AIエージェントとナレッジベース

①AIエージェントのナレッジベースとは

  • AIエージェントが意思決定を行うために外部知識を参照する仕組み
  • RAG（Retrieval-Augmented Generation）の中核を担う
  • 社内文書、FAQ、製品情報などをエージェントに提供
  • LLM単体では持たない正確で最新の情報を補完
ナレッジベースがエージェントの「記憶」と「知識」になる！

②なぜBigQueryを使うのか

  • サーバーレスでインフラ管理が不要
  • PB級のデータを扱える圧倒的なスケーラビリティ
  • SQLで直接ベクトル検索が可能に
  • Google Cloudエコシステムとのネイティブ統合

サーバーレス SQLネイティブ Google Cloud

③アーキテクチャの全体像

  • Gemini: テキストのエンベディング生成
  • BigQuery: エンベディングの保存とベクトル検索
  • Vertex AI: エージェントの推論エンジン
  • ユーザークエリ → エンベディング → BigQuery検索 → 回答生成

ナレッジベースの設計とデータモデリング

④テーブル設計の基本

  • コンテンツ本体とエンベディング列を同じテーブルに保存
  • メタデータ（カテゴリ、更新日、ソース）も併せて管理
  • パーティショニングで大規模データに対応
  • テキストをチャンクに分割してエンベディング化
-- テーブル設計の例
CREATE TABLE knowledge_base (
  content_id STRING,
  content STRING,
  embedding ARRAY<FLOAT64>,
  category STRING,
  source_url STRING,
  updated_at TIMESTAMP
)

⑤テキストのチャンキング戦略

  • 長文を適切なサイズのチャンクに分割する
  • チャンクサイズは500〜1000トークン程度が一般的
  • オーバーラップ（重複）を設定して文脈の途絶を防ぐ
  • セマンティック境界（段落・セクション）を考慮
チャンクが大きすぎると検索精度が下がり、小さすぎると文脈が途切れる！適切なバランスが重要

⑥エンベディングの生成と保存

  • GeminiのエンベディングAPIを使用してベクトル化
  • BigQuery MLのML.GENERATE_EMBEDDING関数で直接生成可能
  • 生成したエンベディングはARRAY<FLOAT64>型で保存
  • text-embedding-004モデルが推奨（768次元）
-- BigQuery MLでエンベディング生成
SELECT *
FROM ML.GENERATE_EMBEDDING(
  MODEL `project.ml_model`,
  (SELECT content FROM docs)
)

⑦メタデータの活用設計

  • エンベディング検索だけでなくメタデータフィルタも活用
  • カテゴリ、日付、アクセス権などで検索範囲を絞り込み
  • ハイブリッド検索で精度を向上させる
  • ビジネスロジックに応じたカスタム列を追加可能
「営業部の資料だけ検索」「直近3ヶ月の情報だけ」といった絞り込みがSQLで簡単にできる！
BigQueryのベクトル検索機能

⑧VECTOR_SEARCH関数の使い方

  • VECTOR_SEARCH(): BigQueryのネイティブベクトル検索関数
  • 近似最近傍（ANN）検索をSQLで実行
  • コサイン類似度、ユークリッド距離などに対応
  • テーブルを直接検索できる宣言的なSQL構文
SELECT *
FROM VECTOR_SEARCH(
  TABLE knowledge_base,
  'embedding',
  (SELECT embedding FROM query_embeddings),
  top_k => 5
)

⑨リアルタイム検索の実現

  • BigQueryストリーミング挿入でリアルタイムデータ反映
  • エンベディング生成はCloud Functionsで自動化
  • 新規ドキュメント追加から検索可能になるまで数秒
  • BigQuery Materialized Viewsで事前計算も可能
ストリーミング挿入 + 自動エンベディング生成でほぼリアルタイムの検索が実現！

⑩ハイブリッド検索の実装

  • ベクトル検索 + キーワード検索を組み合わせる
  • SQLのWHERE句でメタデータフィルタを適用
  • 全文検索（FULLTEXT）との併用で精度向上
  • 検索結果のスコアの統合がポイント
-- ベクトル検索 + フィルタ
SELECT *
FROM VECTOR_SEARCH(
  TABLE knowledge_base,
  'embedding',
  query_embedding,
  top_k => 10
)
WHERE category = 'product'
  AND updated_at > TIMESTAMP_SUB(
    CURRENT_TIMESTAMP(), INTERVAL 90 DAY)
Geminiとの統合

⑪Gemini APIとの連携方法

  • BigQuery MLのML.GENERATE_TEXTで直接Geminiを呼び出し
  • 検索結果をコンテキストとしてプロンプトに渡す
  • Pythonクライアントライブラリからもシームレスに統合
  • Vertex AI SDKでエージェントのロジックを構築
-- BigQuery MLでGeminiを呼び出し
SELECT *
FROM ML.GENERATE_TEXT(
  MODEL `project.gemini_model`,
  CONCAT('検索結果: ', search_result,
         '\n質問: ', user_question)
)

⑫エージェントとのRAGパイプライン

  • ステップ1: ユーザー入力をエンベディング化
  • ステップ2: BigQueryで関連ドキュメントを検索
  • ステップ3: 検索結果をプロンプトに挿入
  • ステップ4: Geminiがコンテキストに基づいて回答生成
このパイプライン全体をSQL + BigQuery MLで完結できるのが最大のメリット！
スケーラビリティとコスト管理

⑬スケーラビリティの設計

  • BigQueryは自動スケーリングでデータ量に対応
  • 数百万件のエンベディングでも高速に検索可能
  • パーティショニングでコストとパフォーマンスを最適化
  • クラスタリングで検索範囲を絞り込み

自動スケーリング パーティショニング クラスタリング

⑭コスト管理のベストプラクティス

  • エンベディング生成: 挿入時の1回のみでコスト抑制
  • 検索コスト: top_kを適切に設定（5〜10件程度）
  • 定期的に不要データのパージを実施
  • BigQueryの容量ベースの料金モデルを活用
エンベディングの次元数を下げる（256次元）とストレージと検索コストを大幅削減できる！

⑮クエリの最適化テクニック

  • BRIN索引: ベクトル検索のパフォーマンスを向上
  • 検索対象のテーブルを事前にフィルタして範囲を限定
  • マテリアライズドビューで頻繁なクエリを事前計算
  • キャッシュを活用して重複クエリのコストを削減
-- フィルタ適用で検索範囲を絞り込み
SELECT *
FROM VECTOR_SEARCH(
  (SELECT * FROM kb
   WHERE department = 'engineering'),
  'embedding', query_emb,
  top_k => 5)
実践的な実装とベストプラクティス

⑯実装例: エンドツーエンドのワークフロー

  • 1. データ取り込み: Cloud StorageからBigQueryにロード
  • 2. チャンキング: テキストを適切なサイズに分割
  • 3. エンベディング生成: ML.GENERATE_EMBEDDINGでベクトル化
  • 4. 検索・回答生成: VECTOR_SEARCH + ML.GENERATE_TEXT
# Pythonでの実装イメージ
from google.cloud import bigquery
client = bigquery.Client()

# ベクトル検索の実行
query = """
  SELECT content, distance
  FROM VECTOR_SEARCH(...)
"""
results = client.query(query)
