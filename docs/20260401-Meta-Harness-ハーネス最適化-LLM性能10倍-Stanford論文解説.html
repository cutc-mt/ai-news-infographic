同じAIでも性能が10倍変わる「ハーネス」とは？

Stanford発の最新論文「Meta-Harness」を徹底解説

2026年4月1日 ｜ 論文: Meta-Harness: End-to-End Optimization of Model Harnesses

① 同じAIでも10倍の性能差が生まれる理由

同じGPTを使っていても、あなたより10倍早く仕事を終わらせる人がいる。その差はモデルの違いではない。

モデルを一切変えずに、ハーネスを最適化しただけで6〜10倍の性能変化が起きる。これがMeta-Harnessが証明した衝撃の事実だ。

ハーネス（Harness）＝ LLMを実際のタスクで動かすための外部コード全体のこと

② ハーネスの6つの構成要素

プロンプトはハーネスのほんの一部に過ぎない。ハーネスには以下の要素がすべて含まれる：

システムプロンプト
ツール定義
実行フロー制御
コンテキスト管理
トライロジック
メモリシステム

例：Claude Codeの裏側で動いている仕組み全体がハーネス

③ プロンプト効果 vs ハーネス効果：氷山の一角

プロンプトエンジニアリングはテキスト入力の最適化。この風をこう聞いた方がいい回答が出る、という話。

ハーネスエンジニアリングはシステム全体のアーキテクチャを対象にする。数百万行のコード、複雑なワークフロー、データ構造まで含めて最適化する。

山の一角がプロンプト。
水面下の巨大な部分がハーネス。
スコープが全然違う！

④ 驚きの実データ：編集フォーマットだけで10倍に

Harnessの報告によると、編集フォーマットを変えただけで性能が劇的に変化した：

6.7% → 68.3%
モデルは同じ。プロンプトも大きくは変えていない

変えたのはコードの編集結果をどういう形式でLLMに渡すかだけ。それだけで10倍。

ハーネスの設計次第で、同じモデルの性能が天と地ほど変わる

⑤ Harness社のTerminalBench 2.0での快挙

AIフレームワーク会社のHarnessがTerminalBench 2.0というベンチマークで大躍進：

変更前
52.8%
変更後
66.5%

+13.7ポイントの向上で一気にトップ5入り。モデルは一切変えていない。ハーネスを最適化しただけ。

⑥ ハーネスは「手作業」だった：従来の限界

ハーネスは今までほとんど手作業で設計されてきた。専門家が試行錯誤して何週間も何ヶ月もかけて調整する。

プロンプトの最適化はDSPyやACE、MCEなど自動化の研究が進んでいるが、ハーネス全体の最適化はほとんど手つかずだった。

これがMeta-Harnessが解決しようとしている問題の核心だ。

従来手法の2つの限界

⑦ 限界①：情報圧縮の罠（完結性バイアス）

従来の手法は実行トレースを要約してからLLMに渡していた。これが大きな罠だった。

要約しようとすると、LLMはドメイン知識を削って簡潔にしてしまう。論文ではこれを「完結性バイアス」と呼んでいる。

エラー例:
生のログ → CUDA Out of Memory.
 Tried to allocate 2.5GB
要約後 → GPUメモリーエラーが発生
（2.5GBという具体的な数字が消失！）
数字がないと「バッチサイズを半分にしよう」という具体的な対策が立てられない

⑧ 限界②：長期的信用割当問題

ハーネスは数百万行のコードで構成されていて、実行トレースも膨大。最終的に失敗しても、その原因がコードのどこにあるか特定するのが難しい。

100行目で間違えたせいで3000行目でエラーになる。この因果関係を追うのがとても難しい。

従来手法は最大でも26Kトークン程度の情報しか使えず、圧縮するか切り捨てるしかなかった。

複雑なエージェントの実行トレースには全然足りない情報量だった

⑨ Meta-Harnessの革新的アプローチ：ファイルシステム方式

従来は情報をLLMのコンテキストウィンドウに詰め込もうとしていた。Meta-Harnessは発想を逆転した。

すべてをファイルに保存しておく

必要な時だけ見に行く

つまり、人間がフォルダーを開いて過去のログを見るのと同じ仕組みだ。

26K → 1000万
使える情報量が約400倍に跳ね上がった

⑩ 選択的アクセス：人間のエンジニアと同じアプローチ

1000万トークン全部読んでいたら時間がかかりすぎる。だから選択的アクセスが重要だ。

grepやcatといったターミナルコマンドで必要な情報だけを取りに行く。人間のエンジニアがデバッグする時と全く同じ。

アクセスパターン:
grep → 特定の失敗を検索
cat  → 関連のログを詳細に読む
kit config.json → 設定を確認
OG *.log → ログ全体を俯瞰

1イテレーションあたり中央値で82ファイルを読み取り、過去の候補ハーネスを幅広く参照している。

⑪ Agentic Proposer：Claude Codeがハーネスを自動設計

Meta-HarnessではClaude Codeをプロポーザー（提案機）として使用。Claude Codeがファイルシステムを探索して、新しいハーネスを提案する。

ファイルシステムを探索

失敗モードを特定

ターゲットを絞った修正を提案

すべての提案が過去の実行のエビデンスに基づいている。感覚ではなくデータドリブンだ。

Claude Codeがハーネスを自動で書いて最適化してくれる！

3つのドメインでの実験結果

⑬ テキスト分類：+16ポイント、トークン75%削減

BENTSという中国の法律文書分類215クラスという難しいタスクでの結果：

ACE
35.2%
Meta-H
51.2%

+16ポイントの改善。しかもトークン使用量はACEの1/4。精度が上がってコストは下がった。

何を見せるかを賢く選ぶことで、少ない情報でより正確な判断ができる

⑭ 数学推論：IMOレベルで+4.7ポイント、5モデルで汎化

数学オリンピック（IMO）レベルの問題50問で評価。平均で+4.7ポイントの改善。

Before
34.4%
After
39.1%

IMOレベルで4.7ポイントはかなり大きい。そして5つの異なるモデルすべてで一貫して改善した。

つまり、発見されたハーネスは特定モデルに最適化されすぎておらず、汎用的な原則を捉えている。

⑮ コーディングエージェント：76.4%でGPT-5超え

TerminalBench-2で89のDocker化されたタスクを評価。ソフトウェアエンジニアリング、システムプログラミング、暗号解析など多岐にわたる。

76.4%
Claude Opus 4.6 + Meta-Harness

さらにHaiku（小さいモデル）でも37.6%で最高スコアを獲得。同じコスト帯のモデルの中で1位。

ハーネスを最適化すれば、小さいモデルでも勝てる！コスト効率の面でもインパクトが大きい

⑯ 発見された3つのハーネス設計パターン

テキスト分類 → LARIクエリー（ラベル空間を先に提示し、紛らわしい例を対比的に見せる）

数学推論 → 4ルートエキスパートルーター（代数・数論・整数論・組み合わせ論に振り分け）

コーディング → 環境ブートストラッピング（タスク開始前に環境情報を自動収集してコンテキストに追加）

環境ブートストラッピングだけで探索ターンが4ターン減ってタスク成功率が向上。シンプルだけど効果的
