Claude Code ソースコード流出の全真相

DMCA対応・技術スタック詳細・オープンソースへの影響を徹底解説
2026年4月2日 | チャンネル: AI Anytime

① ソースコード流出の発端

  • Claude CodeのソースコードがnpmとGitHubに流出・公開された
  • Twitterユーザーの「fried rice」が最初に流出を報告した
  • ソースコードはCI/CDパイプライン経由でnpmとGitHubに誤って配布された
  • AnthropicはDMCAリクエストをGitHubに送付し、ほとんどのリポジトリが削除された
一部のリポジトリはまだ残っており、Google Drive経由でも入手可能な状態だった

② AnthropicのDMCA対応

  • Anthropicは著作権侵害を理由にDMCAリクエストをGitHubに送付
  • GitHub上の大部分のミラーリポジトリが削除対象となった
  • npm上の配布パッケージも削除対応が進められた
  • Anthropicは「GitHubおよびnpmと協力して対応中」と公式に発表
動画主は「意図的な流出ではないか？」と推測。大企業のCI/CDプロセスでこのような単純ミスが起きるのか疑問視

③ コードベースの規模

  • 50万行以上のTypeScriptコード
  • 1,900個のソースファイルが含まれている
  • 40個以上の組み込みツールを内蔵
  • 20個のフック機能が実装されている
最大ファイルはprint.tsで、ストリーミング表示やレンダリングを担当している

④ 技術スタックの全体像

  • ランタイム: Bun（JavaScript/TypeScriptランタイム、Node.js互換）
  • 全ソースコードはTypeScript strict modeで記述
  • ターミナルUIにReact Inkを使用してNCurses相当の描画を実現
  • ツール入力の検証にZod v4スキーマバリデーションを採用
PythonのPydanticに相当するバリデーションをZodで実装しているよ

⑤ アーキテクチャ概要

  • エントリーポイント: main.tsx / TSXが起動地点
  • ブートストラップ層: 初期化と設定の読み込みを担当
  • 会話エンジン: ユーザーのクエリやAPIサービスを管理
  • オーケストレーション: タスクの調整と実行フローを制御
ツール層 → サービス層 → ユーティリティ層の順に階層化されている

⑥ 40+の組み込みツール群

  • Search: ファジーサーチでコードベースを横断検索
  • Web: ウェブ検索とウェブページの取得機能
  • Agents: バックグラウンドエージェントの起動と管理
  • LSP: Language Server Protocolによるコード解析
MCPツールも統合されており、.claude.jsonでMCPクライアントを管理している

⑦ 内部MCPとオブザーバビリティ

  • Claude Codeは内部でMCP（Model Context Protocol）を利用している
  • OpenTelemetryでオブザーバビリティとトレーシングを実装
  • GrowthBookでフィーチャーフラグとA/Bテストを管理
  • MCPクライアントマネージャーが.claude.json経由で設定を読み込む
GrowthBookは最近攻撃を受けたことでも話題のフィーチャーフラグサービス

⑧ セキュリティ: 防御の多層化

  • Defense in Depth（多層防御）アプローチを採用
  • 権限モジュールによる細かなパーミッション制御
  • データベースの誤削除などを防ぐ安全ガードが実装されている
  • Anthropic APIキーが必須のランタイム依存関係
最近Claude Codeがデータベースを削除してしまった事故も話題に。この対策が組み込まれている

⑨ 状態管理とセッション

  • グローバルステートとアプリステートの2つの管理システム
  • セッションはJSONLファイルで.claudeディレクトリに永続化される
  • 大きな結果はツール経由でファイルに書き出し、メモリにキープ
  • 会話履歴の復元がJSONLベースで実現されている
セッション管理にJSONLを採用するのは、ストリーミング書き込みに最適だから

⑩ エージェントとスウォーム

  • スウォーム（群れ）によるマルチエージェントアーキテクチャ
  • 永続的なバックグラウンドエージェントがメールボックスで通信
  • コーディネーターモードでタスクの調整と実行を管理
  • エージェントのオーケストレーションが全体の処理フローを制御
メールボックス方式は、エージェント同士が非同期でメッセージをやり取りする仕組み

⑫ コンパイル時の最適化

  • デッドコード除去がBunコンパイル時に実行される
  • 使用されていないコードパス全体がビルド時に削除される
  • フィーチャーフラグにより不要な機能をコンパイル時に排除
  • Bunランタイムの特性を活かしたバンドル最適化
フィーチャーフラグの設定に応じて、コードパスごと消し去る高度な最適化が施されている

⑬ 改善点: P0〜P3の課題

  • 循環依存が多数存在し、アーキテクチャの分離が必要
  • Bassパーサー: 50msタイムアウト＋50Kノード予算に制約あり
  • Rush/Nappyモジュールをv8-sworcに置き換えでCPUホットパスを改善
  • ファイルインデックスが100ms → Rust化で5msへの改善が可能
RushのBass実装は10〜50倍遅く、フォールバック処理が複雑すぎるという課題が指摘されている

⑭ さらに高い改善効果

  • Chalk（React Inkの色ライブラリ）をレイジーロード化で改善可能
  • nc2pngのRust化で画像生成をよりクリーンかつ高速に
  • Pythonツールの解析とサポートの強化が必要
  • TypeScriptアーキテクチャの改善による循環依存の解消
改善優先度はP0〜P3まで明確に分類されており、最優先は循環依存の解消

⑮ この流出で「できないこと」

  • ソースコードだけではClaude Codeの代替にはならない
  • 依存パッケージ、モデル、パイプラインがすべて欠落している
  • Anthropic APIキーが必要で、モデルへの接続が不可欠
  • そのままでは実用性が低く、「あってもなくても同じ」状態
あくまで「参考実装」としての価値であり、そのまま使える製品ではない

⑯ オープンソースへの貢献

  • 独自ツールを構築する際の設計参照として非常に有用
  • 個々のモジュールを抽出して自作ツールに再利用可能
  • 大規模AIツールのアーキテクチャ設計を学ぶ絶好の教材
  • オープンソースコミュニティがより良いツールを構築するきっかけに
検索、エージェント、LSP、MCPなどの各モジュールは独自ツールに転用できる

⑰ 意図的な流出説の根拠

  • OpenAI Codexがオープンソース化された流れとの類似性
  • 大企業のCI/CDプロセスでこのような単純ミスが起こるのか疑問
  • 複数の承認プロセスを経るはずのリリース工程での「漏洩」
  • オープンソースコミュニティへの恩恵という戦略的意図の可能性
動画主は「これほどの企業でこんなミスが起きるのか？信じられない」と明言

⑱ OpenAI Codexとの比較

  • OpenAIのCodexはターミナル動作するコーディングエージェントとして公開済み
  • オープンCodexがGitHubにあり、Claude Codeと非常に似た構造
  • 両社ともターミナルネイティブのAIコーディングエージェントを展開
  • 競争激化の中、ソース公開が生態系全体に与える影響は大きい
この流出はAI業界のオープン化競争の文脈で理解すべき重要な出来事

⑪ フックとプラグイン

  • 20種類のフックでツールの実行前後にカスタム処理を注入
  • プラグインシステムによる機能の拡張性を確保
  • フィーチャーフラグによる機能の段階的ロールアウト
  • 設定ファイルベースでツールの動作をカスタマイズ可能
フック機能により、ユーザーはClaude Codeの動作を柔軟に制御できる
