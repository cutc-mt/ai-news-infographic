🤖 成長するAI秘書「Nina」

Claude Codeで事業企画の仕事をオーケストレーションで回す

Claude Code AI秘書 オーケストレーション DevelopersIO

①事業企画の仕事と課題

著者の仕事は「仮説を作って、ぶつける相手を見つけて、検証して、サービスにする」という4フェーズを一人で回すこと。問題は、情報収集に追われて「考える時間」がどんどん削れていくこと。朝ニュースを読んで仮説を思いついても、Slackの問い合わせで午前が終わり、午後に提案書を書こうとした時には朝の記事を忘れている。

②AIに解決できない問題

AIは「今この瞬間の答え」は出せても、「昨日の失敗を踏まえた答え」は出せない。この問題を解決するために、失敗から学んで自分で賢くなっていく仕組みが必要だった。情報収集・仮説の素案・提案書の下書きまでを誰かに任せて、自分は「判断と関係構築」に集中したい。

③OpenClawからClaude Codeへ

最初はAIエージェントオーケストレーションツール「OpenClaw」を試した。複数のAIに役割を与えてチームとして動かす仕組みで、5日間で動作を確認。しかし自分でサーバーを用意してJSONを手で直す必要があり、非エンジニアには「動かし続けること」自体のコストが高すぎた。

④Claude Codeサブスクリプションの利点

Claude Codeサブスクリプションは月額固定で使い放題。API従量課金が不要なので、エージェントを何回動かしてもコストが変動しない。5時間のローリングウィンドウ制限はあるが、OpenClawのチーム設計コンセプトをClaude Code単体で実現できる。

⑤Claude Codeの5つの機能

Claude CodeをNina構築に活用する5つの機能：CLAUDE.md（人格・判断基準を定義）、agents/（専門エージェントの定義）、skills/（繰り返しワークフロー）、hooks/（イベント自動実行）、memory/（セッションをまたぐ記憶）。これらを駆使してNinaのプロジェクトを設計した。

⑥Ninaのプロジェクト構成

ai-nina/フォルダ以下に、CLAUDE.md、agents/（researcher・writer・reporter）、skills/（morning-news・daily-report）、hooks/（guard-dangerous-bash・log-action・session-end-log）、memory/（nina-level・boss-patterns・guardrails）、rules/（domain-knowledge）を配置。出力先としてhypotheses/、logs/、outputs/、tasks/を用意。

⑦CLAUDE.mdで人格を定義

CLAUDE.mdに「You are Nina, AI secretary for kamedas」と記述。Ninaは上司が誰で、上司の仕事が何かを知った状態で動く。判断の優先順位として「安全性 > 倫理 > ルール遵守 > 有用性」を定義し、アクセスできるディレクトリ範囲とタスクの分類ロジックも設定。

⑧3つのサブエージェント編成

Ninaは自分では作業せず、オーケストレーターとして3つのサブエージェントに委託。researcher（Sonnet）は情報収集・競合分析、writer（Sonnet）はブログ・提案書の作成、reporter（Haiku）は日報・レポート作成。reporterはコストを抑えるためにHaikuモデルを使用。

⑨日中の運用：指示ベースで動く

日中はClaude Codeのセッションで著者がNinaに指示を出す。「この企業を調査して」「この仮説でブログの下書きを書いて」といった依頼に対して、Ninaがresearcherやwriterに委託して結果を返す。よく使うワークフローは「スキル」として定義し、再利用可能にしている。

⑩/morning-newsスキルの仕組み

/morning-newsを実行すると、researcherエージェントが3つの検索クエリで業界ニュースを収集。各記事からGrowth Pack for LINEに関連する仮説を生成し、ファイルに保存する。従来30〜60分かかっていた情報収集の工程が、判断だけですむようになった。

⑪夜間の自律運転スクリプト

夜間はnina-nightshift.shで自律運転。タスク間に10分の間隔を入れ、1晩の最大実行回数は20回（レート制限保護）。Claude Code Maxの5時間制限に対応するため、深夜1時起動・6時前完了のスケジュールに設定。MacBookのスリープはcaffeinateコマンドで防止。

⑫レベルシステムの再設計

OpenClawのPoCでは「ニュース投稿3回連続成功 → レベルアップ」だったが、Claudeに相談したところ「作業の正確性」ではなく「判断の質」を測るべきという指摘を受けた。そこで「仮説生成」「企業調査」「ブログ下書き」など業務ごとにスキルレベルを分け、3回連続で「修正不要」と判断されたら昇格する仕組みに変更。

⑬5トリガーの学習プロトコル

5つの学習トリガーを定義。①上司フィードバック（承認・修正・却下時に記録）、②判断を変える発見（調査中の重要事実）、③失敗→ガードレール（失敗時にルール化）、④業界パターン抽出（同業種3社以上で発動）、⑤仮説の結果記録（使われた/使われなかった時）。①〜③は防御的、④〜⑤は攻撃的な学習。

⑭ガードレールの自動蓄積

失敗が起きるとguardrails.mdに自動記録される。現時点で4つのガードレールがあり、「事実確認の徹底」「成果物の実在確認」「進捗の定期報告」「機密情報のマスク処理」。全て運用中に起きたインシデントへの対策で、事前に想定して書いたものは一つもない。追加後、同じカテゴリの失敗は再発していない。

⑮Observe → Learn → Improveのループ

セッション中に5つのトリガーに基づいてメモリファイルを即時更新。次のセッションでNinaがそれを参照して判断する。夜間の学習セッションでは、各Issueの調査で得た発見ログをメモリに転記し、業界パターンとして昇格させる。hooksはcommand型のシェルスクリプト3つに絞り、学習ロジックはCLAUDE.mdに集約する設計。

⑯成果：事業企画が変わった3つのこと

①情報収集の工程がなくなった：/morning-newsで判断だけですむ。②提案書をゼロから書かなくなった：企業調査→課題仮説→構成案→ドラフトのフローをNinaに委託。③同じミスの再発がなくなった：ガードレール追加後、同カテゴリの失敗はゼロに。一方、仮説の質はまだ「使える」が半分程度。

⑰新人育成と同じアプローチ

振り返ると、やっていることは新人育成と同じだった。役割を明確にして、専門知識を渡して、失敗したらフィードバックして、成功パターンを記録して、次に活かす。AIは一度記録したことを忘れないので、フィードバックが蓄積するほど判断の精度が上がる。事業企画に限らず、情報収集と仮説構築を繰り返す仕事なら同じアプローチが使える。

⑱コード不要で「成長するAI秘書」を構築

コードが書けなくても、Claude Code Maxのサブスクリプション範囲内で設定ファイル（MarkdownとJSON）だけで構築できる。最初から完璧に設計する必要はない。Ninaも最初はレベルシステムという間違った設計から始まった。失敗して、学んで、作り直す。そのプロセスをAIにも組み込んだことが、今回の一番の成果。
