Context Engineeringで要件定義を設計する

AIに「何を作るか」を整理させる5つのプロンプトパターンと実装例

①「何を作るか」が曖昧なままだとどうなるか

AIコーディングツールは確かにコードを書くスピードを10倍にする。しかし「何を作るか」が曖昧なまま10倍速で書くと、10倍速で「間違ったもの」が完成する。これは笑い話ではなく、実際に多くの開発現場で起きている。

②手戻りの根本原因は要件定義にあり

ソフトウェア開発の手戻り原因を調べると、かなりの割合が要件定義フェーズの不備に起因している。設計ミスでも実装バグでもなく、そもそも「作るべきもの」の定義がズレていたことが原因だ。

③Context Engineering（文脈の設計）とは

Prompt Engineeringが「AIへの聞き方」を工夫するのに対して、Context Engineeringは「AIに何を知らせた上で聞くか」を設計する。AIに「どう作るか」を任せる前に、「何を作るか」をAIに整理させるアプローチだ。

④要件定義の3つの文脈レイヤー

要件定義においては文脈を3つのレイヤーで整理する。ビジネス文脈・ユーザー文脈・技術文脈の3つだ。これらをAIに渡した上で「要件を整理して」と頼むと、何も渡さない場合と出力の質が全く違う。

⑤レイヤー1: ビジネス文脈

この機能を作る目的は何か、成功をどう測るか（KPI・売上・ユーザー数）、予算と期限の制約を整理する。例えば「カート離脱率を70%から50%に下げたい。Q3までにリリースが必要」のように具体化する。

⑥レイヤー2: ユーザー文脈

誰が使うのか（ペルソナ）、どんな状況で使うのか、何に困っているのかを整理する。例えば「30代の忙しい共働き家庭。スマホで隙間時間に買い物。操作が3ステップ超えると離脱する」のように具体的に描写する。

⑦レイヤー3: 技術文脈

技術スタック（言語・フレームワーク・インフラ）、既存システムとの連携、非機能要件（パフォーマンス・セキュリティ）を整理する。例えば「Next.js + TypeScript + MongoDB。Stripe決済連携。1秒以内のレスポンスが必須」のように明記する。

⑧パターン1: 曖昧な要望を構造化要件に変換

「カート機能をもっと使いやすくしたい」というふわっとした要望を、要件ID・要件名・説明・優先度・受入条件（Given-When-Then）に変換する。ビジネス文脈を添えることで、「使いやすくしたい」が「ワンタップ数量変更（200ms以内）」に具体化される。

⑨パターン2: ペルソナ駆動のユーザーストーリー

具体的なペルソナ（例：佐藤さん32歳・2児の母・通勤電車内でスマホ操作）を設定し、INVEST原則に従ってユーザーストーリーを生成する。「〜として、〜したい。なぜなら〜だから。」の形式で出力させるのがポイントだ。

⑩パターン3: 受入条件を自動生成

ユーザーストーリーに対して正常系だけでなく異常系・境界値も含めた受入条件を生成する。AIは指示しないとハッピーパスだけを返す。在庫切れ、ネットワークエラー、境界値（最小1・最大99）なども明示的に要求することが重要だ。

⑪パターン4: 要件の矛盾・漏れを検出

「オフライン対応」と「リアルタイム在庫反映」など、要件間の矛盾をAIが見つけてくれる。人間がやると見落としがちだが、AIは全要件を同時に見渡せるため矛盾検出が得意だ。ただし最終判断は人間が行う。

⑫パターン5: MoSCoW法で優先順位を提案

Must（絶対必要）・Should（できれば欲しい）・Could（あったら嬉しい）・Won't（今回はやらない）の4段階で優先度を分ける。AIの提案は「叩き台」であり「正解」ではない。プロジェクト制約とビジネス目標を添えて提案させる。

⑬TypeScriptで要件定義アシスタントを実装

OpenAI APIとZodを使って型安全な要件定義アシスタントを構築する。RequirementsContextインターフェースで3つの文脈レイヤーを構造化し、generateUserStories関数でユーザーストーリーを自動生成する。

⑭ZodスキーマでAI出力を型安全に検証

UserStorySchemaでid・title・story・acceptanceCriteria・priority・estimatePointsを定義し、AIの出力を必ずスキーマバリデーションする。AIの出力は常に不安定なので、検証を省略すると「いつか壊れる」と筆者は警告している。

⑮INVESTフレームワークで品質チェック

生成されたユーザーストーリーがIndependent・Negotiable・Valuable・Estimable・Small・Testableの6つの基準を満たしているかをAIに評価させる。各項目について「OK」か「要改善」を判定し、改善案も出力する。

⑯アンチパターン: AIへの丸投げと機密情報漏洩

「いい感じに要件定義して」と丸投げすると文脈がゼロなので一般論しか返ってこない。また売上数字・顧客情報・APIキーなどの機密情報をフィルタリングせずにLLMに渡すのは危険だ。マスキングと匿名化が必須である。

⑰アンチパターン: AI出力の無検証利用

AIはもっともらしい嘘をつく（ハルシネーション）。要件定義でこれが起きると、存在しない技術的制約やビジネス上ありえない仮定が紛れ込む。違和感を感じたら立ち止まること。ドメインエキスパートのレビューを必ず通す。

⑱人間×AIの役割分担

AIは構造化・網羅・パターン生成が得意。人間は文脈理解・価値判断・最終承認が得意。AIは「思考のブースター」であって「意思決定の代替」ではない。この境界線を間違えなければ要件定義の質もスピードも上がる。
