コンテキストエンジニアリングの基礎

① Gemini.mdファイルとは何か

  • Gemini.mdはGemini CLIに常に読み込まれるテキストコンテキストファイル
  • プロジェクトレベルはプロジェクトフォルダ直下に配置し、そのフォルダ内でのみ有効
  • ユーザーレベルは ~/.gemini/ フォルダに配置し、すべてのセッションで有効
  • システム命令として常にプロンプトに含まれるため、コーディングガイドラインや設計指針の保存に最適
Gemini.mdに保存しておけば、セッションを閉じても次回から自動的に読み込まれるよ！

② 短期メモリと長期メモリの使い分け

  • 短期メモリはセッション内の会話履歴や、その場で読み込んだローカルファイルのこと
  • 長期メモリはGemini.mdに保存され、セッションをまたいでも永続的に利用可能
  • ローカル設計ドキュメントは柔軟な「グレーゾーン」で、使い方によって短期にも長期にもなる
  • プロジェクトをまたぐ汎用的な指針はユーザーレベルに、プロジェクト固有のものはプロジェクトレベルに配置する
コンテキストウィンドウを無駄に消費しないよう、何を永続化するか慎重に選ぶのがポイント！

③ Agent Skillsでオンデマンドな専門知識を呼び出す

  • Agent Skillsは必要なときだけ動的にロードされる専門知識の仕組み
  • 例えれば「緊急時に呼ぶ配管工」のような存在で、常にそばにいる必要はない
  • skill.mdファイルに名前・説明・コンテンツを定義し、~/.gemini/skills/ に配置
  • Gemini CLIが質問を判定し、関連するスキルを自動的にアクティベートする
# skill.md の構造
name: "ADK Agent Design"
description: "ADKエージェント設計の指針"
content: "エージェント設計原則..."
コンテキストウィンドウを節約しつつ、精度を落とさない効率と性能のバランスが取れる仕組み！

④ コンテキストの3つの管理戦略を比較

  • Gemini.md: 毎回自動ロード、汎用的なコーディングガイドラインに最適
  • Agent Skills: 必要な時だけ動的ロード、特定分野の専門知識に最適
  • ローカルファイル: プロンプトで明示的に参照、一度きりの設計ドキュメントに最適
  • 正解はないので、プロジェクトに合わせて柔軟に使い分けるのが良いアプローチ
コンテキストを賢く管理する「コンテキストエンジニアリング」こそが、Vibe Codingを成功させる鍵！
ADKエージェントの構築

⑤ 設計ドキュメントをダウンロードしてエージェントの設計図を準備

  • Gemini CLIに自然言語で指示するだけで、GitHubからagent.design.mdをダウンロード
  • ドキュメントにはルートエージェント、使用モデル、名前、命令、コンポーネント構成が記載
  • ADK（Agent Development Kit）のPythonスクリプト構造が定義されている
  • 「git clone」や「curl」を覚える必要がなく、自然言語で指示できるのがVibe Codingの強み
AIに正しいコンテキストを与えれば、ADKの知識がなくてもエージェントを作れる！

⑥ Gemini CLIにagent.pyを自動生成させる

  • Gemini.mdのコーディングガイドラインとAgent Skillの設計指針を組み合わせてagent.pyを生成
  • インポート、環境変数の読み込み、ルートエージェントの定義が含まれる
  • モデルは確率的なため、指示に従わないこともある。その場合はプロンプトを微調整して再試行
  • Gemini 3モデルは指示追従が改善されており、より正確なコード生成が期待できる
# 生成されたagent.pyの構造
from google.adk import Agent
root_agent = Agent(
    model="gemini-2.5-flash",
    name="shadow_blade",
    instruction="..."
)
モデルが失敗しても、プロンプトを少し変えるだけで成功することが多い。Vibe Codingはアートだ！

⑦ MCP Serverにツールを追加してエージェントを強化

  • MCP Serverはエージェントが使用するツール（武器）を提供する外部サーバー
  • Shadow Bladeエージェント用にハッキング鎌、クイックスタート弩、黄金のガベルなどの武器ツールを追加
  • 既存のツールパターンに従って、Gemini CLIに新しいツールの追加を指示するだけ
  • ADKはMCP 2ツールセット経由でMCPサーバーに接続する仕組み
ツールの追加も自然言語で指示するだけ。既存パターンを再利用するのがコツ！

⑧ ADK Runでエージェントをリアルタイムテスト

  • adk runコマンドでターミナル上のインタラクティブCLIを起動してエージェントと対話
  • adk webを使えばブラウザ上の開発UIでマルチモーダル入力も可能
  • テストでは「完璧主義の怪物を攻撃」「ドグマを排除」などで武器ツールの動作を確認
  • 仮想環境を作成し、必要なパッケージをインストールしてから実行する必要がある
対話テストだけでは不十分。次はプログラムで自動評価する仕組みが必要！
エージェントの評価とテスト

⑨ ADK評価の仕組み: ゴールデントラストデータセット

  • 期待される軌道（どのツールをどの順番で呼ぶか）と期待される応答を事前に定義
  • 実際の軌道と応答をゴールデントラースと比較し、評価レポートを生成
  • eval_set.jsonにユーザーコンテンツ、最終応答、中間ツール使用を記録
  • 評価データはGemini CLIにQAとして作成させることも、ADK Web UIの会話から生成することも可能
// eval_set.json の構造
{
  "user_content": "攻撃を実行して",
  "expected_response": "ターゲットを排除",
  "expected_tool_uses": [
    {"tool": "attack", "params": {...}}
  ]
}

⑩ 評価メトリクス: 軌道分析と応答マッチ

  • 軌道平均スコアはツール使用の正確さを評価。ツールの順序と種類が一致しているかを確認
  • 応答マッチスコアはROUGEアルゴリズムで最終応答の類似度を0〜1で測定
  • しきい値は用途に応じて調整。応答は低めに、軌道はエージェントの複雑さに応じて設定
  • 他にもハルシネーション検出、安全性評価など多数の評価基準がADKに用意されている
医療用途などでは安全性評価が必須。用途に合わせて評価基準を選ぼう！

⑪ Pytestで評価を自動化してCI/CDに組み込む

  • Pytestを使えばadk evalを自動化し、CI/CDパイプラインに統合できる
  • @pytest.mark.asyncioデコレータで非同期テストを定義し、評価基準とテストデータを1ファイルにまとめる
  • 評価が失敗したエージェントはデプロイしないという安全策が実現できる
  • Python以外にもGoやJava用のADKエージェントがあり、それぞれに合ったテストフレームワークを使う
@pytest.mark.asyncio
async def test_agent():
  evaluator = AgentEvaluator(...)
  result = await evaluator.evaluate(
    eval_set="sample_eval_set.json",
    criteria={"response_match": 0.5}
  )
  assert result.passed

⑫ Gemini CLIで開発ライフサイクル全体をAIに委ねる

  • Gemini CLIでADKエージェントを生成
  • Gemini CLIでテストデータを生成
  • Gemini CLIでPytestテストコードを生成
  • Gemini CLIでCI/CDパイプラインを構築
開発ライフサイクルの全工程をGemini CLIで完結できる。これがVibe Codingの真の力！
CI/CDパイプラインとCloud Runデプロイ

⑬ Cloud BuildでCI/CDパイプラインを構築

  • cloudbuild.yamlにビルドステップを定義し、パイプラインを自動化する
  • ステップ1: Pytestを実行してエージェントの評価を自動化
  • ステップ2: コンテナイメージをビルドしてArtifact Registryに保存
  • テストが失敗すればデプロイは行われない。動かないエージェントを本番に置く無駄を防ぐ
# cloudbuild.yaml の流れ
steps:
  - name: "テスト実行"
    entrypoint: "pytest"
  - name: "コンテナビルド"
    args: ["build", "-t", "IMAGE"]
  - name: "プッシュ"
    args: ["push", "IMAGE"]
評価とデプロイを一体化することで、品質を担保しながら継続的にリリースできる！

⑭ Cloud Runにエージェントをデプロイ

  • gcloud run deployコマンドでコンテナイメージをCloud Runにデプロイ
  • A2A通信用のポート番号とwell-knownパスを指定してエージェント間通信を有効化
  • 今回は検証目的で未認証リクエストを許可。本番では認証スキームを設定すべき
  • デプロイ完了には1〜3分かかる。Google Cloud Consoleで進捗を確認可能
コンソールのCloud Runページで2つのサービス（Dungeon + Shadow Blade）を確認できる！

⑮ A2Aプロトコルでマルチエージェント間通信を実現

  • A2Aプロトコルはエージェント同士が通信するための標準プロトコル
  • エージェントカードにスキル、URL、接続方法（ポーリング/ストリーミング）を定義
  • DungeonエージェントとShadow Bladeエージェントがクラウド上で通信し、ボス戦を実行
  • Gemini CLI自体もマルチエージェント能力を持ち、サブエージェント（調査エージェント等）を自動的に呼び出す
エージェント同士がクラウド上で協力して動く。これがマルチエージェントシステムの醍醐味！

⑯ Hooksでエージェントのライフサイクルを制御

  • Hooksはエージェントのライフサイクルにカスタムロジックを差し込む仕組み
  • ツール呼び出しの前にロギング、セッション開始/終了時に処理など、さまざまなタイミングでフック可能
  • toollogger.shスクリプトを作成し、settings.jsonで「before_tool」として登録
  • 正規表現で特定ツールのみにマッチさせたり、タイムアウトを設定したりと柔軟に構成可能
// settings.json のhooks設定
{
  "hooks": {
    "before_tool": [{
      "pattern": ".*",
      "command": "toollogger.sh",
      "timeout": 5
    }]
  }
}
観測可能性やデータガバナンスの観点からも、Hooksは本番運用で必須の機能！
ボス戦とまとめ

⑰ クラウド上のボス戦でエージェントをテスト

  • DungeonエージェントのURLにアクセスし、Crimson Keepに進入してShadow BladeのURLを入力
  • 完璧主義やドグマなどの怪物に対して、MCP Serverで定義した武器ツールを使って攻撃
  • ボス戦では設計ドキュメント、評価データ、Pytestの意義などに関するクイズにも挑戦
  • 全問正解でボスを撃破。実装したエージェントがクラウド上で実際に動作するのを確認
ローカルで作ったエージェントがクラウド上のボスと戦う。これぞVibe Codingの集大成！
