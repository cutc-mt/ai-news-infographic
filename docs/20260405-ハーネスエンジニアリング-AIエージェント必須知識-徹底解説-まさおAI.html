ハーネスエンジニアリングとは？

AIエージェントの品質を決める最も重要な技術分野を徹底解説

ハーネスエンジニアリング AIエージェント Claude Code まさおAI

①ハーネスとは何か：モデル以外の全て

AIエージェントにおいて、モデル以外の全てのことをハーネスと呼ぶ。「モデル ＋ ハーネス ＝ AIエージェント」という構造で理解すると分かりやすい。Anthropicが2025年11月頃のブログでこの概念を共有して以来、Codexのブログなどでも広く言及されるようになり、AI業界全体で注目を集めるテーマになっている。

②ハーネスの5つの役割：分割して理解する

ハーネスが担う役割は大きく5つに分けられる。モデルに対して何を見せるのか（コンテキスト）、何を使わせるのか（ツール）、ガードレールで行動をどう制限するか、外からどう監視するか、そしてフィードバックループで失敗時にどう対応するか。これらを調節していくのがハーネスエンジニアリングの本質だ。

③外部ハーネスと内部ハーネスの違い

ユーザー側で使える部分が「外部ハーネス」で、Claude CodeやCodexの内部にあるシステムプロンプトやオーケストレーションの仕組みが「内部ハーネス」にあたる。Claude Codeはユーザーに自由に外部ハーネスを調整させる思想で設計されているため、干渉しやすく深く掘り下げやすい。Codexも改善を続けているが、思想の面でClaude Codeが優位に立つ。

④CLAUDE.mdの改善もハーネスエンジニアリング

日々の開発で使っているCLAUDE.mdを改善すること自体がハーネスエンジニアリングの1つ。CLAUDE.mdやAGENTS.mdの内容はAIエージェントの注意力が向けられるため、書きすぎると逆効果になる。常にAIの注意力とコンテキストウィンドウは有限であることを意識して、シンプルに保つことが重要だ。

⑤Hooks（フックス）：決定論的な制御の要

Claude CodeのHooks機能を使えば、ツール呼び出しの前後やプロンプト入力時、応答完了時などに自動処理を仕込める。Lintの自動実行、コードフォーマッター、通知の送信、成果物の検証、自動コミットなどが代表的。AIエージェントの出力が決定論的なものについては、LLMにやらせずにHooksで処理するのが鉄則だ。

⑥決定論的と非決定論的のバランス

AIエージェントは「Aを入れたら必ずBが返る」という決定論的な出力をしない。だからこそ、できる限り決定論的な部分に寄せて制御するのが理想となる。よくミスる内容はスキルMDに書き込んで段階的に改善し、頻繁に起こる問題はどんどん決定論的な制御へ移行していく。これがハーネスエンジニアリングの実践的なアプローチだ。

⑦コンテキストエンジニアリングからの進化

コンテキストエンジニアリングは「モデルのコンテキストウィンドウに何を入れるか」という話で、プロンプトエンジニアリングの進化系だった。しかし現在ではより包括的な「ハーネスエンジニアリング」へと定義が拡張されている。コンテキストの最適化だけでなく、制御システム自体の設計と実装を含む広い概念として2026年4月時点では認識されている。

⑧コンテキストの適切な粒度：具体と抽象のバランス

コンテキストに指示を書く際、具体的すぎると制約が厳しくてLLMの想像力を殺してしまい、抽象的すぎると何をしていいか迷ってしまう。この「適切な粒度」を見つけるのが難しく、定量化も難しいが非常に重要な感覚だ。ツールもできる限り絞り込み、MCPを詰め込みすぎるとコンテキストを圧迫するので注意が必要。

⑨メモリーシステム：状態の永続化

ハーネスの重要な要素としてメモリーシステムによる状態の永続化がある。Claude Codeではmemory.mdや個別メモリファイル、サブエージェントのメモリなどが存在し、セッションが変わっても記憶が保持される。OpenClaudeなどもメモリシステムの探索を進めており、この分野はハーネスエンジニアリングの重要な領域だ。

⑩スキルとサブエージェントの設計

スキルはマークダウンファイルとYAMLフロントマターで構成され、「このファイルにスクリプトがあるからこういう時に実行して」のような指示を記述する。スキルを高度化していくことも、サブエージェントにどのような関心を向けるかを意識することもハーネスエンジニアリングの一部だ。さらに上位概念としてエージェントチームスでオーケストレーションする仕組みもある。

⑪LLMの本質的な弱点を知る

モデル単体だと、コンテキストが1万から100万に増えるにつれて制度がどんどん落ちていくことが研究で知られている。自分自身が正しいか間違っているかを評価するのが難しく、アンカリング効果で少しのバイアスが後の意思決定を引っ張ってしまう。早期に完了を宣言してしまう特性もあり、これらをコントロールするのがハーネスの役割だ。

⑫ジェネレーターとエバリュエーターの分離

作る人と評価する人を分けることが絶対にやるべき設計パターン。自分が正しいと思い込んでいるジェネレーターに対して、別の視点からフィードバックを回す必要がある。Claude Codeでは自然にプランサブエージェントやエクスプロアサブエージェントなどで分離が進んでいるが、主観と客観の突き合わせという本質を意識することが重要だ。

⑬Planner・Generator・Evaluatorの3層パターン

Anthropic自身が実践しているのがプランナー、ジェネレーター、エバリュエーターの3分割。俯瞰と詳細を同時に見るのは難しいため、プランナーとジェネレーターは分けるべき。さらにエバリュエーターで主観と客観のズレを補正する。この3エージェント分割はコストが上がっても品質が高くなり、Claude Codeの内部ハーネスもこの本質を徹底的に取り入れている。

⑭モデルの違いを活かした役割分担

ジェネレーターにはGemini、エバリュエーターにはCodexというように、モデルごとの得意分野に応じて役割を割り当てるのも有効なハーネス設計だ。UIの美しさならGemini 2.5 Proが優秀で、レビューならCodexが向いているなど、条件によって適材適所がある。Web開発ベンチマークなどの知見も参考になる。

⑮OpenAI Codexチームの実績：ハーネス設計の力

OpenAIのコデックチームは5ヶ月で100万行超えのコードを人間が1行も書かずに生成したと報告している。その背景には優れたハーネス設計がある。同チームは最も困難な課題が「環境・フィードバックループ・制御システムの設計」に集中していると明言しており、モデルだけでなくハーネス設計力の重要性を示している。

⑯長期タスクの失敗対策：早期終了と検証不足

長期的なタスクでよく失敗するのが、早期終了宣言と検証不足のバグ。対策としては、feature-list.jsonで進捗を外部に書き出して管理する、Gitの履歴や進捗ノートでセッション間の状態を引き継ぐ、Playwrightなどの検証ツールをAIに持たせて動作確認させる、といった方法が推奨されている。

⑰アンチパターン：過剰な制御は逆効果

制約を増やせば確実性は上がるが、LLMの非決定論的な強みである想像力を殺してしまう過剰なハーネス構築は推奨されない。モデルが進化しても生き残るようなハーネスを作ることが大切で、「Claudeが賢い」という前提を持つことで過剰な制御を避けやすくなる。ドキュメントが腐っていたら適切に処理することも重要だ。

⑱ハーネス設計の始め方：4つのステップ

まずCLAUDE.mdとAGENTS.mdをきれいに整備する。次にHooksでLintやフォーマッターを自動化し、決定論的に寄せられる部分はLLMにやらせない。そして検証手段としてPlaywright CLIなどを導入してAIに動作確認させる。最後にジェネレーターとエバリュエーターを分割した評価ループを導入し、継続的にハーネスを改善していく。これが無難な始め方だ。
