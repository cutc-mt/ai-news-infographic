🧠 Claude Code記憶システム - Karpathy方式で自己進化メモリを実現

Andrej KarpathyのLLMナレッジベース設計をClaude Codeの内部記憶として実装

Claude Code Karpathy 記憶システム Obsidian Hooks 自己進化

①KarpathyのLLMナレッジベース構想

Andrej Karpathyが提唱する個人用ナレッジベースをLLMで構築するアイデア。外部情報を収集し、エージェントが効率的にクエリできるように整理する仕組み。Obsidianをコアスタックとして使用し、AIセカンドブレインの構築に注目が集まっている。

②コンパイラ方式の4ステップアーキテクチャ

Karpathyのシステムは「ソース→コンパイラ→実行可能→テストスイート→ランタイム」というコンパイラのアナロジーで説明される。ソースコードに相当する論文や記事を、LLMがコンパイルしてWikiを作成し、テストを経てランタイムでクエリを実行する。

③Rawフォルダ: 情報の入り口

Rawフォルダがシステムのエントリーポイント。記事、論文、文字起こしなどを生のマークダウン形式でそのまま投入する場所。Obsidian Web Clipperを使えば、インターネット上の情報をワンクリックでVaultのRawフォルダに直接取り込める。

④コンパイラ: LLMによる情報処理

コンパイラ段階でLLMが生情報を処理して要約を作成し、ドキュメント間のリンクを確立。知識の構造化を自動的に行う。Karpathyのシステムでは実際にスクリプトを使用し、生情報をLLMに渡してWikiを生成する。

⑤実行可能: Wikiとバックリンク

コンパイルされたWikiが「実行可能」な形式。ここにはLLMが生成した要約記事とバックリンクが含まれ、知識同士が接続されている。Obsidianのグラフビューを使用すると、異なるマークダウン文書がどのように接続されているかを可視化できる。

⑥グラフ構造の力

バックリンクによるグラフ構造がエージェントに強力な能力を与える。グラフをトラバースしてより良い検索を行い、異なる知識同士を接続して包括的な回答を生成できる。これは単なるベクトル検索よりも優れたアプローチ。

⑦テストスイート: リンティングとデータ整合性

コードにおけるテストスイートと同様に、ドキュメントのリンティングを実行。研究のギャップ特定、古いデータの発見、RawフォルダにあるのにWikiにない情報の検出、壊れたリンクの修正などを行い、データの整合性を確保する。

⑧ランタイム: クエリ実行とインデックス

ランタイムではWikiを使ってクエリを実行。最も重要なのはインデックスファイルで、エージェントに対して「アクセス可能なフォルダとリソース」を記述する。エージェントはこれを開始点として使用し、ファンシーなRAGなしでナビゲートできる。

⑨ベクトル検索不要のシンプルさ

Karpathyは「ファンシーなRAGが必要だと思っていたが、LLMはインデックスファイルを自動維持するのがかなり得意」と言及。ベクトルデータベースなしで、エージェントはすべてのファイルをナビゲートできる。このシンプルさがこの戦略の美点。

⑩Coleの実装: 内部データへの適用

Cole MedinはKarpathyの外部データアプローチを内部データに適用。Claude Codeにコードベースと共に進化する記憶を与えるシステムを構築。インターネットから情報を取り込む代わりに、会話ログを自動的にキャプチャして知識を抽出する。

⑪Claude Code Hooksによる自動化

システム全体がClaude Code Hooksのみで駆動。追加のインストールや統合設定は不要。settings.jsonにフックを定義するだけで、セッション開始時や終了時に自動的に記憶システムが動作する。美しくシンプルな設計。

⑫セッション開始フック

新しいClaude Codeセッションを開始するとagents.mmdとindex.mdが自動的にロードされる。agents.mmdにはシステム全体の説明が含まれ、index.mdはアクティブに維持されるファイルリスト。エージェントはメタ推論を実行できる。

⑬Daily Logs: Rawフォルダの等価物

Daily LogsフォルダがRawフォルダの役割を果たす。Claude Codeとの会話の要約をすべてここにキャプチャする。会話で行った決定、学んだレッスン、アクションアイテムが標準フォーマットで保存される。

⑭Wikiフォルダ: 構造化された知識

Wikiフォルダには適切にフォーマットされリンクされた知識が格納される。Obsidianのグラフビューで知識の接続を可視化でき、エージェントはここを検索する。Daily Logsから抽出された概念と接続が含まれる。

⑮Pre-Compactとセッション終了フック

メモリコンパクションやセッション終了時、Claude Agent SDKが自動的に要約を作成。最新のメッセージが別のLLMプロセスに渡され、Daily Logファイルに保存される。Anthropicサブスクリプションをそのまま使用でき、APIキー設定は不要。

⑯Flushプロセス: 知識の昇格

1日1回FlushプロセスがLogsから概念と接続を抽出してWikiに移動。検索はKnowledgeフォルダにフォーカスするが、必要に応じてDaily Logsも検索可能。スクリプトのプロンプトをカスタマイズして抽出方法を調整できる。

⑰知識の複利ループ

自己進化の核心が「知識の複利ループ」。質問を始めるとナレッジベースを活用し、エージェントが多くのWiki記事を検索して回答を生成。その回答が単一のファイルとして保存され、Wikiが成長する。会話ごとに知識が蓄積され、検索精度が向上し続ける。

⑱時間とともに進化するエージェント

このシステムではメンテナンスはほぼ不要。質問をするたびに回答が良くなり、新しいセッションからの情報も自動的に蓄積される。コードベースの決定や進化を記憶し、プロジェクトごとに賢くなっていく。Claude Codeの標準メモリシステムよりもシンプルで効果的。
