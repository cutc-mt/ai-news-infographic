三体問題: ユーザー・UI・モデル

従来のWebアプリはユーザーとUIの2者関係だった。しかしChatGPT Appsではモデル（AI）という第3の主体が加わる。

  • ユーザー: 意図を入力し、結果を確認する
  • UI（ウィジェット）: 視覚的な表示と操作
  • モデル: 会話の「脳」として推論を行う
ユーザーがボタンを押しても、
モデルはそれに気づかない！
コンテキスト非対称性が最大の課題

① すべてのコンテキストを
共有すべきではない

「すべてを共有する」は最も早い段階での失敗だった。ChatGPT Appsでは、各パーツが意図的に異なるビューを必要とする場合がある。

  • パフォーマンス: UIの画像・価格データを全てモデルに送るとトークン消費とレイテンシが増大
  • ロジック: 情報は意図的に非対称にする設計が必要
フィールド可視性
structuredContentウィジェットとモデルの両方
_metaウィジェットのみ（モデルには非表示）
ミステリーゲーム「Murder in the Valleys」では、
犯人はモデルだけが知っている！

② レイジーローディングは
AIアプリに合わない

Web開発では当たり前のレイジーローディングだが、ChatGPT Appsではパラダイムが逆転する。ツール呼び出しはセキュリティサンドボックスとモデル推論の影響で数秒の遅延が発生する。

  • NG: ユーザーがクリックしてからデータを取得する
  • OK: 初期のツールレスポンスに可能な限り多くのデータを含める
  • window.openai.toolOutputでウィジェットにデータを注入
最初からガンガン数据を積み込め！
待たせるな、が鉄則

モデルの可視性と通信パス

③ モデルに可視性を与える

ユーザーがウィジェットで商品を選択してから質問しても、モデルがどの商品を見ているか知らなければ正しく回答できない。この問題を解決するために2つのアプローチを活用する。

  • 命令型: window.openai.setWidgetState(state)で状態をモデルに共有
  • 宣言型: data-llm属性でUIコンテキストを自動記述
<div data-llm={
  selectedTab === "details"
    ? "User is viewing product details"
    : "User is viewing reviews"
}>
data-llm属性を付けるだけで、
Viteプラグインが自動的に
ウィジェット状態を同期！

④ 操作ごとに異なるAPIを使う

ChatGPT Appsにはウィジェット、サーバー、モデルの間に複数の通信パスが存在する。それぞれが異なる操作を担当しており、互換性はない。

  • ウィジェット → モデル: setWidgetStateでUI状態を共有
  • モデル → サーバー: ツール呼び出しでバックエンドと通信
  • サーバー → ウィジェット: toolOutputでデータを注入
  • ウィジェット → サーバー: callToolで直接ツールを実行
どの通信パスを使うかを
明確に設計することが重要！
「なんとなく」はNG

⑤ UIは複数の表示モードに対応する

ChatGPT Appsは単一のレイアウトではなく、3つの表示モードに対応する必要がある。同じウィジェットでも呼び出し方によって異なるサイズで表示される。

モード特徴用途
インライン会話内に表示素早い操作
フルスクリーン画面全体を占有地図など複雑なUI
ピクチャーインピクチャー会話の上に浮遊会話を続けながら参照
  • モバイルでは閉じるボタンなどのセーフゾーンに注意
  • デバイスごとのUIオーバーレイを考慮する

⑥ 組み込み環境での
UI一貫性が重要

ChatGPT Appは単独の製品ではなく、ChatGPTの既存インターフェース内に組み込まれる。視覚的な不整合はユーザーにすぐ目立つ。

  • Apps SDK UI Kitがベースラインを提供
  • Tailwind CSSベースのコンポーネント・アイコン・デザイントークン
  • カスタムコンポーネント（例: Mapbox統合）でも一貫性を維持可能
UI Kitを使えばChatGPTと
自然に統合できる！
一貫性 = 信頼感

AIのためのUI再設計

⑦ 言語ファーストのフィルタリング

従来のダッシュボードはチェックボックスやスライダーが並んでいる。しかしユーザーが自然言語で直接意図を表現できる環境では、UIフィルターは摩擦になる。

  • NG: サイドバーにフィルターを並べる
  • OK: LOV（List of Values）をツールパラメータに提供
  • ユーザーが「晴れのヨーロッパで200ドル以下」と言えば、モデルが自動的にweather="sunny"をマッピング
フィルターはいらない！
ユーザーは言葉で伝えればいい
No Filtersが正解

⑧ ファイルが豊かな操作を解放する

ChatGPT Appsでは、ファイルは二次的な入力ではなく新しい操作の起点になる。フォームやフィルターではなく、ユーザーが持っているものから体験を始められる。

  • ECアプリで商品写真をアップロード → モデルが識別 → ウィジェットで商品発見へ
  • モデル側: openai/fileParamsでファイルを直接消費して推論
  • UI側: window.openai.uploadFileとgetFileDownloadUrlで双方向に操作
写真1枚で商品検索が始まる！
ファイルは入口に変身する

⑨ CSPは新しいCORS

OpenAIはAppsを二重ネストされたiframe内でレンダリングする。Content Security Policies（CSP）が厳格に適用され、「ローカルでは動くが本番で壊れる」現象が頻発する。

フィールド用途よくあるミス
connectDomainsAPI/XHRステージングAPIの忘れ
resourceDomains画像・フォント・スクリプトCDNのホワイトリスト忘れ
frameDomainsiframe埋め込みYouTube/Mapboxの登録漏れ
redirectDomains外部リンクOAuthコールバックの未設定
CSP設定は最優先で取り組め！
本番デバッグの時間を
大幅に節約できる

⑩ ウィジェットフラグが
大きな影響を持つ

CSP以外にも、ウィジェットレベルの小さな設定がモデル・ウィジェット・ホスト環境間の制御を決定する。見落としがちだが、動作に決定的な影響を与える。

  • widgetDomain: 提出「アプリで開く」ボタンの宛先。提出時に必須
  • ツールアノテーション: readOnly、destructiveHint、openWorldHintが必須
  • ツール可視性: モデルが呼び出してはいけないツールは明示的にprivateに
  • widgetAccessible: ウィジェットがcallToolで自律的にツールを呼べるかを制御

高速な開発反復のための工夫

⑪ ホットリロードが必須

ChatGPT Appsでは、長TTLリソースキャッシュとJSON-RPCによるリソース転送のため、ViteやNext.js標準のHMRがそのままでは動かない。

  • Viteの内部構造を理解した上で、専用Viteプラグインを開発
  • MCPサーバーへのリソースリクエストを傍受し、ChatGPT iframeにリアルタイム更新を注入
  • IDEでの変更が即座にChatGPT内に反映されるように
保存した瞬間にChatGPTで確認！
フィードバックループが爆速に

⑫ すべてのテストを
ChatGPTで行う必要はない

ChatGPTでのテストはゴールドスタンダードだが、初期段階の反復にはローカルエミュレーターが有効。特にツール定義の作業では、Developer Modeでのアプリリロードが必要になる。

  • 軽量ローカルエミュレーターを構築してChatGPTホスト環境をモック
  • デバッグツールとApps専用ログを完備
  • Reactの状態やレイアウトの反復はミリ秒単位で実行
  • 本番テストはモデルの相互作用とエッジケースの検証用に温存

⑬ モバイルテストには
明示的な対応が必要

ChatGPTでテストするにはローカルサーバーのトンネリングが必要だが、Viteはデフォルトでlocalhostを使うため、他のデバイスからはアクセスできない。

  • Viteプラグインを拡張してドメインフォワーディングをサポート
  • iOSとAndroidの両方でテストが可能に
  • モバイル検証を通常の開発ワークフローに組み込めるように
モバイルテストは後回しにしない！
プラグインで解決しよう

⑭ 使い慣れた抽象化で
フロントエンドを加速

Apps SDKは強力な機能を提供するが、主に低レベルのJavaScript APIで構成されている。Reactユーザーには馴染みのある抽象化を導入することで、開発効率を大幅に向上させた。

  • useCallTool、useWidgetState、useLocaleなどのReact Hooks
  • createStore: Zustandベースの高度な状態管理
  • ボイラープレートが減り、モダンなWeb開発に近い感覚でウィジェット開発が可能に
React Hooksがあれば
既存のスキルをフル活用！
useCallTool最高

教訓をツールに変える: Skybridge & Codex Skill

⑮ 教訓を再利用可能な
ツールにエンコードする

同じパターンを複数のアプリで繰り返し発見するのは時間の無駄。そこで教訓を直接ツールに組み込むことを決断し、コミュニティにも公開した。

  • Skybridge Framework: オープンソースのReactフレームワーク
  • Hooks、devツール（HMR・エミュレーター）、data-llm属性をパッケージ化
  • chatgpt-apps-builder Codex Skill: アプリライフサイクル全体を支援
# インストールは1コマンド
npx skills add alpic-ai/skybridge
同じ轍を踏むな！
教訓はコードに落とし込んで
再利用しよう

Skybridgeの
Codex Skill機能

Codex SkillはChatGPT App開発の全ライフサイクルを支援する。アイデア出しからデプロイまで一貫してサポートする。

  • アイデア出し: アプリを「エージェント的」にする方法をブレインストーミング
  • コード生成: ReactフロントエンドとMCPサーバーバックエンドを同時生成
  • ローカルテスト: 開発サーバーを起動し、ホットリロードでリアルタイム反復
  • QAと公開: CSP検証・セーフゾーン確認・本番テストを実行
  • デプロイ: アプリのリリースと反復開発を支援
アイデア → コード → テスト →
公開まで全部自動化！
