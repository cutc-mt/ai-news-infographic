①ChatGPT Apps：AIファースト体験の新しい設計

Alpicは、次世代のプロダクトはAIファースト体験で構築されると考えている。ユーザーが従来のUIを操作するのではなく、モデルと協働するインターフェースだ。OpenAIがApps SDKをリリースすると、すぐに開発に着手。3ヶ月で旅行・小売・SaaSなどB2B・B2C向けに2ダースのChatGPT Appsを開発した。しかし、従来のWeb開発の定石がエージェント環境では通用しないことにすぐ気づいた。

②三体問題：ユーザー・UI・モデルの3者関係

従来のWebアプリではユーザーとUIの2者だけで完結していた。しかしChatGPT Appでは、3番目の主体「モデル」がシステムに入ってくる。最も難しいのは、この3者の間で情報をどう流すかを管理することだ。ユーザーがウィジェットで「選択」ボタンを押しても、モデルはそれに気づかない。この「コンテキストの非対称性」を解決することが、良いアプリと粗雑なアプリの違いを生む。

③教訓1：すべてのコンテキストを共有すべきではない

最初は「すべてを共有する」方針だったが、それが最も早い段階での間違いだった。実際には、同じ状態でも意図的に異なるビューが必要になる場面が多い。

パフォーマンス面：UIウィジェットには画像や価格のバリエーションなど大量のデータが必要だが、モデルには不要。すべて送るとトークン消費とレイテンシが増大。

ロジック面：ミステリーゲーム「Murder in the Valleys」では、モデルは犯人を知る必要があるがユーザーは知ってはいけない。「Time's Up」ゲームでは逆に、UIがユーザーに秘密の単語を見せ、モデルは知らない状態にする。

④教訓1の解決策：フィールドで可視性を制御

「誰が何を知る必要があるか」を明示的に決定するため、異なるtool出力フィールドを活用した。

フィールド目的可視先
structuredContent型付きデータ（ウィジェットとモデル用）ウィジェットとモデルの両方
_metaレスポンスのメタデータウィジェットのみ（モデルから隠蔽）

例えば「Time's Up」ゲームでは、秘密の単語を_metaフィールドだけでウィジェットに渡し、モデルはユーザーのヒントから推測するように設計した。

⑤教訓2：レイジーローディングはAIアプリに合わない

Web開発の経験から、データは必要な時に取得するレイジーローディングが常識だった。しかしChatGPTではパラダイムが逆転する。ツール呼び出しにはセキュリティサンドボックスとモデル推論による数秒の遅延が伴うため、データを後回しにすると体験が悪化する。

教訓：可能な限り初期のツールレスポンスに多くのデータを詰め込み、window.openai.toolOutputでウィジェットにデータを供給する。この方が高速でレスポンシブな体験になる。

⑥教訓3：モデルに可視性を与える

ユーザーがウィジェットで商品を選択してからチャットで質問しても、モデルがUIの状態を知らなければ正しく回答できない。この解決にはwindow.openai.setWidgetState(state)を使用し、次回のユーザーとモデルのやりとりで状態データがコンテキストに追加される。

<div data-llm={selectedTab === "details"
  ? "ユーザーは商品詳細を閲覧中"
  : "ユーザーはレビューを閲覧中"
}>

さらに、宣言的なdata-llm属性をコンポーネントに直接付与する方法を導入。Viteプラグインがこれらの属性を自動収集してwidgetStateを更新する。

⑦教訓4：異なる操作には異なるAPIを使う

ChatGPT Appsでは、ウィジェット・サーバー・モデルの間に複数の通信パスが存在する。これらは互換性がなく、それぞれが異なる種類の操作を支える。

ウィジェット → モデル：ユーザーの操作をモデルに伝える
モデル → サーバー：ツール呼び出しでデータを取得
サーバー → ウィジェット：結果をUIに反映
ウィジェット → サーバー：直接XHRで公開APIにアクセス

どの通信メカニズムがどの操作を担うのかを明示的にマッピングすることが重要。

⑧教訓5：UIは複数の表示モードに適応する

ChatGPT Appsは単一のレイアウトではなく、3つの表示モードに対応する必要がある。

モード外観用途
インライン会話履歴内に表示（デフォルト）素早い操作向け
フルスクリーン画面全体を使用、下部にチャットバー複雑なUI（地図など）
ピクチャーインピクチャーインラインと同サイズだが会話の上に固定会話の続きでもUIを参照したい時

特にモバイルでは閉じるボタンなどデバイス固有のセーフゾーンに注意が必要。コンテンツが見切れないように配慮する。

⑨教訓6：組み込み環境でのUI一貫性が重要

ChatGPT Appは独立したプロダクトではなく、既存のインターフェースの中に組み込まれる。そのため、視覚的な不整合がすぐに目立つ。どれだけ独自コンポーネントを作っても、周囲のChatGPTと調和していなければならない。

OpenAI Apps SDK UI Kitがこの問題を解決する。Tailwind CSSベースで、ChatGPTのデザインシステムに合致するコンポーネント・アイコン・デザイントークンを提供。AlpicはMapbox統合などのカスタムコンポーネントを作る際も、このUI Kitを基準にしてネイティブな一貫性を維持した。

⑩教訓7：言語ファーストのフィルタリング

従来のダッシュボードはチェックボックスやスライダーだらけだが、エージェントUIではそれが逆効果になる。ユーザーは「ヨーロッパの日照のある目的地で200ドル以下」と自然言語で表現できるのに、UIコントロールを通すのは摩擦でしかない。

アプローチ：「フィルターなし」の設計

サイドバーの代わりに、ツールパラメータに
List of Values (LOV) を提供。
モデルが自然言語をバックエンドのAPI要件に
直接マッピングする。
ユーザー「晴れ」→ モデルが weather="sunny" で呼び出し

モデルに選択肢を推測させるのではなく、直接マッピングさせるのがポイント。

⑪教訓8：ファイルでリッチな操作を実現する

ファイルは二次的な入力ではなく、新しい操作の起点になる。フォームやフィルターから始めるのではなく、ユーザーが既に持っているものから体験を始められる。

例：ECアプリでユーザーが商品の写真をアップロード → モデルが商品を特定 → ウィジェットで直接マッチングや発見に進める。

モデル側：openai/fileParamsでチャットにアップロードされたファイルを直接消費
UI側：window.openai.uploadFileとgetFileDownloadUrlでファイル操作が可能

⑫教訓9：CSPは新しいCORS

OpenAIはセキュリティ上の理由でAppsを二重ネストされたiframe内にレンダリングする。Content Security Policy（CSP）が厳格に適用され、「ローカルでは動くが本番では壊れる」問題が頻発する。

フィールド目的よくあるミス
connectDomainsAPI・XHRリクエストステージングAPIと本番の混同
resourceDomains画像・フォント・スクリプトCDNをホワイトリストに追加忘れ
frameDomainsiframeの埋め込みYouTubeやMapboxの追加忘れ
redirectDomains外部リンク（警告なしで開く）OAuthコールバックドメインの忘れ

CSP設定を早い段階から最優先課題として扱うことで、本番デバッグの手間を大幅に削減できる。

⑬教訓10：小さなウィジェットフラグの影響は大きい

CSP以外にも、ウィジェットレベルの小さな設定がウィジェット・モデル・ホスト環境間の制御分担を決定する。見落としがちだが、ナビゲーションやツールアクセス、公開の重要な境界を定義する。

ホストとナビゲーションの境界：
widgetDomainは提出に必須。「Appで開く」ボタンの行き先を定義。setOpenInAppUrlでコンテキストに応じたルーティングが可能。

モデルとツールの境界：
ツール注釈（readOnly、destructiveHint、openWorldHint）は提出時に検証される。モデルが呼び出してはいけないツールは明示的にprivateにする。

ウィジェット実行の境界：
widgetAccessibleがウィジェット自体のcallToolによるツール呼び出しを制御する。

⑭教訓11：高速な反復にはホットリロードが必要

Apps SDKは急速に進化しているが、長TTLのリソースキャッシュとJSON-RPC経由のリソース転送により、ViteやNext.jsの標準的なHMRがそのままでは使えない。

解決策：Viteの内部を深く理解した上で、ChatGPT内でウィジェットをライブリロードできる専用Viteプラグインを構築。MCPサーバーへのリソースリクエストをインターセプトし、ChatGPTのiframeにリアルタイムの更新を注入する。

IDEでの変更が即座にChatGPTに反映され、フィードバックループが劇的に短縮された。

⑮教訓12：すべてのテストをChatGPTで行うべきではない

ChatGPTでのテストはゴールドスタンダードだが、最初の反復にはローカルエミュレータが適している。特にツール定義の作業中は、Developer Modeでアプリをリロードする必要があるため効率が悪い。

Alpicの解決策：ChatGPTのホスト環境をモックする軽量なローカルエミュレータを構築。デバッグツールとApps固有のログも完備。

Reactの状態やレイアウトはミリ秒単位で反復
ChatGPTでのテストはモデルの操作とエッジケースの検証に温存

この使い分けで開発スピードが格段に向上した。

⑯教訓13：モバイルテストには明示的なサポートが必要

モバイルテストには別の課題がある。ローカルサーバーをトンネリングする必要があるが、Viteのデフォルトのlocalhost設定では他のデバイスからアクセスできない。

対応：Viteプラグインを拡張して、トンネリングされたポートでのドメインフォワーディングに対応。iOSとAndroidの両方のデバイスでテストが可能になり、モバイル検証が日常のワークフローに組み込めた。

モバイル特有のセーフゾーン（閉じるボタンなど）の検証も、この仕組みで継続的に行えるようになった。

⑰教訓14：React Hooksでフロントエンド開発を加速

Apps SDKは強力な機能を提供するが、主に低レベルのJavaScript APIとして公開されている。React使いにとっては、使い慣れた概念に近づけたいところ。

React向けの抽象化を導入：
useCallTool：ツール呼び出しのカスタムフック
useWidgetState：ウィジェット状態管理
useLocale：ロケール情報の取得
createStore（Zustandベース）：複雑なデータフロー向けの状態管理

おなじみのフロントエンドパターンを再導入することで、ボイラープレートが削減されモダンな開発体験に近づいた。

⑱教訓15：SkybridgeフレームワークとCodex Skillで再利用化

複数アプリで同じパターンが浮かび上がる中、同じ問題を再発見し続けるのが開発を遅らせる原因になった。そこで教訓をツールに直接組み込むことにした。

Skybridge Framework（オープンソース）：
カスタムフック、開発ツール（HMR・エミュレータ）、data-llm属性などを再利用可能なブロックとしてパッケージ化。

chatgpt-apps-builder Codex Skill：
フルライフサイクルを支援 — アイデア出し→コード生成→ローカルテスト→QA・公開チェック→デプロイ。

npx skills add alpic-ai/skybridge で導入可能。

最も魅力的なChatGPT Appsは、既存プロダクトの移植ではなく、AIファースト体験として意図的に設計されたものになるだろう。
