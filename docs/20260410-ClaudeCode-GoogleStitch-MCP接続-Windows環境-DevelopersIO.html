Claude Code × Google Stitch MCP 接続奮闘記

Windows環境で「Failed to connect」と戦った記録

2026-04-10 | DevelopersIO | 田中聖也

①

① 目標: AIデザインをAIコードに変換

  • Google StitchはAIでUIデザインを生成できるサービスです。
  • Claude Codeからデザインデータにアクセスすれば、AI生成デザイン→AIコード変換のワークフローが実現できます。
  • この2つの橋渡しをするのがMCP（Model Context Protocol）です。
公式の @_davideast/stitch-mcp パッケージを使えば簡単に繋がるはず...でした
②

② アーキテクチャ: 連携の流れ

  • Claude Code ← (MCP) → stitch-mcp
  • stitch-mcp ← (HTTP) → Google Stitch API
  • Claude CodeがMCP経由でStitchにアクセスし、デザインデータを取得してコードに変換する仕組みです。
// 期待されるデータフロー Claude Code → MCP Protocol → stitch-mcp stitch-mcp → HTTPS → Stitch API
③

③ 開発環境

  • OS: Windows 11
  • Claude Code: v2.1.86
  • Node.js: nvm4wで管理
  • @_davideast/stitch-mcp: v0.5.1
Windows環境特有の問題が後に発覚することに...
④

④ 前提条件: gcloud CLIの設定

  • gcloud auth application-default loginで認証を設定
  • gcloud config set projectでプロジェクトを指定
  • set-quota-projectでクオータプロジェクトを設定
  • gcloud beta services mcp enableでStitch MCPを有効化
これらを全部やらないと後で認証エラーが出るよ！
⑤

⑤ 失敗1: 公式手順通りに設定

  • 公式READMEに従い、mcpServersの設定を行いました。
  • command: "npx"で@_davideast/stitch-mcpを実行する設定です。
// 公式READMEの設定 { "mcpServers": { "stitch": { "command": "npx", "args": ["@_davideast/stitch-mcp", "proxy"], "env": { "STITCH_API_KEY": "your-api-key" } } } }
結果: Failed to connect...繋がらない！
⑥

⑥ 失敗2: Windowsの.cmdラッパー問題

  • MCPはstdin/stdoutで純粋なJSONデータをやり取りします。
  • しかしWindowsの.cmdラッパー（npx.cmd）がこのパイプを妨害していました。
  • 対処: .cmdを経由せず、Node.jsで直接JavaScriptファイルを実行する設定に変更。
// .cmdを回避する直接実行設定 { "command": "C:\\nvm4w\\nodejs\\node.exe", "args": ["...\\stitch-mcp.js", "proxy"] }
⑦

⑦ 失敗3: stderrへのログ出力

  • stitch-mcpは接続状況のログをすべてstderrに出力していました。
  • 接続自体は成功しているように見えるが、Claude Code側では認識されない状態。
// stderrに出力されるログ [stitch-proxy] Connecting to https://stitch.googleapis.com/mcp... [stitch-proxy] Connected to Stitch, discovered 12 tools [stitch-proxy] Proxy server running
ログ上は「Connected」なのに、Claude Codeには繋がっていると認識されない...
⑧

⑧ 失敗4: 4つの仮説で原因特定

  • 仮説A: Claude Codeがstitch-mcpにリクエストを送れていない
  • 仮説B: stitch-mcpがリクエストを受け取れていない
  • 仮説C: stitch-mcpがレスポンスを返していない ← 正解！
  • 仮説D: Claude Codeがレスポンスを受け取れていない
「Claude Codeの代わりに自分がMCPクライアントになる」という検証方法で切り分け！
⑨

⑨ 検証結果: 仮説Cが正解

  • stdinにinitializeリクエストを送信して検証しました。
  • stdout: 空（何も返ってこない）
  • stderr: 「Connected to Stitch, discovered 12 tools」と出力
// PowerShellでの検証 $proc.StandardInput.WriteLine($initializeMessage) // 結果: // stdout: 空（何もなし） // stderr: [stitch-proxy] Connected...
stitch-mcpはリクエストを受け取っているのに、stdoutにレスポンスを一切返していない！
⑩

⑩ 失敗5: 認証問題

  • API Key認証では一部の操作で権限不足エラーが発生。
  • 正しくはOAuth2認証 + quota projectヘッダーが必要でした。
  • 公式のAPI Key方式だけでは不十分で、より強力な認証方式が求められます。
単純な認証問題かと思いきや、他にも問題が潜んでいた...
⑪

⑪ 解決策: 自作stdioプロキシの作成

  • 独自のstitch-proxy.jsを作成しました。
  • Claude Code ← (stdio) → stitch-proxy.js ← (HTTPS) → Stitch API
  • 公式パッケージのバグを回避し、直接HTTPSでStitch APIと通信する方式です。
公式パッケージの代わりに自作プロキシで接続成功！
⑫

⑫ 自作プロキシ: アクセストークン管理

  • gcloud auth application-default print-access-tokenでトークンを取得します。
  • トークンは30分間キャッシュして再利用します。
  • OAuth2 + x-goog-user-projectヘッダーで認証。
function getToken() { const now = Date.now(); if (cachedToken && now < tokenExpiry) { return cachedToken; } cachedToken = getAccessToken(); tokenExpiry = now + 30 * 60 * 1000; // 30min return cachedToken; }
⑬

⑬ 自作プロキシ: リクエスト転送

  • Claude CodeからのstdinリクエストをHTTPSでStitch APIに転送します。
  • Stitch APIからのレスポンスをstdoutでClaude Codeに返します。
  • 初期化リクエスト（initialize）は自前で処理し、それ以外はStitch APIに転送。
// stdin → HTTPS → Stitch API → stdout const options = { hostname: 'stitch.googleapis.com', headers: { 'Authorization': 'Bearer ' + token, 'x-goog-user-project': QUOTA_PROJECT } };
⑭

⑭ 自作プロキシ: MCP初期化処理

  • 起動時にStitch APIへinitializeリクエストを送信して接続を確立します。
  • プロトコルバージョンは2024-11-05を指定。
  • 12個のツールが利用可能になります。
async function initStitch() { await sendToStitch({ jsonrpc: '2.0', id: 0, method: 'initialize', params: { protocolVersion: '2024-11-05', capabilities: {}, clientInfo: { name: 'stitch-proxy' } } }); }
⑮

⑮ claude.jsonの設定

  • 自作プロキシを使うためにclaude.json（MCP設定）を更新します。
  • commandにnode.exeのフルパスを指定。
  • argsにstitch-proxy.jsのパスを指定します。
// claude.json の設定 "mcpServers": { "stitch": { "type": "stdio", "command": "C:\\nvm4w\\nodejs\\node.exe", "args": ["C:\\Users\\...\\stitch-proxy.js"], "env": {} } }
⑯

⑯ ディレクトリ構成

  • .claude/フォルダにMCP設定を配置します。
  • mcps/フォルダに自作プロキシを配置します。
  • 必要なファイルはstitch-proxy.jsだけです。
C:\Users\your-name\ ├── .claude\ │ └── claude.json ← MCP設定 └── mcps\ └── stitch-proxy.js ← 自作プロキシ
たった2つのファイルで完結！シンプルな構成です。
⑰

⑰ 最小手数でやるべきこと

  • Step 1: gcloud CLIの認証設定（login、project、quota）
  • Step 2: 自作プロキシスクリプト（stitch-proxy.js）の作成
  • Step 3: claude.jsonの再設定（node.exe + proxy.js）
振り返ると、この3ステップだけでよかった！
⑱

⑱ まとめと学び

  • 公式パッケージでもバグはある。Windows環境では特に注意が必要です。
  • 仮説検証で原因を体系的に切り分けることが重要です。
  • 問題があれば自作プロキシで回避できる柔軟性を持つ。
  • トラブルシュートを通じてMCPの技術理解が深まります。
