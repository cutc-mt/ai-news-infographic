Claude Codeは育成ゲーム

  • Claude Codeは単なるツールではなく、育成ゲームのようなものです。設定してから自分好みに育てることが大切です。
  • 記憶を与えたり、タスクを覚えさせたり、フォルダ構成を工夫したり、MCPを繋いだりと考えることがたくさんあります。
  • この動画で紹介する9つの設定を行うだけで、Claude Codeの性能が劇的に向上します。

① Cursor × Claude Codeで環境構築

  • 非エンジニアにはCursorが最適です。IDE（統合開発環境）としてVS Codeのような使い勝手です。
  • 下にターミナル、右にファイルプレビュー、左にカーソルエージェントを配置できます。
  • 通常の暗いターミナル画面と違い、視覚的に開発できるのが大きなメリットです。
テキストや資料を作る時に
プレビューが見えるのは便利！

Cursorの画面構成と便利な機能

  • 複数ターミナルを起動でき、名前も付けられます。1つのウィンドウで画面分割も可能です。
  • 「Open Project」から自分の作業場所を選択。ローカル環境で作業を進めます。
  • Cursorのエージェント機能も活用可能。別のAIモデルも20ドルプランで利用できます。
エンジニアの方はターミナルでもOK。
CMAXやGhosttyなども選択肢です。

② CLAUDE.mdを手書きで作成

  • CLAUDE.mdはフォルダを開いた瞬間にClaudeが読み込む指示書ファイルです。
  • 自動生成もありますが、手書きで具体的に書く方が圧倒的に良い結果になります。
  • 海外の複雑なテンプレートより、軽くサクッと作る方がおすすめです。
鉄則: その行を消しても
Claudeの動作が変わらなければ不要です！

CLAUDE.mdの書き方のコツ

  • 「全て日本語で回答して」や「完成したら通知音を鳴らして」などの基本的な指示を書きます。
  • 「GitHubへはこうやってプッシュして」や「コミットメッセージはこの形式で」などの作業手順を記載。
  • Cursorならプレビュー機能でMDファイルをすぐ確認できます。 
# CLAUDE.mdの例
- 全て日本語で回答してください
- 完成したら通知音を鳴らしてください
- GitHubへのpushは以下の手順で...
- コミットメッセージの形式: feat: ...

③ CLAUDE.mdを階層設計する

  • CLAUDE.mdは1つだけではなく複数、フォルダごとに設置します。
  • 4〜5階層の構造で設計することで、プロジェクトやクライアントごとに適切な指示が適用されます。
  • 上位フォルダのMDも確認できるため、共通ルールと個別ルールを階層で管理できます。

階層設計の具体例

  • ユーザー全体 → 共通ルールのCLAUDE.md
  • work/ → 仕事関連のルール
  • development/ → 開発の技術ルール
  • client/ → クライアントごとの個別指示

│   │   └── project-b/CLAUDE.md

④ フォルダ構成を整える

  • フォルダ構成が悪いとClaude Codeの精度が下がる原因になります。
  • デスクトップに散らばさず、work/development/personal等で整理しましょう。
  • Claude Codeに「フォルダ構成を見直して」と頼めば、最適な構成を提案してくれます。

  • ├── development/ → 開発プロジェクト
  • ├── memory/ → 記憶・ナレッジ
  • ├── personal/ → 個人作業
  • └── work/ → 仕事関連

⑥ スキルを設定して業務を自動化

  • 議事録の作成など、毎回同じ指示を入力するのは手間です。スキル化で解決しましょう。
  • /議事録のようにスラッシュコマンド一つで定型作業を実行できます。
  • 作業が気に入ったら「スキル化してください」と言うだけで、スキルとして保存されます。

スキルの管理と活用方法

  • 定期的にスキル一覧を確認し、弱い部分は修正していくのがおすすめです。
  • どこに何のスキルがあるかを把握することが、スキル活用の第一歩です。

⑦ MCPで外部ツールを連携

  • MCP（Model Context Protocol）で外部ツールをClaude Codeに接続できます。
  • 連携可能なツール: Gmail、Slack、Discord、Googleカレンダー、Notion、Google Drive、Figmaなど。
  • 会社の制限がある場合は、まず個人環境で試すのがおすすめです。
GmailとSlackを連携するだけでも
仕事がかなり楽になります！

MCPとCLIの違い

  • MCP: 接続が簡単。初心者向け。サクッと外部ツールと連携したい場合に最適。
  • CLI: トークン使用量を圧縮できるメリット。慣れてきたらこちらも試しましょう。
  • 初心者はまずMCPで始めて、慣れてきたらCLIにも手を出すのがおすすめです。
どちらもAPIの一種ですが、
MCPの方が圧倒的に簡単！

⑧ セキュリティルールで危険を回避

  • denyルールを設定して、危険なコマンドの実行をブロックします。
  • Claude Codeはdeny → ask → allowの順で処理されるため、denyが最優先です。
  • 何百回OKを押しても、denyに入れたコマンドは絶対に実行されません。
セキュリティを固めてから
Auto Modeに進みましょう！

ブロックすべき危険コマンド

  • rm -rf: 確認なしでフォルダの中身を全削除。絶対にブロック必須です。
  • git push --force: チーム全員のコード変更を上書き破壊。復旧不可能です。
  • .env読み取りブロック: プロンプトインジェクションでAIが勝手に読むのを防ぎます。 
# settings.jsonのdeny例
{
  "permissions": {
    "deny": [
      "rm -rf *",
      "git push --force",
      "cat .env"
    ]
  }
}

セキュリティ設定の実践方法

  • settings.jsonファイルにdenyルールを直接記述します。
  • 設定は3段階: ユーザー全体 / プロジェクト共有 / ローカルのみ。
  • 月1回はパーミッションリストの棚卸しをして、allowリストが肥大化しないように管理しましょう。
「denyの設定方法を教えて」と
Claude Codeに聞けば全部教えてくれます！

Auto MemoryでAIを育てる

  • 記憶のタイプ: ユーザーの好み、フィードバック、プロジェクト、外部リンクなど多様。
