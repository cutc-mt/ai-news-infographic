① OpenHarnessとは：香港大学が開発したオープンソースエージェントフレームワーク

OpenHarnessは香港大学のデータサイエンス研究室（HKUDS）が公開したオープンソースのエージェントフレームワークだ。MITライセンスで完全無料、ソースコードもすべて見ることができる。Claude Codeのような自立型AIエージェントを、Anthropicに依存せずに自分で構築できる画期的なプロジェクトだ。

② 名前の由来：ハーネス＝馬具が意味するもの

ハーネス（Harness）とは馬につける馬具のことだ。馬に馬具をつけると人間が乗って制御できるようになるように、LLMにもハーネスをつけることで制御可能なエージェントに変換できるという設計思想が名前に込められている。LLMはそのままだと賢いけど何もできない存在だが、ハーネスが手足と記憶とルールを与える。

③ LLMの根本的な課題：賢いけど何もできない

ChatGPTやClaudeなどのLLMは、そのままだと話すことしかできない存在だ。ファイルが読めない、コマンドを実行できない、外部APIも叩けない。いくら賢くても実用的な作業が一切できない。この「脳みそはあるけど体がない状態」を解決するために、LLMに手足と記憶とルールを与えるインフラが必要になる。

④ 手足（Tools）：LLMに行動能力を与える

手足とはファイルの読み書き、シェルコマンドの実行、Web検索といった行動能力のことだ。OpenHarnessは43以上のツールを最初から組み込んでおり、LLMが自ら判断してこれらのツールを使いこなせるようになる。MCP（Model Context Protocol）にも対応していて、SlackやGitHubなどの外部サービスとも連携可能だ。

⑤ 記憶（Memory）：メモリがないと毎回0からやり直し

メモリ管理は実用的なエージェントに不可欠だ。メモリがないと今どこまで作業したか、過去に何を学んだかを一切記憶できず、毎回ゼロからやり直しになってしまう。OpenHarnessはCLAUDE.mdとMEMORY.mdの2種類のメモリを使い分けて、プロジェクトの文脈と作業の進捗をしっかり保持する仕組みを備えている。

⑥ ルール（Rules）：AIの暴走を防ぐ権限制御

ルールとは権限制御の仕組みだ。勝手にシステムファイルを消したりしないように、危険な操作をブロックすることができる。3段階の権限モード（許可・確認・拒否）を操作ごとに設定可能で、ファイルを読むのは「許可」、シェルコマンドは「確認」が必要、rm -rf などの危険な操作は「拒否」に設定できる。AIが暴走した時のリスクを根本から防ぐ設計だ。

⑦ エージェントループ：思考→実行→判断を自動で繰り返す

エージェントループは思考→実行→結果受信→次の判断というサイクルを自動で繰り返す仕組みだ。人間が毎回指示を出さなくても、自分で判断して作業を進める。例えば「このバグ直して」と頼むと、コードを読んで怪しい箇所を特定し、修正を書いてテストを実行し、通ったら報告する。この一連の流れがすべて自動で行われる。

⑧ エラー耐性：指数バックオフとストリーミング対応

APIエラーが起きてもOpenHarnessは諦めない。指数バックオフという仕組みで、最初は1秒待ってリトライし、ダメなら2秒、次は4秒と待ち時間を倍に増やしながら再接続を試みる。またストリーミング型の通信にも対応しており、すべて終わるまで待つのではなく、結果が来るたびにリアルタイムで処理を進められるため体感速度もかなり向上する。

⑨ 43以上のツールキット：オンデマンドで必要な分だけ読み込む

ファイルの読み書き、コードの実行、Web検索、MCP対応、Gitの操作など43以上のツールが最初から組み込まれている。ただし全てを常に読み込んでいるわけではなく、オンデマンドローディングという仕組みで必要な時だけ読み込む。使わないツールでメモリを消費しない、実用的なエージェントには欠かせない効率的な設計だ。

⑩ 2種類のメモリ：CLAUDE.mdとMEMORY.mdを使い分ける

1つ目はCLAUDE.mdファイルで、プロジェクトのルートに置いておくと自動で読み込まれる。本家のClaude Codeと同じ仕組みで、プロジェクト固有のルールや設定を書いておける。2つ目はMEMORY.mdファイルで、セッションをまたいで保存される永続メモリーだ。エージェントが作業中に発見した構造情報などを記録し、次回から調べ直さずに済む。

⑪ プレ／ポストツールフックで安全な運用を実現

ツールを呼ぶ前と後にカスタムの処理を挟めるフック機能がある。例えばgit pushの前にメインブランチかどうか確認するチェックを入れたり、事後にどのファイルを変更したかを全て記録したりできる。企業で使う時に監査が必要なケースにも対応でき、ただ動くだけでなく安全に運用できる設計が行き届いている。

⑫ マルチエージェント調整：自分で部下を作る仕組み

メインエージェントが必要に応じてサブエージェントを自動生成できる。大きなタスクが来た時、自分1人でやるより専門家に分担した方が早い。例えば「このウェブアプリをテストして」と頼むと、フロントエンドのテストはAに、バックエンドはBに、セキュリティチェックはCにという具合に分担する。依存関係がなければ並列で同時実行される。

⑬ チームレジストリー：カンバンボードのような進捗管理

複数のエージェントがバラバラに動くと管理が大変になる。そこでチームレジストリーという仕組みで各エージェントの役割と進捗を一元管理する。誰が何をやっていて、どこまで進んでいるかが一目でわかる。バックグラウンドで動いているタスクも追跡できるため、1人で1時間かかる作業も3人で分担すれば20分で終わるという発想で効率化できる。

⑭ 2つの互換レイヤーでマルチモデル対応

OpenHarnessはAnthropic専用ではない。レイヤー1はAnthropic互換でClaude、GLM、MiniMax、Kimiなどが使える。レイヤー2はOpenAI互換でGPT-4.5やo1、オープンソースモデルも対応だ。モデルを自由に切り替えて用途に合わせて選べる。コストが安いモデルで試して本番は高性能なモデルに切り替えるといった使い方も可能だ。

⑮ GitHub Copilot OAuth認証にも対応

面白いことにGitHub CopilotのOAuth認証にも対応している。デバイスフローで認証すれば、Copilotの契約でエージェントを動かせる可能性がある。VS Codeで使っているCopilotの契約をそのまま活用できるため、既存の開発環境とシームレスに統合しやすい設計になっている。

⑯ 技術スタック：Python + React（Ink）で構築

ベースはPython 3.10以上が必要だ。ターミナルUIにはReactとInkというライブラリを使っている。Inkを使うとReactのコンポーネントでターミナルUIを描画できる面白い技術アプローチだ。テストは114件が付属しており、出力形式はテキスト、JSON、ストリーミングJSONに対応。JSONで出力できるためCI/CDパイプラインへの組み込みも簡単だ。

⑰ 1行でインストール：5分あれば手元で動かせる

依存関係が複雑そうに見えるが、たった1行のコマンドでインストールできる。インストールスクリプトがすべて自動でやってくれる。5分もあれば手元で動かせるほど試すハードルが低い。MITライセンスだから商用利用も含めて自由に改造・再配布が可能で、企業が自社向けにカスタマイズして使うことも想定されている。

⑱ 学習教材としても最適：ブラックボックスの内部がすべて見える

Claude Codeはどう動いているのか中身が全然見えないブラックボックスだが、OpenHarnessならエージェントの内部構造がコードレベルで全部見える。エージェントループがどう動くのか、権限管理がどう実装されているのかが理解できる。学んで改造して自分だけのエージェントを作り、それを共有してコミュニティで発展させていく。オープンソースの良さが詰まったフレームワークだ。
