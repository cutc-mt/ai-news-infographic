Claude Codeの使用量制限を回避する10のテクニック

トークン消費を抑えて効率的にClaude Codeを使いこなす方法

2026年4月10日

①「使用量制限に達しました」の正体

  • Claude Codeを使っていると突然「usage limit reached」というメッセージが表示される
  • いつ発生するかは決まっておらず、使用方法によってタイミングが異なる
  • これはランダムなグリッチではなく、非常に特定の計算に基づいている
  • メッセージを送信するたびにトークンが計算されて蓄積していく仕組み
単純なカウンターではなく、使用方法に依存する動的な制限なんだ！

②トークンとは何か？

  • トークンは文字の並びのことで、ざっくり言えば「1単語 ≒ 1トークン」
  • 送信時も受信時もトークンが消費される
  • すべてのコストはトークンに紐づいている
  • メッセージ、MCPツール定義、システムプロンプト、参照ファイルなどがトークン消費の要因
トークンを理解することが、制限回避の第一歩！

③トークンの複利効果を理解する

  • Claudeは毎回会話全体を最初から読み直す
  • 例: 最初のメッセージが100トークン → 3回目のメッセージは約7,000トークンに
  • 単純加算ではなく複利的に蓄積される
  • 3回目の短いメッセージでも、過去の履歴すべてが含まれる
メッセージ1: 100トークン
メッセージ2: 100 + 返信分
メッセージ3: 全履歴 + 新規入力
→ 指数的に増加！

④ティップス1: 新しい会話を始める

  • /clear コマンドでセッション全体をクリアして新規開始
  • ターミナルを再起動しても同じ効果
  • /compact で過去のメッセージを圧縮
  • 可能な限り毎回新規セッションで始めるのが推奨
clear: 全消去 / compact: 圧縮。目的に応じて使い分けよう！

⑤ティップス2: MCPサーバーを切断する

  • MCPサーバーのツール定義は毎回のプロンプトに付加される
  • 100個のツールを定義すると、100個分の定義が毎回送信される
  • 使っていないサーバーは必ず切断する
  • サーバーを見直し、必要な時だけ起動する
// 悪い例: 不要なMCPも起動したまま
// 良い例: 必要な時だけ接続、終わったら切断
/mcp disconnect server_name

⑥ティップス3: エージェントを監視する

  • プロンプトを送って席を離れるのは最もよくあるミス
  • Claudeがループに陥り、同じ間違った修正を繰り返す可能性がある
  • 同じファイルを何度も読み直して不正確な出力を続けることも
  • 繰り返しを検知したら即座に停止して、より焦点を絞ったプロンプトで再開
コーヒーを取りに行くのはOK。でも監視を怠らないこと！

⑦ティップス4: CLAUDE.mdを最小限に保つ

  • 200行以下に抑えることが重要
  • CLAUDE.mdは毎回のプロンプトに付加されるため、内容がトークン消費に直結
  • 含めるべき内容: 技術スタック、コーディング規約、GitHub Actions、テストルール
  • 含めるべきでない内容: 会話履歴、条件付きルール
CLAUDE.mdは「決定事項」のためのもの。「会話の保存」用ではないよ！

⑧CLAUDE.mdに書くべき内容

  • 技術スタック: プログラミング言語、バージョン、パッケージ
  • コーディング規約: チームで従うべきルール
  • GitHub Actions: 実行したいワークフロー
  • テストルール: 新機能追加時のテスト要件
条件付きルールはNG！「この時はこれを使う」は書けない。 全プロンプトに共通する内容だけを記載してね。

⑨ティップス5: 具体的なファイルを参照する

  • @ファイル名 で特定のファイルを直接指定する
  • リポジトリ全体を渡すのではなく、必要なファイルだけを指定
  • 200ページのPDFから必要な5ページだけを抽出して渡す
  • Claude Codeがリポジトリ全体をスキャンするのを回避できる
@src/auth/login.ts のように指定すると、そのファイルに直接アクセス！

⑩ティップス6: 使用量を監視する

  • /cost: 実際のトークン使用量と推定コストを表示
  • /context: コンテキストとして送信されている内容を確認
  • /compact: 会話を圧縮してトークンを削減
  • 個人サブスクリプションの場合は特に注意が必要
/cost     → トークン使用量とコストを確認
/context  → 送信中のコンテキストを確認
/compact  → 会話を圧縮

⑪5分間のキャッシュシステム

  • Claude Codeは直近5分間の会話をキャッシュする
  • 5分以上の休憩を取るとキャッシュが失われる
  • キャッシュがない状態では全セッション履歴が再送される
  • 長い休憩の後は、セッションをクリアして再開するのが推奨
5分以上離れるなら、セッションをクリアしてから戻ろう！ キャッシュが切れると全履歴が再送されてトークンが無駄になるよ。

⑫ティップス7: メッセージをバッチ化する

  • 5つの別々のメッセージを送るのではなく、1つにまとめる
  • プロンプトは全フローを実行できるように明確に記述する
  • 基本ツール、システムプロンプトなどが1回だけ送信される
  • 同じ内容を5回送る代わりに1回で済むため、大幅に節約
複数の指示は1つのプロンプトにまとめて送るのが鉄則！

⑬「Lost in the Middle」問題

  • コンテキストが大きすぎると中間の情報が欠落する問題
  • LLMは先頭の情報をよく覚えている
  • 末尾の情報も比較的把握している
  • しかし中間部分の情報は失われやすい
コンテキストが多すぎると、大事な情報が抜け落ちる危険があるよ！ 必要最小限の情報だけを渡そう。

⑭ティップス8: 適切なモデルを選ぶ

  • Haiku: 日常的な質問や基本的なタスクに最適
  • Sonnet: Haikuで不十分な場合の次の選択肢
  • Opus: 深い研究と計画にのみ使用する最終手段
  • /model コマンドでモデルを切り替え可能
// モデル選択の優先順位
1. Haiku   → 基本タスク
2. Sonnet  → 中程度のタスク
3. Opus    → 深い研究・計画（最後の手段）

⑮ティップス9: Planモードを活用する

  • Planモードでは実行前に計画を立てて確認できる
  • Askモードはシンプルなやり取りにのみ使用
  • 計画の修正は実装の変更よりはるかにコストが低い
  • 信頼度が95%以上の場合のみ実行するルールも有効
複雑なアプリ開発では絶対にPlanモードを使おう！ 実装してからやり直すより、計画段階で修正する方がトークン節約になる。

⑯ティップス10: エージェント設計に注意する

  • マルチエージェント環境では各エージェントが全コンテキストを受信
  • 単一エージェントで済む場合は、あえて複数エージェントにしない
  • エージェントフローは慎重に設計する
  • 複数エージェント = 単一エージェントより大幅にコストが増加
エージェントを増やすと、全員が全コンテキストを受信するから要注意！

⑰10のティップスまとめ

  • 毎回新規セッションで始める（/clear）
  • 使わないMCPサーバーは切断する
  • エージェントの動作を常に監視する
  • CLAUDE.mdを200行以下に保つ
  • @ファイル名で特定ファイルを参照
  • /cost、/context、/compactで監視
  • メッセージをバッチ化する
  • 適切なモデルを選ぶ
  • Planモードを活用する
  • エージェント設計を慎重に

⑱最も重要なポイント

  • トークンは複利的に増加する。セッションをリセットすることが最優先
  • コンテキストは最小限に。多すぎると「Lost in the Middle」が発生
  • モデルはHaiku → Sonnet → Opusの順で選ぶ
  • 個人のサブスクリプションなら、/costで常にコストを確認
トークンを意識すれば、Claude Codeを賢く効率的に使える！ 毎回のセッションを新しく始めるのが一番の節約術。
