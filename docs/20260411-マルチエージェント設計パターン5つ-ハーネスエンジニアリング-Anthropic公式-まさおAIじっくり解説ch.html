  • Anthropicが公式ブログでマルチエージェントの設計パターン5つを紹介
  • 「どのような場面でどのパターンを選ぶべきか」を実践的に解説

設計パターンを知っておくと、ハーネスの最適化に生かせるよ！

② 5つの設計パターンの全体像

  • Generator-Verifier 品質重視の出力に
  • Orchestrator-Subagent タスク分解が明確な場合に
  • Agent Teams 並列で独立した長期作業に
  • Message Bus エージェント数が増加する場合に
  • Shared State 相互参照が必要な場合に
上に行くほどシンプル、下に行くほど複雑だよ！

③ パターン選びの基本原則: シンプルから始めて進化させる

  • 最もシンプルなパターンから始めるのが推奨される
  • どこでつまずくかを観察し、必要に応じて進化させる
  • 「複雑そうだから」という理由で高度なパターンを選ぶのはNG
  • まずはジェネレーターベリファイヤーから試すのがおすすめ
問題に応じて合うパターンは変わるから、まずはシンプルに！

④ パターン1: ジェネレーターベリファイヤーの仕組み

📋 タスク入力
生成
検証
✅ 合格
  • 不合格の場合はフィードバックを生成者に返してループを回す
  • 最大反復回数（例: 10回）で必ず停止する設計が必須
  • 最も普及したマルチエージェントパターンの一つ
生成 → 検証 → フィードバック → 改善のループが基本だよ！

⑤ パターン1の具体例: カスタマーサポートでの活用

  • メールチケットに対して、ドキュメントから返信を自動生成
  • ベリファイアーが正確性・トーン・網羅性をチェック
  • 不合格時は具体的な問題点（価格の誤記など）をフィードバック
  • コード生成にも有効: コード書く → テスト書いて実行
Claude Codeの最後に「レビューして」と振るのもこのパターン！

⑥ ベリファイアーの検証基準が最重要

  • 「出力が良いか」だけではゴム印を押すだけになる
  • 基準が抽象的すぎるとダメ、具体的すぎると使いにくい
  • ドメイン知識を評価基準に反映することが鍵
  • フォールバック戦略（人間へのエスカレーション等）も必須
ループを作っただけでは品質管理の illusion に過ぎない！

⑦ パターン2: オーケストレーターサブエージェントの仕組み

👔 リーダー
タスク分解
🤖 サブエージェント群
結果統合
  • 階層構造が特徴: リーダーが計画・委託・統合を担当
  • サブエージェントがそれぞれ関心ごとの分離で特定の責任に集中
  • サブエージェントは自分のコンテキストウィンドウ内で完結
リーダーが全体を見渡し、部下に仕事を振るイメージ！

⑧ Claude Codeが実際に使っているパターン

  • Claude Codeはオーケストレーターサブエージェントパターンを採用
  • メインエージェントがコードを書き、ファイルを編集、コマンドを実行
  • 大規模コードベースの検索にはバックグラウンドでサブエージェントを起動
  • 100万コンテキストの親を節約しつつ、探索は並列で実行
意識せずとも Claude Code が自然にサブエージェントを使ってるよ！

⑨ パターン2の利点と注意点

  • 利点 タスク分解が明確な場合に最適
  • 利点 関心の分離で各エージェントが特定の責任に集中
  • 注意 オーケストレーターが情報のボトルネックになる可能性
  • 注意 並列化しなければ速度面のメリットがない
情報の中継が増えると、詳細が失われやすい点に注意！

より高度なパターン

⑩ パターン3: エージェントチームスの仕組み

  • コーディネーターが独立したワーカープロセスを複数生成
  • チームメイトが共有キューから自分でタスクを取得して自律的に作業
  • オーケストレーターとの違い: ワーカーが継続的に存続する
  • コンテキストを蓄積し、ドメイン専門性が向上していく
サブエージェントは1タスクで終了、チームメイトは継続して働く！

⑪ パターン3の適用場面と課題

  • 適用 大規模なコードベース移行（各サービスを独立して移行）
  • 適用 独立性が高く、長期的なマルチステップ作業
  • 課題 完了検出が難しい（2分で終わるタスクと20分かかるタスクが混在）
  • 課題 複数エージェントが同時編集で競合が発生する可能性
独立性が高いタスクなら恩恵が大きいけど、管理変数も増えるよ！

⑫ パターン4: メッセージバスの仕組み

  • Publish / Subscribeの2つのプリミティブで通信
  • エージェントが関心のあるトピックを購読し、ルーターが配信
  • 新しいエージェントの追加時に既存の配線を変更する必要がない
  • スケーラビリティに優れ、エージェントエコシステムの成長に対応
疎結合だから、後からエージェントを追加しやすい！

⑬ パターン4の具体例: セキュリティ運用システム

🚨 アラート
🔍 トリアージ
調査エージェント
対応決定
  • トリアージエージェントが深刻度と種類で分類・ルーティング
  • ネットワーク調査、アイデンティティ分析などの専門エージェントが連携
  • 課題 イベント駆動のためトレーサビリティが難しい
ルーターの誤分類はサイレント障害になるから要注意！

⑭ パターン5: シェアドステートの仕組み

  • エージェントが共有DBやファイルを直接読み書きして協調
  • 中央コーディネーターが存在しない完全に自律的な設計
  • 初期化で質問やデータを作成し、終了条件で完了
  • 1つのエージェントが止まっても他は継続動作（単一障害点なし）
リアルタイムでエージェント同士が発見を共有できる、お気に入りのパターン！

⑮ パターン5の適用例と課題

  • 適用例 リサーチ統合システム（学術文献・業界レポート・特許・ニュース）
  • 学術エージェントの発見が、業界エージェントの調査に即座に反映
  • 課題 作業の重複や矛盾するアプローチの発生
  • 課題 リアクティブループ（エージェント同士が無限に反応し合う）の危険
終了条件（時間制限・収束しきい値）の設計が超重要！

パターンの使い分けと組み合わせ

⑯ パターン使い分けの判断基準

判断軸選ぶパターン
コンテキストを長く保ちたいAgent Teams
短いタスクで明確な出力Orchestrator-Sub
ワークフローが事前に予測可能Orchestrator-Sub
イベント駆動で動的Message Bus
エージェント間で発見を共有Shared State
独立したパーティションで作業Agent Teams
コンテキストの保持期間・ワークフローの予測性・共有の必要性で決める！

⑰ パターンは組み合わせて使うもの

  • 実運用では複数パターンの組み合わせが一般的
  • 例: 全体はOrchestrator + 一部はShared State
  • 例: Message Busでイベント配信 + Agent Teamsで処理
  • Claude CodeのAgent Teams機能は実は複数パターンの統合
Claude Codeのチーム機能 = Agent Teams + Message Bus + Shared State！ 
// Claude CodeのAgent Teams分解
Agent Teams機能 = {
  設計パターン: Agent Teams,
  メンバー間通信: Message Bus,
  ファイル共有: Shared State,
}

⑱ まとめ: まず押さえるべき2つのパターン

  • 優先度高 ジェネレーターベリファイヤー: 品質ゲート
  • 優先度高 オーケストレーターサブエージェント: 関心の分離
  • この2つをハーネスに取り入れるだけで大きく改善
  • 5つのキーワード: 品質ゲート 中央分離 並列自律 イベント駆動 共有知識
まずはシンプルに始めて、つまずいたら進化させるのがコツ！
