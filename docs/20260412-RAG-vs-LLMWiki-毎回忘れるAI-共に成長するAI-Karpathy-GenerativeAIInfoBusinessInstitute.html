RAG vs LLM Wiki

「毎回忘れるAI」から「共に成長するAI」へ — 概念比較ガイド

RAG LLM Wiki Karpathy Hermes Agent Obsidian 自己進化AI

①イントロダクション：今回のテーマ「RAG vs LLM Wiki」

アンドレイ・カパシーが2026年4月4日に公開したLLM Wikiという概念が海外で大バズり。わずか48時間でGitHub上で5000以上のスターを獲得し、フォークも3200を超える異例の人気ぶり。一方で、自己学習ループを回す「Hermes Agent」も登場し、AIの使い方が根本から変わるパラダイムシフトが起きている。

今回のメインテーマ：
「毎回忘れるAI（RAG）」と「共に成長するAI（LLM Wiki）」の構造的な違いを徹底比較

②リセットループ：現在のAIの「健忘症」問題

現在のAI利用の主流は、チャットを閉じると記憶がリセットされる仕組みに基づいている。どれほど重要なドキュメントを読み込ませても、それは一時的なコンテキストに過ぎず蓄積されることはない。これは「リセットループ」と呼ばれ、AIとの対話で誰もが抱く不満の根本原因となっている。

「さっき教えたばかりなのに、また1から説明し直して」— この不満には明確な名前がありました

③従来のRAG：情報の「その場しのぎ」検索

RAG（検索拡張生成）は現在多くの企業や個人が採用している技術。いわば高性能な検索エンジンだが、学習効率において大きな欠陥を抱えている。質問するたびにゼロから情報を再構築する仕組みで、システム自体は決して賢くなることはない。

RAGの非効率な3つのR：
Reload（再読み込み）：毎回関連しそうな文書片をゼロから読み直す
Refine（洗練）：その場で必要な情報だけを抽出する
Reconnect（再接続）：断片をつなぎ合わせてその場限りの回答を作る

④情報オーバーロードの現実：知識労働者の悲劇

RAGの「その場しのぎ」な性質がもたらす損失は深刻。知識労働者の80%が深刻な情報オーバーロードを報告しており、情報を探すためだけに毎日約1.8時間、年間にして約2万ドルもの損失を出している。何千回質問してもシステム自体は1mmも賢くならないのが現状だ。

同じ資料に対して前回質問しても、システム自体は1mmも賢くなりません

⑤カパシーの鋭い批判：「情報の墓場」

AI研究の第一人者であるアンドレイ・カパシーは、現在のRAGをこう批判している。整理されていない書類の山、つまり「墓場」の中を以前より少し早く検索しているだけに過ぎないと。この知識の墓場から抜け出し、情報を「生きた資産」に変えるアプローチが次世代のLLM Wikiである。

現在のRAGは整理されていない書類の山（墓場）を、ただ以前より早く検索しているだけに過ぎない — Andrej Karpathy

⑥LLM Wiki：AIを「検索者」から「編集者」へ

LLM WikiはAIを検索者ではなく編集者（エディター）として定義し直すパラダイムシフト。情報は単なるデータの蓄積ではなく、相互に関連付けられた複利的な成果物へと進化する。使うほどに知識が蓄積され、永続的な理解を構築していく全く新しい仕組みだ。

パラダイムシフトの核心：
その場限りの回答を得るのではなく
AIと共に使うほどに賢くなる永続的な理解を構築する

⑦ソフトウェア開発で例える3つの役割

カパシーはLLM Wikiの構造をソフトウェア開発の現場に例えて説明している。ObsidianがIDE（統合開発環境）で人間が知識を閲覧・ナビゲートするツール。LLMがプログラマーでWikiコードを書き換える主体。そしてWiki自体がコードベースで、常に更新される実行可能な知識の集合体。

開発メタファー：
生のPDF・記事 ＝ ソースコード
LLM ＝ コンパイラー
出来上がったWiki ＝ 実行ファイル（いつでも一瞬で引き出せる）

⑧RAG vs LLM Wiki：徹底比較

従来のRAGと新世代のLLM Wikiの決定的な違いを5つの項目で比較。基本姿勢、記憶の持ち方、時間の経過による効率、メンテナンス、作業単位のすべてにおいてLLM Wikiが優位性を持つ。

基本姿勢：RAG＝ゼロから再構築 → LLM Wiki＝既存のWikiを常に更新
記憶：RAG＝一時的・断片的 → LLM Wiki＝永続的・構造的
効率：RAG＝一定（蓄積なし）→ LLM Wiki＝使うほど賢く（複利）
メンテナンス：RAG＝不要だが整理もされない → LLM Wiki＝AIによる自動メンテナンス
成果物：RAG＝その都度のQ&A → LLM Wiki＝Wikiという永続的アーティファクト

⑨LLM Wikiを支える3つのレイヤー

LLM Wikiは単なるフォルダーではなく、3つのレイヤーで美しく設計されている。第1層が「Raw Sources（生の情報源）」で、PDFやメモなど一切改変しない普遍の真実。第2層が「Wiki Layer」でAIが書き換えるマークダウンによる管理された知識。第3層が「Schema」で、AIへの指示書が定義された設定ファイル。

レイヤー1：Raw Sources — 普遍の真実（PDF・メモ・議事録）。AIは直接変更しない
レイヤー2：Wiki Layer — 管理された知識。index.md（目次）とlog.md（時系列記録）
レイヤー3：Schema — agent.mdなどの指示書。リンクのルールや矛盾の扱いを定義

⑩LLM Wikiの3つのオペレーション

LLM Wikiは3つの動作で機能する。Ingest（取り込み）は新資料から重要事項を抽出しWiki全体に反映。たった1つの記事の追加が15のページを同時に更新することもある。Query（探索）は整理されたWikiから回答を導き、その回答自体も新しいページとして書き戻される。

Ingest：新資料から抽出 → 15ページ一気に更新 → 相互リンク自動生成
Query：探索自体が知識を増幅 → 回答をWikiに保存
Lint：自己修復ヘルスチェック → リンク切れ・矛盾を自動検出

⑪知識の同期：1つの追加が15ページを更新する魔法

LLM Wiki最大の特徴は知識の同期と矛盾の解消。新しい研究結果を取り込むと、AIはそれに関連する10〜15個の既存ページを自動的に書き換える。人間にとって苦痛でしかなかった書類整理やブックキーピングをAIが代行することで、知識の整合性が常に保たれる仕組みだ。

情報をただ消費するだけのフェーズから抜け出し、新しい知を生み出す段階へ進める

⑫ハルシネーション対策：人間が「編集長」

AIにすべてを任せると幻覚（ハルシネーション）のリスクがある。そのため第1レイヤーの生のソースは絶対に改変させない設計になっている。また、Lint（自己修復チェック）でAIが矛盾を検出してフラグを立てるが、最終的な判断を下すのは常に人間。AIは優秀なライブラリアンだが、編集長は人間であるという主従関係は崩れない。

AIは膨大な矛盾を見つける優秀なライブラリアンだが、編集長は常に人間。この主従関係は絶対に崩れません

⑬Hermes Agent：自己進化するオープンソースAIエージェント

Nous Researchが開発したHermes Agentは自己学習ループを自律的に回せるAIエージェント。VPSやGPUクラスター上で24時間稼働し、Telegramなどマルチプラットフォームでアクセス可能。使えば使うほど賢くなり、オープンソースのClaudeのような存在とも言われている。

Hermesの特徴：
24時間稼働（VPS/GPUクラスター）
Telegram等のマルチプラットフォーム対応
自己学習ループで自律的に成長
オープンソースで誰でも利用可能

⑭GAPA：プロンプトの「バックプロパゲーション」

Hermesの最大の特徴がGAPA（Gradient-guided Adaptive Prompt Augmentation）という仕組み。ニューラルネットワークの学習で使う誤差逆伝播法の概念を、テキストのやり取りに適用したもの。約15回のツール呼び出しごとに一旦停止し、自分自身の行動を振り返って何が成功し何が失敗したかを自己分析する。

GAPAの仕組み：
15回ツール呼び出し → 一時停止 → 振り返り
→ 何が成功/失敗したか分析 → プロンプトを自己更新
→ 次の行動計画を動的に書き換え

⑮スキルの自動生成：苦労した経験を「筋肉の記憶」に

Hermesは複雑なタスクを完了すると、それを再利用可能なスキルとして自動保存する。例えば、あるユーザーの環境でManim（アニメーションライブラリ）のスキルを試行錯誤の末に自ら獲得した事例が報告されている。次からはテキストだけでなくアニメーションで説明してくれるようになる。

1度苦労して解決したタスクはスキル化される — 次からは一発でこなす「筋肉の記憶」のようなもの

⑯Hermes × LLM Wiki：2つの革命が合体

Hermes Agentの最新アップデートで、カパシーのLLM Wikiがビルトインスキルとしてパッケージ化された。「/llm-wiki」というコマンドを打つだけで、成長し続けるエージェントが知識ベースの構築を全自動で引き受けてくれる。Obsidianと組み合わせれば、AIの脳内でシナプスが繋がっていく様子をリアルタイムで可視化できる。

/llm-wiki コマンド一つで：
知識の取り込み → 相互リンク自動生成 → 矛盾チェック → 継続的成長
すべてHermesが全自動で実行

⑰複利で爆発する知識ベース：実際の成果

カパシー自身のパーソナルWikiの例では、100の記事を取り込んだ結果40万文字以上の完全に相互リンクされた知識ベースが構築された。10のソースが40ページに、50のソースが200ページへと複利で爆発的に成長していく。マーケティング会社Goldie Agencyの事例では、キャンペーンごとにゼロからやり直す事前調査を不要にした。

複利成長の実例：
10ソース → 40ページ
50ソース → 200ページ
100ソース → 40万文字以上の完全相互リンク知識ベース

⑱今日から始めるアクションプラン

1945年、ヴァネヴァー・ブッシュが夢見た「個人の知識を連想で繋ぐデバイス（Memex）」。彼が解決できなかった唯一の課題は「誰が膨大なメンテナンスを行うか」だった。2026年、私たちはAIという最高の師匠を手に入れ、ついにその夢を実現しようとしている。

Obsidianを導入：知識の繋がりを可視化するIDEとして
Webクリッパーで記事収集：Markdown形式でRaw Sourcesに取り込む
ローカル資産を管理：画像等をローカル保存しAIが参照できる環境を整える
/llm-wikiコマンドを実行：Hermes AgentやClaude Codeで知識の自動同期を体感
