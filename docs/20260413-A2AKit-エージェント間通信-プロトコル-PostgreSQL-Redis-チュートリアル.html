A2A-Kit完全チュートリアル：基礎から本番環境まで

PostgresとRedisを使ったエージェント間通信の標準プロトコル「A2A」を3行のコードで簡単実装するフレームワークを徹底解説

A2A Protocol エージェント間通信 PostgreSQL Redis FastAPI LangGraph 本番運用

①A2Aがエージェント間通信の標準プロトコルに

A2A（Agent-to-Agent）は、Google、Linux Foundation、そして150以上のパートナー企業が推進する標準規格であり、エージェント同士が通信するための共通言語として注目されている。今後、エージェント同士が連携するシステムを構築する際には、このプロトコルがデファクトスタンダードになる可能性が高い。

②公式SDKの学習コストは非常に高い

公式のA2A SDKは低レベルなプロトコルプリミティブをそのまま提供しており、学習の壁が高い。リクエストの構造、イベントキュー、タスクアップデーター、リクエストコンテキスト、インメモリタスクストレージ、デフォルトリクエストハンドラー、A2A Starletteアプリケーションといった多数のオブジェクトを理解しなければならない。Hello Worldエージェントを作るだけでも大量のコードが必要になる。

③A2A-Kitならたった3行のコードでエージェントが完成

A2A-Kitを使えば、タスクのインスタンス化、イベントキューの管理、アップデーターの構築、キャンセルハンドラーの実装がすべて不要になる。開発者がやるべきことは「ctx.complete()」を呼び出すだけ。A2A-Kitが背後でタスクのステートマシン、SSEイベント、ストレージへの書き込み、楽観的排他制御、プッシュ通知をすべて自動で処理してくれる。

④フレームワークに依存しない完全な独立性

特定のフレームワークに縛られるソリューションとは異なり、A2A-Kitは完全にフレームワーク非依存（agnostic）で設計されている。独自のLLM、Agent Framework、ビジネスロジックを自由に持ち込むことができる。A2A-Kitはプロトコル層だけを担当するため、開発者は「エージェントが実際に何をするか」という本質的な部分に集中できる。

⑤3つの文字列を変更するだけで本番環境へ移行

A2A-Kitは本番運用を前提に設計されており、開発環境から本番環境への移行が極めて簡単。ストレージをPostgresに変更し、ブローカーとイベントバスをRedisに設定するだけ。同じWorker、同じctx.complete()のまま、水平スケーリングが可能な本番構成に切り替えられる。

⑥プラガブルなストレージで用途に合わせて選択

ストレージはインメモリ、PostgreSQL、SQLite、Redisから用途に応じて選択可能。開発時はインメモリで素早く動作確認を行い、本番環境ではPostgreSQLに切り替えて永続的なデータ管理ができる。設定の変更はパラメータを渡すだけなので、コードを書き換える必要がない。

⑦Redisブローカーで水平スケーリングを実現

プラガブルなブローカーにより、水平スケーリングに対応した構成を簡単に構築できる。Redisをブローカーとイベントバスに設定すれば、複数のWorkerインスタンスが協調してタスクを処理可能になる。1台のサーバーから複数台へのスケールアウトも、設定の変更だけで対応できる。

⑧本番運用に必要な高度な機能を標準搭載

A2A-Kitの背後にはフル機能の本番アーキテクチャが存在する。分散キャンセル（cooperative cancelとforce cancel、タイムアウト付き）、SSEの再接続（last event IDによるリコネクション）、楽観的排他制御、OpenTelemetryインストルメンテーション、Authミドルウェア、Webhookプッシュ通知などを標準で提供している。

⑨基本セットアップ：Workerクラスでエージェントを構築

エージェントの基本構成はWorkerクラスを継承してhandle()メソッドを実装するだけ。handle()の中でタスクコンテキスト（ctx）にアクセスし、ctx.complete()を呼び出せばタスクが完了する。A2AServerクラスにWorkerを渡してFastAPIアプリを生成し、Uvicornで起動すればエージェントサーバーのできあがり。最低限のコードで動作する。

⑩Agent Cardでエージェントの情報とプロトコルを設定

エージェントの説明やメタデータはAgentCardConfigクラスで設定する。エージェントの名前、説明文、そして通信プロトコル（JSON-RPCまたはHTTP+JSON）を指定できる。HTTP+JSONを選択するとSwagger UIが利用可能になり、開発中のテストが容易になる。gRPCは実装が難しいため現在は未対応。

⑪ストリーミングサポートを1行で有効化

CapabilitiesConfigでstreamingをtrueに設定するだけでストリーミングが有効になる。ストリーミングエンドポイントを利用すると、エージェントからの応答がリアルタイムで届く。SSE接続が切れた場合でも、イベントIDを使って再接続時に情報を取りこぼさない仕組みが組み込まれている。

⑫便利なクライアントクラスでsend()とstream()を活用

A2A-Kitにはサーバー側だけでなくクライアント側にも便利なクラスが用意されている。A2AClientをコンテキストマネージャーとして使い、send()でテキストメッセージを送信、stream()でストリーミング受信が可能。結果のテキスト、タスクの状態、タスクIDなどが簡単に取得できる。Pythonスクリプトから数行でサーバーと対話できる。

⑬開発用のDebug UIでブラウザから動作確認

localhostの/chatエンドポイントにアクセスするとデバッグ用のチャットUIが表示される。ストリーミングの有無、エージェントのURL、タスクの状態遷移（submitted -> working -> completed/failed）をリアルタイムで確認可能。グラフの開発初期段階で出力を直接確認できるため、開発効率が大幅に向上する。ただし本番運用は想定していない。

⑭LangGraphとの統合でカスタムストリーミングイベントを送信

LangGraphとの連携例では、get_stream_writer()を使ってカスタムストリーミングイベントをクライアントに送信する方法を紹介。イベントタイプごとに「done」「error」を判別し、send_status()でステータス更新、emit_text_artifact()でテキスト成果物を送信できる。LangGraphのストリーミングをA2A-Kit経由でシームレスにクライアントに届けられる。

⑮ライフサイクルフックで状態変化をフック可能

LifecycleHooksクラスを使えば、Workerのロジックに触れずにすべての状態遷移をフックできる。on_state_change（すべての状態変化）、on_working（タスク開始時）、on_turn_end（入力待ちになった時）、on_terminal（タスク終了時）の4種類のフックが用意されている。ログ出力、メトリクス収集、ダッシュボード更新、Slack通知などに活用できる。

⑯Webhook経由のプッシュ通知で長時間タスクに対応

数分かかるディープリサーチのようなタスクでは、プッシュ通知を使って別のサーバーに結果を送信できる。A2AサーバーがWebhook経由で外部サーバー（スマホや別のサービス）に通知を送る仕組み。CapabilitiesConfigでpush_notificationsとpush_allow_httpを有効にし、クライアント側でpush_urlを設定するだけで動作する。シークレットトークンによる認証もサポート。

⑰Docker ComposeでRedis + Postgresの本番構成を構築

本番デプロイの例として、Docker ComposeでRedis、PostgreSQL、エージェントの3つのサービスを構成する方法を紹介。Redisはキャンセルレジストリとブローカーとして、PostgreSQLはストレージとして機能する。サーバーの設定でbroker、event_bus、storageのURLを指定するだけで、インメモリ版から本番構成に切り替えられる。LangGraphを使ったディープリサーチエージェントの例も含まれている。

⑱アーキテクチャ図とAPIリファレンスを完備したドキュメント

リポジトリには詳細なドキュメント、コンポーネント図、APIリファレンスがすべて用意されている。FastAPIサーバーを中心に、ミドルウェア、リクエストエンベロープ、タスクマネージャー、ブローカー、キャンセルレジストリ、Workerクラス、イベントエミッター、ストレージ、イベントバスといった各コンポーネントの関係が視覚的に理解できる。環境変数の設定方法や追加の依存パッケージのガイドも充実している。
