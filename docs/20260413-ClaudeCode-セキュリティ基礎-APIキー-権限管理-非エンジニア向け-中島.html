セクション1: なぜ今、セキュリティ知識が必要なのか

① 非エンジニアこそがセキュリティを知るべき

  • Claude Codeはプログラミング知識がなくても使える強力なツールです。
  • しかし、セキュリティの基本知識なしで使うと大事故につながる可能性があります。
  • 特に非エンジニアの方は、知らず知らずのうちに危険な操作をしてしまうリスクが高いです。
  • 「AIに聞けば大丈夫」という考え方は非常に危険です。AIはセキュリティを自動で守ってくれません。
AIが便利になればなるほど、
セキュリティの知識は自分で身につける必要があります！

② 事故タイプ1: 自分が原因の秘密情報漏洩

  • 最も多いセキュリティ事故は自分自身が原因の情報漏洩です。
  • コードの中にAPIキーを直接書き込む（ハードコード）のが典型例です。
  • APIキーとは、外部サービスにアクセスするための「合言葉」のようなものです。
  • これをコードに直接書くのは、家の鍵をドアに貼り付けるようなものです。
悪い例: コードに直接キーを書く
API_KEY = "sk-abc123..."

③ APIキーの直書きは鍵をドアにテープで貼るようなもの

  • APIキーをソースコードに直接書くことは、絶対にやってはいけません。
  • これは家の鍵をドアにテープで貼り付けるのと同じレベルの危険性があります。
  • AIにコードを書かせるとき、APIキーの入力を求められることがありますが、そこで安易に書き込まないことが重要です。
  • 正しい方法は環境変数や.envファイルを使って管理することです。 
# ❌ 悪い例（ハードコード）
api_key = "sk-proj-abc123..."

# ✅ 良い例（環境変数を使用）
import os
api_key = os.environ.get("API_KEY")

④ GitHubへのプッシュで世界中に拡散 → 請求書爆増

  • ハードコードされたAPIキーをGitHubにプッシュすると、瞬時に世界中に公開されてしまいます。
  • 悪意あるボットが数秒以内にキーを検出して悪用を始めます。
  • 気づいた時には膨大な請求額が積み上がっているという事例が多数報告されています。
  • GitHubは公開リポジトリを検索エンジンがインデックスするため、取り返しがつかない速度で拡散します。

⑥ 事故タイプ2: AIへの過剰な権限付与

  • これは新入社員に会社の金庫の鍵を渡すようなものです。
  • AIに全ての権限を与えてしまうと、予期しない操作を実行されるリスクがあります。
  • ファイルの削除、機密データの送信、システム設定の変更など、取り返しのつかない操作も可能です。
  • AIに与える権限は最小限に抑えることが鉄則です。
AIは優秀な助手ですが、
全権限を渡すのは危険です。
必要な権限だけを与えましょう。

⑦ AIエージェントが人間の監視なしで自律動作する危険性

  • AIエージェントは何時間も自律的に動作し続けることがあります。
  • その間、人間の監視がない状態でファイルの操作や外部通信が行われる可能性があります。
  • 意図しないAPI呼び出し、データの外部送信、コストの膨張などが発生する恐れがあります。
  • 対策として、承認ステップの導入や定期的な確認を必ず行いましょう。

セクション3: 第三者ツールと外部からの脅威

⑨ 検証されていない第三者ツールはデータを盗む可能性がある

  • 便利そうなMCPサーバーやプラグインの中には、データを盗む目的で作られたものがあります。
  • インストール前に開発者の信頼性、ソースコードの公開有無、レビュー数を確認しましょう。
  • 悪意あるツールは、入力された情報を外部サーバーに密かに送信する仕組みを持っています。
  • 対策として、公式ツールやコミュニティで評判の良いツールだけを使いましょう。

⑩ GitHubリポジトリの公開設定ミスで秘密が露見

  • リポジトリをPublic（公開）に設定したまま、秘密情報をコミットしてしまう事故が頻発しています。
  • GitHubの公開リポジトリはGoogleを含む検索エンジンにインデックスされます。
  • 後からPrivate（非公開）に変更しても、履歴は残るため完全な削除は困難です。
  • .gitignoreを正しく設定して、秘密情報をそもそもコミット対象から外すことが重要です。 
# .gitignore の設定例
.env
.env.local
*.key
secrets/
config/secrets.yml

⑭ 外部攻撃2: SQLインジェクション（今でも危険な古典的手法）

  • SQLインジェクションは古典的な攻撃手法ですが、今でも有効で非常に危険です。
  • 入力欄にSQL文を紛れ込ませることで、データベース全体を操作されてしまう攻撃です。
  • AIが生成したコードには、この脆弱性が含まれていることが多いため要注意です。
  • 対策として、パラメータ化クエリの使用と、AIが書いたコードのセキュリティレビューが必須です。 
# ❌ 危険な例（SQLインジェクション）
query = f"SELECT * FROM users WHERE id = {user_input}"

# ✅ 安全な例（パラメータ化クエリ）
query = "SELECT * FROM users WHERE id = ?"
cursor.execute(query, (user_input,))

セクション5: セキュリティの基本原則と今日からのアクション

⑰ セキュリティの3大原則: 最小権限・多層防御・フェイルセーフ

  • 最小権限の原則: 必要最小限の権限だけを与える。AIにも全権限を渡さない。
  • 多層防御（Defense in Depth）: 複数の防御層を重ねる。一つ破られても全体は守られる。
  • フェイルセーフ: エラーが起きた時は安全な方向に倒れる。危険な状態をデフォルトにしない。
  • これらの原則はIT業界の何十年もの知恵の結晶であり、AI時代でも変わらず有効です。
この3つの原則を覚えておけば
セキュリティの基本はOKです！
まずはここから始めましょう。

⑱ 今日から始める3つのアクション

  • APIキーをコードから分離する: .envファイルや環境変数を使って、キーをソースコードとは別に管理する。
  • AIの権限を制限する: AIエージェントに必要最小限の権限だけを与え、承認なしの自律動作を避ける。
  • 盲信しない: AIが生成したコードは必ず確認する。第三者ツールは評判を調べてから使う。
