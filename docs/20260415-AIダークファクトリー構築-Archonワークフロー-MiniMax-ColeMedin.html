AIダークファクトリーを構築する

AIが自律的にコードを書き、レビューし、リリースする世界

2026年4月15日 | Cole Medin ライブ配信

① ダークファクトリーとは何か

  • 「ダークファクトリー」という概念は、1980〜90年代の中国の無人工場が由来です。
  • 人間がいないため電気すら不要という意味の「暗い工場」を、ソフトウェア開発に応用した考え方です。
  • Dan Shapiroがこの概念をコードベース管理に適用。AIが要件定義から実装、レビュー、リリースまで全てを自動化します。
人間はハンドルを握らない。AIに全てを任せる究極の形。それがダークファクトリー！

② AIコーディングの5段階（レベル0〜2）

レベル0：スパイシー・オートコンプリート
AIはあくまで参考ツール。開発者自身が全てのコードを書く。スマートなStack Overflowのような存在。
レベル1：コーディング实习生
クルーズコントロールのようなもの。AIは定型コードやボイラープレートを担当。まだハンドルは握っている。
レベル2：ジュニア開発者
ペアプログラミング。開発者とAIが交代でコントロール。片手でハンドルを握る感じ。
車の運転に例えると分かりやすい！レベルが上がるにつれて、人間の介入が減っていく

③ AIコーディングの5段階（レベル3〜5）

レベル3：自動運転車
AIが大部分のコードを生成。人間は全てをレビューする。Coleのおすすめレベルで、最も信頼性が高い。
レベル4：エンジニアリングチーム
ハーネスを使った長時間の自律作業。寝ている間にAIが働く。ただし最終確認は必要。
レベル5：ダークファクトリー
ハンドルなし 人間は英語で目標を指定するだけ。実装・テスト・修正・リリースは全てAIが担当。

④ ダークファクトリーの全体アーキテクチャ

GitHub Issue（入力）
トリアージ（分類・判定）
実装（並列処理）
PR検証（ホールドアウト）
修正（失敗時）
マージ＆リリース
全ての入力はGitHub Issueから。バグ報告も機能要望も、ここからスタートする

⑤ なぜMiniMax M2.7を選んだのか

  • Anthropic定期購読は使えない。ダークファクトリーを公開すると、他のユーザーのIssueが消費されてしまうため規約違反の恐れがある。
  • OpusとSonnetは高すぎる。HaikuはMiniMaxより性能が低いのにコストが高い。
MiniMax M2.7 $0.30 / 100万入力トークン
Claude Haiku $1.00 / 100万入力トークン
Sonnet 高額すぎて不適切
MiniMaxはHaikuの3分の1の価格で、性能はさらに高い！コストパフォーマンス抜群

⑥ Claude Codeを他のモデルに切り替える方法

  • 環境変数を変更するだけで、Claude Codeのバックエンドモデルを簡単に切り替えられる。
  • MiniMax、GLMなどはAnthropic互換エンドポイントを提供。OpenRouter経由でも利用可能。
  • Ollamaを使えばローカルモデルも利用可能。Gemma 4やQwen 3などが無料で使える。
  • Archonワークフロー内でモデル名をマッピング。Opus→MiniMax M2.7、Haiku→MiniMax M2.7高速版のように設定。
GLMはエンドポイントに「anthropic」を追加するだけで対応完了。とても簡単！

⑦ Archon：初のオープンソースハーネスビルダー

  • Archonは、AIコーディングのプロセスをパッケージ化できる初のオープンソースハーネスビルダー。
  • ワークフローを定義すれば、どんなコードベースでも同じ手順で確実な結果を得られる。
  • 複数のワークフローを並列実行可能。各ワークフローは独立したワークツリーで隔離される。
  • 近日中にPi（π）エージェント対応も予定。さらに多様なプロバイダーをサポートする。
プロセスを「ワークフロー」として定義。再現性と信頼性を両立するのがArchonの強み！

⑧ ガバナンスレイヤー：AIの行動規範

  • Mission.md：リポジトリの使命と目的を定義。どの方向に進むべきかの指針。
  • Factory Rules：開発ルールと制約。許可される機能と拒否すべき機能を明確化。
  • スコープの定義。実装すべき機能と除外すべき機能を明確に区別する。
  • 全ワークフローにこれらの文脈が常に注入され、AIの判断の基盤になる。
ガバナンスが不十分だと、AIが勝手に方向性を変えてしまう危険がある！

⑨ トリアージワークフロー：Issueを分類する

  • オーケストレーターが新しいIssueを検出すると、トリアージがガバナンス文書と照らし合わせて判定。
  • 判定結果はラベルで管理。factory:accepted、factory:rejected、needs-human など。
  • AIが分類した結果をJSON出力し、BashスクリプトがGitHub CLIを使って確定的にラベルを適用。
  • 連続して2回失敗したIssueにはneeds-humanラベルを付けて人間の介入を促す。
分類はAIが行い、ラベル付けはスクリプトで確定的に実行。一貫性を重視した設計！

⑩ ホールドアウトパターン：バイアスを排除する検証

  • StrongDMが考案した手法。実装を知らない別のエージェントが検証を行う。
  • 自分の宿題を自分でチェックさせるようなバイアスを排除する仕組み。
  • システム全体の回帰テストを実施。何が実装されたかを知らせずに、全機能を中立に評価する。
  • 実装と検証は別々のワークフローで実行。コンテキストの引き継ぎなし。
AIは自分のミスを見逃しがち。だからこそ、実装者と検証者を分けることが重要！

⑪ 実装ワークフロー：並列でIssueを解決

  • トリアージで承認されたIssueは、Archonワークフローで並列に処理される。
  • 各Issueは独立したワークツリーで作業。コードベースの完全なコピーで互いに干渉しない。
  • Issueを分類（バグ/機能/改善など）し、分類に応じた対応フローを選択。
  • 実装完了後は自動的にプルリクエストを作成。レビュー待ち状態に。
10個のIssueが同時に来ても、並列処理で全て同時に対応できる。これがArchonの真価！

⑫ PR検証と回帰テスト

  • Validate PRワークフローがプルリクエストを徹底的にレビュー。
  • ブラウザ自動化ツールを使って、実際のユーザーと同じ操作でE2Eテストを実行。
  • 会話の送信、出典リンクの確認、インターフェース操作など、全ユーザージャーニーをテスト。
  • 問題が見つかった場合は、修正ワークフローまたは新規Issueを作成して対応。
総合テストは1日1回または週1回。全ユーザージャーニーを網羅するからトークン消費が多い！

⑬ GitHub Issueのテンプレートとラベル管理

  • バグ報告用と機能要望用のIssue テンプレートを作成。必須フィールドで十分な文脈を確保。
  • ラベルでIssueのライフサイクルを管理：factory:accepted、in-progress、needs-fix、factory-rate-limitなど。
  • テンプレート不使用のIssueは自動的にコメントして閉じる厳格な運用。
  • ラベルの整合性により、オーケストレーターが正確に状態を追跡できる。
一貫性こそが信頼性の鍵。全てのIssueを同じ形式に統一することで、AIの判断精度が向上する

⑭ VPS上でのオーケストレーター構成

  • cronジョブが定期的に起動し、未処理のIssueがないか確認。
  • オーケストレーターがArchon CLIを呼び出して、各ワークフローを自動的に実行。
  • VPS上にArchon、RAGチャットアプリ、Claude Code（MiniMax設定済み）を構成済み。
  • レート制限保護あり。クレジットを使い果たした場合は翌日まで待機。
# オーケストレーターの基本動作
1. cron が毎時間トリガー
2. 未処理の Issue を検出
3. トリアージ → 分類
4. 承認された Issue を並列実装
5. PR検証 → マージ
自動課金は無効化済み。クレジットを使い切っても勝手に課金されない仕組み！

⑯ ダークファクトリーのリスクと課題

  • 茹でガエル現象：小さな問題が見過ごされ、蓄積して大きな障害になる危険性。
  • アライメント問題：AIが人間の意図と異なる実装をしてしまうリスク。ガバナンスで軽減。
  • セキュリティリスク。AIコーディングは熟練エンジニアより多くの脆弱性を生む傾向がある。
  • AIの迎合的（sycophantic）な性質。自分のミスを見逃し、問題を「良い」と言ってしまう。
信頼性を最重視するならレベル3がおすすめ。ダークファクトリーはあくまで「実験」！

⑰ Claude Codeのレート制限と最新動向

  • Anthropicのレート制限が最近厳しくなった。以前よりClaude Codeの利用量が制限されている。
  • 100万トークンのコンテキストウィンドウオプションが利用不可に。20万トークンに制限。
  • Routines（旧スケジュール機能）が正式リリース。定期的なプロンプト実行が可能に。
  • 週次制限は50%に到達。金曜にリセット。日常的な使用でかなり厳しく感じる状況。
Routinesはcronジョブのような機能。一度設定すれば定期的に自動実行される。ダークファクトリーと相性抜群！

⑱ 公開実験と今後の展望

  • ダークファクトリーは公開実験。完全に失敗する可能性もあるが、どこまで信頼性を高められるかに挑戦。
  • 計画文書とアーキテクチャは全て公開予定。月末までに誰でもIssueを投稿できるようにする目標。
  • Dynamisコミュニティで4時間のセカンドブレイン講座を提供。独自のAIセカンドブレイン構築を指導。
  • 4月26日はDynamisの1周年記念。4月28日にAI変革ワークショップをLeor Weinsteinと共同開催予定。
上手くいけばMVPの自動生成が可能に。失敗しても学びは多い。それが公開実験の醍醐味！
