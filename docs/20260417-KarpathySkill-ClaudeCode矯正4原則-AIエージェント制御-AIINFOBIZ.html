② AIエージェントの3つの問題

  • 暴却（忘却） — セッションが終われば全てを忘れ、毎回0から学習し直す
  • 暴走 — 質問せずに推測で実装を進め、曖昧な指示でも勝手に書き殴る
  • 過剰 — 100行で済む処理に対し、専門知識を駆使した過剰なシステムを構築
「自信過剰なジュニアエンジニア」とRedditで呼ばれるくらい… 知性はあるのに文脈を読む力とブレーキが足りない状態！

③ 文脈を与えても解消されない4つの暴走

  • 隠蔽された混乱 — 曖昧さを放置し、人間に質問せずに独自の誤った前提で実装を進める
  • 肥大化する抽象化 — 1回しか使わないコードに不要なデザインパターンを適用
  • イトセルの副作用 — 依頼されていないフォーマット変更やコメント削除を勝手に行う
  • 曖昧なゴール — 終了条件がないため、早々に切り上げるか永遠に無駄な修正を繰り返す

④ 解決策の全体像

  • プレーンテキストの規律 が複雑なインフラを凌駕する
  • 判断力とルールをマークダウンに集約し、シンプルで高速な実行環境を維持
  • 4つの原則を CLAUDE.md に記述するだけで、AIが自己修正する
  • セッション開始時に自動で読み込み、行動を最適化する仕組み
複雑なインフラよりも、シンプルなルールブックの方が効果的！

⑤ 原則1: Think Before Coding

推測するな、混乱を隠すな、トレードオフを提示せよ

  • 曖昧な指示に対して勝手に実装を始める 悪習を排除
  • 実装前に前提条件を明示し、複数の解釈肢を提示
  • 不確実な場合は必ず人間に確認を求める
「こういう風にも取れるし、ああいう風にも取れる」→ 人間に確認してから進めるのが正解！
2

⑥ 原則2: Simplicity First

100行で済むなら100行で書け

  • 推測的な機能は不要。念のための柔軟性も不要
  • 1度しか使わない抽象化を作らない
  • 業界標準のコードをバグのないシンプルな構造へ強制的にリファクタリング
「犬小屋を作って」と頼んだら勝手に帝国の柱を立てて宮殿を建て始める…それがAIの過剰化！
3

⑦ 原則3: Surgical Changes

外科手術のような精密さで触れるべき場所だけ触れ

  • 必要な箇所のみを変更。自分の散らかしたものだけ片付ける
  • 頼まれていないフォーマット変更やコメント削除を 完全に禁止
  • 「ドライブバイ・リファクタリング」を禁止し、既存のコードスタイルを厳格に遵守
ピンポイントで指示されたことだけやって！余計なお節介は不要！
4

⑧ 原則4: Goal-Driven Execution

成功基準を定義し、検証されるまでループせよ

  • 「バグを直して」ではなく「バグを再現するテストを書き、それをパスせよ」と指示
  • 検証可能なゴール を設定することで、LLMが独立してループを回す
  • スタンフォード大学の研究で、構造化プロンプトによりエラーが 40%削減 された
何をすべきか指示するな。成功基準を与えて、あとはAIが回す！指示の仕方で結果が劇的に変わる

⑨ LLM Wiki — 知識の自動構造化

AIチャットの「金魚の記憶」から脱却する仕組み。Karpathyのアイデアを具体化する。

  • Raw Data — Webクリッパーや指導で収集したPDF、テキスト、YouTube字幕などをフォルダーに格納
  • LLM Processor — AIが自立的に文脈を解釈し、論理的なチャンクに分割
  • 意味的な関係性を発見し、バックリンクでネットワーク化
  • Wiki構造化ナレッジ — マークダウンファイル群にインデックス、目次、更新履歴を定着
銀行の利子のように知識が複利で増える！使えば使うほど賢くなるパーソナルナレッジベース

⑩ Wiki Graph vs RAG の比較

項目 RAG（従来） Wiki Graph
検索方式 埋め込みベクトル検索 インデックス＋リンク
インフラ ベクターDB + チャンキング マークダウンフォルダー
理解の深さ 意味の類似度のみ 関係性の深い理解が可能
コスト 計算資源の継続維持が必要 クエリー時のトークンのみ
スケール 数百万ドキュメントに最適 数万規模までは有効
適材適所で使い分けが大事！エンタープライズはRAG、個人〜チームはWiki Graphがおすすめ
