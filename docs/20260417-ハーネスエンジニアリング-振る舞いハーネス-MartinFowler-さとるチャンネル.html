① AIコードは本当に信用できるか？

AIにコードを書かせると動くけれど、どこまで信頼していいか不安になる。LLMは確率的に次のトークンを繋いでいるだけで、プロジェクトの文脈やコードの本当の意味を理解していない。変数名の判断基準すらない状態で出力しているため、そのまま本番に投入するのは危険だ。

確率的にそれっぽいコードを出力するだけだから、
文脈を理解しているわけではないよ！
確率的トークン予測 プロジェクト文脈なし

② ハーネスエンジニアリングとは

Agent = Model + Harness。Martin Fowlerが提唱した方程式だ。ハーネスとは、AIモデルを除く全ての周辺環境（プロンプト、ツール、実行環境、制約ルール、フィードバック）を指す。馬に轡（くつわ）をつけてコントロールするイメージで、モデルという「馬」をハーネスで安全に操る考え方。

Agent = Model + Harness
ハーネス = プロンプト + ツール + 実行環境 + 制約 + フィードバック
Martin Fowler Agent方程式 馬の轡（くつわ）

③ ビルダーのハーネス

ツール提供側が用意する安全策のことだ。Claude Codeならforce-pushはデフォルトで実行しない。Cursorならファイルの自動保存や差分確認が組み込まれている。ツールを選ぶ時点で、ハーネスの一部がすでに決まっている。製品の設計者が組み込んだ「安全装置」のレイヤーだ。

  • Claude Code: force-pushをデフォルト拒否
  • Cursor: ファイルの自動保存と差分確認
  • 各ツール固有の安全設定が組み込み済み
デフォルト安全策 ツール選択

④ 外部ハーネス

ユーザーが自分で設定するハーネスのレイヤーだ。CLAUDE.mdに「TypeScriptを使う」「テストはpytestで書く」など、プロジェクト固有のルールを記述する。製品のデフォルト設定と、ユーザーのカスタム設定の二重構造になっているのがポイント。

CLAUDE.mdがまさに外部ハーネス！
プロジェクトごとにルールを書いておこう。
CLAUDE.md カスタム設定 二重構造

⑤ ガイド：事前の制御

フィードフォワード制御の仕組みだ。作業開始前に「こうして欲しい」と伝える。CLAUDE.mdに「型定義は必ず書く」等を指定すれば、1回目から正しいコードを生成できる。暗黙知を形式知に変換するプロセスでもある。

  • 事前にルールを渡す（フィードフォワード）
  • CLAUDE.md / CURSOR Rules に記述
  • 暗黙知 → 形式知への変換
  • 1回目の成功率を引き上げる
フィードフォワード 事前指示 形式知化

⑥ センサー：事後の制御

フィードバック制御の仕組みだ。実行後に「ここ間違ってるよ」と指摘する。リンターやテストがこの役割を担う。AIがエラーを見て自己修正するループを形成する。ただし、無限ループを避けるために回数上限を設けて、人間にエスカレーションする仕組みも必要だ。

リトライ回数に上限を設けて、
人間にエスカレーションする仕組みが必須！
フィードバック リンター テスト

⑦ ガイドとセンサーの組み合わせ

ガイドで1回目の成功率を上げ、センサーで最終的な品質を担保する。ガイドだけでは想定外のミスをカバーできず、センサーだけでは毎回エラーが起きて非効率になる。両方のバランスが重要で、どちらか一方では不十分だ。

Guide (事前) → 成功率 UP
Sensor (事後) → 品質保証
Guide + Sensor → 最強の組み合わせ
ガイド＋センサー バランス重要

⑧ 計算的実行（Computational）

CPUで高速・決定的に処理する方式だ。リンター、型チェッカー、Prettierなどが該当する。1万ファイルあっても数秒で完了する。同じ入力なら必ず同じ結果が返る。AIに聞くより圧倒的に早く、リソースの消費も少ない。

  • リンター: 数秒で1万ファイルをチェック
  • 型チェッカー: 決定的な結果を即座に返す
  • Prettier: フォーマットを瞬時に統一
CPU処理 高速・決定的 低コスト

⑨ 推論的実行（Inferential）

AI自体がコードの意図を解釈して判断する方式だ。「この設計は筋がいいか」「この命名は分かりにくくないか」など、曖昧な評価が可能になる。ただしGPUを使うため、時間もコストもかかる。計算的実行で済むことを推論的実行に頼るのは非効率だ。

曖昧な判断はAIの得意分野。
でもコストと時間を意識して使おう！
GPU推論 曖昧な評価 高コスト

⑩ 計算的 vs 推論的の使い分け

インデントを揃えるのにAIを使うのはもったいない。Prettierで1秒で直せることにトークンを消費する必要はない。スタイルチェックはCPUに、アーキテクチャの妥当性判断はAIに任せる。リソースの最適配分が鍵になる。

計算的 (CPU): スタイル、フォーマット、型検査
推論的 (AI): 設計判断、命名、アーキテクチャ評価
→ 適材適所でリソースを最適化！
リソース最適配分 適材適所

⑪ 3つの規制カテゴリ

Fowlerは規制を3つのカテゴリに分類している。保守性（リンターで検出）、アーキテクチャ適合性（fitness functionsで検査）、そして振る舞いの正当性。最初の2つはツールの進化でかなり制御可能になったが、3つ目が一番やっかいな問題だ。

  • 保守性: リンターで自動検出
  • アーキテクチャ適合性: fitness functions
  • 振る舞いの正当性: ★最大の未解決課題
保守性 アーキテクチャ 振る舞いの正当性

⑫ 振る舞いハーネス：最大の課題

Fowlerが「部屋の中の象（Elephant in the Room）」と呼ぶ未解決課題だ。AIがコードもテストも両方書くと、間違った前提で自己完結してしまう。「足算を頼んだのに引き算を実装＋引き算のテストを書く → 全て通るが要件は満たしていない」という恐怖が常にある。

テストが全部通っても、
それが本当に正しい振る舞いかは
誰も保証できない…これが最大の恐怖！
自己完結の罠 要件不一致 Elephant in the Room

⑬ ステアリングループ

ハーネスは一度作って終わりではない。問題が起きたらハーネスを調整して改善するサイクルが必要だ。「DBマイグレーションは人間の承認を得てから実行」などのルールを追加していく。面白いことに、AI自身にルール作りを手伝わせることも可能だ。

問題発生 → ルール追加 → ハーネス改善 → 再発防止
※ AI自身にルールを提案させることも可能！
AIが自分のハーネスルールを
書くのを手伝うという面白い逆転現象！
継続改善 ルール追加 AI参加型

⑭ ハーネス可能性（Harnessability）

ハーネスが効きやすいコードと効きにくいコードがある。強い型付け、明確なモジュール境界、テスト充実 = ハーネスしやすい。逆に、動的型付け、スパゲッティコード、テストなし = ハーネスしづらい。従来の「良いコード」の条件が、そのままAI制御しやすさに直結している。

  • 良いコード → ハーネスしやすい
  • スパゲッティ → ハーネスしづらい
  • 品質の良いコードほどAI制御も容易
強い型付け モジュール境界 テスト充実

⑮ 人間の役割

ハーネスは人間を不要にするものではない。機械的なチェックから人間を解放し、本当に人間が決めるべきことに集中させる仕組みだ。要件の正しさ、設計の妥当性の確認こそが人間の本質的な仕事。振る舞いハーネスが未解決である以上、人間の目は必須だ。

人間は「何を作るか」を決め、
AIは「どう作るか」を担当する。
この分担が理想的！
要件確認 設計判断 人間必須

⑯ アクション①：ガイドの整備

CLAUDE.mdやCURSOR Rulesでプロジェクトのルールを明文化しよう。「Prismaを使う」「エラーは必ずロギングする」など、チームの当たり前を書き出す。これは新メンバーへのオンボーディング資料と同じ役割を果たす。

# CLAUDE.md の例
- TypeScript strict mode を使用
- エラーは必ずロギングする
- DB マイグレーションは承認必須
- Prisma をORMとして使用
CLAUDE.md CURSOR Rules 明文化

⑰ アクション②③：センサー強化とテスト文化

リンターやチェッカーを強化しよう（ESLintルールの追加、TypeScript strict modeの有効化など）。そして、AIにコードを書かせる前に人間がテストケースを定義する習慣を構築する。テストファーストで、AIには実装だけを任せるのが理想だ。

  • ESLintルールを追加してセンサー強化
  • TypeScript strict mode を有効化
  • 人間がテストケースを定義 → AIが実装
  • テストファースト文化の構築
ESLint strict mode テストファースト

⑱ まとめ：信頼は制約から

ガイド、センサー、テストの3つを意識して整えることがハーネスエンジニアリングの第一歩だ。AIを信用するために、まず制御の仕組みを整える。人間が「何を作るか」を決め、AIが「どう作るか」を担当する。信頼は正しい制約から生まれる。

明日からできる3つのアクション：
① CLAUDE.mdにガイドを書く
② リンター・チェッカーを強化する
③ テストケースは人間が定義する
ガイド センサー テスト 信頼＝制約
