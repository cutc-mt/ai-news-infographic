①2つのテーマ: チーム強化とプレMythos

  • テーマ1：エージェントチームの飛躍的強化。複数AIの連携能力が大幅向上。
  • テーマ2：プレMythosの衝撃。封印された最強モデルの安全テスト版。
  • Opus 4.7はこの2つの側面を併せ持つ重要なモデル。

②マイクロマネジメントからの脱却

  • 従来はステップバイステップの細かな指示が必要だった。
  • Opus 4.7では自立的かつ忠実に動くため、段階的指示が不要に。
  • AIが複数エージェントを並列管理するパラダイムシフトが発生。
  • 人間とAIの1対1から、複数AIを並行管理するワークフローへ。
指示が細かくなくても
勝手に動いてくれる時代に！

③オーケストレーターの進化

  • コアオーケストレーターのツール呼び出しと計画精度が2桁向上。
  • メールボックス連携でサブエージェントと直接やり取り可能に。
  • ただし連携すべきでない箇所での連携を防ぐ制御が重要。
  • 複雑なマルチステップワークで14%のパフォーマンス改善。
プランニング能力が格段に向上！
複数エージェントを上手く束ねる。

④逐次処理と並列処理の使い分け

  • 逐次処理：企画が決まらないと次に進めない場合など。
  • 並列処理：独立したタスクを同時に進める場合。
  • タスクに応じてルーターが逐次か並列かを判定する必要がある。
  • Meta AIでもエージェントチームの練習が可能。
間違えると厄介！
分からない場合は逐次処理で安全に。

⑤エージェントを増やすリスク

  • Googleの論文によると、エージェントを増やすと最大70%の性能低下。
  • 「エージェントを増やせば賢くなる」は幻想。
  • ルーターが正しくタスクを振り分けないと精度が落ちる。
  • 最初に間違えると、後続もすべて間違える連鎖エラーの危険。
料理で砂糖と塩を間違えたら
全部台無しになるのと同じ！

⑥バグ検出テスト: 実験設計

  • カレンダーアプリに30個のバグを意図的に埋め込む。
  • Codex、Gemini、Claudeの3者協談でアプリを作成。
  • サンプルバグを3つ提示し、それをヒントに全30個を発見させる。
  • Sonnet 4.6とOpus 4.7を同一条件で比較。

バグ検出能力を検証！

  • サンドボックスに閉じ込めても脱出しかねない危険なモデル。
  • Project Glasswingとして防衛目的の組織にのみ限定公開。

  • Opus 4.7はMythos級モデルの安全テスト版。
  • 新しいサイバーセーフガードを本番環境でテストする役割。
  • 危険なリクエストの自動検知・ブロック機能の実証。
  • 段階的リリースでMythosを安全に世界に出すための入り口。

⑨バグ検出結果: Sonnet vs Opus

  • Sonnet 4.6：18点／30点満点。推論が長く、時間がかかった。
  • Opus 4.7：23点／30点満点。5点多くのバグを発見。
  • Opus 4.7は推論が非常に短く高速。なのに精度が高い。
  • Codexによる採点で公平に評価。
思考が短いのに精度が高い！

⑩バグ修正能力の比較

  • Sonnet 4.6は修正スピードは速いが、構文チェックが甘い。
  • Opus 4.7は構文チェックまで含めて丁寧に処理。
  • バージョン更新→構文チェック→成果物配置まで実行。
  • 自己検証能力が高いからこそ、間違いにくい。
Opusは頭で考えるより
手足を動かすタイプ！

⑪推論モデル vs タスク処理モデル

  • Sonnet 4.6は推論モデル寄り。考える時間が長い。
  • Opus 4.7はタスク処理モデル寄り。実行が丁寧。
  • 今後は推論モデルよりタスク処理モデルが強くなる可能性。
  • Pausis Benchが58%→70%に向上。Cursor Benchでも大幅改善。
考えるより動く方が
これからのモデルは強い！

⑫ルーターモデルの重要性

  • タスク振り分けを行うルーターが不可欠。
  • メインのコンテキストを汚さずリソースを節約。
  • 「並列か逐次か」「何人で処理するか」を事前判定。
  • 軽量な別モデルをルーターとして使う実務的アーキテクチャ。
ルーターがないと
エージェントチームは精度が下がる！

  • Mythosのような強力なAIにはネット切断の対策が必要。
  • ローカルモデルでオフライン環境にデータを置く選択肢。
  • 医療データ、企業秘密などは特にローカル環境が重要に。
  • ローカルAIへの需要がこれから急増する可能性。
本当に大事なデータは
ローカルモデルで守る時代へ！
