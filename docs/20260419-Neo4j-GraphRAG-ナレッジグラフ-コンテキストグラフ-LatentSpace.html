Neo4j GraphRAG: ドキュメントをナレッジに変える現代AIのグラフ戦略

Emil Eifrem (Neo4j CEO) が語る、グラフデータベースがAIシステムで再び重要になる理由

2026年4月19日

Neo4jとは何か

①Neo4jの全体像: データベースからプラットフォームへ

  • Neo4jはグラフデータベースとして広く知られているが、現在はより広いプラットフォームに進化している。
  • Emil Eifrem CEOは「データをナレッジに変換する」というフレーズで紹介している。ノイズからシグナルを抽出し、知識密度の高い形で表現する。
  • 世界中の企業が利用しており、Transport for London（ロンドン交通局）などもユーザー。地下鉄ネットワーク自体がグラフ構造。
「グラフ脳」になると、すべてのものがグラフに見えてくる！

②GraphRAGが解決する3つの課題

  • 高精度: グラフによる豊かなデータ表現で、より正確な回答が得られる。
  • 開発生産性の向上: ベクトル空間は不透明だが、グラフは明示的で視覚的に検査可能。
  • 説明可能性: なぜそのドキュメントが選ばれたのかを監査できる。ベクトル空間の「コサイン類似度0.7」と違って明確。
りんごとオレンジは「果物」という関係で繋がる。でもりんごとテニスボールは「丸い」というだけでは区別できない——ベクトル検索だけではこの違いが分からない！

③ベクトル検索 + グラフトラバーサルの組み合わせ

  • クエリ時の流れ：ベクトル検索で開始点を見つけ → グラフを辿るで完全なコンテキストを取得。
  • カスタマーサポートの例：自然言語で検索 → BM25 + ベクトル検索で約100件の文書を発見 → グラフから著者情報や評価を取得。
  • 単なるベクトル検索ではなく、著者の信頼性（PageRankやスター評価）も考慮して最終的なトップKを選択。
グラフ「または」ベクトル検索ではなく、両方の組み合わせが成功のパターン！

ベクトルデータベースとエンタープライズ導入

④ベクトルデータベース単独カテゴリは終わった

  • Emil氏は数年前から「スタンドアロンのベクトルDBは持続可能なカテゴリではない」と主張。現在はほぼ全てのDBにベクトル検索機能が組み込まれている。
  • 本来の位置づけは「データベース」ではなく検索プラットフォームに近い（TurboPufferのSimonも同様の見解）。
  • 「十分に良い」ものがほとんどのケースで十分。毎年、汎用DBのベクトル機能が向上し、スタンドアロンの余地は減っている。
小規模なら何でもいい。でもスケールではDBのアーキテクチャが本当に重要になる！

⑤ライフサイエンスでの大規模導入

  • 製薬・ライフサイエンス業界での採用が急増。研究者が日常的に使う科学的インテリジェンスツールとして活用。
  • Novo Nordiskのケース：内部研究、特許、学術論文など6,000万件以上のドキュメント、数十億のノードとリレーションシップ。
  • NER（固有表現認識）とエンティティ解決が、膨大なデータを構造化する鍵技術。AI業界でまだ十分に語られていない重要領域。
ライフサイエンス企業は博士号保持者が大量に在籍。GraphRAGは彼らの生産性向上の要！

⑥金融業界での爆発的成長

  • 2026年のNeo4jのAI関連会話の30%がグローバル銀行とのもの。北米のトップ20銀行すべてが顧客。
  • 大手住宅ローン会社のケース：若手バンカーの早期育成を目的に、過去の最適な案件フローを分析。コンバージョン率が20%向上。
  • 当初はバンカーへのドラフト提案機能だったが、2026年には完全自動化へ移行。人間を介さず顧客に直接メッセージを送信。
ROIは「Neo4jへの支払いより遥かに多い」とEmil氏。バンカーの平均在籍期間は1年未満だから、迅速な育成が超重要！

エージェント開発のパラダイムシフト

⑦2025年12月のターニングポイント

  • 2025年12月にあらゆる指標が急上昇。コンピュート、データベース、コーディングエージェントの全てのチャートが急増。
  • 以前は「ドラフトを作成して」だったのが、「送信して」に変化。顧客対応の完全自動化が信頼のティッピングポイントに到達。
  • 2025年夏の時点ではほぼ誰も顧客向け本番運用をしていなかったが、直近3ヶ月で劇的に変化。
何かが起きている。GPTやOpus 4.5の品質向上が要因かもしれないが、モデル品質だけではない変化が起きている！

⑧エージェント開発パターンの逆転

  • 旧パターン: 専用のCypher関数を優先 → 失敗時に汎用text-to-Cypherにフォールバック。
  • 新パターン（直近3〜6ヶ月）: 汎用text-to-Cypherから開始 → 失敗時にエッジケースを専用関数に抽出。
  • LLMの性能向上により、ワンショットでほとんどのクエリを生成可能に。パターンが完全に逆転した。
CypherのようなDSLは学習コストが高いのが弱点だった。でも今はLLMが代わりに書いてくれるから問題なし！

⑨CypherとGQL: DSLの新時代

  • Cypherは2015年のOpenCypherから始まり、長い標準化プロセスを経てGQLとしてISO標準に。SQLの初の兄弟言語。
  • ISO標準化されたことでLLMの学習データとしての信頼性も向上。Cypherは長年蓄積された豊富なトレーニングデータがある。
  • Neo4jは内部でGeminiモデルをファインチューニングしてtext-to-Cypherに使用。さらに正規表現による後処理で矢印の方向などを修正。
モデルがそのままでは完璧ではない。関係の方向を逆に書くことがあるので、正規表現で修正する裏技がある！

コンテキストグラフと4つのデータソース

⑩エージェントに必要な4つのデータソース

  • ①オペレーショナルDB: 「現在」の記録。顧客数や現在の価値など。システムオブレコードの現在形。
  • ②クラウドデータウェアハウス: 「過去」の記録。Q3のラテンアメリカ地域の売上など。OLAPの領域。
  • ③エージェントメモリ: エージェント自身の状態と過去の軌跡。短期的・長期的な記憶。
  • ④コンテキストグラフ: 組織の意思決定の軌跡。「なぜ」その判断をしたのかを記録。
Swyx氏の見解: ①②④は強固な大カテゴリ。③のエージェントメモリはまだ小さいが、個人に閉じた記憶として重要！

⑪コンテキストグラフとは何か

  • コンテキストグラフは意思決定の軌跡（Decision Traces）を記録するもの。例: 20%割引の理由、誰が承認したか。
  • 組織の暗黙知をデジタル化。Slackや電話での承認など、どこにも記録されていない意思決定プロセスを捕捉。
  • 人間の脳からエージェントの脳へ、意思決定を移行するための基盤。フライホイール効果で自己改善が可能に。
営業担当者が車から電話で割引の承認を得て契約——この記録されていない意思決定こそがコンテキストグラフの対象！

⑫ナレッジレイヤー: エンタープライズの新常識

  • 企業内の全データソースにMCPエンドポイントができても、データの矛盾が発生する問題。同じ「顧客数」でもシステムごとに違う値に。
  • ナレッジレイヤーでメタデータを統合管理。スキーマ、S3バケット、ビジネスオントロジーをグラフ形式で表現。
  • ゼロコピー方式: データを複製せず、仮想化レイヤーで元のソースをポイント。または部分的にインライン化してクエリ可能に。
「顧客とは何か？」を5人に聞くと6つの答えが返ってくる。でもエージェントを成功させるには定義を統一しないといけない！

スタートアップ戦略と未来展望

⑬AIネイティブスタートアップへの投資

  • 10年前は売上の3分の1ずつが大企業、中堅、スタートアップ。大企業セグメントの全指標が圧倒的に優秀だった。
  • OracleなどのDB企業の80%以上の収益はGlobal 2000から。エンタープライズでマネタイズするのがDBビジネスの王道。
  • でもCEO帽子を外せば「スタートアップの人は私の仲間」。次世代AIネイティブスタートアップのアーキテクチャにNeo4jを組み込むことが重要。
Bank of Americaからは常に多くの収益が上がる。でも時代の精神（Zeitgeist）に参加するには次世代のスタートアップが不可欠！

⑭create-context-graph: グラフ版Create React App

  • uvx create-context-graphコマンド一つで、フル機能のNeo4j環境とフロントエンドが立ち上がる。
  • 22の業界に対応したドメインモデルを内蔵。合成データ生成にも対応し、実際のデータ連携も可能。
  • 8〜9のエージェントプラットフォームと統合。短期記憶、長期記憶、コンテキストグラフを全て含むツールキット。
Cloud Co-WorkのWilliam氏が日曜日の午後にハッキング。わずか6日でコミュニティから大人気に！

⑮LLMベースのレコメンデーション革命

  • YouTubeのレコメンドがLLMベースに。全動画をトークン化してコードブックに入れ、LLMで次に見る動画を予測。
  • X（旧Twitter）の新しいアルゴリズムやPinterestも同じ手法を採用。レコメンデーション業界で大流行中。
  • レコメンド界隈の専門家Eugenia氏も非常に強気。ただし「普通のシステムでは絶対に推薦しない奇妙な推薦」も出てくる副作用も。
YouTubeはA/Bテストの王者。彼らが採用しているということは、効果は確実ということ！

業界展望と今後の展望

⑯SaaS-pocalypseと買い vs 作るの転換

  • SaaS-pocalypse: AIにより「買う vs 作る」のバランスが急激に「作る」側にシフト。Clouderaが一時的にシャットダウンした例も。
  • Emil氏自身、年に$200,000の会議管理ソフトに不満。「$2,000で自分好みのものを作れる時代になった」。
  • でも簡単ではない。社員がエージェントネイティブでない場合、既存のバグだらけのシステムの方がマシなことも。移行にはトレーニングと段階的なプロセスが必要。
リーダーへの忠告: AIで何でもできると「バイブコーディング」しても、社員が尻拭いをすることになるかも。本当の職人技への敬意を忘れずに！

⑰エージェントメモリとMCPのグラフ的性質

  • MCPの初期リリースには約200〜300行のPythonで実装された小さなインメモリグラフDBが含まれている。
  • エージェントメモリはグラフと相性が良いが、実用上はオーバーキルかも。ほとんどの人のメモリは「1つのファイルに収まる」規模（Emil氏の冗談）。
  • LLMのコンテキスト窓は100K→100万に拡大したが、10億や1兆には到達しない。外部のコンテキストグラフが長期記憶の解決策になる。
Emil氏: 「私はそんなに多くのトークンを出力してないよ（笑）」。でもコンテキストグラフには強く賛成！

⑱ソフトウェアが再び柔軟になった興奮と警鐘

  • Emil氏は10年間「ポストテクニカル」な立場だったが、ソフトウェアが再びモールダブル（塑形可能）になったことに大興奮。
  • 一方でバランスへの警鐘も。OpenCode CEO Daxの言葉: 「何でもバイブコーディングできる時代こそ、抑制と焦点が価値を持つ」。
  • データベースのテストは非常に厳密だからこそ、バイブコーディングと相性が良い。テスト可能性が高い領域ではAI生成コードの活用がしやすい。
Steve Jobsのように「プロダクトの編集者」としての役割。最も重要なのは「ノー」と言うこと——機能を削る勇気！
