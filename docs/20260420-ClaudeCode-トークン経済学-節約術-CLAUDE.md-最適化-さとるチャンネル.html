Claude Code トークン経済学 — 99%のトークンが無駄に消えている

キャッシュ読み取りの正体、CLAUDE.mdの罠、そして5つの節約術を徹底解説

①衝撃の事実: 99%のトークンが無駄に消えている

  • GitHub Issue #24147 のデータを分析すると驚きの結果が
  • I/Oトークンは388万、キャッシュ読み取りは50億9250万
  • 計算すると99.93%がキャッシュ読み取りで消費されている
  • あなたが使っているトークンのほぼ全部が「履歴の再読み込み」に消えている
100回トークンを使っているつもりでも、実質的な作業に使われているのはわずか0.07回分！

②100ドル課金の体感: 実質20ドル分しか使えない

  • 100ドル課金しても体感20ドル分の作業しかできない
  • 月額200ドルのMaxプランでも1日で使い切る投稿が続出
  • この感覚の正体は「キャッシュ読み取りがクォータを食う」問題
  • ユーザーは「高すぎる」と感じるが、原因は出力トークンではなく履歴の再送にあった
課金額の5分の4は「過去の会話を読み直す」ために消えている状態。もったいない！

③みんなが誤解していること: 原因は出力じゃない

  • コストがかかる原因は「出力トークンが多いから」と思いがち
  • 38セッションを分析した結果、コード出力はわずか0.6%
  • 残り99.4%は「会話履歴の再読み込み」が占めている
  • AIは毎回、過去の会話を全部読み直してから回答を始めている
出力を減らしても根本解決にならない。入力側（キャッシュ読み取り）を減らすのが正解！

④LLMはステートレス: 毎回ゼロからやり直し

  • AIは前回のリクエストの内容を一切覚えていない
  • 毎回「こういう会話をしてきましたよ」と全部教え直す必要がある
  • CLAUDE.md、システムプロンプト、会話履歴、ツール定義が毎メッセージでフル再送信
  • 会話が長くなるほど再送信量が雪だるま式に増加していく
LLMの仕組み上、避けられない問題。だからこそコンテキストをスリムに保つことが重要。

⑤CLAUDE.mdの罠: 57KBがトークンを食い潰す

  • 57KBのCLAUDE.mdは約15,000トークンに相当
  • 往復の会話で 15,000 × 50 = 75万トークンをCLAUDE.mdだけで消費
  • 何も作業していないのに75万トークンが消えている状態
  • 不要な記述を削るだけで劇的な節約が可能
50往復の会話でCLAUDE.mdだけで75万トークン！これが一番の無駄遣い。

⑥キャッシュの落とし穴: 安いけど量が圧倒的すぎる

  • キャッシュ読み取りの単価は0.3ドル（入力3ドルの10分の1）
  • しかし量はI/Oの1310倍もあって圧倒的すぎる
  • Anthropicのクォータにはキャッシュ読み取りもフルカウントされる
  • 安いから大丈夫と思っていたら、量で圧倒されて結局高コストに
単価は10分の1でも、量が1310倍なら合計は131倍！クォータもしっかり消費される。

⑦コスト構造の残酷な数学

  • 入力: 100万トークンあたり3ドル、出力: 15ドル
  • キャッシュ読み取り: 100万トークンあたり0.3ドル
  • 入力と出力の比率は3対1、キャッシュ読み取りはI/Oの1310倍
  • 0.3 × 1310 = 393ドル vs 3 × 1 = 3ドル → 130倍の差
393ドル vs 3ドル。キャッシュ読み取りのコストがI/Oの130倍という残酷な現実。

⑧雪だるま式の増加: 3週間で5倍に爆増

  • 1月11日週: 2億7600万トークン
  • 2月1日週: 14億7000万トークンに増加
  • わずか3週間で5倍以上に増加
  • 使い方は変わっていないのに、履歴が溜まってコンテキストが肥大化するから
使い方を変えなくてもトークンが増え続ける。これが雪だるま式コスト増の正体。

⑨最悪のセッションコスト: 1回で約900円

  • 最悪のセッションで約6.30ドル（約900円）
  • 1日10セッションで9,000円、月20日で18万円
  • 中央値は0.41ドル（約60円）で、同じ通信量でも15倍の差
  • セッションの管理方法次第で天と地ほどのコスト差が生まれる
同じ作業をしていてもセッションの長さで15倍もコストが違う！リセットが鍵。

⑩節約術第1位: CLAUDE.mdの最適化

  • これが一番効く節約術
  • 古いプロジェクトのルールを消す、汎用的な説明は簡潔にする
  • サンプルコードは最小限に抑える
  • 実際にユーザーがCLAUDE.mdを整理して入力トークン40%削減できたケースも
CLAUDE.mdを整理するだけで40%削減！一番手軽で一番効果的な節約術です。

⑪節約術第2位: /clearの戦略的使用

  • 無関係なタスクの間で会話履歴をリセット
  • APIの実装が終わってUIに移るタイミングなど自然の区切りでリセット
  • 古いコンテキストは毎メッセージでトークンを浪費し続ける
  • 使わない古い履歴は消すのが一番。迷わず/clearを
タスクが変わったら/clear！古い履歴を引きずるのが一番のコスト浪費です。

⑫節約術第3位: .claudeignoreの活用

  • node_modules、dist、.next、build などのビルド産物を除外
  • Claude Codeがファイルを探索する際もコンテキストに含まれる
  • 不要なファイルを見ると無駄にトークンを使ってしまう
  • .gitignoreと同じ要領で.claudeignoreを設定するだけ
ビルド産物や巨大な依存フォルダは必ず除外しよう。見るだけでトークンを浪費する。

⑬節約術第4位: 60分ルール

  • 長いセッションは入力が指数関数的に溜まっていく
  • 60分を超えるとコスト効率が急激に落ちる傾向
  • 60分経ったら/clearして新しいセッションで始めるのが正解
  • 健康にも良くて、トークンにも良い。一石二鳥のルール
60分で区切るのが黄金ルール！タイマーをセットして定期的にリセットしよう。

⑭節約術第5位: サブエージェント活用

  • メインエージェントから別のエージェントを呼び出して並列処理
  • コンテキストが分離されてメインセッションが肥大化しない
  • 大きなプロジェクトで複数タスクを同時進行する時に効果的
  • 各エージェントが独立したコンテキストを持つので履歴が混ざらない
メインの会話はスリムに保ちつつ、作業は別エージェントに任せるのが賢い使い方。

⑮Caveman Promptの嘘: 出力削減は全体の一部だけ

  • 「Respond like a caveman」等で出力75%削減と話題に
  • 実際に出力は61〜68%削減できたことは確認
  • しかし総セッションコストの削減はわずか15〜25%にとどまった
  • 入力（キャッシュ読み取り）は一切変わらないから、根本的な解決にはならない
出力を減らしても入力は変わらない。全体の4分の3は入力なんだから当たり前。

⑯数字のマジック: 幹を太いまま枝葉を切るだけ

  • 入力3・出力1の比率で、Caveman Promptは出力（全体の1/4）のみ削減
  • 出力の半分を削っても全体では1/8しか減らない
  • 幹を太いままにして枝葉を切るだけの状態
  • しかも説明が雑になるリスクもあって、副作用も考えられる
幹（入力）を細くしてから枝（出力）を切るのが正しい順序。逆では効果が限定的。

⑰正しい優先順位: 幹を細くしてから枝を切る

  • ①CLAUDE.mdの最適化（幹を細くする — 最大の効果）
  • ②セッション管理（/clearと60分ルールで履歴をスリムに）
  • ③その後でCaveman Promptを検討（枝を切る仕上げ）
  • 幹を細くしてから枝を切るのが正しい順序
順番が大事！CLAUDE.md → セッション管理 → 出力削減の順で取り組もう。
