AIが作るUIはなぜ全部似る？
taste-skillで"slop"を止める

判断基準を渡すだけで、AIの出力が劇的に変わる仕組みを解説
2026年4月23日
①

①全部似るUI問題: AIが作る画面がどれも同じ

  • AIでUIを作ると出来上がりは早いのに全部似てしまう
  • 角丸に句点、青いCTAボタン、余白の取り方までどこかで見たデザインに
  • アニメーションも「安っぽいか派手すぎるか」の2択に偏る
  • 動かないと古臭いし、動きすぎると子供っぽくなる
原因は「どの見た目を採用するか」の判断が弱いからなんです
②

②slop（スロップ）の正体: AIが量産する薄味コンテンツ

  • slop = AIが量産する薄味で凡庸なコンテンツへの蔑称
  • UI版のslopは「パターンの寄せ集め」で判断の芯がない状態
  • AIがそちらに寄る理由は学習データの多数派に引っ張られるから
  • よく見る構成が統計的な正解になり、安全に寄せるほど差分が消える
ヒストグラムの山が全部真ん中に寄ってるイメージ。特徴が消えてしまうんです
③

③判断基準不足が根本原因: 知識不足ではない

  • 実装まで一気にやらせると細かい判断を全部規定値で埋める
  • 判断すべき場面を全部スキップして平均に逃げる
  • 人間のデザイナーは必ず「選ぶ理由」を持っている
  • 例：「この画面は密度を下げて重要な数字だけ拡大する」という理由がある
  • 理由があると余白1つとっても意味が変わる
色や角丸の好みだけ伝えてもslopは止まらない。好みじゃなくて判断基準ごと渡す必要がある
④

④taste-skill登場: センスを渡すスキル

  • taste-skill = AIに「判断基準」を渡す仕組み
  • 中身は単なるテンプレではなく、完全なルール集
  • UIの優先順位、レイアウトタイプ、モーション強度の意思決定ルールを定義
  • 出来上がりを縛るのではなく、判断軸を縛る設計
  • だからReactでもViewでも動くフレームワークとして機能する
出力の形じゃなく考え方を渡してるから、スタックを選ばないのがポイント
⑤

⑤SKILL.mdの中身: ルールの構造

  • taste-skillの本体はSKILL.mdというMarkdownファイル
  • AIが読み込んで判断の基準として使う
  • 見た目の好みを指示するのではなく、選択のルールを配布する仕組み
  • Claude Codeとの相性が良く、追加設定が不要
  • 環境を乗り換えても同じ基準を持ち込める
テンプレじゃなく選択のルールを渡してる。ここがキモです
⑥

⑥導入はコマンド1発: 表紙を開くより簡単

  • npx skills add の後にtaste-skillのURLを渡すだけ
  • スキルMarkdownが自動で入って、AIが読み込む
  • Claude Code専用ではなく、CursorやCopilot、Codexでも使える
  • 環境を乗り換えても同じ判断基準を持ち込めるのが強み
// 導入コマンド（1行だけ）
npx skills add https://github.com/Leonxlnx/taste-skill
たった1コマンドで判断基準が注入される。圧倒的な手軽さ
⑦

⑦GitHub 11.9kスター: 実務の痛みに刺さる

  • taste-skillはGitHubで11.9kスターを獲得（UI系スキル集として相当）
  • フォークも1.1k、ベータ版も近日リリース予定で勢いが続いている
  • 流行ってるというより実務の痛みに刺さってる評価
  • 「UIを出すAIじゃなく、判断を持たせる仕組み」として共感を集めている
スターは人気じゃなく共通の痛みの証拠。「AIのUIなんか違うんだよな」の共感がやっと形になった
⑧

⑧3つのダイアル: DESIGN_VARIANCEで独自性を調整

  • スキルMarkdownの上部に1から10の数値を書くだけで方向性が変わる
  • DESIGN_VARIANCE（デザインの独自性）
    • 2〜3 → 中央寄せでクリーンな雰囲気
    • 8〜9 → 一気に非対称で配置も大胆に
  • 数字1つで出力が変わるので、同じプロンプトで比較できる
// ダイアル設定例
DESIGN_VARIANCE: 7   // 独自性の高さ
MOTION_INTENSITY: 4 // アニメの強さ
VISUAL_DENSITY: 6  // 情報密度
⑨

⑨MOTION_INTENSITY: アニメーションの強さを制御

  • MOTION_INTENSITYでアニメの強さを段階的に調整
  • 1〜2（弱） → ホバーで色が変わる程度の控えめな動き。B2Bの業務アプリなら正解
  • 7〜8（強） → 磁石っぽい吸着の動きが入る。ボタンが少し寄ってくる演出
  • さらに上げるとスクロール連動の演出も追加
  • ブランドLPならそのくらい遊びたい場面がある
動かないと古臭いし、動きすぎると子供っぽい。案件に合わせて調整するのが大事
⑩

⑩VISUAL_DENSITY: 画面の情報密度を調整

  • VISUAL_DENSITYで詰めるか広げるかを制御
  • 低い → 余白多めのランディングページ的デザイン
  • 高い → 情報ぎっしりのダッシュボード的デザイン
  • 管理画面を余白多めで作ると「スクロール大すぎる」と怒られる
  • 逆にLPで密度を上げすぎると見せたい数字が埋もれる
要件と密度が合ってないと痛い目を見る。大事なのはダイヤルを回すだけで判断が切り替わる仕組み
⑪

⑪ダイヤルの実践例: 案件ごとに顔を変える

  • B2B管理画面の設定例:
    • VISUAL_DENSITY → 高め
    • MOTION_INTENSITY → ほぼなし
    • 硬いプロダクトに派手さは不要
  • ブランドLPの設定例:
    • VISUAL_DENSITY → 低め
    • MOTION_INTENSITY → 中くらい
    • 余裕のある空間でブランド感を演出
案件ごとに顔を変えられるのが強い。「高級感のあるモダン」が毎回ぶれない
⑫

⑫9つの変種: 用途別にスキルを差し替える

  • taste-skillには用途別の変種が9種類以上用意されている
  • 特定の見た目を強制しない万能型をベースに、案件の性格で差し替え
  • 迷ったらデフォルトのtaste-skill（万能型）を使うのが基本
  • 刺さるやつだけ選べばOK、全部使う必要はない
万能プロンプトを探すよりスキルを付け替える発想が効く
⑬

⑬softスキル: 上質に寄せる切り札

  • softスキルは上質な雰囲気に寄せたい時の切り札
  • 色味を落として白多め、フォントもプレミアム寄りに
  • ホテルや香水ブランドのあの「品の良さ」を再現できる
  • モーションも滑らかで主張しない、全体のトーンが落ち着く
退避を落として白多め、品の良さを一発で演出。高級ブランド案件に最適
⑭

⑭redesignスキル: 既存プロダクトを直す

  • redesignスキルは既存のUIを改善したい時に地味にいい仕事をする
  • いきなり作り直さず、まず現状の監査から始める
  • レイアウトの不揃い、余白の不整合、整合性の崩れを洗い出す
  • 設計レビューのような動きで、良さを壊さない方向で修正案を提示
いきなり全画面にニューアルを提案されないのがありがたい。現場では0から作り直すのが一番難しい
⑮

⑮outputスキル: 完成品質を強制する安全装置

  • outputスキルはAIが「ざっくり出力で逃げる」のを封じる
  • TODO IMPLEMENTやプレースホルダーで済ますのを禁止
  • 完成コードで修正を強制する安全装置
  • 見た目じゃなく出力品質にフォーカスしたスキル
レビューでTODOを見つけるとガッカリするあれ。これを根絶できる
⑯

⑯スキルの付け替え発想: 万能プロンプトより道具選び

  • taste-skillは見た目の好みだけじゃない、目的に応じた道具選びの仕組み
  • 新規・改善・品質担保など、種類で道具を差し替える
  • 「万能プロンプトを探す」よりスキルを付け替える発想が効く
  • taste-skillの本質は仕組み配布である
「スロップの正体って知識不足より判断基準不足だった」という気づきが大事
⑰

⑰UI以外にも効く: 判断基準を渡す普遍的ノウハウ

  • AIには「入力の指定」と一緒に最低票（採点表）を渡すのが重要
  • 採点表とは観点の明文化。「こういう状態が合格」と先に渡す
  • 人間の新人教育でやることと同じ発想
  • 命名規約、例外設定、性能予算など、全部基準があると迷いが減る
  • レビュー観点をまとめたMDを渡すだけでも効果あり
見るべき点を先に渡すと精度が1段上がる。UIに限らない普遍的なノウハウ
⑱

⑱まとめ: AIには採点表ごと渡せ

  • 今日の学びを1行にすると：「AIには採点表ごと渡せ」
  • 完璧なプロンプトを探すより、まずtaste-skillを入れて3ダイアルで回す
  • 違和感が減る感覚は一発で分かる
  • そこから先は現場に合わせて採点表を育てるフェーズ
  • AIに判断基準を渡す発想を明日から試してみてほしい
taste-skillはUI領域の採点表としてよくできた成功例。この発想はすべてのAI活用に応用できる
