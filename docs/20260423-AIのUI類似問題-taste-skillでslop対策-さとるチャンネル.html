AIのUIが全部似る問題 - taste-skillで"slop"を止める

AIに判断基準を渡すだけで出力が激変する仕組みを解説

taste-skill slop対策 Claude Code UI設計 3ダイアル 判断基準

①AIのUIが全部似てしまう問題

最近AIでUIを作っていると、出来上がりは早いのになんか全部似てしまう。角丸3つと青いCTAボタン、余白の取り方までどこかで見たようなものばかり。言われてみると既視感の塊で、オーバーレイや動きも安っぽいか派手すぎるかの2択になってしまう。

②「slop」とは何か

この現象は最近「slop（スロップ）」と呼ばれている。直訳は「残飯」で、AIが量産する薄いコンテンツへの辛辣な命名。UIにおけるスロップとは、パターンの寄せ集めで判断の芯がない状態。見た目は整っているが、選んだ理由がないまま作られている。

③スロップの根本原因：統計的平均

AIがそっち寄りになる理由は、学習データの多数派に引っ張られるから。よく見る構成が統計的な正解になる。安全に寄せるほど差分が消えて、ヒストグラムの山が全部真ん中に寄ってしまう感じ。個性が統計的に削られていくのだ。

④実装も規定値で埋めてしまう

実装まで一気にやらせると、細かい判断を全部規定値で埋める。判断すべき場面を全部スキップして平均に逃げる。色、丸み、余白、回層、モーションまで、全部「なんとなく規定値」で埋められてしまうのが問題の深さだ。

⑤人間のデザイナーとの違い

人間のデザイナーは必ず選ぶ理由を持っている。例えば「この画面は密度を下げて重要な数字だけ拡大する」とか、理由がある。余白1つとっても意味が変わる。逆に色や角丸の好みだけ伝えてもスロップは止まらない。好みではなく判断基準ごと渡す必要がある。

⑥対策の本丸：選ぶ理由をAIに持たせる

対策の本丸は選ぶ理由をAIに持たせること。出力の形を縛るのではなく、判断軸を縛る。どんなレイアウトにするか、どの程度のアニメーションにするか、情報密度はどのくらいにするか。その「考え方」を渡すことで、ビューでもスタックでも動くフレームワークになる。

⑦taste-skill登場：センスを渡すスキル

登場するのがtaste-skill。「センスを渡すスキル」という名前だが、中身は完全にルール集。レイアウトの優先順位、回層タイプ、モーション強度の意思決定ルールを配っている。テンプレではなく選択のルールを渡すのがキモだ。

⑧コマンド1発で導入完了

導入は驚くほど簡単。コマンド1発で完了する。実行するのは以下のコマンドだけだ。

npx skills add https://github.com/Leonxlnx/taste-skill

これでSKILL.mdが入って、AIが読み込む。追加設定は一切不要。

⑨GitHub 11.9kスターの実力

taste-skillはGitHubで11.9kスターを獲得。フォークも1.1kに達している。UI系のスキル集としては相当な数字。人気ではなく実務の痛みに刺さっている証拠。UIを出すAIではなく、判断を持たせる仕組みとして評価されている。「AIのUIなんか違うんだよな」という共通の悩みにようやく答えが出た。

⑩Claude Code以外でも使える

Claude Codeと相性が良いが、Cursor、GitHub Copilot、Codexでも使える。出力の形ではなく考え方を渡す設計なので、スタックを選ばない。環境を乗り換えても同じ判断基準を持ち込めるのが強み。日々ベータも近日リリース予定で勢いが続いている。

⑪3つのダイアルで振る舞いを制御

taste-skillの面白い特徴が3つのダイアル。SKILL.mdの上部に1から10の数値を書くだけで、方向性が変わる。数字1つで出力が切り替わるので、プロジェクトごとに微調整が可能だ。3つのダイアルとは以下の通り。

  • DESIGN_VARIANCE（レイアウトの冒険心）
  • MOTION_INTENSITY（アニメーションの強さ）
  • VISUAL_DENSITY（情報密度）

⑫DESIGN_VARIANCE：レイアウトの冒険心

1つ目のダイアルはDESIGN_VARIANCE。値を2や3に下げると中央寄せでクリーンな配置になる。8や9に上げると一気に非対称で大胆な配置に変化する。同じプロンプトでも、この値を変えるだけで全く違うレイアウトが生成される。冒険したい度合いを数値で指定できる。

⑬MOTION_INTENSITY：アニメーションの強さ

2つ目はMOTION_INTENSITY。1や2だとホバーで色が変わる程度の控えめな動き。B2Bの業務アプリならそれが正解。7や8まで上げると磁石っぽい吸引の動きが入る。ボタンが少し寄ってくるような演出だ。さらに上げるとスクロール連動の演出も入る。ブランドLPならそのくらい遊びたい場面がある。

⑭VISUAL_DENSITY：画面の情報密度

3つ目はVISUAL_DENSITY。低くすると余白多めのLP、高くすると情報詰めのダッシュボードになる。管理画面を余白多めで作ると「スクロール大杉」と怒られる。逆にLPで密度を上げすぎると見せたい数字が埋もれる。要件と密度が合っていないと違和感が出る。

⑮案件別のダイアル設定例

ダイヤルを回すだけで判断が切り替わる。具体的な当て方は以下の通り。

  • B2B管理画面：分散・密度高め・モーションほぼなし
  • ブランドLP：密度低め・分散中高・モーション中

「高級感のあるモダンな」という漠然とした指定でも、ダイヤルのおかげで再現性が出る。「モダン」の解釈が毎回ぶれる問題が解消されるのだ。

⑯用途別バリアント9種類

taste-skillには用途別のバリアントが9種類ある。案件の性格に合わせて差し替えが可能だ。

  • taste-skill：万能型（迷ったらこれ）
  • soft-skill：上質に寄せる切り札
  • redesign-skill：既存プロダクトの改善
  • output-skill：完成コード強制

⑰soft-skillとredesign-skillの特徴

soft-skillは上質に寄せる時の切り札。退色を落として箔多め、フォントもプレミアムよりになる。ホテルや香水ブランドのシックな雰囲気が作れる。モーションも滑らかで主張しない。

redesign-skillは既存プロダクトの改善用。いきなり作り直さず、まず現状の問題を監査する。回層の不揃い、余白の不整合、精緻さの崩れを洗い出し、既存の良さを壊さない方向で修正案を出す。現場で0から作り直すのが一番難しいから、このアプローチが重宝する。

⑱最も重要な学び：判断基準を渡せ

今日の学びを一行にすると「AIには最低表（評価基準）ごと渡せ」。人間の新人教育でやることと同じだ。こういう状態が合格かを先に渡しておく。この発想はUIに限らない。命名規約、例外設定、性能予算など、すべて基準があれば迷いが減る。レビュー観点をまとめたMDを渡すだけでも効く。見るべき点を先に渡すと制度が1段上がる。
