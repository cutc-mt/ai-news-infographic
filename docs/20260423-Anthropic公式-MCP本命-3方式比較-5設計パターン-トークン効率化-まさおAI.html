Model Context Protocol徹底解説 - API直叩き vs CLI vs MCP

2026年4月23日

①Anthropic公式ブログ記事の背景

  • Anthropicの公式ブログでMCP（Model Context Protocol）に関する詳細な記事が公開された。
  • 本番環境でAIエージェントを運用するなら、外部システムとの接続はMCPを中心にするのが合理的という主張。
  • AIエージェントが実際の業務で役立つには、外部APIやデータベースなどと接続する仕組みが不可欠。
  • その接続方式として3つの方法を比較検討し、MCPが最適であることを論理的に説明。
Anthropic自身が「MCPこそが本命」と宣言したことで、プロトコルの信頼性が大きく向上した！

②MCP（Model Context Protocol）とは

  • AIモデルと外部ツール・データソースを接続するための標準化されたプロトコル。
  • 一度MCPサーバーを作れば、対応するすべてのクライアントでそのまま利用可能。
  • AIエージェントフレンドリーな設計で、モデルがツールを理解しやすく、正確に呼び出せる。
  • 認証やセキュリティも標準化されており、OAuthによる安全なアクセス管理を実現。
USBのようなもの！一度規格に合わせて作れば、どのパソコンでも使えるのと同じ仕組み。

③接続方式①: API直叩き

  • メリット: 実装がシンプルで、REST APIを直接呼び出すだけ。
  • デメリット: 共通レイヤーがなく、各クライアントごとに個別にAPI統合が必要。
  • M×N問題が発生する。M個のモデルとN個のAPIを組み合わせると、M×N個の統合作業が必要になる。
  • 評価: スモールスタート向き 本番不向き
小規模なプロトタイプならAPI直叩きで十分。でもスケールすると破綻するよ！

④接続方式②: CLI経由

  • メリット: ローカル環境では最適化されており、既存のCLIツールをそのまま活用可能。
  • デメリット: 共通層が薄く、認証やパース処理が独自実装になりがち。
  • クラウド環境ではCLIが使えない場合が多く、本番エージェントとの相性が悪い。
  • 評価: ローカル最適 クラウド不向き
Claude Codeなどローカルツールでは優秀。でも本番のクラウドエージェントには不向きなんだ。

⑤接続方式③: MCPプロトコル（推奨）

  • 標準化された共通レイヤーにより、一度作れば全クライアントに届く。
  • AIエージェントフレンドリーな設計で、モデルがツールの意味を正確に理解できる。
  • クラウド環境でも問題なく動作。本番エージェントはクラウドで動くためMCPが自然な選択肢。
  • 評価: 本番最適 スケーラブル
Anthropicが推すのはこれ！MCPこそが本番AIエージェントの「本命」接続方式。

⑥3方式の比較まとめ

項目 API直叩き CLI経由 MCP
実装の容易さ ◎ ◎ △
共通レイヤー なし 薄い 厚い
スケーラビリティ M×N問題 ローカル限定 高い
クラウド対応 対応 不向き 最適
AIフレンドリー 普通 普通 高い

本番環境の総合評価ではMCPが圧倒的。初期コストは高いが長期的には圧倒的にお得！

⑦設計パターン①: リモートMCPサーバー

  • どこからでもアクセス可能なリモートサーバーとしてMCPをホストする設計。
  • ローカル環境に依存せず、複数のユーザーやエージェントが同じMCPサーバーを共有。
  • クラウド上で動く本番エージェントが自然にアクセスでき、運用コストが下がる。
  • Anthropicの公式ガイドでも、本番運用ではリモートMCPサーバーが基本と推奨されている。
本番エージェントはクラウドで動くから、ローカルMCPサーバーでは意味がない。リモートが王道！

⑧設計パターン②: 意図単位でツールをまとめる

  • アンチパターン: APIを1対1でツールにマッピングすると、ツール数が膨大になる。
  • 正しい設計: 一連の操作を1つのツールにまとめる。例えば「ファイルを作成してコミットする」を1ツールに。
  • ユーザーの意図（インテント）を単位にすることで、AIがツールを正確に選択できる。
  • ツール数が減ることでトークン消費も削減され、レスポンスも高速化される。
API 100個＝ツール100個はNG！
「やりたいこと」単位でまとめれば、AIも迷わずユーザーもハッピー！

⑨設計パターン③: コード指揮設計

  • AIに直接操作させるのではなく、スクリプトを生成させてからサンドボックスで実行する設計。
  • サンドボックス実行により、安全性を確保しながら柔軟な操作が可能。
  • 生成されたコードをレビュー・検証してから実行するため、予期しない動作を防止。
  • より複雑な操作や条件分岐も、コードとして表現できるため柔軟性が高い。
AIに直接データベースを操作させるのは怖い。コードを生成→確認→実行なら安心！

⑩設計パターン④: Interactive UI

  • MCPツールがUIコンポーネントを返すことで、インラインレンダリングが可能に。
  • 単なるテキストではなく、グラフやフォームなどのリッチなUIを直接表示。
  • ユーザーがその場で操作・確認できるため、インタラクティブな体験を提供。
  • 結果の可視化や確認をシームレスに行え、ユーザビリティが大幅に向上する。
テキストだけじゃなくて、ボタンやグラフがそのままチャット内に表示される的未来な仕組み！

⑪設計パターン⑤: 標準化されたOAuth

  • MCPプロトコルに標準化されたOAuth認証フローが組み込まれている。
  • 初回のみOAuth認証が必要で、再認証の頻度が激減。ユーザー体験が大幅に改善。
  • Anthropicが公式に推奨する認証方式で、セキュリティと利便性を両立。
  • トークンの自動リフレッシュも標準対応しており、認証切れによる作業中断を防止。
毎回パスワード入力…から解放！
初回だけ認証すれば、あとは自動で管理してくれる優れもの。

⑫トークン効率化: ツールサーチ

  • ツールサーチは、実行時に必要なツールだけを読み込む仕組み。
  • すべてのツール定義を常にコンテキストに含めるのではなく、動的に検索して必要なものだけをロード。
  • ツール数が100個以上あっても、一度に読み込むのは数個だけなのでコンテキストが肥大化しない。
  • トークン消費を大幅に削減でき、APIコストの節約にも直結する。
100個のツール定義を全部読むのは無駄！必要な時、必要な分だけ読み込むのが賢いやり方。

⑬プログラマティックツールコーリング

  • 生のデータをそのまま返すのではなく、整形して必要な部分だけを返す仕組み。
  • 37%のトークン削減を実現。これはAPIコストとレスポンス速度の両方に大きく貢献。
  • プログラム的にツールの結果を加工するため、AIが処理しやすい形式でデータを受け取れる。
  • 不要なメタデータや冗長な情報を削ぎ落とし、本質的な情報だけを抽出。
37%削減はデカい！大量のAPI呼び出しをするエージェントほど恩恵が大きいよ。

⑭スキルとMCPの関係

  • スキルはMCPの上位概念。複数のMCPツールを組み合わせた「手続き的知識」。
  • 例: 「GitHubでPRを作成する」というスキルは、複数のMCPツール（ブランチ作成、コミット、PR作成）を連携させる。
  • MCPが「個別のツール」を提供するのに対し、スキルはワークフロー全体を定義。
  • スキルレイヤーを導入することで、より複雑なタスクの自動化が可能になる。
MCP＝道具、スキル＝道具の使い方。
いい道具（MCP）を揃えて、使い方（スキル）も整理するのが理想！

⑮なぜ本番エージェントにMCPが最適なのか

  • 本番AIエージェントはクラウドで動く前提。ローカル前提のCLI方式とは相性が悪い。
  • MCPならリモートからアクセス可能で、クラウドエージェントと自然に統合できる。
  • 標準化により、新しいAIモデルやクライアントが登場しても再実装が不要。
  • セキュリティ・認証・エラーハンドリングも統一されており、運用上の安心感が高い。
「クラウドで動く」＝「ネットワーク越しにツールを使う」＝ MCPの出番！という論理。

⑯MCPエコシステムの展望

  • MCP Registryが充実しており、様々なサービスのMCPサーバーが公開されている。
  • コミュニティ主導でオープンソースのMCPサーバーが続々と登場。カバレッジが急速に拡大中。
  • Slack、GitHub、Google Drive、Notionなど主要SaaSのMCPサーバーが既に利用可能。
  • Anthropicの公式推奨により、今後さらに標準化が進むことが期待される。
エコシステムが育てば、自分で作らなくても既存のMCPサーバーを組み合わせるだけで済む未来が来る！

⑰実践: MCPサーバー開発のポイント

  • 意図単位の設計を最優先に。APIの1対1マッピングは避ける。
  • ツールの説明文は明確で具体的に。AIが正しくツールを選択できるようにする。
  • 返却データは最小限に絞り、プログラマティックに整形してから返す。
  • 認証は標準OAuthを利用し、独自実装を避ける。 
// 良い例: 意図単位のツール
tool: "create_github_pr",
description: "ブランチを作成し、
  変更をコミットしてPRを作成"

// 悪い例: APIの1対1マッピング
tool: "git_create_branch"
tool: "git_add_files"
tool: "git_commit"
tool: "git_create_pull_request"
API設計の基本は「利用者の視点」。AIエージェントにとって使いやすい設計を心がけよう！
