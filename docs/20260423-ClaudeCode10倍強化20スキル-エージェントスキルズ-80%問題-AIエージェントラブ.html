AIコード生成の進展と80%問題

① 80%問題とは何か

  • AI研究者のアンドレ・カーパシーは、80%のコードがAIによって書かれ、20%が人間によって編集されるようになったと述べています。
  • この変化は2025年後半に急速に進行しました。
  • 以前は「70%問題」と言われていたAIの限界が、80%以上に到達してより大きな課題に変化しています。
人間はもはや英語でプログラミングしているに過ぎない、という指摘も！

③ AIエラーの変化: 構文から概念へ

  • AIのエラーは構文エラーから概念的な失敗へと変化しました。
  • コードとしては完璧に動きテストも通るが、作っているものが根本から間違っている状態が増えています。
  • カーパシー氏は「AIが間違った過程を行わずに実行することがある」と警告。
問題はバグではなく、AIの性質そのものに起因する！

④ 理解の負債（Comprehension Debt）

  • ジェレミー・ハワードはこの現象を「理解の負債」という用語で表現しています。
  • LLMが実装したものが機能するように見えると、単に鵜呑みにしてしまう危険性。
  • エージェントは疲れず、自信を持って実装を続けます。コードは妥当に見え、テストもパスします。
  • 時間が経つほど自分のコードベースを理解できなくなっていきます。
10%間違えていても「あと1つのプロンプトで直せる」と思い込む心理的フックが実在する

  • 「AIにコードを書かせているけど、その後の確認がかなり大変」という声が増加。

⑧ 反合理化メカニズム（Anti-Rationalization）

  • AIに機能追加を指示すると「とりあえず実装だけして、テストは後で追加します」と言い訳をする。
  • しかしその「後で」は永遠に来ない。
  • エージェントスキルズではテストの後回しを一切許容しない。必ず実装とテストをセットで出力することをシステムレベルで強制。

⑨ チェスタトンの柵（Chesterton's Fence）

  • 元々は哲学の概念。なぜそのルールが存在するのかを完全に理解するまで決して取り払ってはいけないという原則。
  • AIにリファクタリングを指示すると、一見無駄に見える条件分岐をスパッと消してしまう。
  • でも実はその条件分岐が特定のタイムゾーンのユーザーを救う重要なエッジケース処理だったりする。
既存の振る舞いを1mmも変えずにコードを簡略化する！まず「なぜ存在するか」をドキュメント化。

⑩ 盲目的同意（Sycophancy）の危険

  • AIは人間に反論するよりも、もっともらしい出力を生成するように最適化されていることが多い。
  • 例：間違った前提「このアルゴリズムはO(N)最適化」を書くと...
  • シニアエンジニアなら「ネストがあるからO(N²)だ」とツッコミを入れるが、AIは「素晴らしいアプローチですね」と同意してしまう。
  • その間違った前提の上に何百行ものコードが積み上がる「前提条件の暴走」が発生。

⑪ 宣言的開発へのシフト

  • 命令型から宣言型への完全な移行。
  • 旧: 「Xを取りYを返す関数を書け。このライブラリを使え」
  • 新: 成功基準とテストケースを明確に定義してAIに解決策を探させる
  • LLMは特定の目標を満たすまでループすることが非常に優れている。

⑬ 連続コーディングループ（LLMガムテクニック）

  • タスクを極小のアトミックユーザーストーリーに徹底的に分割。
  • AIは「タスクを選ぶ → 実装する → テストをパスさせる → コミットする」のサイクルを実行。
  • 1つのタスク完了後、AIのセッションコンテキストを完全にワイプして真っさらな状態にリセット。
# ループの流れ
1. タスクを選択
2. 実装
3. テストをパス
4. コミット
5. コンテキストをワイプ ← ここが重要
6. 次のタスクへ（1に戻る）

⑭ 短期記憶と長期記憶の分離

  • 短期記憶: progress.txt — 直近のタスクで何に成功したか、どのファイルを変更したかを記録。
  • 長期記憶: agents.md — プロジェクト特有のルールや回避すべきエラーを永続的に記録。
  • 新しいループが始まるたびに、agents.mdの内容がシステムプロンプトとしてAIに再注入される。
映画「メメント」の主人公のように、メモで記憶をつなぎ合わせる仕組み！

⑮ 平行ワークスペースで競合を防止

  • Gitのワークツリー機能を活用して、完全に分離された物理的ワークスペースを構築。
  • 実装用、テスト用、レビュー用のワークスペースを同時に展開。
  • 複数のエージェントが同時に5〜10個のループを走らせてもお互いの変更を上書きしない。
別々のサウンドステージで同じ映画の別シーンを同時撮影するようなイメージ

⑯ 特化型サブエージェントの使い分け

  • プランナー: アーキテクチャーの計画を立てるエージェント
  • ワーカー: 実際にコードをタイピングするエージェント
  • レビューアー: セキュリティの脆弱性を探すエージェント
  • 1つのAIモデルに全ての役割を持たせるのは非効率。特化型ペルソナを使い分けることが鍵。
それぞれが専門の部屋で仕事をするイメージ！

⑰ CI/CDガードレール: AIの暴走を防ぐ

  • AIにシェルコマンドやGitのコミット権限を渡すことは、本番環境のデータベース消去や幻覚によるバグのデプロイの危険を伴う。
  • サンドボックス化された環境での実行と、基本権限を読み取り専用に保つガードレールが必要。
  • AIが書いたコードはCI/CDパイプライン、厳格なリント、自動テストを完全通過しない限りメインブランチにマージされない物理的な壁を構築。
AIを自由に走らせるためには、かつてないほど厳格なシステムの土台が必要！

⑱ エンジニアの未来: 二極化と新たな道

  • カーパシー氏: LLMによってコードを書くことを楽しむエンジニアとビルディングを好むエンジニアに分かれると予測。
  • DORAレポート: AIはプロセスの増幅器。良いプロセスはより良くなり、悪いプロセスはより悪くなる。
  • 高パフォーマンスチームは55〜77%高速化、一方で技術負債は前例のない速度で蓄積。
  • 成功するのは、キーボードから手を離しても理解を維持する方法を知っている開発者。
AIは強力なツールに過ぎない。そのツールが創り出す世界をデザインするのはいつだってあなたなのです。
