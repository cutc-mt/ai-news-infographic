①Qwen 3.6 27Bとは: オープン系の強力なコーディングモデル

  • Qwen 3.6 27Bはオープンモデルとして、エージェント型のコーディングに特化して設計されたモデルです。
  • 長いコンテキストでの推論能力と、リアルなコーディングワークフローへの対応が強化されています。
  • リポジトリレベルの理解や、複数ファイルにまたがる変更にも対応できる能力を持っています。
27Bというサイズは、ローカル運用と性能のバランスが絶妙なんだ！

②エージェント型コーディングに最適化: 実際のワークフローに強い

  • 単なるコード補完ではなく、エージェント型ワークフローに最適化された設計です。
  • リポジトリ全体の構造を理解して、的確な変更を提案する能力が向上しています。
  • 長いやり取りでも文脈を維持したまま対話を続けられるのが最大の強みです。
従来のモデルは長い対話で文脈を失いがちだったけど、Qwen 3.6は違う！

③Ollamaの現状: 27Bはまだ未対応

  • 現時点でOllamaが対応しているのは35B A3Bという変種のみです。
  • 本命の27Bモデルは、Ollamaではまだ利用できません。
  • そのため、27Bを使いたい場合は別の方法でサーブする必要があります。
Ollamaファンは残念だけど、27Bを使うなら別ルートが必要だよ！

④vLLMがベストルート: 27Bを安定してサーブ

  • vLLMを使うのが、27Bモデルをサーブする最良の方法です。
  • OpenAI互換のエンドポイントを提供するため、各種ツールとの連携がスムーズです。
  • 高速な推論と安定したパフォーマンスを期待できます。
vLLMならOpenAI API互換だから、既存のツールチェーンがそのまま使える！

⑤インストール手順: Python + UV + vLLM

  • まずPythonとUV（パッケージマネージャー）を準備します。
  • vLLMのインストールは以下のコマンド一発で完了します。 
# vLLMのインストール
uv pip install vllm

# サーバーの起動
vllm serve <model-name> --port 8000
UVを使うと依存関係の解決が爆速！pipより圧倒的に速いよ。

⑥ツール呼び出しが超重要: 設定を間違えると動かない

  • ツール呼び出し（Tool Calling）の設定が、エージェント動作の成否を分けます。
  • auto tool choiceとパーサーを正しく有効にしないと、モデルがツールを呼び出さずに説明文を返してしまいます。
  • フラグを正しく設定すれば、モデルが確実にツールを呼び出すようになります。
ここを間違えると「ツールを呼ぶべきなのに、ツールの説明を書くだけ」になっちゃう！

⑦vLLMの完全コマンド: 起動オプションを解説

  • サーバー起動に必要な主要オプションをまとめています。 
# vLLMのフル起動コマンド
vllm serve Qwen/Qwen3.6-27B \
  --port 8000 \
  --tensor-parallel-size 1 \
  --max-model-len 32768 \
  --enable-auto-tool-choice \
  --tool-call-parser hermes
tensor-parallel-sizeはGPUの枚数に合わせて調整してね！

⑧コンテキスト長は最大に: ハードウェアの限界まで

  • max-model-lenは、ハードウェアが許す限り大きく設定すべきです。
  • 長いコンテキスト = 強いエージェントワークフローという関係にあります。
  • エージェントが多数のファイルを読み込みながら作業するには、コンテキストウィンドウが広いことが不可欠です。
VRAMが許すなら32K以上を設定しよう！コンテキストが広いほど賢く動くよ。

⑨MLX対応がまもなく: Macユーザーにも期待

  • Apple Silicon向けのMLX対応が近づいています。
  • MacユーザーはMシリーズチップの統合メモリを活かしたローカル推論が可能になります。
  • リリース情報を随時チェックすることを推奨しています。
Macユーザーはもう少し待てば、GPUなしでも27Bが動くかもしれない！

⑩Kilo CLI連携: OpenAI互換で簡単接続

  • Kilo CLIのインストールはnpmコマンド一つで完了します。 
# Kilo CLIのインストール
npm install -g @kilocode/cli
  • OpenAI互換プロバイダーとして設定して、localhost:8000を指定するだけです。
  • vLLMサーバーへの接続は数ステップで完了します。
OpenAI API互換だから、エンドポイントのURLを変えるだけで使える！

⑪Kilo Claw: ホスト型エージェント体験

  • Kilo Clawは、ホストされたエージェント体験を提供します。
  • ローカルサーバーの管理なしに、永続的なエージェントをセットアップできます。
  • 手軽さを重視するならKilo Clawが最適です。自分でサーバーを立てる手間が省けます。
サーバーの設定が面倒な人は、Kilo Clawでサクッと始めるのがおすすめ！

⑫Hermes Agent: 最も魅力的な連携先

  • Hermes Agentが、Qwen 3.6 27Bにとって最も魅力的な統合先です。
  • ツール呼び出しのパーサーもHermes形式に対応しており、相性が抜群です。
  • Hermes + Qwen 3.6 27Bの組み合わせが、ローカルAIコーディングのベストプラクティスです。
ツールパーサーもHermes対応だから、vLLMの起動フラグともピッタリ合う！

⑬Hermesの設定手順: カスタムエンドポイントの指定

  • ターミナルでhermes modelコマンドを実行して設定を開始します。
  • カスタムエンドポイントを選択して、ベースURLを設定します。 
# Hermesの設定
hermes model
# → Custom Endpointを選択
# → Base URL: http://localhost:8000/v1
ベースURLの最後に/v1を忘れないようにね！

⑭設定ファイルで管理: .hermes/config.yaml

  • 設定は.hermes/config.yamlファイルで一元管理できます。 
# config.yamlの例
provider: openai-compatible
base_url: http://localhost:8000/v1
default_model: Qwen/Qwen3.6-27B
context_limit: 32768
  • プロバイダー、モデル名、コンテキスト制限を明示的に指定することで、設定ミスを防げます。
YAMLファイルで管理すれば、チーム内で設定を共有できるよ！

⑮ツール使用の強制: 確実に呼び出させる設定

  • モデルがツールを「説明」するだけで「呼び出し」しない問題を解決する必要があります。
  • vLLMのenable-auto-tool-choiceフラグとtool-call-parserを確実に有効にします。
  • これらの設定により、モデルは確実にツールを呼び出すようになります。エージェントとして正しく機能するための必須設定です。
ツール呼び出しが動かないと、エージェントはただのチャットボットになっちゃうからね！

⑯サブエージェントの継承: 一貫したワークフロー

  • Hermes Agentではサブエージェントがメインモデルの設定を自動的に継承します。
  • 個別にモデルやエンドポイントを設定し直す必要はありません。
  • 一貫したワークフローを維持しながら、複雑なタスクを分担できます。
メインエージェントとサブエージェントが同じモデルを使うから、動作が安定する！
