Claude Context でコード理解の摩擦を消す設計

ハイブリッド検索 + ASTチャンキング + Merkle Tree が実現する次世代のコンテキスト管理

2026年4月24日

問題提起: コードをLLMに渡す3つの失敗パターン

① 手動ペーストの罠

  • ファイルの内容をコピペしてLLMに投げる方法は、最初は手軽に思える
  • しかし更新のたびに貼り直しが必要で、すぐに継続コストが膨らむ
  • 数ファイルなら耐えられても、プロジェクト全体をカバーするのは現実的ではない
コピペ地獄に陥ると、更新頻度が下がってLLMの回答も古くなっていく…

② エージェント探索ループの無限地獄

  • grep → read → grep → readの無限ループに陥る
  • 命名揺れ（同じ概念なのに名前が違う）で検索が的外れになる
  • 実装が複数ファイルに分散していると、関連箇所の特定が困難
  • 古いコードが残存していると、LLMが間違った実装を参照してしまう
ツールコールを大量に消費して、トークンも無駄に増えるのが最大の問題

③ ディレクトリ丸投入のノイズ問題

  • プロジェクト全体をそのままLLMに投げる「全部入り」アプローチ
  • node_modulesなどのノイズが大量に混入する
  • 本当に重要なファイルがノイズに埋もれて、LLMの精度が下がる
「渡してから選ぶ」ではなく「選んでから渡す」のが正しいアプローチ

④ 3つの失敗に共通する根本原因

  • どの失敗パターンにも関連性のフィルターが後付けという共通点がある
  • 「全部渡してからLLMに選ばせる」→ 本当に必要な情報が埋もれる
  • 正しい設計は「選んでから渡す」→ 関連性を事前に判定して必要な分だけ提供
この「選んでから渡す」を実現するのが Claude Context の設計思想

⑤ Claude Context とは何か

  • Zilliz（Milvus開発元）が開発したオープンソースMCPプラグイン
  • MilvusはGitHubスター4万超のベクトルデータベース
  • 2025年8月リリース、わずか8ヶ月で7000スターを獲得
  • 15以上のクライアントに対応（Claude Code / Cursor / Windsurf / Cline 等）
役割は「インデックス作成＋検索」のみ。生成は呼び出し元のLLMが担当する賢い設計

⑥ Claude Context の3本の技術柱

  • 柱1: ハイブリッド検索（BM25＋ベクトル検索）+ RRF統合
  • 柱2: ASTチャンキング（Tree-sitterで関数・クラス単位に分割）
  • 柱3: Merkle Treeによる増分インデックスで変更分のみ再計算
この3つが連携して「必要なコードだけを必要な分だけ」LLMに渡す仕組みを実現

技術解説: 3本の柱を深掘り

⑦ 柱1-A: BM25 キーワード検索

  • キーワードの完全一致に強い古典的な検索アルゴリズム
  • 関数名やエラー文字列など、正確な文字列を探す場面で威力を発揮
  • 「getUserById」という関数名を検索すると、確実にその関数を特定できる
BM25単体だと表記揺れに弱いのが弱点。だからベクトル検索と組み合わせる

⑧ 柱1-B: ベクトル意味検索

  • テキストをベクトル（数値の配列）に変換して意味の近さで検索
  • 表記揺れに強いのが最大の強み（「ユーザー取得」≠「getUser」でも意味でHit）
  • 「ログイン処理」というクエリで、signin()やauthenticate()も見つけられる
ベクトル検索単体だとキーワード完全一致に弱い。だからBM25と組み合わせる

⑨ 柱1-C: RRF で2つの検索を統合

  • Reciprocal Rank Fusion（逆順位融合）でBM25とベクトル検索の結果を統合
  • 各検索結果の順位の逆数を足し合わせて、統合スコアを算出
  • スケール差を吸収して、どちらの検索の良さも活かせる
RRFのおかげで「キーワード完全一致」と「意味的近さ」の両方をカバーできる

⑩ 柱2: ASTチャンキング（意味的分割）

  • Tree-sitterを使ってコードの構文木（AST）を解析
  • 関数単位・クラス単位でチャンクを切り出す（文字数分割ではない）
  • 対応言語: Python / JavaScript / TypeScript / Java / C++ / Rust 等
  • 未対応言語はLangChainのCharacterTextSplitterにフォールバック
文字数で分割すると関数の途中で切れる→ASTなら意味のまとまりが保たれる

⑪ 柱3: Merkle Tree による増分インデックス

  • ファイルハッシュで変更を高速に検出
  • 変更があったファイルだけ再計算するので、差分更新が超高速
  • スナップショットを永続化するので、PCを再起動しても整合性を維持
Gitがコミットの差分を管理するのと同じ仕組みをインデックスに応用したイメージ

⑫ 導入効果: トークンとツールコールを大幅削減

  • トークン約39.4%削減を達成
  • ツールコール約36.3%削減を達成
  • 計測ベンチマーク: SWE-bench Liteの30サブセットで検証
  • 探索ループの発生率が下がり、コストも回答精度も改善
トークン約40%削減＝コストも速度も大きく改善する実用的な効果

セットアップと他ツールとの比較

⑬ セットアップ: クラウド構成

  • Zilliz Cloud + OpenAI埋め込みを組み合わせた構成
  • コマンド1発でベクトルDBと埋め込みモデルの登録が完了
  • 自前でインフラを構築しなくていいので、最速で使い始められる
クラウド構成なら数分でスタート。コードに外部APIを送っていいプロジェクトに最適

⑭ セットアップ: ローカル構成

  • Milvus Docker + Ollamaを使った完全ローカル構成
  • コードを外部APIに送りたくない場合は、この構成が必須
  • 埋め込みモデルにはNOMIC Embed Textを使用
注意点: Node.js v22推奨 / Milvusアドレスは明示指定 / .gitignoreの確認が必須

⑮ 他ツールとの関係: Context7

  • Context7（Upstash開発）は外部OSSドキュメントの参照に特化
  • Claude Context → 自分のプロジェクト内のコード検索
  • Context7 → 外部ライブラリのドキュメントを検索
両方を併用すると「自前コード＋外部ドキュメント」の両方を最強にカバーできる

⑯ 他ツールとの関係: Serena / Cursor内蔵

  • Serena: LSPベースのシンボル参照。リファクタリングに最適
  • Cursor内蔵検索: 個人でCursorを使うならClaude Contextは不要な場合も
  • Claude Contextの価値はチーム統一基盤としてIDEを問わず同じ検索を提供できる点
個人開発ならCursor内蔵で十分。チームで横断的に使いたい時にClaude Contextが輝く

⑰ 導入判断: 3つの質問で決める

  • Q1: コードを外部APIに送ってOK？ → YESなら選択肢が広い
  • Q2: 10万行以上の横断検索が必要？ → YESなら導入価値あり
  • Q3: IDEを問わず同じ検索が欲しい？ → YESなら統一基盤の価値あり
3つともNOなら無理に導入しなくてOK。小規模プロジェクトは他の手段でも十分

⑱ まとめ: コード理解の摩擦をゼロに

  • ハイブリッド検索でキーワードも意味も両方カバー
  • ASTチャンキングでコードの意味を保ったまま分割
  • Merkle Treeでインデックス更新を最小限に維持
  • 結果としてトークン約40%削減・ツールコール約36%削減を達成
核心: 「渡してから選ぶ」を「選んでから渡す」に変える設計転換こそが、LLMコーディングの効率を根本から改善する
