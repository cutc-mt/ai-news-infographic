RIP Roo & Cursor: AIコーディングツールの終焉と新時代

RooCodeの終了とCursorのSpaceX買収問題 — Kiloが描くモデル自由の未来

📅 2026年4月24日

①RooCodeが終了: サービス停止の発表

  • RooCodeがVS Code拡張機能、クラウド、ルーターの全サービスを5月15日に終了すると発表
  • 数百万のインストールを誇り、エージェント型コーディングの概念を先駆けて開拓したツール
  • Kiloのブログタイトルは「Thank you Roo, we'll take it from here」
  • カスタムモード、アーキテクト/コード/デバッグモード、ファイルやターミナルへのアクセスなどが機能として存在
数百万のユーザーが頼りにしていたツールが、わずか1日で終了の知らせに。
時代の転換点

②Rooの主張: IDEは未来ではない

  • RooはIDE（統合開発環境）は未来ではないと判断し、リモート・クラウドへ移行する方針
  • VS Code拡張機能という形態から脱却し、クラウドベースのエージェントへシフト
  • Rooが開拓したエージェント型コーディングの概念は、他のツールに引き継がれていく
開発の未来はクラウド上のエージェントにあるというRooの信念。でも本当にそうだろうか？
IDE vs クラウド

③Kiloが引き継ぐ: VS Codeに残る理由

  • KiloはRooの判断に反対し、VS Code拡張機能をゼロから再構築した
  • CLIとクラウドエージェントと同じコア基盤の上にVS Code拡張機能を構築
  • Kiloの主張: エディタは開発者が生活する場所だからこそ重要
  • 差分の確認、判断の決定、ローカルコマンドの実行はすべてエディタ上で完結する
「開発者はエディタに住んでいる」— Kiloの強い信念がVS Code残留の理由
エディタ第一主義

④Kiloの新機能: 真の並列実行

  • 真の並列実行により、複数のタスクを同時に処理できる
  • サブエージェント委譲で、メインエージェントが作業を細分化して委任
  • エージェントマネージャーで複数のエージェントを統合的に管理
  • インライン差分レビューでエディタ内で直接変更を確認
  • セッションをターミナルとエディタの間でシームレスに移動可能
Rooが去った後、KiloがVS Code拡張機能を最強に進化させた！

⑤SpaceXがCursorの買収権を獲得

  • SpaceXが今年後半にCursorを600億ドルで買収する権利を獲得
  • または、共同作業に対して100億ドルを支払う選択肢も存在
  • まだ完全に買収されたわけではなく、SpaceXが巨大なオプションを持っている状態
  • コーディングアシスタントがモデルの配信層になりつつある懸念
600億ドルという巨額の買収オプション — AIコーディング市場の地殻変動が始まった

⑥市場の構造変化: モデル配信層の争奪戦

  • OpenAIはCodexを自社で保有
  • AnthropicはClaude Codeを自社で展開
  • Googleもコーディング製品を独自に展開
  • SpaceXはxAIを保有し、Cursorとの契約で勢力拡大
各社がコーディングツールを囲い込み — ユーザーはどのエコシステムに属するか選択を迫られている

⑦Cursorの強みだったモデル自由度

  • Cursorが人気だった理由はモデルを自由に選べることにあった
  • Claudeが最適ならClaudeを、OpenAIが最適ならOpenAIを、柔軟に切り替え可能だった
  • しかしSpaceX/xAIとの提携により、モデルの自由度が徐々に失われる可能性
ユーザーがCursorを選んだ最大の理由がモデルの自由度だったという皮肉

⑧Windsurfの前例: Anthropicがアクセス切断

  • WindsurfがOpenAIに買収された際、AnthropicがClaudeへのアクセスを突然切断
  • これがCursorの将来を考える上で重要な前例となる
  • もしCursorがSpaceX/xAIと完全に結びつけば、他社モデルの利用に制限がかかる可能性
Windsurfの例が示している — 買収＝モデルの制限という構図

⑨モデル自由の重要性: 最適なモデルは常に変わる

  • Claudeはリファクタリングに最適
  • GPT-5 Codexはコードレビューに最適
  • Geminiはコストパフォーマンスに優れる
  • Grokはスピードが速い
  • Qwen / Kimiは基本的な作業に十分
タスクごとに最適なモデルは違う — 自由に選べることが開発効率の鍵

⑩オープンソース・モデル中立ツールの価値

  • Kilo — エディタに残り、モデルを押し付けない
  • Kline — オープンソースのコーディングアシスタント
  • OpenCode — モデルに依存しない開発ツール
  • Aider — ターミナルベースのAIコーディングツール
特定のモデルに縛られないモデル中立のツールが、これからの時代に必要

⑪Kiloの立ち位置: 独立した中間層

  • RooはIDEを去ったが、KiloはVS Codeに残留
  • CursorはSpaceXと結びつくが、Kiloは推すモデルを持たない
  • 自分がコーディングする場所で動き、最高のモデルを選べるツールを使うべきという主張
RooともCursorとも違う第三の道 — ユーザーの自由を最優先する立場

⑫Kiloの技術再構築: ゼロから作り直した理由

  • RooCodeのVS Code拡張機能をゼロから再構築した
  • CLI、クラウドエージェントと同じコア基盤を共有する設計
  • 拡張機能とCLIとクラウドの間で、一貫した体験を提供できる構造
  • Rooの遺産を引き継ぎつつ、エディタ中心の開発体験を強化
単なる引き継ぎではなく、根本的な再設計でより強力な基盤を構築

⑬エージェントマネージャー: 複数エージェントの統合管理

  • 複数のAIエージェントを一元管理するエージェントマネージャーを搭載
  • メインエージェントがサブエージェントにタスクを委譲する階層構造
  • 並列実行により、一度に複数のファイルやモジュールを処理可能
  • エディタとターミナルを行き来しながら、セッションを維持したまま作業を継続
1人のエージェントだけでなく、チームで開発を進める感覚

⑭AIコーディング市場の3つの陣営

  • クラウドエージェント企業: リモートでの自律的なコーディングを目指す（Rooの方向性）
  • モデル研究室系エディタ: OpenAI Codex、Claude Code、Googleのコーディング製品、SpaceX/xAI + Cursor
  • モデル中立プラットフォーム: Kilo、Kline、OpenCode、Aiderなど
市場が3つの陣営に三分割 — どの陣営に属するかが開発者の選択になる

⑮開発者が取るべき安全策: 依存リスクの回避

  • 1つの閉じたスタックへの依存を避けることが最も安全
  • 自分のワークフローを移植可能な状態に保つ
  • モデルを自由に変更できるツールを優先して選ぶ
  • ベンダーロックインは、買収や終了のリスクを直接的に意味する
今日の味方が明日の敵になるかもしれない — 自己防衛が開発者に求められる

⑯Windsurf事件の詳細: 買収によるアクセス遮断

  • OpenAIがWindsurfの買収を発表
  • Anthropicが直ちにWindsurfのClaudeアクセスを切断
  • 競合他社に買収されたツールへのモデル提供を拒否する動き
  • 同じことがCursorとxAIの間でも起こる可能性を示唆
モデル提供の切り札 — 買収の発表と同時にアクセス遮断が実行された

⑰RooCodeが残した遺産: エージェント型コーディングの基礎

  • カスタムモードの概念を初めて実装 — アーキテクト、コード、デバッグの3つの役割
  • エージェントがファイルシステムにアクセスし、自律的にファイルを操作する概念
  • エージェントがターミナルを操作し、コマンドを実行する基盤を構築
  • これらの概念はKiloをはじめとする後続ツールの基盤として引き継がれた
RooCodeは消えても、その革命的概念はAIコーディングのDNAとして生き続ける

⑱今後の展望: モデル自由が生き残る道

  • AIコーディング市場は急速に統合されていく — 大企業がツールを独占する動きが加速
  • オープンソースとモデル中立のツールが、長期的に開発者の最良の選択肢に
  • 最適なモデルは常に変化する — 今日の最強が明日はそうとは限らない
  • Kiloの主張: 開発する場所で動き、最高のモデルを選べるツールこそが未来
RooもCursorも時代の象徴 — 次の時代は自由度を武器にしたツールが勝つ
