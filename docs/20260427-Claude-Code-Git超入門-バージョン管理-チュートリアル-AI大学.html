Gitとは

①Gitとは: 分散型バージョン管理システム

  • バージョン管理とは、ファイルの変更履歴を記録・管理する仕組みです。
  • 複数人で同時に開発しても、変更が衝突しません。
  • 過去のどんなバージョンにも、いつでも戻ることができます。
  • 分散型なので、ネットがなくてもローカルで作業が可能です。
Gitは世界中の開発者が使う、
最もポピュラーなバージョン管理ツールです！

②Gitの誕生: Linus Torvaldsが開発

  • Linus TorvaldsがLinuxカーネル開発のために2005年に作成しました。
  • 既存のツールに不満があったため、わずか2週間でプロトタイプを完成させました。
  • 現在は世界中のソフトウェア開発で標準的に使われています。
  • オープンソースで、誰でも無料で利用できます。
Linuxの生みの親が作ったツールだから、
信頼性と実用性は折り紙付きです！

③なぜClaude CodeにGitが重要か

  • Claude Codeはコード変更をGitコミットで自動管理します。
  • AIが生成したコードを安全にレビュー・ロールバックできます。
  • ブランチ戦略でAI生成コードを安全に実験できます。
  • 変更履歴がすべて残るので、AIの出力を追跡できます。
Gitを知らないとClaude Codeの
便利さを半分しか活かせません！

④Claude Code×Gitのワークフロー

  • 1 Claude Codeがコードを生成します。
  • 2 生成されたコードをテストします。
  • 3 問題なければコミットで確定します。
  • このサイクルを繰り返すことで、安全に開発が進みます。

Gitの基本概念

⑤Working Directory（作業ディレクトリ）

  • 自分が実際に作業しているフォルダのことです。
  • ファイルを編集・追加・削除する場所です。
  • Gitはここでの変更を「未追跡」または「変更あり」として認識します。
  • 変更をコミットするには、まずStaging Areaに追加する必要があります。
普段エディタで開いている
プロジェクトフォルダがこれです！

⑥Staging Area（ステージングエリア）

  • コミットする前の準備エリアです。
  • git addコマンドでファイルをここに追加します。
  • 「この変更をコミットに含める」という選択ができる仕組みです。
  • 変更の一部だけをステージングすることも可能です。
商品をレジ袋に詰めるような
イメージで考えてみてください！

⑦Local Repository（ローカルリポジトリ）

  • 自分のPC内に保存される変更履歴のデータベースです。
  • git commitでステージングの内容がここに保存されます。
  • インターネット接続がなくても完全にローカルで動作します。
  • すべてのコミット履歴が時系列で記録されます。
オフラインでも作業できるのが
分散型バージョン管理の強みです！

⑧Remote Repository（リモートリポジトリ）

  • GitHubなどのサーバー上にある共有リポジトリです。
  • git pushでローカルの変更をリモートにアップロードします。
  • git pullでリモートの最新状態をローカルに取得します。
  • チーム開発でコードを共有・同期するための中心的な存在です。
  • GitHub ActionsでCI/CDパイプラインを構築できます。 
# GitHubとの連携手順
$ git remote add origin https://github.com/user/repo.git
$ git push -u origin main
GitHub連携で世界中のどこからでも開発可能です！
インストールと初期設定

⑨Gitのインストール方法

  • macOS: ターミナルで git --version を実行すると、Xcode Command Line Toolsが自動インストールされます。
  • Windows: git-scm.com からGit for Windowsをダウンロードしてインストールします。
  • Linux: Ubuntu/Debianなら sudo apt install git でインストールできます。 
$ git --version
git version 2.43.0

# インストール確認完了！
インストール後はバージョン確認で
ちゃんと入ったかチェックしましょう！

⑩Gitの初期設定（最初にやること）

  • ユーザー名の設定が必須です。
  • メールアドレスの設定も必須です。
  • デフォルトブランチ名をmainに設定しましょう。 
$ git config --global user.name "あなたの名前"
$ git config --global user.email "あなたのメール"
$ git config --global init.defaultBranch main
この設定は最初の1回だけでOK！
--globalですべてのリポに反映されます。
基本コマンド

⑪リポジトリの開始: init と clone

  • git init で新しいリポジトリを初期化します。
  • git clone URL でリモートリポジトリをローカルに複製します。
  • 新規プロジェクトはinit、既存プロジェクトはcloneを使います。 
# 新しいプロジェクトを開始
$ mkdir my-project && cd my-project
$ git init

# 既存のプロジェクトを複製
$ git clone https://github.com/user/repo.git
Claude Codeは自動でgit initを
実行してくれることもあります！

⑫変更の登録: add と commit

  • git add . ですべての変更をステージングに追加します。
  • git add ファイル名 で特定のファイルだけ追加もできます。
  • git commit -m "メッセージ" でステージングの内容をコミットします。
  • コミットメッセージは変更内容が分かるように書きましょう。 
$ git add index.html style.css
$ git commit -m "ヘッダーのデザインを修正"
コミットは「セーブポイント」を
作るようなものです！

⑬状態の確認: status・log・diff

  • git status で現在の変更状態を確認します。
  • git log でコミットの履歴を一覧表示します。
  • git diff で変更された差分を詳細に確認します。
  • Claude Codeを使う際は、git diffでAIの変更内容を確認しましょう。 
$ git status        # 何が変更されたか確認
On branch main
Changes not staged for commit:
  modified: index.html

$ git log --oneline  # 履歴を簡潔に表示
a1b2c3d ヘッダー修正
e4f5g6h 初期コミット
ブランチ操作

⑭ブランチの作成と切り替え

  • git branch でブランチの一覧を表示します。
  • git branch 名前 で新しいブランチを作成します。
  • git checkout -b 名前 で作成と切り替えを同時に行います。
  • ブランチは並行世界のようなもので、mainに影響しません。 
# ブランチを作成して切り替え
$ git checkout -b feature/new-design

# 現在のブランチを確認
$ git branch
* feature/new-design
  main
新機能は必ずブランチを切ってから
開発するのがベストプラクティスです！

⑮ブランチの統合と削除

  • git merge ブランチ名 でブランチの変更を統合します。
  • git branch -d 名前 で不要なブランチを削除します。
  • マージ時にコンフリクト（衝突）が発生することがあります。
  • コンフリクトは手動で解決するか、Claude Codeに解決を依頼できます。 
# mainに切り替えて統合
$ git checkout main
$ git merge feature/new-design

# 統合済みのブランチを削除
$ git branch -d feature/new-design
統合が終わったブランチは
こまめに削除してきれいに保ちましょう！
Claude Code×Git ベストプラクティス

⑯Claude Codeが自動で行うGit操作

  • Claude Codeは自動でgit init/add/commitを実行してくれます。
  • 新機能の開発は、ブランチを切ってからClaude Codeに指示しましょう。
  • AIが生成したコードは必ずテスト後にコミットする習慣をつけましょう。
  • git diffで変更内容を確認してから承認するのが安全です。
🎯 まとめ: Gitの基本コマンド（init, add, commit, push）を使いこなして、ブランチで安全にAIと協働しましょう。コミット履歴があればいつでも戻せます！
Claude Codeを信頼しつつも、
変更内容は必ず確認しましょう！

⑰.gitignoreで管理を最適化

  • .gitignoreファイルでGit管理から除外するファイルを指定します。
  • 除外すべきもの: node_modules/、.env、ビルド成果物など。
  • AIが生成する一時ファイルも除外しておくと安心です。
  • パスワードやAPIキーなどの機密情報は絶対にコミットしないように！ 
# .gitignoreの例
node_modules/
.env
*.log
.DS_Store
dist/
.claude/
.envファイルを間違えてコミットすると
大事故につながるので要注意です！
エラー解決とGitHub連携

⑱よくあるエラーと解決法

  • merge conflict: 同じ箇所を別々に編集した場合に発生。手動で解決するか、AIに解決を依頼しましょう。
  • detached HEAD: 特定のコミットにいる状態。git checkout mainで戻せます。
  • push rejected: リモートに新しい変更がある場合。git pullしてからpushし直しましょう。 
# コンフリクト解決の流れ
$ git status           # 衝突ファイルを確認
# ファイルを手動で修正
$ git add .
$ git commit -m "コンフリクト解決"
エラーが出ても焦らない！
メッセージをよく読めば解決策が見つかります。
