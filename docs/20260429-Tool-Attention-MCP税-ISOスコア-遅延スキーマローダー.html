Tool Attentionが暴くMCP税の正体

Tool Attention Is All You Need ─ MCPサーバーのツール定義が引き起こすコンテキスト圧迫問題と、その画期的な解決策を論文から読み解く

MCP税 Tool Attention ISOスコア 二段階遅延ローダー

① 200kコンテキストでも破綻する: ツールを増やすと壊れる

フロンティアモデルであるGPT-4oやClaudeなどは、200kトークンのコンテキストを持っている。そのため「MCPサーバーのツールを全部入れても余裕だろう」と思われがちだ。しかし現実は厳しく、ツールを増やすほどモデルの性能は壊れていくという結論に至る。

直感に反する事実

コンテキストに余裕があっても、ツール定義が増えるとツール選択の精度が下がり、回答品質が悪化する。

② MCP税とは何か: ツール定義がコンテキストを食いつぶす

MCP（Model Context Protocol）でツールを接続すると、各ツールの定義（JSON Schema）をシステムプロンプトに含める必要がある。これが「MCP税」と呼ばれる問題の正体だ。ツールが増えるほど、そのスキーマ情報がコンテキストを圧迫していく。

1 MCPサーバーにツールを接続する
2 各ツールのJSON Schemaがシステムプロンプトに追加される
3 コンテキストがツール定義で埋め尽くされる

③ MCP税の実態: ツール定義だけで1.42Mトークンを消費

複数のMCPサーバーを接続した環境での実測値が衝撃的だ。ツール定義だけで1.42Mトークンを消費し、コストが通常の17倍に膨れ上がる。コンテキストの大部分がツール定義で埋まり、ユーザーの実際の質問に使える余裕がほとんど残らない。

④ 70%のラインで崩壊: コンテキストが限界を超える瞬間

コンテキストの約70%がツール定義で占められると、エージェントの性能が急激に低下する。ユーザーの実際の質問に割けるコンテキストが不足し、ツール選択の精度も大幅に下がる。この70%というラインが、MCP運用における実質的な限界点として示されている。

急激な劣化

70%を超えたあたりから性能が急カーブで落ち込む。緩やかな低下ではなく、閾値をまたいだ瞬間に崩壊が起きる。

⑤ ISOスコア: ユーザーの意図とツールの関連性を定量化

ISO（Intent-Schema Overlap）スコアは、ユーザーの意図（Intent）とツールスキーマの重なり度を定量化する指標だ。各ツールが現在のユーザー要求にどれだけ関連しているかをスコア化し、ISOスコアが高いツールだけをコンテキストに残すことで、無駄なツール定義を排除できる。

ISOスコアの役割

「このツールは今の質問に関係あるか？」を数値で判定し、関係のないツールをコンテキストから追い出すガードレールとして機能する。

⑥ 状態認識ゲーティング: エージェントの状態でツールを動的に切り替え

エージェントの現在の状態（思考中・実行中・完了など）を認識し、状態に応じて必要なツールを動的に切り替える仕組みだ。たとえば思考フェーズでは検索ツールを外し、実行フェーズではコード実行ツールを有効にする。不要なツールを積極的に排除することで、コンテキストを節約する。

思考 → 分析・計画に必要なツールのみ有効化
実行 → アクション実行用のツールに切り替え
完了 → ツール定義を最小限に圧縮

⑦ Summary Pool: 全ツールのスキーマを要約で保持

Summary Poolは、全ツールのスキーマを短い要約に圧縮して保持する仕組みだ。詳細なJSON Schema全体をコンテキストに置くのではなく、各ツールの「名前と概要」だけを軽量に保持する。詳細なスキーマは必要になったタイミングで後から遅延読み込みするため、コンテキストを大幅に節約できる。

従来

全ツールのフルスキーマを常にコンテキストに展開

→

Summary Pool

要約だけを保持し、必要なものだけ詳細を展開

⑧ Schema Promotion: 必要なツールだけを要約からフルスキーマへ昇格

Schema Promotionは、実際に使用される可能性が高いツールだけをSummary Poolからフルスキーマへ昇格させる仕組みだ。ISOスコアに基づいて「今の質問に関係のありそうなツール」を判定し、そのツールだけ詳細なスキーマを読み込む。これが二段階遅延ローダーの核となる機能だ。

昇格の判断基準

ISOスコアが閾値θを超えたツールが昇格の対象になる。スコアが低いツールは要約のまま残り、コンテキストを圧迫しない。

⑨ 二段階遅延スキーマローダーの仕組み

Summary PoolとSchema Promotionを組み合わせた二段階遅延ローダーが、MCP税の解決の中核を担う。第1段階では全ツールの要約のみを保持し、第2段階でISOスコアに基づいて必要なツールだけフルスキーマへ昇格する。これによりコンテキスト消費を最小限に抑える。

1 第1段階: Summary Poolで全ツールの要約のみを保持
2 ISOスコアで関連ツールを特定
3 第2段階: 必要なツールだけSchema Promotionで昇格

⑩ Tool Attentionアーキテクチャ: Attentionをツール選択に応用

Tool Attentionは、TransformerのAttentionメカニズムをツール選択に応用したアーキテクチャだ。「どのツールに注意を向けるべきか」を動的に決定する。論文タイトル「Tool Attention Is All You Need」は、あの有名な「Attention Is All You Need」へのオマージュとなっている。

Attentionの応用先

トークン間の関係性ではなく、ツールとユーザー意図の関係性にAttentionを向けるという発想の転換。

⑪ Transformerとの構造的類比: 同じ原理を別の対象に適用

TransformerとTool Attentionの間には明確な構造的類比がある。Transformerはトークン間のAttentionを計算し、Tool Attentionはツール間のAttentionを計算する。両者とも「関連性の高いものに注目し、低いものを無視する」という同じ原理で動作している。

両者とも「関連性の高いものに注目し、低いものを無視する」という同じ原理で動作する。対象がトークンからツールに変わっただけだ。

⑫ 実験結果: ツール定義のトークン消費を95%削減

Tool Attentionを適用した結果、ツール定義のトークン消費を95%削減することに成功した。従来は1.42Mトークンを消費していたツール定義が、二段階遅延ロードにより大幅に圧縮される。コンテキストをユーザーの実際の会話に解放できるようになる。

⑬ 実験結果: ツール選択の成功率が22ポイント向上

ツール選択の成功率が22ポイント（percentage point）向上した。特に多数のツールが存在する環境で効果が顕著に表れている。ユーザー要求の正確な理解と、それに基づくツール選択が大幅に改善されたことを意味する。

⑭ 実験結果: APIコストが86%削減

APIコストが86%削減された。これはトークン消費の減少に直接連動する成果だ。特にエンタープライズでの運用において、多数のMCPサーバーを常時接続する環境でのコスト改善効果は絶大で、実用的なMCP運用の障壁を大きく下げる結果となっている。

⑮ 失敗ケース: 敵対的パラフレーズに弱い

ユーザーが意図を回りくどく表現すると、ISOスコアが正しく機能しなくなる。敵対的なパラフレーズ（間接的な表現）に対して、ツールの関連性判定が不正確になるリスクがある。たとえば「データを永遠に消し去って」と言われた場合、削除ツールとの関連性を正しく判定できるかが課題だ。

弱点の本質

ISOスコアはユーザーの意図を直接表現に依存するため、暗喩や比喩が入ると精度が落ちる。

⑯ 限界と課題: 今後の改善ポイント

Tool Attentionにもいくつかの限界がある。ISOスコアの精度向上が引き続き必要で、1階層あたりのファイル数が多い場合の対応や、リアルタイムなスキーマ更新への対応が今後の課題として挙げられている。また、複雑なマルチステップワークフローでの検証も不足している。

今後の課題

ISOスコアの精度向上／大規模ディレクトリ対応／リアルタイムスキーマ更新／マルチステップワークフローの検証

⑰ 明日から実装できる最適化パラメータ

論文の実験で最適化されたパラメータが示されており、すぐに実装に活用できる。ゲーティング閾値θ=0.28は、ISOスコアが0.28以上のツールだけを有効化する値だ。Schema Promotionで同時に昇格させる最大ツール数はk=10に設定するのが最適とされている。

// 論文で最適化されたパラメータ
const θ = 0.28; // ゲーティング閾値
const k = 10; // Schema Promotionの最大ツール数

// ISOスコア >= θ のツールだけを有効化
if (isoScore >= θ && promoted < k) {
  promoteSchema(tool);
}

⑱ まとめ: MCP税は払わなくてよくなる

Tool AttentionはMCP税問題の画期的な解決策だ。二段階遅延ロードと状態認識ゲーティングの組み合わせにより、コンテキストを圧迫せずに多数のツールを扱えるようになる。実用的なパラメータ（θ=0.28, k=10）で即導入可能であり、MCPエコシステムのスケーラビリティを根本から改善する。
