Tool Attention Is All You Need

arXiv:2604.21816 / Sadani & Kumar (2026年4月)

MCP税 Tool Attention ISOスコア 遅延スキーマローダー

① 200kコンテキストでも破綻する理由

LLMのコンテキストウィンドウは200kトークンに拡大されたが、MCP（Model Context Protocol）でツールを接続するとあっという間に上限に達する。ツールのスキーマ定義だけで膨大なトークンが消費され、本来のタスクに使えるコンテキストが圧迫される。

根本的な矛盾

ツールをたくさん繋ぐほど便利になるはずが、逆にコンテキストが足りなくて賢く動けなくなるというジレンマ。

② MCP税が燃える: 1.42Mトークン消費の衝撃

論文が実測した「MCP税」の正体は1.42Mトークンの消費だ。ツールのスキーマを全てコンテキストに読み込むと、タスクに関係のないツール定義まで毎回送信される。結果としてコストが17倍に跳ね上がる。

③ ツール数が70%を超えると崩壊する

実験で判明した臨界点がある。利用可能なツール数が70%を超えたあたりから、エージェントの成功率が急激に低下し始める。コンテキストがノイズで溢れ、LLMが正しいツールを選択できなくなるのだ。

なぜ70%なのか

ツールのスキーマがコンテキストの大部分を占めると、LLMの「注意力」が分散してしまい、本来のユーザーの意図を正確に把握できなくなる。

④ 論文「Tool Attention Is All You Need」の全体像

SadaniとKumarが2026年4月に発表したこの論文は、MCP税を排除するためのTool Attentionアーキテクチャを提案している。タイトルはTransformerの有名論文「Attention Is All You Need」へのオマージュであり、構造にも深い類比がある。

論文情報

⑤ ISOスコアで意図を測る: Intent-Schema Overlap

論文が導入したISO（Intent-Schema Overlap）スコアは、ユーザーの意図とツールのスキーマがどれだけ重なっているかを定量化する指標だ。このスコアが高いツールだけを優先的に読み込み、低いツールは遅延させる。

ユーザーのクエリから意図ベクトルを抽出
↓
2
各ツールのスキーマから特徴ベクトルを計算
↓
3
両者の重なりをISOスコアとして算出

⑥ 状態認識ゲーティング関数の必然性

ISOスコアだけでは不十分だ。論文は状態認識ゲーティング関数を導入し、現在のタスクの進行状況に応じてツールの有効・無効を動的に切り替える。これにより「今の状態で本当に必要なツールだけ」をコンテキストに乗せる。

ゲーティングの本質

単なるフィルタリングではなく、状態遷移に連動した動的な開閉が重要。これがTransformerのアテンション重みとの構造的類比になる。

⑦ 二段階で遅延する: Summary PoolとSchema Promotion

Tool Attentionの中核を担うのが二段階遅延スキーマローダーだ。第一段階の「Summary Pool」でツールの要約だけをコンテキストに置き、第二段階の「Schema Promotion」で必要なツールの完全なスキーマだけを遅延読み込みする。

⑧ 実験結果: トークン消費を95%削減

Tool Attentionを適用した結果、トークン消費が95%削減された。全ツールのスキーマを毎回読み込む従来手法に対し、必要なツールだけを動的に選択する手法が圧倒的な効率性を示した。

⑨ 成功率が+22ポイント向上

効率だけでなく精度も向上した。タスク成功率が+22ポイント改善されている。コンテキストからノイズが排除されることで、LLMが本来のタスクに集中でき、的確なツール選択と実行が可能になった。

精度と効率の両立

通常はコスト削減と精度向上はトレードオフだが、Tool Attentionは両方を同時に達成する珍しい手法。

⑪ Transformerとの構造的類比で腹落ちさせる

論文タイトルの「Attention Is All You Need」は単なるオマージュではない。Tool AttentionはTransformerのセルフアテンションと構造的に対応している。ツールの選択が「どのトークンに注目するか」の決定と同じ仕組みで動く。

対応関係

クエリ＝ユーザーの意図、キー＝ツールのスキーマ要約、バリュー＝完全なスキーマ。QKVアテンションそのもの。

⑫ QKVアナロジー: Tool Attentionの中身

TransformerのQ（Query）、K（Key）、V（Value）をツール選択に当てはめると明確になる。Q=ユーザーの意図ベクトル、K=ツールの要約ベクトル、V=完全なスキーマ。QとKの内積がISOスコアに相当し、閾値を超えたVだけが読み込まれる。

// Tool AttentionのQKV対応 Q = ユーザーの意図ベクトル K = ツールの要約ベクトル (Summary Pool) V = 完全なスキーマ (Schema Promotion) // ISOスコア = Q・K^T (内積) if ISO(user_intent, tool_summary) > θ: load(full_schema) // Schema Promotion

⑬ 限界も直視する: 失敗ケースの分析

論文は成功だけでなく失敗ケースも正直に報告している。ISOスコアの閾値設定が難しいケースや、ユーザーの意図が曖昧な場合にゲーティングが誤動作する場面がある。特に複数ツールの組み合わせが必要な状況では、単一ツールの選択だけでは対応できない。

既知の弱点

意図が曖昧なクエリではISOスコアが分散し、正しいツールを特定できない場合がある。

⑭ 敵対的パラフレーズ脅威: セキュリティの課題

敵対的パラフレーズ攻撃のリスクも指摘されている。ユーザーのクエリを意図的に変換してISOスコアを操作し、不要なツールを強制的に読み込ませたり、必要なツールを除外したりする攻撃が理論上可能だ。

セキュリティへの示唆

ゲーティング機構は新しい攻撃面を生む可能性があり、ISOスコアのロバスト性が課題。

⑮ 明日から実装できるパラメータ: θ=0.28、k=10

論文はすぐに使える具体的なパラメータを提示している。ゲーティングの閾値θ=0.28、Summary Poolのサイズk=10が最適値として実験で導出された。これらを既存のMCP実装に組み込むだけで効果が得られる。

// 明日から使える設定 const config = { gating_threshold: 0.28, // θ: ISO閾値 summary_pool_size: 10, // k: 要約プール上限 };

⑯ MCPの未来: スケーラブルなエージェントへ

Tool Attentionが示したのは、MCPがスケールするための道筋だ。ツール数が数十から数百に増えてもコストが線形に増加しない設計により、エージェントが多様なツールを組み合わせて複雑なタスクをこなす未来が現実味を帯びる。

スケールの鍵

遅延読み込み＋動的ゲーティングの組み合わせが、ツール数に依存しないコスト設計を実現する。

⑰ 既存手法との比較: なぜTool Attentionが違うのか

従来のツール選択手法は、ツールの説明文をベクトル化してコサイン類似度で選ぶシンプルなものが多かった。Tool Attentionはそれを状態認識＋二段階遅延で大幅に進化させている。単なる類似度判定ではなく、タスクの進行状況に応じた動的な切り替えが決定的な違いだ。

全スキーマを常にロード

⑱ まとめ: MCP税をなくすための設計の必然

Tool Attentionは、Transformerの構造をツール選択に転用したシンプルで強力な設計だ。ISOスコアによる意図測定、状態認識ゲーティングによる動的切り替え、二段階遅延ローダーによるコスト抑制。これら3つの要素が組み合わさることで、95%のトークン削減、+22ppの成功率向上、86%のコスト削減を同時に達成している。
