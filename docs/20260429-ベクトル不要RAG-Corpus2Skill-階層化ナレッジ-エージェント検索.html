Corpus2Skill ベクトルDB不要 階層化ナレッジ エージェント検索

① Corpus2Skillとは: ベクトルDBを使わないRAGの新手法

MTRIの研究者らが2026年4月に提案したCorpus2Skillは、ベクトルデータベースに依存しない全く新しいRAGのアプローチだ。従来のベクトル検索の限界を打ち破り、ナレッジ全体を階層構造に整理して、LLMエージェントがその階層を辿りながら検索する仕組みを実現している。

核心アイデア

膨大な文書をピラミッド状の階層に整理し、エージェントが「どこに何があるか」を俯瞰しながら的確に探せるようにする。

② 従来RAGの根本的な問題: 「木を見て森を見ず」

従来のベクトル検索は、質問と似たベクトルを持つ文書を上位X件だけ取得する仕組みだ。つまり「答えがありそうな断片」をいくつか拾ってくるだけで、全体像を把握してから答えを探すことはしない。これが「木を見て森を見ず」の問題と呼ばれている。

何がまずいのか

取得したX件が十分な情報量かどうかを誰も判定しない。必要な文書が上位X件に入っていなければ、永遠に見つからない。

③ ベクトル検索が苦手な「網羅的検索」

「過去の〇〇に関する案件を全部検索して」といった網羅的な検索は、ベクトル検索と非常に相性が悪い。ベクトル検索は「意味が似ているもの」を探すのが得意だが、「特定カテゴリに属するものを全件取得する」という要求には応えられない。カテゴリ分けや分類の概念がそもそもないからだ。

得意なこと

意味的に類似する文書を検索する

vs

苦手なこと

カテゴリ別に全件を網羅的に取得する

④ Corpus2Skillの全体像: 2つのフェーズ

Corpus2Skillは大きく2つのフェーズに分かれる。事前準備フェーズでは、全文書を階層構造に整理してナレッジベースを構築する。検索フェーズでは、LLMエージェントがその階層を辿って必要な文書を的確に見つけ出す。つまり、検索の「勘」をあらかじめ構造化して与えるアプローチだ。

1
事前準備: 文書をクラスタリング → 階層構造を構築
2
検索時: エージェントが階層を辿って文書を特定

⑤ 事前準備①: 文書のベクトル化とクラスタリング

まず、全文書を埋め込みモデルでベクトル化する。ただし、ここでベクトルを使うのは分類のためだけで、検索時には使わない。次に、k-means法で文書をグループ分け（クラスタリング）する。意味的に似た文書が同じグループに集まる仕組みだ。

補足

ベクトル化は「事前の整理」にだけ使う。検索時には一切使わないのが従来RAGとの大きな違い。

⑥ 事前準備②: LLMによるクラスタ要約

クラスタリングされた各グループに対して、LLMが要約を作成する。要約の内容は「このグループは何についての文書か」「どんな質問に答えられるか」の2点だ。これにより、各グループの「中身の概要」が分かるようになる。

1
k-meansで文書をグループ分け
2
各グループをLLMが要約
3
「何についての文書か」「どんな質問に答えられるか」を記録

⑦ 事前準備③: 要約のクラスタリングでピラミッド構造を構築

要約自体をさらにクラスタリングして、その上の階層の要約を作る。これを繰り返すことで、ピラミッド状の階層構造が出来上がる。一番上にはナレッジ全体の概要があり、下に降りるほど具体的な文書に近づいていく。文書数が10万件でも階層が1段増えるだけで済む（O(log N)）。

ナレッジ全体の要約（トップ階層） ├── カテゴリAの要約 │ ├── サブカテゴリA-1の要約 │ │ ├── 文書ID: doc_001 │ │ └── 文書ID: doc_002 │ └── サブカテゴリA-2の要約 │ └── 文書ID: doc_003 ├── カテゴリBの要約 │ └── ... └── カテゴリCの要約 └── ...

⑧ 事前準備④: ディレクトリ構造で保管 ─ SKILL.mdとINDEX.md

構築した階層は、ディレクトリ構造としてファイルシステムに保管される。各階層のディレクトリには2種類のファイルが含まれる。SKILL.mdには「このディレクトリ以下で何ができるか」を記述し、INDEX.mdには「下位ディレクトリの概要一覧」を記述する。一番下の階層には文書IDだけが並ぶ。

knowledge_base/ ├── SKILL.md # 全体で何ができるか ├── INDEX.md # 下位カテゴリ一覧 ├── category_A/ │ ├── SKILL.md # カテゴリAの能力 │ ├── INDEX.md # サブカテゴリ一覧 │ └── sub_A1/ │ ├── SKILL.md │ └── docs.txt # doc_id一覧

⑨ 検索フェーズ: エージェントがトップから降りていく

検索時は、LLMエージェントがトップ階層の要約を見て全体を俯瞰する。そこから「この質問に関係ありそうなディレクトリ」を判断して、一段ずつ降りていく。SKILL.mdで「ここで何ができるか」を確認し、INDEX.mdで「どの下位ディレクトリに行くべきか」を判断する。

1
トップのSKILL.mdを読んで全体を俯瞰
2
INDEX.mdから関連ディレクトリを特定
3
末端の文書IDに到達するまで繰り返す

⑩ 文書の取得: get_document(doc_id)でピンポイントに取り出す

エージェントが末端の文書IDまで辿り着いたら、get_document(doc_id)という関数を呼び出して、その文書の全文を取得する。不要な文書は一切読み込まないため、無駄なコンテキスト消費を抑えられる。必要な文書だけをピンポイントで取り出す仕組みだ。

# エージェントの検索フロー skill = read_file("SKILL.md") # 全体俯瞰 index = read_file("INDEX.md") # 下位一覧 sub = navigate_to("category_A/") # 関連ディレクトリへ doc = get_document("doc_003") # 文書を取得 answer = generate(doc) # 回答生成

⑪ Corpus2Skillのキモ: 検索の「勘」を構造化して与える

この手法の本質は、膨大なナレッジを捉えるための「勘」を階層構造で与えている点にある。「このカテゴリの文書はこのあたりにまとまっている」という直感的なナビゲーションをAIが行える。人間が図書館で「棚の分類を見て本を探す」のと同じ感覚だ。

重要な洞察

ベクトル検索は「意味の近さ」だけを頼りにするが、Corpus2Skillは「どこに何があるか」の地図を提供する。検索の質が根本から変わる。

⑫ コーディングエージェントと同じ発想

Claude Codeなどのコーディングエージェントが、プロジェクトのディレクトリ構造を辿りながらgrepでファイルを探すのと全く同じ発想だ。まずディレクトリ構造を把握し、関連しそうなディレクトリに降りて、ファイルの中身を確認する。この自然な探索パターンをRAGに応用したのがCorpus2Skillだ。

コーディングエージェント

ディレクトリ構造 → grep → ファイル閲覧

Corpus2Skill

階層構造 → INDEX → 文書ID → 取得

⑬ WixQAベンチマークで既存手法を全指標で上回る

WixQAは、エンタープライズ顧客サポート向けのRAGベンチマークだ。Corpus2Skillは、このベンチマークにおいて全ての評価指標で既存手法を上回る成績を収めた。ベクトルDBを使わないという「制約」があるにもかかわらず、検索精度と回答品質の両方で優れた結果を達成している。

⑭ ベクトルDBが不要 ─ 必要なのはLLMだけ

Corpus2Skillの大きな利点の一つは、検索時にベクトルデータベースが不要なことだ。必要なのはLLMだけ。階層構造のテキストファイル（SKILL.mdとINDEX.md）を読みながらナビゲートするだけで検索が完結する。インフラのシンプルさとコスト削減にもつながる。

コスト面のメリット

ベクトルDBの運用・保守コストがゼロに。階層構造はプレーンテキストで管理できる。

⑮ O(log N)のスケーラビリティ: 文書が増えても階層が1段増えるだけ

階層の深さはO(log N)で増加する。つまり文書数が10倍になっても、階層が1段増えるだけで済む。10万件の文書があっても、エージェントは数回のステップで目的の文書に到達できる。この対数的なスケーリングが、大規模ナレッジベースでの実用性を支えている。

O(log N)
階層の深さ ─ 文書数が増えても探索ステップは最小限

⑯ PageIndexとの比較: 文書の読み方の違い

PageIndexは、単一の長い文書を「人間が目次を辿るように読む」手法だ。一方、Corpus2Skillはナレッジ全体を「ファイルシステムを辿るように探索する」手法。PageIndexが「1冊の本を読む」イメージなら、Corpus2Skillは「図書館の中から本を探す」イメージだ。

PageIndex

1冊の本を目次から辿って読む

vs

Corpus2Skill

図書館の棚を階層から辿って探す

⑰ なぜCorpus2Skillが優れているのか: 検索の質が違う

従来のベクトル検索は「質問に似ている文書」を探すだけだが、Corpus2Skillは「どこに何があるか」を理解した上で検索する。網羅的な検索にも強く、「〇〇カテゴリの案件を全部出して」という要求にも正確に応えられる。これは地図を持った検索と手探りの検索の違いだ。

格差の理由

ベクトル検索は「推測」、Corpus2Skillは「構造化された知識に基づく判断」。情報の網羅性が根本的に違う。

⑱ まとめ: RAGのパラダイムシフト

Corpus2Skillは、RAGのあり方を根本から見直す手法だ。ベクトルDBへの依存を脱却し、ナレッジを階層構造に整理することで、エージェントによる高精度な検索を実現する。WixQAでの全指標優越は、この手法の有効性を強く示唆している。

今後の展望

エージェントベースの検索が主流になる中、ナレッジの階層化はRAGだけでなくAIシステム全体の設計にも影響を与える可能性がある。
