GPT-5.5 公式プロンプトガイド

「プロセス指定」から「アウトカム指定」へ — プロンプトエンジニアリングのパラダイムシフト

📅 2026年4月30日 | 出典: OpenAI Developer Documentation

①パラダイムシフト: プロセス指定からアウトカム指定へ

GPT-5.5最大の変更点。これまでの「こうやってステップを踏んで」というプロセス指定から、「こういう結果を出して」というアウトカム指定へ移行。モデルが最適な経路を自律的に選択する。

❌ 従来（GPT-4時代） 「まずユーザーの意図を分析して、次に関連情報を検索して、最後に要約を作成して…」
✅ GPT-5.5 「ユーザーの質問に対する最適な回答を作成して。検索や分析は必要に応じて実行して。」

②デフォルトスタイル: 効率的・直接的・タスク指向

GPT-5.5はデフォルトで効率的で直接的なスタイルを持つ。無駄な前置きや冗長な説明は不要。ユーザーのタスクを最短で完遂する方向に最適化されている。

  • 無駄な前置き（「もちろんです！」「素晴らしい質問ですね！」）は自動的に省略
  • 簡潔な回答を好むユーザーには自然に短い回答が返る
  • タスク完了を最優先に行動する

💡 余分な装飾を削る指示は、もはやプロンプトに書く必要がない。

③Personality: トーン・温かさ・率直さを短く定義

ペルソナは簡潔に定義する。長い性格記述よりも、トーン・温かさ・率直さ・フォーマリティの4軸で短く指定するのが推奨。

// ✅ 推奨されるPersonality定義 tone: professional but warm warmth: high directness: medium-high formality: low

💡 長いバックストーリーは不要。モデルの判断力を信じて短く。

④Collaboration Style: 質問タイミングとプロアクティブ度合い

モデルがユーザーとどう協力すべきかを制御する。質問するタイミング、仮定の仕方、プロアクティブな提案の度合いを明確に指定する。

  • 質問するタイミング: 「要件が不明確な時だけ質問する」か「常に確認する」か
  • 仮定の仕方: 「不明点は合理的に仮定して進める」か「必ず確認する」か
  • プロアクティブ度: 改善案を自発的に提案するか、指示されるまで待つか

⑤Preamble: 知覚応答性を向上させる中間更新

長いタスクやツール呼び出しの前に、短いユーザー向け更新を送信する「Preamble」機能。モデルが「今何をしているか」を伝えることで、ユーザーの体感待ち時間を大幅に削減する。

  • 長時間の処理前に「関連情報を検索しています…」と通知
  • 複数ステップのタスクで進捗を逐次報告
  • ユーザーは「何も起きていない」不安を感じない
UX向上 新機能

⑥Goal-first: 目的地を描いてステップは任せる

「~を解決する」「~を達成する」といったゴールを先に定義するアプローチ。具体的な手順はモデルに委ねることで、より効率的なルートを発見できる。

❌ 手順指定 「CSVを読み込んで、欠損値を補完して、正規化して、モデルに学習させて…」
✅ Goal-first 「このデータセットから最も予測精度の高いモデルを構築して。前処理とアルゴリズム選択は任せる。」

⑦ALWAYS/NEVERを避ける: 判断ルールを優先

「常に~する」「絶対に~しない」という絶対ルールは、セーフティに関わるもの以外は避けるべき。代わりに状況に応じた判断ルールを指定する。

❌ 絶対ルール 「ALWAYSコードを書く前にテストを先に書いて」「NEVER外部ライブラリを使わないで」
✅ 判断ルール 「複雑なロジックの変更時はテストを先に書く。シンプルな修正なら直接コードを変更して。」

💡 絶対ルールはモデルの柔軟性を制限し、逆効果になることが多い。

⑧Stop Rules: 明示的な停止条件で過剰実行を防止

GPT-5.5の目玉機能の1つ。「どこで止めるか」を明示的に定義することで、不必要な継続実行や過剰な検索を防ぐ。

  • 回答可能ルール: 「答えられるならすぐ答える。わからなければ検索する。」
  • 証拠充足ルール: 「十分な証拠が3つ集まったら検索を止める。」
  • 段階的停止: 「要約は3段落以内。それ以上は不要。」
最重要概念

⑨text.verbosity: 回答の長さをAPIレベルで制御

APIパラメータ`text.verbosity`で回答の長さを直接制御。プロンプトで「簡潔に」と書く代わりに、APIレベルで指定できる。

// verbosity レベル "low" // 最短の回答。事実のみ。 "medium" // デフォルト。バランス重視。 "high" // 詳細な説明付き。

`low`を指定すれば「簡潔に」のプロンプトが不要になり、トークン消費も抑えられる。

⑩フォーマット: 段落がデフォルト、装飾は控えめに

GPT-5.5のデフォルト出力形式は段落（パラグラフ）テキスト。ヘッダーや箇条書き、太字などのマークダウン装飾は控えめに使用するのが自然なスタイル。

  • 段落テキストが最も読みやすいデフォルト形式
  • 箇条書きは手順や列挙が必要な時だけ使用
  • ヘッダーは長いドキュメントでのみ階層構造に利用
  • 過度な装飾（Emoji多用、太字の連続）は不自然

⑪編集のpreserves first: 既存成果物を維持して改善

既存のコードやドキュメントを編集する際、「既存のアーティファクトを維持したまま改善する」がデフォルトの動作。破壊的変更ではなく、漸進的改善を優先する。

  • 書き直しではなく、既存コードを尊重した修正
  • ユーザーの意図しない変更を避ける
  • 変更範囲は最小限に、効果は最大限に
  • 既存のテストやAPIとの互換性を保持

⑫引用と証拠: 事実ベースとクリエイティブを分ける

タスクの種類に応じて証拠の要件を変える必要がある。事実確認は引用を必須にし、クリエイティブ生成は自由度を残す。

  • 事実タスク: 出典を明示し、検証可能な情報のみ使用
  • 分析タスク: データに基づく推論を明確に区別
  • クリエイティブタスク: 出典不要、自由な発想を許容
  • 混合タスク: 「事実部分は引用、推論部分は仮説と明記」

⑬Retrieval Budget: 検索の停止ルールで効率化

検索ベースのタスクでは「十分な証拠が集まったら再検索しない」という停止ルールが重要。無駄な検索を繰り返さず、最適なタイミングで回答に移行する。

// Retrieval Budget の例 stop_rules: - 「3つの信頼できる情報源が見つかったら検索終了」 - 「回答の確信度が85%以上になったら出力」 - 「5回検索しても見つからなければ現状で回答」

⑭Phase値: commentaryとfinal_answerを明確に分離

GPT-5.5では出力のフェーズを明示的に区別できる。中間の思考プロセス（commentary）と最終的な回答（final_answer）を分けることで、ユーザー体験が向上する。

  • commentary: 「検索中です」「該当記事を3件見つけました」など進捗
  • final_answer: ユーザーに提示する最終的な回答
  • Preambleと連携して、UXをシームレスに
  • API経由でフェーズを判定し、UIに適切に表示可能

⑮プロンプト構造: 7つの要素で体系的に組む

GPT-5.5推奨のプロンプト構造は7つの要素で構成される。すべて必須ではないが、複雑なタスクほど網羅的に指定するほど精度が向上する。

1. Role // 誰として振る舞うか 2. Personality // トーン・スタイル 3. Goal // 何を達成するか 4. Success criteria // 成功の定義 5. Constraints // 制約条件 6. Output // 出力形式 7. Stop rules // 停止条件

⑯検証: テスト実行とレンダリング確認を必須に

生成された成果物の品質保証として、実行可能なものは必ずテストする。GPT-5.5はツール呼び出しで検証を自動化できる。

  • コーディング: テストスイートを実行して通過を確認
  • UI: スクリーンショットやDOM検証でレンダリング確認
  • データ処理: 入出力のサンプルで正確性を検証
  • ドキュメント: リンク切れやフォーマット違反の自動チェック

⑰実装計画: 要件トレーサビリティと障害時挙動を定義

大規模なタスクでは実装計画を構造化して、各要件がどこに対応しているかを追跡可能にする。障害発生時の挙動も事前に定義する。

  • 要件トレーサビリティ: 各要件と実装の対応を明確に
  • データフロー: 入力から出力までの流れを視覚化
  • 検証コマンド: 各ステージで実行するテストを定義
  • 障害時挙動: エラー時のフォールバック戦略を明記

⑱フロントエンド: デザインシステム整合とアクセシビリティ

フロントエンド開発では既存のデザインシステムとの整合性を最優先する。ファーストスクリーンの表示速度とアクセシビリティも重要な品質指標。

  • 既存のコンポーネントライブラリとカラーパレットを継承
  • ファーストスクリーンは1秒以内の表示を目標
  • WCAG準拠のコントラスト比とキーボードナビゲーション
  • レスポンシブ対応はモバイルファーストで設計
