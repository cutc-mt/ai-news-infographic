Claude Code vs Codex vs GLM vs Kimi

イントロダクション

① 比較の目的: 単なるベンチマークではない

この動画では、AIコーディングツールを単なるモデルの性能だけで評価しない。モデルの性能はもちろん重要だが、それ以外にも「プランニング能力」「制限」「ツール連携」「エコシステム」「使用量の監視」など、多くの要素が実際の開発体験に影響する。本音で語る4つのサブスクリプションの徹底比較。

ターゲットユーザー

② 誰のための比較か: 毎日コードを書く人

カジュアルなユーザーなら、どのツールの無料枠やトライアルでも十分。しかし、毎日コードを書く開発者にとって重要なのは「どれが十分な使用量を提供しているか」「フロントエンドとバックエンドの両方に対応できるか」「お気に入りのツールで使えるか」「コードベースが複雑になっても信頼できるか」、そして「毎月の料金に見合う価値があるか」である。

GLM Coding Plan

③ GLM Coding Plan: 柔軟性が最大の武器

GLM 5.1はフロントエンドにもバックエンドにも強く、大きなコーディングタスクでも崩れにくい。UIの判断も的確で、既存のコードを理解し、プロジェクト構造を安価なモデルより正確に把握できる。「デモ用」ではなく、実務で使えるモデルという位置づけ。

GLM Coding Plan

④ GLMの柔軟性: ツールを選ばない自由

GLM Coding Planの最大の強みは公式アプリにロックインされないこと。Kilocode、Claude Code、Cursor、OpenCode、Klineなど、プロバイダー設定をサポートする複数のコーディングエージェントで利用可能。自分の好みのワークフローに合わせてツールを選べるのが大きな魅力。

GLM Coding Plan → Kilocode / Cursor / Claude Code / Kline
GLM Coding Plan

⑤ GLMの料金: 安くはないが価値はある

GLMは料金を大幅に引き上げ、現在は月額$18 / $72 / $160の3ティア制。年額や四半期プランで割引あり。以前のような「安い選択肢」ではなくなったが、複数ツールをまたいで使える柔軟性を考えると、1つのアプリに縛られない価値は十分にある。

Claude Code

⑥ Claude Code: ターミナルコーディングの革命児

Claude Codeはターミナルベースのコーディングエージェントとして、多くの人の考え方を変えた。リポジトリ内でエージェントと対話し、ファイルの検査、コードの編集、コマンドの実行、イテレーションを自然なワークフローで実現。モデルの性能は依然としてトップクラス。

ワークフローが自然で、エージェントとの対話がスムーズ！
Claude Code

⑦ Claude Codeの強み: フロントエンドとデザイン

フロントエンド開発において、ClaudeはGPTモデルより優れたデザイン感覚を持つことが多い。UIの見た目がきれいで、視覚的な選択が洗練されており、使いやすいフロントエンドがどうあるべきかをよく理解している。バックエンドでもコードベースの読解力とバグの推論能力に優れる。

Claude Code

⑧ Claude Codeの課題: 価値と価格のバランス

$20プランは本格的な日常コーディングには使用量が少なすぎる。$100や$200プランは本格的だが、推奨しにくい価格帯。最大の問題はClaude Codeにロックインされること。プレミアム製品であることに変わりはないが、多くの人にとって正当化が難しくなっている。

Codex (OpenAI)

⑨ Codex: 単なるターミナルツールを超える存在

Codexは単一のツールではなく、複数の機能が統合されたプラットフォーム。ローカルコーディングワークフロー、ChatGPT連携、アプリ体験、クラウドタスク、コードレビュー機能、そしてOpenAIの広大なエコシステムがすべて含まれている。ターミナルツールではなく、完全なコーディングプラットフォーム。

Codex (OpenAI)

⑩ Codexの料金: フリーティアから始められる

無料枠ですぐに試せるのが魅力。$20プランで使用量が大幅に増加し、$100や$200プランでは制限がさらに拡大。OpenAIは使用量のリセット方法やサブスクリプションへの統合方法も継続的に改善している。多くの人にとって$20からのCodexは非常に合理的な選択肢。

Free → $20 → $100 / $200
Codex (OpenAI)

⑪ Codexの強み: コードベース理解とリファクタリング

Codexはコードベースの理解、リファクタリング、処理内容の説明が得意。長いコーディングセッションでも、プロジェクト全体を把握していて、毎回のプロンプトを独立したタスクとして扱わない。バックエンド開発、デバッグ、テスト、スクリプト、アーキテクチャ設計など一般的なエンジニアリングタスクでも優秀。

Codex (OpenAI)

⑫ Codexの弱点: フロントエンドのセンス

Codexの唯一の弱点はフロントエンドのデザインセンス。技術的には正しいUIを作れるが、見た目が平凡になりがち。ジェネリックなレイアウト、余白、カード、配色を使いすぎる傾向がある。ただしこの弱点は適切な指示やデザインルールを与えれば改善可能。

良い指示・スキル・デザインルール・具体例を与えれば、Codexも大幅に改善する！
Codex (OpenAI)

⑬ Codex vs Claude Code: 集中 vs 包括

すでにChatGPTをリサーチ、執筆、画像生成、音声、計画、一般的なAIタスクに使っているなら、Codexはさらに正当化しやすい。単なるコーディングエージェントではなく、コーディングを含む包括的なAIサブスクリプション。Claude Codeは「集中型」だが、Codexは「包括型」。多くの人にとって、包括型のほうが実用的。

Kimi K2.6

⑭ Kimi: 強力だが、もう一つの個性

KimiプランはKimi K2.6へのアクセスを提供。コード生成や特定の問題解決では優秀だが、一貫性が常に同じレベルではない。コーディングプランに課金する際、たまに素晴らしい結果を出すことよりも、地味なタスクでも安定していることのほうが重要。Kimiは強いが、最も安全な第一選択ではない。

Kimi K2.6

⑮ 実務で求められる「一貫性」とは

実際のコーディング作業で必要なのは、「一度だけすごい」ではなく「毎回確実に動く」こと。正しいファイルを編集する、指示に従う、簡単な修正を複雑化しない、テストを適切に書く、無関係なコードに触れない、フロントエンドもバックエンドもデバッグもリファクタリングも安定していること。Kimiは改善のスピードが速いので、今後に期待。

Kimiモデルの進化スピードは速い！今後のバージョンアップに注目。

最終結論

⑰ おすすめの2つ: Codex と GLM 5.1

最終的に推奨を2つに絞るなら：

  • Codex — ChatGPTを使い続けていて、コーディング以外も含めた完全なAIエコシステムが欲しい人向け。クラウドタスク、レビュー機能、包括的なサブスクリプションが魅力
  • GLM 5.1 — Kilocode、Cursor、Klineなど複数のツールをまたいで使いたい人向け。自分の好きなワークフローにモデルを持ち運べる柔軟性が最大の利点
結論

⑱ 結論: モデル比較からエコシステム比較へ

今回の比較は単なるモデルの性能比較ではない。フロントエンドの出来だけで言えばClaudeが勝つ場面も多い。1つのモデルを複数ツールで使うならGLMが圧倒的。最大のAIバンドルならCodex。実験的な代替モデルならKimiが興味深い。しかし、大多数の人は4つも購読しない。お金をどこに投じるか——その答えはCodexとGLM 5.1の2つに絞られる。

「完全なAIエコシステムが欲しいならCodex。柔軟性と強力なモデルが欲しいならGLM 5.1」
