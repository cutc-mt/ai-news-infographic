Archon ロードマップライブセッション

ワークフロー市場の未来をライブで構築

2026年5月5日 | Cole Medin

①Archonとは: AIコーディングのハーネスビルダー

  • オープンソースのAIコーディング用ハーネスビルダー
  • ワークフローをYAML形式で定義して、複数のコードベースで並列実行できる
  • エージェント駆動の非決定的ノードと、スクリプト駆動の決定的ノードを混在可能
  • プロバイダーをまたいだ柔軟なワークフロー構築が可能
AIコーディングを決定的で再現可能にするのが Archon の使命です！

②ワークフロー市場: コミュニティで共有する構想

  • ユーザーが構築したワークフローをコミュニティと共有できる市場を構築予定
  • 1コマンドで他の人のワークフローをインストール可能に
  • デフォルトワークフローが最初から同梱されていて、すぐに使い始められる
  • npmやClaude Codeプラグインのようなモデルを参考に設計中
archon.diy/workflows に市場ページを構築中！

③ロードマップページ: 縦型タイムライン

  • 縦型タイムライン＋GitHub風カードのレイアウトを採用
  • roadmap.shのようなノードベースの依存関係を表現
  • スティッキーガラスモーフィズムのナビゲーション付き
  • マイルストーン: セットアップ改善、市場、評価システムなど
roadmap.shのレイアウトは参考にしつつ、よりモダンなデザインに！

④セキュリティの課題: 悪意あるワークフロー対策

  • 最大のエンジニアリング課題はワークフローのセキュリティ
  • 承認後に悪意あるバージョンに更新されるサプライチェーン攻撃のリスク
  • コミットハッシュのピン留めで特定バージョンを固定する案も検討
  • 独自のセキュリティスキャナーを構築する方針
市場に出す前に全ワークフローを審査する体制を構築します

⑤ホスティング方式の議論: PR vs 自己ホスト

  • Archonリポジトリ内にPRでワークフローを追加する方式
  • ユーザーのリポジトリで自己ホストし、マーケットがリンクする方式
  • トレードオフ: セキュリティ vs 自律的なバージョン管理
  • 結論: Phase 1はArchonリポジトリ内でPR管理する方針に決定
Homebrewがformulaをhomebrew-coreで管理するのと同じメンタルモデルです

⑥149件のIssue分析: サブエージェントでトリアージ

  • Haikuモデルを使ったサブエージェントを大量に起動してIssueを分析
  • ワークフロー領域が全Issueの40%を占める最多ドメイン
  • 重複Issueや類似ドメインを特定して、優先順位をグループ化
  • 他ドメインは一桁%: UI、アダプター、プロバイダーなど
ワークフローはバグ最多かつ機能要望最多の両方を占めています

⑦主要バグ: サイレント失敗の信頼性問題

  • サイレント失敗が最大のバグ: 成功に見えて実は何も届いていない
  • ワークフローのループ制御フローに関する問題が多発
  • 複数ステップのワークフローで途中でエージェントが脱落
  • 複雑さが増すほどエージェントがステップを忘れる・スキップするリスクが上昇
エージェントは本質的に非決定的なので、ハーネスで制御する必要があります

⑧Archon vs Hermes: リンゴとオレンジの違い

  • Hermesはパーソナルエージェント（セカンドブレイン）
  • ArchonはAIコーディングのワークフロービルダー
  • 両者は組み合わせて使える: HermesがコーディングタスクをArchonワークフローに委任
  • Archonは動画生成ワークフローなど、コーディング以外の用途にも活用可能
Hermesが「やるべきこと」を決めて、Archonがどうやるかを実行します

⑨SDK要件: プログラムからのエージェント呼び出し

  • ArchonはコーディングエージェントにSDKが必要（プログラム的に呼び出すため）
  • Kiro CLIとGemini CLIはSDKがなく未対応
  • Pi コーディングエージェントはSDKがあり、最近サポートを追加
  • CLIのヘッドレスモード（-pフラグ）だけでは不十分
TypeScriptコードからプログラム的にエージェントセッションを起動する必要があります

⑩セットアップ改善: 5分で始めるArchon

  • 目標: 5分以内にArchonを使い始められるようにする
  • バイナリインストールでセットアップを簡素化
  • より良いクイックスタートガイドの作成
  • ワークフロー仕様のドキュメント化を強化
導入の摩擦を最小限にするのが最優先です

⑪高度なワークフロー制御フロー

  • マルチロード・ループボディの制御を強化
  • 条件分岐や反復処理の柔軟な制御を追加
  • ワークフロー実行の信頼性を向上させる
  • 現在のループ機能はバグ最多の領域
ループ制御はコミュニティからの要望が最も多い機能拡張です

⑫永続的プロジェクトオーケストレーター

  • コードベースごとに1つのステートフルな会話を維持
  • 実行をまたいでコンテキストを保持
  • プロジェクトを一つのエンティティとして管理
  • マルチリポジトリワークスペースの抽象化
これはArchonにとってもすごく便利な機能になるとCole氏が語っています

⑬ローカルLLMとプロバイダー抽象化

  • ローカルLLMのサポートを追加予定
  • プロバイダー抽象化で複数のAIプロバイダーを統一的に扱う
  • マルチモーダルサポートはスコープ外と判断（将来検討）
  • 本番デプロイメントの対応も将来的に予定
プロバイダーを自由に切り替えられるのがArchonの強みです

⑭ワークフロー評価システム: Eval構想

  • Archonワークフローの評価（Eval）システムを構築予定
  • オブザーバビリティで実行状況を可視化
  • ハーネススコア診断: リポジトリがAIエージェント作業にどれだけ適しているかを採点
  • ワークフロー実行の信頼性を定量的に評価
診断自体もArchonワークフローとして構築できる可能性があります

⑮コミュニティワークフローの構造

  • コミュニティワークフローフォルダにYAML + workflow.mdを格納
  • workflow.mdにメタデータ（名前、説明、タグ、バージョン）を記載
  • archon workflow installコマンドが実際に動作するように実装
  • スター数は機械管理の別ファイルで管理する方針 
# ワークフローインストール例
archon workflow install piv-loop

# コミュニティフォルダから
# ユーザーのworkflows/へコピー

⑯ワークフローのカスタマイズ方法

  • Archonスキルをコーディングエージェントに読み込ませてワークフローを構築
  • 既存のコードベースを分析してテックスタックに特化させる
  • デフォルトワークフローを参考にしつつ、独自にカスタマイズ
  • Claude Codeでも他のエディタでも利用可能
テックスタックに最適化されたワークフローが一番使いやすいです

⑰エージェント vs ハーネス: アプローチの違い

  • スキル方式: システムをエージェントに組み込む→エージェントの推論に依存
  • Archon方式: エージェントをシステムに組み込む→制御はハーネスが管理
  • コーディングエージェントのセッションをつなぎ合わせてフローを定義
  • 各セッションはよりフォーカスされて、見落としが減少
フローを定義し、必要な時だけエージェントの推論能力を借りる設計です

⑱今後の展望: 段階的な市場の進化

  • Phase 1: 全ワークフローをArchonリポジトリ内でPR管理
  • セキュリティシステムの構築後、自律的なバージョン管理を許可
  • GitHub OAuth、マルチ組織対応を将来的に追加
  • Enterprise対応や本番デプロイも長期ロードマップに
セキュリティを第一に、段階的にコントロールを緩めていく方針です
