マルチエージェントが上手く動く3つのパターン

背景・前提知識

① Cognitionが語るマルチエージェントの現在地

  • Devin開発元のCognition社が「マルチエージェントが上手く動く条件」に関する最新ブログを公開
  • 約10ヶ月前に「マルチエージェントを作るな」という有名な記事を発表していたが、今回はその対比として位置づけられている
  • Claude Codeを中心とするマルチエージェント・オーケストレーションが実は上手く回り始めている現状を報告
  • ハーネスエンジニアリングの視点から、本気で自律エージェントを設計したい人に向けた内容
「作るな」から「上手く動く条件」へ。
モデルの進化が設計思想を変えた！

② モデル性能の劇的進化と前提の変化

  • 10ヶ月前に比べてモデルの性能が爆上がりしている。Claude Sonnet 3.7などの高性能モデルが登場
  • 以前はAgenticなサーチやターミナルベンチマークが注目されていなかったが、今ではファイルサーチも軽量モデルでこなせるように
  • 複数エージェント間で情報をやり取りする精度が向上し、100回中100回に近い精度で欲しい情報を取りに行けるように進化
  • モデルの性能が上がったことで、マルチエージェント構成の前提そのものが根本的に変わった
昔は「壊れやすい」だった。
今は「使える」に変わった！

③ マルチエージェントに向かう2つの力学

  • 力学1: 人間の限界 - AIエージェントの活用が進むと、人間が計画・管理・レビューするのが追いつかなくなる。DevinがDevinを監督する構成が自然と求められる
  • 力学2: コスト最適化 - フロンティアモデル（OpusやGemini OS）は値上がり傾向。性能は取りたいがコストは抑えたいというジレンマ
  • CognitionもSWE 1.5とSonnetを組み合わせる方法を採用。フロンティア性能を安く引き出す手段としてのマルチエージェント構成
  • どちらの力学も、結果的にマルチエージェント構成に向かう方向にある
人間が追いつけない ＋ コストを抑えたい
＝ マルチエージェントが必要！

④ 前回「マルチエージェントを作るな」の原則を振り返る

  • 原則1: コンテキストの共有 - 目指すToDo、計画、必要に応じた介入などをエージェント間でちゃんと共有する。この原則は今でも有効
  • 原則2: 暗黙の判断の意識 - サブエージェントが返す結果の中には暗黙の判断が含まれており、それを意識して設計する必要がある
  • 現在でもマルチエージェントの多くはRead-Only（読み取り専用）の構成が基本。コード検索、Web検索、レビュー、視点注入など
  • これらがうまく回っているパターンであることは、現在でも前提として共有されている
コンテキストの共有は超重要。
これは絶対に忘れないで！
パターン1: コードレビューループ（実装者 × レビュアー）

⑤ 結論: 書き手は1本に絞るのが成功の鍵

  • 現在うまくいっているパターンは、書き込みが1本に絞られている構成
  • 複数のエージェントは判断や検討だけを行う役割。Claude CodeのExplorerエージェントがまさにこの構成
  • 同時にファイルを書き込むとぶつかって壊れやすくなるため、書き手を一本化するのが安全
  • この構成はDevinのチームが実際に検証し、3つの成功パターンとして共有している
書き手は1人。知性を足すのは複数。
これが基本原則！

⑥ パターン1: コードレビューループの仕組み

  • 実装者（Coder）と評価者（Reviewer）を分ける設計パターン
  • 同じClaude Opusモデルが実装したとしても、レビューさせると抜け漏れを発見できる
  • Devinは「Devinレビュー」のループを回しており、プルリクエスト時には多くのバグが解消されている
  • このパターンは絶対にやった方がいいと強く推奨。実際に有効であることが検証済み
実装 → レビュー → 修正 のループで
バグを大幅に減らせる！

⑦ コンテキストロード: フレッシュな視点の重要性

  • LLMの注意力は有限。長い文脈になるほど重要な細部に注意が届かなくなる
  • コーディング側は作業を続けて膨らんだコンテキストを抱えており、気づけない部分が出てくる
  • レビュアーには事前のコンテキストを共有しないのが重要。レビューして欲しいことだけを伝えて、鮮度の高い視点を維持する
  • コンテキストロード現象: コンテキストが長くなるほどモデルの判断力が落ちる。これは超重要な真実
コンテキストが長い ＝ 判断力が下がる
フレッシュな視点がバグを見つける！

⑧ 改善ループの設計: レビュー指摘をどう扱うか

  • コーディング側がレビューから戻ってきた指摘を適切に選択できるかが重要
  • 言われた通りにそのままやると、無駄な修正をしたりユーザーの指示と矛盾したりして無限ループに陥る危険がある
  • この設計をきれいにすれば、現状のモデルでもかなり妥当な判断が可能になる
  • 改善ループをどう設計するかはハーネスエンジニアリングの見せどころ
レビュー指摘をそのまま実行しない！
適切にフィルタリングする設計が鍵。
パターン2: スマートフレンドと大きいモデルの組み合わせ

⑨ スマートフレンド: モデルを使い分ける設計パターン

  • プライマリー（軽量モデル）を中心に使いながら、スマートフレンド（大型モデル）に助けてもらう構成
  • SWE 1.5（軽量・高速）をプライマリーとし、Claude Sonnetをアドバイザーとして組み合わせる設計
  • OpusやGemini OSのような最強クラスのモデルは日常使いには高すぎるため、安いモデルで回して要所だけ大型モデルを呼ぶ
  • フロンティア性能を安く引き出す手段として、マルチエージェント構成が需要を増している
普段は軽量モデル、困ったら大型モデル。
コストと性能のバランスが取れる！

⑩ 最大の課題: 小さいモデルが自分の限界に気づけるか

  • 通常は賢い方が仕事を振るのが自然。しかしプライマリー構成では逆方向になる
  • 小さいモデルが「ここで詰まった」「ここは相談すべきだ」と自分で気づけるかが最大の課題
  • 軽量モデル（SWE 1.5）は判断する側が賢くないため、いつ助けを求めるべきかの判断自体が難しい
  • この「限界への気づき」をどう設計に組み込むかが鍵になる
弱いモデルが「自分はここが分からない」
と気づくのって難しいんだよね…。

⑪ 限界への気づきを促す3つの解決策

  • 強制相談 - 毎回必ず1回はClaude Sonnetなどの大型モデルに相談して見落としを潰す方法
  • 判断の学習 - プロンプトの書き方を改善したり追加学習をして、迷うべき場面をモデルに覚えさせる。かなり強力なやり方
  • ルールベースの条件設定 - マージは必ず相談するなど、決定論的に処理するフックスのような仕組み
強制・学習・ルールの3本柱で
「気づき」を仕組み化する！

⑫ 賢い側へのコンテキスト共有という課題

  • 軽量モデルが相談しても、相談を受ける側もコンテキストを完璧に把握しているわけではない
  • 基本的にはコンテキストを全部丸ごとスマートモデルに渡して、何が重要かを賢い方に選ばせる方法を採用
  • 工夫1: 分からない時に推測して適当なことをさせずに、安全に切り替える仕組み
  • 工夫2: 聞かれていないことにも答えて、質問の意図を超えて回答するように工夫
相談先も完璧ではない。だからこそ
工夫が必要なんだ！
パターン3: 書き手を増やす ＋ 総括

⑬ パターン3: マネージャー＋子エージェントで書き手を増やす

  • マネージャーDevinがタスクを分解して、子Devinを生成し、進捗を調整する構成
  • これまでのパターンは1つのメインエージェントに周辺エージェントが知性を足す構造だったが、パターン3は書き手そのものを増やす試み
  • ただし、この構成では3つの落とし穴に直面することが分かった
  • 成功にはマネージャーが判断を分裂させずに、子のスコープをまとめる共通の考え方が必要
書き手を増やすのは難しい。
でも挑戦する価値はある！

⑭ 書き手を増やす際の3つの落とし穴

  • 過剰指示の罠 - マネージャーが子に細かく指示しすぎると、目的からずれた指示になり逆効果になる
  • 状態共有の不足 - 複数の子エージェント間で状態を適切に共有できているかが常に課題になる
  • 発見の共有欠如 - ある子が発見したことを他の子に共有する仕組みがデフォルトでは存在しない
過剰指示・状態共有・発見の共有
この3つに注意して設計しよう！

⑮ 総括: マルチエージェントが一番うまく動く構成

  • 書き手が1本で、知性を足す構造が一番うまく動く。クリーンなコンテキストを持つレビュアーが見落としを発見する
  • ジェネレーターとレビュアーの分割パターンが最も確実な成功パターンとして挙げられている
  • フロンティア級のOpusやSonnetが、軽量モデルの見落しの判断を補う構成が効果的
  • GPT 5.5とGeminiなどの補い合いもかなりはまる。LLMディベートのような手法も有効
書き手1本 ＋ 知性を足す構造
これが現状のベストプラクティス！

⑯ コンテキストエンジニアリング: 残る最大の課題

  • 残る問題はコンテキストのやり取り、つまり全ての通信問題。いつ相談するべきか、どう反映させるか
  • 子エージェントが見つけたことを他の子にどう反映させるかの設計がデフォルトでは存在しない
  • どういうコンテキストをどういうサブエージェントに渡していくかはコンテキストエンジニアリングの見せどころ
  • マネージャーは判断を分裂させずに、子のスコープをまとめる共通の考え方が重要
コンテキストエンジニアリングが
マルチエージェントの成否を分ける！

⑰ Claude Codeでの実践: すでに動いている仕組み

  • Claude CodeではExplorerエージェントがすでにRead-Onlyの補助エージェントとして活躍中
  • Claude CodeにAdvisor機能が登場。スラッシュコマンドでSonnetやOpusにアドバイザーを切り替え可能
  • Opusに実装させて、最後にサブエージェントでレビューさせるパターンが実用的に機能している
  • ClaudeのモデルとGPT 5.5、Gemini 3.1 Proを組み合わせるクロスモデル運用も効果的
Claude CodeのExplorerやAdvisorは
まさにこの設計パターンの実践例！
