GStack - あなたのPCに仮想開発チームを構築

Y Combinator CEO Garry Tanが公開した、Claude Codeを23人の専門チームに変えるオープンソースツールキット

2026年5月7日 | GitHub ★90,000

① GStackとは何か

  • Garry Tan（Y Combinator CEO）が公開したオープンソースツールキット
  • GitHubで瞬時に9万スターを獲得する異常なスピードで話題に
  • Claude Codeを、CEOやデザイナー、エンジニアリングマネージャーといった23種類の専門的な役割に進化させる
  • あなたのノートパソコンの中に、フルスタックのスタートアップ企業が立ち上がるようなもの
単なるプロンプト集じゃない！
製品設計からデプロイまで
一貫したワークフローをAIに実行させる仕組み

② 23人のAIエージェント組織

  • CEO：製品の方向性を決定し、戦略を練る
  • デザイナー：UI/UXの設計とレビューを担当
  • エンジニアリングマネージャー：開発の進行管理
  • QA：品質保証とテストを自動実行
  • セキュリティ責任者：脆弱性の監査を行う
  • 合計23の専門的な役割をシミュレート
各エージェントは前の
エージェントの出力を受け取って
自分の専門の仕事だけをする！

③ 厳格なスプリント構造

  • Think：問題を深く分析して思考する
  • Plan：実装計画を詳細に立てる
  • Build：コードを構築する
  • Review：コードレビューを実施
  • Test：テストを自動実行
  • Ship → Deploy：リリースとデプロイ
  • Reflect：振り返りと改善
工場の組み立てライン
のように構造化された流程で、
後戻りの少ない開発が可能に！

④ /office-hour: AIが前提を疑う

  • あなたがアイデアを出すと、AIがその前提を疑って押し返してくる
  • 例：「カレンダーアプリを作りたい」→ AIが「本当に必要なのはチーフオブスタッフ（参謀AI）では？」と提案
  • ただのイエスマンではない。プロダクトの方向性を議論して、時にはダメ出しをする
  • タイピングが超絶に早い優秀なCTOを雇ったという次元ではなく、取り締まり役をPCの中に立ち上げたような感覚
人間の上司であるあなたに対して
AIがプロダクトの方向性を
議論してくれる！

⑤ リダクションモード: スコープを削る

  • /plan-ceo-reviewにリダクションモードが搭載されている
  • LLMはユーザーを喜ばせようとして、要求以上の過剰な機能を追加しがち
  • このモードはあえて製品のスコープを削り落とせという強烈な制約を与える
  • 最短で価値を提供できるMVP（最小限のプロダクト）を見つけ出すために、AI自身が「この機能は今はいらない」と提案
余計なボタンや画面を
作らないように制御！
過剰設計を防ぐ仕組み

⑥ LOC劇場の論争

  • Garry Tanの主張：「60日間で60万行」のプロダクションコードを書いた
  • コミュニティはこれをLOC劇場（コード数の茶番）だと一蹴
  • 過去に「30万行のコードを書いた」と誇っていたアプリが、ただのブログだったと指摘された
  • 2026年にもなってコードの行数で生産性を自慢するのは、工場が「煙をたくさん排出しました」と自慢するようなもの
本当に測るべきなのは
コードの量ではなく
生み出された価値！

⑦ AIスロップ問題

  • AIはデフォルトで冗長で無駄の多いコード（AIスロップ）を生成する傾向がある
  • LLMは本質的に確率的に最もらしい次のトークンを予測しているだけ
  • エレガントで短いコードよりも、ステップバイステップで冗長な平均的なコードパターンを出力する方が予測が容易で安全
  • 1万行の冗長なコードを数秒で出力できる時代に、行数を誇ることは将来のメンテナンス負債を自慢しているようなもの
無意識に無難な道を
選んでしまうのが
AIの性質なんだ！

⑧ 自分で宿題を採点する問題

  • GStackの多くのエージェントが同じ基盤モデル（Claude）を使っている
  • AIはモデルのアーキテクチャごとに特定の論理的盲点を持っている
  • Claudeが書いたバグのあるコードを、別の「シニアエンジニア」の人格を与えられたClaudeがレビューしても、見落としのパターンが同じ
  • ユーザーの批判：「自分で自分の宿題を採点するようなもの」
役職の名前を変えて
「君はCTOだ」と言っても
中身が同じクローンなら
自分のミスには気づけない...

⑨ 革新的なQAスキル

  • 通常、AIエージェントにブラウザーテストをさせると数十秒のコールドスタート遅延が発生
  • GStackは永続的なChromiumデーモンとコンパイル済みPlaywrightをバックグラウンドで動かし、遅延を完全に排除
  • AIが即座にブラウザを開いて、ユーザーと同じようにボタンをクリックしてバグを発見
  • 操作から将来のための回帰テストのコードまで自動生成
ハッカーニュースの辛口批評家でさえ
「純粋に素晴らしい」と
賛辞を送った機能！

⑩ /design-review: AIっぽさを排除

  • AIは放っておくと汎用的で退屈なUIデザインを作り出しがち
  • 典型的な例：どこにでもあるInterフォントやありきたりな青いグラデーション
  • このスキルは平均的なデザインを検知して、主張のあるCSSを注入して洗練させるようプロンプトが組まれている
  • 「無難なデザインは排除対象」とAIに教え込んでいる
AIの「平均を好む癖」
を意図的に壊して、
個性のあるデザインに！

⑪ /codex: 敵対的監査で品質担保

  • Claudeが出力したコードに対して、ライバルのOpenAI Codexにセカンドオピニオンを求める
  • 異なるアーキテクチャと学習データを持つ別のAIモデルに弱点を見つけさせる
  • 敵対的監査（Adversarial Auditing）で、単一モデルの盲点を見事にカバー
  • AI同士を戦わせることで、より高品質なコードを実現する仕組み
Claude vs OpenAI の
AI対決で
お互いの弱点を暴き出す！

⑫ プロンプトインジェクション防御

  • 悪意のあるWebページがAIのプロンプトを乗っ取るプロンプトインジェクション攻撃の脅威
  • ブラウザ自体に22MBのローカル機械学習分類機を組み込み
  • AIが見るすべてのページと出力をリアルタイムで監視
  • 攻撃を検知したら即座にブロックするセキュリティレイヤーを構築
AIが操作するブラウザでも
安全を担保する
本格的なセキュリティ設計！

⑬ GBrain: 永続的な知識ベース

  • 23人のAIエージェントが過去のプロジェクトの文脈をどう記憶するか？その答えがGBrain
  • 完全にローカルで動くSQLiteか、クラウド上のSupabaseを選択して記憶を保存
  • 数ヶ月前に出した技術的な決定も、リポジトリごとに記憶として永続化
  • 人間のチームなら「このライブラリは半年前にセキュリティ上の理由で外したよ」と共有するのと同じ仕組み
AIの物忘れ問題を
解決する外部脳！
知識を永続化する仕組み

⑭ 信頼の三要素（Trust Triad）

  • 複数クライアントを抱えるコンサルタントがA社の機密アルゴリズムをB社で出力する危険性
  • Read-Only：過去の文脈を検索はできるが、学習して記憶を書き換えることは禁止
  • No Access：特定のAIエージェントからのアクセスを一切禁止する設定も可能
  • リポジトリごとに知識の境界線を引けるように設計されている
AIが勝手に機密情報
を持ち出さないようにする
情報漏洩防止の仕組み！

⑮ ベンダリングのセキュリティ問題

  • GStackのインストールはベンダリングという手法を採用
  • 他人が書いたランタイム、コンパイル済みのバイナリーファイル、テレメトリーのフックをソースコードに直接コピー
  • 出所不明のバイナリーを自社のリポジトリにコピーすることは、エンタープライズでは絶対に許されない
  • 「AIが作った超便利なツールです」と言って会社のシステムに他人の実行ファイルを入れると、セキュリティ部門が卒倒するレベル
中身がよくわからない
ブラックボックスを
現金な金庫にくっつけるようなもの！

⑯ 個人向けvs企業向けの使い分け

  • 個人開発者や、0から新しいアプリを作るグリーンフィールド開発には最適なスターターキット
  • すでにCI/CDパイプライン、独自のセキュリティ基準、厳格なコーディング規約を持つ既存チームには不適合
  • 既存チームのルールと23人のAIマネージャーのルールが根本から衝突してしまう
  • Redditの結論：間違ったレイヤーにある間違った抽象化
便利さとセキュリティの
トレードオフを
理解して使うことが重要！

⑰ 本当に学ぶべきこと

  • Garry Tanと全く同じツールをそのままインストールする必要はない
  • 工場全体を買う必要はない。本当に価値があるのは組み立てラインの設計図を盗むこと
  • 役割に基づいたプロンプトの設計や、エージェント同士の明確な境界と制約というパターンを学ぶ
  • 自分の組織の基準やワークフローに合わせた、自社専用のAIエージェントを構築することこそが真の価値
設計図を学んで
自分専用にアレンジするのが
正しい使い方！
