複雑なAIエージェントを捨てろ ― Ralph Loopが最強の理由

Chris Parsons (Cherrypick CTO) がAI Engineer 2026で発表した愚直なループの哲学

2026年5月7日

①複雑な構成図の罠

  • Xを見ているとマルチエージェント連携や計画グラフが毎日流れてくる
  • ダングラフでノード20個繋いで分岐10個作ると「頭良さそう」と思ってしまう
  • でも実際は何ヶ月もデバッグして毎週壊れている状態
  • 「すごい構成、本当に動いてます？」という本質的な問い
複雑な構成図を見ると
「すごい」と思っちゃうけど
本当に動いてるのかが大事！

②Ralph Loopとは何か

  • Ralph Loop＝同じプロンプトを繰り返すだけの愚直なループ
  • 提唱者はGeoffrey Huntleyさん、世間に広めたのはChris Parsonsさん
  • Chris ParsonsはCherrypickのCTOでユーザー万人以上のスタートアップを運営
  • AI Engineer 2026で「複雑なワークフローより愚直なループの方がよく動く」と発表
2026年はRalph Loop Agent
の年と呼ばれ始めている！

③名前の由来：ラルフ・ウィガム

  • 名前はザ・シンプソンズのラルフ・ウィガムというキャラクターから
  • ラルフは同じことを何度も何度も繰り返して、最終的に何かを成し遂げるキャラクター
  • ちょっと抜けてるけれど一途な性格が、エージェントの姿と重なる
  • Huntleyさんの名言：「同じプロンプト繰り返すだけで勝手に進む」
ちょっと抜けてるけど一途な
ラルフの姿がエージェントに
ぴったり当てはまる！

④Huntleyの哲学：ダサい方が効く

  • Huntleyさんの名言：「確定的な世界ではダサいやり方が効く」
  • 構成は3 Phases, 2 Prompts, 1 Loopだけ
  • 1つ目のプロンプト：コードと仕様を比べてギャップを見つけて優先順位付きTODOを出力
  • 2つ目のプロンプト：TODOから1つタスクを取って実装→テスト→コミット
計画と実行を分けて
それをループで回すだけ！
プロンプトは2個だけでOK

⑤while true で始まるループ

  • ターミナルでwhile trueと書いてその中でClaude Codeを叩く
  • たった1行のBashスクリプトで終わりというシンプルさ
  • Claude Code自体も内部でループで動いている：スキルを読む→次のステップを決める→ツールを呼ぶ→結果を見て最初に戻る
  • つまりwhile trueで囲むのはループの上にループを足すだけ 
while true; do
  claude -e "implement the next
  most important ticket"
done
プログラミング入門で
最初に習う無限ループが
そのまま最強の手法！

⑥並列エージェント6-7台の悲劇

  • Parsonsさんの実体験：プロジェクトを大量のチケットに分解して6-7個並列でエージェント起動
  • 結果は悲惨：エージェント同士が同じチケットを重複実装し始めた
  • 片方が変えたファイルをもう片方が上書きするコンフリクトの嵐
  • 本人の振り返り：「AIに最悪の労働環境を与えていた」
指示はバラバラ、調整は不在、
依存関係も完璧じゃない…
崩壊するのは当然！

⑦ウォーターフォール開発の再現

  • 事前に全部のチケットを切って、依存関係を決めて、並列で流す
  • これは90年代のウォーターフォール開発と全く同じ構造
  • AIエージェントの世界で90年代の失敗を再演しているという指摘
  • 新しい技術だと思ってやっているが、構造は古い失敗の繰り返し
新しい技術でも
設計思想が古ければ
同じ失敗を繰り返す…
ちょっと笑えない！

⑧n8nワークフローも毎週壊れた

  • Parsonsさんがn8nでニュースレター作成のワークフローを組んでいた
  • 情報を集約して要約して整形して送る流れを毎週月曜午前2時に自動実行する設定
  • しかし毎週必ず壊れていた：APIが変わっていたりJSONのスキーマがずれていたり
  • 複雑なほど壊れる箇所が増えるという根本的な問題
毎週月曜の午前2時に
壊れた通知が来るのは
一番嫌なタイミング！

⑨制約論：並列よりシリアルで回せ

  • Theory of Constraints（制約論）がここで活きる
  • 古典的ビジネス書『The Goal』（Eliyahu Goldratt）がベース
  • 教え：並列を急ぐな。まずシリアルで自分が追いつけるか試せ
  • 多くの人は1個のループが回せてないうちから6個並列にして崩壊している
1個のループが間に合わなく
なってから初めて並列を
考えればいいんだ！

⑩たった1行のプロンプトで動く

  • プロンプトは本当に1行： 
"Implement the next most
important ticket using TDD
principles. Make docs.
Ticket commit when done."
  • 日本語訳：「TODOから次に大事なチケットを取ってTDDで実装してコミットしろ」
  • これをループで回すとAIがチケットを次々に消化していく
  • 完了したらステータスをDoneに書き換えて次のチケットに進む
1回でほぼ完璧に通ることも
GPT-5.12以降や
Claude Opus 4.6以降だと多い！

⑪AIが依存関係をその場で判断

  • 「次に大事なチケット」の判断は誰がする？→AIがその場で判断
  • AIがTODO配下のファイルを全部読んで優先順位を評価
  • 依存関係を見て「これだな」と決めるので、事前の依存関係グラフは不要
  • 最新のモデルは人間が想像する以上にちゃんと判断する
これまでのエージェント設計の
前提がひっくり返る発見！
事前設計はいらない

⑫Claude Codeの/loopコマンド

  • Claude Codeに/loopコマンドが追加された
  • 例：/loop every 5min util do maxtask from...
  • 同じセッションが維持されるため文脈も引き継がれる
  • 安全装置が2つ：タスクは7日後に自動失効、1セッション最大50タスクまで
  • 間隔を省略すると動的スケジューリング：AIが1分〜1時間の間で適切に判断
「今やることないから
30分後でいいか」みたいに
AIが自分で判断する！

⑬安全装置と運用方法

  • 7日失効：週跨ぎで放置しても勝手に止まる
  • 最大50タスク：上限を超えたら人間に戻ってくる設計
  • プロセスバウンドなのでセッションを閉じたらループも停止
  • 完全自立したい場合は--dangerously-skip-permissionsとヘッドレスでサーバー上で動かす
  • 推奨は公式の/loopコマンド（安全装置付き）
暴走対策もバッチリ！
週跨ぎ放置でも
勝手に止まってくれる

⑭24時間ループ生活の実例

  • 朝のループ：毎朝6時にその日のブリーフィングを自動生成（メール未読、カレンダー予定、優先タスク）
  • ハートビートループ：15分ごとにカレンダーをチェックして次の予定が近づいたらSlackで通知
  • カンバンループ：自作カンバンアプリと連携、プロジェクトの過去の決定を読んで続きから作業
  • スタートアップ運営ループ：Ash MauryaのフレームワークをAIに与えて経営判断の叩き台に
コーディングだけじゃない！
生活の全部をループ化する
アイデアが面白い

⑮Lethal Trifecta（致死の三要素）

  • Simon Willisonが提唱したLethal Trifecta：3つの要素が揃うとデータリスクが急増
  • ①プライベートデータへのアクセス：社内リポジトリ、メールなど
  • ②信頼できないコンテンツへの露出：誰でも書けるGitHub Issue、Webページなど
  • ③外部通信能力：メール送信、API呼び出し、PR作成など
実例：GitHubのMCPサーバー経由で
悪意あるIssueがAIに読まれて
プライベートリポジトリの
中身がPRで公開される攻撃！

⑯Lethal Trifectaの防御策

  • VPS上で動かす：メインマシンから物理的に隔離
  • APIキーを分離：普段使いのキーをそのまま渡さない。万が一漏れても影響を限定
  • 細かな権限制限：メールなら「ドラフト作成はOK、送信はNG」など操作単位で制御
  • Devbox / Dockerでコンテナの中に閉じ込める
--dangerously-skip-permissions
は絶対に使わない！
便利だからって全許可はダメ

⑰人間は何をやるのか

  • Parsonsさんの判断基準：「それは不可逆で恥を書くか？」
  • YES＝人間がやる：顧客への重要メール（間違えたら取り消せないし恥を書く）
  • NO＝AIに任せる：テスト修正やリファクター（間違えても直せる）
  • 残るのは戦略思考、人と話すこと、そして自分がやりたい仕事
シンプルだけど強力な基準！
面倒な仕事がほぼ全部
AIに行くことになる

⑱明日から始める：while true だけ

  • スタートラインは1つだけ：1個の単純なwhile trueループから始める 
while true; do
  claude -e "implement the next
  most important ticket"
done
  • 最初は短いチケット5個くらいで試してみる
  • 動いたらプロンプトを少しずつ進化させる：完了条件を厳しく、リカバリ手順を追加
  • ボトルネックになってから初めて並列を考える
いきなり完璧じゃなくて
回しながら育てる！
まず体験して世界を変えよう
