Claude CodeはMarkdownよりHTML！？
読まれるドキュメントと認知負荷の本質を解説

📅 2026年5月9日 | 🎬 まさおAIじっくり解説ch

① Claude CodeとHTML出力の話題

  • ネット上で「MarkdownよりHTMLが効く」という議論が話題に
  • Claude Codeチーム内でもHTML出力について議論されている
  • AIエージェントの進化に伴い、出力形式の見直しが進んでいる
MarkdownはAIとのやり取りには最適だけど、人間が読むには限界が…

② Markdownの限界とは

  • 100行超えのMarkdownを人間が本当に読んでいるのか？
  • 文字がツラツラ並んでいるだけでは読みにくい
  • 表現力に制約がある（レイアウトや視覚要素）
  • モックやスクショベースの情報を表現しきれない
長いMarkdownは人間がスルーしてしまう問題がある…

③ 図解の力で理解度アップ

  • 読みやすさの解決策は図解を入れること
  • カード形式で情報を整理すると見やすい
  • ブログ風のレイアウトで視認性が劇的に向上
  • モデルが大幅に向上しており、HTML生成の品質も高い
図解を入れるだけで認知負荷が下がる！

④ HTMLファイルをURLで一発共有

  • CloudflareやFirebaseなどにデプロイしてURLシェア可能
  • スラッシュコマンドでHTMLを一発生成できる
  • 画像も含めて1枚のHTMLファイルにまとめられる
  • チームメンバー全員に即座に共有
スラッシュコマンド1発で生成→デプロイ→URLシェア！

⑤ AIが書いて人間が読む時代

  • 人間が編集する部分が減少している
  • AIがドキュメントを生成するなら、人間の読みやすさを優先すべき
  • 人間があまり介入しないならMarkdownの手軽さが不要に
  • 読みやすさと共有しやすさを優先 → HTMLが自然な選択
人間が読むもの → HTML
AI同士の指示 → Markdown

⑥ CLAUDE.mdはMarkdownのまま

  • 重要: CLAUDE.mdをHTMLにする話ではない
  • AI同士の指示はMarkdownのままが最適
  • 人間が読む最終成果物はHTMLで
  • 適材適所で使い分けることが前提
全部をHTMLにしよう！ではないよ。適材適所が大事！

⑦ 情報密度が高いのに読みやすい

  • HTMLは情報密度が高いのに読みやすさを実現
  • タブやカラムレイアウトで情報を整理可能
  • ウェブページのような表現力を持つ
  • 配色やスタイルで視覚的な階層を作れる
情報が詰まっているのにスッと読めるのがHTMLの強み！

⑧ インタラクションも追加可能

  • スライダーでユーザーが操作して表示を変えられる
  • エージェンティックサーチの要素も追加可能
  • どの要素が何を言っているか可視化できる
  • 圧倒的な表現力を持つキャンバスとして機能
静的な資料じゃなくて動的な資料が作れる！

⑨ 1Mコンテキストとトークン効率

  • 1MコンテキストウィンドウがあるClaude Opusでは誤差レベル
  • トークン効率はMarkdownより悪いが、人間が読むなら意味がない
  • 生成には時間がかかるが、レビューの品質が劇的に上がる
読まれないドキュメントを作るより、読まれるものに投資しよう！

⑩ HTMLが強い4つの理由

  • 情報密度が高いのに読みやすい
  • 視覚的に見やすい（タブ・レイアウト・ウェブページ風）
  • 共有しやすさ（デプロイしてURLで即シェア）
  • 双方向インタラクションが可能
この4つを兼ね備えた最強のキャンバス！

⑪ 実践の5大ユースケース

  • 仕様書をHTMLで作成
  • コードレビューもHTMLで表示
  • スライド資料の作成
  • デザインプロトタイプの作成
  • レポートの作成
スーパーパワーズなども実際にHTMLで実装されている！

⑫ 使い捨てエディターの発想

  • JSONデータをポチポチ編集するような使い捨てアプリを作れる
  • 1回きりのデータ加工用にカスタムエディターを生成
  • JSONをコピペして整形するHTMLを即作成
  • テキストエディタでやりたくない作業を自動化
簡単なアプリみたいなものをサッと作れるのが面白い！

⑬ デザインシステムは不要

  • デザインシステムを作らなくてもある程度成立する
  • AIに「HTML作って」と頼むだけではじめられる
  • 画像が入っているだけで読みやすさが向上する
  • テンプレート作成は便利だが、必須ではない
まずは「HTML作って」から始めよう！

⑭ トークンを先に投資する発想

  • 読まれないドキュメントを作るより読まれるものに投資すべき
  • プランモードでの戻りを減らすなら、初めにトークンをかける
  • 先に品質へ投資すれば結果的に効率的
  • クオリティラインを意識することが重要
読まれるドキュメントにするための先行投資！

⑮ 目標ドリフトを防ぐ本質

  • 人間がスルーすると目標ドリフト（目標のズレ）が起きる
  • 視覚情報で人間がちゃんと認知できるようにする
  • Human-in-the-loopで目標のズレを防止
  • 認知の圧をかけて正しい方向に進む
読まないと目標がズレる！視覚情報で防ごう！

⑯ 認知負荷を下げるアプローチ

  • 本質は人間の認知負荷をどう下げるか
  • テキストを読まなくても済む仕組みを作る
  • 画像や動画もHTMLに組み込める
  • 情報を視覚的に構造化して理解を助ける
認知負荷を下げることが最重要！

⑰ PRレビューでのHTML活用

  • GitHubのメッセージだけでなく、HTMLページで過程を可視化
  • HTMLを作成→デプロイ→リンクを添付で詳細に共有
  • Claude Codeが様々な情報にアクセスできるからこそ実現可能
  • レビュアーの理解度が劇的に向上する
PRの説明をHTMLページにしてリンクで共有するのが良さそう！

⑱ まとめ: 適材適所で認知負荷を下げる

  • AI同士の指示はMarkdownのままでOK
  • 人間が読む最終成果物はHTMLで出力
  • 視覚情報で認知負荷を下げて目標ドリフトを防止
  • トークンを先に投資して読まれるドキュメントを作る
  • Claude CodeのトレンドとしてHTML出力が注目される
本質は「認知負荷を下げること」！
適材適所でHTMLを活用しよう！

⑲ 1枚のHTMLに全部詰め込める

  • SVG統合で図形やアイコンも自由に描画
  • コードスニペットもプレビュー形式で表示
  • アニメーションや動画の埋め込みも可能
  • 全てが1つのファイルに完結するのが最大の強み
1枚のHTMLが最強のキャンバス！

⑳ スライド vs HTMLの使い分け

  • スライドは途中の解説コンテキストが省略されがち
  • HTMLは詳細な説明も含めて1ページに表現可能
  • 用途に応じて適材適所で選ぶのが正解
  • HTMLの方が情報量を損なわずに共有できる
解説を含めるならHTML形式がおすすめ！
