CLI — ターミナルの相棒

④ Codex CLI (TUI) — インタラクティブUI

  • インストールはnpm i -g @openai/codexだけ。Node.js 18以上が前提
  • Mac派はbrew install codexでもOK。WindowsはWSL経由で3パターンのインストール方法がある
  • 起動はターミナルでcodexと打つだけ。シンプルすぎて拍子抜けするレベル 
# 基本コマンド3つ
codex              # 起動（TUI）
codex --resume     # 前の会話を再開
codex --last       # 直近のスレッドに復帰
  • フォーク機能：Gitのブランチのように会話を分岐できる。この方向と別案の両方を同時に試したい時に便利
エイリアスの小ネタ：cdxをbashrcに登録すると
3文字で起動できるよ！1日30回叩く人なら
年間で数時間の削減になる（エンジニアあるある）

⑤ codex exec — パイプで自動化

  • codex exec "プロンプト"で対話モードに入らずにプロンプトだけ実行して結果を返す
  • シェルスクリプトに組み込めるから、自動化の強い味方になる
決定的な使い方はパイプ連携 
# テスト失敗を自動修正（最強のコンボ）
npm test 2>&1 | codex exec "失敗を修正して"

# ESLintのエラーも自動修正
eslint . | codex exec "エラーを直して"
  • --jsonフラグでJSON Lines出力に対応。GitHub Actionsにもそのまま流し込める
  • ヘッドレス環境ではcodex login --device-authでデバイス認証が可能
要注意フラグの順番！
❌ codex --json exec "..." ← ダメ
✅ codex exec --json "..." ← 正解
間違えてもエラーが出ないから気づかない…

⑥ Goal設定とフルオートモード

  • V0.1280で追加されたGoalコマンドが革命的。タスクをCodexに永続化できる
  • 「テストを全部通すまで頑張って」みたいなゴールを与えると、ターミナルを閉じても継続して動いてくれる
  • UIにステータスが表示され、--resumeで再開すると途中経過も確認できる
フルオートモード：--dangerously-allow-full-disk-accessでサンドボックスを完全解放
  • @YunKzLKさんの事例：6時間放置して実装させたら、結果はちゃんと完成していた

Appserver & SDK — 組み込みとプログラマブル

⑩ Codex Appserver — JSON-RPC 2.0

  • JSON-RPC 2.0のサーバー。Codexハーネスを自前のアプリに埋め込むためのインターフェース
  • 起動はcodex --endpoint server。標準入出力モードで動く 
# WebSocketで待ち受け
codex --endpoint server \
  --listen ws://127.0.0.1:4500
  • 状態管理は3層構造：
    ・Thread = 1つの会話セッション全体
    ・Turn = その中の一往復
    ・Item = さらに細かいメッセージや操作
  • 承認フローもこの階層で動いて、accept/reject/cancelをターン単位で送れる
種明かし：VS Code、JetBrains、Xcodeの
拡張機能も全部このAppserverで動いてる！
自分でIDE拡張を作る時も同じ仕組みに乗っかれる

⑪ Codex SDK — TypeScript & Python

  • TypeScript版：@openai/codex-sdk（Node.js 18以上）
  • Python版：実験的（3.10以上）。メインはTypeScriptで、Node.jsエコシステムと相性が良い 
// 基本は3行で動く
import { Codex } from '@openai/codex-sdk';
const codex = new Codex();
const thread = codex.startThread();
const result = await thread.run("修正して");
  • ストリーミング対応：thread.runAndStream()でチャットGPT風の段階的表示が作れる
  • 画像入力もサポート。Figmaスクショからコード生成なども可能に
PRレビューBOTが100行で作れる！
MVPなら30行。Zodで型固めて絵文字つけて
複数ファイル対応まで全部いけるよ

⑫ Zodで構造化出力 — 型安全なAI出力

  • SDKの推し機能：Zodで構造化出力ができる。型を定義しておくと、Codexが返すJSONがその型に必ず合うように制御される
  • AI出力を本番で使う時に一番怖いのは、JSONパースで落ちたりフィールド名が変わったりすること。それをZodが全部防ぐ 
// Zodで型定義
import { z } from 'zod';
const schema = z.object({
  title: z.string(),
  severity: z.enum(['low','mid','high']),
  suggestion: z.string()
});
// 出力が必ずこの型に合う！
  • マルチエージェント：スレッドAが設計、スレッドBが実装、スレッドCがレビューという分担を1スクリプトで組める
本物のSaaSバックエンドにも組み込める品質！

ハマりどころ & ベストプラクティス

⑯ ハマりどころTOP5

  • 1 OAuth コールバック問題
    リモートサーバーでコールバックURLがlocalhost固定だと届かない。回避策：codex login --device-authを使う
  • 2 Windows WSLでフルAUTOが効かない
    GitHub Issuesでも盛り上がってる。Windows勢はバージョンを最新にしてからじゃないと痛い目を見る
  • 3 Zshエイリアスがサンドボックス内で継承されない
    Codexはサブシェルで動くので、.zshrcのエイリアスが効かない。AGENTS.mdに「ls -laを使う」みたいに書いておく
  • 4 execのフラグ位置の罠
    グローバルフラグはサブコマンドの後ろに置かないと無視される。エラーも出ないのが怖い
  • 5 承認モードの選択ミス
    Read-only → 安全だけど作業が止まる
    Workspace write → プロジェクト内は自由（推奨）
    Dangerous allow → 本番では絶対NG
ここまで知っておけば初見でハマるリスクはほぼゼロ！
特に4番は1回ハマると一生忘れない系…
