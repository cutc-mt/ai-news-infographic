サブエージェントの使い方完全解説

Claude Code / Codex マルチエージェントの仕組みと実践的な活用法

2026年5月9日

1サブエージェントとは何か

  • シングルエージェントは1つの言語モデルで全ての処理を行う従来の使い方です。ChatGPTやClaudeを普通に使う場合がこれに該当します。
  • マルチエージェントは複数の言語モデルが連携してタスクを処理する仕組みです。メインエージェントが全体を管理し、サブエージェントが個別のタスクを担当します。
  • メインの言語モデルをメインエージェント、その下で動く複数の言語モデルをサブエージェントと呼びます。
普段ChatGPTやClaudeのWebアプリを使っている場合は、マルチエージェントを意識することはほとんどありません。でもClaude CodeやCodexでは結構使われています！

2Claude Codeでのサブエージェント起動例

  • 「サブエージェントを使って英語にしてください」と指示すると、メインエージェントが自動的にサブエージェントを起動して処理を行います。
  • Ctrl+Oでサブエージェントがどういう履歴で処理を行ったかを確認できます。
  • メインエージェントがサブエージェントにプロンプトを渡し、結果を受け取ってユーザーに返答する流れになっています。

3Codexでのサブエージェント起動例

  • Codexではサブエージェントに名前が付与されて起動します（例：「Plank」という名前のエージェント）。
  • バックグラウンドエージェントとして動作し、メインエージェントからプロンプトを受け取って処理を実行します。
  • 結果をメインエージェントに返し、メインエージェントがユーザーに最終的な回答を表示する流れはClaude Codeと同じです。
Claude CodeもCodexも、最初はメインエージェントで対応し、必要に応じてサブエージェントを使ってくれます。ユーザーが明示的に指示しなくても勝手に判断してくれるんです！

4マルチエージェント構造① Independent

  • Independent型は通信なしで並列に動き、最後に結果を並べるだけの構造です。
  • 例えば「Web検索で〇〇について調べて」と入力すると、4体のエージェントが並列してそれぞれ調べてくれます。
  • 最後に4体の答えを統合する仕組みで、投票や多数決に近いアプローチと言えます。
エージェントA エージェントB エージェントC エージェントD
結果を統合

5マルチエージェント構造② Centralized

  • Centralized型は中央にメインエージェントがいて、サブエージェントに指示を出す構造です。
  • メインエージェントが各サブエージェントに指示を出し、結果を受け取って統合して返答を生成します。
  • Claude Codeで使われているのがこの構造で、ユーザーの指示をメインが分解してサブに振り分けます。
これが一番よく使われている構造です。Claude Codeの基本的なマルチエージェントの仕組みがこれに当たります！

6マルチエージェント構造③ Decentralized

  • Decentralized型はエージェント同士が会話しながら処理を進める合議型の構造です。
  • エージェント同士で議論し、最終的に投票を行って結論を決めます。
  • 会議室で全員が議論して最終的な判断を決めるようなイメージの構造になっています。
エージェントA ⟷ エージェントB
↕
エージェントC ⟷ エージェントD
全員が議論 → 投票で結論を決定

7マルチエージェント構造④ Hybrid

  • Hybrid型はCentralizedとDecentralizedの混合構造です。
  • 1つのメインエージェントがいて、サブエージェント同士が通信できる仕組みになっています。
  • Claude Codeのエージェントチームズ機能がこのハイブリッド型に該当します。メインエージェントがいて、サブ同士が通信できる構造です。
多くのマルチエージェントシステムは、この4種類の構造のいずれかに近い形になっています。用途によって使い分けが重要です！

8Googleの研究論文の概要

  • 論文名：Towards a Science of Scaling Agent Systems（arxiv.org/abs/2512.08296）
  • GoogleのGemini、OpenAIのGPT、ClaudeのOpusを使って、マルチエージェント構造ごとの性能を網羅的に調査した研究です。
  • 4つのベンチマーク × 5つの構造 × 9つのモデルで180パターンの実験を実施しています。
一般的なマルチエージェントの研究は単発のQ&Aタスクで評価されることが多いですが、この論文は真にエージェント的な挙動が必要なタスクで評価している点が画期的です。

9「エージェントが多ければ良い」は神話

  • エージェントをたくさん連携させれば賢くなるという考え方は必ずしも正しくないことが実験で示されています。
  • GPTとGeminiはマルチエージェントにすることで精度が上がりましたが、Sonnet 4.5では逆に精度が下がる結果になりました。
  • モデルのグレードが最も高いものを見ると、マルチエージェントにすることが常にプラスに働くわけではないことが分かります。
「AIエージェントをたくさん連携させればもっと賢くなるはず！」と思われがちですが、実は逆効果になることもあるんです。

10並列タスクと直列タスクの使い分け

  • 並列分割可能なタスク（金融分析など）では、売上分析・コスト分析・競合比較などを独立に調べるため、マルチエージェントが有効です。
  • 直列処理が必要なタスク（ゲーム内のプランニングなど）では、順番にタスクを行う必要があるため、マルチエージェントだと精度が下がります。
  • ポイントは「タスクが分割してメリットがあるかどうか」です。これによってエージェント構成で性能が上がるか下がるかが決まります。

1145%ルールと損益分岐点

  • 180パターンの実験結果から統計モデルを構築し、エージェント数とパフォーマンスの関係を分析しました。
  • シングルエージェントで45%以上の精度が得られるタスクは、エージェントを追加するとネガティブなリターンになります。
  • 逆に45%未満の難しい問題については、マルチエージェントを検討する価値があります。
シングルエージェントで半分以上正解できるタスクなら、無理にマルチエージェントにする必要はないという目安です。最近のモデルは高性能なので、シングルで十分なケースが多いですね。

12協調コストと情報ロスの問題

  • シングルエージェントは全ての作業を1人で行うため、全体の流れが見える状態です。
  • マルチエージェントにすると伝達コストが発生し、それぞれのエージェントが何をやったかを圧縮して伝える必要があります。
  • 情報の圧縮過程で表現税（情報ロス）が発生し、精度の低下につながることがあります。
最近の言語モデルは性能がどんどん上がっているので、シングルエージェントで対応できるケースが増えています。協調コストを考慮すると、無理にマルチにしなくてもいい場面が多いです。

13AIエージェント会社運営への懐疑的見解

  • 最近SNSでAIエージェントで会社運営するという投稿がバズっています（ホーム、経理、営業などをエージェントに分担させる構想）。
  • 人間組織では分業が効率的ですが、AIは全ての能力が既にめちゃくちゃ高いので、わざわざプロンプトで分けるメリットは限定的です。
  • 論文の結果と同様に、シングルエージェントで高精度が出せる場合はエージェントを分ける必要がないという考え方です。
内容としては面白いですが、本当にマルチエージェントにする意味があるのかは疑問です。1人のエージェントで十分色々と対応できます。

14実践①：並列調査ワークフロー

  • 複数の情報源（arXiv論文、X、Hacker News、公式ブログ、YouTube）をサブエージェントに分担させて並列で最新情報を調査します。
  • 各サブエージェントが調べ終わったら、重複を確認してユーザーに伝えるべき情報をフィルタリングして統合します。
  • 1日1回の定期実行などで使うのが現実的です。トークン消費が大きいので頻繁な実行には注意が必要です。

arXiv X HN Blog
Obsidianのマークダウンファイルと組み合わせると、ニュースを非常に分かりやすくまとめてくれます！

15並列調査のプロンプト例

  • サブエージェントを使う場合は、以下のようなプロンプトを入力します。 
最新のAIニュースについて調べたいです。
ハッカーニュースやX、公式ブログ、
アーカイブの論文などを調べるための
サブエージェントをそれぞれ作成して
最新の情報を調べてください。

それぞれのサブエージェントが調べ終わったら、
重複などを確認してユーザーに伝えるべき
情報をフィルタリングして教えてください。
このプロンプトだけで、Claude Codeが自動的に4つのサブエージェントを並列起動して調査してくれます。約5分で全プラットフォームの最新ニュースをまとめてくれます。

16実践②：忖度回避の客観評価

  • 言語モデルにはSycophancy（追従）という性質があり、ユーザーの気分を害さないような返答をしがちです。
  • 自分の成果物を評価させる際、サブエージェントを使うと直接やり取りしていないため、率直なフィードバックが得られます。
  • メインエージェントに直接評価させると「優秀ですね」と好意的な評価になりがちですが、サブエージェント経由だと10点満点中8点など厳しい評価も出ます。
完璧に忖度を防ぐことは言語モデルの構造上難しいですが、サブエージェント経由にするとかなり的確なフィードバックが得られます。自分の成果物を評価させる時は基本サブエージェントを使っています。

17実践③：作成と検証ループで品質向上

  • 作成エージェントと評価エージェントを分けると、自分で書いたコードを採点する際の甘さを防げます。
  • 評価エージェントに採点基準を設定し、基準を満たすまでループを回すと自動的に品質が向上します。
  • 最大3回のループ制約を設けることで、無限ループを防ぎつつ自己改善させることができます。

② 評価

基準を満たすまで繰り返す（最大3回）
実際に3ループ回した結果、ラウンド1から9点達成までスコアが向上しました。1発で書かせたコードより、ループを回した方が品質が確実に良くなります。

18サブエージェント活用のまとめ

  • 並列で分割できるタスク（調査や分析など）ではサブエージェントが有効です。複数の情報源から独立して情報収集できます。
  • シングルエージェントで高いパフォーマンスが出る場合は、無理にマルチエージェントにするとコストも精度も悪化する可能性があります。
  • 忖度回避や検証ループなど、客観的な評価が必要な場面ではサブエージェントが特に効果を発揮します。
  • サブエージェントはトークン消費が大きいので、コストに注意しながら適切に活用することが重要です。
エージェントを増やす前に、まずはシングルエージェントで試してみる。それで足りない場合にマルチエージェントを検討するという順番がおすすめです！
