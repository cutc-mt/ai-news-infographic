Archonの基礎
①

①Archonとは: AIコーディングのハーネスビルダー

  • Archonはオープンソースのハーネスビルダーで、AIコーディングエージェントをソフトウェア開発システムに組み込むためのツール
  • 従来の「スキルやプロンプトで開発プロセスをエージェントに詰め込む」のではなく、LLMをシステムに組み込む逆転の発想
  • Claude Code、Codex、Piなど複数のプロバイダーを混在させて利用可能
ワークフローを作って反復可能で信頼性の高いAIコーディングプロセスを実現するのがArchonの役割です！
②

②生産性の進化: 1倍 → 10倍 → 100倍

  • 従来の開発 = 1倍: AIコーディングアシスタント登場以前のクラシックな開発
  • Claude Code導入 = 10倍: コーディングエージェントを使い始めた段階
  • Archon導入 = 100倍: ワークフローの並列実行で飛躍的に向上
  • 速さの秘訣は「処理速度」ではなく並列実行とFire & Forget
必ずしも「正確に100倍」というわけではないけれど、体感的にはそれぐらい圧倒的な生産性向上なんです！
③

③決定論的 vs 非決定論的: Archonの哲学

  • 従来のアプローチでは、開発ライフサイクル全体をLLMに任せるため非決定論的になりがち
  • Archonはコーディングエージェントからの決定権を可能な限り奪うことで信頼性を担保
  • 推論能力が必要な「コードの記述」だけにLLMを使い、それ以外はスクリプトで確実に実行
重要なのは「コーディングエージェントにすべて任せる」のではなく、信頼できるプロセスの中でLLMを活用することです！
④

④ワークフローの構造: ノードとYAML

  • ワークフローは1つのYAMLファイルで定義され、各ステップを「ノード」として構築
  • 条件分岐やルーティング、ループ構造に対応。バグか機能追加かで処理を分岐可能
  • 決定論的ノード（Bash/TypeScript/Pythonスクリプト）とAIノード（プロンプト実行）を混在
  • ノード間でコンテキストを引き継ぐか、新規セッションで開始するかを制御可能 
# ワークフローの例
nodes:
  - id: explore
    type: ai
    fresh_context: true
  - id: plan
    depends_on: [explore]
  - id: implement
    fresh_context: true
    depends_on: [plan]
ブラウンフィールド開発
⑤

⑤GitHubイシューを入力にした修正フロー

  • ブラウンフィールド開発の入力は常にGitHubイシューのコンテキスト。決して「バイブコーディング」ではない
  • イシューの記述に十分なコンテキストがあることを確認してからArchonワークフローを起動
  • ワークフローの出力はPull Request。すべての変更はフィーチャーブランチで管理
イシューには「問題」「提案された解決策」「ユーザーフロー」を含めておくと、エージェントがより正確に対応できます！
⑥

⑥包括的なイシュー修正ワークフロー

  • コンテキスト収集: イシューの説明を読み込み、関連コードを調査
  • リサーチ: バグなら原因調査、機能リクエストなら実装計画
  • 実装: 計画に基づいてコードを変更
  • 自己検証: 別セッションのエージェントがレビュー（バイアス防止）
  • PR作成 → 包括的PRレビュー → コメント追加で完了
検証は「自分の仕事を自分でチェック」ではなく、別のエージェントセッションで実施するのがポイントです！
⑦

⑦ワークフローのカスタマイズ: 自分だけの harness を構築

  • デフォルトのワークフローは「インスピレーション」として使い、自分用にカスタマイズするのがベストプラクティス
  • GitHub Issuesの代わりにJira、Linear、Monday.comなどに差し替え可能
  • 各コードベースの技術スタックやアーキテクチャに合わせた独自ワークフローを構築
  • コミュニティメンバーがLinearやJira向けにカスタマイズした事例も報告されている
⑧

⑧並列実行とWorktreeの分離

  • ArchonはWorktree（独立したコードベースのコピー）を自動作成して各ワークフローを分離
  • 複数のエージェントが互いに干渉せずに同時に作業可能
  • 信頼できるワークフローならFire & Forgetで、20個のイシューを同時に対応することも理論上可能
  • ワークフローの状態はSQLiteまたはPostgreSQLに保存され、クラッシュ後も再開可能
トークン消費とレートリミットだけが制約。それ以外は何個でも並列で回せます！
PIVループとグリーンフィールド開発
⑨

⑨PIVループ: Plan → Implement → Validate

  • Plan（計画）: コーディングエージェントと一緒にスコープを計画するマークダウン文書を作成
  • Implement（実装）: 計画に基づいてエージェントにコーディングを委任
  • Validate（検証）: エージェントの自己チェック + 手動のコードレビューとテスト
  • コーディングエージェントの初回パスの出力品質はかなり低い。ループでの反復改善が不可欠
PIvループはColeの最も基礎的なAIコーディング手法。すべての複雑なタスクで必ず使っています！
⑩

⑩Exploreフェーズ: エージェントとの対話的探索

  • ワークフローはまずコードベースの探索からスタート。実装方法の全体像を把握
  • 探索の後、エージェントが人間に質問を投げかける。回答を待つためにワークフローが一時停止
  • 回答を与えるとHuman-in-the-Loopゲートを通過して次フェーズへ
  • 必要に応じて何度でも探索を繰り返し可能（最大10回まで）
⑪

⑪構造化プラン文書: 実装の唯一のコンテキスト

  • プラン文書は毎回同じフォーマットで作成。レビューのしやすさと信頼性を確保
  • 内容: 要約・ミッション・成功基準・スコープ内外・コードベースコンテキスト
  • プラン作成後もリファインループで何度でも修正可能
  • 実装ノードはプラン文書だけをコンテキストとして受け取る（ Fresh Context = true ） 
# プラン文書の構造
## Summary（要約）
## Mission（ミッション）
## Success Criteria（成功基準）
## In Scope / Out of Scope
## Key Files to Create/Update
## Architecture & Task List
⑫

⑫Workflow Marketplace: コミュニティで共有する仕組み

  • Archonの新機能としてWorkflow Marketplaceを構築中
  • コミュニティメンバーが独自のワークフローを共有し、1つのコマンドでインストール可能に
  • パッケージ構成: YAML + スクリプト + スキルをバンドルして自己ホスト（npmパッケージと同じ仕組み）
  • レジストリにPRを送信して登録。バージョニングは各自のリポジトリで管理
archon workflow install <id> のコマンド1つで他人のワークフローを導入できるようになります！
セカンドブレインと高度な活用
⑬

⑬セカンドブレインとの統合: Obsidianベースの指揮システム

  • Obsidianベースのセカンドブレインをコマンドセンターとして活用
  • repositories.mdというインデックスで、すべてのプロジェクトのメタデータを管理
  • 各コードベースに対応するファイルで、Archonの設定・カスタムワークフロー・ディスパッチ履歴を記録
  • 自然言語で「イシュー600、7610、1612をArchonで処理して」と指示するだけで自動実行
KarpathyのLLM Wikiにインスパイアされたシステムで、セカンドブレインが自動的に適切なワークフローを選択してくれます！
⑭

⑭ディスパッチ履歴で学習するプロアクティブな推薦

  • セカンドブレインにディスパッチ履歴を蓄積していく仕組みを構築
  • 過去の利用パターンから、タスクに応じて最適なワークフローを自動的に推薦
  • 例：「Coleは過去20回、このコードベースのイシュー修正にこのワークフローを使っている」
  • 日次ログと振り返りシステムを組み合わせて、メモリを継続的に更新
⑮

⑮ノードごとのモデル指定でトークン効率を最適化

  • 各ノードでプロバイダーとモデルを個別に指定可能
  • リサーチにはHaiku（高速・低コスト）、プランニングにはOpus（高精度）といった使い分け
  • プログレッシブ・ディスクロージャー: スキルは必要なノードにだけ注入してコンテキストを節約
  • Coleの実例: 5時間枠の41%消費で多数のワークフローを実行。かなり効率的
実装フェーズでブラウザ自動化スキルを渡しても無駄なトークン消費になるだけ。必要な場所に必要なスキルだけを配るのがコツです！
⑯

⑯他ツールとの比較: N8N / Pydantic AI / Symfony

  • N8N: ワークフロービルダーとしては似ているが、AIコーディングには特化していない。Worktreeの分離やプロバイダー混在は不可
  • Pydantic AI Harness: エージェント構築フレームワーク。AIコーディング特化ではなく、ディープリサーチ向け
  • Symphony（OpenAI）: Codexオーケストレーション仕様。Archonで同等のワークフローを構築可能
  • Archonの強み: AIコーディングに特化し、Worktree分離・複数プロバイダー・ヒューマンインザループを統合
実践Tipsと今後の展望
⑰

⑰実践Tips: 信赖できる結果を得るための重要ポイント

  • Human-in-the-Loopは計画後と実装後の2箇所に必ず配置。エージェントの「勘違い」を防ぐ
  • コーディングエージェントの失敗の半分は「悪いコード」ではなく「意図とのズレ」
  • ループは必須: 初回パスの品質は低い。計画→レビュー→実装→検証の反復が信頼性の鍵
  • Worktreeなしフラグ（--no-worktree）でコーディング以外のワークフローも実行可能
  • リソース消費は軽い（1GB未満）が、ブラウザ自動化スキルを使う場合はChromeインスタンスに注意
