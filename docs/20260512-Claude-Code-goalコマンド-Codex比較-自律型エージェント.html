/goalコマンドの仕組み
1

① /goalコマンドとは何か

  • Claude Codeに新しく追加された/goalコマンド
  • 完了条件を設定すると、AIが自動で条件達成に向けて動作する
  • 各ステップごとにプロンプト指示が不要で、自動で次ターンに進む
  • 条件を満たすとユーザーに制御を戻す設計
「ここまでやってね」と指定するだけで
AIが勝手に進めてくれる！
2

② Haikuが採点する仕組み

  • 各ターンの後にHaiku（高速小型モデル）が条件成立をチェック
  • 条件が満たされていない場合は、自動で次のターンを開始
  • 小型モデルが判定するため、高速かつ低コストで動作
  • メインのSonnetが作業、Haikuが審査という二段構え
別のAIが採点することで
品質を担保する仕組み！
3

③ 本質は「他人の目」でブロック

  • 「勝手に自走するコマンド」ではないというのが公式の説明
  • 別のAI（Haiku）が採点して、終了をブロックする構造
  • ユーザーが設定した条件を満たすまで、AIが勝手に終了できない
  • 人間のレビューを自動化したような安心感
自走ではなく「採点されて
合格するまで終われない」
という設計思想がポイント！
Codex版/goalとの違い
4

④ Claude Code版の特徴

  • ターンごとのHaiku採点で品質を担保する即時性重視の設計
  • メモリ上で動作するため、セッション内でのみ有効
  • CLI環境で手軽に使えるのが強み
  • Haikuを使うためコストも低く抑えられる
さくっと使いたい時に
ぴったりな軽量アプローチ！
5

⑤ Codex版の特徴

  • brief.md、goals.json、ledger.jsonlに成果物を保存
  • セッションをまたいで利用可能なため、長期プロジェクト向き
  • 計画→実行→テスト→レビュー→反復のループを永続的に回す
  • 大規模で複雑なタスクの管理に適している
ファイルに保存されるから
セッションが終わっても
続きができる！
6

⑥ Claude Code vs Codex 比較まとめ

  • 即時性：Claude Codeはターン単位で高速に判定
  • 永続性：Codexはファイル保存でセッションをまたぐ
  • 軽さ：Claude Codeはメモリ上でサクッと動く
  • 規模：Codexは計画→実行→テスト→レビューの大規模ループ
用途によって使い分けるのが
ベストプラクティス！
/goalの活用例

  • 自動でテスト→修正ループを回し続ける
  • 全テストが通ったらHaikuが判定して自動で完了
  • 何度も手動でテストを走らせる必要がなくなる
テストが通るまで勝手に

  • コードの品質が自動的に改善されていく
  • Linterがクリーンになった時点で終了
コードレビュー前の
品質チェックが自動化できる！

⑨ CIパイプラインの自動整備

  • 「CIを通るようにして」と/goalに指定するだけでOK
  • 自動でパイプラインの設定を調整してCIをグリーンに
  • CIが正常に通ったら自動で完了報告
  • 複雑なパイプライン構成も自動で最適化

⑩ Subagent @mention機能

  • @mentionでサブエージェントにタスクを委任できる機能
  • メインセッションから特定のサブエージェントを直接呼び出し
  • タスクを分割して並列で処理できるようになる
  • 複雑な作業を複数のエージェントに振り分ける設計が可能
エージェント同士で
チームワークができる！

⑪ Auto-memory（/memory on）

  • /memory onで自動記憶機能を有効化
  • プロジェクト固有のルールや好みを自動で記憶していく
  • 次回セッションでも同じ設定が反映される
  • AIがプロジェクトの文脈を学習していく仕組み
毎回同じ説明をしなくて
よくなるから超捗る！

  • PowerShell上でClaude Codeがネイティブに動作
  • Linux環境の構築不要でWindowsユーザーも簡単に導入
  • より多くの開発者がClaude Codeを使えるように

Claude Codeが身近に！
