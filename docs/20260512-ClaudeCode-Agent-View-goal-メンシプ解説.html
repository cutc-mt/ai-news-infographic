/goal：終わりの定義を先に渡す
1

① /goalとは：完走するAIの仕組み

  • /goalは、完了条件を先に渡すコマンド。AIが止まってもゴールが生きている限り、自動で再開する
  • これまでの「止まる → 続き → また止まる」というストレスから解放される
  • 寝る前に投げて、朝に結果を見るという運用が現実味を帯びる
  • 走っている間はオーバーレイが表示され、ターンカウントや経過時間がリアルタイムで見える 
/goal find all flaky tests
and fix them
止まるAIから走り続けるAIへ！
これが体験としてでかい
2

② ゴールは成果物の形まで書くのがコツ

  • ゴールが曖昧だと、AIも「できた」と言いがちになる
  • 「テストが連続で通る」のように、成果物の形まで明確に書くのがポイント
  • AIに任せるのではなく、完了条件を設計するという発想の転換
曖昧なゴールは曖昧な結果を
生む。具体性が命！
3

③ /goalの制御コマンド一覧

  • /goal status ─ 今どこまで進んでいるかを確認
  • /goal pause ─ 一旦止めて呼吸できる。心理的に大きい
  • /goal resume ─ 落ち着いたら再開
  • /goal clear ─ やめたい時はスパッと外せる
ポーズ中に要件が変わったら
ゴールも一緒に更新しないと事故るよ！
安全装置：監査と暴走ガード
4

④ トークン予算を最初に設定できる

  • 放置して気づいたらトークンを使い果たしていたという恐怖がある
  • /goal max-tokens 250のように最初から予算を渡せる
  • 予算があるだけで、任せる側の心が落ち着く。地味に効く 
/goal max-tokens 250
find all flaky tests and fix them
予算を渡すだけで
安心して任せられる！
5

⑤ 完了監査（Completion Audit）

  • 完了マークの前にCompletion Auditが走る
  • 条件を満たしたかを自己検証する仕組みが入っている
  • 「終わったふり」を減らすことで、信じる根拠を提供
  • 根拠がないと、結局あとで自分が全部やり直すことになる
気持ちの話じゃなくて運用の話！
放置には根拠が必要
6

⑥ 暴走ガード（Runaway Guard）500回

  • 止まっても回し続ける仕組みだと、無限ループが怖い
  • Runaway Guardが標準装備。デフォルト上限は500回
  • 設計思想：「止まっても続く」を許しつつ、「無限に走らない」を挟む
  • アーキテクチャとして/goal実行管理レイヤーが1本できたイメージ
500回は多いように見えて
長時間タスクなら納得の数字
Agent View：複数セッションを1画面で管制
7

⑦ Agent Viewの概要と起動方法

  • claude agentsと打つだけで起動する専用画面
  • 上に接触欄、下に入力欄のシンプル構成。たったそれだけなのに効く
  • 実行中・Needs Input・Completed・Failedの状態が1画面で見える
  • 現在はリサーチプレビュー扱い。UIやショートカットは変わる可能性あり
ターミナル10枚開く時点で
もう情報負けしてる！
Agent Viewで一括管理しよう
8

⑧ 新セッションのディスパッチ

  • 下のニュークラウンドにプロンプトを打ってEnterで新セッションを即ディスパッチ
  • 1個投げたら終わりではなく、思いついた順にポンポン投げられる
  • 各セッションにラベルがつき、何をしているかがざっくり分かる
調査・リファクタ・バグ修正…
同時に複数投げて
放置するだけで運用っぽい！
9

⑨ Peek / Attach / Detachの操作術

  • Peekで覗き見しながら、ニュークラウンドも残るのが気が利いている
  • Enterか右矢印でAttach。普通の対話に切り替わる
  • 空プロンプトで左矢印を叩くとDetachしてAgent Viewに帰ってくる
  • 「?」でショートカット一覧が表示される。覚えなくていいのが一番強い
往復が早いのが
並列のストレスを消すポイント！
ピークで1呼吸置いてから入ろう
シェル直投げと合わせ技運用
10

⑩ シェルから直接バックグラウンド投げ

  • claude --bgでシェルから直接バックグラウンドにディスパッチ
  • 投げた瞬間にセッションIDが返ってくる。遠隔操作の取っ手になる
  • claude attachで取り付けて会話、claude stopで停止も可能 
# 夜に投げて朝に結果を読む
claude --bg "調査して"
claude agents   # 朝に結果確認
夜に投げて朝に読む。
これが遠隔操作の基本パターン！
11

⑪ Worktree自動隔離で衝突を防止

  • 並列で投げると一番怖いのは、同じファイルを触って衝突すること
  • ファイル編集が必要になると、そのセッションは.claude/worktrees/配下に自動で分岐
  • 同じ机で同時に書かないよう、仕組みで強制している
  • セッションごとに作業場が分岐し、最後に合流できる設計
物理的に衝突を防ぐから
安心して並列投げできる！

⑫ 合わせ技：Agent View × /goal

  • Agent Viewで並列にセッションを立てて、各セッションに/goalを載せる
  • この時点であなたは指示する人から監督する人に役割が変わる
  • 運用のコツ：一覧でNeeds Inputが出たやつだけ返す。他は放置
  • 朝に開くとCompletedが緑、Failedが赤で並ぶ。触る順番が色で決まる
これが夜に投げて朝に揃う

⑯ メンバーシップ開設のお知らせ

  • スラチャンネルのメンバーシップがすでに始動済み。準備中ではなく、もう走り始めている

  • 同じClaude Codeでも、権限・パス・環境で挙動が変わる。その違いを順番に片付ける
  • 答えだけ置かず、「どこで判断したか」まで一緒にやるのがコンセプト
