BEADSとは何か
1

① BEADSの概要: 依存グラフ型Issue Tracker

  • BEADS（bd）はSteve Yeggeが公開したCLIベースのツール
  • タスク間の依存関係をグラフ構造で管理するIssue Tracker
  • AIコーディングエージェントの「記憶喪失問題」を解決するために生まれた
  • オープンソース: github.com/gastownhall/beads
AIがセッションを開き直しても
「どこまで進んだか」を覚えている
魔法のツール！
2

② 実際に使ってみた感想

  • 以前の体感: TODOリストを上から順に消化する感覚
  • BEADS導入後: readyが空になるまでAIが次々タスクを取って終わらせる
  • AIが自律的にタスクを獲得して完了させる体感が劇的に変わった
  • bd rememberの永続メモリも面白い機能
人間が指示を出さなくても
AIが勝手に次のタスクを
見つけてやってくれるのよ！
3

③ 解決する課題: 記憶喪失問題

  • セッションをリセットすると会話のコンテキストが全て失われる
  • 「どこまで進んだか」「何をやったか」が分からなくなる
  • 同じタスクを何度もやり直すハメになる悪循環
  • max-turns制限でタイムアウトすると全てがパーになる
Claude Codeがタイムアウトしたら
それまでの作業が全部消える…
それが最大の悩みだった！
従来ツールの限界と依存グラフの革新性
4

④ 既存TODOツールの限界

  • MarkdownのTODOリスト: タスク間の依存関係を表現できない
  • GitHub Issues: エージェントがCLIから直接操作しづらい
  • 線形リスト: タスクの順序や優先順位を動的に管理できない
  • 「Aが終わらないとBに着手できない」という関係が表現できない
TODOリストは「やること」は
書けても「関係性」は
書けないんだよね！
5

⑤ 依存グラフが画期的な理由

  • タスク同士の先行・後続の関係をグラフ構造で表現
  • 「タスクAが完了しないとタスクBに着手できない」を自動的に管理
  • 前提条件を満たしたタスクを自動的に「ready」として提示
  • グラフを動的に更新し最適な実行順序を常に提示してくれる
線形リストは1番→2番だけど
依存グラフは並列で
どんどん進められる！
6

⑥ リスト型と依存グラフ型の比較

  • リスト型: 1番目→2番目→3番目と直列にしか処理できない
  • 依存グラフ型: 前提条件を満たしたタスク全てを並列に「ready」にする
  • AIエージェントにとって「次にやるべきこと」が明確になる
  • 結果としてタスクの消化スピードが大幅に向上する
AとBが終わったらCとDが
一気にreadyになるから
AIはサクサク進められる！
内部構造とプロジェクトセットアップ
7

⑦ 内部構造: Doltデータベース

  • Dolt: Gitのようにバージョン管理できるリレーショナルデータベース
  • コミット・ブランチ・マージが可能なDBとしてBEADSのバックエンドに採用
  • 依存グラフを保存し変更履歴も自動的に管理
  • 「Git for Data」というキャッチコピー通りデータのGitが実現
データベースなのに
Gitみたいにコミットや
マージができるの！すごい！
8

⑧ JSON APIでの統合

  • CLIコマンドだけでなくJSON APIも提供されている
  • 他のツールからの統合が容易で拡張性が高い
  • エージェントがAPI経由でタスク状態を確認・更新可能
  • MCPサーバーとしても機能しよりシームレスな統合ができる
CLIでもAPIでも使えるから
どんなエージェントとも
相性が良いのが魅力！
9

⑨ bd initでプロジェクト初期化

  • bd initを実行するだけでプロジェクトが初期化される
  • プロジェクトディレクトリに.beads/フォルダを自動作成
  • Doltデータベースを初期化してグラフ構造のひな形を作成
project/
├── .beads/          # BEADSデータベース
│   └── (Dolt files)
├── src/
└── ...
bd init 1発で
準備完了！超シンプル！
BEADSのコアコマンド
10

⑩ タスクの5つの状態とbd ready

  • todo: まだ着手できない（前提条件が未完了）
  • ready: 前提条件を満たして着手可能な状態
  • active: 現在作業中のタスク
  • done: 完了したタスク
  • blocked: 何らかの理由でブロックされている
bd readyで「今やれること」が
一覧で出るからAIも
迷わず次のタスクへ！
11

⑪ bd addで依存関係付きタスク追加

  • bd addコマンドでタスクを依存関係付きで追加
  • --before: このタスクの後に実行すべきタスクを指定
  • --after: このタスクの前に完了すべきタスクを指定
bd add "実装する" --before "テスト" --after "設計"
--before と --after で
タスクの関係を宣言するだけで
グラフが自動で構築される！
12

⑫ claim→closeの作業サイクル

  • bd ready で着手可能なタスクを確認
  • bd claim でタスクを獲得（activeに移行）
  • 作業を実行（コード変更など）
  • bd close でタスク完了（doneに移行）
  • 依存していたタスクが自動的にreadyに昇格！
このサイクルを回し続けるだけで
AIが自律的に全タスクを
完了させていくのよ！

⑬ Claude Codeへの統合方法

  • CLAUDE.mdにBEADSの使い方を記述するだけで統合完了
  • エージェントに自動的にready → claim → work → closeをループさせる
  • max-turnsでタイムアウトしても次のセッションでbd readyすれば続きから再開
  • コンテキストを引き継がなくてもタスク状態は永続化されている
CLAUDE.mdに書くだけで
「タイムアウトしても続きから」が
自動でできるようになるの！

永続メモリと実践ワークフロー

⑯ bd remember: 永続メモリ機能

  • bd rememberでプロジェクトの重要情報を永続的に記憶
  • 新しいセッション開始時に自動的にコンテキストとして読み込まれる
  • コーディング規約・アーキテクチャ決定・学んだことなどを記録
bd remember "React Router v6を使用"
bd remember "テストはVitestを使うこと"
AIのメモ帳みたいなもの！
でもコンテキストが膨張すると
コスト増になるから注意！
17

⑰ rememberの注意点: 貯めすぎに注意

  • rememberしすぎるとコンテキストが膨張してコスト増・精度低下
  • 定期的に整理・要約が必要なポイント
  • 重要なことだけに絞るのがコツ。何でもかんでも記録しない
  • 面白い機能だけど試す価値は十分にある
お部屋の片付けと同じ！
溜め込みすぎないように
定期的に整理しよう！

  • 永続メモリ: プロジェクトのコンテキストを保持し続ける
