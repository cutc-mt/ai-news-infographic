BEADS（bd）- AIエージェント向け依存グラフ型タスク管理

Steve Yegge氏が公開したCLIベースの永続メモリ＆タスク管理ツール
2026年5月14日
BEADS の概要と背景

① BEADSとは: AIエージェント向けのタスク管理ツール

  • BEADS（bd）は、AIコーディングエージェント向けに設計された依存グラフ型のissue trackerです。
  • CLIベースで動作し、bdコマンドを使ってタスク管理を行います。
  • Claude Code、Codex、Cursorなどのエージェントフレンドリーな設計になっています。
  • Steve Yegge氏が公開したオープンソースツールです。
「ビーズ（Beads）」という名前は、タスクがビーズのように連なって進んでいくイメージから来ています！

② 生まれた背景: AIエージェントの記憶問題

  • Claude Codeなどでセッションを開き直すと記憶を失う問題があります。
  • TODOリストを順番に消化していたが、時間が経つと「これやったっけ？」と整理ができなくなる。
  • コンテキストウィンドウの中でどんどん記憶を追加する手法も、引き継ぎ問題が発生。
  • セッションのゼロリセット問題を解決するためにBEADSが誕生しました。
セッションを開き直すたびに、AIが「何をやっていたか」を忘れてしまうのが大きな課題でした。

③ Markdown TODOとの違い: 依存関係が管理できる

  • Markdown TODOは「上から順番にやってください」という単純なリストでした。
  • BEADSでは「この条件をクリアしないと次に進めない」という依存関係が設定できます。
  • 「これはこれに関係している」「探索中に見つけた」など、多彩な依存関係を設定可能。
  • ブロックされているタスクは自動的に表示されないので、AIが迷うことがありません。

④ GitHub Issuesとの違い: ローカル＆オフライン対応

  • GitHub IssuesやLinearはクラウドサービスなので、オフラインでは使えません。
  • BEADSはオフライン操作が可能。ローカルLLMを使う場合でも動作します。
  • ブランチごとに作業記憶を独立して持つことができます。
  • マージ時には両方の履歴を綺麗に残せるので、複数ブランチでも安心です。
クラウド依存なしでローカル完結できるのが強み！
内部構造とコマンド解説

⑤ 内部構造: DoltとJSON API

  • 内部でDoltというバージョン管理付きSQLデータベースを使用しています。
  • DoltはローカルSQLファイルとして動作する特殊なデータベースです。
  • CLIからJSON APIでやり取りして、タスクデータを管理しています。
  • Doltを採用した理由はマルチエージェント・マルチブランチで同じタスクグラフを操作するためです。
Doltは「Gitのようなバージョン管理ができるSQLデータベース」という珍しい存在です！

⑥ bd init: プロジェクトの初期化

  • bd initコマンドでプロジェクトにBEADSを導入します。
  • 実行すると隠しフォルダが作成され、必要なファイルが配置されます。
  • プロジェクトごとに独立して動作します。（画像編集アプリ、SNSアプリなど別々に管理可能）
  • 同時にCLAUDE.mdやAGENTS.md、Claude hooks設定ファイルも自動生成されます。 
# プロジェクトディレクトリに移動
cd my-project
bd init

⑦ bd ready: タスク状態の確認

  • bd readyで、現在取り組めるタスクの一覧を確認できます。
  • ブロックされているタスクは表示されないので、AIエージェントが迷いません。
  • ステータスにはopen、in_progress、closedなどがあります。
  • 優先度（P1、P2など）も設定でき、JSON形式でリストを返すことも可能です。 
bd ready
# JSON形式で取得
bd ready --json

⑧ タスクのライフサイクル: claim → close

  • bd claimでタスクに取り組むことを宣言（ステータスがin_progressに変更）。
  • 作業が完了したらbd closeでタスクを終了します。
  • AIエージェントは「readyが空になるまで次々とタスクを取って終わらせる」体感で動作します。
  • 人間がTODOリストの順番を気にする必要がなく、ビーズが連なって自動的に進むイメージです。
bd ready → claim → 作業 → close → 次のready... のループが回る！
依存関係の設定と記憶機能

⑨ 依存関係の種類: blocks / relates / parent

  • blocks: 「このタスクが終わらないと次に進めない」という依存を設定。
  • relates: 「このタスクに関係している」を示す参照リンク。
  • parent: 親子関係を設定して、タスクの階層化が可能です。
  • タスクIDはハッシュベースで発行され（3文字→4文字→5文字と拡張）。
依存関係を設定しておけば、AIが「次に何をすべきか」を自動で判断できる！

⑩ Claude Codeへの組み込み方

  • まずBEADSをインストールして、プロジェクトディレクトリでbd initを実行。
  • セッションスタートフックとプレコンパクトフックの2箇所にbdコマンドが設定されます。
  • セッション開始時にbd primeが自動実行され、コンテキストが流れ込みます。
  • hooksの設定内容には「bdでタスク管理せよ」「独自のtodoやmemoryは使うな」という指示が含まれています。 
# インストール
pip install beads-ai
# 初期化
bd init

⑪ bd prime: セッション開始時のコンテキスト読み込み

  • Claude Codeのセッション開始時にbd primeが自動的に呼び出されます。
  • bdコマンドの使い方や、過去に記憶した内容がコンテキストとして流れ込みます。
  • 1件以上の永続メモリがある場合、rememberした内容も一緒に読み込まれます。
  • Hooks設定はシンプルで、許容範囲の軽量な設定にとどまっています。
bd primeのおかげで、セッションを開き直してもAIが「覚えている」状態に！

⑫ 永続メモリ: remember / forget

  • bd rememberで、AIエージェントが覚えておきたい情報を永続的に保存できます。
  • bd forgetで、不要になった記憶を削除することも可能です。
  • 注意: めちゃくちゃ貯めすぎると軽量なコンテキストとは言えなくなるので、適度にforgetを使うのが重要です。
  • 一定以上の件数になると追加ができなくなる仕様もありますが、それまででもコンテキストが重くなる可能性があります。
メモリは便利だけど、貯めすぎ注意！定期的に整理しよう！
MCP vs CLI と実践的ワークフロー

⑬ MCP vs CLI: 公式推奨はCLI

  • MCPサーバーも提供されていますが、Claude Skillsは意図的に提供されていません。
  • MCPを使うと使える機能一覧まで渡されてしまうため、コンテキストを消費してしまいます。
  • 公式はCLI推奨。Hooks経由で必要なコマンドだけを流し込む方が軽量です。
  • CLAUDE.mdに書かれている内容で十分という判断で、あえてシンプルに保たれています。
CLIの方がコンテキストの節約になるので、公式推奨に従うのがベスト！

⑭ マルチエージェント・マルチブランチ対応

  • 複数のブランチで独立してタスクを積み上げても、マージ時に両方の履歴を綺麗に残せる。
  • 複数のAIエージェントが「僕これやります」とタスクを取る分散処理モデル。
  • サーバーモードを立てれば、チームでの共有も可能な設計になっています。
  • Doltのバージョン管理機能により、複数環境からの同時アクセスにも対応しています。

⑮ Windsurf（Cursor系）にも対応

  • 調査の結果、Windsurfにも自然に導入できることが確認されています。
  • CursorやCodexなど、他のAIコーディングエージェントにも対応。
  • CLIベースなので、エージェントの種類を問わず幅広く活用できます。
  • 各エージェントの設定ファイル（AGENTS.md等）にHooks設定を追加するだけ。

⑯ サーバーモード: チーム共有も可能

  • サーバーを立てれば、複数人・複数環境から同じBEADSにアクセスできる設計です。
  • リモートフラグも用意されており、リモートサーバー接続も想定されています。
  • チーム開発で複数のエージェントや人が共通のタスクグラフを操作可能。
  • ただし、動画作者はまだサーバーモードは試していないとのことです。
チーム開発での活用が期待できる機能です！
実際のワークフローとまとめ

⑰ 実際のワークフロー: 初期化から完了まで

  • Step 1: プロジェクトディレクトリで bd init を実行して初期化。
  • Step 2: タスクを作成し、P1/P2で優先度を設定。IDが発行されます。
  • Step 3: bd readyでブロックされていないタスクを確認。
  • Step 4: bd claimでタスクに着手。作業完了後にbd close。
  • Step 5: 必要に応じてbd rememberで記憶を保存。
  • Step 6: セッション開始時にbd primeが自動実行され、コンテキストが復元される。

⑱ まとめ: シンプルで強力なAIエージェント向けツール

  • 概念は少し複雑に見えるが、1回覚えると癖になるシンプルなツールです。
  • 依存グラフでタスクの順序を自動管理。AIが「次に何をすべきか」を迷わない。
  • 永続メモリでセッションをまたいだ記憶を保持（ただし貯めすぎに注意）。
  • マルチエージェント・マルチブランチに対応し、チーム開発にも使える設計です。
  • 他のプロジェクトにもどんどん導入したくなる優れたツールと評価されています。
TODOリストを順番に消化する感覚から、「readyが空になるまでAIが次々タスクを取って終わらせる」体感に変わる！かなり良い！
