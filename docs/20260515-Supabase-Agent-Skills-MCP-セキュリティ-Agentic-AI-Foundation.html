Supabase Agent Skills

AIエージェントに正しいSupabase開発を教えるオープンソースの取り組み
2026年5月7日 | Agentic AI Foundation "The Context" Episode 37

① Supabase Agent Skillsとは

  • AIコーディングエージェントにSupabaseの正しい使い方を教えるオープンソースの指示セット
  • Claude Code Codex Cursor GitHub Copilot に対応
  • ワンコマンドでインストール可能: npx skills add supabase/agent-skills
SupabaseはGitHubで10万スターを達成！モデルはSupabaseを知っているが、正しく使えていないのが問題

② AIエージェントが犯すよくあるミス

  • 公開スキーマでRLSポリシーをスキップしてしまう
  • 存在しないCLIコマンドを幻覚（ハルシネーション）する（例: supabase db execute）
  • security_invoker = true なしでビューを作成し、RLSを黙ってバイパスする
  • ドキュメントを無視して、数ヶ月古い学習データに頼る

③ スキルがカバーする4つの領域

  • ドキュメントアクセス — 実装前に最新ドキュメントを確認する仕組み
  • セキュリティ — RLSポリシーや認証の必須チェックリスト
  • ツールワークフロー — CLIとMCPサーバーの使い分け
  • スキーマ管理 — マイグレーションと直接変更の使い分け
重要な情報はすべてSKILL.mdの約100行に集約されている！

④ ドキュメントアクセスの優先順位

  • MCP search_docs ツール — 関連スニペットを関連度順で返す
  • Markdownとして取得 — URLに .md を追加してcurlで取得
  • ネイティブWeb検索 — 上記2つが使えない場合のフォールバック
スキルは「現在のAPIがどうなっているか」ではなく「どうやって調べるか」を教える。これがポイント！

⑤ セキュリティチェックリスト（1）

  • user_metadata を認可に使わない — ユーザーが編集可能だから危険
  • 代わりに app_metadata を使用する
  • service_role キーをフロントエンドに公開しない
  • Next.jsでは NEXT_PUBLIC_ 変数がブラウザに送信されるので注意

⑥ セキュリティチェックリスト（2）

  • ビューはデフォルトでRLSをバイパスする — security_invoker = true が必須
  • UPDATEにはSELECTポリシーが必要 — ないと黙って0行を返す（エラーなし！）
  • Storageのアップサートには INSERT + SELECT + UPDATE の3つの権限が必要
  • ユーザー削除してもJWTは無効にならない — 先にセッションを取り消す必要がある

⑦ なぜセキュリティをインライン化したか

  • 当初はセキュリティガイダンスを別の参照ファイルに置いていた
  • しかし、エージェントは参照ファイルを読み飛ばすことがあった
  • そこですべての重要情報をSKILL.md本体に移動
  • スキルがロードされた瞬間にエージェントが全情報にアクセスできる仕組みに
エージェントは「怠け者」— 複数のファイルをまたぐ知識は忘れられがち

⑧ CLIツールワークフロー

  • CLIは --help でコマンドを発見できる — 自己ドキュメント化されている
  • エージェントは --help を使って推測ではなく確認すべき
  • 既知の落とし穴: supabase db query はCLI v2.79.0以降が必要
  • コマンドを推測せず、必ずヘルプで確認することが原則

⑨ MCPサーバーワークフロー

  • MCPはエージェント専用に設計されたインターフェース
  • サービスを構造化されたツールとして公開し、ターミナルアクセス不要
  • ただし、MCPサーバーは人間が事前に設定する必要がある
  • スキルには接続問題のトラブルシューティングガイドも含まれている
CLI（人間向け）とMCP（エージェント向け）の両方に対応！

⑩ スキーマ管理のアプローチ

  • 開発中は execute_sql や supabase db query で直接スキーマを変更
  • すべてのDDL操作でマイグレーションを作成すると反復速度が落ちる
  • スキーマが安定したら Database Advisors でセキュリティと性能をチェック
  • チェック通過後にマイグレーションとしてコミットして正式化

⑪ 評価結果：モデル別のスコア

  • Claude Code Opus 4.6 — ベースライン58% → MCP+Skillで67%
  • Claude Code Sonnet 4.6 — ベースライン46% → MCP+Skillで71%
  • Codex GPT-5.4 — ベースライン71% → MCP+Skillで88%
  • Codex GPT-5.4 Mini — ベースライン42% → MCP+Skillで71%
すべてのモデルで一貫した改善を確認！特にCodex GPT-5.4は+17ポイント向上

⑫ 評価の3つのテスト条件

  • ベースライン — ツールもスキルもなし（事前学習知識のみ）
  • MCPのみ — MCPツールあり、スキルなし
  • MCP + Skill — MCPツールとスキルの両方あり
  • 6つのSupabaseシナリオで評価 — LLMジャッジがトランスクリプト全体を審査

⑬ 学び（1）：MCPだけでは不十分

  • MCPツールだけではワークフローやガイドラインがないため、エージェントがツールの組み合わせを推測するしかない
  • MCP onlyの条件では、エージェントはsearch_docsを一度も呼び出さなかった
  • ドキュメント確認ツールがあっても、使い方がわからないと意味がない
ツールを提供するだけではダメ — 使い方を教える必要がある！

⑭ 学び（2）：エージェントは学習データに頼る

  • search_docs が利用可能でも、MCP onlyのエージェントは一度も呼び出さなかった
  • エージェントは学習データのスナップショットで作業する傾向がある
  • スキルは「まず現在のドキュメントを確認する」ようエージェントを誘導

⑮ 学び（3）：ボトルネックは能力ではなくコンテキスト

  • スキルがあればすべてのモデルが security_invoker を正しく適用できた
  • 実装方法は知っていた — 知らなかったのは「いつ適用すべきか」
  • つまりコンテキストが提供する判断が、技術的な能力の壁を超えている
モデルは「やり方」は知っているが、「やるべきタイミング」を知らない！

⑯ スキル開発のベストプラクティス

  • ドキュメントを複製しない — 代わりに「見つけ方」を教える
  • SKILL.mdに必須情報を集約 — エージェントは参照ファイルを読み飛ばす
  • 意見を持つ — 学習データにはない判断基準をエンコードする
  • 複数のエージェントでテスト — あるモデルに効果的でも別のモデルには違うかも
  • シンプルに始める — v0.1.0から評価結果とフィードバックで拡張

⑰ インストール方法

  • すべてのスキルをインストール: 
npx skills add supabase/agent-skills
  • 特定のスキルのみ: 
npx skills add supabase/agent-skills --skill supabase
npx skills add supabase/agent-skills --skill supabase-postgres-best-practices
  • Claude Codeプラグインとして: 
claude plugin marketplace add supabase/agent-skills
claude plugin install supabase@supabase-agent-skills
