②事前インデックス/RAGの限界

  • 従来のRAGは全コードを事前にベクトル化してインデックスを作成する手法。
  • コードベースが大きくなるとインデックスの更新コストが膨大になる。
  • 検索精度も限界があり、関連性の低い結果が混ざる問題がある。
  • コード変更のたびに再インデックスが必要で、リアルタイム性に欠ける。
あらかじめ全部インデックス
してるけど、検索結果が
イマイチ…という経験あるかも。

③agentic searchとは

  • 事前インデックスが不要な新しいコード探索アプローチ。
  • AIエージェントが必要な情報を逐次探索しながらタスクを進める仕組み。
  • 人間の開発者と同じようにファイルを開いて読んで判断することを繰り返す。
  • インデックス構築・保守が不要なので、どのコードベースでも即座に使える。
まるで人間がコードを
読んで理解していくのと
同じアプローチ！

④agentic searchの仕組み

  • Step1: コードの探索（grep、ファイル一覧など）
  • Step2: 見つけた情報から次に何を読むべきか判断
  • Step3: 関連ファイルを実際に読み込む
  • Step4: 十分な情報が集まるまで探索ループを繰り返す
探索→判断→読込→探索…
このループがAIに
「賢い探索」を可能にする！

⑤弱点① 探索の手がかり不足

  • 検索するためのキーワードや手がかりがないと、関連ファイルを見つけにくい。
  • 間接的に関係するファイルは探索ループから漏れる可能性がある。
  • 対策: CLAUDE.mdでプロジェクト構造や重要ファイルを事前に教えておくと改善。
「このファイル見て」って
教えてあげないと見つけられない
こともあるんだね。

⑥弱点② ファイル名の乖離

  • 実装内容とファイル名が一致していない場合、探索が困難になる。
  • 例: `utils.ts`に重要なビジネスロジックが入っている場合など。
  • 対策: 意味のあるファイル名にリネームするか、CLAUDE.mdで構成を明示。
`misc.ts`に全て詰め込んでる
プロジェクトあるある…
ファイル名は大事！

⑦ハーネスがモデルと同じくらい効く

  • Anthropicの検証: モデルの性能以上にハーネス設計が成果を左右する。
  • 同じClaudeモデルでも、ハーネスの有無で出力品質が劇的に変わる。
  • ハーネス = Claude Codeを取り巻く設定・ツール・環境の総合的な仕組み。
  • 「使い方」を極めることが最大のパフォーマンスアップになる。
モデルが賢くても周りが
ダメだと結果が出ない！
これが一番重要なメッセージ。

⑧ハーネス① CLAUDE.md

  • プロジェクトのルール・制約・前提条件を記述するファイル。
  • 階層ロード対応: ルート＋サブディレクトリのCLAUDE.mdが自動で読み込まれる。
  • ルートにはグローバルな知識、サブディレクトリにはローカルな規約を記載。
  • 注意: ファイルはできるだけ軽く保つ。広く適用できる内容に絞ることで性能を維持。
ただの設定ファイルじゃない！
Claudeにとっての
「プロジェクトの道しるべ」。

⑨ハーネス② Hooks

  • ファイル保存前後・コマンド実行前後に自動処理をフックする仕組み。
  • 本来は「制限」ではなく継続的改善のツールとして使うのが正解。
  • `stop`フックで自己振り返りとCLAUDE.mdの自動更新を促せる。
  • `start`フックでチーム固有のコンテキストを動的ロード可能。
Claude自身が
自分を改善していくための
仕組みでもあるんだ！

⑩ハーネス③ Skills

  • 再利用可能なタスク定義。定型作業を自動化する仕組み。
  • 必要な時だけ読み込まれるので、コンテキストの無駄な消費を防ぐ。
  • 大規模コードベースで作業タイプごとの専門知識を段階的に開示。
  • 常に使わない情報をスキルに切り出すことで、コンテキストの肥大化を防止。
必要な時だけ知識を取り出す
「引き出し」のような存在！
無駄がないのがポイント。

⑪ハーネス④ Plugins

  • 外部ツールの統合パッケージ。スキル・Hooks・MCP設定を1つにまとめる。
  • 新規エンジニアが即座にベテランと同じ環境を入手できる。
  • 部門固有の知識（暗黙知）をパッケージ化して組織全体で共有。
  • ホスト型マーケットプレイスでプラグインの更新を全社に配布可能。
「この人しか知らない作業」を
パッケージ化して
チーム全体に展開できる！

⑫ハーネス⑤ MCP

  • Model Context Protocol: 外部リソースへの標準接続プロトコル。
  • 組織の独自システムやツールにClaude Codeを接続する仕組み。
  • 社内のデータベース・API・CI/CDなどに安全にアクセス可能。
  • 標準プロトコルなので様々なツールを統一的に接続できる。
Claude Codeを
社内インフラの中心に
接続する「標準規格」。

⑬ハーネス⑥ LSP

  • Language Server Protocol: IDEと同じシンボルレベルのナビゲーションを提供。
  • 関数の呼び出し追跡・参照の横断検索が高精度に実行可能。
  • 既存のLSPサーバーを活用し、異なる言語の同名関数も正確に区別。
  • 開発者がIDEで使っている機能と同等の能力をClaude Codeに付与。
IDEの「定義へジャンプ」とか
「参照を検索」とかが
Claude Codeでも使えるようになる！

⑭ハーネス⑦ Subagents

  • 大規模タスクを並列分散実行するサブエージェントの仕組み。
  • 探索タスクと編集タスクを分離して効率を向上。
  • 複雑な操作を専門のエージェントに委譲することで集中力を維持。
  • 1つのタスクを複数のサブタスクに分割し、並列処理で高速化。
Claudeが「私」を何人も
生み出して並列で作業する
イメージ！

⑮Context Rotを防ぐ設計

  • コンテキストウィンドウが埋まると、以前の情報を忘れて精度が低下する。
  • これをContext Rot（コンテキスト腐敗）と呼ぶ。
  • ハーネス7要素の設計自体がContext Rotを防ぐように構成されている。
  • Skillsの按分読込やSubagentsのタスク分割がContext Rot対策の要。
AIの「忘れ」を防ぐ設計こそが
ハーネスの最大の価値！
これを意識するだけで全然違う。

⑯組織導入のベストプラクティス

  • 責任者を1人任命し、ハーネスの設計・運用を一元管理する。
  • プラグインの再利用で、部門横断のナレッジ共有を実現。
  • 定期的な見直しサイクルを設けて、CLAUDE.mdやスキルを最新に保つ。
  • 段階的に導入し、成果を測定しながら拡大していくのが成功の秘訣。
技術だけでなく
組織の仕組み作りも
セットで考えないとね！

⑰CLAUDE.md書き方のコツ

  • 基本スタンス: Claudeが賢い前提の「思いやり」で書く。
  • 過度な指示は不要。「ファイルを開く時は…」など常識的なことは省く。
  • プロジェクト構造やアーキテクチャの全体像を簡潔に伝えるのが重要。
  • 広く適用できる内容に留め、詳細はSkillsに切り出すのが正解。
「Claudeは賢い」前提で
書かないと、余計な指示で
コンテキストを圧迫しちゃう！

⑱まとめ: ハーネス7要素を極めよう

  • CLAUDE.md + Hooks + Skills + Plugins + MCP + LSP + Subagents
  • この7要素を極めれば、Claude Codeは最強の開発パートナーになる。
  • 最初はCLAUDE.mdから始めて、段階的に他の要素を追加していくのがおすすめ。
  • agentic search × ハーネス7要素の組み合わせが、巨大コードベース攻略の鍵。
