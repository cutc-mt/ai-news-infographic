Claude Codeの仕組みをやさしく解説

AIエージェントの動作原理からコンテキスト管理まで

2026年5月18日

第1部：Claude Codeの大枠の仕組み

①根本の仕組みはシンプル

  • Claude CodeはPC内のプログラムとして動いている
  • AnthropicのサーバーにいるAIにプロンプトを送ってテキストを受け取るを繰り返すだけ
  • ユーザーのプロンプトに様々な要素をくっつけて組み立てる
  • 仕事が終わったと判断するまで送信→受信→操作→追加を延々と繰り返す
複雑な仕事をこなすけど、
動き自体は超シンプル！

②プロンプトのやり取りサイクル

  • 人間がAIに実現したいことをプロンプトとして送る
  • Claude Codeがプロンプトに様々な要素を付加して組み立てる
  • APIを通じてAnthropicサーバーに送信
  • サーバーからAIの結果テキストが返ってくる
  • プログラムがPC内のファイルを読み書き・操作
  • 操作結果をプロンプトに追加して再送信

③ブラウザ版Claudeとの違い

  • 本質的にはプロンプト送受信の繰り返しは同じ
  • 最大の違いは動ける環境の広さ
  • Claude Code → PCのファイルシステムにアクセス可能
  • ブラウザ版 → PCのファイルをいじる権限がない
  • GoogleカレンダーやSlackなども「環境」になり得る
「環境」＝ AIが情報を読み取ったり
働きかけたりする対象のこと

④Claude Codeを構成する2つの要素

  • 環境とのやり取り → どのように操作しているか
  • 思考の中身 → プロンプトに何が含まれ、どう変化するか
  • この2つを解剖すれば全体像が見えてくる
CodexやCursorなど
他のAIエージェントも
ほぼ同じ仕組み！

第2部：環境とのやり取り（ツール）

⑤AIはテキストしか生成できない

  • 大事な前提：AIができるのはテキストを生成することだけ
  • テキストだけでどうやってファイル編集や各種操作を？
  • それを実現するのがツールという仕組み
テキストだけど実は
いろんな操作ができる！

⑥ツールとは

  • ツール＝AIが呼び出せる、あらかじめ用意された処理の塊
  • プログラムが「こんなツールが使えますよ」とAIに教える
  • 使いたい時は指定フォーマット通りにテキストを書く
  • プログラムがそのテキストを解釈して実際の処理を実行
ツール呼び出しの概念 
// AIが出力するテキスト
{ "tool": "web_search",
  "query": "Claude Code 最新情報" }

// ↓ プログラムが解釈して実行
→ Web検索が実行される

⑦基本ツール：Read・Write

  • Read → ファイルの中身を読み込む
  • Write → ファイルを新規作成、または中身を丸ごと書き換え
  • Claude Codeの操作の大半は基本ファイルツール + Bashで成り立っている
ファイルの読み書きが
Claude Codeの基本動作！

⑧万能ツール：Bash

  • Bash＝パソコンに文字で命令を出す仕組み（黒い画面）
  • ファイル編集、情報取得、動画編集などほぼあらゆる操作が可能
  • 決まったコマンドを入力するだけで実行できる
  • ツールの中でも特に汎用性が高い存在
Bashがあれば
たいていのことはできる！

第3部：機能もすべてツールで動いている

⑨質問機能：AskUserQuestion

  • Claude Codeを使っていると「どうしたらいいですか？」と質問してくる
  • これはAskUserQuestionというツールが呼び出されている
  • すべてのユーザーとのやり取りもツールで実現

⑩プランモード：EnterPlanMode

  • Plan Mode＝事前に綿密な計画を立ててから実装に取りかかる機能
  • ユーザーが計画をレビューした後に実装開始
  • EnterPlanModeツールが呼び出されることで動作
安全に実装を進めるための
大切な機能！

⑪すべての機能がツールとして定義されている

  • サブエージェント → タスクの一部を別のAIに委任
  • スキル → AI向けの業務マニュアル
  • 検索 → コードベースやWebの検索
  • 公式ツール一覧ですべての機能がツールとして定義されていることを確認できる
Claude Code ≒
ツールの集合体！

⑫ツールの拡張方法

  • ユーザーが直接ツールを追加することはできない
  • Bash活用 → コマンドをAIに教えるだけで拡張可能
  • スキル → 実行するコードをマニュアルに書けばOK
  • MCP → ツール呼び出しフォーマットの統一規格
Bashとスキルは汎用性が
めちゃくちゃ高い！

第4部：AIの頭の中身とコンテキスト

⑬セッション開始時に読み込む情報

  • システムプロンプト → より良く動くための指示
  • メモリー → 過去セッションで学んだこと
  • スキルリスト → 使えるスキル（業務マニュアル）の一覧
  • CLAUDE.md → プロジェクトのルールや説明
仕事を始める前から
結構な量が読み込まれている！

⑭プロンプトの膨張問題

  • ユーザーのプロンプトが全体のプロンプトに追加される
  • AIがファイルを読み込むたびに内容がコンテキストに追加される
  • いろんなファイルを読むとあっという間に膨れ上がる
  • これがずっと働いていると調子が悪くなる原因
/contextコマンドで
今のコンテキストの中身を
確認できるよ！

⑮コンテキストウィンドウと制限

  • コンテキストウィンドウ＝AIに渡せる最大テキスト長
  • Claude Opus 4.7だと100万トークン（約200万文字）
  • 上限を超えていなくても関係ない情報が多いと性能が低下
  • この現象はコンテキストロストと呼ばれている
状況に応じた情報の取捨選択が
AIの力を引き出すカギ！

⑯コンパクション機能

  • Compaction＝長くなりすぎた会話ログを丸ごと要約
  • 要約で重要情報が抜け落ちる可能性もある
  • しかしパフォーマンス低下や上限超過を防ぐために重要
情報の損失と
パフォーマンスのバランス！

⑰スキルの段階的開示

  • 最初はスキルの名前と説明文だけが読み込まれる
  • 「今の仕事に使えそう」とAIが判断した時だけ全文が読み込まれる
  • スキルを100個作っても一気に全部は読み込まれない
  • 能力を拡張しつつコンテキストも圧迫しない仕組み
必要な時だけ読み込むから
効率的！

⑱サブエージェントの活用

  • サブエージェントは独自のコンテキストを持っている
  • 途中の作業ログはメインに伝えず、最終成果だけを渡す
  • これによりメインのコンテキストを汚さずに結果だけ得られる
  • すべての機能がコンテキスト効率化という共通の問題に取り組んでいる
各機能がいろんな角度から
コンテキスト問題にアプローチ！
