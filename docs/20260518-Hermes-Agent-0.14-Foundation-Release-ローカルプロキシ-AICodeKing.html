Hermes Agent 0.14 Foundation Release

インストールからプロキシまで、エージェントプラットフォームの基盤が進化

2026年5月18日 | AICodeKing

インストール・パフォーマンス改善

①

PyPIパッケージで簡単インストール

  • Hermes Agentが正式なPyPIパッケージとして公開された。リポジトリをクローンする必要がなくなった。
  • pip install hermes-agent のコマンド一つでインストールが完了する。
  • インストール後は hermes を実行するだけで起動できる。
  • ツールの導入が困難だと、多くのユーザーが「良い部分」にたどり着く前に離脱してしまう。この変更で大幅に敷居が下がった。 
pip install hermes-agent
hermes
従来はリポジトリのクローンとカスタムセットアップが必要だったのが、1コマンドで完結するように！
②

依存関係のスリム化（Debloating）

  • Hermesは以前、メッセージングアダプタやブラウザ機能、音声ツール、画像ツールなど使わない可能性のある依存関係を多数抱えていた。
  • 今回から重いコンポーネントは遅延インストール（Lazy Install）方式に変更され、実際に必要な時だけ読み込まれる。
  • 基本セットアップが軽くなり、通常のマシン、小型VPS、ノートPCでも快適に動作する。
  • Hermes本体がクリーンになり、不要なパッケージに邪魔されない体験が実現した。
全機能を最初から詰め込むのではなく、必要な時だけ追加する設計に切り替えたのがポイント！
③

起動速度の改善（Cold Start）

  • Hermesの起動が高速化され、不要なものを startup 時に読み込まなくなった。
  • セッションを素早く開始したい時に、待ち時間のストレスが大幅に削減される。
  • 派手な機能ではないが、日々の使用体験に大きく影響する重要な改善。
  • 起動時に読み込むモジュールを最小限にし、実際に使用する機能だけを段階的に初期化する設計に変更。
派手さはないけど、毎日使うものだからこそ起動速度は超重要！
④

「Foundation Release」の意味

  • このアップデートは1つの目立つ機能ではなく、基盤全体の強化がテーマ。
  • Hermesをインストールしやすく、軽く、実用的にすることに焦点を当てている。
  • モデル、サブスクリプション、コーディングツールの間に立つローカル層としての役割を明確にしたリリース。
  • 今後の機能追加に向けた「土台」として位置づけられており、拡張性の確保が意識されている。
インストール→軽量化→高速起動の3本柱で基盤を刷新！

ローカルプロキシ・プロバイダー統合

⑤

OpenAI互換のローカルプロキシ

  • Hermesをローカルエンドポイントとして動かせるようになり、OpenAI APIフォーマットで通信できる。
  • 背後で Claude Pro、ChatGPT Pro、SuperGrok などの既存サブスクリプションをルーティング可能。
  • Codex CLI、Aider、Klein、Continue などのツールがHermesをエンドポイントとして利用できる。
  • ツールごとにAPIキーを設定する必要がなく、Hermesがローカルルーターとして一元管理する。 
// ツール側のエンドポイント設定
endpoint: http://localhost:PORT
// Hermesが自動的にルーティング
複数のサブスクリプションを1つのローカル層で管理できるのが最大の魅力！
⑥

xAI Grok統合（SuperGrok OAuth）

  • SuperGrokユーザーは、OAuth経由でGrokに接続できるようになった。別途APIキーは不要。
  • Grokがより大きなコンテキストウィンドウに対応し、大規模なコードベースや長いリサーチタスクに強くなった。
  • プロバイダーの柔軟性が高まり、異なるモデルを用途に応じて使い分けやすくなった。
  • 有料サブスクリプション、ホスティッドモデル、ローカルワークフローの連携をよりシームレスに。
Grokの大規模コンテキストがコードベース全体の理解に役立つ！
⑦

新しい検索ツールの追加

  • X検索ツールが追加され、OAuthまたはAPIキーを使ってX上の投稿やスレッド、反応、製品アップデートを直接検索できる。
  • Brave Search が新しい検索プロバイダーとして統合された。
  • DuckDuckGo スタイルの検索プロバイダーも追加され、予算に優しいオプションが増えた。
  • すべてのワークフローに高価な検索APIが必要ではなく、用途に応じてコストを抑えられるのが魅力。
BraveやDuckDuckGoでコストを抑えつつ、X検索でリアルタイム情報もキャッチ！
⑧

プロバイダールーティングの構想

  • Hermesの目標は、異なる有料サブスクリプション、ホスティッドモデル、ローカルワークフローを統合すること。
  • ユーザーのサブスクリプションとコーディングツールの間に立つ柔軟なローカル層として機能する。
  • ローカルプロキシにより、Codex CLI や Aider など複数のツールで同じエンドポイントを使い回せる。
  • ただし、各プロバイダーの利用制限や規約には引き続き注意が必要。
AIコーディングのエコシステムは既にカオス。1つの柔軟な層で整理するアプローチが秀逸！

メッセージング・ワークフロー統合

⑨

Microsoft Teams統合

  • Microsoft Teamsがエンドツーエンドで統合された。HermesがTeamsからメッセージを受信し、返信を送信できる。
  • 企業環境でのエージェント活用がより現実的に。チームが既に作業している場所にエージェントを配置できる。
  • メッセージの送受信が双方向でリアルタイムに行われる。
ターミナルを開かなくても、Teams上でエージェントと対話できるのが実用的！
⑩

LINE・SimpleX Chatの追加

  • LINE がHermesのメッセージング対応プラットフォームに追加された。
  • SimpleX Chat も統合され、プライバシー重視のユーザーにも対応。
  • これにより、Discord、Telegram、Teams、LINE、SimpleX の5つのメッセージングプラットフォームに対応。
  • ユーザーが既に使っている場所でエージェントを動かすという理念の一環。
LINE対応で日本のユーザーにも使いやすくなった！
⑪

セッションのライブハンドオフ

  • /handoff コマンドで、アクティブなセッションを別のモデルやプロファイルにリアルタイムで引き継げるようになった。
  • 最初は高速なモデルで作業を開始し、タスクが複雑になったら強力なモデルにシームレスに切り替え可能。
  • 会話のコンテキストを維持したままモデルを変更できるため、最初からやり直す必要がない。
  • コーディングワークフローで特に有用。軽い修正には軽量モデル、複雑なリファクタリングには高性能モデルという使い分けができる。
速いモデル→強いモデルへ途中で切り替えできるのが革命的！
⑫

Discord・Telegramの改善

  • ネイティブボタンがTelegramとDiscordに追加され、エージェントの質問にタップで回答できる。
  • 手動でテキストを入力する代わりにボタンをタップするだけ。モバイルでの操作性が大幅に向上。
  • Discord履歴バックフィルがデフォルトで有効化。Hermesがチャンネルに参加した際、直近の会話コンテキストを自動的に理解する。
  • 「今まで何を話していたのか分からない」状態を回避し、自然に会話に参加できる。
ボタンで回答＋履歴を自動理解で、エージェントが「天然で会話に参加」する感覚！

コーディング体験・セキュリティ

⑬

言語サーバー診断の統合

  • Hermesがファイル編集後に言語サーバーの診断情報を自動表示するようになった。
  • 未解決のインポート、未定義のシンボル、型エラーなどを早期に発見できる。
  • コードを書いた直後にフィードバックが得られるため、エラーの蓄積を防げる。
  • IDEのようなリアルタイム診断体験が、エージェントのコンテキストでも実現。
ファイル保存後にエラーを即座に検出。手動で確認する手間が省ける！
⑭

ファイル変更の検証機能

  • 新しいファイル変更ベリファイアが、ディスク上で実際に何が変わったかを明確にまとめる。
  • エージェントの最も厄介な問題の1つである「編集したつもりが実際には変更されていない」問題に対処。
  • エージェントの意図と実際の結果の乖離を可視化し、信頼性を向上させる。
  • diff サマリー形式で変更内容が表示されるため、大量のログを読む必要がない。
「編集したはずなのに...」問題に実用的な解決策がついに登場！
⑮

Computer Useの改善と拡張

  • Computer Useのバックエンドが改善され、Anthropic以外のプロバイダーでも動作するようになった。
  • GUI自動化が特定のプロバイダーに縛られなくなり、自由度が大幅に向上した。
  • 統合動画生成ツールも追加され、エージェントの表現力が広がった。
  • クリッカブルなターミナルURLや、Z統合の改善も含まれている。
GUI自動化がプロバイダー非依存になったのは、柔軟性の面で大きな前進！
⑯

セキュリティレイヤーの強化

  • 危険なコマンドの検出が改善され、より多くのリスクのある操作を事前にブロックできる。
  • sudo保護が強化され、管理者権限を伴う操作に追加のガードレールが設けられた。
  • ツールエラーのサニタイズにより、機密情報がエラーメッセージに漏れるのを防ぐ。
  • エージェントが強力になるほど安全層も慎重にあるべきという設計思想。
強いエージェントには強い安全対策が必要。バランスが良い！
