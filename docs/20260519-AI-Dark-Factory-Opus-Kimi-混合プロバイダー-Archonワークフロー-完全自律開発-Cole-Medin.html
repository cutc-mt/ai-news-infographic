1Dark Factoryとは

  • 人間ゼロ — コードを一切書かない・レビューもしない完全自律型のソフトウェア開発システム
  • Issue作成 → PR作成 → 別エージェントによるレビュー → マージ → 本番デプロイまで全自動
  • 開発者の関与は「この機能を作って」の指示を出すだけ。実装の品質チェックすらAI同士で完結
  • Cole Medinが「工場の暗闇」＝Dark Factoryと命名。人間の介入 lights off の象徴
「ダークファクトリー」＝
工場の電気を消しても
勝手に生産が進む状態。
これが次世代の開発現場。

2Dark Factoryの4つのワークフロー

  • ① Issueトリアージ — 自動で優先順位を判定。どのIssueを先に解くべきかAIが決める
  • ② 個別Issueの修正実装 — トリアージ結果に基づき各Issueを個別に実装
  • ③ PR検証 — 作成されたPRを別エージェントが独立してレビュー
  • ④ リグレッションテスト — 週1回全テストを自動実行。新機能が既存を壊していないか確認
4つのワークフローが自律的に連動。
人間が介入するのは
「この機能が欲しい」と言うだけ。

3Archonワークフローとは

  • 各ワークフローはArchonワークフローで構成。モジュール式のパイプライン設計
  • 各ノード（計画・実装・検証）でモデルとプロバイダーを自由に切り替え可能
  • 例えば「計画はOpus」「実装はKimi」「検証はClaude」とノード単位で最適なモデルを割り当て可能
  • プロバイダーの追加・差し替えも簡単。APIキーを変えるだけで別のモデルに切り替え
Archonの強みは柔軟性。
プロバイダーロックインなしで
いつでも最強の組み合わせを追求できる。

4レート制限問題が深刻化

  • Anthropicの5時間制限と週次制限が最近さらに厳格化
  • OpenAIも同様に制限強化中。サブスクリプションの過剰消費ユーザーが削られつつある
  • Dark Factoryのように24/7エージェントを動かす用途では、レート制限が最大のボトルネック
  • 制限に達するとワークフロー全体がストップ。自動化の恩恵が一瞬で失われる
フラッグシップを
24時間回し続けるのは
現実的じゃない。
だからこそミックス戦略が重要！

5コスト効率の考え方

  • OpusやGPT-5.5を全工程で使うのは非現実的。コストが天文学的になる
  • 計画・調査・実装・検証の全フェーズでフラッグシップを使うとトークン過多に
  • 重要度に応じてモデルを段階的に使い分けるのがコスト効率の鍵
  • 安価なモデルをワークホースとして活用しつつ、最重要工程だけフラッグシップに任せる
全工程フラッグシップ＝
金食い虫。
賢い使い分けがDark Factoryの生命線。

7API信頼性の比較

  • Claude API — ダウンタイムこそ多いが、動作中は非常に安定している
  • Kimi/Minimax — 動作中でもたまにハングする。応答が返ってこないことがある
  • ワークフロー内ではリトライもされない問題がある。ハング＝プロセス停止
  • 自動化システムではAPIの安定性が生死を分ける。ハンドリングが必要

8ミックスプロバイダーの2つの戦略

  • 戦略A：強モデルで計画→弱モデルで実装 — 計画を完璧にして、実装は安価なモデルに従わせる
  • 戦略B：弱モデルで計画→強モデルで実装 — 実装時に強いモデルが自律的に修正
  • 戦略Aのメリット：計画の質が高ければ、実装モデルの能力不足をカバー可能
  • 戦略Bのメリット：強いモデルなら計画の不備も実装段で補正できる
計画を完璧にするか、
実装力でカバーするか。
AI開発の二大流派。

9採用戦略：Opusで計画、Kimiで実装

  • 戦略Aを採用 — 計画を完璧にしてKimiに従わせる方針
  • Opusが計画フェーズを担当。問題分析・方針策定・タスク分解
  • Kimi K2.6が実装と検証のワークホースとして担当。大量のIssueを安価に処理
  • これによりOpusの高コストなトークン消費を計画に限定。全体コストを大幅抑制
最強の頭脳（Opus）で
設計図を描き、
がんばれる作業員（Kimi）で
量産する。合理的！

10Provider切り替えの技術

  • Claude CodeをKimi向けに環境変数でリルートしていた過去あり
  • ミックスするにはClaudeとPi（オープンソースのコーディングエージェント）を分ける必要あり
  • 1つのエージェントに複数プロバイダーを共存させるのは技術的に困難
  • エージェントごとに専用のプロバイダーを割り当てる設計が現実解
Claude Code → Anthropic (Opus)
Pi Agent → Moonshot (Kimi K2.6)
環境変数1行でリルートは
手軽だけど、ミックスは
エージェント分離が必要。

  • Kimi K2.6用のハーネスとして採用。Anthropicエコシステムの外でKimiを動かす
  • Claude Codeより軽量。Dark Factoryの実装フェーズに最適化して使える
Piは「どんなモデルでも

12Adversarial Reviewの仕組み

  • 実装と検証でセッションを完全に分離。クリーンコンテキストを維持
  • PRレビュー時は実装プロセスを一切知らない状態でGit diffのみをレビュー
  • 実装者とレビュアーが別エージェントなので、先入観ゼロの冷徹なレビューが可能
  • 実装のバイアスがレビューに影響しない。人間の同棲レビューの弱点を克服
「誰が書いたか」を知らない
レビュアーが品質だけを
ジッジする。これぞ真の
Adversarial（敵対的）。

13PR検証ワークフロー

  • 検証エージェントにIssueとGit diffだけを知らせる。実装プロセスにはバイアスを持たせない
  • 最大リトライ回数を設定。一定回数レビューOKが出たら次へ進む
  • 準備が整ったら自動マージ＆デプロイ。人間の承認は不要
  • 最大リトライに達しても合格しない場合は、安全のためマージしない仕組み
Issueの要件を満たすかだけを
客観的にチェック。
感情ゼロの品質管理。

14リグレッションテストの自動化

  • 週1回全アプリケーションを自動テスト。スケジュール実行で手間ゼロ
  • 新しい機能追加で既存機能が壊れていないかを全面的に確認
  • 問題を発見したら自動でIssueを作成。修正が必要な箇所をリストアップ
  • 作成されたIssueはDark Factoryの通常ワークフローに流れ、自律修正ループが回る
壊れる→見つける→直す
完全な自動修復サイクル。
