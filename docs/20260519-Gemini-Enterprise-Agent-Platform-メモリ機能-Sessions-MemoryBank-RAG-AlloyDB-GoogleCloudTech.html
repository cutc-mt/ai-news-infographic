AIエージェントにメモリを追加する — Sessions / Memory Bank / RAG の3本柱

エージェントが「賢くなる」ための
具体的な方法が実装レベルで解説されています！

② Agent Sessions（エージェントセッション）

  • 20行未満のコードでセットアップ・設定が可能。最小限の労力で導入できる
  • すべてのADKセッションがクラウドに自動保存される
  • セッションの途中終了 → 後で再開が可能。途中で離れても続きから始められる
  • ステートに情報を保存できる。例：マラソンルートのランドマーク情報を保持
人間との会話みたいに、
「また続きをしよう」ができる。
ステートフルなエージェントの実現！

③ セッション再開の実力

  • デモではマラソンプランナーエージェントが実演。ルート計画の途中でセッションを終了しても、再開時に前回の情報を完全に覚えている
  • マラソンルート上のランドマーク（水補給地点・観光スポットなど）をステートに保存
  • 再開後は「前回の続きから」ではなく、文脈を引き継いだ自然な対話が継続される
  • 開発者はステート管理を意識する必要がほぼない。プラットフォームが自動で処理
従来のステートレスな
API呼び出しとは別次元。
長期的なタスクに最適。

④ Agent Memory Bank（メモリバンク）

  • セッションの次のレベルに引き上げる機能。単なるステート保存ではなく、長期的な記憶を提供
  • エージェントとのターン（リクエスト→レスポンス）ごとに、有用な情報を自動でメモリに保存
  • ローカルファイルではなくクラウドに全自動保存。開発者が手動で保存作業をする必要なし
  • 設定不要の自動化 — エージェントが勝手に「これは覚えておくべき」と判断して保存
人間の「あ、これ覚えとこ」
をAIが勝手にやってくれる。
記憶の自動化がすごい！

⑤ メモリバンクの実例 — ラクダルール

  • デモではラスベガス旅行の計画中、「ラスベガスの公道にラクダを乗せてはいけない」というルールが自動保存される
  • このルールは単なるログではなく、今後の計画で自動的に参照される
  • 後で別の都市の旅行計画を立てた際も、「あ、この都市も公道にラクダはダメだね」と推論に活用される
  • 人間が明示的に「これ覚えて」と指示しなくても、文脈から重要な情報を抽出・保存
ラクダのルール、笑えるけど
実際に法律違反なんですよね。
AIが賢く回避してくれる！

⑥ セッション vs メモリバンクの違い

  • セッション：1つの対話のステートを保持。中断→再開がメインのユースケース
  • メモリバンク：複数のセッションをまたいで知識を蓄積。ターンごとに自動抽出・保存
  • セッションは「短期的記憶」、メモリバンクは「長期的記憶」という人間の脳のモデルに似ている
  • 両者は組み合わせて使う。セッションの情報がメモリバンクに昇華される仕組み
人間の脳と同じように、
作業記憶（セッション）と
長期記憶（メモリバンク）の
二層構造で設計されている。

⑦ RAG — 独自データで正確な情報検索

  • Retrieval Augmented Generation（RAG）で、エージェントに企業独自のデータを提供
  • LLMのハルシネーションを防ぎ、正確で最新の情報に基づいた回答を生成可能に
  • 社内ドキュメント、製品マニュアル、FAQなど独自のナレッジベースを検索して回答に活用
  • セッション・メモリバンクと組み合わせることで、3層のコンテキストが完成
「なんでも知ってる」ふりをする
AIではなく、確実な情報を
元に回答する賢いAIに。

⑧ AlloyDB × Auto Embeddings

  • Google CloudのAlloyDBを使用したRAG実装例がデモで紹介
  • Auto Embeddings機能：レコードを挿入するだけで自動で埋め込みベクトルを生成
  • データエンジニアの負荷軽減：新しいレコードを入れるだけでRAG用埋め込みが利用可能に
  • 従来なら埋め込み生成パイプラインを別途構築が必要だったが、データベース層で完結
埋め込み生成が「INSERTしたら
終わり」になる。運用コストが
劇的に下がる！

⑨ チャンキングとドキュメント処理

  • PDFやドキュメントをデータベースに投入する際のチャンキング処理が重要
  • コンテキスト管理と構造化が鍵。適切に分割しないとRAGの検索精度が落ちる
  • エージェントがRAGの情報を適切に消費できる形に整形する必要がある
  • チャンクサイズ・オーバーラップ量・メタデータ付与など、品質に直結する設計判断が多数
RAGは「データを入れるだけ」じゃない。
どう分割するかが
検索精度の命運を分けます。

⑩ Data Agent Kit（データエージェントキット）

  • VS Code、OSS Code、Cloud Shellで動作する拡張機能として提供
  • VS Codeから直接データベースと対話可能。エディタを離れずにデータ操作
  • 75以上のCode Labが利用可能。手を動かしながら学べる豊富な教材
  • 全コードがオープンに公開。自分のプロジェクトに自由に組み込める
75以上のCode Labってすごい量。
まずはVS Code拡張を入れて
手を動かすのが一番！

  • ワンクリックデプロイでマラソンプランナーのデモを自分で試せる
  • クレジット付きで実行可能。無料枠で始められる
  • 全エージェントコールがスケーリング＆セキュア。本番環境-ready

  • データエンジニアリングコミュニティに大きなインパクト。手作業の大幅削減が期待

⑭ 3つのコンテキスト改善要素まとめ

  • セッションステート：ステートフル化と再開機能。1つの対話の文脈を保持
  • メモリバンク：自動記憶保存とクラウド活用。複数セッションをまたいだ知識蓄積
  • 独自データ（RAG）：AlloyDB + Auto Embeddings。企業データに基づく正確な回答
  • これら3つの組み合わせでエージェントの行動が大幅に改善
