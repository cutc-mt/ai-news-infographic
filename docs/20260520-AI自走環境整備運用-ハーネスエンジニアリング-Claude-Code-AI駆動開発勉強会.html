AI自走環境整備・運用スペシャル #2

ハーネスエンジニアリングでAIを「自分のチーム」にする

📅 2026年5月19日配信 | AI駆動開発勉強会

①ハーネスエンジニアリングとは

  • AIエージェントが自律的に作業できるよう、周囲の環境を設計・管理する技術の総称
  • HashiCorpのMitchell Hashimotoが2026年2月に提唱した概念
  • AI導入6段階の第5段階「Engineer the Harness」として位置付け
  • 核心：エージェントがミスするたびに環境側に恒久的な修正を施す
ハーネス＝馬具。馬（AI）を正しい方向に導く装備。プロンプトエンジニアリングの次のステージ！

②業界主要企業の採用実績

  • OpenAI 手書きコードゼロで約100万行を構築
  • Spotify Honkシステムで1,500件以上のPRを本番マージ
  • LangChain モデル変更なしでTerminal Bench 30位→5位に改善
  • Martin Fowler Guide（事前制御）とSensor（事後制御）の枠組みを体系化
モデルの性能を上げるだけでなく環境を整えるだけで劇的に改善できる！

③なぜハーネスが必要なのか

  • LLMは本質的にステートレスな関数。推論中に何も「覚えない」
  • 同じ入力でも毎回出力が違うのはサンプリング時の確率的揺らぎによるもの
  • 会話が続いているように見えるのは、アプリ層が毎回履歴全体をプロンプトに詰め込んでいるから
  • LLMにとって毎回が「初めまして」。ルールも失敗も毎回外から注入が必要
「注入する仕組み」全体を設計するのがハーネスエンジニアリング！

④3つのエンジニアリング比較

  • プロンプト：入力テキストの最適化。「何を聞くか」
  • コンテキスト：渡す情報全体の最適化。「何を見せるか」（RAG・メモリ等）
  • ハーネス：外側の実行環境全体の最適化。「どう動かすか」（ツール・権限・ガードレール）
段階的に広がるイメージ。ハーネスが一番包含範囲が広い！

⑤Layer 1: CLAUDE.md（憲法）

  • Claude Codeが最初に読み込む指示書。プロジェクトの目的・構成・ルールを記述
  • 「やらないこと」を明記するのが最も費用対効果が高い
  • 書くべき：ビルド・テストコマンド、コードスタイル、ブランチ命名規則、アーキテクチャ
  • 書かない：コードを読めば分かること、自明な指示、頻繁に変わる情報
公式：「全行について『削除したらエージェントがミスをするか？』と問え」

⑥個人設定のimport方式

  • チーム開発ではCLAUDE.mdがgit管理されるため、個人の好みを直接書けない
  • 解決策：CLAUDE.md末尾に@my-preferences.mdを追記
  • my-preferences.mdは.gitignoreに追加。各メンバーが独自設定を管理
  • 参照先が存在しない場合は無視されるので、チームへの影響はゼロ
チームルールは共有＋個人の好みは各自管理。両立できる仕組み！

⑦Layer 2: Rules（ドメインルール）

  • .claude/rules/にMarkdownを置くと自動読み込み。CLAUDE.mdの肥大化を防止
  • globsなし：セッション開始時に常時ロード
  • globsあり：マッチするファイルを読んだ時点で遅延ロード
  • globsで条件付きにしてコンテキストを節約するのがベスト
CLAUDE.mdは「憲法」であり「百科事典」にすべきではない！

⑧Layer 3: Skills（ワークフロー）

  • /コマンド名で呼び出せるワークフロー。SKILL.mdにMarkdownで記述
  • 重要：LLMとスクリプトの責務を分離する
  • 決定的な処理（API・パース・I/O）はスクリプト。LLMは「判断」と「生成」に限定
  • 実例：スクリプト化でトークン消費1/6・処理時間1/3に改善
全部をLLMに任せるな！確定的な処理はスクリプトが正解

⑨Layer 4: Agents（サブエージェント）

  • @名前で呼び出す専門特化のサブエージェント
  • メインと別のコンテキストで動作。大量情報を扱ってもメインを汚さない
  • 数百ファイルの横断検索や要約に最適。要約だけがメインに返る
  • 「〜について何かあったっけ？」のような曖昧な問いに対応
コンテキスト分離が最大のメリット。長時間タスクでも安心！

⑩Layer 5: Settings（権限制御）

  • settings.jsonでツールの使用権限を制御。ハーネスの「安全装置」
  • allow：業務に必要な範囲で広く許可。毎回確認が出るとリズムが崩れる
  • deny：破壊的操作（リポ削除・アーカイブ・PR強制クローズ）を明示的に禁止
  • 汎用許可はsettings.json（共有）。URL固定curl等はsettings.local.json（ローカル）
最初から完璧な設定は不可能。実運用で徐々に収束させるのが現実的！

⑪Explore→Plan→Implement

  • Explore：Plan Modeでファイルを読み、コードベース構造を理解。変更はしない
  • Plan：詳細な実装計画作成。テキストエディタで編集も可能
  • Implement：計画に従いコードを書き、テストを実行し結果を検証
  • 結果ではなくアプローチを指定する。「どうやってほしいか」を伝える
3段階で確実に進める。いきなり実装するなってことだね！

⑫テスト駆動のフィードバックループ

  • テストがエージェントへの精密な仕様書として機能する
  • Edit→Test→Fixの自動修復ループがエージェンティクスの本質
  • Spotifyは3チャンネル検証：決定論的検証器・ストップフック・LLMジャッジ
  • テスト実行速度がループ品質に直結。高速化はコスト最適化
テストが遅いとトークン消費が膨らむ。高速化＝コスト削減！

⑬コンテキストエンジニアリング4戦略

  • Write：外部ファイルに記録してセッションをまたいで保持
  • Select：必要な情報を適切に取得（スクラッチパッド・メモリ活性化）
  • Compress：情報を要約して圧縮。95%に達すると自動コンパクション
  • Isolate：コンテキストを複数エージェントやランタイムに分割
Karpathy曰く「コンテキストウィンドウを満たす繊細な技術と科学」

⑭マルチエージェントパターン

  • Writer/Reviewer：一方が実装し、もう一方が新鮮なコンテキストでレビュー
  • 並列スペシャリスト：セキュリティ・スタイル・パフォーマンスを同時に実行
  • Fan-out：マイグレーション対象ファイルごとにclaude -pを並列実行
  • 57%の組織がマルチステップのエージェントワークフローを導入済み
複数エージェントを協調させて大規模タスクをこなす！

⑮ガードレールとセキュリティ

  • 最小権限の原則。デフォルトで書き込み・Bash・MCP実行時に許可を求める
  • サンドボックスでOSレベル制限：gVisor・Firecracker・WASIなど
  • ルールファイルインジェクション攻撃に注意！外部リポを無条件に信用しない
  • プロンプトで意図を表現しコードでルールを強制。セキュリティをLLMに任せない
最大反復回数を設定して暴走を防ぐ。PRは自動マージしない！

⑯コストとパフォーマンス最適化

  • 未最適化だと1セッション$10〜$100+。戦略的最適化で80%以上削減可能
  • プロンプトキャッシュ：静的部分を記憶。レイテンシ最大80%・コスト最大90%削減
  • モデルルーティング：80%の単純クエリを安価なモデルに振り分け。コスト40〜60%削減
  • 日常はSonnet、複雑な推論のみOpusにエスカレーションする運用が推奨
全部Opusにするな！簡単なタスクは安いモデルで十分

⑰ダブルループモデル

  • ループ1：探索（Vibing）：コード品質を気にせず高速にプロトタイピング
  • ループ2：洗練（Refinement）：エンジニアリングモードに切り替えて改善
  • ループ1の出力をベースラインとして扱い、PRレビュー・フィードバック・改善を反復
  • Anthropicの2エージェント構成：初期化エージェント＋コーディングエージェントで長時間実行タスクを遂行
まず広く探索してから、しっかり洗練させる2段構え！

⑱まとめ：ハーネスでAIをチームメイトに

  • CLAUDE.mdで憲法を定め、Rulesでドメインルールを分離し、Skillsでワークフローを定義
  • Agentsで専門タスクを委譲し、Settingsで安全装置を整備する
  • これらを積み重ねるとClaude Codeは文脈を理解しルールに沿って動く「自分のチーム」になる
  • OpenAIは手書きゼロで100万行、Spotifyは1,500件のPRをマージ。ハーネスが整っていれば扱える領域はどこまでも広がる
ハーネスエンジニアリングは「賢いチャットボット」を「自分専用の開発環境」に変える！
