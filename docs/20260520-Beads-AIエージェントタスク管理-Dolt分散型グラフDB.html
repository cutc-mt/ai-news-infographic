MEMORY.md卒業！
AIエージェント専用タスク管理「Beads」

分散型グラフデータベースでエージェントの記憶を構造化

2026年5月20日

①Beadsとは: AIエージェント専用タスク管理DB

  • AIコーディングエージェント専用のタスク管理データベース。コマンドは bd の2文字で起動。
  • Claude Code、Codex CLI、Aider、Cline、Cursor など、主要なエージェントツールに対応。
  • 単なるタスク管理ではなく、依存関係まで把握するグラフ構造のDB。
「Git for Tasks」がキャッチコピー！タスク管理の常識を変えるツール🌟

②MEMORY.mdの限界: コンテキストリセットで記憶が飛ぶ

  • コンテキストウィンドウがリセットされると、昨日までの作業内容をエージェントが忘れる。
  • 仕方なく MEMORY.md や TODO.md に状況を書き込むのが現状。
  • 放置すると無限に膨らんでいき、気づいたら数千行のファイルが完成する。
  • 自由形式テキストだから、エージェントが解釈ミスして同じタスクを2回実行することも。
「続きを頼んだら、どこからでしたっけ？」→ エージェあるある😭

③複数エージェントの衝突問題: TODO.mdが壊れる

  • Claude CodeとCodexを同じプロジェクトで並列実行すると大惨事が発生。
  • 両方が同時に TODO.md を書き換えてGitで衝突し、最悪の場合ファイルが壊れる。
  • せっかく並列で高速化しようとしているのに、コンフリクト解消で逆に時間がかかる。
  • テキストファイルベースでは、マルチエージェントの排他制御が原理的に不可能。

④Beadsの設計思想: 分散型・グラフ型・エージェントファースト

  • 分散型: クラウド不要。基本ローカルで動作し、Gitのようにピア同期可能。
  • グラフ型: タスク同士をリンクで繋いで管理。DAG（有向非巡回グラフ）構造。
  • エージェントファースト: 人間UIではなく、エージェントがCLIから直接操作できる設計。
bd ready → 着手可能タスクをJSONで出力
bd create --priority P0 → 最優先タスク作成
GitHub IssuesやLinearは人間中心。Beadsは機械中心の設計🎯

⑤Doltとは: バージョン管理可能なSQLデータベース

  • DoltはGitのバージョン管理機能を持ったSQLデータベース。SQLでクエリ可能。
  • データに対してブランチ作成・チェックアウトができる。Gitと同じ感覚で操作。
  • ピア同期: チームや別マシンとGit感覚でデータ共有可能。
  • SQLクエリ: 構造化データなので誤読がほぼゼロ。エージェントと相性抜群。
SQLite＋Gitバージョン管理＝Dolt。シンプルで強力な組み合わせ✨

⑥セルレベルマージ: Dolt最大の強み

  • 表計算ソフトのようにセル（1マス）を最小単位として競合判定を行う。
  • Gitは行単位だから、同じ行を2人が違う風に編集すると必ずコンフリクトする。
  • Doltなら、同じ行でも違うカラムを編集していれば両方の変更を自動でマージする。
  • 例: Claudeが優先度を変更し、Codexがアサイン先を変更しても衝突しない。
これがマルチエージェント調停の安全弁！TODO.mdでは絶対無理🔥

⑦ハッシュベースのタスクID: マージ時の衝突を回避

  • IDは bd:A1B2 のようなランダムハッシュで生成される。連番ではない。
  • 連番だと、ブランチAとブランチBで同じ番号が衝突するリスクがある。
  • ハッシュなら衝突確率が天文学的に低い。Gitのコミットハッシュと同じ哲学。
  • 階層構造も可能: bd:A1B2 のサブタスクは bd:A1B2.1。さらに深く bd:A1B2.11 まで。

⑧5種類のリンク: タスク間の関係を明示的に表現

  • BLOCKS: これが終わらないと進められない（依存関係の主役）
  • COMPLETED_BY: 完了タスクの記録
  • DUPLICATES: 重複タスクのマーキング
  • SUPERSEDES: 新しい設計が古い設計を置き換える
  • REPLIES_TO: メッセージのスレッド接続
A→B→C のブロックチェーンで依存関係を可視化！🔗

⑨bd readyコマンド: 着手可能タスクを一瞬で抽出

  • bd ready を叩くと、ブロッカーが全て解除されたタスクだけが一覧で出力される。
  • エージェントは「次に何をやるか」を迷わず即座に決定できる。
  • MEMORY.mdだと、LLMがブロック状態を見落として誤ったタスクに着手するリスクがある。
  • Beadsなら依存関係グラフを解析して100%確実に判定可能。
bd ready --format json
→ P0: ログイン機能実装 (bd:A1B2)
→ P1: DBマイグレーション (bd:C3D4)

⑩エージェント直結: JSON出力でパースが簡単

  • bd ready --format json でマシンリーダブルなJSONを出力。
  • エージェントがパースして次のアクションを自律的に決定できる。
  • bd create --priority P0 で優先度付きタスク作成。P0 なら最優先。
  • 優先度順に bd ready から取得して実行、という運用が自然に組める。
「準備OK」＝bd ready。名前も直感的で使いやすい😊

⑪Compaction機能: 完了タスクを自動圧縮要約

  • 古い完了済みタスクを意味的に要約して、コンテキストの圧迫を防ぐ。
  • 全詳細を残すとコンテキストウィンドウを圧迫するが、消すと履歴が失われるジレンマ。
  • メモリ減衰という考え方。人間の記憶と同じで、古いほど抽象化される。
  • 最新のアクティブタスクに集中できるので、エージェントの判断精度が向上。
10年前の細かい記憶は忘れるけど概要は残る＝人間の記憶と同じ！🧠

⑫永続メモリ: bd rememberとbd prime

  • bd remember: PostgreSQLのように永続的なキーバリュー保存が可能。
  • bd prime: Markdown形式で保存メモリを出力。エージェント起動時に呼ぶ設計。
  • MEMORY.mdの置き換えとして機能するが、構造化されているからノイズが入りにくい。
  • 自由形式テキストの「解釈ガチャ」がなくなり、毎回安定した読み取りが可能。
# エージェント起動時に
MEMORY=$(bd prime)
# → 構造化されたメモリを取得

⑬MEMORY.md vs Beads: 構造化データの圧倒的勝利

  • MEMORY.mdは自由形式 → LLMの解釈に依存しすぎる。毎回違う読み方をされる。
  • Beadsはグラフ構造 → BLOCKS REPLIES_TO で明示的に関係を表現。
  • Markdownには本質的にIDがない。タスクの並び替えで参照が壊れる致命的問題。
  • 排他制御: Beadsは原子的クレームで安全。TODO.mdは書き込み途中の上書きリスクあり。
  • 変更履歴: Doltのコミットログで誰がいつ何を変更したかを完全に追跡可能。

⑭インストールと初期セットアップ: 数分で完了

  • Mac: brew install beaded-dev/tap/beads で1発インストール。
  • Linux/Mac: npm install -g @beads-io/beads または curl でもインストール可能。
  • プロジェクトディレクトリで bd init を叩くだけ。ローカルにDolt DBが立ち上がる。
  • .beaded/dolt ディレクトリが作成される。Embedded Modeで個人利用に最適。
  • ファイルロックで排他制御。1人で使う分には全く問題なし。
cd my-project
bd init
# → .beaded/dolt/ が作成される ✅

⑮エージェント連携: bd setupで自動設定生成

  • bd setup claude でClaude Code用の設定ファイルが自動生成される。
  • AGENTS.md が更新され、エージェントにBeadsの使い方を自動で教える。
  • 対応ツール一覧は bd setup --list で確認可能。新エージェントも随時追加中。
bd setup claude     # Claude Code用
bd setup codex      # Codex CLI用
bd setup aider      # Aider用
bd setup cursor     # Cursor用
bd setup max        # マルチエージェント管理用
コマンド1発でエージェント教育完了！面倒な設定なし🎉

⑯マルチエージェント協調: Claude×Codexの並列実行

  • ターミナルAで bd setup claude、ターミナルBで bd setup codex。同じBeadsを見る。
  • 両方で bd ready を叩くと同じタスクが表示される。
  • Claudeが bd update bd:A1B2 --claim で原子的にクレーム。その瞬間ロックがかかる。
  • もう一度 bd ready を叩くと、クレーム済みタスクは表示されない。自然とタスクが分散する。
  • 裏側ではDoltのセルレベルマージが衝突を防いでいる。
ClaudeとCodexが仲良く協力できる世界が来た🤝

⑰特殊モード: ステルス・Git-free・ ephemeral

  • Stealth Mode: bd init --stealth。共有リポジトリに公開しない個人用モード。
  • NO_GIT 設定でGit操作も無効化。完全にローカル・ノーコミット。
  • Git-free Mode: 環境変数 BEADS_DIR で保存先を自由に指定。
  • Ephemeral DB: テストやベンチマーク用の使い捨てデータベース。毎回クリーンな状態。
  • Contributor Mode: OSS貢献者向け。bd init --contributor でプランニングを別リポに分離。
SupabaseやGoogle Piperなど企業内VCSでも対応可能！🏢

⑱Beadsの世界観: エージェント時代のGit for Tasks

  • Git登場前は各自バラバラのバックアップだった。今のMEMORY.md/TODO.mdはまさにその段階。
