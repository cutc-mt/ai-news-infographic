Archon × Jira 連携ライブビルド

チケットをドラッグするだけでプルリクエストが完成する世界へ

Archonとは

① Archonとは: オープンソースのAIコーディングハーネスビルダー

  • Archonは開発プロセスを再現可能なワークフローとしてパッケージ化するツールです。
  • エージェントにシステムを組み込むのではなく、システムにエージェントを組み込むという逆転の発想。
  • 決定論的ノード（コマンド実行）と非決定論的ノード（AI推論）を組み合わせて、プロセスを再現可能にします。
  • ワークツリーによる並列実行、モデル・プロバイダーの混在使用にも対応。
従来の「プロンプトを積み上げる」手法は脆い。ArchonはDAG（有向非巡回グラフ）で確実に実行するアプローチ！

② 7つのDAGノードプリミティブ: ワークフロー構成要素

  • Prompt - AIエージェントに推論をさせるノード
  • Command / Bash / Script - 決定論的なコマンド実行
  • Loop - 繰り返し処理の制御
  • Approval / Human in the Loop - 人間の承認ゲート
  • Cancel - ワークフローの中止
  • Orchestration - 全体の調整と管理
各ノードはエージェントかコマンドのどちらか。この組み合わせでワークフローを構築します。

③ 既存のプラットフォームアダプター: 6つの接続先

  • Web Interface - ブラウザから直接ワークフローを起動・ログ確認
  • CLI - Claude Codeやセカンドブレインからコマンドラインで操作
  • Slack - スレッドごとに独立した会話、並列処理が可能
  • GitHub - ソースコントロールとPR管理
  • Telegram / Discord - チャットベースの連携
今回はこのリストにJiraを追加するライブビルドを実施！
Jira連携の目的と設計

④ Jira連携の目標: チケットごとにエージェントと会話

  • 各Jiraチケットに専用の会話を持たせる。Slackのスレッドと同じ仕組み。
  • チケット内で@Archonとメンションするだけでワークフローが起動。
  • Archonがコメントとして返信し、チケット上で対話が完結。
  • Stack Overflow調査によると、80%の組織がJiraを使用。企業にとって必須の連携先。
自動でチケットを処理するのではなく、チケット内で会話して一緒に進めるアプローチ！

⑤ 公式アダプター vs コミュニティアダプター

  • 公式アダプター - メンテナーが責任を持って更新。Web、CLI、Slack、GitHubなど。
  • コミュニティアダプター - コミュニティメンバーが構築。Gitea、GitLab、Discordなど。
  • Jiraアダプターはコミュニティアダプターとして分類。将来はAsana、Monday.com、Linearも想定。
  • カテゴリはタスク管理（Task Management）。チャット（Chat）カテゴリとは独立。
AsanaやMonday.comなど、将来的に他のタスク管理ツールも同じカテゴリにまとめる設計です。

⑥ 双方向通信の仕組み: WebhookとAPI

  • Webhook（インバウンド） - ユーザーがコメントした瞬間にJiraからArchonへ通知を送信。
  • API（アウトバウンド） - ArchonからJiraへ返信を投稿し、コメント履歴を取得。
  • Webhookは共有シークレットで認証。APIはHTTP Basic Authで認証。
  • ngrokを使ってローカルサーバーにパブリックURLを公開してWebhookを受信。 
// Webhookの流れ
ユーザーがコメント → JiraがWebhook発火
→ Archonが受信 → ワークフロー実行
→ APIでJiraに返信
認証と環境設定

⑦ Atlassian認証: APIトークンで接続

  • JIRA_DOMAIN - 自分のAtlassianドメイン（例: myteam.atlassian.net）
  • JIRA_EMAIL - Atlassianアカウントのメールアドレス
  • JIRA_API_TOKEN - Atlassianで生成するAPIトークン
  • JIRA_WEBHOOK_SECRET - Webhookの検証用シークレット（ランダムなhex文字列）
  • JIRA_BOT_MENTION - トリガー文字列（デフォルト: "Archon"） 
// 環境変数の設定
JIRA_DOMAIN=myteam.atlassian.net
JIRA_EMAIL=user@example.com
JIRA_API_TOKEN=<token>
JIRA_WEBHOOK_SECRET=<hex>
JIRA_BOT_MENTION=Archon

⑧ Webhookの設定手順: Jira側の構成

  • ngrokを起動してローカルのポート3000をパブリックURLに公開。
  • Jiraの設定からシステムWebhookを作成。URLにngrokのURL + /webhooks/jiraを指定。
  • シークレットには環境変数と同じ値を設定。
  • イベントはissue createdとcomment createdを有効化。
  • JQLクエリを空にすると全プロジェクトの全チケットが対象に。
URLのスペルミス（3つの"o"など）に注意！404エラーの原因になります。

⑨ Bot Mentionの仕組み: シンプルな文字列マッチ

  • @Archonとコメント内でメンションするとトリガーが発動。
  • BotのアカウントIDではなく、プレーンな文字列として処理。大文字小文字は区別しない。
  • 環境変数が未設定の場合は"Archon"がデフォルト。
  • 初期実装ではアカウントIDを前提としていたが、プレーンテキストに修正した。
ユーザーはBotアカウントを作成する必要なし。ただ@Archonと書くだけでOK！
PIVループによる開発プロセス

⑩ PIV System Evolution: 12ステップの開発サイクル

  • Prime（準備） - コードベースの文脈を事前読み込みして理解を深める。
  • Plan（計画） - 包括的な実装計画を作成。人間が承認ゲートで確認。
  • Implement（実装） - 計画に基づいてコードを変更。
  • Validate（検証） - 型チェック、リント、手動レビューを実行。
  • System Evolution（進化） - 実装中の問題から学習してAIレイヤーを改善。
PIV = Plan・Implement・Validateのループ。毎回の反復で品質が向上します。

⑪ モデルの使い分け: Opusで計画、Sonnetで実行

  • Opus（計画フェーズ） - より強力な推論能力で包括的な実装計画を作成。
  • Sonnet（その他のフェーズ） - 高速でトークン効率の良い処理。
  • Kimi K 2.6（実装フェーズ） - コーディング専用の軽量モデル。
  • ベンチマーク結果: 計画にOpusを使うだけで、全体の品質が大幅に向上。
全工程に最強モデルは不要。計画の質が結果を左右するから、計画だけに投資する戦略。

⑫ 人間の承認ゲート: 3つのレビューポイント

  • ゲート1: プライムの確認 - エージェントの理解が正しいか検証。追加質問にも回答。
  • ゲート2: 計画の承認 - 9タスク・8ファイルの変更計画を人間が確認して承認。
  • ゲート3: 実装の検証 - コードレビュー、テスト結果、修正の最終確認。
Fire and forgetではなく、人間が各段階で介入して品質を担保する設計です。
実装・テスト・デバッグ

⑬ Dark Factory実験プロジェクト: RAGベースの検索アプリ

  • Dino Chat - YouTube動画とコースコンテンツを横断検索するRAGエージェント。
  • 階層型RAG - チャンクを検索後、動画全体を拡張読み込みして包括的な回答を生成。
  • PostgreSQL + pgvector でベクトル検索。グラフDBは使わずシンプルに。
  • Geminiでトランスクリプトを抽出し、マークダウンとして保存→チャンク化→ベクトルDBへ。
Dark Factory自身が自分自身を構築。GitHub Issueで戦略を指示して自律開発した実績あり。

⑭ Jiraテスト用バックログの作成: 10の実チケット

  • バグ修正 - コピー機能の不具合やUI表示の問題を特定。
  • 動画ライブラリのソート機能 - アルファベット順、日付順のデフォルトソートを追加。
  • リアルタイム検索フィードバック - エージェント処理中に取得チャンクをリアルタイム表示。
  • コピー-to-クリップボード - アシスタントメッセージのコピー機能を追加。
AIにコードベースを分析させ、実際の改善機会をJiraチケットとして自動作成した。

⑮ デバッグの過程: 5つのハードルを乗り越えて

  • 環境変数名の誤り - JIRA_BASE_URLではなくJIRA_DOMAINが正解。
  • Webhook URLのタイプミス - 3つの"o"を入力して404エラーが発生。
  • Webhookシークレットの設定漏れ - シークレットパラメータが空で送信される問題。
  • メンション検出の失敗 - Jiraがプレーンテキストで送信、ADF形式を期待していた。
  • API認証エラー（401） - 古いAPIキーが原因。新しいキーを生成して解決。
外部APIのペイロード形式を事前に把握しておくことが最重要という教訓。
