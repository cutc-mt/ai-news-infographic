導入 ─ チャットUIの限界と次の時代

①クロードが描いたのは漫画ビューアー

  • クロードに漫画を読ませてと頼んだところ、要約ではなく本物の漫画ビューアーが返ってきた
  • ページ送りや見開き表示、解説ボタンまで付いた完全なUI
  • 「質問に答えているのではなく、体験そのものを描してきた」という驚きの体験
  • 吹き出しでテキストが流れるチャット体験ではない。中にアトリエが住んでいる感覚
要約じゃない！体験そのものを返してくる。これこそが今日のテーマの入り口だよ〜！

②本日のテーマ ─ 2つの矢印が交差

  • ゴールは2つ：MCP Appsが何を描いたのか、WebMCPが何を狙っているのか
  • 名前が似て混ざりやすいが、完全に別物
  • 2つの矢印が逆向きに交差している
  • エージェントの中にWebが動く＋Webサイトがエージェントに呼ばれる、両方向の流れが同時に始まっている
MCP AppsとWebMCPは別物！ここを腹落ちするところまでご案内しますぽん！

③チャットUIの限界 ─ 認知負荷の丸出し

  • チャットGPTを前にして何を聞けばいいか分からなくなる瞬間は、あなたのせいじゃない
  • 入力欄の前方向に腕が伸びる構造 ─ どの腕を伸ばすかは毎回ユーザーが完売させられる
  • イニアならチケットのサービスだから…、ボックスならストレージだから…と前提を全部頭に入れないと質問が組めない
  • これが認知負荷の丸出し。地味に疲れる体験
入力欄を前にして手が止まる。あれ、エージェントのせいじゃなくてUIのせいだったんだ！

④チャットはエージェント時代のCLI

  • チャットはエージェント時代のCLI（コマンド画面）に過ぎない
  • 歴史は繰り返す：CLI→GUI、コマンド→タップ、テキスト→視覚
  • プログラマーはCLIが好きでも、一般多数派はiPhoneのような見れば分かるUIに流れる
  • 「チャットボットは死ぬ」のような見出しがメインストリームで流れ始めている
DOS画面→Windows→スマホと同じ流れがAI体験でも起きている！液晶史の繰り返しだよ〜

⑤移行の数字 ─ 5%から40%へ

  • パートナーの予測：26年末までに企業アプリの40%がエージェントUIに向かう
  • 前年は5%未満。5%から40%は劇的なシフト
  • 「移行の年」と呼んで遜色ない規模の変化
  • 背景にあるユーザーの不満は本物。数字が裏付けている
5%→40%って8倍増！もう戻れない流れが来てるぽん！
MCP Apps ─ エージェントの中にアプリを描く

⑥MCP Appsの核心 ─ 画面を返す

  • これまでのツールはJSONやテキストつまりデータを返していた
  • MCP AppsはHTML＋CSS＋JSをまとめたファイルを返せる
  • データではなく画面を返す。出力が根本から変わった
  • 頼み事をすると、エージェントが返事の代わりに専用の画面を組み立てて差し出す
テキストじゃなくて画面が返ってくる！聞く→押すへの転換がここで起きる！

⑦サンドボックスiframe ─ ミニアプリ召喚

  • 技術的にはサンドボックス化されたiframeとして描画される
  • iframe＝ウェブページの中に別ページを埋め込む仕組み
  • 会話の途中に本物のミニアプリがぽんと召喚される
  • ユーザーは「何を聞くか」から「どこを押すか」に移れる
認知負荷がガッツリ下がる！聞く→押すへの転換がチャットUI限界の直撃ポイントだね！

⑧MCP Appsのリリースと対応環境

  • 2026年1月26日に正式リリース
  • AnthropicとOpenAIの共同開発という位置づけ ─ これだけでも事件
  • 対応範囲：ChatGPT、Claude、Gemini ─ 普段の作業面にすでに入っている
  • 実験おもちゃではない。メインストリームのエージェントで動いている
AnthropicとOpenAIが組んでる！競合が共同開発って超大事件ぽん！

⑨漫画ビューアー ─ アプリが呼ばれて出てくる

  • 自分のウェブコミックをクロードの中で完全動作させることに成功
  • ページ送り、解説、テキストモードなど必要な機能が全て揃ったUI
  • 別タブに飛ぶ必要も別アプリを開く必要もない
  • 漫画アプリ、動画アプリ…アプリが消えるのではなく呼ばれて出てくる発想への転換
アプリの切り替えが当たり前だった世界が、必要な時だけ現れる形に変わる！

⑩MCP Appsのセキュリティ設計

  • ローカルストレージが使えない ─ 勝手にデータ保存不可
  • ネットワークも制限付きアクセス ─ 外部APIはサーバーツール経由
  • CSPやCORSのブラウザセキュリティ要件を満たす必要がある
  • 自由にしすぎると悪意アプリが暴れる。縛りすぎると面白いUIが作れない。サンドボックスでバランスを引いている
アプリは出すけれども権限はガッチリ縛る。設計の妙だね〜
WebMCP ─ Webサイトがエージェントに呼ばれる（逆向きの矢印）

⑪WebMCPとは ─ 矢印が完全に逆

  • MCP Apps: エージェントの中にWebを描く
  • WebMCP: エージェントがWebサイトを直接動かす
  • あなたの運営しているサイトのボタンがAIエージェントから直接押せるようになる
  • もう一度言う。あなたのサイトのボタンがAIから直接押される
感覚的にぞっとしませんか？サイトのボタンがAIに直接押される世界！

⑫WebMCPの仕組み ─ 最小コストで対応

  • Webサイト側がフォームや要素にツール名と説明を属性としてつけるだけ
  • ブラウザ内蔵のエージェントがその要素を道具として認識して呼べるようになる
  • JavaScriptから命令的に登録するAPIも用意されている
  • 既存のサイトを最小コストでエージェント対応にできる
属性を足すだけ！既存サイトの改装が不要なのがポイントだね〜

⑬WebMCPの現在地 ─ 実験段階

  • 2026年2月頃からChrome 146カナリーでフラグ付き提供開始
  • カナリー＝Chromeの実験版。フラグ付き＝設定でオンにしないと使えない
  • W3Cコミュニティグループでドラフト。正式な規格ではない
  • ここを混ぜると判断が謝るから注意が必要
実験はできるけど当たり前ではない。正確に位置づけを把握して！

⑭WebMCP vs MCP ─ JavaScriptとJavaの如く

  • 名前が似ているのでついたくなるが中身は別物
  • WebMCP: WebMCP ─ ブラウザ内蔵のエージェントがWebを操作するための仕組み
  • MCP: Model Context Protocol ─ エージェントと外部ツールを繋ぐプロトコル
  • 関係はJavaScriptとJavaみたいなもの。同じ仕様だと思って話すと必ず事故る
ここだけは覚えて！名前似てるけど中身は別物。事故を防ぐために！

⑮なぜWebMCPが必要か ─ トークン燃費の改善

  • 現在のブラウザ内蔵エージェントはスクリーンショットを取ってDOMを読んで推測
  • それを大量のLLMに撫でて推論。トークンを猛烈に燃やしている
  • WebMCPの発想：最初から呼べる関数として渡せ
  • ボタンに「購入処理です」と書けば推測する必要がない。呼ぶだけ。これがトークンの燃費改善
スクショで推測するのをやめて、最初から関数として渡す。シンプルで強力な発想の転換だね！
圧倒的な成長と向き合うべき課題

⑯MCPの圧倒的な成長 ─ 970倍

  • MCPは18ヶ月で970倍成長。聞き間違いではない
  • 月間ダウンロード数は9,700万〜1億まで目前
  • エンタープライズ採用率78%（前年31%から2.5倍以上）
  • 25年12月にCNCF（Linux Foundation傘下）に寄託。中立な最短が持つ技術は長くいける
970倍！もう趣味レベルじゃない。企業として組織として採用されている段階だよ〜

⑰主要パートナーと標準化の波

  • ランチャーパートナー：Amazon、Box、Canva、Figma、Slack、Salesforce
  • 本丸にもう入っている。これは誰かが新しい恋愛を作るレベルの話ではない
  • 標準化された実装が配られ始めている
  • 個別企業の方針転換で消える確率もぐっと下がった
ここまで来るともう戻れない！標準化の波は止まらないぽん！

⑱リスクと課題 ─ 魔法ではない

  • セキュリティ: 認証情報の盗難、プロンプトインジェクション、ツールポイズニングのリスク
  • 悪意あるページにエージェントの勝手な操作指示を仕込まれる可能性
  • パフォーマンス: ツールを呼んでまた別を呼ぶ往復で待ち時間が伸びる（ダブルホップ問題）
  • コンテキスト汚染: UI操作を全てエージェントに送ると1文字打つたびにトークン消費。便利にしたつもりで同じ罠を踏む
MCP AppsもWebMCPも魔法ではない！設計を雑に扱うとチャットUIの不便さを別の形で増幅させるよ！
