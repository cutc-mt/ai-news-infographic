②HTMLがMarkdownを超える5つの理由

  • 情報密度: テーブル・SVG・CSS・JS・画像を1ファイルに
  • 長文可読性: 100行超えでもナビで読める
  • 共有性: ブラウザで即開ける、S3やCDNでリンク共有
  • 双方向操作: スライダーやボタンでUIを操作できる
  • Joy Factor: よく作られたHTMLは読む喜びがある
Markdownはシンプルだけど、エージェント時代には物足りなくなってるんだよね〜

③情報密度の違いを実感

  • スティッキーヘッダー付きの複雑テーブルを単一ファイルで
  • SVGイラスト・グラフ・図解をインラインで埋め込み
  • カラー表現: MarkdownはUnicode文字で色を推測するしかない
  • Base64画像で外部URLに依存しない自己完結型
  • 実装計画のレビュー時間を大幅短縮 ─ 視覚化で素早く判断

長い仕様書もHTMLならPCでは8セクション、スマホでは縦に最適化！

⑦トークンコストの現実

  • Markdown: ~1,140トークン ($0.017)
  • セマンティックHTML: ~2,760トークン ($0.041)
  • フルHTML: ~5,480トークン ($0.082) ─ 100万窓の0.55%に過ぎない
  • 真のコストは生成時間: フルHTMLはMarkdownの5倍
  • 月30回使っても$3未満 ─ Pro/Maxなら固定料金内
100万トークン窓のおかげで、コスト差はほぼ誤差範囲！

⑨セキュリティリスクと対策

  • HTMLにJSが含まれる＝ブラウザで実行されるリスク
  • リフィード攻撃: 前回のHTMLを次のコンテキストに渡すと注入の恐れ
  • 共有レポート: 認証ドメイン内で実行＝Cookie漏洩リスク
  • 対策: サンドボックスサブドメイン or CSP: script-src 'none'
  • 単一ファイル原則: CDN依存なし、オフライン完結が理想
エージェント生成のHTMLは「信頼できない入力」として扱うべし！

分散マルチエージェントシステム ─ AI協調のアーキテクチャ

⑫マルチエージェントとは？

  • 複数の独立したAIエージェントが協調して課題を解決する仕組み
  • 「分散」＝エージェントが別々のコンピュータで動作可能
  • 並列実行で高速化・耐障害性・スケーラビリティを実現
  • 単一LLMが「壁にぶつかる」複雑タスクで真価を発揮
1つのすごいAIより、役割分担したチームの方が強いって話！

⑬オーケストレーション戦略

  • スーパーバイザ型: 中央エージェントがタスク配分・管理
  • ネットワーク型: エージェント同士が直接通信・高い柔軟性
  • プランナー型: 実行時にエージェントを動的選択
  • LoopAgent: 自動フィードバックループで継続改善
  • 階層型スーパーバイザで大規模システムにも対応
用途によって最適な戦略が変わる。正解は一つじゃない！

⑭A2Aプロトコル × MCP

  • A2A(Agent-to-Agent): Google提供のエージェント通信規格
  • メッセージ送信・助け依頼・結果共有の共通フォーマット
  • MCP(Model Context Protocol): データソース統合のオープン規格
  • もろいAPIコードを排除し、エラー処理・キャッシュを標準化
  • Google Cloud Run + A2Aで本番デプロイが可能
A2Aは「エージェント界のEmail」、MCPは「エージェント界のUSB」！

⑮専門エージェントと検証

  • 役割例: Researcher / Writer / Critique / Evaluator
  • 検証コントラクト: 実装前に「完了の定義」を明確化
  • テストスイート / タイプチェックで品質担保
  • ユーザー検証者エージェント = QAエンジニア的役割
  • 敵対的検証で脆弱性やエッジケースを発見
検証が厳しいほど、出力品質が高くなる！大事な設計思想だね

⑯ハーネスこそが鍵

  • エージェントを包む「ハーネス」コードが最も重要で困難
  • テスト / 共有コンテキスト / タスクリスト ─ 全部ハーネスの仕事
  • 永続メモリ: 過去のやり取りを記憶して文脈維持
  • 可観測性: エージェント進捗のモニタリングが必須
  • モデル選択の戦略: 役割に応じて最適なLLMを割り当て
エージェントそのものより、周りの土台作りが9割の労力！

⑰課題と限界

  • 最適アーキテクチャは不存在 ─ ユースケース依存
  • エージェント間の暗黙の前提がバグの温床に
  • 調整コストが増大: 単一エージェントで45%以上成功するなら見直し
  • 構造化ハンドオフ: 長期コンテキストの一貫性維持が難題
  • 本番運用にはマイクロサービス的設計パターンが必要
マルチエージェントにする前に、単体でできるかも確認してね！

⑱2つのトピックの接点

  • HTML出力はエージェントの可観測性ツールとして最適 ─ レポート・ダッシュボード
  • 双方向HTMLはエージェントの意思決定支援に直結
  • A2A + MCPでエージェント間通信を標準化し、HTMLで結果を可視化
  • 分散システムの複雑さをHTMLの情報密度で表現する共通言語に
  • 「単一ファイルHTML」がエージェント出力の最強フォーマットになる未来
