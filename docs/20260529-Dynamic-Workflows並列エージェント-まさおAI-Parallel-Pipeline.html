Claude Code Dynamic Workflows

大量並列エージェントを決定論的に操作する新機能を徹底解説
2026年5月29日

① Opus 4.8と一緒に登場した新機能

  • Opus 4.8と同時にリリースされたClaude Codeの新機能「Dynamic Workflows」
  • Claude Code v2.1.154以降で利用可能なJavaScriptオーケストレーションスクリプト機能
  • 現在はリサーチプレビュー段階。Pro/Max/Team/Enterpriseで対応
  • Proユーザーは/configコマンドで有効化する必要がある
単発のサブエージェントとAgent Teamsの中間を埋める存在として設計されています。

② Dynamic Workflowsとは何か

  • Claudeがタスクに応じてJavaScriptのスクリプトを自動で記述する仕組み
  • バックグラウンドで独立したランタイムがそのスクリプトを実行する
  • スクリプトが大量のサブエージェントを並列で制御して検証を回す
  • 結果を相互にチェックして、最終的に単一の回答に収束させる
「段取りスクリプトをClaude自身が書いて、
その上で何十個ものエージェントが動く」
というイメージです。

③ オーケストレーションの仕組み

  • オーケストレーター（Claude本体）が段取りスクリプトを生成するのが第一歩
  • スクリプトが数十〜数百のサブエージェントを並列で制御・起動する
  • 各エージェントの結果を相互チェック・検証して品質を担保
  • メインセッションの応答性は維持。オーケストレーションはバックグラウンドで動く
Claude本体はあなたとの会話を続けながら、
裏でスクリプトが大量のエージェントを
自律的に動かすのがポイント！

⑤ Parallel — 待たない同時実行

  • Parallelは待たない・同時に並列実行するモード
  • 互いに独立したサブタスクに最適。前の結果を待つ必要がない
  • 例：「ファイルA・B・Cをそれぞれ別アプローチで検証」のような並行作業
  • Parallelなら実行時間は最も遅いタスクに収まるので高速
それぞれが独立して動けるから
スピード重視のタスクに向いています。

⑥ Pipeline — 順次で結果を渡す

  • Pipelineは待つ・順次実行するモード
  • 前段の結果を後段が使う場合に必須の実行方式
  • 例：「リサーチ → 分析 → レポート作成」のように段階的に処理が進む流れ
  • Pipelineは段階ごとに品質が上がるので精度重視のタスクに最適
前の結果が次の入力になるなら
迷わずPipelineを選びましょう。

⑧ トーナメント方式で品質を担保

  • 複数のエージェントが同じ問題に異なるアプローチで取り組む
  • お互いの結果を相互にレビュー・検証する（敵対的レビュー）
  • 最も信頼性の高い回答を自動的に選択する仕組み
  • 複数の視点から計画を策定できるので、盲点を減らせる
トーナメントのようにバトルしながら
品質を担保する仕組み。
1人の天才より5人の审查家。

⑨ Dynamic Workflowsの起動方法

  • プロンプトに「workflow」という言葉を含める
  • /deep-researchコマンドを使う
  • ultracode設定を有効化する
  • /goalコマンドを使うとワークフローが自動トリガーされる
一番手軽なのはプロンプトに
「workflow」と書くだけ。
Claudeが判断して起動します。

⑩ ワークフローの監視と制御

  • /workflowsコマンドでフェーズ・トークン消費・エージェント出力を検査
  • タスクパネルで進捗概要を確認できる
  • 一時停止・再開・再起動・停止などのライブ制御が可能
  • Ctrl+Gでスクリプトを確認・承認できる
Ctrl+Gは緊急ブレーキにもなります。
暴走したら即確認・停止が可能です。

⑪ ワークフローの保存と再利用

  • 生成されたスクリプトは.claude/workflows/に自動保存される
  • 保存したスクリプトは後から再利用可能。同じ作業の繰り返しに便利
  • JavaScriptなので手動で編集してカスタマイズもできる
  • チームでワークフローのテンプレートを共有する運用も現実的
一度作ったワークフローを
資産として蓄積していくのが
この機能の本当の価値です。

⑫ セキュリティ監査に最適

  • コードベース全体を一気に監査できる。数百ファイルを並列でスキャン
  • バグスイープ — あらゆるファイルからバグを探し出して報告
  • 脆弱性パターンを複数エージェントで交差検証して見逃しを防ぐ
  • 単独エージェントより検出率が格段に高いのが最大の強み
セキュリティ監査は盲点が致命傷。
トーナメント方式で
見逃しを大幅に減らせます。

⑬ 大規模マイグレーションを一括処理

  • 500ファイル超の大規模なコードベース変更に対応
  • APIのバージョンアップやライブラリの全面置き換えを並列で実行
  • 変更後のコードを相互に検証して整合性をチェック
  • 手作業なら何日もかかる作業を大幅に短縮できる
「全部のファイルを変えたい」
そんな時にWorkflowの出番です。
手作業では絶対無理なスケール。

⑭ 複数ソースの交差検証調査

  • 複数のドキュメント・コード・ログを同時に読み込んで分析
  • 各ソースを別のエージェントが担当して結果を突き合わせる
  • 矛盾する情報をトーナメントで判定し、最も信頼できる結論に到達
  • 根拠のある調査レポートを短時間で生成できる
1人で全部読むのは時間がかかる。
並列で分担すれば圧倒的に早い。
AI時代の「チーム調査」ですね。

⑮ コストに注意 — トークン消費が激しい

  • トークン消費が著しく多い。並列エージェントの分だけ倍増する
  • 最初は小さいタスクで試すことが推奨されている
  • /usageコマンドで消費をリアルタイムに監視できる
  • /modelで小さいモデルにルーティングしてコストを管理する
力は強力なぶんコストも強力。
使い所を見極めるのが重要。
まずは小さなタスクから！

⑯ JavaScriptスクリプトの構造

  • JavaScriptで書かれたオーケストレーションスクリプト
  • Parallel/Pipelineの呼び出しを宣言的に記述する
  • 中間結果はスクリプト変数に保持。次のフェーズで参照可能
  • エージェントへのプロンプト・タスク・条件を動的に設定できる
// 例: 並列で3つ検証して最良を選ぶ
const results = await parallel([
  { task: "アプローチAで検証" },
  { task: "アプローチBで検証" },
  { task: "アプローチCで検証" }
]);
const best = selectBest(results);
Claudeが自動生成するから
JavaScriptが書けなくてもOK。
手動でカスタマイズもできます。
