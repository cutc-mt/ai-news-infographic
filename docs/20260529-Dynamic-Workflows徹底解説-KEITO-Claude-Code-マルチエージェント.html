Claude Code Dynamic Workflows 徹底解説

マルチエージェント実行の革命 — Claudeが自分でチームを組む
2026年5月29日 | Claude Code v2.1.154以降対応

① Dynamic Workflows とは何か

  • Claude Codeにマルチエージェントのオーケストレーション機能が追加された
  • JavaScriptのスクリプトで実行計画・分岐・中間結果をプログラマティックに管理する仕組み
  • 現在はリサーチプレビュー段階。Claude Code v2.1.154以降で利用可能
  • 従来の単一エージェントから数十〜数百のエージェントを並列実行できる世界へ
Claude Codeが「1人で作業する」から
「チームを組んで作業する」に
進化したと言えます！

② なぜこれがすごいのか

  • 従来のClaude Codeは1つのターンで1つのタスクを処理。Workflowsは複数タスクを並列で同時に進行
  • 途中で条件分岐ができる。エージェントAの結果次第でBかCに振り分ける柔軟性
  • 一時停止・再開・再起動・停止が自由自在。長時間のタスクでも安心
  • 成功したワークフローは保存して再利用可能。何度も実行する作業に最適

③ ワークフローの4つのトリガー方法

  • プロンプトに「workflow」を含める — 最もシンプルな起動方法
  • /deep-research コマンド — 複数ソースの交差検証調査に特化した調査ワークフロー
  • ultracode 設定を有効化 — xhigh推論エフォート＋自動オーケストレーション
  • /goal コマンド — 目標を指定してClaudeに最適な実行計画を立てさせる

④ Subagents vs Skills vs Workflows

  • Subagents — Claudeがターンごとにオーケストレーション。中間結果はコンテキスト内に保持
  • Skills — Claudeがプロンプトに従って実行。既存の拡張機能の仕組み
  • Workflows — スクリプトが実行計画・分岐・中間結果を管理。数十〜数百のエージェント並列実行
  • Subagentsが数人のチームなら、Workflowsは数十人規模の組織を動かすイメージ
小さい作業はSubagents、
大規模な作業はWorkflowsと
使い分けるのがポイント。

⑤ ワークフローの監視と制御

  • /workflows コマンド — フェーズ・トークン・エージェント出力をリアルタイムで検査
  • タスクパネルで進捗の全体像を確認可能。どのフェーズが終わって次が何か一目で分かる
  • 一時停止・再開 — 途中で止めて確認してから再開できる安心感
  • 再起動・停止 — 失敗したフェーズだけやり直すことも可能

⑥ ワークフローの保存と再利用

  • 成功したワークフローは.claude/workflows/ディレクトリに自動保存
  • 保存したスクリプトを別のプロジェクトでも再利用可能。一度作れば何度でも使える
  • チームで共有すれば、ベストプラクティスな実行パターンを組織全体に展開できる
  • JavaScriptで書かれているため、自分でカスタマイズも自由自在

⑦ コード監査のユースケース

  • コードベース全体の監査 — セキュリティの脆弱性をエージェントごとに分担して検出
  • 500ファイル超の大規模マイグレーションを並列で一気に処理。時間を大幅に短縮
  • バグスイープ — 複数エージェントが別々のモジュールを担当してバグを網羅的に洗い出す
  • 1つのエージェントが見落とすバグも、複数の視点から検査すれば見逃しを減らせる

  • セキュリティ監査も並列で。認証・権限・データフローなど項目ごとに専門エージェントを割り当て

⑮ Workflowsの実行の流れ

  • フェーズ1: 計画 — Claudeがタスクを分析してサブタスクに分割し、実行スクリプトを生成
  • フェーズ2: 承認 — ユーザーが Ctrl+G でスクリプトを確認・承認（安全性の担保）
  • フェーズ3: 並列実行 — 複数のエージェントが同時にタスクを処理。進捗を監視
  • フェーズ4: 統合 — 各エージェントの結果をまとめて最終出力を生成
計画→承認→実行→統合の
4フェーズ構成で
安心・安全に大規模実行。

⑯ メリットと注意点のまとめ

  • 圧倒的な並列処理力 — 数十〜数百のエージェントで時間を大幅に短縮
  • 柔軟な分岐制御 — 中間結果に応じて動的にルートを変更できる
  • 中断・再開の自由 — 長時間タスクでも途中で確認できる
  • トークン消費が激しい — コスト管理には /usage と /model が必須
  • リサーチプレビュー — 仕様が変わる可能性がある点に留意
