1000のAIエージェントがマラソンを走る

Google Cloud Nextで披露された「Race Condition」プロジェクトのアーキテクチャを徹底解説
📅 2026年5月31日
🔥 Google Cloud Next 2026 × ⚡ Race Condition × 🎯 ADK

①Race Condition：1000エージェントのマラソンシミュレーション

  • Google Cloud Next 2026の開発者キーノートで披露された大規模マルチエージェントシステム
  • ラスベガスのマラソンコースを舞台に、1000人のAIランナーエージェントがリアルタイムで走るシミュレーション
  • 各ランナーは水分補給レベル速度を独立して管理し、給水所で自分で判断して飲む
  • 全てオープンソースで公開。ローカルマシンで試すことが可能
マップ上のドット1つ1つがAIランナー。それぞれが独立して走っている！

②4層のポリグロットアーキテクチャ

  • Angularフロントエンド：シミュレーションUIと管理ダッシュボードを提供
  • Goゲートウェイ：高並行トラフィックの管理。何千ものWebSocket接続を軽々と処理
  • Python ADK：全てのAIロジックがここに。LLMアクセスと豊富なエコシステムを活用
  • インフラ層：Redis・Pub/Subなどでエージェント間の通信を支える
各層に最適な言語を選ぶ「ベストツール・フォー・ジョブ」の哲学！

③2つのディスパッチモード：Subscriber と Callable

  • Subscriber（Redis Pub/Sub）：ランナーエージェント専用。サブミリ秒で応答可能
  • Callable（HTTP）：プランナー・シミュレーター向け。標準のA2Aプロトコルで通信
  • シミュレーターがティックメッセージをブロードキャストすると、全ランナーが同時に起動して応答
  • Callableはスケールダウン可能。アイドル時はゼロにスケールしてコストを節約
思考系エージェントはHTTP、実行系はPub/Sub。使い分けがポイント！

④Goゲートウェイ：メッセージパッシングの中核

  • 全てのフロントエンドとエージェントの間に位置する中央通信ハブ
  • エージェントのAgent Cardを読み取り、ディスパッチモードを自動判別して適切にルーティング
  • Goの並行処理能力により、数千のWebSocket接続とメッセージングを効率的に処理
  • BFF（Backend for Frontend）パターンを採用。Cloud Run上で実行される
ゲートウェイのルーティング
├ Agent Cardを読み取り
├ dispatch = "subscriber" → Redis Pub/Sub
└ dispatch = "callable"  → A2A HTTP

⑤A2Aプロトコルとエージェントのメッシュネットワーク

  • Agent-to-Agent（A2A）プロトコルを活用。エージェント同士が互いを発見・通信する
  • 指向性グラフではなく、メッシュネットワーク。どのエージェントも他のどのエージェントとも通信可能
  • ハードコードされた設定ではなく、動的ディスカバリーでエージェントを見つける
  • コールドスタート時の自動リトライ機能も内蔵。通信が確実に届く仕組み
エージェント同士が自由に会話できるメッシュ型。柔軟性が高い！

⑥Agent Card：エージェントの身分証

  • Agent Cardはエージェントの「名刺」のようなもの。名前・説明・スキル・URLが含まれる
  • A2Aプロトコルの拡張機能としてディスパッチモードを定義可能
  • プランナー：callable。ランナー：subscriber。Cardを見れば通信方式が分かる
  • スキル情報も定義。「マラソンルートの計画」「シミュレーション実行」などを記載
{
  "name": "planner-agent",
  "description": "マラソンルート・イベント計画の専門家",
  "dispatch": "callable",
  "skills": ["plan_marathon_route", "report_route"]
}

⑦レジリエントモデル：自動リトライとバックオフ

  • 429エラー（クオータ超過）やネットワーク障害が発生しても自動でリトライ
  • 指数的バックオフ＋ジッターを搭載。リトライ間隔をランダムに変動させ、一斉アクセスを回避
  • GoogleのGen AI クライアントライブラリの機能を活用してADKに統合
  • これがデモがスムーズで安定している理由の一つだとのこと
デモが安定して見えたのは、裏で堅牢なエラーハンドリングが動いていたから！

⑧ADKのスキルシステム：遅延読み込み

  • スキルの遅延読み込み（Lazy Loading）でコンテキストウィンドウを効率管理
  • プロンプトに「必要な時だけスキルをロード」と指示するだけでOK
  • GIS空間エンジニアリングスキル：マラソンルートの計画に使用
  • 各スキルはskill.markdownにメタデータ・説明・関連ツールを定義
必要な道具だけを使う。無駄がないから高速でコストも安い！

⑨プランナーエージェント：マラソンルートの設計

  • マラソンルートとイベント計画の専門アナリストとして設計されたエージェント
  • 計画プロセス：GISスキル読み込み→ルート計画→レポート作成
  • Google Maps MCPが利用可能なら天候データも取得して最適な開始時刻を検討
  • MCPはAPIキーがオプション。持っていなくても動く親切設計
プランナーの計画フロー
├ ステップ1: GISスキルをロード
├ ステップ2: ルートを計画
├ ステップ3: レポートを作成
└ ステップ4: Maps MCPがあれば天候を確認

⑩シミュレーターエージェント：ティック駆動の実行

  • シミュレーションの各ティック（刻み）ごとにRedisにメッセージをブロードキャスト
  • 全1000人のランナーが同時にメッセージを受信し、各自の状態を更新
  • 位置・速度・水分補給レベルなど、各ランナーのティレメトリを追跡
  • _callable_モードで動作。使用頻度が低いためゼロスケール可能
ティックごとに全員に一斉送信。Pub/Subだからこそ可能なリアルタイム性！

⑪ランナーエージェント：独立した意思決定

  • 各ランナーは独立したセッションを持ち、自身の状態を自律的に管理
  • 給水所を通過する時に自分で水分補給するか判断。これがエネルギーに影響する
  • ペースは個々に異なる。ウォーキングジョギング全力疾走など多様な走法
  • Cloud Runにデプロイ。常時ホットインスタンスを維持して即応性を確保

⑫Redis Pub/Subによる超高速通信

  • Google CloudのMemory Store（マネージドRedis）のPub/Subを活用
  • 1つのノードで約10万件の同時メッセージを処理可能。本プロジェクトには十分すぎる性能
  • HTTPで1000件同時送信するとネットワークオーバーヘッドが問題に。これがPub/Sub移行の決定打
  • 全ランナーの応答を約10秒以内に収集する必要がある。ボトルネックは「ネットワーク」
HTTPの限界に気づいてRedis Pub/Subに切り替えた。いい判断！

⑬ADKエージェントの構造：レジリエントモデルの実装

  • ADKのLlmAgentクラスでエージェントを定義。名前・モデル・説明・命令を設定
  • static_instructionsはキャッシュ可能。全セッションで共通の命令を効率よく送信
  • utils/retry.pyにレジリエントモデルの実装。最大試行回数・初期遅延・ジッター等を設定
  • global_gemini.pyでプレビューモデルに対応。エージェントランタイムでも最新モデルを使える
レジリエントモデルの設定例
max_attempts: 5
initial_delay: 1.0秒
max_delay: 30.0秒
jitter: true
backoff_base: 2（指数的バックオフ）

⑭call_agent：エージェント間通信の統一インターフェース

  • call_agentツールが全エージェント間通信の単一エントリーポイント
  • プランナーはシミュレーターの場所を知らなくて良い。名前で指定するだけ
  • 失敗時の自動リトライ、ティレメトリの送信、A2Aディスカバリーも自動で処理
  • A2Aプロトコルをラップしてネットワークの複雑さを隠蔽
プランナーは「シミュレーターを呼んで」と言うだけ。後は全部自動！

⑮なぜ手続き型ではなくエージェントにしたのか

  • 手続き型コードでもシミュレーションは作れるが、LLM周辺のライフサイクル管理が必要
  • ADKがセッション管理コールバックツール呼び出しの解釈ティレメトリを無料で提供
  • 分散システムの配管（プラミング）をゼロから書かなくて済む
  • 開発期間が数ヶ月から数週間に短縮。開発者は知能の部分に100%集中できる
月数 → 週数

⑯Cloud Run vs Agent Runtime：デプロイ戦略

  • ランナーエージェントはCloud Runにデプロイ。スケーラビリティを細かく制御
  • Cloud Runは並行数設定ホット/コールドインスタンス数など調整ノブが豊富
  • プランナー・シミュレーターはAgent Runtimeにデプロイ。ゼロスケールでコスト最適化
  • 用途に応じて適切なインフラを選択する柔軟な設計
高頻度通信はCloud Run、低頻度はAgent Runtime。適材適所！

⑰拡張ビジョン：より多くのエージェントの可能性

  • 公共安全エージェント：ランナーの安全確保や交通規制のシミュレーション
  • ビジネスエージェント：マラソンの影響を受ける店舗をモデル化（ドライブスルーvsカフェ）
  • ランナーに個性とアイデンティティを付与。ルール遵守者とチーターなど多様な行動
  • 監視エージェント：不正行為を検知するシステム。予測不能なLLMの振る舞いを活用
LLMだからこそ生まれる予測不能な面白い振る舞い。映画みたいだ！
