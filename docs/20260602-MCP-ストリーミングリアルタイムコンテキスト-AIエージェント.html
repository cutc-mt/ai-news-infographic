Section 1: MCPの現在と進化

① MCPとは — AIと外部世界をつなぐUSB端子

  • Anthropicが2024年11月に導入したオープン標準
  • 2025年12月にLinux FoundationのAAIF（Agentic AI Foundation）に寄贈
  • AIエージェントと外部ツール・データソースの接続を標準化
  • 「N×M問題」を解決 — 各ツールごとのカスタムコネクタが不要に
  • 「AIの世界のUSB端子」— プラグアンドプレイでツール接続
MCPがあれば、AIはどんなツールにも
標準手順で接続できる。
カスタム連携開発が過去のものに！

② 2026年の普及状況 — 全社採用のde facto standard

  • Anthropic・OpenAI・Google・Microsoft・Amazonの全主要企業が採用
  • 月間数百万SDKダウンロード — 急成長を記録
  • 企業のAIエージェント統合におけるde facto standardに
  • Python・TypeScript・Goなど主要言語で公式SDKが利用可能
  • オープンソースのMCPサーバーが数千のツール統合を提供
2024年の誕生からわずか1年半で
業界標準に。
Linux Foundationへの寄贈が決定打！

③ N×M問題の解決 — 連携手工の終焉

  • 従来: N個のAIモデル × M個のツール = N×M個のカスタム連携が必要
  • MCP導入後: ツール側に1つのMCPサーバーを用意するだけで全AI対応
  • 開発者は個別のAPIラッパーを書かずに済むようになった
  • これまで数週間かかった連携開発が数時間に短縮
  • 「AI側」ではなく「ツール側」に統合の責任が移るパラダイム
N×M → N+Mに。
拡張性の革命だ。
新しいツールを追加するだけで、
全てのAIエージェントが利用可能に！

④ MCPがもたらすビジネスインパクト

  • オープン標準によるベンダーロックイン回避 — どのAIモデルにも切り替え可能
  • AIツールチェーンの構築スピードが劇的に向上
  • エコシステムの形成 — コミュニティ主導でツール統合が爆発的に増加
  • ガバナンス・セキュリティがプロトコルレベルで統一
  • 企業は社内データへのAIアクセスを安全かつ標準的に提供可能に
MCPの真の価値は「接続」じゃなくて
「標準化」にある。
世界中のツールが同じ言語を話す！

Section 2: 従来のRequest-Responseの限界

⑤ 古いパラダイム — リクエスト・レスポンスの限界

  • ユーザーがプロンプトを送信 → AIが応答。1回きりの対話
  • バッチ処理中心のデータ → 古い情報に基づく応答でハルシネーション発生
  • ステートレス — 会話の文脈が途切れ、継続的なタスクが不可能
  • 単一エージェント — 複数エージェントの協調や複雑なワークフローに不向き
  • リアルタイム性なし — 動的に変化する環境に対応できない
「聞かれたことだけ答える」AI。
今起きていることに
全く気づいていない。

⑥ データの鮮度問題 — AIの最大の弱点

  • RAGの限界: インデックスされたデータは取得時点で既に古い
  • 在庫数が変化 → 注文が入る → 顧客がクレーム。AIは気づかない
  • ハルシネーションの根本原因の多くは情報の古さ
  • バッチ更新を頻繁にしてもミリ秒レベルの鮮度は不可能
  • リアルタイムビジネスでは「数分遅れのデータ」は致命的
AIが「賢い」だけではダメ。
「今この瞬間」のデータが
なきゃ正しい判断ができない！

⑦ ステートレスの壁 — 持続的なタスク実行が不可能

  • リクエストごとにコンテキストがリセット — 継続タスクが断絶
  • 「在庫を監視して閾値を下回ったら発注」のような継続的タスクを実装不能
  • 複数人での共同作業や段階的承認フローにも対応できない
  • サーバー側にセッション状態を保持する仕組みが必要
  • → これこそがStreamable HTTP Transportが解決する問題
AIが「記憶」と「継続性」を持てば
監視・判断・実行の
ループを自律的に回せる！

Section 3: ストリーミング・リアルタイム・コンテキスト

⑧ リアルタイムコンテキストエンジンとは

  • DeltaStream RTContext / Confluent Real-Time Context Engine
  • ミリ秒レイテンシで運用データ・状態・シグナルをAIに直接ストリーム
  • 在庫・注文・顧客情報・システムイベントなどをライブで提供
  • AIが「今この瞬間」の状況に基づいて意思決定できるようになる
  • イベントストリーム → コンテキストエンジン → AIエージェントのデータフロー
「AIに質問する」のではなく
リアルタイムデータが
AIに「今の状況」を教える。
受動から能動へ！

⑨ Streamable HTTP Transport — MCPの次世代トランスポート

  • MCPの新しいトランスポート層 — 長時間接続をサポート
  • プログレス更新をストリーム — 長時間実行タスクの進捗をリアルタイム通知
  • セッション管理でステートフルな対話を実現 — 文脈が途切れない
  • Elicitation機能: センシティブ操作の前に人間の承認を求める仕組み
  • 双方向通信: サーバーからクライアントへのプッシュも可能
HTTPベースだからファイアウォールも
通りやすい。
企業導入の大きなアドバンテージ！

⑩ コンテキストエンジニアリングの台頭

  • プロンプトエンジニアリング → RAG → コンテキストエンジニアリングへ進化
  • AIへのリッチでガバナンスされたリアルタイムデータ提供が基盤
  • データの質・鮮度・ガバナンスがAI性能の決定要因に
  • 「どんなデータを・いつ・どの形式で」AIに渡すかが新たな職種スキル
  • プロンプトの最適化よりコンテキストの設計の方が遥かにインパクトが大きい
「プロンプト職人」は過去の職業。
次は「コンテキスト設計者」が
AIの性能を握る！

⑪ Elicitation — AIと人間の協調の新たなパターン

  • AIがセンシティブ操作を実行する前に人間の承認を求める仕組み
  • 対話型の承認フロー — 単なるYes/Noではなく、条件付き承認も可能
  • ガバナンスと自律性のバランスを取るための重要な概念
  • 金額の閾値・対象範囲などで自動/手動を動的に切り替え
  • 「AIに任せすぎる」問題への実用的な解決策
AIが勝手に動くのは怖い？
大事な時だけ人間に聞く。
これがElicitationの考案。

Section 4: 3層プロトコルスタック

⑬ 2026年のAIエージェント通信 — 3層プロトコルスタック

Layer 3: WebMCP — 構造化ウェブアクセス
Layer 2: A2A (Agent-to-Agent) — マルチエージェント協調
Layer 1: MCP — エージェント ↔ ツール通信
  • MCP: AIと外部ツール・データの標準インターフェース
  • A2A: エージェント同士が自律的に協調・交渉するプロトコル
  • WebMCP: ウェブ上の構造化データにAIが直接アクセス
3つの層が積み重なって
AIエージェントの通信インフラを
完成させる！

⑭ A2Aプロトコル — エージェント同士の自律協調

  • Agent-to-Agent: Googleが提唱するエージェント間通信の標準
  • エージェント同士が能力・可用性を宣言し、自律的にタスクを割り振り
  • 人間の仲介なしでネゴシエーションやタスク委譲を実行
  • マルチエージェントシステムの実装を劇的に簡略化
  • MCP（Layer 1）と組み合わせて完全なエージェント生態系を形成
「Aさんにお願いして、Bさんに伝えて…」
もう人間が橋渡し不要。
AI同士で勝手にチーム組む！

⑮ ハイブリッドアーキテクチャ — 確定性×非確定性の融合

  • 非確定性入力をAIが解釈 → コンテキストに応じた意思決定
  • 従来の確定的APIアクションと相補 — 両者の長所を活用
  • AIが判断し、確定的システムが実行するハイブリッド型
  • 金融取引・注文処理などは確定的APIで安全に実行
  • 次世代AIシステムは両アプローチを統合したアーキテクチャが主流に
「AIだけ」でも「APIだけ」でもダメ。
AIが判断してAPIが実行。
この融合が鍵！

Section 5: リアルタイムエージェントのユースケース

⑯ 自律的ビジネスエージェント — リアルタイム意思決定

  • 例: 価格変動を検知 → 自動で発注・価格調整を実行
  • 24時間365日稼働の自律型ビジネスオペレーター
