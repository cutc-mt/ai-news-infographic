① AIエージェントは「Web操作」で壁にぶつかる

  • Claude Code、Cursor、Codex、Gemini CLI — どれもコードを書くのは得意だが、Webサイトを操作するのは苦手
  • ログイン、フィルター操作、データ収集、フォーム入力、エクスポート — これらが要求されると止まる
  • 現代のWebはもはやHTMLだけではない。ログイン壁、JavaScriptの重いページ、ポップアップ、無限スクロール、多段階フローがある
  • ブラウザのフィンガープリント、行動パターン、CAPTCHAまで検出される時代に、シンプルなスクレイピングは通用しない
WebFetchは「テキストをリクエスト」するだけ。
ログインやJSが必要なページでは即座に失敗する。

② BrowserAct — AIエージェントに「手」を与える層

  • 無料・オープンソースのブラウザ制御レイヤー。Claude CodeやCursorの下に敷いて使う
  • エージェントが実際のブラウザを開いて、ナビゲート、クリック、タイピング、データ抽出、スクリーンショットを人間のように実行できる
  • AIそのものを置き換えるものではない。エージェントの「頭脳」はそのままで、「手」を追加するイメージ
  • VS Codeのターミナルから1つのインストールコマンドで導入可能。エージェントのプロンプトに組み込んで使う

⑪ リージョン別プロキシ — 地域ごとの正確なデータ取得

  • 地域固有のプロキシIPを使えば、その地域のローカルユーザーとしてサイトにアクセスできる
  • USの顧客には20%割引が見えるが、UKの顧客には異なる価格や送料メッセージ、別のプロモコードが表示されることがある
  • 1つのIPからチェックすると本当の地域別オファーが見えない問題を解決
  • これによりリージョン対応の競合調査が可能に。単一地域のスクレイプでは絶対に見つけられないインサイトを得られる

⑭ WebFetchの限界 — なぜブラウザ制御が必要か

  • WebFetchは基本的に「ページのテキストをリクエスト」するだけ。以下のケースで壊れる
  • ログインが必要なページ → アクセス不可
  • JavaScriptの描画が必要なページ → コンテンツが空白
  • ユーザー操作が必要なページ → クリックやフィルター操作ができない
  • 動的読み込みのページ → ロード前に取得してしまう
現代のWebサイトのほとんどが
これらの条件の少なくとも1つを
満たしている。WebFetchだけでは不十分。
