② 開発の全工程 — 素材準備から完成まで

  • 素材準備 — AIでスプライトシート、背景、タイル、敵キャラの画像を生成
  • プランニング — Opus 4.8に全体設計と開発計画を任せる
  • 実装・コーディング — 当初はSonnetで試したが画像処理で行き詰まり、Opusに切り替え
  • テストプレイ — 完成版を全ステージプレイして動作確認

③ 画像AIで素材を一括生成 — スプライトシート方式

  • 今回はAbacus AIのエージェントを使用。NanoBanana 2やGPT Imageでも代用可能
  • スプライトシートを作成。ラスボス、プレイヤーキャラ、敵キャラ（ゾンビ・レッドデビル・ラーデーモン）などを1枚画像に
  • プレイヤーキャラはダークナイトとダークメイジの2種類を用意
  • 背景タイル、樹木オブジェクト、草のパターンなど、ステージ構築用の多彩な素材を生成
自分のイメージに合わせて
仕上げてくれる画像AIなら
どれでもOKです。

④ AIにとって最大の壁 — スプライトの切り出し処理

  • スプライトシートからピクセル単位の外科手術のような切り出しが必要。AIにとって非常に難易度が高い作業
  • ジャンプアクションの切り取り範囲が微妙に異なる。文字が被るリスクもあるため、正方形に切れない場合もある
  • ドラゴンは右向きしかないので反転が必要。ガーゴイルは24×24でテキスト要素が混入
  • フレーム数もバラバラ（2フレームや4フレーム）。AIの画像認識能力とピンポイントの微調整が同時に求められる
プロンプトエンジニアリングで
どうにもならない部分。
モデルの能力を頼るしかない。

⑤ Sonnetでは崩壊 — Opusにしかできなかった作業

  • 当初はプランニングをOpus、コーディングをSonnetで分業する予定だった
  • しかしSonnetにスプライトの切り出しをやらせるとぐちゃぐちゃになって崩壊
  • 仕方なくOpusに引き継ぎ。Opusは綺麗に処理を完遂してくれた
  • 結論：ソネットじゃ全然話にならないレベルの仕事だった。画像処理の精度で圧倒的な差が出た
「なんでこんなに手こずってるんだ？」
と思っていたら、スプライト処理が
AIにとって鬼門だった。

⑥ ステージ構築の罠 — 分割素材 vs 巻き物方式

  • 分割したタイルやオブジェクトを組み立ててステージを作らせると破綻する。素材の渡し方が肝
  • 巻き物方式が最適解。横長の背景画像を1枚渡して、くるくる回すだけでそれっぽく動く
  • 各ステージごとに背景画像を1枚ずつ用意する方法が、AIにとって最も扱いやすい
  • Rosebud AIなどのゲーム開発プラットフォームも1素材1ファイル方式を採用。余計な作業をさせない設計
AIに組み立て作業をさせないことが
成功の鍵。
1枚画像で済ませるのが正解。

⑦ Opus 4.8の印象 — 失敗しない、頼れるエンジニア

  • 失敗しないのが最大の特徴。不具合修正よりも、イメージと違った部分のバランス調整がメインに
  • 「これとこれとこれを直して、ついでにこの機能も追加して」みたいに4〜6個のタスクをまとめて渡すと整理して順次実行
  • 全て最小ステップで完結してくれる。途中で破綻することがない
  • キャラが白くチカチカするなどの不具合も、原因を特定して修正してくれた
さすがにOpusすごいなと
思いました。
本当にこいつ、失敗しない。

⑧ 4.7から4.8への進化 — より少ないステップで解決

  • 4.7から4.8へのアップデートで、より少ないステップで問題解決に至る能力が向上
  • 余計な手戻りが減り、初回のアプローチで正解に近づく確率が上がっている
  • 特に画像認識の精度が向上。スプライトのピクセル単位処理が4.7より格段に安定
  • 複数タスクの整理能力も进化。渡された順序を最適化して効率的に処理する
少ないステップで解決＝
コスト削減にも直結。
トークン消費が減るのでお財布に優しい。

⑨ コスト管理と運用 — 安いプランでも十分回せる

  • 使用しているのは一番安いプラン。Opusだから高コストだと思われがちだが、工夫すれば問題ない
  • Anthropicが利用枠を2倍に拡大。以前よりかなり多くのボリュームが使えるようになった
  • うまく運用すれば1〜2時間連続で回せる。頻繁に制限が来る印象はない
  • 惜しみなくバンバン使っても、実用上の支障はほぼないレベル
Opusは高級なイメージがあるけど、
最安プランでも実用的。
Anthropicの枠拡大に感謝！

⑩ 無茶ぶりの正体 — スプライトシートをそのまま渡す

  • 今回の開発アプローチは無茶ぶりだったと振り返っている。スプライトシートをそのまま渡してしまった
  • Opusならそのままでもやってくれた。能力が高いからこその結果
  • しかし他のモデルだと絶対にうまくいかないアプローチ。素材の渡し方が開発成功の鍵
  • 正攻法は1素材1ファイル方式。ボスならボスの画像2〜3枚だけを渡すのが鉄則
Opusの能力が高すぎて
自分が無茶ぶりしてたことに
気づいてなかった。

⑪ 他モデルでも成功させる正攻法 — 素材の渡し方

  • 1素材1ファイルで渡す。ボスキャラならボスの画像を通常パターンと火を吹くパターンの2枚
  • プレイヤーキャラも同様。実際に使うアクションだけを個別の画像として切り出してから渡す
  • ステージ背景は巻き物方式で1枚の横長画像。くるくる回すだけで動く
  • Antigravity + Gemini 3.5 Flashでも同等のゲームが作れた。素材の渡し方次第でロジックの難易度は下がる
素材準備の手間を人間がかけるほど
AIは安定して動く。
作業を振り分けるのが大事です。

⑫ ゲーム「Grave Reaper」— 横スクロールアクション

  • プレイヤーキャラはダークナイトかダークメイジを選択可能

  • 残りのTODO：ボスごとの見た目変更、バランス調整、HPバー、ボス専用BGM、効果音追加など

⑬ セッション管理のコツ — 適切なリスタート

  • コンテキストウィンドウの管理は重要。セッションログが長くなると圧縕が必要だが、リスタートが一番確実
  • 霧の良いところでセッションをリスタートするのがベストプラクティス
  • リスタートの際にハンドオフドキュメントを作成しておくと、次セッションがスムーズに始まる
  • ログの圧縮機能もあるが、重要な情報が落ちるリスクがある。ドキュメントベースの引き継ぎが安全
バイブコーディング初心者ほど
セッションの切り方を
意識すると効率が上がります。

⑭ ハンドオフドキュメント — 1行の指示で完璧に作成

  • 「次のセッションで開発をスムーズに始められるように、重要なポイントをプロジェクトファイルに保存して」の1行で十分
  • Opusがプロジェクトフォルダ内にHANDOFF.mdを自動生成してくれる
  • ファイルにはこれまでの経緯、現状のステータス、今後の注意点がまとまる
  • 次セッションでは「続きやろうか」の一言で、TODOリストを読み込んで即座に開発を再開
ハンドオフ.mdがあるおかげで
「続きやろうか」だけで
前回の続きから再開できる。

⑮ 実際のハンドオフ手順 — 3ステップで完結

  • セッション終了前に「ハンドオフドキュメントを作って」と指示。プロジェクト内にHANDOFF.mdが生成される
  • 次セッションを開始して「Grave Reaperの開発作業を再開します」と一言伝える
  • Opusがハンドオフ.mdを読み込み、「プロジェクトの状態を確認しました。V12時点で以下のTODOが残っています」と即座に状況把握
彼らは賢いので、
ドキュメントがあれば適切な
行動を取ってくれます。

⑯ Opusの立ち位置 — 最後の切り札

  • 普段はGemini 3.5 FlashやAntigravityで十分なゲームも作れる。ロジック的には難しくない

  • 高度な画像認識、ピクセル単位の微調整、複雑なタスクの整理が必要な場面でOpusは唯一の解

⑰ 実践的まとめ — Claude Codeでゲームを作る5原則

  • 素材は1素材1ファイルで渡す。スプライトシート方式はOpus以外では破綻する
  • 背景は巻き物方式で1枚画像。AIに組み立てさせない
  • プランニングはOpusに任せる。全体設計の質が段違い
  • タスクはまとめて渡す。4〜6個の指示を一度に渡すと整理して実行してくれる
  • ハンドオフ.mdを必ず作成。セッションの切り方が開発効率を左右する
この5つを守れば、
Claude Codeで安定したゲーム開発が
可能です。
