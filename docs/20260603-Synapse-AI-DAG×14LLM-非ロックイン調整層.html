非ロックイン調整層で業務AIを本番に乗せる時代が来た
2026年6月3日

  • 48回のリリースと535コミット。OSSとしては異常な活発さ

② Synapse AIってそもそも何?

  • ワンライナーで言うと「決定論DAGマルチLLM対応のローカルファースト・オーケストレーター」
  • 公式スローガン: Build AI agents that actually work（実際に仕事をするエージェントを作れ）
  • 公式のキャッチフレーズ: Frozen Exact Path。あなたが設計したパターンの通りに走る
  • Twitterではキラキラなのに社内に入れたら3日で詰まる — そんなPoCの墓場問題を解決するポジション
「すごいけど業務じゃ動かないAI」を 実際に本番運用に乗せるための調整層です。

③ 開発元と技術スタック

  • GitHub組織名はSynapseOKAI。オープンな開発で数字もすべて見える状態
  • 言語構成: Python 53% + TypeScript 43%のハイブリッド構成
  • バックエンドはPython、フロントエンドはTypeScriptで明確に分かれている珍しい設計
  • 配布手段: Docker / npm / pipの全てに対応。社内の標準スタックに合わせて選べる
OSSなのにエンタープライズ目線がある。 守備範囲が広い設計ですね。

④ なぜ「調整層」として注目されるのか

  • 前回の動画で「核エージェントは強いが、それを束ねて業務に流す仕組みがない」という課題を提起
  • その束ねる層を独自の世界観なしで自由にやろうというのがSynapse AIのポジション
  • Synapse AI自体は線路を引く側。何を走らせるかは利用者次第という分離設計
  • LangGraphは線路＋車両セット販売、CrewAIは車両だけ自由に選びたい層。線路だけ作るポジションはずっと放置されていた
ノンロックイン思想で14のLLMに HITL付きで実現。
意外と開いていたポジションを狙い撃ち。

⑤ 軸1: 決定論DAG — 脱線しない電車

  • DAG = Directed Acyclic Graph（有効非巡回グラフ）。矢印が逆流しないネットワーク
  • あらかじめ設計した線路の上しか走らない電車。同じ入力なら同じ経路・同じ出力が保証される
  • 自動車で言えばガードレールの内側しか走れないように物理的に制約する設計
  • 業務システムなら「3回やったら3回とも同じ手順で動いてほしい」。監査で崩壊しないことが担保される
決済APIに勝手なリクエストを 飛ばされたら泣くしかない。
ガードレールが経営への説明になる！

⑥ 決定論 vs ReAct — 安全面の違い

  • ReAct系は毎回その場で次の手を考えるスタイル。柔軟性が売りだが、出発するまでどこに行くか分からない
  • いわばヒッチハイカー。たまにいい寄り道してくれる時もあれば、突然砂漠に向かい始めることもある
  • ハルシネーション経路の問題。AIが妄想で別の道を作ってしまう。金融系なら一発でアウト
  • 決定論DAGは矢印が引かれていないところには物理的に行けない。理由がないからこそ安全という構造
ガードレール＋非常停止ボタン＋ 運転士のレビュー = 3重の安心構造。 社内倫理審査も通しやすい。

⑦ 軸2: マルチLLMオーケストレーション

  • ノードごとにLLMを変えられる。ルーティング処理は爆速のHaiku、文章生成は重めのOpusみたいな割り当てが可能
  • まるで野球の継投システム。先発・中継ぎ・抑えで役割とモデルを分けられる
  • 対応プロバイダー14個以上: OpenAI / Anthropic / Google / Gemini / xAI / Groq / AWS Bedrock / Ollamaなど
  • プロバイダー間のAPI差異はSynapse側が吸収。新モデルが出たらLLM指定だけ書き換えれば移行完了
LLMの世代交代が早すぎる時代に めっちゃフィット。
コード書き換えの地獄から解放。

⑧ 軸3: ローカルファーストとノンロックイン

  • Ollamaと組み合わせればネット切断状態でも100%ローカルで動かせる。社秘データを触る現場に刺さる
  • ハイブリッド構成が同じDAGの中で組める。機密書類の処理はOllama、最終確認だけAnthropicにするなど
  • バックインなし = 独自の流儀を覚えないとかけない。LangGraphのような学習コストの壁が低い
  • 抜けたくなったら抜ける自由がある。ノードに何を置くかは完全に利用者の自由
中身は外に出さないけど品質だけ 任せる。上層が頷く構成が AWS Bedrock対応で実現。

⑨ HITL — 人間の介入ポイントを設計

  • HITL = ワークフローを途中で一時停止して人間がレビューし、OKなら再開する仕組み
  • ベルトコンベアの途中に品質チェック担当者がいるイメージ。完全自動と完全手動の中間を自然に設計できる
  • Synapse AIではHITLが標準機能として組み込まれている。後付けのレビュー画面ではなく、最初から止める前提で組める
  • 決定論DAGがガードレール、HITLが非常停止ボタンという関係。エージェントの暴走を物理的に止める安全装置
最初からHITLを入れた状態で 組み始められる。
設計し直しが不要なのが大きい。

⑩ 承認チャネル — 普段使う場所で承認

  • 承認手段としてUI / Slack / Telegramが用意されている。普段使っているツールで承認できる
  • ワークフローが止まった瞬間にSlackに通知 → ボタン押せば再開という体系が構築可能
  • 通知チャネルも幅広くDiscord / Microsoft Teamsもサポート。日系大企業案件で聞きそうなラインナップ
  • WhatsAppが入ってるあたり、海外の現場運用も意識。現場の人にWhatsAppで承認依頼を飛ばすのも普通に組める
出張先にスマホからポチっと。 モバイル承認が 日常の延長でできる。

⑪ REST API × Cron × Webhook — 業務システムと連携

  • REST APIを公開することで、社内の業務システムからエンドポイントを叩いてSynapse AIのDAGを起動できる
  • 内蔵のCron機能で決まった時間にワークフローを自動起動するのも標準サポート
  • 逆方向も対応。DAGの中から外部APIを叩くノードもWebhookもOK。双方向で既存システムに繋がる
  • Synapse AIは閉じたAIツールではなく業務システムの部品として振る舞える設計。エンタープライズ目線で入り込みやすい
PoCで動いたものが半年眠るリポジ トリに溜まる……。これが墓場問題。 Synapse AIは最初から本番の部品が揃う。

⑫ PoCから本番へのハードルを下げる

  • DAGで設計 → HITLで審査 → Cronで自動起動 → REST APIで連携 → 通知で可視化。フルセットが揃っている
  • 最初から本番想定の設計ができる。PoCが終わってから設計し直す手間がゼロ
  • 「AIがすごいけど業務じゃ動かない」から「AIが実際に仕事をする」への橋渡し役
  • DAG + HITL + Cron + REST + 通知。これらがPoCの墓場から脱出するキットとして機能する
未マージのPRブランチが30本も 眠ってる……。
PoCの墓場脱出の候補が やっと見つかったかもしれない。

⑬ ユースケース1: コンテンツライティングの自動化

  • フロー: 調査 → 執筆 → レビュー → Google Docsに共有まで1本のDAGで通す
  • 途中にHITLを入れて構成案だけ人間が叩く運用も可能。AIが書いたものを最終チェックしてから共有
  • 公開先がGoogle Docsなのも面白い。社内レビューやコメント機能とそのまま接続できる
  • 完全に既存のオフィスワーカーの作業の延長線として組まれている設計
調査だけAI、書くのは人間とか 中途半端なやつじゃなくて、
最初から最後までDAGで繋ぐ。

⑭ ユースケース2: HITL付き自立コード開発

  • HITL付きマルチエージェントが連携して、リポジトリに来るクエストを作成するところまで自動実行
  • マージは人間。ここがHITLとして組み込まれているポイント。自動生成されたPRをレビューしてマージするだけ
  • Claude Codeは個人の手元の相棒。Synapse AIのコードDAGはチームの開発工程の一部
  • 個人のリスクと組織のリスクは違うという設計で表現。ガードレールとしての決定論とHITLが両方必須
一人で動かすのではなく ラインに組み込むイメージ。
レイヤーが全然違う。

⑮ ユースケース3: AI Builder — 自然言語からDAG生成

  • AI Builderは自然言語からDAGを設計するメタエージェント。DAGを作るためのDAGみたいな存在
  • 「こういうワークフローが欲しい」を日本語で書くだけで叩き台が生成される。DSLすら書かなくていい
  • LLMの配置までAIがやってくれる。例: 問い合わせメールの仕分け・テンプレ作成・人間への確認送信など
  • 出来上がったものはコードでもグラフでも編集できる。AIに叩き台を作らせて人間が直すが普通になるかも
これからはツールの書き方を 覚えるのではなく、
AIに設計させて直すが普通に。 前提が変わるやつです。

⑯ 競合との住み分け — 対立ではなく地図を描く

  • LangGraph: エコシステムの規模は頭1つ抜けている。Chainの資産が丸ごと使える。ただし構文が独特で学習コストが高い
  • CrewAI: 役割ベースで組む。プロトタイプの爆速さはリーチだが、決定論で動かしたい業務用途では制御しにくい
  • Hermes: 自己改善のエージェント。失敗を記憶して次は同じ失敗をしない。個体の強さを上げる方向
  • OpenManus: エージェントの記憶レイヤーにフォーカス。コールドスタート解消。脳に外付けメモリを足すイメージ
どれが勝つかじゃなく どう住み分けるか。
戦争じゃなく共存が鍵です。

⑰ Synapse AIの立ち位置 — 工場のレイアウト図

  • Synapse AIはHermesやOpenManusの競合ではなく、それらをノードとして抱える調整層
  • 工場で言うとSynapse AIはレイアウト図と配線設計。各エージェントは個別の機械
  • 個別の機械を磨くフェーズから工場全体を設計するフェーズへ。単体エージェント時代は終わり、層の時代が来る
  • LangGraphで個別エージェントを書きつつSynapse AIで束ねる組み合わせもあり。共存がエージェント後半戦の合言葉
「LangGraphもCrewAIもいらない」の 答えは使い分ければ全部いる。 層の高さが違うから直接競合じゃない。

⑱ 強み・弱点・始め方 — 30分で試せる

  • 強み: 決定論×マルチLLM×HITLの3重構造。POCから本番への移行パスが最初から用意されている
  • 弱点: スター数がまだ少ない。エコシステムの規模ではLangGraphに勝てない。AGPL 3.0のSaaS注意点あり
  • インストールは2コマンド: Macはcurlでsetup.shをbashにパイプ、WindowsはPowerShellのIRM | IEX
  • おすすめ: ①Ollamaをローカルで立てる → ②AI Builderに自然言語を投げる → ③出てきた叩き台を手直し → ④実際に動かして確認
まずOllamaでAPI代ゼロで試して、 後でクラウドモデルに差し替える。
コストと品質のチューニングが 同じUIで完結するのが強い。
