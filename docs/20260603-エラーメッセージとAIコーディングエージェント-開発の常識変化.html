エラーメッセージは誰のため？

AIコーディングエージェントが変える開発の常識 — エラー省略は損だった
2026年6月3日

① エラーを省略して渡してませんか？

  • 「エラーが長すぎるからスタックトレースを消してエラー本体だけ渡してた」— 多くの開発者がやっていたこの习惯、実はAIにとってマイナスだった
  • 人間の感覚だと「情報が多すぎると混乱しそう」と思いがち。でもAIは人間とはまったく逆の性質を持っている
  • これは最近の論文で実験データで裏付けられた発見。「情報を減らしてAIに渡す」は完全に間違っていた
  • 結論から言うと：AIにとってあなたが消していた詳細情報は宝の山だった
省略していたあの情報こそ、
AIが正しく修正するための
最大の手がかりだったんです！

② エラーメッセージの「第2の読者」が登場

  • エラーメッセージはこれまで人間のためだけに設計されてきた。コンパイラを作る人たちはずっとデバッグしやすい文明を構築してきた
  • しかし今、第2の読者が現れた。それがClaude CodeのようなAIコーディングエージェントだ
  • エラーを読んで自分で修正するAI。人間と同じエラーを見ているのに、求めている情報の質がまったく違う
  • 人間は要点を1行欲しい。AIは冗長でもいいから全部出してくれが欲しい。ニーズが真逆に分かれた
同じエラーを一緒に読むけれど、
欲しい情報は正反対。
これが全話の起点です。

③ 削除実験で証明された事実

  • アブレーション実験（削除実験）を実施。エラーメッセージの一部をわざと削ってAIに修正させ、成功率を比較
  • 実験にはMLKのSvelteKitのような型付き言語を使用。ユニフィケーション（型パズル合わせ）が噛み合わない時にどこがどう合わなかったかを全部記録するのがリッチなエラー
  • 結果：情報を消すとAIの修正成功率が明確に落ちた。料理から塩を抜いたら味が変わるのと同じ
  • 「俺がスタックトレース消して渡していたのは、AIの修正能力をわざわざ下げていたってことか！？」— 罪悪感の声
人間が混乱するからといって
AIも混乱するとは限らない。
勝手な思い込みだった！

④ AIは情報が多いほど賢くなる

  • AIは大量のテキストを読むのが苦手ではない。むしろ文脈の手がかりが多ければ多いほど正解にたどり着きやすい
  • 人間が「うるさい」と感じるユニフィケーションのトレースやスタックトレースこそ、AIにとっては位置情報と矛盾の構造を伝える重要なデータ
  • 画面いっぱいにスクロールするくらいのエラーでも、Claudeのコンテキスト窓に収まる量なら全部入れた方がいい
  • 人間の直感「情報は少ないほどいい」はAIには通用しない。反直感的だが実験で証明済みの事実
AIにとって
「うるさいエラー」は「詳細な地図」
と同じ価値を持っている。

⑤ 型エラー vs テスト失敗：意外な結果

  • 「型エラーとテスト失敗、どっちの方がAIが修正しやすい？」— 型エラーの方が圧倒的に情報量が多い
  • テスト失敗は「結果」しか伝えない。「こうなるはずがこうならなかった」という事実。原因の特定はそこからAIが自分で探す必要がある
  • 型エラーは「構造の矛盾」を直接示す。「この行のこの引数がおかしい」と住所まで書いてある状態
  • テスト失敗は犯人探しの最初の1個から始める感じ。型エラーは犯人の住所まで地図で示してくれる
人間にとってはテスト失敗の方が
原因が分かりやすいけど、
AIには逆なんです！

⑥ 型修正は「本質的バグ修正」に近い

  • 論文のさらなる発見：型エラーを直したコードはその後のテストにも通る傾向があった
  • 「型を直したらバグが直っていた」というケースが多発。型エラーの修正は表面的な辻合わせではなく本質的な修正に近い
  • 型システムが「ちゃんと意味の正しさを守っていた」— 語源（型）の本来の役割がAI時代にも証明された
  • 型付き言語を使うメリットが「人間の安全」だけじゃなく「AIの修正精度」にも直結するという発見
型システムが正しい意味を守っていた。
ちょっと感動する結果ですね。
型の価値が再確認された瞬間。

⑦ 言語選びに新たな評価軸が誕生

  • 今までTypeScriptなどの型付き言語を選ぶ理由は人間のメリットだった。補完が効く、リファクタしやすいなど
  • そこに「AIが直しやすい」という新たな理由が1つ大きく増えた
  • Rust, Kotlin, Haskell, TypeScriptなど型情報が豊富な言語がAIとの開発で有利に
  • TypeScriptでも型ヒントを書くか書かないかでClaude Codeの仕事の質が変わる。Pydanticでスキーマ定義していると、AIが構造を完全に把握できる
言語を変えなくても
型ヒントを足すだけで
効果があるなら
動的型付け派にも朗報！

⑧ エラーを型で表現する設計思想

  • ユーザーIDをただのstringとして扱わず、ブランデッドタイプ（User型のID専用型）として定義する
  • 注文IDをユーザーIDの引数に渡すバグがコンパイル時点で型エラーとして表に出る
  • テストを書いて落ちたところからAIに修正させるより、型で先に弾く方が断然AIには優しい
  • ZodやPydanticでスキーマ定義しておくと、AIが「このフィールドはstringのはず」を完全に把握できる。テスト間も減る一石二鳥
テストで後から見つけるのではなく、
型で先に弾く。
AIにとっても人間にとっても優しい！

⑨ 変数名を全部消してもAIは理解できる

  • 衝撃の実験結果：変数名を全部bar1, bar2, bar3みたいに置き換えても、AIはコードの意味を正しく理解できた
  • username, priceといった手がかりを全部消した状態でも、型エラーがあれば正しく修正できた
  • つまりAIは変数名から意味を読んでいるのではなく、コードの「形」から読んでいた
  • 型構造、関数の呼び出し関係、引数の数 — AST（抽象構文木）のような骨格レイヤーで理解している
変数名が意味のない名前でも
構造が見えてれば意味は推測できる。
AIは形読みなんです！

⑩ でも命名のレビューは無駄じゃない

  • 「プルリクで変数名をもっと意味のあるものにって指摘するのはAI観点では無駄なのでは？」
  • AI観点だけならそう。でも人間が3ヶ月後に読み返すことを考えたら命名は当然大事
  • 良いレビューの基準は変わらない。ただAIの存在を知った上で優先順位の付け方が少し変わる
  • 命名 → 人間の可読性重視。型の充実 → AIの修正精度重視。それぞれの役割が明確になった
レビューの意味が
なくなったわけじゃないです。
人間とAIで役割分担が明確に。

⑪ コードの形が説明書になる

  • コードの意図をAIに伝えるためにコメントやドキュメントを書く — これも大事だがさらに上位にあるのが「型と構造で語ること」
  • 例えば findUser(id): Promise<User | null> というシグネチャがあればコメント不要。戻り値がユニオン型な時点で「見つからないケースがある」ことが分かる
  • コメントで書くより型で表現する方がAIには100倍伝わる。コメントは読まれない可能性もあるが型は絶対に評価される
  • コメントだと嘘をつくこともできるが、型は嘘をつけない。型には強制力がある
コメントはオプション、
型は契約。
コードの形そのものが
説明書になる時代に！

⑫ 実践アクション①：エラーは全部貼る

  • エラーをClaude Codeに渡す時は全部ペーストする。省略しない。冗長な部分も含める
  • コンパイラのverbose出力やデバッグレベルを上げる設定があればそれも使う
  • TypeScriptなら extendedDiagnostics や noErrorTruncation。普段はうるさいから切っている設定こそ、Claudeに渡す時はオンにする
  • Rustなら RUST_BACKTRACE=1 も該当。エラーだけじゃなく関連するソースファイルも合わせて渡すと修正成功率がさらに上がる
Claude Codeは自分でファイルを
読みに行ってくれるけど、
最初から渡した方が早い。
ファイル探索のターンを節約！

⑬ 再現コマンドも一緒に渡す

  • エラーと一緒に再現コマンドも渡すのが正解。npm testやcargo runなど「このコマンドでこのエラーが出る」を伝える
  • 再現コマンドを渡すと、Claudeが自分で再実行して確認できる。修正後に本人がもう1回叩いて治ったかチェックできる
  • これだけで自動で回るループが完成。エラー → Claudeが修正 → 自分でテスト → 治ったか確認
  • ターミナルでエラーを見たらつい「Error:」から下だけハイライトしてコピーしがちだが、コマンド実行前まで選択に変えるだけで成功率が変わる
「Error:」の行から下を選ぶのではなく、
コマンド実行前から選ぶ。
地味だけど効果は大きいです。

⑭ プロジェクト設計に「AI修飾性」を入れる

  • 言語選定に「AIと一緒に開発する速度」という新しい評価軸を1つ足す
  • チェックすべき3つのポイント：エラーメッセージの質、型情報の豊富さ、ドキュメントの構造化具合
  • ドキュメントの構造化とは？Rustのstd::docsのように関数の引数・戻り値・Safetyセクションが構造化されている形式
  • 今までパフォーマンス、エコシステム、チームのスキルで語っていた言語選定議論に「AIが読みやすいか」を加えるべき
チームでこの話をすると
設計の優先順位が
変わってくるはず。

⑮ 型で先に弾く設計の癖をつける

  • エラーを型で表現できる場所は型で。テストで後から検知するのではなく、コンパイル時にAIに「ここが間違っている」と伝える
  • APIのレスポンスをanyで受けないでZodでスキーマ定義する。Pydanticでモデルを定義する
  • スキーマ定義しておくと「ズレたら型エラーで即座に出る」→テスト間も減る→ AIにも優しい遺産になる
  • これは設計レビューで意識的に議論したい話。テストカバレッジだけでなく「型カバレッジ」も気にする時代が来るかも
テスト → 実行時
型 → コンパイル時
早い段階で弾くほど
AIにも優しい。

⑯ 2人の読者は理屈が噛み合う

  • コードが読まれる相手が1人増えた。人間とAIが二重ユーザーになった
  • 重要なのは2人の利害が割と噛み合うこと。型をしっかり描くと人間にとっても安心。良いコメントは人間にもAIにもヒントになる
  • 人間は要約を欲しがる、AIは冗長を欲しがる — ニーズは違うが方向性は同じ
  • 「型のために頑張るんじゃなくて、型も見るからいつもより1段ちゃんと書く感覚でいい」— 特別な努力は不要、いつもの延長線上
AIのために別のスタイルを
用意する必要はない。
良いコードはAIにも通じる。
素晴らしいことに！

⑰ コードに対する見方が変わる

  • エラーメッセージの設計者、言語デザイナー、そして開発者自身が2人の読者を意識する時代に入った
  • コードの形に対する見方が変わると、毎日のレビューも設計判断もちょっとずつシフトしていく
  • エラーメッセージの設計者も言語デザイナーも人間用だけじゃなくAI用の読みやすさも考えるようになる未来が来る
  • 小さな一歩が大きな変化に。「明日のエラー出た時に省略しないで全部渡す」— それだけで開発体験が変わる
Claude Codeの使い方を見直す
機会にもなった。

⑱ 明日からの3つの小さな一歩

  • ①エラーメッセージは省略せず全部貼る。これだけでも明日の開発体験が変わる。一番効果の大きいアクション
  • ②型ヒント・スキーマ定義を雑からちゃんと。anyやエニーを1個ずつ型付けしていく。ZodやPydanticでAPIレスポンスを定義
  • ③言語選定で「AI修飾性」を話題に出す。チームの設計議論にエラーメッセージの質、型の豊富さ、ドキュメントの構造化を加える
  • 本当に意識を変えるだけで開発体験が変わる。「多分それだけで何かが変わる気がしてきた」— 型を意識して書いてみる
