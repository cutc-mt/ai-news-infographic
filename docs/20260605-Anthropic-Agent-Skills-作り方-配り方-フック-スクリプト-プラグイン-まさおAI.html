Anthropic公式が語る Skillの作り方・配り方

数百のスキルを運用して掴んだベストプラクティスを解説！
📅 2026年6月5日 | まさおAIじっくり解説ch

①Anthropic公式ブログが大放出

  • Anthropicが公式ブログで数百のスキルを運用した知見を公開
  • スキルの作り方から配り方まで、ボリューム満点の内容
  • 社内で実際に使いながら見つけたベストプラクティスが中心
  • 公式ならではの読み応えのある記事として話題に
公式からこのボリュームの記事は珍しい！スキル使いの必見コンテンツだね

②スキルは1枚のMarkdownじゃない

  • スキルはSKILL.mdだけではありません
  • スクリプト・フック・リファレンス・アセットなども含むフォルダー構成
  • メモリファイルもスキルフォルダーに入れて管理可能
  • 構造化された情報でエージェントも内容を把握しやすい
1枚のMarkdownだと思ってた人も多いはず。実はもっとリッチな構造なんだね！

③フックでエージェントを制御

  • フックを使うとスキルが読み込まれたタイミングで処理を追加できる
  • セッション中ずっと発動し続けるフックも設定可能
  • 既存のフックにプラスアルファで機能を追加できる仕組み
  • YAMLフロントマターの中にもフック設定が存在する
フックを使えばエージェントの動きを細かく制御できる。これがスキルの強力さの秘訣！

④スクリプトで決定論的な処理を

  • AIの推論に頼らずコードで確実に実行したい処理をスクリプトに
  • 例えば「ステップ2でこのスクリプトを発動」のような段取りも可能
  • JSONやYAMLなどの定型フォーマットを読み込ませて処理させる使い方も
  • プロンプトだけではブレる部分をコードでガッチリ固める
AIに「こうやってね」で任せるより、コードで書いちゃう方が確実なんだよね

⑤リファレンス資料もスキルに内包

  • テンプレートや参考資料をスキルフォルダーに入れておける
  • APIリファレンスやライブラリ解説も一緒に管理すると便利
  • 専門知識をスキル経由でエージェントに渡す仕組みとして活用
  • エージェントが必要な情報を自分で見つけやすくなる
資料をスキルの中にまとめとけば、エージェントが自分で必要な情報を見つけられる！

⑥Anthropicが定義する6分類

  • 基本ライブラリ — API参照・ツールの使い方
  • 自作ツール — 独自CLIやスクリプトの使い方
  • 動作検証 — アプリのテスト手順（★一番効果が高かった）
  • データ取得分析 — データ収集の再現性を担保
  • ランブック — インフラ運用手順
  • 運用タスク — 定期業務の自動化ワークフロー
動作検証系のスキルが社内で1番効果的だったって共有があったよ！

⑦動作検証スキルがダントツで有用

  • Anthropic社内で運用した結果、動作検証系のスキルが最も効果的だった
  • 「このアプリはこういう形で動作確認を進めましょう」といった手順を記述
  • テストのやり方をスキル化しておけば、毎回同じ品質で検証が可能
  • バグの見つけ方や検証パターンもスキルに蓄積できる
数百のスキルを使った結果、動作検証がトップ。テスト自動化の最強ツールかも！

⑧Claude Codeスキルの独自性

  • 他のエージェントハーネスとは概念的な取り扱いが異なる
  • YAMLフロントマターにモデルの努力レベルやシェルの設定が含まれる
  • ユーザーフラグ系の設定もスキル単位で管理可能
  • 全部使い回せるわけではないので、移行時は注意が必要
Codexや他のエージェントにすぐ持っていけるわけじゃない。ここ重要！

⑨役割ごとに分ける、まとめすぎない

  • 関連する機能を1つのスキルにまとめすぎないことが大切
  • 役割ごとに分けておいた方が使いやすくなる
  • まとめすぎるとエージェントが必要な情報を見つけにくくなる
  • 適度な粒度で分割するのがベストプラクティス
1つにまとめちゃうと結局使いにくなる。「何をどこに置くか」の設計が大事！

⑩ツール・APIの使い方を教えるスキル

  • エージェントブラウザーや自作CLIツールの使い方をスキル化
  • 「このパターンの時にはこのコマンドが良さそうです」を事前に教える
  • APIドキュメントをリファレンスとして置いておくのも効果的
  • ヘルプコマンドにスキル情報を入れておくとさらに便利
ツールの使い方をプロンプトで毎回教えるのではなく、スキルにまとめちゃおう！

⑪データ取得分析をスキルで再現

  • データの取り方を再現性高くスキルに記録しておく
  • 「こういうデータが欲しい時はこうやれば取れます」という流れを教える
  • 忘れちゃう前提でスキルに書いておくのがコツ
  • 分析の手順や出力形式も統一できる
「あれ、どうやってデータ取ったっけ？」をスキルで解決。忘れても大丈夫！

⑫ランブックでインフラ運用を記録

  • インフラ運用の手順をスキルとしてあらかじめ作っておく
  • いざという時に「そういえばこのスキル作ってたな」と呼び出せる
  • 月1回のタスクなど、定期業務をスキル化して自動化
  • バイブコーディングで大量にアプリを作る時代に必須の対策
作ったアプリの運用手順を忘れないようにスキルに残しておく。運用の外部記憶だね！

⑬デスクリプションで発火タイミングを制御

  • デスクリプションには「いつ発火するか」を意識して書く
  • 「これは〇〇をするツールです。こういう時に発火しましょう」と簡潔に
  • エージェントは常にデスクリプションを頭に入れているので、内容が重要
  • エージェントに対する思いやりがスキル品質の向上につながる
デスクリプションの書き方一つで、エージェントがスキルを使ってくれるかが変わるんだね

⑭スキルの展開テクニック

  • 冒頭にびっくりマークをつけるとスキルを展開できるテクニックがある
  • 保存先の指定方法もスキル内で取り決め可能
  • 「!スキル名」のように明示的に展開を促す記法も活用できる
  • エージェントが自動で展開するのを待つだけでなく、能動的に使える
「!」で明示的に展開。待つだけじゃなくて自分から呼び出せるのが便利！

⑮リポジトリ内かプラグインかの選択

  • リポジトリ内 — プロジェクト固有のドメイン知識に使う
  • プラグイン — 再利用可能な汎用的なスキルに使う
  • 再利用性の高いものはプラグイン化してメンテナンス
  • プラグインはClaude Code限定。他のエージェントには持っていけない
使い分けが大事！プロジェクト専用ならリポジトリ、汎用ならプラグイン！

⑯ユースログでスキルの健康診断

  • PostToolUseのフックでスキルの利用ログを収集
  • よく使われているスキルと使われていないスキルが分かる
  • 使われていないスキルの整理や改善の判断材料になる
  • 実装例のGistが公式から共有されていてすぐ試せる
「どのスキルが実は使われてない？」をログで把握。良いスキル作りのために超重要！

⑰クロードのフックが持つ独自の力

  • Claude Codeのフックはいろんなところに介入できるのが強み
  • エージェントの動きを制御・調整できる点が他のハーネスと違う
  • コンテキストの調整もフック経由で可能
  • すべてのエージェントで同じ仕組みが使えるわけではない（注意点）
Claude Codeのフックは介入ポイントが多くて制御力抜群。他のハーネスにはない強さ！

⑱まとめ: 3つの重要ポイント

  • スキルはフォルダー構造 — スクリプト・フック・リファレンスを含む
  • 6分類で整理 — 動作検証が最強、役割ごとに分ける
  • 配り方を考える — リポジトリ or プラグイン、ログで改善
  • Anthropic公式からの貴重なベストプラクティス。スキル作成者の必読記事！
公式ブログ＋解説者の深い知見で、スキルの本質がよく分かった！今すぐスキル作ろう！
