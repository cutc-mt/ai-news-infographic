Opus × Gemini マルチプロバイダー・ワークフロー

最強のUI（Gemini）× 最強の推論（Opus）を組み合わせてフルスタックWebアプリを一発構築！
📅 2025年6月5日 | 🎬 Antigravity

①Gemini 3.5 Flash 登場: 安くて速くてUIが綺麗

  • GoogleがGemini 3.5 Flashをリリース。低コスト・超高速のLLM
  • LLMにしては珍しく、人間が手作りしたような美しいフロントエンドを構築できる
  • 入力100万トークンあたり$1.50と圧倒的低コスト。大量のトークンを消費しても安上がり
LLMって普段UIを作るの下手だけど、Geminiは別格！

②Opus 4.8 登場: 推論力で圧倒的性能差

  • AnthropicがOpus 4.8をリリース。長時間のエージェントエンジニアリングタスクで圧倒的な力を発揮
  • ベンチマークではGeminiも健闘しているが、実際の使ってみた感想ではOpusが一段上
  • Opusの単価は高いが、推論品質では他の追随を許さない実力を持つ
推論力ならやっぱりOpus一強！長時間タスクで差が歴然。

③両者のいいとこ取り: 混在プロバイダーの概念

  • 「どちらかを選ぶ」のではなく、両方のモデルを組み合わせるワークフローを構築
  • GeminiでUIデザインを担当し、Opusでページコピー・統合・推論を担当する役割分担
  • コスト効率も最適化。UI作成にトークンを大量投入してもGeminiのおかげで安く済む
強みを組み合わせる。まさに最強のタッグ！

④Geminiの弱点: ページコピーのハルシネーション

  • GeminiはUIの見た目が素晴らしいが、テキスト内容や情報がハルシネーションしやすい
  • 例えばベンチマークの数値など、架空のデータをでっち上げてしまう問題が頻発
  • だからこそOpusに情報の正確性を担保させる役割が重要になる
見た目は完璧でも中身が嘘だったら困るよね！そこをOpusでカバー！

⑤ワークフロー設計: 各ステップは別セッション

  • すべてのステップが独立したコーディングエージェントセッションとして実行される
  • ステップ間の通信はハンドオフ文書（Markdown）で行う設計
  • 各ステップの出力Markdownが次のステップの開始時に読み込まれる仕組み
一つの会話の中でプロバイダーを切り替えられないから、この設計に！

⑥単一タスク集中: LLMに一度に一つのことだけ

  • Opusのような最強モデルでも、一度に太多のことをさせると失敗する
  • 探索・計画・UI構築・統合・検証をそれぞれ別のセッションに分割するのが成功の鍵
  • ステップごとにモデルを簡単に入れ替え可能。実験性が高く、最適な組み合わせを見つけやすい
一つのことに集中させる。これがLLM活用の基本中の基本！

⑦ステップ1: 探索フェーズ（Sonnet）

  • ワークフローの最初の入力は仕様書（spec document）。手書きでもエージェントで作ってもOK
  • Sonnetを使用。探索には高性能推論モデルは不要で、コスト効率を重視
  • 出力はcontext.md。リポジトリ状態・フレームワーク推奨・ブランド資産などをまとめる
Claude Codeで /model で Sonnet に切り替えて実行！

⑧ステップ2: 計画フェーズ（Opus）

  • ここでOpusの出番。context.mdを読み込み、3セクション構成の詳細な計画を作成
  • セクションA: サイトのコンテンツと意図（音声・トーン・メッセージング）
  • セクションB: 統合スコープ（API・DB・認証などバックエンドの設計）
  • セクションC: デプロイ計画（Vercel・DigitalOceanなど）
UIの構造（グリッドやカラム）は計画には含めない。Geminiに任せるのがポイント！

⑨ステップ3: UI構築フェーズ（Gemini）

  • Gemini 3.5 Flashが計画のセクションAだけを読み込んでUIを構築
  • PiまたはAntigravityでGeminiを利用。OpenRouter経由で接続
  • OpusがUI構造を指定するとGeminiの強みが活かせない。計画にはコンテンツのみ記載する
  • 出力はUIサマリー。次の統合フェーズで活用される
Geminiに任せると、エージェント製とは思えない美しいUIが完成！

⑩ステップ4以降: 統合・検証・デプロイ（Opus）

  • UIの次は再びOpusに戻り、バックエンド統合・API構築を実行
  • 検証フェーズでSonarを利用してユニットテスト・リントを実行
  • 問題があればOpusが自動修正して再検証
  • デプロイフェーズでVercelやDigitalOceanへ自動デプロイ可能
Vercelへの自動デプロイもテスト済みで好結果！

⑪ステップ最後: スモークテスト（ブラウザ自動化）

  • デプロイ後にエージェントブラウザスキルで自動的にウェブサイトをテスト
  • 実際のユーザーが操作するようにブラウザ自動化で動作検証を実施
  • ハンドオフ文書を介して人間が任意のステップで介入・確認することも可能
ブラウザで実際に動かして確認。完全自動で！

⑫Archon: ワンショット実行ツール

  • Archonはオープンソースのハーネスビルダー。すべてのスキルを統合して一発実行
  • エージェントが自律的に全ステップを順次実行。手動で一つずつ回す必要なし
  • ただし本動画ではスキルごとの使い方を解説。どのハーネスでも応用できる知識を提供
全自動で回したいならArchon！手動で確認したいならスキル単位で！

⑬対応ハーネス: Pi・Antigravity・Claude Code

  • Pi: Claude Codeに似たエージェントハーネス。/model で Gemini に切り替え可能
  • Antigravity: 新しいバージョンでGeminiとの親和性が大幅向上
  • Claude Code: 探索・計画・統合・検証フェーズで活用。OpenRouterでGeminiにも接続可能
  • スキルはプロンプトとして直接読み込むことも可能。どんなハーネスでも使える
特定のツールに縛られない設計が素晴らしい！

⑭仕様書の重要性: 入力はすべてspec.mdから

  • ワークフローの唯一の入力は仕様書（spec.md）
  • 手書きでもエージェントと協力して作成してもOK。リポジトリにサンプルあり
  • 大規模なアプリを構築するほど、事前に具体的な仕様を定義することが重要
ガバッと書いた仕様でも、Opusがちゃんと解釈してくれる！

⑮デモ: Deep Space Catalogを一発構築

  • 「Deep Space Catalog」という宇宙カタログサイトをワークフローで一発構築
  • UIはGeminiが作成した美しいデザイン。Claude CodeやLovableのワンショットより格段に良い
  • バックエンドもOpusが構築。惑星データベースやAPI統合がフルスタックで完備
  • Vercelへの自動デプロイも成功。PoCやMVP構築に最適
エージェント製とは思えないクオリティ！でもまだヘッダーだけはLLM感あり（笑）

⑯セキュリティ検証: SonarQubeで依存パッケージを守る

  • 動画スポンサー: SonarQube Advanced Security
  • 最近のAxios NPMパッケージの脆弱性のように、エージェントが悪意あるバージョンを導入するリスク
  • SonarQubeはサーロットブレーカーのように機能し、未検証の依存パッケージをブロック
  • SCA SAST シークレット検出を一つのプラットフォームで統合
AIが機械速度でコードを書くなら、検証も機械速度で！

⑰ベストプラクティス: マルチプロバイダー活用のコツ

  • 各ステップのハンドオフ文書を必ず検証する。人間をループに入れて確認可能
  • ステップごとにモデルを簡単にスワップできる設計にする。実験性が重要
  • ハンドオフのコンテキストエンジニアリングに多くの労力を注ぐ。コミュニケーションの質が結果を左右する
  • ワークフローは汎用的で、どんなデプロイプラットフォームやバックエンドにも対応可能
自分なりのハーネスにもこの設計思想を応用できる！

⑱まとめ: 最強のフルスタック開発ワークフロー

  • GeminiのUI構築力 × Opusの推論力を組み合わせた画期的なアプローチ
  • Geminiの安さを活かしてUI作成にトークンを大量投入しつつ、Opusは重要な部分だけに使用してコスト最適化
  • リポジトリはGitHubで公開中。READMEにエージェント向けセットアップガイド付き
  • PoCやMVPの構築に最適。どんなハーネスでも応用可能な知識を提供
どちらかを選ぶ必要はない。両方使えば最強になれる！
