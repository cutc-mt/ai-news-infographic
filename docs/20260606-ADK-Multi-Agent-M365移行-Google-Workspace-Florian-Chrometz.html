② エンタープライズ移行の地獄と地味な作業

  • Microsoft 365からGoogle Workspaceへの移行は、企業の生産性スイーツ全体の切り替え
  • ライセンスとアドオンのマッピングが純粋な肉体労働。数日かけてドキュメントを読み、最新の変更を確認し、対応表を作る必要がある
  • ユーザーにとって分かりやすい10章構成のクリーンなレポートを出力する必要がある
  • 手作業ではミスも発生しやすく、人的コストも膨大。まさにAIエージェントが解決すべき領域

③ ADKで構築する移行アセスメントエージェント

  • ユーザーがソフトウェアライセンスのテキストや画像を入力すると、自動でリサーチ→構造化データ出力まで実行
  • M365のライセンスとアドオンをGoogle Workspaceの同等製品に正確にマッピングする10章構成のレポートを生成
  • オーケストレーションパイプラインとしてADKを活用し、入力→研究→出力の流れを構築
  • ADKがアーキテクチャをクリーンでモジュール式に保つ仕組みを実演

④ Google Agent Development Kit（ADK）とは

  • Googleが提供するオープンソースフレームワーク。マルチエージェントシステムの構築に特化
  • 複雑なプロンプトに頼らず、複数のシンプルなエージェントが協力してタスクを分割・実行する設計パターンを実現
  • エージェントの振る舞いとオーケストレーションを精密に制御できる。設計と整理と信頼性が向上する
  • GeminiモデルやVertex AIとの統合が最適化。Google Cloudエコシステムへのデプロイパスも用意

⑤ SequentialAgentで直列パイプラインを構築

  • ADKのSequentialAgentを使って、ワークロードを直列のパイプラインに分割
  • 入力パーサーがライセンスデータを抽出、リサーチャーがドキュメントを検索、レポーターがレポートを作成
  • 各エージェントには非常に具体的なタスクと独自のシステムプロンプトが割り当てられる
  • タスクが細分化されることでハルシネーションが減少し、デバッグも格段に楽になる
1つの巨大プロンプト vs
複数の小さなエージェント。
後者の方が確実。

⑥ 3つのサブエージェントの役割分担

  • Input Parser（入力パーサー）: ユーザーが入力したテキストや画像からライセンス情報を正確に抽出する
  • Researchers（リサーチャー）: 内蔵ドキュメントやナレッジベースを検索してM365とGWSの対応関係を調査する
  • Reporter（レポーター）: 調査結果を整理して10章構成のクリーンなレポートとして出力する
  • それぞれのエージェントが専任で担当するため、専門性が高く精度も向上
チーム開発と同じ発想。
役割分担で品質と効率を両立。

⑦ LoopAgentで自己修正サイクルを実現

  • LLMにゼロショットやワンショットで完璧な10章レポートを出力させるのはほぼ不可能
  • だからバリデーションが必要。ADKのLoopAgentで自己修正ループを作る
  • ライセンスマッパーとレポートレビューアーをペアリング。レビューアーが結果を検査して問題があればマッパーに戻す
  • 不足している機能や不要な絵文字、フォーマットの崩れなどを自動検出して修正する
「生成 → チェック → 修正」
のループが完成する前に
出力を確定させない。
これが信頼性の秘訣。

⑧ リアルタイム検索とローカルナレッジベースの融合

  • Google Searchをリアルタイムグラウンディングツールとして活用し、最新の製品情報を取得
  • ローカルに保存されたエンタープライズ機能マトリクスのCSVファイルをナレッジベースとして参照
  • 2つのデータソースを組み合わせることで、正確で最新のマッピング結果を実現
  • 検索結果だけでなく、社内ナレッジも活用するハイブリッドなアプローチがポイント

⑨ ADKの組み込みコンテキストキャッシュでコスト削減

  • 巨大なローカルCSVファイルを毎回プロンプトに注入すると、コンテキストウィンドウを消費してAPIコストが急増
  • ADKの組み込みコンテキストキャッシュがこの問題を解決。ナレッジベースを一度ロードすれば再利用可能
  • ADKがTTL（Time To Live）とトークン制限を自動的に管理。開発者は意識する必要がない
  • 結果としてレイテンシの低減とトークンの節約が毎回の実行で実現される
同じデータを何度も送らない。
キャッシュでコストと速度を
同時に改善できるのは大きい。

⑩ ロギングプラグインで本番運用の可観測性を確保

  • ADKにロギングプラグインを追加するだけで、プロンプトとレスポンスのデータを直接ログにストリーミング
  • ロギングシンクへのストリーミングで、本番環境のオブザーバビリティをサポート
  • どのエージェントがどのタイミングで何を出力したかを追跡可能。デバッグと監査に不可欠
  • プラグインアーキテクチャにより、カスタムのモニタリングツールとの統合も容易
本番環境では可観測性が命。

⑪ ワンコマンドでローカルIDEから本番へデプロイ

  • ローカルIDEで開発したエージェントを、カスタムDockerファイルなしで本番にデプロイ
  • ADKで構築したエージェントはGoogleのAgent Engineに1つのコマンドで直接デプロイ可能
  • 複雑なホスティングインフラの構築は不要。インフラ構成の悩みから解放される
  • 「ローカルで動いた」から「本番で動く」までのギャップが最小。開発者体験が大幅に向上
Dockerファイル書かなくてOK。
1コマンドデプロイは
開発者の時間を大幅に節約。

⑫ HTTP 429リソース枯渇エラーの対策法

  • 実運用で最も学んだことはHTTP 429「リソース枯渇」エラーの回避方法
  • 解決策は2つのHTTPヘッダーを追加するだけ。プライオリティ推論を利用する
  • プライオリティ推論を有効化すると、モデルエンドポイントで優先スロットが保証される
  • これにより、高負荷時でもエージェントの安定した実行が可能になる
ヘッダー2行追加で
レート制限を回避。
実務で大違いのTips。

⑬ 信頼できるエージェントの3つの設計原則

  • 状態管理: エージェントがタスクの進行状態を正しく追跡し、失敗時のリカバリーも可能にする
  • ルーティング: 入力を適切なサブエージェントに分配する仕組み。SequentialAgentとLoopAgentがこれを担う
  • ガードレール: 出力品質の検証ループ。レビューアーエージェントが成果物を検査し、不十分なら修正を要求
  • この3つが揃って初めて本番で信頼できるエージェントが完成する

⑭ パイプライン全体の流れを整理

  • ステップ1: ユーザーがライセンスのテキストや画像を入力する
  • ステップ2: Input Parserがライセンスデータを抽出して構造化する
  • ステップ3: ResearchersがGoogle SearchとローカルCSVを検索して対応関係を特定
  • ステップ4: Reporterが10章のレポートを生成し、Reviewerが検証して必要に応じて修正ループに入る
  • ステップ5: 品質基準を満たしたレポートがユーザーに提示される
入力 → パース → 検索 → 生成 → 検証
この流れが自動で完結する。

⑰ ADKエージェント構築のレシピまとめ

  • 分割: 1つの巨大なプロンプトではなく、小さくフォーカスされたサブエージェントにタスクを分割する
  • 検証ループ: LoopAgentで自己修正サイクルを組み込み、品質を自動的に担保する
  • キャッシュ: コンテキストキャッシュで大きなデータの再注入を回避し、コストとレイテンシを削減する
  • デプロイ: Agent Engineへのワンコマンドデプロイで本番運用へスムーズに移行
  • HTTPヘッダー: プライオリティ推論用のヘッダー追加で429エラーを回避する
