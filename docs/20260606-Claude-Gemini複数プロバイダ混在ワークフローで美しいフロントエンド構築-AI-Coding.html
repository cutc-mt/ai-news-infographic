Claudeが計画し、Geminiがデザインする

複数プロバイダ混在ワークフローで美しいフロントエンドをワンショット構築
2026年6月6日

① 2つの新モデルが同時期に登場

  • GoogleがGemini 3.5 Flashをリリース。安価で超高速、LLMとは思えない美しいUIを生成
  • わずか1週間後にAnthropicがOpus 4.8を投入。長時間のエージェントタスクで高い推論力を発揮
  • どちらも一長一短があるため、片方を選ぶのではなく両方を組み合わせるアプローチが最適
どちらのモデルも強みが違う。GeminiはUI制作に優れ、Opusは推論力が高い。この2つを掛け合わせるのが鍵！

② Gemini 3.5 FlashのUI制作力

  • AntigravityやPiを使ってワンショットで手作りのような美しいUIを生成可能
  • Claude CodeやLovableなどの従来のツールと比べて、はるかに自然な仕上がり
  • ただしLLM特有のヘッダーデザインなど、まだわかる部分は残っている
  • 高速かつ安価なため、UIの試行錯誤に大量のトークンを消費できるのが強み
Claude Codeで作るUIは「AIっぽさ」が出がちだけど、Geminiはもっと人間が作ったように見える。この差はでかい！

③ Geminiの弱点：ハルシネーション問題

  • UIは美しいが、ページのテキスト内容がハルシネーションする問題が顕著
  • ベンチマーク上ではOpusやSonnetに近いスコアだが、実使用では推論力でOpus 4.8が明らかに上回る
  • 情報の正確性が求められるページコピーやバックエンド構築にはOpusを使うべき
見た目は完璧でも、書いてある内容がデタラメだったら意味ないよね。だからOpusの推論力が必要なんです。

④ コスト面でもGeminiが圧倒的に有利

  • Gemini 3.5 Flashは100万入力トークンあたり$1.50と極めて安価
  • Opus 4.8ははるかに高額。UIデザインに大量のトークンを消費するならコスト差が大きい
  • 安いからこそ何度もやり直せる。GeminiでUIを作り直すコストは低く抑えられる
UIデザインは試行錯誤が多いから、コストが安いGeminiを活用するのは理にかなってます！

⑤ 混在ワークフローの核となる考え方

  • GeminiにはUIデザインを担当させ、Opusには計画と情報の正確性を担当させる
  • 各ステップは別々のコーディングエージェントセッションとして実行される
  • ステップ間の通信はハンドオフドキュメント（Markdown）で行う
  • 1つのセッション内でプロバイダを切り替えることはできないため、この分割設計が必須
「全部Opusでやればいいや」は大間違い。1つのLLMに全部やらせると必ず破綻します！

⑥ ハンドオフドキュメントの仕組み

  • 各ステップがMarkdownファイルを出力し、次のステップがそれを読み込む仕組み
  • 例えば探索ステップではcontext.mdを出力し、計画ステップがそれを読み込む
  • 計画ステップではplan.mdを出力し、デザインステップがそれを読み込む
  • ドキュメントを人間が確認できるため、途中で介入・修正が可能
各ステップの出力をチェックできるから、エージェントが脱線してもすぐ軌道修正できます！

⑦ 1タスク＝1セッションの原則

  • LLMは1つのセッションで複数のタスクをこなすと性能が落ちる
  • 全体を1つのセッションで実行しようとすると、複雑なアプリでは必ず失敗する
  • 各セッションを単一タスクに絞ることで、最高の結果が得られる
  • 分割すれば、各ステップのモデルを簡単に入れ替えて実験可能
「探索にGeminiを使ってみよう」「統合にClaude Kimmy K 2.6を試そう」など、柔軟に実験できます！

⑧ 最初のステップ：探索（Exploration）

  • ワークフローの入力はスペックドキュメント（仕様書）
  • このステップにはSonnetを使用（推論力は不要で高速・安価だから）
  • リポジトリ状態、フレームワーク推奨、ブランドアセット、環境変数、データ層を調査
  • 結果をcontext.mdとして出力し、計画ステップに渡す
Claude Codeなら /model でSonnetに切り替えてから /front-end-mix-explore を実行するだけ。

⑨ 2番目のステップ：計画（Planning）

  • このステップはOpus 4.8に切り替えて実行
  • 3セクション構成の詳細な計画書plan.mdを作成
  • セクションA：サイトのコンテンツと意図（音声・トーン・メッセージ）
  • セクションB：統合スコープ（API、データベース、バックエンド）
  • セクションC：デプロイ計画（Vercel、DigitalOcean等）
重要：OpusにはUIの構造（グリッドやカラムのレイアウト）を一切説明させないこと。それをやるとGeminiの良さが消える！

⑩ 3番目のステップ：UIデザイン（Gemini）

  • ここでGemini 3.5 Flashに切り替えてPiまたはAntigravityを使用
  • plan.mdを読み込むが、セクションAのみに注目してUIを構築
  • Geminiは独自のデザインセンスで美しいフロントエンドを生成する
  • 結果としてUIサマリーを出力し、次の統合ステップに渡す
Piでも /frontend-mix-design とするだけ。Claude Codeと同じ感覚で使えるのがいいところ。

⑪ OpusがUI構造を指定すると逆効果

  • 初期のワークフローでOpusがGeminiにUI構造を指示してしまった問題
  • Opusがグリッドやカラムなどを詳細に指定すると、Geminiのデザイン力が阻害される
  • 解決策：Opusにはコンテンツと意図のみを記述させ、レイアウトはGeminiに任せる
  • この修正により、Gemini本来のUI制作力が最大限に発揮される
Opusがレイアウトまで指示すると「OpusのUI」になってしまう。Geminiの強みを活かすには任せるしかない！

⑫ 統合ステップ：Opusでバックエンドを構築

  • GeminiがUIを作ったら、Opusに戻ってバックエンドやAPIの統合を実行
  • plan.mdのセクションB（統合スコープ）に基づいてデータベースやAPIを構築
  • Opusの推論力により、ハルシネーションのない正確な統合が実現
  • UIサマリーとplan.mdの両方を参照して、フロントとバックを接続
例えば宇宙カタログサイトなら、惑星のデータベースモデルやAPIエンドポイントをOpusが正確に構築します。

⑬ 品質検証ステップ：Sonnetでテスト

  • 統合が終わったらSonnetを使って単体テストとリンティングを実行
  • 問題が見つかったらOpusで修正するサイクルを回す
  • Sonnetを使う理由はコスト効率が良いから。テストに高い推論力は不要
  • この「検証→修正」のループで品質を確保
高価なOpusをテストに使うのは勿体ない。Sonnetで十分カバーできるならそちらが正解！

⑭ デプロイとスモークテスト

  • デプロイはVercelやDigitalOceanのMCPを接続すれば自動化可能
  • Agent Browserスキルを使ってブラウザ自動化で実際のサイトをテスト
  • ユーザーが実際に使うようにクリックやナビゲーションを検証
  • これでデプロイ済みの完成アプリがワンショットで完成
ブラウザ自動化で「人間がテストする」のをエージェントにやらせる。これが最後の仕上げです！

⑮ 実例：Deep Space Catalog

  • このワークフローで構築した「Cosmic Explorer」デモサイトを紹介
  • Geminiが作ったUIは手作りのような美しいデザインに仕上がっている
  • Opusが書いたページコピーはハルシネーションなしで正確な内容
  • Opusが構築したフルバックエンド付きの完全なフルスタックアプリ
UIの美しさと情報の正確さが両立できている。これこそが混在ワークフローの価値！

⑯ ワークフロー全体のモデル構成

  • 探索 → Sonnet（高速・安価）
  • 計画 → Opus 4.8（高推論力）
  • UIデザイン → Gemini 3.5 Flash（美しいUI・安価）
  • 統合 → Opus 4.8（バックエンド構築）
  • 検証 → Sonnet（テスト実行）
  • 修正 → Opus 4.8（バグ修正）
  • デプロイ＋スモークテスト → 自動化
高価なOpusは本当に必要なステップにだけ使う。賢いコスト配分がこのワークフローの肝です！

⑰ PiとOpenRouterのセットアップ

  • GeminiにアクセスするにはOpenRouterを経由するのがおすすめ
  • Piでは/modelコマンドでGemini 3.5 Flashに切り替え可能
  • Antigravityを使う場合は、スキルをプロンプトとして直接読み込むことも可能
  • GitHubリポジトリにはREADME付きでセットアップ手順が完備されている
Claude CodeユーザーならPiの操作感はほぼ同じ。移行コストはほぼゼロです！

⑱ ベストプラクティスと今後の展望

  • 各ステップに最適なモデルを選ぶことが最高の結果を生む
  • 1セッション1タスクを守り、ハンドオフドキュメントで連携する
  • 必要に応じて人間をループに介入させ、品質を担保する
  • このワークフローはPoCやMVPの構築に最適
  • Archonを使えば全ステップを自動実行することも可能
「AIコーディングはどのモデルが最強か」ではなく、「どう組み合わせるか」の時代が来た！
