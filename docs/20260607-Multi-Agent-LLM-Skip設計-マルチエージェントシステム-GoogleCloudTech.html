LLMをスキップするマルチエージェント設計

1000のAIエージェントがマラソンを走る — そのほとんどはAIを使わない

📅 2026年6月7日 | Google Cloud Tech「Hands-On AI」

①「Race Condition」プロジェクトとは

  • オープンソースで1000のAIエージェントがマラソンを同時に走るシステム
  • 3D可視化でリアルタイムに全ランナーの動きを確認できる
  • ラスベガスを舞台に26.2マイル（約42.2km）のコースを構築
  • Google Cloud TechのAnnieと、システム構築を主導したCasey Vestが解説
  • GoogleのADK（Agent Development Kit）で構築された実プロジェクト
前回はアーキテクチャ構築、今回は「LLMをいつ使うか」という核心に迫る！

②設計の核心: LLM vs 決定論的ロジック

  • 最重要判断：「いつLLMに任せ、いつコードを書くか」
  • この判断は3つの場所で行われた
  • ① プランナー — マラソンルートの計画
  • ② シミュレーター — レースシミュレーションの実行
  • ③ ランナー — 個々のランナーの行動
  • 共通パターン：LLMが得意なところで使い、そうでないところではスキップ
フルLLMがベストではない！Googleの面接でも同じ結論が出るらしい...

③マラソンルート計画の制約

  • 正確に26.2188マイルのコースを作らなければならない
  • 市の承認済み道路ネットワーク上でのみ構築
  • ラスベガスの全道路をモデリングできない — 一部データのみ使用可能
  • Google Mapsの車ルートは一方通行を考慮して不適切
  • 歩行ルートは橋や建物の中を通る問題があった
  • GeoJSON形式で道路の座標データを管理
マラソンは道路を閉鎖できるから一方通行は関係ない！Google Mapsは全く使えなかった

④最短経路はNP困難問題

  • ルート構築はNP困難問題 — CSで最も難しい問題の一つ
  • LLMだとノイズが多く、コンテキストの維持が困難
  • 制約が増えるほどLLMは混乱し精度が劇的に低下
  • 決定論的ならユニットテスト可能で結果が常に正確
  • 数十年の既存のCSアルゴリズムを使うのが正解
「これがGeminiを使わず決定論的にした最大の理由！全部CSのアルゴリズムだから」— Casey

⑤採用アルゴリズム: Spine & Sprout

  • フェーズ1: Dijkstra法 — Haversine距離で加重された辺でランドマークを接続
  • フェーズ2: 蛇行ウェイポイント — 経路を蛇行パターンで拡張
  • フェーズ3: トリミング — 最終セグメントを調整して正確に26.2188マイルに
  • 処理は完全に決定論的 — 実行のたびに同じ結果
  • Haversine距離は地球表面の球面距離を計算する公式
「複雑な数学だからこそ、決定論的にしてユニットテストできることが重要」— Casey

⑥AIはルート構築で使ったか？ — 設計時のみ

  • LLMはアルゴリズムの選定に活用 — どのアルゴリズムを使うべきかの調査
  • Google AI Studioを使用 — 高速で使いやすかった
  • コード実行を有効化 — GeminiがPythonを書いて動かせる
  • Google検索で最新ルーティングアルゴリズムを調査
  • GeoJSONと制約条件をプロンプトに渡して出発点を生成
AI Studioの成果物をGemini CLIに渡して「ユニットテスト可能にして、深いコードレビューして」と指示！

⑦研究→決定論的コードへの変換フロー

  • AI Studioでアルゴリズム調査 + プロトタイプ生成
  • Gemini CLIで制約適用 + ユニットテスト化
  • 深いコードレビューで品質を向上
  • このプロセスを何度も反復して完成度を高めた
  • 非開発者もAI Studioでアルゴリズム調査に参加できた
  • 「次からこの方法でリサーチする！」— Annieも大賛成
「LLMは設計時」で「実行時は完全に決定論的」というパターン！

⑧シミュレーターの内部設計

  • ゲーム開発由来のティックを活用 — 数ミリ秒ごとに状態計算
  • 毎ティックでシステム全体の状態を追跡
  • シミュレーター＝ゲームサーバー、ランナー＝ゲームプレイヤー
  • 全ランナーの位置・状態・プロパティを継続監視
  • ADKのワークフローエージェント — ループ、シーケンシャル構成
  • マルチエージェントでサブエージェントを含む構成
全エージェントの中で最も洗練された設計！ゲームエンジンのアプローチを採用

⑨秘密兵器: before_model コールバック

  • ADKのbefore_model コールバックが全ての呼び出しをインターセプト
  • モデルは作成に必須だが一度も呼ばれない
  • 代わりに決定論的ツール呼び出しを直接返す
  • Pre-race: 環境セットアップ — ツール呼び出しが固定
  • Tick: 「ティックを進める」→「レース完了確認」の2呼び出しのみ
  • 毎回同じ処理ならLLM不要 — でもエージェントの恩恵は維持したい
「モデルは必要だけど一度も呼ばない」— これぞハイブリッド設計の真骨頂！

⑩LLMスキップの3大メリット

  • 💰 コスト削減 — 不要なAPI呼び出し排除でトークン消費を大幅抑制
  • ⚡ レイテンシ低減 — LLM応答待ちがなくなりミリ秒単位で実行
  • 🔒 信頼性保証 — 決定論的処理は常に同じ結果を返す
  • ライフサイクル管理・コールバック・テレメトリーは維持
  • ショートサーキットで自前実装すると可視性が失われるので注意
コールバック方式ならADKの全恩恵を維持したままLLMだけスキップできる！

⑪オートパイロット: 1000人をミリ秒で

  • AutopilotRunner — RunnerAgentを継承してLLMをヒューリスティクスに置換
  • 行動データはGemini Deep Researchから収集したヒューリスティクス
  • 全てコードベースの決定論的処理 — LLMは呼ばれない
  • ミリ秒単位で実行 — 1000人の同時動作を実現
  • バックエンドは最大10,000セッションをサポート
  • フロントエンドはChrome制約で1000表示が限界
「AIが決定し、コードが実行する」ランナーを増やしてもトークンは増えない！

⑫トークン使用量: 約35万の現実

  • Gemini 3 Flash Previewで約350,000トークン使用
  • ルート計算の数学的処理からのトークンはゼロ
  • トークンはAIの判断に使用：ランドマーク選定、タイミング確認、安全性評価
  • モデルはレース前に一度だけ推論し、後は1000人が自律動作
  • ランナー追加でもトークンは増えない — スケーラビリティの秘密
「AIは判断し、コードが実行する」明確な分離がコスト効率最大化の鍵

⑬テレメトリーダッシュボード

  • 組み込みの開発者ツールUI — エージェントのライフサイクルを可視化
  • 各行がリアルなADKイベント — starting/finished/model start等
  • セッションIDでグループ化しメッセージ順序を追跡
  • Pub/Sub → WebSocketでフロントエンドにリアルタイム配信
  • レイテンシ、総イベント数、総トークン数をリアルタイム確認
  • 67ターン目など各ランナーの実行状況をデバッグ可能
Pub/Sub→WebSocketで1000セッションをリアルタイムストリーミング！

⑭Cloud Runでのスケーリング: 50インスタンス

  • Google Cloud Run上でステートレスに実行
  • ランナー用に50+のサーバープロセスが稼働
  • ロードバランサーがどのインスタンスに割り当てるかは不定
  • 最大の課題：セッション状態をどこに保存するか？
  • 50のステートレスインスタンス間で状態を共有する必要
  • これが最初に壊れた部分 — スケール時の状態管理は最難関
スケールを上げたら真っ先に壊れたのは状態共有！誰もが直面する壁

⑮セッションストア: 3つの選択肢の問題点

  • ① インメモリ — 複数インスタンスで使えない（状態がローカルに閉じる）
  • ② AlloyDB/Cloud SQL — SQLオーバーヘッドが遅すぎてボトルネック
  • ③ Agent Platform — Cloud RunではなくAgent Runtimeが必要で不適合
  • どのデフォルトオプションも要件を満たせなかった
  • 既にMemory Store（Redis）をアーキテクチャで使用中だった
「インメモリは×、DBは遅い、Platformは使えない...どうする？！ 」

⑯解決策: Redis + コミュニティの力

  • Google ADK Communityリポジトリにコミュニティ製Redis Session Serviceを発見！
  • サブクラス化してイベントプルーニング + ブロブ成長防止を追加
  • 極低レイテンシとスケーラブルなセッション管理が実現
  • 1000セッションにPub/Subで一斉送信 → 全レスポンスを高速収集
  • コミュニティコントリビューターに大感謝！オープンソースの力で壁を突破
ADK CommunityはGoogleがメンテナンス。OSSエコシステムが実プロジェクトを支える！

⑰AIスキル付きオープンソース

  • プロジェクトにAIハーネス用スキルが同梱されている
  • "Exploring the Codebase"スキル — AIにコード構造を解説
  • Gemini CLIやCursor等でコードベースを直接質問できる
  • 例：「極低レイテンシのRedisセッションサービスはどう実現した？」
  • レポはGitHubで完全公開 — 誰でも探索可能
AIコーディングツールにスキルを読み込ませるだけで、 巨大コードベースでも迷子にならない！

⑱学びとベストプラクティスまとめ

  • 「LLMが得意なこと」と「決定論的コードが得意なこと」を明確に分ける
  • LLMは設計・調査時に使い、実行時はスキップする
  • ADKのbefore_model コールバックでハイブリッド設計を実現
  • エージェントのテレメトリー・可視性を犠牲にしない
  • スケーリングではセッション状態の保存先が最難関
  • コミュニティリソースを活用 — Google ADK Communityは必見
  • 「AIが決定し、コードが実行する」がマルチエージェント設計の最適解
1000エージェントが走るマラソンで学んだ本番スケールの知見を 全てオープンソースで共有！これは必見です！
