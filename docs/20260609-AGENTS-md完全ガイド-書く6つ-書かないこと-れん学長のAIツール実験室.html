①AGENTS.mdとは何か

AGENTS.mdは、AIエージェントが作業開始時に最初に読むファイルです。Claude CodeではCLAUDE.mdという名前になり、CodexではAGENTS.mdとして扱われます。

  • ここにベースとなる指示を書いておくことが重要
  • しかし何でもかんでも書けばいいわけではない
  • 毎回の会話で最初に読み込まれる
ここに書いた内容は毎回読まれるからこそ、中身の取捨選択が大事なんです！

②書きすぎると逆効果になる

たまにしか使わない細かい手順を書いておくと、それがノイズになって肝心のルールが埋もれてしまう。結果として、本当に守ってほしい指示までAIに無視されてしまいます。

  • ノイズが増えると重要な指示が見落とされる
  • Anthropicの公式ガイドにも明記されている
  • 本当に必要なものだけを残すべき
書きすぎ＝逆効果。大事なのは何を書かないかです！

③置き場所で効く範囲が変わる

AGENTS.mdはどこに置くかで効く範囲が変わります。大きく分けて2つの置き場所があります。

  • グローバル パソコン全体の設定場所に置くと、すべての作業に効く
  • プロジェクト 特定フォルダ内に置くと、そのフォルダだけに効く
  • 両方あるときは、近い方（プロジェクト）が優先される
「近いメモが勝つ」が原則です！

④グローバル設定の場所

どのプロジェクトでも守ってほしい共通ルールは、パソコン全体の設定場所に置きます。

# Claude Code の場合
~/.claude/CLAUDE.md

# Codex の場合
~/.codex/AGENTS.md
  • すべての作業に共通するルールを記述
  • 「日本語で回答して」などの基本方針
  • 「外部に投稿しないで」などの安全ルール
ツールごとに場所が違うので注意！

⑤プロジェクト設定の書き方

案件やプロジェクトごとのルールは、そのフォルダの中にAGENTS.mdを置くだけで設定できます。

  • プロジェクトのルートに配置するだけ
  • グローバルと重複した場合はプロジェクトが優先
  • 「近いメモが勝つ」というシンプルな原則
my-project/
├── AGENTS.md  ← プロジェクト専用
├── src/
└── README.md
グローバルとプロジェクトを分けるのがコツ！

⑥書くべき①: 概要と地図

プロジェクトが何の場所なのかの説明と、主要なファイルの置き場所の地図を数行で書きます。

  • プロジェクトの目的を1〜2行で説明
  • 主要なディレクトリ構造を簡潔に記述
  • ファイルを1個ずつ説明するのはやりすぎ
全ファイルを列挙する
主要な場所だけを数行で
地図は大まかでいいんです！

⑦書くべき②: 動かし方のコマンド

ビルドやテストなどの実行手順をコピペでそのまま動く形で残します。コードの作業ではこれが一番効くとのこと。

# 例: ビルドとテスト
npm run build
npm test

# 例: 開発サーバー起動
npm run dev
  • プログラミングする人向けの項目
  • 正式な形で実行コマンドを記述
  • コード作業で最も効果を発揮する
コード書かない人はスキップしてOK！

⑧書くべき③: 自分たちだけの決まり

デフォルトと違うルールだけを書きます。世間の標準ルールはAIがすでに知っているので書く必要がありません。

  • 独自のコーディング規約など
  • 文章でダラダラ説明するより、お手本を1つ置く方が伝わる
  • 標準的な慣習は書かない
「PEP8に従って」など標準ルールを書く
うちだけの独自ルールを書く
AIが知ってることは書かないのがポイント！

⑨書くべき④: リポジトリーの作法

プロジェクトの管理フォルダーの変更を記録する際のルールを書きます。

  • コミットメッセージの書き方
  • 「頼まれるまで勝手に確定・保存しない」
  • 完了報告で何を伝えるか
# 例: コミットメッセージのルール
feat: 新機能の追加
fix: バグ修正
docs: ドキュメント更新
お作法を書いておくと事故が減る！

⑩書くべき⑤: 触ってはいけない境界線

絶対にやってほしくないことを明確に書きます。AIにとってのガードレールです。

  • 機密情報（APIキーなど）には触らせない
  • 本番環境の設定は変更禁止
  • 外部への勝手な投稿や送信を禁止
これは絶対に書いておくべき安全のフェンスです！

⑪書くべき⑥: 知らないと事故る罠

そのプロジェクト特有の事故りやすいポイントや癖を書いておきます。

  • ハマりどころや注意点
  • 一見正しく見えて実はダメなパターン
  • 過去に事故った経験から得た教訓
コード書かない人は①③⑤⑥を抑えれば十分！

⑫書かない①: 読めば分かること

公式が明確に「書かないもの」も挙げています。まずは読めば分かることは不要です。

  • コードを見れば分かること → 書かない
  • 世間の標準的な慣習 → AIがすでに知っている
  • 一般的なルール → わざわざ書く必要なし
「変数名はsnake_caseで」
標準ルールはAIが既に知っている
AIが知ってることを書くのはムダなんです！

⑬書かない②: 変わる情報はURLで

API仕様など時間的に変化していく情報を直接書くと、古い情報を参照して間違えてしまいます。

  • APIの仕様は頻繁に変わる
  • URLを書いてそこから参照してもらう
  • 規約や仕様が書いてあるページをリンク
# 悪い例
APIの認証はOAuth2.0で...

# 良い例
API仕様はこちらを参照:
https://docs.example.com/api
URLを書くだけでOK！

⑭書かない③: 手順はスキルに逃がす

長い手順やたまに使う作業は、「スキル」という仕組みに逃がします。AGENTS.mdには書きません。

長い手順
スキルに移動
AGENTS.mdはスッキリ
  • 料理のレシピのようなもの（必要な時だけ呼び出す）
  • AGENTS.mdが膨らまない
  • コンテキスト窓を圧迫しない
毎回読むファイルだからこそ短く！

⑮両方使うなら正本を1つに

CodexとClaude Codeでは見るファイルが違います。普通にやると同じ内容が2つに分散してしまいます。

  • Codex → AGENTS.md を見る
  • Claude Code → CLAUDE.md を見る
  • 分散するとルールがズレていく
2つのファイルに別々に書く
正本を1つにまとめる
片方だけ古くなると事故の元！

⑯CLAUDE.mdは転送するだけ

AGENTS.mdが正本で、CLAUDE.mdは「ナイフ（写し）」にするのがベストプラクティスです。

# CLAUDE.md の中身（これだけ！）

共通指示は AGENTS.md を
見てください。
  • CLAUDE.mdには「AGENTS.mdを見て」とだけ書く
  • AGENTS.mdに本当のルールを集約する
  • ツールを変えても使いやすい
ルールがバラけないし、メンテもラク！

⑰よくある失敗: スキル一覧の二重管理

AGENTS.mdにスキルの一覧を書いておくと親切と思いがちですが、これが二重管理の罠になります。

Before:
12個のスキル表が入っていた。スキルとAGENTS.mdで内容が重複。
After:
方針を1行だけ残してスキル一覧は削除。
  • スキル内容と重複してしまう
  • スキルを更新するとズレが生じる
  • 一覧は消して、方針だけ1行書く
二重管理は確実にズレるのでやめましょう！

⑱まとめ: 足すより、引く

今日のポイントは3つです。AGENTS.mdを効かせるための黄金ルールです。

  • 1 AGENTS.mdは毎回読まれる → 広く常に聞くものだけを入れる
  • 2 判断基準は「よく使うか」ではなく「消したらAIが事故るか」
  • 3 スキルの一覧は不要 → 手順はスキルに逃がす、方針だけ書く
何を書きたすかより何を書かないかが大事！まずは1つ消してみよう！
