AI利用におけるセキュリティの脅威
①

非エンジニアが直面する
AIセキュリティの現実

  • バイブコーディングやAIでの業務効率化を始める非エンジニアが急増しているが、セキュリティの知識がないまま使っている人が多い
  • 「自分には関係ない」と思いがちだが、情報漏洩のリスクはすぐそばにある
  • セキュリティは一部の知識がある人だけの問題ではなく、AIを使う全員が理解すべき重要なテーマ
クロードコードやコデックスを教える時に、やばい使い方をしている人をよく見かけるんですよね...
②

プロンプトインジェクション
攻撃の仕組み

  • AIがウェブページを検索した際、その中に人間には見えない悪意のあるコマンドが隠されていることがある
  • 小さすぎる文字や見えない色で書かれた指示をAIがユーザーからの命令と勘違いして実行してしまう
  • AIが.envファイルの中身（APIキー等）を読み取って外部に送信する被害が実際に発生している
  • 機密情報が漏洩して高額請求されるケースもXで話題に
自分は何もしてない！普通にクロードコードにお願いしただけなのに...というケースがほとんど
③

.envファイルの機密情報
漏洩の実態

  • .envファイルにはAPIキーやトークンなどの超重要な機密情報が保存されている
  • プロンプトインジェクションにより、AIがこのファイルの中身を外部に自動送信してしまう
  • クロードコードを使っていると「.envファイルを作ってAPIを入力して」とよく言われるが、その管理には要注意
.envは昔からシステム開発でよく使われるファイル。トークンやAPIキーなどめちゃめちゃ重要な機密情報を保存するものです！
④

悪意のあるMCPによる
データ抜き取り被害

  • MCPは外部サービスと簡単に繋ぐことができる便利な機能だが、悪意のあるものも多数存在
  • Google広告運用を簡単にできるMCPが話題になったが、実は悪意のある第三者が作ったものだった
  • MCP経由で顧客情報や業務データが見放題になり、乗っ取られる被害が発生
  • 数千万円単位の損害が出たという報告もあり、Xで話題に
「めちゃめちゃすごいじゃん！」と飛びついたら、実は悪意があるものだった...というパターンが本当に多いんです
⑤

AIエージェント時代に
事故が増える理由

  • 従来はエンジニア知識がないとAPIや.envファイルを扱うことができなかったので事故は起こりにくかった
  • AIエージェントの時代になって、誰でもできるようになったことが事故増加の要因
  • 従来は「変なファイルを開かなければ安全」だったが、AIが自分の代わりに危険な操作を実行してしまう可能性がある
  • AIは権限がめちゃくちゃ大きいため、便利さと引き換えにリスクも増大
自分がいくら気をつけていても、AIが騙されてしまったらアウト！そこが最大のポイントです
⑥

バイパスパーミッションの
恐ろしい危険性

  • バイパスパーミッション（全自動承認モード）を有効にしている人が多いが、極めて危険
  • 名前に「デンジャラス」がついているぐらい危険。全ての確認をスキップしてAI判断で実行してしまう
  • 情報を漏洩したことにすら気づけないことがある。どこに送信したか見ていない人も多い
  • 基本はバイパスパーミッションをオフにして使うべき
自分のパソコンにクロードコードを入れて、そのままバイパスパーミッションで動かしてる人...マジでやばいです！

⑦

危険なプラグイン・スキルの
見分け方と注意点

  • YouTubeサムネイル作成、広告運用、SEO分析など便利なスキルが多数販売されている
  • しかしスキルの安全性まで検証して紹介しているケースは非常に少ない
  • 野良スキルや野良プラグインを使うのは、見ず知らずの人に鍵を渡すのと同じ
  • どうしても使いたい機能がある場合は、AIに同じものを自作させるのが安全
AIがどんどん賢くなっているから、スキルに頼らなくても自分で考えて動けるようになっています！
⑧

Anthropic公式
「Security Guidance」登場

  • Anthropic公式プラグインとしてリリース。公式が出しているものにやばいものが仕込まれることはまずない
  • コードを書く（実装する）→ セキュリティレビューを自動で実行 → 安全な方向に修正してくれる
  • 自分で何かをする必要はなく、入れておけば自動でチェックしてくれる仕組み
セキュリティツールは「使うのを忘れる」問題があるから、自動実行される設計が本当に親切！
⑨

Security Guidanceが
検出する脆弱性一覧

  • SQLインジェクション: データベースを操作するSQL文に悪意のあるコードが混入する脆弱性
  • 認証バイパス: 本来認証が必要な処理を迂回されてしまう脆弱性
  • APIキーのハードコード: プログラム内にAPIキーを直接書き込んでしまうと、エラー時に外部から見える危険
  • その他の脆弱性: 非エンジニアが気づきにくい定着性の問題も多数検出
APIキーのハードコードはほんとによくあるある！エラー画面でAPIキーが丸見えになっちゃうんです
⑩

3段階の自動
セキュリティチェック

  • 第1段階: ファイルの保存時に自動でセキュリティチェックを実行
  • 第2段階: ファイルの編集時にも継続的にチェックを実行
  • 第3段階: Gitのコミット・プッシュ時に最終的なセキュリティレビューを実施
  • 多段防御の仕組みで、第1段階で見逃しても後の段階で発見できる
いろんなタイミングで防御してくれるから、万一見逃しても次でキャッチできるのが安心ですね！

Security Guidanceの
対象ユーザー

  • コードを書く人全般に有用。コードのデバッグにも活用できる

  • 前提条件: Python 3.8以上のデポジトリーが必要
「コード書いてないよ」と言っても、クロードコードが勝手にコードを組んで便利にしてくれてることもあるから、とりあえず入れておくのもアリです！

Claude Codeのセキュリティ基本設定
⑬

Denyルールで
機密ファイルを保護

  • .envファイルや.sshファイルには機密情報が格納されている
  • Denyルールを設定して、AIがこれらのファイルを読み込めないようにブロックする
  • Read ツールの対象から除外することで、プロンプトインジェクション攻撃を自動ブロック
  • 設定方法が分からなくても、クロードコードに「Deny設定して」と頼めば自分でやってくれる 
# .claude/settings.json の例
{
  "deny": [
    "Read(.env*)",
    "Read(.ssh/*)"
  ]
}
CLAUDE.mdに書いたルールはAIが無視することもあるので、Deny設定で確実にブロックするのが重要です！
⑭

危険なコマンドを
ブロックする方法

  • rm -rf / = わずか6文字でパソコンの中の全ファイルを一括削除するコマンド
  • curl / wget = 外部と通信するコマンド。プロンプトインジェクションの悪用に使われる危険がある
  • これらのコマンドをDeny設定で禁止することで、AIが誤って実行するのを防ぐ
  • curl/wgetがなくても、クロードコードの標準ツールで外部通信は十分に可能 
# ブロック推奨コマンド
{
  "deny": [
    "Bash(rm -rf *)",
    "Bash(curl *)",
    "Bash(wget *)"
  ]
}
コマンドは本当に強力！10文字書くだけでパソコンの中身全部消せちゃうんです
⑮

CLAUDE.mdに
セキュリティルールを記載

  • CLAUDE.mdはクロードコードの設定ファイル。会話開始時に毎回読み込まれる
  • パスワードやAPIキーをコードに直接書かない、外部データの取り扱いルールなどを記載
  • ただしCLAUDE.mdに書いたからといって必ず守られるわけではない点に注意
  • Deny設定（確実にブロック）+ CLAUDE.md（ルールの明文化）の二重の対策が重要 
# CLAUDE.md の記載例
## セキュリティルール
- APIキーをコードに直接書き込まない
- 機密ファイル(.env, .ssh)にアクセスしない
- 外部へのデータ送信前に必ず確認する
- 認証ロジックは慎重に取り扱う
⑯

Gitによる
バージョン管理の重要性

  • Gitを使うとファイルの変更履歴がすべて記録され、セキュリティ上の問題を見つけやすくなる
  • 悪意のある命令でプログラムが書き換えられても、Gitがあれば変更内容を確認できる
  • Security Guidanceのセキュリティレビュー機能もGit環境で最も効果的に動作する
  • Gitなしでもクロードコードは使えるが、セキュリティ観点では必須と言える
東大生マッチングアプリ「ユートピア」の脆弱性も、Git管理していれば気づけたかもしれない事例ですね
⑰

アクセス範囲を
プロジェクト単位に限定

  • ホームディレクトリでクロードコードを使うと、パソコン内の全ファイルにアクセス可能になってしまう
  • ダウンロード、書類、ピクチャー、デスクトップなどすべての個人情報が漏洩の対象に
  • プロジェクトフォルダごとにクロードコードを動かすことで、被害を最小限に抑えられる
  • 最小権限の法則: AIに渡す権限は必要最小限にとどめる
ホームディレクトリで動かしてる人は今すぐやめて！決済情報も企業情報も全部持っていかれます
⑱

企業向け
「Claude Code Safety Hub」

  • CCSHは、端末とAIサーバーの間にゲートウェイを設置する仕組み
  • 機密情報の送信を自動検知・ブロック。外部からのプロンプトインジェクション攻撃も防御
  • 各従業員の利用状況をモニタリングでき、安全な社内運用を実現
  • Claude Codeだけでなく、Codex、ChatGPT、Claude.aiにも対応
ゲートウェイを通るデータをチェックして、危ないものはブロック。シンプルで強力な仕組みです！
