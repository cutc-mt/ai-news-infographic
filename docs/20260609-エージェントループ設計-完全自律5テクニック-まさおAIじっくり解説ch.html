AIエージェントは「ループ設計」の時代

完全自律で回すためのループ設計の4ポイントと5つのテクニック
2026年6月9日

①オープニング: ループ設計とは何か

2026年6月、AI界隈で「エージェントのループ設計」が大きな話題になっている。

  • OpenClawを開発したPeter Steinbergerさんが提唱
  • 「エージェントにプロンプトを与えるのではなく、ループそのものを設計するべきだ」と主張
  • Claude Codeの中の人もインタビューで言及し、さらに注目が集まっている
これからはプロンプトではなく、ループを設計する時代だ！

②エージェント進化の歴史: 3世代の変遷

第1世代（2023年）
人間が手でコードを書くのが普通だった時代
第2世代（2024年）
タブ補完が中心。CursorのComposer β版が登場し「これヤバい」と話題に
第3世代（2026年）
ループ設計の時代。エージェントにタスクを任せ、ループで回すのが当たり前に
Opus 4.5が分岐点だったと感じる — モデルの自立性が急激に高まった

③なぜ今ループ設計が重要なのか

より完全自律に向けて進化していく中で、振り返りとフィードバックが不可欠になっている。

  • コンテキストウィンドウが伸び、より多くの情報を保持できるように
  • 指示追従性が向上し、エージェントが意図を正確に理解できるように
  • ロングコンテキストでも情報を失いづらくなった
  • モデルの自律性が高まり、ループを回すだけで良い仕事ができるように

④ループ設計の4つのポイント: 概要

ループを設計する上で押さえるべき4つの要素がある。

  • 1 ゴールの明確化 — 計測できる形で目標を設定
  • 2 エージェントの割り当て — スキルやサブエージェントを定義
  • 3 進捗の確認 — Todoリストやタスク管理で進行を把握
  • 4 継続条件の判定 — 合格基準を満たすまでリトライ
変なゴールや変な検証機を作ると、意図からずれていくので注意！

⑤ポイント1: ゴールを明確にする

ループ設計の最初のステップは、計測できる形でゴールを設定すること。

  • LLMジャッジで合格を判定する方法も有効
  • 3体のLLMが合格と言うまで繰り返す手法も存在
  • ゴール条件の質が、成果物の品質に直結する
// ゴール設定の例 goal: { "quality_score": >= 90, "llm_judge": 3体中3体合格, "max_retries": 5 }

⑥ポイント2: エージェントを割り当てる

目標に対して、どのエージェントに何を実行させるかを定義する。

  • スキル定義 — エージェントが実行できる機能を明確化
  • サブエージェント — タスクを分割して複数のエージェントに割り当て
  • エージェント間の連携と役割分担を設計
スキルとサブエージェントの定義が、ループの品質を左右するよ！

⑦ポイント3&4: 進捗確認と継続条件

エージェントの進捗を確認し、継続するかどうかの条件でループを制御する。

  • Todoリストやタスク管理ツールで進捗を可視化
  • 70点なら不合格 → 自動でリトライが走る仕組み
  • 合格基準を満たしたらループを終了
while (!goal.met()) { result = agent.execute(task); score = evaluate(result); if (score >= threshold) { break; // 合格! } task = refine(result, feedback); }

⑧Boris Chernyさんが語る自律化の5テクニック

Claude Codeの中の人であるBoris Chernyさんが、完全自律に向けた5つのテクニックを公開。

  • 1 オートモード（自動承認）
  • 2 ダイナミックワークフロー
  • 3 ループコマンド＋スリープ防止
  • 4 自己検証ツールの搭載
  • 5 Ralphループの活用
これらを組み合わせることで、長時間の自律実行が可能になる！

⑨テクニック1: オート承認モード

エージェントをより自律的に動かすための第一歩は、自動承認（オートパーミッション）の設定。

  • エージェントがファイルの読み書きやコマンド実行を自動で承認
  • 人間の介入を最小限にして、連続的な処理を可能にする
  • セキュリティを考慮しつつ、許可する操作の範囲を設定
承認を自動化するだけで、エージェントの動きが劇的に変わる！

⑩テクニック2: ダイナミックワークフロー

最近登場したダイナミックワークフローは、エージェントの能力を飛躍的に高める。

  • Claude Codeでultracodeなどのモードを呼び出せる
  • 大量のサブエージェントが並列で動く仕組み
  • ロングランのタスクでより高品質な成果物を生み出す
サブエージェントが大群で動くのは、かなり強力なパターンだ！

⑪テクニック3: ループコマンド＋スリープ防止

エージェントが止まらずに動き続けるための環境を整えることも重要。

  • ループコマンド — 60分間隔などで定期実行を設定
  • クラウド版のClaude Codeを使う方法
  • ローカルではAmphetamineというMacアプリがおすすめ
  • MacBookを閉じても稼働し続ける設定が可能
スリープ対策をしないと、長時間ループが途中で止まってしまう！

⑫テクニック4: 自己検証ツールを持たせる

Claude公式も推奨する「自分自身で検証できるツールを渡す」という手法。

  • スクリーンショットを取らせて、自分のミスに気づかせる
  • レイアウトが崩れているスクショを見れば「100点」とは言えない
  • エージェントが正しい仕事をしているか自己確認する仕組み
視覚的なフィードバックを与えることで、エージェントの勘違いを大幅に減らせる！

⑬テクニック5: Ralphループ

最近話題になっているRalphループは、Claudeのプラグインにも登場している。

  • オフィシャルプラグインとして簡単に導入可能
  • ループ設計を手軽に体感できる
  • 定期的な間隔でタスクを自動実行する仕組み
プラグインに入れるだけですぐにループ設計を試せる！

⑭ゴールとループの違いを理解する

ループとゴールは似ているが、動き方が違う。

  • ループ: 定期的な間隔でもとにタスクを繰り返す仕組み
  • ゴール: 別モデルがゴール条件を定義し、満たしているかを判定
  • エージェントループ = 目標から継続条件で繰り返す新しい概念
// ループ: 定期実行 loop.run(task, interval="60min") // ゴール: 条件達成まで goal.achieve(target, judge="model") // エージェントループ: 両方の融合 agent_loop.run(goal, retry_on="fail")

⑮コミュニティの指摘と残る課題

Peter Steinbergerさんの発言に対し、コミュニティから鋭い指摘も上がっている。

  • 「ループを設計するエージェント自体が欲しい」という意見
  • ループはエージェントの動き方を直すが、知っている内容は直さないという問題
  • 知識レイヤーが欠けていると、ループもうまく機能しない
  • 本質的に様々な課題が残っている
ループ設計だけでなく、知識の質も重要という鋭い指摘！

⑯Opus 4.5が分岐点だった理由

ループ設計が現実的になったのは、Opus 4.5の登場が大きな転換点だった。

  • モデルの自立性が急激に高まった
  • その後Opus 4.6 → 4.7 → 4.8と順調に進化
  • 自立的に動いて品質を保てるレベルに到達
  • 進化のスピードが予想をはるかに超えている
AIの進化が自分の予想以上に速くて驚きを隠せない！

⑰結論: 完全自律に向けたループ設計

ループを設計しておけば、エージェントが自立的に回っていく時代が来ている。

  • モデルが十分賢くなったので、あまり縛らなくても動く
  • 適度な制約をつけることで、高品質な自律動作が可能に
  • 検証ツールを自分で作ることで、さらに精度が向上
  • エージェントに指示を出さなくても勝手に動くように
完全自律の切り口があってこそ、より高度なハーネスが作り込める！

⑱まとめ: 自分の手をどこまで離せるか

最終的な目標は、どれだけ自分が手を離して、高品質にエージェントを動かせるかを追求すること。

  • ループ設計の4つのポイントを押さえる
  • 自律化の5つのテクニックを組み合わせる
  • ループを意識してハーネスを作り、日常的に活用する
