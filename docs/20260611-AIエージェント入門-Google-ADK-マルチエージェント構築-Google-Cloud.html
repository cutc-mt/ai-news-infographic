AIエージェントとは？実践ガイド

Google ADKを使って最初のマルチエージェントシステムを構築する
📅 2026年6月11日作成

① AIエージェントの定義: 答えるだけでなく決断する

  • AIエージェントとは、ただ答えるだけでなく、決断して行動できるソフトウェアのこと
  • 従来のチャットボットは単一の応答を生成するだけ。エージェントはリクエストを見て、何が必要かを判断する
  • APIを呼び出したり、コードを実行したり、その結果を見て次の行動を決める
  • この「推論→行動→観察→調整」のサイクルがモダンAIエージェントの基盤
チャットボットとの違いは
自律的に行動できるかどうか！
ただ喋るだけじゃない。

② ReAct論文: 推論と行動の統合

  • 「ReAct」論文 — Synergizing Reasoning and Acting in Language Models がエージェントの理論的基盤
  • 言語モデルはテキストを一括で生成するだけでなく、ステップバイステップで推論できる
  • ツールやAPIの呼び出しという「行動」をとり、結果を「観察」してから次を決める
  • この推論・行動・観察のループこそが現代のAIエージェントの設計原理
ReAct = Reasoning + Acting
この2つを統合するアイデアが
すべてのエージェントの原点。

③ エージェントの3要素: 推論・計画・記憶

  • 推論能力 — 与えられた情報から論理的に判断を下す
  • 計画能力 — 複雑な目標を小さなタスクに分解して実行順序を決める
  • 記憶能力 — 過去のやり取りや結果を覚えて、将来の行動に活かす
  • これらにユーザーに代わって自律的に意思決定する能力が加わって初めてエージェントと呼べる
推論・計画・記憶の3つが
揃ってはじめて自律型AI。

④ 順次型エージェント: 流れ作業のように

  • ステップ1→ステップ2→ステップ3と、決まった順序で実行するパターン
  • 予測可能でデバッグしやすいのが利点。組み立てラインのようなイメージ
  • ただし柔軟性は低い。シンプルで予測可能なフローに最適
  • データパイプラインや定型処理などに適している
1→2→3と進むだけなら
ワークフローツールで十分。
でもエージェントの基礎。

⑤ 反応型エージェント: 状況に応じて柔軟に

  • 現在の状態を見て「次は何をすべきか」をその場で判断するパターン
  • 今回はツールA、次はツールBと柔軟に切り替えられる
  • 動的な状況に対応できるのが強みだが、事前の計画はしない
  • リアルタイムのデータ分析やチャットボットなどに適している
その場の判断で動くから
予測不可能だけど
柔軟性はピカイチ。

⑥ 計画型エージェント: まず計画を立ててから実行

  • いきなり行動するのではなく、まず全体の計画をスケッチしてから実行に移る
  • 旅行の例：いきなり航空券を買わず、日付・ホテルを決めてから順番に手配する
  • 複数ステップ・依存関係のある目標に最適。全体像を見てから動ける
  • 複雑なタスクをこなすなら、このパターンが一番強力
「まず計画を立てる」
人間と同じですね。
旅行の計画がいい例！

⑦ パターンの選び方: 問題に合わせて最適な設計を

  • シンプルで予測可能なフロー → 順次型がベスト
  • 動的で変化の多い状況 → 反応型が適している
  • 複数ステップ・依存関係がある目標 → 計画型が圧倒的に有利
  • 重要なのは「どれが正しいか」ではなく「問題に合わせて選ぶこと」
正解は1つじゃない。
問題の性質で選ぶのが
一番大事な判断。

⑧ Google ADK: エージェント開発キット

  • Google Agent Development Kit (ADK) — エージェント構築・デバッグ・デプロイ用のオープンソースフレームワーク
  • Python、TypeScript、Go、Java、Kotlinの複数言語に対応
  • LLMエージェント、ループエージェント、カスタムエージェントの3つのカテゴリ
  • ベースエージェントを継承して独自のカスタムエージェントも作成可能
ADKはGoogleのAgentspaceや
CESでも使われてる大本命。

⑨ 実践プロジェクト: ブログ執筆エージェント

  • トピックを渡すと、構造化アウトライン→全文ブログ→代替タイトル提案まで自動で生成
  • プランナーとライターはLLMエージェント（モデル駆動）で、ループエージェントでラップする
  • バリデーションが失敗するとループエージェントが自動で最大3回リトライ
  • 安全ネット付きの自動生成システムで、品質を保証しながら効率的に執筆
トピック渡すだけで
ブログ1本完成！
自動リトライ付きで安心。

⑩ 環境セットアップ: UVとADKのインストール

  • まずUV（Pythonパッケージマネージャー）をインストール
  • 次にGoogle ADKをインストールして準備完了
  • IDEで agent.py ファイルを作成して、必要なライブラリをインポート
  • モデルの設定は環境変数ファイル（.env）から読み込む設計
pip install uv google-adk
UVは高速な
Pythonパッケージマネージャー。
pipより速いよ。

⑪ Blog Planner: アウトライン生成エージェント

  • LlmAgentクラスで定義。名前は「blog_planner」、モデルは環境変数から取得
  • instructionでMarkdown形式のアウトラインを指定：タイトル・導入・4〜6セクション・結論
  • output_key = "blog_outline" — 出力を共有ステートに保存して次のエージェントが使えるようにする
  • 詳細な指示を出力形式に明記することで、生成結果の信頼性が大幅に向上する
出力形式を明確に指定するのが
信頼性を上げるコツ！
曖昧な指示はダメ。

⑫ バリデーションチェッカー: 品質を自動保証

  • アウトライン検証用の第2のエージェント。新しく書くのではなくチェックだけを行う
  • ステート内のアウトラインにタイトル・導入・4〜6セクション・結論があれば「OK」
  • 何かが欠けていれば「RETRY」と何が不足しているかを説明
  • この厳格なチェックによって、OK/RETRYの自動判定が可能になる
2つのエージェントで
作成と検証を分離。
責任分担が明確になる。

⑬ Loop Agent: 自動リトライで堅牢なプランニング

  • プランナーとチェッカーをLoopAgentでラップして「robust_blog_planner」を作成
  • 最初にプランナーを実行→チェッカーが検証。OKなら完了、RETRYなら再実行
  • 最大3回までリトライ可能。モデルが要件を忘れてもチャンスがある
  • この安全ネットのおかげで、品質を保証しながら自動生成を実現
3回リトライ＝安全網。
1発で決まらなくても
チャンスがある！

⑭ Blog Writer: アウトラインを全文ブログに変換

  • 共有ステートのblog_outlineを読み取り、フルのMarkdownブログ記事に変換する
  • ターゲット読者はソフトウェアエンジニア。基礎はスキップして実践的なインサイトに集中
  • 「方法と理由」の両方を説明し、短いコードスニペットを適宜含める
  • アウトラインの構造に従って適切な見出しで構成。完成原稿は「blog_post」に保存
エンジニア向けに
実践的で深い内容を。
コード例も入れてね。

⑮ 記事バリデーション: 執筆品質の最終チェック

  • blog_post_validation_checker — 完成した記事の品質を検証する第4のエージェント
  • 記事に導入・セクション・結論・明確な説明があるかを確認
  • 合格なら「OK」、不合格なら「RETRY」と修正点をリスト
  • ライターとチェッカーもLoopAgentでラップして「robust_blog_writer」に。最大3回の自動リトライ
プランも執筆も
二重の安全網で守る。
ADKのループが便利！

⑯ ツール化: ループをツールとしてラップ

  • robust_blog_plannerとrobust_blog_writerをツールとしてラップするのが次のステップ
  • これによりルートエージェントが内部構造を気にせず、ツールとして直接呼び出せる
  • カプセル化の恩恵。プランニングの内部ロジックを知らなくても利用可能
  • ツール化することで複雑なサブシステムを簡単なインターフェースで提供
中身を隠して
シンプルなインターフェースに。
おしゃれな設計！

⑰ ルートエージェント「Blogger」: 全体を統括

  • ルートエージェント「blogger」が全体ワークフローを統括するオーケストレーター
  • ユーザーからトピックを受け取ると、まずプランナーツールでアウトラインを作成
  • 次にライターツールで全文ブログを執筆
  • 最後に代替タイトル3案とツイート風フック2案を追加して完成
トピック→プラン→執筆→仕上げ
自動ワークフローの完成！
人間の手が要らない。

  • ローカルのADK Webリンクを開いて、トピックを入力すればサブエージェントが協力して自動でブログを生成
  • 今回の成果：概念→コード→実行まで一気通貫でエージェントを理解・構築
