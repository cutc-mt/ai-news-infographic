  • 2026年現在: ドキュメントを1秒も読む必要なし。コマンド1つでCLIとスキルをインストール

③ Agent CLIの基本: CLI＝能力、スキル＝指示

  • CLIはコーディングエージェントに与える「能力」。プロジェクトの足場構築、評価、プロンプトの最適化などを実行
  • スキルはCLIの使い方をコーディングエージェントに教える「指示書」
  • CLIが能力、スキルが指示。この組み合わせが圧倒的に強力
  • READMEから確認できるコマンド: scaffold、evaluate、single prompt実行、prompt最適化
# 主要なCLIコマンド
$ agent scaffold    # プロジェクト作成
$ agent evaluate    # エージェント評価
$ agent run         # プロンプト実行
$ agent optimize    # プロンプト最適化
コマンドは自分で打たない。
AIが全部やってくれる！

④ スキルの仕組み: ドキュメントが組み込み済み

  • スキルにはGoogle ADKのドキュメントがすべて組み込まれている。コーディングエージェントが自動的に参照
  • エージェント構築の各段階（scaffold、evaluate、deployなど）に対応するスキルが存在
  • コーディングエージェントはエンドツーエンドのプロセスを自動で駆動。ユーザーの指示と仕様に従うだけ
  • 他のフレームワークのスキルは開発止まりだが、Agent CLIはデプロイ＋本番監視まで完備
ドキュメントを読む手間ゼロ。
スキルに全部詰め込まれてる

⑥ 構築するエージェント: Ask Your Data

  • CSVデータをアップロードして質問すると、Pythonコードを生成・実行して回答するエージェント
  • デモ用にシンプルに設計だが、複雑なエージェントにも同じプロセスが適用可能
  • コード実行を選択した理由: 本番環境での信頼性確保が課題だから。ADKのセキュアなサンドボックスが活きる
  • ADKのビルトインサンドボックスを使用。任意のコードを実行しても本番環境で安全
コード実行はセキュリティが命。
サンドボックスで隔離！

⑦ ビルドプロセス: Claude Codeが自動で構築

  • プロンプトを貼り付けると、Claude CodeがCLIワークフロー、scaffold、ADKコードのスキルを自動ロード
  • エージェント自ら質問を返してくる。「どのコードエグゼキューターを使いますか？」→ ビルトインサンドボックスを選択
  • あとはClaude Codeが実装を一気に駆け抜ける。5〜10分でエージェント完成
  • 実運用では仕様書の作成を先に行うことを推奨。simpleなものならワンショットでも可能

⑨ 部屋の象: フレームワークはまだ必要？

  • よく聞かれる質問: コーディングエージェントSDKがあるのに、なぜADKやLangChainが必要？
  • コーディングエージェントSDK（Antigravityなど）は強力だが、トークン効率が悪くレスポンスが遅い
  • 個人のセカンドブレイン用途ならSDKで十分。出力品質を重視すればOK
  • しかしプラットフォームへの組み込みや本番環境ではフレームワークがほぼ必須
用途で使い分けるのが大事。
個人 vs 本番

⑩ 4秒ルール: 速度とスケーラビリティが鍵

  • 研究によれば、ユーザーがエージェントからの最初のトークンを4秒以内に受け取れないと離脱する可能性が高い
  • ADKなどのフレームワークはミニマリスティック。プロンプトとツールを制御できるから高速・軽量
  • Agent CLIで構築したエージェントは数百万ユーザーに対応可能な環境にデプロイできる
  • トークン効率が良い = コスト削減にも直結。大量リクエストには必須の要件
4秒遅いだけでユーザーは逃げる。
本番環境では速度命！

⑫ 評価システム: 自然言語でevalを実行

  • 新しい会話で「ADKスキルを使って構築したエージェントを評価して」と自然言語で指示するだけ
  • 構築時の会話コンテキストは不要。既存のコードベースを自動認識して評価を実行
  • エージェント構築時に自動生成されたテストケースを実行。デフォルトは3テストケース
  • 各テストでエージェントにクエリを送り、期待値と一致するかを検証
評価結果が悪ければ自動で反復改善。
プロンプトやツールを調整！

⑬ プロンプト最適化: 自己反復で改善

  • 評価データを基にプロンプトを自動的に最適化する機能がCLIに組み込まれている
  • システムプロンプト、ツールの定義、エージェントの挙動を評価結果に応じて調整可能
  • デプロイ前の品質保証として重要。eval → 改善 → 再evalのサイクルで信頼性を向上
  • デプロイまでの全プロセスがCLI + スキルで駆動されるため、一貫したワークフローを維持
人間が手動でプロンプト調整？

⑭ デプロイメント: 「デプロイして」の一言で完了

  • 「このエージェントをデプロイして」と言うだけでGoogle Cloudへ自動デプロイ

  • 約6分でGCPにインスタンスが構築され、エージェントが本番環境で稼働開始
  • 設定はカスタマイズ可能。コンソールのプレイグラウンドURLでWebからテストもできる

⑮ 本番環境の機能: 監視・トレース・権限管理

  • メモリ機能: 本番環境でもエージェントが会話の記憶を保持
  • 評価ビルトイン: 本番環境に統合されたeval機能で継続的な品質監視
  • フルトレーサビリティ: トレース機能でエージェントの各ステップを監査可能
  • コード実行のサンドボックス: 本番環境でも安全に任意のコードを実行
  • エージェントのアイデンティティ: ユーザーやサービスアカウントと同じように権限を設定可能
