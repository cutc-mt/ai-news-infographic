OpenRouter Fusion の「Fable超え」は過大宣伝だった

2026年6月14日 / AICodeKing

Compound Model・DracoBench検証・実タスクで振るわないFusion APIの実態を解剖

① Fusion APIの華麗なデビュー: 市場最賢を宣言

  • OpenRouterが新API「Fusion」をローンチ
  • 公式に「市場で最も賢いコンパウンドモデル」と自称
  • 単一モデルではなく複数モデルを統合した新しい形式
  • Fable相当の知能を半額で提供すると主張
  • 発表時から派手なマーケティングが話題に
「最も賢い」という言葉の裏付けをこれから検証します🔥

② Fable競合は誰? 答えはOpenRouter

  • Fable対抗ならOpenAIやDeepSeekを思い浮かべがち
  • 実際に出してきたのはルーター企業のOpenRouter
  • FusionはモデルではなくAPIとして設計
  • 既存モデルを束ねるコンパウンド方式を採用
  • ルーティング事業者がモデル開発領域に踏み出した形
モデルを自作せず既存モデルを組み合わせるアプローチ🤔

③ コスト訴求: Fable同等を半額で?

  • 「Fable level知能を半額で」が売り文句
  • 性能を保ったまま価格を半分に圧縮すると主張
  • ベンチマークの数字を前面に出して優位性を強調
  • しかし測定範囲には重大な限定があった
  • コスト比較だけでも実タスクでは逆転する可能性
「半額」も条件次第では逆転するので注意⚠️

④ DracoBench比較: Fable 5は66%どまり

  • 公開ベンチでFable 5は66%を記録
  • Opus 4.8 単体 FusionはFable 5を上回る
  • Opus 4.8 + GPT 5.5のFusionはさらに高得点
  • Fable 5 + GPT 5.5のFusionが最高得点を獲得
  • モデルを重ねるほどスコアが伸びる設計に見える
確かに数字だけ見ればFusionが勝っています📊

⑤ ベンチの中身はDeep Research特化

  • 使用ベンチはPerplexity作成のDracoBench
  • ドメインはDeep Researchタスクに特化
  • コーディングや数学などの総合能力は未測定
  • 限定分野で勝っただけで「全てで勝つ」と主張
  • 誤解を招くマーケティングとの指摘が続出
Deep Researchが得意なだけでは??🚨

⑥ Fusion構成: 複数モデルを重ねて強化

  • 基本単位はOpus 4.8
  • そこにGPT 5.5を加えた2モデルFusion
  • Opus 4.8 + Gemini 3.1 Pro + GPT 5.5の3モデル構成
  • 最強はFable 5 + GPT 5.5のFusion
  • 構成次第でスコアが段階的に上がる設計
モデルを並べれば並べるほど強くなる理屈🧩

⑦ 仕組み: パネル×ジャッジ×最終回答

  • プロンプトをパネルモデル群へ並列送信
  • 各モデルにweb search・fetchを有効化
  • ジャッジモデルが全回答を読んで構造化分析
  • 合意点・矛盾・ユニーク洞察・盲点を抽出
  • 呼び出し元モデルが最終回答を執筆
アイデア自体は新しくないが筋は通っています🔧

⑧ 処理フロー例: 注意機構の質問

  • 例「アテンション機構とは?」を入力
  • 質問がパネル内の各モデルへ同時送信
  • 各モデルが個別回答を生成・返信
  • ジャッジが全回答を統合
  • 最終的に一本の回答として出力
prompt → panel (N models)
        ↓
   judge analysis
        ↓
   final answer
シンプルだが分かりやすいパイプライン🛤️

⑨ OpenAI API互換で使いやすい

  • 基本的にOpenAI APIと同じ使い勝手
  • SDKやクライアントを大きく変えずに導入可能
  • 統合ハードルが低い点は評価できる
  • 裏側の並列処理を意識しなくて済む
  • ただし分かりやすさと中身の差が激しい
インターフェースだけ見れば優等生です✅

⑩ マーケティングの根本問題: 比較が偏っている

  • Fable超えを主張しつつDeep Researchのみ比較
  • テキストタスクなら複数モデル利用が常に有利
  • コーディングは全く別領域なのに測定なし
  • 総合優位を装う表現が「悪い見栄え」
  • ユーザーの期待を意図的に誘導している感
都合のいい指標だけ拾っている印象⚠️

⑪ エレベーターシミュレータ: バグだらけ

  • 実タスク1本目: エレベーターシミュレータ
  • 動作はするがバグだらけで実用性低い
  • Opus単体やGLMと比較しても良さがない
  • Fusionならではの優位性は感じない
  • 複数モデルを束ねた意味が見えない結果
高価なFusionが単体モデルに負ける事態🐛

⑫ コンタクトレンズケースSVG: 凡庸

  • コンタクトレンズケースのSVG生成を依頼
  • 比例が崩れたいまいちな出来
  • Opus単体でも同等レベルで描ける
  • Fusion独自の強みが見当たらない
  • 悪くはないが特別でもない結果
「悪くない」以上のものが欲しい👀

⑬ 3JS折りたたみテーブル: 実用不可

  • 3JSで折りたたみテーブルのシミュレータを依頼
  • 脚を折りたたむと脚同士が重なる状態に
  • 物理的にありえない動きで実用性ゼロ
  • 複数モデル統合の意味をなさないレベル
  • 全体を通して酷い出来と評価
物理が破綻しているのは致命的❌

⑭ パンダのバーガーSVG: Geminiの使い回し?

  • 「パンダがバーガーを食べる」SVGを生成
  • 見た目がGeminiの生成そのものに酷似
  • 統合せずGeminiの回答をそのまま出した感
  • 出来自体はまあまあ程度
  • ジャッジが機能していない可能性を示唆
統合感のない一枚だった🐼

⑮ 弓矢シミュレータ: 極めて悪い

  • 弓と矢のゲーム風シミュレータを作成
  • 的の配置が奇妙にスタックされている
  • ゲームとして全く意味をなさない状態
  • 全体テスト中で最悪の出来
  • 「極めて悪い」と断言するレベル
ゲーム性が破綻していると話にならない💀

⑯ 数学問題とローカル学習: 両方失敗

  • 数学問題でも正答に到達できず
  • ローカルモデルトレーナーの実行を試みる
  • 対応するエージェントが存在しない
  • Fusion API非対応で実行すら不可能
  • 数学もエージェント連携もダブル敗北
エージェント未対応は実用上の致命傷🚫

⑰ コストと遅延: 高くて遅い最悪组合せ

  • コストが高く単体モデルより割高に
  • 複数モデルを呼ぶため応答が遅い
  • 結果の品質も下がる三重苦状態
  • 大多数のエージェントが未対応
  • 対応していても時間がかかり過ぎる
高い・遅い・悪いの三重苦…💸

⑱ 結論: ルーティング業に専念すべき

  • Fableの真の強みは圧倒的コーディング力
  • Deep Research向きに作られたモデルではない
  • 「複数ターンで精査」は diminishing returns の世界
  • OpenRouterはルーティングの改善に集中すべき
  • 過大宣伝に惑わされず冷静な評価が必要
得意分野で勝負するのが結局一番🎯
