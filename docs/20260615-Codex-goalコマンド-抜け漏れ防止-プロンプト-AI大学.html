Codex /goalコマンドでAIの抜け漏れを激減させる

依頼の最後の「1行」が成果物を決める — 実験データ付き
2026年6月15日

① AIに任せた作業で抜け漏れが多発する悩み

  • AIにちゃんとスキルを仕込んで手順も全部書いたのに、成果物を見たら肝心なものが抜けていた経験はないか
  • 学長自身がこの前やらかした。最初はCodexの性能を疑ったが、原因を追っていくと犯人はCodexではなかった
  • 同じ作業を2通りのやり方で試した結果、片方は画像が0枚、もう片方は8枚と成果物がまるで別物になった
  • 別れ目は依頼の最後に完了条件を1行渡すかどうかというたった1行の差だった
AIの能力ではなく「頼み方」が
成果を決める。これは目からウロコ
1行の差で成果が激変

② 学長の失敗体験 — 画像が抜ける司令スキル

  • 使ったのは記事のURLを渡すと翻訳から図鑑まで一気にやる自作の司令スキル
  • 翻訳→HTML作成（6回）→図鑑→音声生成→デビューまでの一連の作業をCodexに任せた
  • スキル内の段取りにはちゃんと書いてあったのに、途中の画像生成がすっぽり抜けていた
  • Claude Codeだとスキルに書いてあることを割と忠実に全部やってくれるが、Codexだと抜け漏れが多いという違いが判明
原因はスキル内に「必要に応じて
画像を作ってください」という
弱い指示があったから。

③ 原因究明 — 「必要なら」が見つかった犯人

  • 調べてみるとスキル内に「必要に応じてインフォグラフィックを作ってください」という記述を発見
  • この「必要に応じて」という表現がオプショナル扱いされてしまい、AIが画像生成をスキップ
  • CodexはClaude Codeよりも曖昧な指示を飛ばす傾向が強いことが判明
  • こういった抜け漏れをなくすために/goalコマンドの活用が有効だと気づく
「必要なら」「適宜」「可能なら」
この3語は要注意！
AIに飛ばされる原因No.1です。

④ /goalコマンドとは — AIの完了条件契約書

  • 簡単に言うとAIの完了条件契約書みたいなもの。AIが最後までやり遂げるための仕組み
  • 通常の依頼は「頼む→AIが作業→結果待ち」だが、/goalは完了条件を元にAIが自律的にやり遂げる
  • 条件を満たしているかチェック→まだなら続ける→達成なら完了。このループで最後までやり切る
  • 毎回「もう1回テストして」を伝える必要がなく、最初に完了条件を書くだけでAIが自律運転
AIをエージェンティックに
使う上でかなり重要な機能。
特にCodexでは必須レベル。

⑤ /goalの使い方 — CLIとデスクトップアプリ

  • 入力欄に直接/goalと打ち込むだけ。例えば「plan.MDの計画通りテストが通るまで実装」と指定可能
  • デスクトップアプリでも左側のプラスボタンから「目標を目指す」を選んでゴールモードに切り替えられる
  • CLIベースでもデスクトップアプリでもどちらでも使える。使い勝手は同じ
  • Codex 128.0以降で使用可能。設定ファイルに goal=true を書けば有効化できる
/goalコマンドが見つからない時は
Codexを最新版に更新！
128.0以降で使えます。

⑥ 状態管理コマンド — 一時停止・再開・クリア

  • /goal — ゴールの設定と実行開始。完了条件を指定してAIに作業させる
  • /goal pause — 一時停止。作業の途中で止めて確認したい時に使う
  • /goal resume — 再開。一時停止したゴールを再び動かす
  • /goal clear — クリア。ゴールの設定を破棄してリセットする
4つの状態管理コマンドで
AIの作業を自在にコントロール。
長時間のタスクでも安心。

⑦ プラン→ゴールの推奨フロー — 作戦会議から実行へ

  • まずプランモードでAIに計画を立てさせる。コードベースや関連情報を見ながら方針を決める作業
  • 方針が決まっていない時や設計・比較が必要な時にプランモードが特に強い
  • 立てた計画を元に/goalコマンドを実行する。計画→ゴールの2段構えが推奨される流れ
  • プランモード単体だと時間がかかるため、プランまでいかない細かい作業ではゴール単体でも十分
作戦会議（plan）→
契約書（goal）→実行の
3ステップが理想の流れ。

⑧ ゴールに書くべき7つの要素 — 完了条件の骨組

  • ①目的 — どんな状態なら終わりか。最終的なゴールを明確に定義する
  • ②入力 — URLやファイル名など、AIが作業のベースにする情報
  • ③成果物 — 期待されるファイル名やURL。何が存在すれば完成かを具体化
  • ④完了条件 — 何をもって完了と判定するかの明確な条件
  • ⑤確認方法 — ファイルやコマンドでどうやって確認するか
  • ⑥制約 — やらないことや公開範囲の制限
  • ⑦停止条件 — やめて報告すべき状況。無限ループを防ぐための安全装置
公式は6つだったが学長が停止条件を追加して7つに整理。
安全装置は大事！

⑨ ゴール設定のコツ3選 — 精度を上げる書き方

  • コツ1: 確認方法を具体化 — 「確認する」だけで弱い。「ファイルの末尾に X が存在すること」など何を見て確かめるかまで書く
  • コツ2: 曖昧な言葉を排除 — 「必要なら」「適宜」「可能なら」は要注意。AIがこれらの文字を見ると飛ばす
  • コツ3: 存在確認レベルに落とし込む — 「画像を作る」ではなく「画像が存在すること」まで具体化。AI自身もチェックできる
ゴールはお願いではなく
完了を判定するチェックリストを
作ってあげる、というのがポイント。

⑩ 実験A — /goalだけだと画像0枚の結果

  • 同じ記事URLを入力し、Codex公式のダイナミックワークフローを/clode nitroideで試す

⑪ 実験B — ゴールプロンプトビルダーで画像8枚

  • その弱い記述を改良して明確に必須化し、再度/goalで実行
  • 結果は画像8枚を含む完全パッケージが完成。0枚→8枚の劇的な改善

⑭ Codex vs Claude Codeの/goal比較

  • Codexの/goalは長い作業を完遂させるための本体機能。途中でサボるのを防ぐのが主目的
  • Claude Codeの/goalはすでに割と最後までやり遂げてくれる中での保険的な役割
  • Claude Codeはスキルに書いてあることを忠実に全部やってくれる傾向が強い
  • CodexはClaude Codeより途中で抜け漏れが多い分、/goalの効果がより顕著に出る

⑮ 無料プロンプト配布 — 汎用ゴールテンプレート

  • note記事で汎用のゴールテンプレートを無料公開中。3種類のテンプレートを用意
  • テンプレート1：作業を丸ごと任せたい時のフォーマット。目的→入力→成果物→完了条件の順に埋める
  • テンプレート2：スキル実行前にゴールだけ確認したい時のプロンプト。「実行しないでください」で下書き確認
  • テンプレート3：不具合修正。再現条件と修正根拠付きで片付けたい時の形式

⑯ 注意点 — /goalより先にやるべきこと

  • どう作るかがまだ固まっていない時は/goalより先にプランモードを使う。方針を先に固めるのが正解
  • ゴールに「必要なら」「適宜」を残したまま実行しないこと。AIがその手順を飛ばす原因になる
  • 停止条件を必ず書くこと。無限ループや不要な作業を防ぐための安全装置として機能する
  • プランモードは時間がかかる。細かい定型作業なら/goal単体で十分実用的
設計段階はプラン、
定型作業はゴール。
