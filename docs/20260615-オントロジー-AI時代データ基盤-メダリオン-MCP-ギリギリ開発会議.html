② AI-Readyなデータ基盤とは？

  • アプリやシステムはそこそこ簡単に作れるようになってきた。となると守るべきものはデータ
  • データが一番価値を持つという認識は、業界で共通認識として定着しつつある
  • AIが正しくデータを理解して活用するためには、ただデータをためるだけでは不十分
  • 「AI-Readyデータ基盤」という言葉もそこそこ定着してきた。フレームワークが固まってきている

③ AIのジレンマ — 数字は正しいのに意味が足りない

  • 「先月の売上は？」→ AIはSQLを生成して数字を返す。クエリも完璧、実行もできる
  • でも返ってきた金額は税込み？税抜き？送料は含んでる？売上は注文時点？出荷時点？
  • 正しい数字が帰ってきているのに、業務の文脈が含まれていない状態。最もらしい数字だが使い物にならない
  • つまり生のデータとAIの間に「意味の層」が必要になる
「税込みですか税抜きですか？」
誰も聞かないけど全員気になってる。
これがAIの限界

④ AI-Readyデータ基盤の三層アーキテクチャ

  • 第1層 正確なデータ基盤 — メダリオンアーキテクチャでデータ品質を段階的に向上させる土台
  • 第2層 意味の層 — セマンティックレイヤーとオントロジーで業務の意味を定義するコア部分
  • 第3層 安全なアクセス層 — MCP（Model Context Protocol）でAIが安全にデータにアクセスする入り口
  • これは特定の企業が勝手に言っているのではなく、業界全体で固まりつつあるアーキテクチャ
この3層構造が
AI-readyデータ基盤の
基本形として固まっている！

⑤ メダリオンアーキテクチャ — データ品質の段階的向上

  • IBMが提唱し、Microsoft（Databricks）も推奨するデータ品質向上のフレームワーク
  • Google Cloudも正式採用。メジャーなテック企業がこぞって採用している概念
  • データをブロンズ→シルバー→ゴールドと段階的に品質を上げていく考え方
  • リレーショナルデータベースでも基本はビュー等で実現可能。既存の設計思想に近いもの
概念や考え方なので
特別なソフトウェアは不要。
実践はRDBから始められる！

⑥ ブロンズ層 — 生データの無加工蓄積

  • Bronze CSV、JSON、在庫データなど生データをそのまま無加工で蓄積する層
  • データレイクや生のテーブルに近いイメージ。ETLの最初の取り込み先
  • 変更履歴を残すことが重要。「いつ」「どう変わったか」をトレースできる状態にする
  • ここには加工を加えないのがポイント。生の事実をそのまま保持する
ここで加工しないのが
大切！生データを
そのまま大事に保管。

⑦ シルバー層 — 洗浄・統一・正規化

  • Silver ブロンズ層のデータを洗浄・統合・正規化して整える層
  • フォーマットを統一したり、重複を排除したりする。商品名がバラバラなのを商品マスターで直す
  • リレーショナルデータベースでうまく設計すれば、シルバーくらいまでは自然に到達するレベル
  • ただしここまでは集計データはまだ含まれない。個別の正規化済みデータがメイン
要は「綺麗なデータ」を
用意する層。
RDBの正規化と同じイメージ。

⑧ ゴールド層 — 集計・整形したデータ

  • Gold 日時・カテゴリー別の売上マートなど、集計や整形を施したデータ
  • 設計パターンとしてスタースキーマが定番。デファクトスタンダードとして使われている
  • リレーショナルDBならビューを使って実現。BIツールやレポートのデータソースになる
  • ゴールド層まで整えると正確なデータ基盤が完成。これが三層アーキテクチャの土台
ゴールドまでできたら
土台完成！次は
意味の層を乗せる。

⑨ なぜ意味の層が必要か — 売上定義の崩壊

  • 「2026年5月の売上は？」→ ゴールド層から各自がSQLを書いて取得
  • 担当者A：税込み・返品含めて集計 → 140万円
  • 担当者B：税抜き・返品除いて集計 → 120万円
  • AIエージェント：出荷時点で集計 → 130万円。3人で3つの違う数字が返ってくる事態
ちゃんとした数字を取ってるのに
見え方がバラバラに！
売上の定義を固定しないと大混乱。

⑩ セマンティックレイヤー — 指標の統一定義

  • セマンティックレイヤー = 集計された数値を返す層。バラバラにならないように指標を固定・定義する
  • 「売上」は返品を除いた税抜きの金額の合計、と1箇所で明確に定義する
  • 平均注文額（AOV）= 売上 ÷ 注文数。リピート率 = 期間内に2回以上購入した顧客の割合
  • そうすると「5月のカテゴリー別売上は？」と聞いても誰が聞いても同じ答えが帰ってくる
現場でも数値が統一される。
これは BIやKPI監視の
大前提になる！

⑪ セマンティックレイヤーの定義例

  • Total Revenue = 返品を除いた税抜きの金額の合計
  • Average Order Value = 総売上 ÷ 総注文数
  • Repeat Rate = 期間内に2回以上購入した顧客の割合
  • Monthly Category Sales = 月次×カテゴリー別の売上集計
これらは概念なので
実現方法はいっぱいある。
コードでもツールでもOK！ 
// 例: dbtでのセマンティックレイヤー定義
metrics:

    label: "総売上"
    description: "返品除いた税抜き合計"
    type: sum
    sql: amount * (1 - is_returned)

⑫ セマンティックレイヤーとオントロジーの使い分け

  • セマンティックレイヤーの得意分野 → いくつ？いくら？をどの切り口で測るか
  • オントロジーの得意分野 → 何と何が繋がっているかの関係性をたどる
  • セマンティック → 主にダッシュボード・BI・KPI監視で活用
  • オントロジー → 主にレコメンデーション・横断調査で活用
返すものが違う：
セマンティックは数値、
オントロジーは関係データ！

⑬ オントロジーとは？ — 関係性の定義

  • 元々は哲学から来ている概念。特定領域の概念や関係性を体系的に表現するフレームワーク
  • 3つの主要構成要素を持つ：クラス・関係・属性
  • クラス = データに存在する全ての物事（例：人、会社）
  • 関係 = クラス間のつながり（例：人が会社に所属する）
  • 属性 = クラスを記述するプロパティ（例：人の名前・年齢）
出演者：「これオブジェクト指向じゃん！」
感覚的には似てるけど
使える場所が違うらしい。

⑭ オントロジーの活用例 — 関係性から答えを導く

  • 質問：「タンブラーと一緒に買われる商品は？」
  • オントロジーの手順：タンブラーを含む注文を関係性から洗い出す
  • → タンブラーを発注した顧客を特定
  • → 顧客の他の注文を見る → 一緒に買われた商品を洗い出す
  • 結果：冷却ボトル34%、シリコース21%、食器洗剤18%。関係をたどって答えが帰ってくる
セマンティックレイヤーの
返し方とは全く別！
関係性をたどって初めて
答えが出る。

⑮ オントロジーはどう実装するか

  • セマンティックレイヤー・オントロジー用の専用ツールが存在する
  • dbt（セマンティックレイヤー構築ツール）やProtégé（オントロジーエディター）
  • Protegé（プロテージ）はグラフDB。関係データベースの上に乗せるソフトウェア
  • ただし基本的には概念・考え方なので、RDBだけで頑張って組むことも可能
大規模になってデータが分散すると
グラフDBが必要になるけど、
小規模ならRDBで十分！

⑯ Palantir式オントロジー — アクションの追加

  • Palantirはオントロジーに「アクション」という要素を追加している。より複雑な定義
  • アクション（行動）を含めることで、データの関係性だけでなく実行可能な操作も定義できる
  • ただしこれは拡張系。デフォルトのオントロジーにはアクションは含まれない
  • シンプルな形（クラス・関係・属性）がデフォルト認識として広まっている
Palantirはさらに複雑。
でも基本は同じ。
拡張は段階的にでOK！

⑰ MCP — AIの安全なアクセス層

  • MCP（Model Context Protocol） = 第3層。AIがデータに安全にアクセスする入り口
  • 生のSQLではなくツールとして命令を渡す仕組み。AIが直接DBを操作しない安全設計
  • 意味の層まで整えた正しいデータを、MCPを通じてAIに渡すのが流れ
  • これによってAIが安全かつ正確なコンテキストを得られるようになる
SQLを直接叩かせない。
ツール経由で
安全にアクセスさせるのが正解。

⑱ まとめ — どこから始めるか

  • Step 1 ゴールド層を整える — 正規化して集計データを取りやすくする
  • Step 2 セマンティックレイヤーを載せる — 業務の意味付けと数値の意味付けをしっかり行う
  • Step 3 MCPで安全な入り口を作る — 正しいデータをAIに渡せるようにする
  • Step 4 複数システムがある場合はオントロジーを組んで関係性を定義する
小さく始めて大きくする。
セマンティックレイヤーも概念だから
マテリアライズドビューでも作れる。
まずは1つのユースケースから！
