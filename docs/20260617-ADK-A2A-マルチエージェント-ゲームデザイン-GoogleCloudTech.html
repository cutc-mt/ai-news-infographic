② ADK（Agent Development Kit）とは

  • Googleが開発したオープンソースのエージェント開発フレームワーク
  • Python、Java、Go、Kotlin、Androidなど複数プラットフォームに対応
  • 推論・計画・ツール活用など自律的なAIエージェントの構築に必要な機能をすべて提供
  • 単一エージェントからマルチエージェントシステムまで、階層的な構造を組める
AIエージェントを作るための
「SDK」みたいなもの。
Geminiを「頭脳」として使う。

③ A2A（Agent-to-Agent）プロトコルとは

  • A2A = エージェント同士が直接やり取りできる標準プロトコル
  • 各エージェントはAgent Cardという情報を公開。自分の能力を他のエージェントに知らせる仕組み
  • 異なるアーキテクチャや別々のサーバーで動いていても、標準仕様で通信が可能
  • ADKにはRemoteA2aAgentというコンポーネントが組み込み済みで、すぐにA2A通信を利用できる
エージェント同士が
「誰が何をしてくれるか」を
自動的に見つけて協力する！

④ なぜモノリシックはダメなのか

  • 1つの巨大な「なんでも屋」エージェントは命令過多で崩壊しがち
  • 指示が多すぎると出力の精度が下がり、システムが脆くてスケール不可能になる
  • 代わりに専門特化型のエージェントチームを組めば、高精度で制御しやすくスケール可能
  • 例：旅行プランなら飛行機担当・ホテル担当・観光案内担当に役割を分ける
全部できるエージェントは
結局何もまともにできない。
専門家チームが正解！

⑤ ゲームデザインのデモ概要

  • この動画のメインのデモは、複数のAIエージェントが協力してゲームのコンセプトをデザインするプロセス
  • ユーザーからのリクエストを受け、オーケストレーターが各専門エージェントにタスクを振り分ける
  • 各エージェントがそれぞれの分野で出力を生成し、最終的に一つのゲームデザインに統合される
  • 動画では実際のコードとライブデモを見せながら、各ステップを解説
「ゲームを作って」→ AIたちが

⑥ オーケストレーターエージェント

  • オーケストレーターはシステムの中央コーディネーターとして動作する親エージェント
  • ユーザーからのリクエストを受け取り、全体のゴールを理解してタスクを割り振る
  • ADKのサブエージェント機能を使って、専門エージェントをツリー構造で管理する
  • エージェント間の情報の流れを制御し、最終的な結果をユーザーに届ける
プロジェクトマネージャー
みたいな役割。
自分では作業せず管理に徹する！

⑦ エージェントの階層構造

  • 親 Root Agent — ユーザーの要求を理解してタスクを振り分ける
  • 子 専門エージェント群 — 各自の担当領域で専門的に作業する
  • 親エージェントはサブエージェントにタスクを委譲するツリー構造を採用
  • サブエージェントの出力が次のエージェントの入力になる。セッション状態を共有して連続的な対話を実現
組織図みたいに
階層構造で管理。
上位から下位に指示が流れる。

⑧ Agent Tools — ツールとして変換

  • エージェントを「目的地」ではなく「ツール」として扱うのがAgent Toolsの考え方
  • 専門エージェントをAgentToolに変換すると、親エージェントが複数のツールを組み合わせて使用できる
  • 単なる転送ではなく、マルチステップのワークフローを実行できるようになる
  • 「パリへのフライト」と「ホテル予約」を1つのクエリで連続処理できる
転送じゃなくて
道具として使う。
これがゲームチェンジャー！

⑨ 並列実行 — ParallelAgent

  • 依存関係がないタスクは同時に実行して時間を節約する
  • ADKのParallelAgentを使えば、独立したタスクを並列に処理できる
  • SequentialAgentと組み合わせて、直列処理と並列処理を混在させる
  • 例：フライト検索とホテル検索は独立だから同時に実行可能
順番にやる必要がないなら
並列で一気に処理。
速度が劇的に向上する！

⑩ 共有セッション状態

  • Session State = エージェント間で情報を共有する仕組み
  • あるエージェントの出力が次のエージェントの入力として使われる。継続的な対話を実現
  • 例えばTripSummaryAgentが結果をtrip_summaryというキーに書き込み、Reviewerがそれを読み取る
  • この共有状態によって、ステップ間の連携がスムーズに行われる
エージェント同士が
同じページを見ながら
作業を引き継ぐイメージ！

⑪ フィードバックループ — 品質保証

  • Reviewerエージェントを追加して、出力を自動レビューする仕組み
  • 品質チェックで「pass」か「fail」を判定。完全性と構造を検証
  • failの場合は修正ループに入り、合格するまで改善を繰り返す
  • これによってAIシステムが自己調整型になり、常に品質を保証できる
人間がレビューしなくても
AIがAIをチェック。
品質管理も自動化！

⑫ Agent Card — 能力の公開

  • Agent Card = 各エージェントが自分の能力とエンドポイントを記載した名刺のようなもの
  • 決められたURLに公開されるため、他のエージェントが自動発見可能
  • カードには対応機能・入出力フォーマット・エンドポイントURLなどが含まれる
  • これがA2Aプロトコルの相互運用性の要。新しいエージェントがプラグインのように追加できる
エージェントの「履歴書＋名刺」。
これがあれば誰でも
協力相手を見つけられる！

⑬ A2A通信のメッセージ方式

  • SendMessage — エージェント間の基本メッセージ送信
  • Task — 1つの作業単位。Task IDでステータスを追跡
  • TaskStatus — WORKING・COMPLETED・FAILEDなどで進捗を管理
  • プッシュ通知機能もある（現在は一部対応）。非同期処理に対応
メッセージの送受信は
JSON-RPCベース。
標準仕様で統一されている。

⑭ ハイブリッド編成 — クラウド＋デバイス

  • クラウド側ではGeminiのような強力なモデルで複雑な推論を実行
  • デバイス側ではGemini Nanoを使ってプライバシーに配慮したオフライン処理が可能
  • ADKを使えばクラウドとデバイスのエージェントをシームレスに組み合わせられる
  • Androidゲーム開発ではオンデバイスAIでレイテンシを低減しつつ、重い処理はクラウドに任せる構成が有効
軽い処理は端末で、
重い処理はクラウドで。
両方のいいとこ取り！

⑮ Interactions API — 新しい推論エンジン

  • Interactions API = ステートフルなマルチターン推論に特化した新API
  • previous_interaction_idでサーバー側に会話履歴を管理でき、コードがシンプルになる
  • background=Trueで長時間タスクの非同期実行が可能。タイムアウトなしで処理を待てる
  • 「思考（thoughts）」を最終回答と分離して扱い、推論の連鎖を効果的に処理
generateContentの進化版。
エージェントの
内部ループに最適化された！

⑯ 透過的なブリッジ — A2AとInteractions API

  • InteractionsApiTransportがA2AプロトコルをInteractions APIに透過的にマッピング
  • A2A SendMessage → Interactions create、A2A Task → Interaction ID、TaskStatus → Interaction Status
  • Interactions APIをリモートA2Aエージェントとして扱える。既存のクライアントコードを変更不要
  • Deep Researchのような強力なビルトインエージェントも、A2A経由でそのまま利用可能
新しいAPIを既存の
A2Aシステムにシームレスに統合。
コードを書き換える必要なし！
