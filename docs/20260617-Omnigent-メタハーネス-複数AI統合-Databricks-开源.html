Omnigent — 全コーディングエージェントを統合するメタハーネス

Databricksがオープンソースで公開した、Claude Code・Codex・Piなど複数のAIエージェントを束ねる新しいレイヤー
2026年6月13日公開 | Apache 2.0ライセンス

① Omnigentって何？ — 動画概要

  • Databricksが開発・オープンソース化したメタハーネス。複数のAIコーディングエージェントを統合管理する新しいレイヤー
  • Claude Code、Codex、Pi、カスタムエージェントをひとつのシステムで束ねる
  • 各ハーネス（エージェントのラッパー）は互いに独立していて、切り替えや組み合わせが困難だった課題を解決
  • Databricks自身の5000人以上のエンジニアリングチームで毎日使っている実績あるツール
「ハーネスの上にもう一段
新しいレイヤーが必要だ」という
大号令のもとに誕生！

② エージェントを使う現状の課題

  • エンジニアは4〜5個のエージェントを同時に開いている。コーディング、検索、ドキュメント、Slackを行き来
  • エージェント間でテキストをコピペするのが日常。各ハーネスは自分のセッションしか理解しない
  • ツールを切り替えるたびにコンテキストが失われる。同じ情報を何度も入力し直す無駄
  • 最新のハーネスやSDK、モデルを組み合わせるために絶えず再統合の作業が発生
5個開いてコピペ祭り。
これ限界やろ！
というのが出発点。

③ メタハーネスという新しい概念

  • 個別のハーネスのさらに上のレイヤーに位置する共通インターフェース。クラウドがプロセス管理を抽象化したように
  • ハーネス = モデルをエージェント化するラッパー。Claude CodeやCodexがこれに該当
  • どのハーネスも内部のLLM呼び出しは違えど、ユーザーへのインターフェースは「メッセージとファイルが入力、テキストとツール呼び出しが出力」で共通
  • Omnigentはこの共通APIでラップして、ハーネスを差し替え可能な部品にする
Kubernetesが
個別サーバーを束ねたように、
Omnigentはハーネスを束ねる！

④ Omnigentの3つの柱

  • Composition 合成 — 複数のモデル・ハーネス・手法をコードの書き直しなく組み合わせて切り替え可能
  • Control 制御 — 状態を追跡するコンテキストポリシーでコストや権限をガードレールとして管理
  • Collaboration 協働 — ライブセッションをURLで共有。チームメンバーとリアルタイムで共同作業
合成・制御・協働。
この3つがメタハーネスの価値。
単一ハーネスではここまでできない！

⑤ アーキテクチャ — Runner + Server

  • Runner — 任意のエージェントをサンドボックス化されたセッションでラップし、統一APIを提供する
  • Server — 履歴・ポリシー・スキル・成果物を管理し、セッションをターミナル・アプリ・Web APIで公開
  • 1つのコマンドでターミナルにセッションが立ち上がり、同時にlocalhost:6767でWeb UIが起動
  • ブラウザ、スマホ、デスクトップアプリ、API経由で同じセッションにアクセス可能。メッセージやファイルが同期
1コマンドでターミナル＋Web＋
モバイル全部同期。地味に便利そう。

⑥ Composition — エージェントの合成

  • Claude Code、Codex、Pi、カスタムエージェントを1行の変更で切り替え可能
  • カスタムエージェントは短いYAMLファイルで定義。プロンプト・ハーネス・ツール・サブエージェントを宣言
  • 異なるハーネスを使うサブエージェントを同じエージェント内で組み合わせることができる
  • OpenAI Agents SDKやClaude Agents SDKも統一APIでラップ。SDK同士の組み合わせも自由
name: my_agent
prompt: データ分析を手伝って。

executor:
  harness: claude-sdk

tools:
  researcher:
    type: agent
    prompt: 情報を検索して要約して。
YAMLを書くだけで
自作エージェントが動く。
ハーネス切り替えも1行！

⑦ Control — 状態的ポリシーによる制御

  • 従来の「許可／拒否」リストではなく、エージェントの行動を動的に追跡してコンテキストに応じた判断
  • コストポリシー — 例：「$100ごとにエージェントを一時停止して人間の承認を求める」
  • コンテキストポリシー — 例：「npmパッケージをインストールした後はgit pushに人間承認が必要」
  • ポリシーはサーバー全体・エージェント別・セッション別の3レベルでスタック。より厳しいセッションルールが優先
policies:
  budget:
    type: function
    handler: omnigent.policies.builtins.cost.cost_budget
    factory_params:
      max_cost_usd: 5.00
      ask_thresholds_usd: [3.00]
プロンプトに頼らないシステムレベルの制御。
セキュリティ担当者に安心感！

⑧ Collaboration — リアルタイム共同作業

  • ライブセッションをURLで共有。チームメンバーがエージェントの作業をリアルタイムで監視可能
  • ワークスペース内のファイルにコメントを追加したり、エージェントに直接コマンドを送信できる
  • セッションの共同ドライブ（co-drive）やフォークもサポート。複数人で同じエージェントを操作
  • 従来の「Slackにコードを貼ってレビュー」からエージェントのセッション内で直接共同作業へ
コピペ地獄からの脱却！
URLを渡すだけで共同レビュー。

⑨ Polly — マルチエージェントオーケストレーター

  • Omnigentに同梱されているマルチエージェントのコーディングオーケストレーター
  • Polly自体はコードを書かない。タスクを計画して、サブエージェントに委譲する役割
  • コーディングエージェントを並列のGit worktreeで実行。Claude Codeでの実装を委譲する例も
  • 完成したdiffを別のベンダーのエージェント（例えばCodex）にレビューさせる。同じLLMでの実装とレビューのバイアスを防止
実装とレビューを別LLMにする。
これが品質保証の鍵！
同じモデルの自画自賛を防ぐ。

⑩ Debby — 2つの頭脳を持つブレインストーマー

  • もう一つの同梱エージェント。ClaudeとGPTという2つの頭脳を持つブレインストームパートナー
  • すべての質問が両方に送られ、回答が並べて表示される
  • /debateコマンドで両方の頭脳が互いを批判し合い、最終的に一致点に収束する
  • 異なるモデルの強みを活かしたマルチパースペクティブな議論が可能に
ClaudeとGPTが
議論して結論を出す。
面白いユースケースだ！

⑪ Omnibox — OSレベルの強力なサンドボックス

  • Databricksのセキュリティチームが開発したOSレベルのサンドボックス。OSアクセスを柔軟に制限できる
  • ネットワークリクエストを傍受・変換。例えばGitHubのセキュリティトークンをエージェントから隠蔽可能
  • トークンは承認されたリクエストのエグレスプロキシにのみ注入される。エージェントには一切見せない
  • ModalやDaytonaなどのクラウドサンドボックスプロバイダーでも隔離された環境で安全に実行
トークンを見せない仕組み。
セキュリティが仕組みレベルで保証されるのはでかい！

⑫ 単一ハーネス vs Omnigent

  • モデル/ハーネス切替：単一は再統合が必要 → Omnigentは1行の変更で完了
  • インターフェース：単一はターミナルや各ツールUI → Omnigentはターミナル・Web・デスクトップ・モバイル・APIで同一セッション
  • ガバナンス：単一はプロンプトベースの許可リスト → Omnigentは状態追跡型コンテキストポリシー
  • コスト管理：単一は手動 → Omnigentは自動予算ポリシーで閾値停止
  • 協働：単一はコピペ → Omnigentはライブ共有・共同ドライブ・フォーク
全面的にレベルアップ。
特にコスト管理と協働が
実務的で重要。

⑬ 導入方法 — 数分で始められる

  • 必要な環境：Python 3.12+、Node.js 22 LTS、tmux
  • インストールは1コマンドで完了：
    curl -fsSL https://omnigent.ai/install.sh | sh
  • セットアップ：omni setup でモデルの認証情報を設定。環境変数を自動検出もする
  • 対応する認証：Claude・ChatGPTのサブスクリプション、OpenAI/Anthropic互換ゲートウェイ、Databricksワークスペース
インストール1コマンド。
数分で動き出す。
お手軽さ大事。

⑭ 業界トレンド — 複数エージェントの台頭

  • Harveyはオープンソースのワーカーモデルにフロンティアのアドバイザーモデルを組み合わせて、品質とコストでフロンティアモデルを上回った
  • Anthropicは研究製品をリードエージェントが並列サブエージェントを統率する構成で構築
  • Databricks自身のGenieも計画・検索・コード生成で異なるLLMを使用するマルチエージェント構成
  • エンジニアの働き方も変化。1つのエージェントをプロンプトするのではなく、エージェントチーム全体を駆動するループを設計する方向へ
単一モデルの限界。
マルチエージェントが標準に。
Omnigentはその波に乗っている。

⑮ デプロイとインフラ対応

  • Fly.io、Railway、Renderなど幅広いインフラでデプロイ可能
  • クラウドサンドボックス：Modal（サーバーレス）とDaytona（開発環境）を公式サポート
  • ローカルマシンでも、使い捨てのクラウドサンドボックスでも実行できる柔軟性
  • 多数のLLMプロバイダーに対応。コミュニティからのパッチでさらに拡大予定
ローカルでもクラウドでもOK。
ModalとDaytonaがあれば
ノートPCなしでも協働できる！

⑯ 制限事項と注意点

  • プロジェクトはアルファ版。ライフサイクルの非常に初期段階に位置する
  • セットアップにPython・Node.js・tmuxが必要。手軽さの中に環境構築のハードルあり
  • モデル・インフラ・利用料金は利用者が自己負担。Omnigent自体は管理しない
  • Omnigent Server MCPや自動最適化GEPAなどロードマップ項目は未リリース
  • オフネットワークのチームメンバーが参加するには常に稼働するサーバーが必要
アルファ版だから
本番導入は慎重に。
でも方向性は確かに正しい。

⑰ 今後のロードマップ

  • Omnigent Server MCP — エージェントがセッションをまたいで作業できるようにするMCPサーバー
  • GEPA — メタハーネスレベルでの自動最適化。エージェントの選択と構成を自動化
  • MemEx / RLM — エージェント内部のコードベースイントロスペクション。エージェント自身がコードを理解・操作
  • さらなるハーネスの統合やコミュニティからの貢献による機能拡張が期待される
MCPサーバーが来れば
エージェント間の壁が完全に消える。
これが一番楽しみ！

⑱ まとめ — 新しいレイヤーの始まり

  • 業界最大の転換は新しい抽象化レイヤーへの移行。プロセス→Kubernetesのように、ハーネス→メタハーネスへ
  • Omnigentはセッション・ポリシー・スキルをハーネスを問わず保持する。
    ツールを変えても作業環境が維持される
  • モデルとハーネスは変わり続けるが、ユーザーが作業するレイヤーは変わらないべきだという哲学
  • Apache 2.0ライセンスでオープンソース。GitHubとDiscordコミュニティで開発を進行中
「モデルとハーネスは変わり続ける。
作業するレイヤーが変わらないでいい」
この哲学がすごく刺さる。
