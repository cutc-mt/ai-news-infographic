① OpenRouter Fusionとは？

  • OpenRouter Fusionは、複数のAIモデルに同じ問いを投げてから1つの回答に統合する合議モデル
  • APIのエイリアスはopenrouter/fusion。1つのモデル名として呼び出せるのが最大の売り
  • 単体のLLMをそのまま呼ぶのではなく、複数モデルや補助処理を組み合わせて1つの応答を返す
  • 2026年6月13日にAPIがローンチされ、OpenRouterの公式ブログで大きな話題に

② Fusionの3段階の仕組み

  • パネル段階 1つのプロンプトを最大8モデル（標準3モデル）に並列送信。互いの回答を見ずに独立して生成
  • ジャッジ段階 別のAIモデルが各回答を横断分析。合意・矛盾・抜け・独自の洞察を評価
  • 統合段階 シンセサイザーが構造化分析に基づき1つの最終回答を生成
  • LLMディベートや合議として以前から語られてきた設計を、APIとして手軽に使えるようにした

⑦ 検証に使った6種の問題

  • 論理ひっかけ — 詭弁や直感を誘う問題で、推論力を測る

⑧ Fusionの2つのプリセット

  • Quality構成 Opus 4.8・GPT-5.5・Gemini 3.1 Proなどの高性能モデルをパネルに含む。最上位のパフォーマンスを目指す
  • Budget構成 Gemini 3 Flash・Kimi K2.6・DeepSeek V4 Proなどの安価なモデルを組み合わせる
  • Budget構成はFable 5と同等の性能を約半額で狙えると主張されるが、単体フロンティアに勝るとは限らない
  • Fusionはリクエストでパネル構成を指定可能。カスタマイズの自由度が高い

⑨ 結果① — 最上位パネルは強い

  • 本物Fusion vs Opus 4.8単体 → 勝率0.90、95%信頼区間[0.71, 0.97]、p=0.0002
  • 内訳は5勝1分0敗。偶然の勝ちとは見なしにくい結果
  • GPT-5.5単体との比較でも勝率0.75、p=0.023で有意差あり
  • 特に意思決定系の問題では、複数回答を束ねた保守的な整理が効いていた
素直に強い！
複数視点の抜けを拾いやすく

⑩ 結果② — 勝因はモデルの質

  • 安いFusion vs Opus 4.8単体 → 勝率0.17、p=0.003、内訳は0勝6分0敗
  • 安いFusionはGPT-5.5単体にも勝率0.05、自作合議にも勝率0.21、本物Fusionにも勝率0.08と全面的に敗北
  • 段取りが答えの質を底上げする範囲には限界があるという事実が判明
  • 安いモデルの誤読や浅い推論を、ジャッジとシンセサイザーだけで最上位レベルへ変える力は未確認
段取りだけでは強くなれない。

⑬ Fusionの3つの使いどころ

  • 急がないここぞの難問 — 99秒待てて$0.26を払ってもよい意思決定なら、本物Fusionの0.90は検討に値する
  • 複数視点が欲しいレビュー — 仕様の抜け・意思決定の反論・コード方針の見落としなど、1人のレビュアーでは偏る場面
  • 変数が多いタスク — 条件・例外・制約・コストが絡む判断で、複数回答の差分を見る意味がある
  • 対象外：締切直前の下書きや何度も回す日常タスクは単体frontierで十分
正解が明確な短い変換や要約なら
単体frontierを1回呼ぶ方が
速くて安い！

⑭ Best-of-N — 軽い代替案

  • Best-of-Nは、複数の候補回答を出してあとから良いものを選ぶ手法
  • Fusionより軽いアプローチ。難問だけ複数候補を作り、最後に強いモデルで選ぶ形
  • Fusionの発想を手軽に取り入れられる設計パターン
  • ただし合議は常用の節約テクニックとしては弱い。レビュー多重化の設計パターンとして残すのが現実的
合議は節約のためじゃなく
品質保証のための
設計パターンと考えるべき！

⑮ fusion-fable — 自作の再現例

  • GitHubにfusion-fableというオープンソースの自作合議スキルが存在
  • Opus 4.8が主オーケストレーター兼ジャッジ。GPT-5.5はCodex CLI、Gemini 3.1 Proはagy CLI経由で並列呼び出し
  • パネルはOpusを2本独立で走らせて3モデル構成まで可変
  • CLIが無い環境ではパネルを縮退して続行。完全停止しにくい設計
最終判断は常にOpus 4.8。
統合役の質が低いと
良い部分を拾い切れない！

⑯ 合議設計の重要な2つのポイント

  • 統合役の質が最も重要。パイプラインを逆にして安いモデルに最終統合させる形は避けるべき
  • 独立性の確保。各モデルに人為的な役割を割り当てすぎず、互いの回答を見せずに走らせることで違う切り口を狙う
  • APIの使いやすさもFusionの強み。キュー・並列実行・採点・統合を自分で組まずに済む
  • サーバーツールとしても使えるし、プラグインとして詳細なモデル構成をカスタマイズも可能
役割を固定しすぎると
多様性が失われる。
独立性が合議の命！

⑰ Fusion APIの呼び出し方法

  • openrouter/fusionというエイリアスでAPIから簡単に呼び出し可能
  • リクエストでパネル構成を指定できる。Quality・Budget以外にも自作プリセットが設定可能
  • 特定のタスクでのみ合議を起動するサーバーツールとしても利用できる
  • より詳細なモデル構成をカスタマイズするプラグインとしても利用可能
// Fusion API呼び出しの例
const response = await fetch(
  "https://openrouter.ai/api/v1/chat/completions",
  {
    model: "openrouter/fusion",
    messages: [{role: "user", content: query}]
  }
);
model名を変えるだけで
合議モードに切り替えられる！
APIの設計が美しい。

⑱ まとめ — Fusionの評価と今後

  • 本物Fusion（Quality構成）は単体frontierに勝つ。勝因はモデルの質
  • 安いFusionは非推奨。安いモデルを束ねるだけでフロンティア級は狙えない
  • OpenRouterの「約半額」はFable 5比で成り立つ見せ方。Opus 4.8やGPT-5.5比では高くつく
  • 合議はレビュー多重化の設計パターンとして残すのが現実的。節約テクニックではない
  • 今後は長い調査タスクや数日がかりの作業での検証が待たれる
