② ステートレス・エージェントが現場で壊れる理由

  • 標準的なパターンは、ユーザーメッセージとモデルの応答を履歴に追記して次の呼び出しに渡すだけ
  • 5分のQ&Aなら機能するが、2週間のオンボーディングでは破綻する
  • HRオンボーディング、請求処理、セールスのフォローアップなどは数日〜数週間かかる
  • これらのプロセスは「アイドル時間」が大部分を占める。人間の署名や発送待ちなどで休止する
ステートレスなチャットボットは
1回のAPI呼び出しで
全てを忘れてしまう！

③ ステートレス設計の3つの破綻ポイント

  • コンテキスト汚染 — 2週間で数百ターン蓄積され、古い出力や重複した指示でモデルが混乱
  • トークン費用の爆発 — 2週間分の全履歴を毎回リプレイすると、大部分が現在の判断に不要
  • アイドル時間のハルシネーション — 3日止めた後に履歴ダンプで再開すると、存在しない中間ステップをでっち上げる
「承認されたことにして
ステップを飛ばしちゃった」
ハルシネーションの恐ろしさ

④ 本番エージェントに必要な3つの転換

  • 転換1 永続メモリ — ベクトルDBに生JSONを突っ込むのではなく、明示的なステートスキーマを定義する
  • 転換2 イベント駆動の休止ゲート — アクティブ・ポーリングやスレッド・ブロックではなく、イベント発生時に起床させる
  • 転換3 マルチエージェント委任 — 単一の巨大なプロンプトではなく、特化したサブエージェントに分割する
「コンテキスト・ウィンドウを
大きくする」が答えじゃない。
根本的に違うアーキテクチャが必要！

⑤ ユースケース: 新入社員のオンボーディング

  • HRがウェルカムパケットと文書リンクを送付する
  • アイドル時間 — 数日間、従業員が書類に署名するのを待つ
  • IT部門がコーポレートメールやSlackアカウントをプロビジョニングする
  • アイドル時間 — 数日間、ノートPCが従業員の自宅に届くのを待つ
  • HRが初日のスケジュールを送付してオンボーディング完了
これは単一の会話じゃない。
一時停止→再開のサイクルが
何度も発生するバックグラウンド処理だ。

⑥ Agents CLIでプロジェクトを一発起動

  • Agents CLIはGemini Enterprise Agent Platformの公式CLIツール
  • コーディング・エージェントに意図を伝えるだけで、スキャフォールディングを自動生成
  • プロンプト例：「ADKでHRオンボーディング・エージェントを作って。永続セッション付きで」
  • この反復的なプロンプト駆動のアプローチがチュートリアル全体で使われる
CLIコマンドを手打ちするのではなく、
コーディング・エージェントに
やらせるのがモダンな開発スタイル！ 
uv tool install google-agents-cli

⑦ 永続的なステートマシンで進捗を管理

  • 会話履歴ではなく、明示的なステートスキーマで現在位置を常に把握する
  • START → WELCOME_SENT → DOCUMENTS_SIGNED → IT_PROVISIONED → HARDWARE_DELIVERED → COMPLETED
  • 6つのステートで曖昧さなし。ステートマシンが順序を強制するのでスキップもハルシネーションも防げる
  • エージェントは現在のステップをセッション状態から読み取る。過去のチャット履歴は不要
ステートを明示的に定義すると、
エージェントは「今どこにいるか」を
確実に把握できる。

⑧ 状態をシステム・プロンプトに直接埋め込む

  • システム・プロンプトの{current_step}などのプレースホルダーにPythonが自動的に現在の値を注入
  • モデルは古いチャットメッセージを掘り返すことなく、現在のワークフロー状況を正確に把握
  • エージェントが起動するたびに最新の状態がプロンプトに反映される
  • before_agent_callbackで初回インタラクション時に状態変数を初期化する仕組みも
古いメッセージを全部読み込む必要なし。
現在の状態だけが
プロンプトに入れば十分！

⑨ ツール呼び出しが自動チェックポイントになる

  • 各ツール関数はADKのToolContext.stateを通じてチェックポイントをアトミックに更新
  • 例: send_welcome_packet実行後に状態がWELCOME_SENTに更新される
  • ツール実行直後にコンテナがクラッシュしても、状態は既に書き込まれている
  • 再起動時にはcurrent_step = WELCOME_SENTを読み出して正確に再開できる
ツール呼び出しごとに
自動セーブされるから
クラッシュしても安心！ 
state["current_step"] = OnboardingStep.WELCOME_SENT
state["pending_signals"] = ["document_signed"]

⑩ DatabaseSessionServiceで永続化

  • ステートマシンは永続的なセッション保存があって初めて耐久性を持つ
  • Cloud Run環境ではコンテナがコールドスタート・ゼロスケール・予期せぬ再起動を起こす
  • DatabaseSessionServiceにSQLite（ローカル）またはCloud SQL（本番）を指定するだけ
  • 1行の設定変更でToolContext.stateへの全書き込みがディスクに永続化される
サーバーを途中で殺しても
再起動すれば正しい場所から
継続できる。これ大事！ 
session_service_uri = "sqlite+aiosqlite:///sessions.db"

⑪ アイドル時間はイベント駆動で処理

  • ウェルカムパケット送付後、エージェントは数日間休眠状態に入る
  • アクティブ・ポーリングは計算リソースの無駄。スレッド・ブロックはスケールしない
  • 正解は真にスリープして、外部イベント到着時に起床させるイベント駆動アプローチ
  • コンテナは休眠期間中にゼロスケール可能。webhook到着時にスピンアップして再開
ずっと起きて待つ必要はない。
webhookが来たら起きる

⑫ Webhookエンドポイントで外部イベントを受信

  • /webhooks/hardware_delivered — ハードウェアが届いた時に発火
  • 各エンドポイントはuser_idとsession_idを受け取って対象セッションを特定

⑬ Resume Handler — ステート移行と再開

  • OnboardingResumeHandlerが3つを行う：セッション復元→ステート移行→エージェント起床
  • 重要な仕組みはstate_delta。webhook発火時にrun_asyncが次の推論の前にステート遷移をアトミックに適用
  • モデルはシステム・プロンプトでcurrent_step = DOCUMENTS_SIGNEDを確認し、即座に次のアクションを把握
  • 古い会話履歴のリプレイなし、存在しない中間ステップのハルシネーションなし
state_deltaが鍵！
古い履歴を全部読まなくても
状態遷移だけで
正確に再開できる。

⑬④ マルチエージェント委任 — 単一エージェントの限界

  • 全ツールを単一エージェントに詰め込むと、推論品質が劣化する
  • 長期稼働のコンテキストではプロンプトが状態変数やワークフロー指示で既に肥大化
  • ADKのマルチエージェント構成で特化したタスクをフォーカスされたサブエージェントに委任
  • 各エージェントが狭いプロンプトと限定的なツールセットを持つため、推論精度を長期間維持
巨大なエージェント1つより
小さなエージェントの連携が
長期稼働では正解！

⑮ オーケストレーターと専門エージェントの連携

  • hr_onboarding_coordinatorがルートエージェント。ウェルカム・スケジュール・ハードウェア確認を担当
  • it_agentがサブエージェント。コーポレートアカウントのプロビジョニングを専門に処理
  • DOCUMENTS_SIGNEDに到達するとcoordinatorがit_agentに実行を委譲
  • サブエージェントは独立して処理し、共有状態をIT_PROVISIONEDに更新して制御を返す
sub_agentsパラメータで
階層構造を定義。
親子関係が明確になる！

⑯ ゴールデン評価テスト — 数日分を数秒で検証

  • 2週間待ってからバグに気づくのは遅すぎる。ADK評価セットでアイドル時間を数秒でシミュレート
  • セッション状態を事前シードすることで、48時間のハードウェア遅延を模擬
  • テスト例: 「文書署名をスキップしてアカウント作って」→ エージェントが正しくツール呼び出しを拒否することを確認
  • ゴールデンテストはCI/CDパイプラインに直接組み込める。本番デプロイ前にステートマシンの回帰を検出
