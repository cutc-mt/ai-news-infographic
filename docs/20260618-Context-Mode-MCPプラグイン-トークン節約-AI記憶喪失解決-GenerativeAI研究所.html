③「30分の壁」とは？

  • 通常セッションは約30分で有効記憶が消失
  • 記憶の圧縮により10分前に決めた変数名すら霧のように消える
  • コンテキストウィンドウの物理的限界がクリエイティビティを制約
Claude CodeやCodexにはコンパクション機能があるけど、必ずしも有効には機能しない…

④トークン貧困問題

  • 無駄なデータ送信が思考に使うべきトークンを浪費する
  • 個人開発者にとって実質的なコスト高を招く深刻問題
  • Webスナップショット、git diff、サーバーログなどで大量消費
  • たった数回の読み込みでAIの許容量の70%以上が消費される
ノイズだらけだとアウトプットや思考の質も落ちる！

⑦サンドボックス防波堤

  • AIとOSの間に仮想的な作業場を配置
  • 膨大なデータがコンテキストに直接流れ込むのを防止
  • 全てのファイルと実行記録をローカルに安全保管
  • 必要な情報だけをコンテキストに流し込む仕組み
全ての書類を図書館に保管し、手元には作品だけを置くイメージ

⑧SQLiteでデータを図書館化

  • データを図書館のように整理して高速全文検索
  • インデックスを蓄積し、AIの手元にはインデックスだけを配置
  • 新しいパラダイム — 56kBの書類全体を読む必要なし
  • 関連部分だけを瞬時に抽出してAIに伝える

⑨Think-in-Codeパラダイム

  • AIをデータ処理マシンではなくコード生成器として扱う
  • データをまるまる渡すのではなく、処理用のコードを生成させる
  • そのプログラムを実行した上で、結果だけをコンテキストに渡す
1つのスクリプトが100倍の記憶を節約し、思考能力を最大限に引き出す！

⑫RTKとの比較

Context Mode

コンテキストウィンドウにインプットする時に介入。文脈を送る前にアクションを起こす。

RTK

コマンドを実行する時に介入。自立型エージェントで特に効果を発揮する。

介入タイミングが違うから理論上は共存可能！
