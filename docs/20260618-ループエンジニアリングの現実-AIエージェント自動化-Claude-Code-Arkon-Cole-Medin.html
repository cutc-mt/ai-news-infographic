ループエンジニアリングの現実

もうプロンプトを書かない？ AIコーディングエージェントを24時間働かせる新しい手法

📅 2026年6月18日

🎬 Claude Code / OpenClaw の開発者たちが推進する「ループ」の世界

第1章：ループエンジニアリングとは？

①新時代のAI活用: プロンプト不要の世界がやってきた

  • Peter Steinberger（OpenClaw開発者）とBoris Cherny（Claude Code責任者）が「もうプロンプトを書かない」と宣言
  • 代わりにループを書いて、AIエージェントを自律的に動かす手法
  • エージェントが24時間365日休まず作業する仕組み
  • 「ループエンジニアリング」が次のバズワードになりつつある
「プロンプトエンジニアリングは終わった、ループの時代だ」
— って、本当にそうかな？ 🤔

②キーパーソン: ループ推進の二人の巨人

  • Peter Steinberger — OpenClawの開発者。無限の予算でエージェントを大量展開
  • Boris Cherny — Claude Codeの責任者。「俺の仕事はループを書くこと」
  • 「Claudeにはプロンプトしない。ループを書いて、ループが仕事をする。俺の仕事はループを書くことだ」— Boris
  • 二人ともインタビューやポッドキャストでこの手法を広めている
Borisの発言が注目を集め、
Redditでも議論が白熱中！

③Claude Codeの3つのループ機能

  • /loop — 一定間隔でプロンプトを実行。例：「5分おきにGitHubのIssueを確認して処理」
  • /goal — 完了条件を設定し、AIが達成するまで自律的に作業
  • /routines — スケジュール済みのタスクを定期的に実行
  • これらを組み合わせて大きなタスクを分割して自律的にこなす仕組み
一番シンプルな /loop だけでも
自動化のパワーを実感できる！

④基本的な使い方: Claude Codeでループを動かす

  • タスクリストをplan.mdなどのドキュメントとして準備
  • Claude Codeにloop skillを読み込ませるよう指示
  • 各サイクルで未完了のタスクを1つずつ実行して検証
  • 全タスク完了後にオーケストレーターが報告
# 指令の例
「loop skillを読み込んで、
 plan.mdのタスクを1つずつ
 順番に実行してね」
Claude Code自身が
/loopの実行プロンプトも自動生成！

⑤実演: タスクを自動で片付ける様子

  • リクエストを送ると、Claude Codeがloop skillを自動で読み込み開始
  • 「順次タスクリスト」として認識し、最初のタスクを実行
  • 検証後、/loop wakeupで自動的に次のタスクへ移行
  • ユーザーがプロンプトを手書きする必要は一切ない
数分戻ってくるだけで
もう2つのタスクが終わってる 😳

⑥ループエンジニアリングの仕組み: オーケストレーターとワーカー

  • オーケストレーター — メインエージェント。ユーザーの指示を受け取り、全体を管理
  • ワーカー — 実際の作業を行うエージェントたち
  • オーケストレーターが何人のワーカーを立ち上げるか、何周ループするかを決定
  • ワーカーの結果がオーケストレーターに戻り、次の指示を生成
人間は高いレベルで指示を出すだけ。
あとはシステムが自律的に回る！

第2章：ループエンジニアリングの3つの課題

⑦課題①: 最高品質とは程遠い

  • ループエンジニアリングで最高の結果を得られるという主張は大げさ
  • Borisは「1日に数万のAIエージェントを管理」と言うが、実用性に疑問
  • これだとClaude Codeのバグの原因にもなっているのでは？というツッコミも
  • PoC（概念実証）やアイデア探索には有用だが、全開発を任せるには不適切
自律ループを1日走らせて戻ってきたら
「クソコード」ができてた経験あり 😭

⑧課題②: トークン消費が爆発する

  • オーケストレーターが仕様を分析し、ワーカーを起動し、結果を統合するだけで大量消費
  • 比較的シンプルなアプリ1つの構築で100万トークン以上を消費した実例あり
  • 毎回コンテキストを渡して推論する分散処理の仕組み上、どうしても高コストに
  • 「Peterみたいな無限の予算」がない場合は要注意
コスト追跡ダッシュボードを
見て「うわっ」となった 💸

⑨課題③: コンテキストの肥大化問題

  • /loopを使う場合、同じセッション内でループが継続
  • 長時間ループするとLLMのコンテキストが膨張して overwhelmed 状態に
  • 複数のコーディングエージェントセッションに作業を分散する仕組みが必要
  • 各セッションが全体の位置づけを理解して通信できることが理想
解決策が「ハーネス（制御システム）」の構築だ。
詳しくは第3章で！

第3章：ハーネスで課題を解決する

⑩ハーネスエンジニアリング: ループの進化形

  • ハーネス = エージェントを制御・監視するフレームワーク
  • エージェントに全権を委ねず、決定的なプロセスで管理
  • AIの推論は必要な場面（コード記述など）に限定
  • テスト実行や検証などは人間が保証するプロセスにする
「ループエンジニアリング」というより
「ハーネスエンジニアリング」だね！

⑪Arkon: ワークフロー駆動のハーネスビルダー

  • Arkon — 複数のコーディングエージェントをオーケストレートするツール
  • ワークフローファイルでプロセスを定義し、各ステップで独立したセッションを実行
  • ステップ間はMarkdownでコンテキストを引き継ぎ、セッションを分離
  • 計画 → 実装 → テストの古典的ワークフローを1つのArkonフローに構成可能
Claude Codeの /loop と違って
各ステップが独立セッションで動く！

⑫実例: GitHub Issueの自動修正ワークフロー

  • Issue番号を抽出 → Issueのコンテキストを取得
  • LLMがバグか新機能かを分類し、動的にワークフローを分岐
  • 調査 → 実装 → 検証 → PR作成をステップごとに別セッションで実行
  • 各ステップが決定的で、エージェントが判断しすぎない設計
バグ修正と新機能で
全く違うワークフローが走る！

⑬コスト最適化: モデルを適材適所で使い分ける

  • Arkonでは各ノードでモデルを個別に指定可能
  • 分類・コンテキスト読み込み → 小型モデル（Haiku / Kimmy K2.7）
  • 実装 → Claude Code、レビュー → Codex
  • 全ステップをOpusでやる必要はなく、ステップごとに最適なモデルを選べばコスト大幅削減
Claude Code単体だと1つのモデルで
全部やるから高くつくんだよね 📊

⑭耐性（Durability）: いつでも途中から再開

  • 全ログと実行状態をPostgres（Neon）に保存
  • マシンが落ちても正確に中断したステップから再開可能
  • 会話、コードベースの操作、すべての状態が永続化
  • キャンセルしても作業が失われることはない
外部DBに状態を置くのが鍵。
エージェントのセッションに依存しない！

⑮並列実行: 4つのIssueを同時に自動処理

  • メインのClaude Codeがオーケストレーターとなり、複数のArkonワークフローを並列起動
  • Git worktreeで各エージェントを分離し、互いに干渉しないよう管理
  • 修正 → PR検証 → コードレビューを9つ以上のセッションで同時実行
  • 途中でステータス確認も可能。オーケストレーターが全体を把握
各セッションは軽量だから
全体では大量の作業をこなせる！ 🚀

第4章：観測ダッシュボードとクラウドデプロイ

⑯オープンソースの観測ダッシュボード

  • GitHubでオープンソースとして公開中のループ管理ダッシュボード
  • 全ループの実行履歴、トークン消費、ステータスを可視化
  • NeonのPostgresに状態を外部保存し、オーケストレーターが読み取って次のアクションを決定
  • Human-in-the-Loop: 各ラウンド終了後に人間が確認してから次へ進める機能
コーディングエージェントに
実行ログを分析させて改善もできる！

⑰Retoolでクラウドデプロイ: チーム共有も可能に

  • Retoolを使えばローカルのダッシュボードをクラウドに簡単デプロイ
  • Claude Codeが作ったReactアプリをそのままRetoolにインポート可能
  • 権限グループ設定でワークフローの一時停止・再開に承認が必要に
  • チームメンバーとループのセットアップを共有できるようになる
ローカルで作った管理ツールを
どこからでもアクセスできる世界線へ 🌍

⑱まとめ: ループの良さを取り入れる正しいアプローチ

  • 「ループエンジニアリング」はバズワードに過ぎないが、有用な概念は含まれている
  • ハーネスを構築して、決定的なプロセスでAIを制御するのが正解
  • コストはモデルを適材適所で使い分けることで大幅に抑えられる
  • Human-in-the-Loopで人間が重要な判断ポイントに介入する信頼性を確保
  • 自律性と制御のバランスを取れば、AIコーディングの生産性は劇的に上がる
「ハーネスエンジニアリング」
こそが次の正しい名前かも！ 💜
