Claude Designアップデート解説

Claude Codeとの双方向連携・デザインシステムの活用法を徹底解説
2026年6月20日

① Claude Designアップデートの概要

  • Claude Designに大規模アップデートがやってきた。Claude Codeとの双方向連携が最大の目玉
  • デザインから実装へ、実装からデザインへと双方向でデータが行き来するようになった
  • 使い勝手も大幅に向上。以前よりスムーズにデザイン制作ができるように
  • オパス4.8に進化したことでUIの精度が向上。より実用的なデザイン出力が期待できる
デザインとコーディングの壁がなくなった！今までは片道通行だった連携が双方向になったのが大きな進化です。

② Claude Designの基本構造

  • Claude Designは「プロジェクト」と「デザインシステム」の2つが大枠として存在する
  • デザインシステム：よく使うデザインの定型ルールや部品をまとめたもの
  • プロジェクト：デザインシステムを反映した成果物。UI画面、アプリのUI、スライドなど
  • デザインシステムを作り込んでからプロジェクトに展開していくのが基本フロー
ブランドの統一感を出すために、まずルールを定義してから成果物を作る。この順序が大事です。

③ デザインシステムとは何か

  • 例えば株式会社Aのブランドデザインテーマをルールとして定義する
  • 営業資料の世界観テーマなど、プロジェクトレベルで定義できる
  • フォント、色、UIデザインのルールを細かく設定可能。ボタンやアバターなどのパーツ単位で指定
  • 一度作れば複数のプロジェクトで使い回せる。ブランドの一貫性を保てる
会社のスライドテーマやUIテーマをデザインシステムとして定義しておけば、毎回ゼロから作らなくて済みます。

④ プロジェクトのフォルダ構造

  • プロジェクトの中にはフォルダ構造が用意されている
  • 画面のHTML：実際のUI画面を構成するファイル
  • DSフォルダ：デザインシステムの定義ファイルが格納される
  • 「デザインシステムを元に成果物を作ったよ」という構成になっている
HTMLとDSフォルダが分かれているのがポイント。どこに何が入っているかが明確です。

⑤ デザインシステムの作成手順

  • 「クリエイトデザインシステム」からスタート
  • 「どういう感じのものを作りたいか」をチャットで伝える
  • 参考にするGitリポジトリを指定することも可能
  • フッツロゴなどの連携も組み込める。既存のブランド資産を活用できる
既存のリポジトリを参考に渡せば、それに近い雰囲気のデザインシステムが自動的に生成されます。

⑥ デザインシステムの細かな設定

  • フォント：M+ラウンドなど、ブランドに合ったフォントを選択可能
  • 色：プライマリーカラー、セカンダリーカラー、アクセントカラーを定義
  • UIパーツ：ボタン、アバター、バッチなどのデザインルールを個別に設定
  • ページ図の中でデザインシステムを可視化できる。HTMLも配置可能
チャット欄からエージェンティックに調整できるのが便利。細かい修正も会話だけで済みます。

⑦ チャットでエージェンティック調整

  • チャット欄からエージェンティックにデザインを調整できる
  • 「ボタンをもっと丸くして」「色を少し明るめに」など自然言語で指示が可能
  • デザインシステムにコメントも付けられる。チーム共有時のメモとして活用できる
  • 試行錯誤がしやすい環境。すぐにプレビューで確認できるから効率的
「このボタンの角をもっと丸くして」
という感覚でデザイン調整ができる。
まるでデザイナーと話しているようです。

⑧ デザインシステムの共有とエクスポート

  • シェア機能でHTML出力ができる。他者にデザインシステムを共有可能
  • プロジェクトアーカイブとして一括で吐き出しも可能
  • PDF、パワーポイントなど複数の形式でエクスポートできる
  • Claude Codeにも渡せる。CLIやClaude Code Webから利用可能
Claude Codeに渡して実装に進めるのがこのアップデートの醍醐味。ここが双方向連携の起点になります。

⑨ パブリッシュ機能で公開

  • パブリッシュ機能でデザインシステムをトップ画面に出せる
  • デフォルト設定も可能。よく使うデザインシステムを最初から選べる
  • チーム全体で統一されたデザインを使えるようになる
  • プロジェクトごとに適切なデザインシステムを切り替えて使える
複数のデザインシステムをパブリッシュしておけば、プロジェクトに応じて最適なものを選べます。

⑩ Claude Codeとの双方向連携の全体像

  • Claude Design → Claude Code：デザインからプロジェクトを作ってClaude Codeに持ち込む
  • Claude Code → Claude Design：デザインシンクツールでClaude CodeからClaude Designに渡す
  • 双方向フローが確立。デザインと実装の往復がシームレスに
  • コンポーネント単位で少しずつ同期することが前提で設計されている
オパス談：「コンポーネント単位で少しずつ同期する」が設計思想。大まかな連携ではなく、細かい粒度での同期です。

⑪ デザインシンクツールとは

  • デザインシンクツールはすでにClaude Codeに組み込まれている
  • list_projectsでデザインシステムの一覧を取得できる
  • get_projectで特定のプロジェクトを取得できる
  • CLIからもClaude Code Webからも同じツールが使える
別途インストール不要。Claude Codeを使っているなら、すでにデザインシンクツールが使える状態です。

⑫ 既存コンポーネントからのデザインシステム生成

  • 既存のReactコンポーネントからデザインシステムを作ることも可能
  • ストーリーブック形式で渡すと再現度が高いデザインシステムが生成される
  • ストーリーブックからToアプリを作ってデプロイする例も紹介された
  • 既存資産を無駄にしない。すでにあるUIコンポーネントからスタートできる
ストーリーブックがあれば、それを食わせるだけでデザインシステムが自動生成されます。チームがすでに持っている資産を活用できるのが大きいです。

⑬ ベストプラクティス4ステップ

  • ステップ1 よく使う会社のスライドテーマやUIテーマをデザインシステムとして定義
  • ステップ2 プロジェクトレベルで成果物を作成する（アニメーションでも何でも）
  • ステップ3 Claude Codeに渡して実装を進める
  • ステップ4 Claude Codeで作ったものをデザインシンクでClaude Designに戻す
この4ステップの循環が現在のベストプラクティス。デザイン→実装→確認のサイクルを回していくイメージです。

⑭ オパス4.8による精度向上

  • オパスが4.8になったことでUIの精度が向上している
  • より実用的できめ細かいデザイン出力が期待できる
  • 前のバージョンと比べて、意図した通りのデザインに近づきやすくなった
  • モデルの進化がそのままデザイン品質に直結している
Claude Codeの性能向上が、デザイン面でも大きなメリットをもたらしています。モデルが賢くなれば、それだけ良いデザインを出せるというわけです。

⑮ Claude Design vs オープンソースのデザインハーネス

  • 推奨 Claude Designを使う方がおすすめ
  • Claude Codeとのネイティブ連携が最大の強み
  • オープンソースのデザインハーネスはClaude Codeとの統合が不十分
  • 同じ生態系内で完結できるのがClaude Designの決定的な優位性
Claude Codeを使っているなら、迷わずClaude Designを使うのが正解。連携の手間がゼロに近いです。

⑯ ベータ版の現状と課題

  • まだベータ版なので複雑性と使いにくさは残っている
  • UIの一部に不自然な部分やバグが存在する可能性
  • 学習コストがかかる。最初は使い方に戸惑うかもしれない
  • 今後の進化が大きく期待される。初期段階からの利用者優位も
ベータ版ならではの不満はありますが、方向性は間違っていません。今後のアップデートでさらに良くなるはずです。

⑰ 今後の進化に期待すること

  • UIの使いやすさの改善。複雑性を減らして直感的な操作へ
  • コンポーネント同期の精度向上。より細かい粒度での双方向連携
  • Figmaなどの外部ツールとの連携がさらに深まる可能性
  • チーム開発でのコラボレーション機能の強化も期待
今はベータ版の黎明期。初期から触っておくことで、今後の進化に乗り遅れなくなります。

⑱ まとめ：デザイン×実装の新常識

  • デザインシステムを定義してプロジェクトで活用するのが基本
  • Claude Codeと双方向で連携していくのが現在のベストプラクティス
  • 「デザイン→実装→確認→修正」のサイクルを高速で回せる
  • オープンソースのハーネスよりClaude Design一択。Claude生態系の完成度が高い
デザインとコーディングの分断時代は終わりました。Claude Design + Claude Codeで、1つの流れでデザインから実装まで完結する新しいワークフローが始まっています。
