EGO Light — AIエージェントのためのブラウザ

エージェント専用のワークスペースをブラウザの中に構築する新アプローチ

📅 2026年6月20日

① ブラウザエージェントの現実: デモは完璧だが実務は別

  • デモ映像では見事に動いているように見えるが、実際のウェブサイトは複雑で混沌としている
  • ログイン画面、二段階認証（2FA）、大量のタブが開いている状態など、現実の障壁が山積み
  • エージェントがフォーカスを奪ってしまったり、空っぽのブラウザプロファイルではログイン済みの状態を維持できない
  • 既存のブラウザにエージェントを「後付け」するアプローチには構造的な限界がある

② EGO Lightの発想: ブラウザ自体をエージェント対応に

  • 従来の逆転アプローチ。エージェントにブラウザを後付けするのではなく、ブラウザ自体をエージェントのために設計する
  • Chromiumベースのブラウザで、AIエージェントに専用の隔離スペースを提供する仕組み
  • ユーザー自身のタブやログイン状態はそのまま維持。エージェントには別のワークスペースが割り当てられる
  • 人間の作業とエージェントの作業が可視化されて分離される、安心感のある設計

⑤ SPACE: 同じブラウザ内の並行ワークスペース

  • SPACE（スペース）という概念で、同じブラウザの中に並行して独立したワークスペースを構築できる
  • エージェントの作業は可視化されるものの、ユーザーのタブとは完全に分離されている
  • 隔離された環境なので、エージェントが勝手にユーザーのセッションに干渉することはない
  • SPACEごとに独立したCookie・キャッシュ・セッションを持つため、複数タスクの同時実行が可能

⑥ EGO Browser: CLIではなくコードベースのAPI

  • EGO BrowserはコードベースのAPIを提供する。CLIツールではなく、JavaScriptの関数としてブラウザを操作できる
  • API呼び出しが少なく済むため、トークン消費を抑えつつ高速に実行できるのが特徴
  • 少ないツールコール数で同じタスクを完了できる。結果として実行速度の向上とコスト削減に直結
  • API駆動の設計は、自動化パイプラインやCI/CDへの統合も想定している
ツールコールが減る＝

早くて安い。

⑦ JavaScript API: ブラウザを関数として操作

  • snapshot() — ページの状態を圧縮されたセマンティックビューとして取得する
  • fill() — フォームに入力する。テキストフィールドやドロップダウンにも対応
  • click() — 要素をクリック。ボタンやリンクの操作がシンプルに
  • wait() — 条件を満たすまで待機。ページのロード完了や要素の出現を検知
  • navigate() — URLへ遷移。ページ遷移も一行で記述可能
  • capture() — スクリーンショットを取得。テスト結果の記録に便利
// EGO Browser APIの例
const page = await ego.createSpace();
const snap = await page.snapshot();
await page.fill("@e3", "Tokyo");
await page.click("@e5");

⑧ Snapshot: ページをセマンティックビューに圧縮

  • Snapshotはページ全体を要素参照付きのセマンティック構造に圧縮する。各要素に@e1、@e2のような参照ラベルが付与される
  • 見た目のHTMLではなく、意味的な構造を抽出するため、AIエージェントがページを理解しやすい
  • 不要なスタイルや隠し要素を取り除き、エージェントにとって重要な情報だけを残す仕組み
  • これにより、API呼び出しの回数を大幅に削減。1回のsnapshotでページ全体を把握できる
snapshot出力の例:
[button] @e1 "Search"
[input]  @e2 "Enter city..."
[link]   @e3 "Tokyo" → "/tokyo"
[link]   @e4 "Osaka" → "/osaka"

⑨ カスタムChromium: 困難なページも処理可能

  • EGO Lightは独自のChromiumエンジンをベースにしている。一般的なブラウザ自動化ツールとは異なり、エンジンレベルで対応
  • iframeの内部コンテンツも正しく読み取れる。他のツールが苦手とする領域
  • Shadow DOMの隠れた要素にもアクセス可能。コンポーネントベースのUIにも強い
  • Stripeの支払いウィジェットやSalesforceの埋め込みコンポーネントも正しく操作できる
  • Reactポータルで描画される要素も、エンジンレベルで追跡して正確に参照できる

⑪ 従来のブラウザ自動化との違い

  • Selenium / Playwright: DOMセレクタに依存。ページの微細な変更で壊れやすい
  • Computer Use型エージェント: スクリーンショットから座標を推定。遅くて不正確な場合が多い
  • EGO Light: エンジンレベルの参照（@e1, @e2）を使うため、DOM変更に強い。スクリーンショット不要で高速
  • ツールコール数が少なく済むため、トークン消費とレイテンシを大幅削減。実務での利用コストに直結
セレクタ → 座標 →
参照ラベル。
進化の系譜です。

⑫ 開発者向け: QA・ステージング・管理画面

  • QA（品質保証）: エージェントが自動的にページを巡回し、フォームの動作確認やリンク切れの検出を行える
  • ステージング環境: デプロイ前に管理画面の操作テストを自動化。人的エラーを減らす
  • 管理パネル: データの登録・編集・削除をエージェントに任せることで、定型作業を自動化
  • CI/CDパイプラインに組み込んで、デプロイ後の自動回帰テストを実行する用途にも適している

⑮ 仕組みの全体像: ブラウザの中にエージェント環境

  • ユーザーが普段使うブラウザタブとは別の入り口が用意されている。エージェントはそこからアクセス
  • 各SPACEには独立したCookie・ストレージ・セッションが割り当てられる。完全な隔離を実現
  • エージェントの作業内容はユーザーが可視化できる。何をしているのか見えない「ブラックボックス」ではない
  • 必要に応じてユーザーが介入することも可能。Human-in-the-Loopな設計思想
