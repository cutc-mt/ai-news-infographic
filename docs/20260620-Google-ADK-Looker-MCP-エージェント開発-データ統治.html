エージェント開発とデータ統治を繋ぐ最強のブリッジを徹底解説

① 企業データとAIエージェントの架け橋

  • エージェントが企業データをビジネスの文脈で理解するには、単なるデータベース接続では不十分
  • 統治されたセマンティックレイヤーへのブリッジが必要。それを実現するのがこの技術スタック
  • 目標は「もろくて生のSQLを書くエージェント」から脱却し、「自然言語で定義されたデータモデルと直接やり取りするエージェント」へ移行すること
  • Lookerの集中モデリングとガバナンスを活用することで、エージェントが取得する情報がすべて検証済みの信頼できる情報源に根ざしていることを保証
データベースに繋ぐだけじゃダメ。

セマンティックレイヤーが鍵です。

② MCP（Model Context Protocol）とは

  • MCPはエージェントがLookerの能力を発見・呼び出せるようにする共通言語
  • エージェントは複雑なAPI呼び出しの仕方を知る必要がない。標準化されたツールのセットを見るだけ
  • これらのツールはLookerのモデリングレイヤーに直接マッピングされているため、メタデータの取得、モデルの探索、クエリの実行が可能
  • LookMLで定義済みのディメンション・メジャー・アクセスフィルターを尊重したまま操作できる
ビジネスロジックはLookerに、
実行はエージェントに。
責任の分離ができるのがポイント。

③ 従来のエージェントの問題点

  • 従来のエージェントはもろい生のSQLを直接書くアプローチに依存
  • スキーマ変更やテーブル名の不一致で簡単に壊れる。データベースの内部構造に強く結合
  • セキュリティやアクセス制御のルールをエージェント自身が管理する必要があり、リスクが高い
  • ビジネスロジックがエージェント側に散在し、統治が不可能になる状態
SQLを直書きさせると
ガバナンスが吹き飛びます。
生SQLからの脱却が急務。

④ MCP Toolbox for Databases

  • MCP ToolboxはデータソースをMCPツールとして公開するためのプロダクション対応のオープンソース
  • あらかじめ構築されたLookerツールセットが含まれており、エージェント用の能力のライブラリとして機能
  • 新しいプロジェクトごとにカスタム統合コードを書く必要がない。エージェントをToolboxに向けるだけでOK
  • Lookerプロジェクトファイルの監査から統治されたクエリの実行まで、複雑なタスクをすぐに継承
統合レイヤーを毎回ゼロから
書く必要がなくなりました。
Toolboxが重労働を引き受けます。

⑤ Toolboxによるスケーリングとカスタマイズ

  • エージェントごとに利用可能なツールを指定できる。用途に応じて最適なツールセットを構成可能
  • Toolboxが重労働を処理するため、エージェントシステムをスケールさせても統合レイヤーを毎回作り直す必要がない
  • データ分析専用エージェント、LookML改善専用エージェントなど、目的別にカスタマイズできる
  • オープンソースなので、コミュニティの貢献や拡張ツールも期待できる

⑥ LookML — ビジネスロジックの中心

  • LookMLモデリングが中心に位置し、ビジネスロジックを一元化して統治された信頼できる情報源を確立
  • ディメンション（切り口）・メジャー（集計値）・アクセスフィルターをLookMLで一貫して定義
  • データチームが中央でルールを管理するため、エージェントが勝手に危険なクエリを発行するリスクを排除
  • LookMLを変更すると、エージェントのコンテキストに自動的に反映される仕組み
「定義は1箇所、利用はどこでも」
DRY原則がデータでも実現。

⑦ トランスポート層 — MCPとGenAI Toolbox

  • AIとLookerを接続するトランスポート層はMCPとGenAI Toolboxで構成
  • この層は万能翻訳機として機能。メタデータの取得やクエリの実行など、Lookerの内部機能をすべて標準化されたツールとして公開
  • ツールの定義はtools.ymlという設定ファイルで管理。構成が変更されても設定ファイルを更新するだけ
  • エージェントはAPIの複雑さを意識せずに、発見可能なツール群としてLookerの機能にアクセス
tools.ymlに書くだけで

⑧ アプリケーション層 — Agent Development Kit

  • Google Agent Development Kit（ADK）がLLMとエージェントのオーケストレーションを担当
  • LLMはADKで構成され、MCP経由でツールを発見して呼び出すフロー
  • Lookerの統治された環境と対話しながら根拠のある推論を実行する。技術タスクの実行にも対応
  • 好きなモデルを選択可能。ADKの柔軟性により、用途に合わせて最適なLLMを利用できる

  • 3つの層が疎結合で連携。各層を独立して拡張・交換可能

⑩ グランデッド＆ガバナンスの実現

  • エージェントは常にLookerの統治環境に接続されたまま動作。データへのアクセスが一貫して管理される
  • MCPブリッジはLookerインスタンスと常時同期。LookMLへの変更がすぐにエージェントに反映
  • ダッシュボードの枠を超えて、データをエージェントアプリケーションで利用可能にする拡張
  • アクセスフィルターが自動適用されるため、エージェントが権限外のデータにアクセスすることはない

⑪ ツール発見の仕組み

  • ツールはLookerのモデリングレイヤーに直接マッピングされているため、探索→実行まで一貫したフロー
  • Toolboxに新しいデータソースを追加すれば、エージェントは自動的に新しいツールを認識

⑫ LookML改善の自動化

  • エージェントはLookMLプロジェクトファイルの監査を実行可能。不要な定義や重複を自動検出
  • LookMLの洗練も自動で実行。エージェントが統治された環境にアクセスできることを利用して改善を提案
  • データ分析タスクもエージェントが自律的に実行。Lookerのディメンションとメジャーを尊重したクエリを発行
  • 人間が直接SQLやLookMLを触る手間を大幅に削減。エージェントが安全な範囲で自動対応

⑮ セキュリティとアクセス制御

  • Lookerのアクセスフィルターが自動的に適用されるため、エージェントが権限外のデータに触れることはない

  • ツールごとの権限スコープを指定可能。エージェントに最小限の権限だけを付与
  • すべてのクエリがLookerの統治レイヤーを通過するため、監査ログとトレーサビリティが確保
