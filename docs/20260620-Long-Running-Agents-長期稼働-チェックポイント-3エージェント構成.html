Long Running Agents

長期間稼働エージェントの設計原則 — 数時間から数週間動き続けるAI
2026年6月20日

① Long Running Agentsとは

  • セッションをまたいで状態を維持し続けるエージェントの設計パラダイム
  • 稼働期間は数時間から数日、さらには数週間に及ぶ
  • 一度きりのデモではなく、実際のビジネスプロセス全体を自動化するのが目的
  • Google Enterprise Agent Platformが提唱する次世代のアーキテクチャ
「デモが動く」と「仕事が終わる」は
まるで別の問題なのです。

③ 必須条件1: 真にスリープできること

  • アクティブポーリングは絶対NG。CPUを食い続ける古い方式はスケールしない
  • 正しい起動トリガーはイベント駆動のみ。必要な時だけ目を覚ます
  • Webhook受信で外部システムからの通知に反応
  • スケジュール実行で定期タスクを正確に実行
  • 人間の承認を待機中は完全にスリープ状態へ
  • ツールコールバックで非同期処理の完了を検知
ポーリング＝ずっと起きている。
スリープ＝必要な時だけ覚ます。
イベント駆動が唯一の正解。

④ 必須条件2: 全ステップにチェックポイント

  • コンテナがクラッシュしても、再デプロイ後に正確に再開できる仕組み
  • 各ステップの完了状態を永続ストレージに保存
  • 復帰時に「どこまで終わったか」を正確に認識し、続きから再開
  • 幻覚の記憶は一切なし。状態は全て外部に保存し、LLMのコンテキストに依存しない
  • メモリーバンク＋セッション管理で長期記憶を実現
人間が寝て起きても「昨日の続き」を
覚えているのと同じ仕組みを
エージェントにも必要なのです。

⑤ 必須条件3: 3エージェント構成

  • 自己評価は絶対NG。エージェントは自分の出力を常に過大評価する
  • Anthropicの研究で、LLMは自分の仕事に点数をつけると常に甘くなることが実証済み
  • Plannerが計画を立て、Generatorが作業を実行、Evaluatorが独立して評価
  • 作業者と評価者を分離することで、品質の検証ループが初めて成立する
自分で作って自分で採点＝
ズルの温床。
第三者評価が必須です！

⑥ 3役割の詳細な分担

  • Planner: タスク全体を分析し、段階的な実行計画を作成。何をいつやるかを設計
  • Generator: 計画に従って実際の作業を実行。コードを書いたり文章を作成
  • Evaluator: 出力品質を客観的に評価。要件を満たしているかを独立して判断
  • 評価が不合格ならGeneratorにフィードバックを戻して修正ループを実行
  • この分離により品質保証が初めてエージェントで成立する
ソフトウェア開発の
PM＋エンジニア＋QAと同じ構造。
人間のチーム設計をそのまま応用。

⑦ 壁1: コンテキスト劣化の問題

  • LLMのコンテキストウィンドウには限界があり、いっぱいになると品質が急激に低下する
  • 長時間稼働するほど対話履歴が蓄積し、新しい情報の処理能力が圧迫されていく
  • 「毎回ゼロから始める」のは解決策にならない。過去の成果が全て失われるから
  • 必要なのは「文脈を圧縮して保持しつつ、新しい情報を取り込む」インテリジェントな記憶管理
人間も「寝る前に情報を整理」するように、
エージェントにも
記憶の整理が必要です。

⑧ 壁2: 永続状態の欠如

  • 状態を保存する仕組みがないと、毎回「初対面」の状態に戻ってしまう
  • 状態なしで長時間動かすとドリフトが発生。本来の目的から徐々に逸脱
  • 重大な結果として破壊的動作を引き起こすリスク。作業成果を上書きしてしまう
  • 最悪のケースではエージェントが諦めてしまう。「何をやっていたか」を忘れた状態に
記憶がない人間は毎朝「自分は誰？」状態。
エージェントも同じ。
永続メモリが命綱。

⑨ 壁3: 自己検証の問題

  • エージェントは悪い仕事をした後でも「完璧にできました！」と報告してしまう
  • 品質基準が存在しない、あるいは自己判断に委ねられているのが最大の問題
  • 「本当に終わったか」を自分で判断できない。無限ループや品質低下に陥る
  • Anthropicの研究でLLMは自己評価で一貫して過大評価するバイアスが実証済み
自分の宿題を自分で採点して
いつも100点をつける子。
だから第三者のEvaluatorが必要！

⑩ ブレイクスルー1: ハーネスの進化

  • Plan, Build, Evaluateを分離したハーネス設計が登場
  • 単一エージェントに全役割を押し付ける方式から、責任を分離する方式へ進化
  • ハーネスがオーケストレーションを担当し、各エージェントは単一責任に集中
  • フィードバックループが自動化され、品質の悪い出力は自動的に再生成される
  • Plan→Build→Evaluate→Planの反復サイクルで品質を段階的に向上
ソフトウェア開発の
「計画→実装→レビュー」を
エージェントの世界で実現。

⑪ ブレイクスルー2: 永続メモリパターン

  • ライブプランをMarkdownで保存。計画そのものが状態の「真実の源」になる
  • 変更ログ＝ラボノート。各ステップの意思決定と変更内容を記録し続ける
  • Ralph Loop: 計画→実行→評価の反復を、外部ファイルに状態を保存しながら継続
  • 状態はLLMのコンテキストではなく外部ストレージに保存。再起動しても失わない
  • 人間がいつでもプランの進行状況をMarkdownとして確認可能
プランをファイルに書き出す＝
人間もエージェントも
同じ情報を見られる。
透明性の鍵です。

⑫ ブレイクスルー3: インフラの製品化

  • セッション管理: エージェントの稼働ライフサイクル全体を管理する仕組み
  • メモリーバンク: 長期記憶を保持し、セッションをまたいでアクセス可能にする層
  • セッション＋メモリーバンクの組み合わせでエージェントの長期記憶が実現
  • Gemini Enterprise Agent Platformでこれらのインフラが製品化された
  • 自前でゼロから構築せずとも、プラットフォームとして利用可能になった
Googleがインフラを製品化したことで
Long Running Agentsの導入ハードルが
大きく下がりました。

⑬ 応用例: 社員オンボーディング

  • 新入社員のオンボーディングは数日〜数週間かかる複雑なプロセス
  • 書類手続き、システムへのアクセス権設定、研修スケジュールの調整など多岐にわたる
  • 各ステップで人間の承認が必要。承認待ちの間はスリープして待機
  • Long Running Agentなら承認を得たら自動的に次のステップに進行
  • 進行状況はライブプランでいつでも確認可能。管理者の負担も大幅に削減
「承認待ち」で止まる従来のフローを
エージェントが自動継続する。
HR担当者の夢が叶います！

⑭ 応用例: ローン処理ワークフロー

  • ローンの申請から審査・承認・実行まで複数日にまたがるプロセス
  • 審査ステップでは外部APIとの連携が必要。信用情報の照会や書類の検証
  • 人間による承認判断が必要な段階ではスリープし、Webhookで承認通知を受信
  • 途中でエラーが発生しても、チェックポイントから再開可能。最初からやり直しは不要
  • 処理時間の短縮と人間への手作業の削減。両方の恩恵を同時に享受
金融業界ではコンプライアンスが
必須。人間の承認を挟みつつ
エージェントが効率化する理想形。

⑮ 応用例: マルチ日の署名ワークフロー

  • 複数人の署名が必要な契約プロセスは時間がかかる代表的な業務
  • 署名者ごとのステータス管理、リマインド送信、順番の制御などが自動化の対象
  • 一人が署名したら次の署名者に自動通知。全員完了まで追跡
  • チェックポイントにより署名の順序と状態が正確に保持される
  • 法務部門のボトルネックを解消し、契約のサイクルタイムを大幅に短縮
「署名待ち」はビジネスの大きな
摩擦要因。
Long Running Agentが
その摩擦を取り除きます。

⑯ 共通する要件: 人間をループに含める

  • 全ての応用例に共通するのは人間の承認や判断がプロセスに含まれること
  • 人間の応答を待つ間に完全にスリープし、リソースを消費しない
  • 承認が完了したらWebhookやコールバックで自動的に再開
  • 人間はライブプランを通じてエージェントの状態と進行をリアルタイムに確認可能
  • 必要に応じてエージェントを一時停止・修正・再開できる柔軟性が鍵
人間とAIの協調作業。
AIが自動化でき、人間が判断すべき
部分を明確に分離できるのがポイント。

⑰ アーキテクチャ全体像のまとめ

  • 3層構造: インフラ層（セッション＋メモリーバンク）→ ハーネス層（Plan/Build/Evaluate）→ エージェント層（Planner/Generator/Evaluator）
  • イベント駆動の起動トリガーが、エージェントを無駄なく目覚ませる
  • チェックポイントが各ステップの進行を記録し、クラッシュ後も正確に再開
  • Markdownのライブプランが人間とエージェントの共通の情報源になる
  • Evaluatorの独立評価が品質の最終保障。自己評価の罠を回避する
インフラが状態を守り、
ハーネスがフローを制御し、
エージェントが仕事をする。
責任分離の完成形です。

⑱ まとめ: エージェントの次のステージ

  • Long Running Agentsは「一度動いて終わり」のエージェントから、継続的に稼働し続ける次世代の形
  • スリープ、チェックポイント、永続メモリの3つが必須の設計パターン
  • 3エージェント構成（Planner + Generator + Evaluator）が品質保証の最小構成
  • Gemini Enterprise Agent Platformがインフラを製品化し、導入ハードルを大幅に下げた
  • オンボーディング、ローン処理、契約ワークフローなどビジネス上時間のかかる業務こそが最大の応用領域
  • エージェントの進化は「賢さ」から「継続性と信頼性」の段階へ移行した
AIが「賢い」だけでは仕事は終わらない。
最後までやり切る仕組みこそが
ビジネス価値を生み出します。
