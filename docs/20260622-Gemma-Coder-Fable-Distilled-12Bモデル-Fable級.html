シンプルに「Gemma」と呼んでいます！

②モデルの正体: Gemma 4のファインチューン版

  • ベースはGoogleのGemma 4 12B Instruct
  • コーディング専用にファインチューニング
  • ターミナル作業とツール利用に最適化
  • 推論とマルチステップのエージェントタスク向け
  • パラメータ数はわずか12B（120億）
12Bというサイズは、ローカルPCでも
比較的簡単に動かせる大きさです

③エージェント能力: コーディング＋自律動作

  • 利用可能な情報を自動的に読み取る
  • 情報を分析し、ツールを呼び出して変更を実行
  • 作業結果を自分で検証する能力
  • マルチステップのワークフローに対応
  • ファイルの読み取り・変更・ツール呼び出しを一連で実行
作成者は「小さなローカルのコーディング＆
ツール利用エージェント」と表現しています

④プライバシー重視: 完全ローカル動作

  • すべてローカルで実行、API費用ゼロ
  • データはマシン内に留まり、外部に送信されない
  • ダウンロード後はオフラインでも動作可能
  • Apache 2.0ライセンスでGGUF形式で公開
  • llama.cpp、Ollama、LM Studioで簡単に利用可能

⑤ベンチマーク: Tau 2 Bench Telecom で3.5倍

  • テスト対象：Tau 2 Bench Telecom（マルチステップのツール利用）
  • ベースGemma 4 12Bのスコア：約15%
  • ファインチューン版のスコア：約55%
  • ベースモデル比で約3.5倍の性能向上
  • Q8量子化版での20タスクのローカルテスト
ただし、これは小規模テストの結果。
公式リーダーボードとの直接比較はできません

⑥現実的な評価: フロンティアモデルには遠い

  • 大型フロンティアモデルは同じ領域で90%以上を記録
  • MMLU Pro（一般知識）ではベースモデルよりやや劣る
  • 特定用途に特化したモデル＝一般的なチャットには不向き
  • 作成者も「フロンティアモデルと競合するわけではない」と明言
一般知識を少し犠牲にして、
コーディングとエージェント動作を強化する
トレードオフ設計です

⑦量子化オプション: 4種類のサイズを選択

  • Q3 KM：約5.7 GB（最小・試用向け）
  • Q4 KM：約6.87 GB（推奨のスイートスポット）
  • Q6 K：約9.11 GB（品質重視）
  • Q8：約11.8 GB（最高品質）
  • 多くの環境ではQ4 KMから始めるのがおすすめ
Q4 KMならメモリ使用量・速度・
品質のバランスが最適です！

⑧推奨サンプリング設定: 正しい設定が必須

  • Temperature：1.0
  • Top P：0.95
  • Top K：64
  • Repetition Penalty：1.1
temperature: 1.0
top_p: 0.95
top_k: 64
repeat_penalty: 1.1
repetition penaltyを設定しないと、
「0000」のような壊れた出力が
続いてしまうことがあります

⑨Ollamaで導入: ターミナルから素早くセットアップ

  • Hugging FaceからGGUFファイルをダウンロード
  • Modelfileを作成して推奨パラメータを記述
  • ollama create gemma-for-agentic-f -f Modelfile
  • ollama run gemma-for-agentic で起動
# 手順（3コマンド）
wget <gguf-url>
ollama create gemma-for-agentic-f -f Modelfile
ollama run gemma-for-agentic
Gemma 4の新しいアーキテクチャを使っているので、
Ollamaを最新版にアップデートしてから
インポートしましょう

⑩LM Studioで導入: GUI派にやさしいルート

  • グラフィカルインターフェースで簡単に操作
  • アプリ内からGGUFをインポート、またはターミナルでlms import
  • 同じ推奨サンプリング設定を適用
  • 開発者セクションでローカルAPIサーバーを起動可能
  • OpenAI互換APIでAPIキーなしで外部ツールに接続

⑪外部ツール連携: Zedエディタでコーディング

  • Zedエディタのエージェントパネルやインラインアシスタントで利用
  • Ollama環境ならモデルを自動検出
  • LM StudioならLM Studioプロバイダーから選択
  • コンテキスト長はデフォルト4,096トークン（8,122推奨）
  • ハードウェアのメモリ余裕があればさらに拡大可能
ZedはOllama接続時のコンテキストが
短めなので、設定画面から拡張推奨！

⑫安定性の問題: 実用にはまだ早い

  • 動くときは驚くほど良い結果を出すが、不安定さが目立つ
  • 同じ出力を繰り返したり、フリーズすることがある
  • ツール呼び出し用の特殊トークンが漏出する問題
  • マルチステップのワークフローが正常に完了しないケース
  • レビュー結果：デイリー利用の信頼性に欠ける

⑬バグの原因: モデル重みではなくクライアント側

  • モデルカードによると、クライアントやサンプラーの設定が原因
  • フロントエンドがGemma 4のネイティブツール形式を理解していない可能性
  • 正確なランタイム・チャットテンプレート・ツールパーサーが必要
  • ユーザーにとっては「動かない」なら原因はどうでも同じ
理論的には修正可能ですが、
必要な環境が揃わないと
期待通りの動作になりません

⑭良い面: ワンショットタスクで意外な実力

  • 一発のコーディング質問は比較的高い精度
  • 関数のリファクタリングや自己完結型のソリューション生成に対応
  • 12Bモデルとは思えない高い持続性を発揮
  • 技術的なプロンプトをベースモデルより有用に処理する場面
  • 期待以上の結果が出るときは「Fable級」の感触あり

⑮使い分けガイド: どう活用すべきか

  • 実験・検証：12Bの限界を試すのに最適
  • ワンショット：小規模なコード生成や質問に有効
  • 学習用：ローカルエージェントの動作を理解する教材に
  • 日常のエージェント開発には信頼性不足
  • 大規模リポジトリの変更は必ず監視が必要
「動くときは動く」がキーワード。
信頼性を求めるなら
より大型のモデルがまだ安心です
