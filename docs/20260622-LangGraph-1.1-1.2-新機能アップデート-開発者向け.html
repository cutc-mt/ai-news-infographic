LangGraph 1.1 & 1.2 新機能アップデート

開発者向け最新機能を徹底解説 — ミドルウェア・プロファイル・デルタチャネル・グレースフルシャットダウン
2026年6月22日

① モデルプロファイル: LLMの能力情報を取得

  • LangGraph 1.1で導入されたprofile属性。LLMインスタンスからモデルの能力情報を取得できる
  • 取得できる情報：最大入力トークン、最大出力トークン、ツール呼び出し対応、構造化出力対応、画像入力対応
  • GPT-4o-mini の場合：入力128k、出力16k、ツール呼び出し対応、構造化出力対応
  • モデルの仕様をAPIを呼ばずにローカルで確認できるため、エラーハンドリングに役立つ
プロファイルがあれば
「このモデルはツール呼べる？」
をコードで判定できる！

② プロファイルで条件分岐: ツール対応をチェック

  • 複数モデルを使う場合、プロファイルの値で条件分岐が可能
  • ツール呼び出しに対応している場合だけ bind_tools() を実行
  • 対応していないモデルにはツールを紐付けない。無駄なバインドを回避
  • シンプルなコードで複数モデルの互換性問題を解決できる
if model.profile.tool_calling:
    model = model.bind_tools(tools)
else:
    # ツールをバインドしない
モデルごとの対応力を
自動判定して安全に
ツールを使い分けよう！

③ プロファイルでプロンプト長を事前チェック

  • プロンプトが最大入力トークンを超えるかをAPI送信前にチェック可能
  • 長いプロンプトをAPIに送ってエラーになるのを事前に防ぐ
  • プロファイルは with_structured_config() で上書きすることも可能
  • 上書きのユースケースは限定的だが、レジストリの情報が間違っている場合などに有用
APIに送る前に
トークン数をチェック。
エラーを未然に防ごう！

④ ミドルウェア: LangGraph × LangChain統合

  • LangChainのミドルウェアがLangGraphの create_agent() で利用可能に
  • 元々LangGraphにあった機能がLangChainパッケージにマージされた
  • create_agent() に middleware を第3引数として渡せる
  • リトライ・要約・モデレーションなど、エージェントの振る舞いを宣言的に設定
agent = create_agent(
    model,
    tools,
    middleware=[...]  # 第3引数！
)
ミドルウェアでエージェントの
振る舞いを宣言的に管理。
設定がぐっと楽になる！

⑤ リトライミドルウェア: API障害に自動対応

  • ModelRetryMiddleware でAPI呼び出しの自動リトライを設定
  • max_retries: 最大3回までリトライ。初期遅延1秒、バックオフ係数2
  • リトライ間隔：1秒→2秒→4秒と指数バックオフで増加。最大遅延30秒
  • タイムアウトエラーと接続エラーに限定。一般的なエラーではリトライしない
ModelRetryMiddleware(
    max_retries=3,
    initial_delay=1,
    backoff_factor=2,
    max_delay=30
)
APIが一時的に落ちても
自動リトライで
サービスを継続！

⑥ 要約ミドルウェア: 長い会話履歴を自動圧縮

  • SummarizationMiddleware がプロファイルと組み合わさって超強力に
  • 会話履歴のトークン数が最大入力トークンの80%を超えたら自動で要約を開始
  • 最新20件のメッセージはそのまま保持。21件目以降を要約モデルで圧縮
  • 要約モデルはチャットモデルと別に指定可能。安いモデルで要約すればコスト削減
SummarizationMiddleware(
    summary_model=cheap_model,
    trigger=lambda: token_ratio > 0.8,
    max_history=20
)
履歴が長くなっても
自動で要約して節約。
プロファイル連携が鍵！

⑦ モデレーションミドルウェア: OpenAI専用

  • OpenAIModerationMiddleware はOpenAIのモデレーションAPIを利用
  • 入力・モデルの出力・ツール呼び出しの結果の3箇所をチェック
  • ツールの結果もプロンプトインジェクションの温床になるためチェック対象
  • violation_message を設定可能。「Blocked」などカスタムメッセージで応答
  • カテゴリフィルターでピザ注文など許容する話題を指定可能
「I will find you and hurt you」
→ 暴力カテゴリでBlocked。
ツール結果も要チェック！

⑧ Provider Strategy: ネイティブ構造化出力

  • LangChain 1.2でProviderStrategy という新しい構造化出力オプション
  • プロバイダーのネイティブ構造化出力APIを直接利用
  • カスタムパースやツール呼び出しを使わず、OpenAIの strict JSON schema で直接取得
  • strict=true で厳密なスキーマ検証を強制。型の保証が強力
  • 条件付きで使う場合はbool値をフィールドに追加して分類器として機能させるのが推奨
ネイティブAPIを直叩き。
パース処理を省略して
正確な構造化データを取得！

⑨ Provider Strategyの注意点: 入力不足の罠

  • 「こんにちは」といったスキーマに関係ない入力を渡すと問題が発生
  • 時々 KeyError が発生、あるいはでたらめな推測値が返ってくる
  • 例えば「margaritaを推測、合計$8.5」など、意味のない結果が生成される
  • 対策：bool型フィールドを追加して、注文があったかどうかを分類させる
  • description 属性を付ければLLMがフィールドの意味を理解しやすくなる
boolフィールドをフラグにして
入力不足のケースを
安全にハンドルしよう！

⑩ イベントストリーミングv3: 超シンプルな書き方

  • 従来：astream_events でイベントタイプを確認し、ネストした辞書をアンパックする複雑なコード
  • v3：for message in agent.stream() で message.text に直接アクセス
  • event.data.chunk.content のような多段階のアンパックが不要に
  • 特定のイベントタイプを知る必要がない場合、v3が推奨される新しい書き方
# v3 - シンプル！
for msg in agent.stream(
    {"messages": messages},
    config=checkpointer_config
):
    print(msg.text, end="", flush=True)
アンパック地獄からの
脱却。コードが
劇的に読みやすくなった！

⑪ デルタチャネル: チェックポイントサイズを劇的に削減

  • DeltaChannel は増え続けるリスト全体を再シリアライズせず、差分だけを保存
  • snapshot_frequency=5 なら、5回に1回だけフルスナップショットを書き込み
  • 読み出し時は少数の差分だけをリプレイすれば復元可能
  • 実行速度への影響はほぼなし。メモリとI/Oの節約だけが目的
チェックポイント肥大化の
究極の解決策。
長時間稼働エージェントに必須！

⑫ デルタチャネルの実績: 5.3GB → 130MB

  • LangChainのコーディングエージェント：200ターン以上の長時間実行
  • 従来のスナップショット：5.3 GBものチェックポイントデータを消費
  • デルタチャネル導入後：約130 MBに削減。約97%の削減
  • 重要：パフォーマンス低下はゼロ。メモリ使用量だけが劇的に改善
5.3GB → 130MB
約40分の1に！
しかも速度は変わらない！

⑬ ノード単位のタイムアウト設定

  • add_node() に timeout 引数を追加。各ノードごとにタイムアウトを設定可能
  • API呼び出しをするノードなど、外部通信を伴うノードにピンポイントで設定
  • グローバルな設定ではなくグラフの定義に直接組み込める
  • タイムアウトすると GraphNodeTimeoutError がスローされる
graph.add_node(
    "slow_api",
    slow_api_call,
    timeout=30  # 30秒でタイムアウト
)
APIが応答しないノードを
個別に守る。
グラフ全体が止まるのを防げる！

⑭ リトライポリシー: ノード単位で再試行を設定

  • retry_policy を add_node() に渡してノード単位のリトライを定義
  • max_attempts で最大試行回数を設定。タイムアウトと組み合わせて使うのが効果的
  • エラー情報にはrun_timeout、elapsed time、attempt countが含まれる
  • 0.3秒のタイムアウトに5秒のスリープ → 2回リトライしてタイムアウトエラーを報告
graph.add_node(
    "bake",
    bake_node,
    timeout=30,
    retry=RetryPolicy(max_attempts=3)
)
タイムアウト×リトライの
最強コンボ。
ノードレベルの信頼性が爆上がり！

⑮ キャッシュポリシー: ノードの結果をキャッシュ

  • add_node() にcache_policy も新設。ノードの実行結果をキャッシュ可能
  • 同じ入力でノードが再実行される場合、キャッシュから結果を返す
  • API呼び出しコストの削減や開発中の反復テストの高速化に有用
  • タイムアウト・リトリーポリシー・キャッシュポリシーの3つを組み合わせてノードを堅牢に
ノードレベルで3つのポリシーが
すべて設定可能に。
グラフが一段と堅牢に！

⑯ ノードレベルのエラーハンドラー

  • on_error_handler() 関数でエラー発生時の遷移先ノードを指定
  • Command クラスで「現在のノード → 補償ノード」へのルーティングを定義
  • 例：注文失敗時に「refund issued」補償ノードに自動遷移
  • add_node() の error_handler 引数でハンドラーを登録
def on_error_handler(state):
    return Command(
        goto="compensate",
        update={"error": str(exception)}
    )

graph.add_node("order", order_fn,
    error_handler=on_error_handler)
エラーで全体が死ぬのではなく
補償アクションに遷移。
本格的なエージェント設計に必須！

⑰ カスタムエラーハンドラー vs タイムアウト

  • カスタムエラーハンドラー：ノードのエラーをグラフ内で回復可能な場合に使用
  • 別のノードへのリルーティングや補償アクションの実行ができる
  • タイムアウトハンドラー：応答が遅いだけのノードに使用。リトライを中心に対応
  • カスタムハンドラーがなければ、例外が全体の実行を殺してしまう
  • 例：ピザハワイはメニューにない → 注文ノードでエラー → 補償ノードで返金処理
回復可能ならカスタムハンドラー
応答遅延ならタイムアウト
両方を使い分けよう！

⑱ グレースフルシャットダウン: 安全に停止

  • Kubernetes等でPodが再起動されると、プロセスが強制終了されて作業が失われる
  • RunControl クラスと request_drain() で安全な停止を実現
  • invoke メソッドに run_control を渡す。drainシグナルで次のノードへの遷移を中止
  • 結果："graph was stopped gracefully"。次のノードの情報と共に正常停止
  • クラウド環境での中間状態のデータ損失を防ぐための重要機能
run_control = RunControl()
result = graph.invoke(
    {"messages": messages},
    config={"run_control": run_control}
)
run_control.request_drain()
KubernetesでPodが殺されても
中間作業を守れる。
本番環境での必須機能！
