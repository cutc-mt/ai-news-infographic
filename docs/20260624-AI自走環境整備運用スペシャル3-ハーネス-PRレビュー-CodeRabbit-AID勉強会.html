⑪ボトルネックは消えない、移動する（明り 山本）

  • AIでコードを早く作るのは当たり前になった。差別化の要因ではなくなった
  • ボトルネックは「作る」から「通す」に移動した。レビュー・結合が詰まる場所に
  • デフォルトでは衝突が起きる前提で設計していたが、これでは負債になる
  • 根本解決: 予防の量が後続の重さを決める。衝突が起きない形にコードを作る
  • 差がつくのはボトルネックの移動を読んで予防で先回りする設計力
本番コードの8割以上がAI生成という時代。通す設計が問われる！

⑫衝突予防設計: 機能の縦割りと「核」の直列化

  • 機能を縦割りにして依存をなくす。複数エージェントが同じ場所を触らない構造
  • どうしても残る共通箇所＝「核」を少数に絞り込む
  • マルチエージェント安全実行の2条件:
    ①派生元を1つの仕様で固定　②核に触るタスクだけを直列化
  • 核だけ直列、残りは全部並列。これが両立する唯一の構造
  • 機能ごとの依存軸を明示的に記載し、結合ポイントを特定
並列開発を安全に行うには、衝突し得る場所を事前に特定して直列化！

⑬ブランチ戦略とレビューのリスクベース切り替え

  • ブランチの深さは衝突管理ではなく、検証の深さで決める
  • 検証が1段で良ければトランクベースでOK。結合を分けたいなら分割する
  • レビューの重さ＝ブランチ段数ではなく核に触れたかどうかのリスクで決まる
  • スライスレベルのPR（核未接触）→軽いレビュー、コードラビットで自動
  • リポ全体を読む必要があるPR（核接触）→重いレビュー、AI+他メンバーで要件確認しながら
差分だけ見れば足りるか？全体を読む必要があるか？リスクで判断！

⑭人間に残る出番: 要件の方向性確認と境界判断だけ

  • AIは仕様通りに動くが、仕様が要件として正しいかは判定できない
  • AIが要件を取り違えるとテストごと整合して綺麗に間違うコードが完成する
  • それを止められるのは要件を分かっている人間だけ。これが唯一人間の出番
  • 人間が握るべきは不明瞭な部分。明確な箇所は全部エージェントに任せる
  • マージの判断は最後の1戦だけ絶対に人間がやる。エージェントにはマージ権限を持たせない
要件の方向性確認と境界判断。これが人間の究極の役割！
