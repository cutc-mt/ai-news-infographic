② 長時間作業を支える4つの柱

  • 持続するスレッド — コンテキストを維持したまま作業を継続
  • 共有メモリ — プロジェクト横断で情報を保管・参照
  • ツールへのアクセス — ブラウザー、コンピューター、コネクターなど
  • 繰り返し実行 — 1つのプロンプトを超えてループとして進み続ける
  • レビュー — 成果物を確認する場所も整っている
この4つが揃うことで
作業がループとして進み続けるんです！

③ 持続するスレッド: ピン止めして使い続ける

  • Codexデスクトップアプリの左側にスレッドをピン止めして配置できる
  • よく戻るプロジェクトや作業をスレッド単位でピン止めして使い続けられる
  • 繰り返し出る質問やフィードバック、世間の反応を見続けるためのスレッドを作成可能
  • 後述のハートビート機能と組み合わせて、スレッド単位でループを回し続けられる
  • 長く続けるとスピードが遅くなったり、コストが高くなることはある点に注意
スピード低下はあるものの
過去のコンテキストを維持したまま
回答してくれるのが嬉しい！

④ 音声入力: 話言葉でリクエストできる

  • AquaVoiceやAmicaなど外部の音声入力サービスもあるが、Codex自体にも音声入力機能が内蔵されている
  • デスクトップアプリを通じて音声で指示を出せる環境が整っている
  • 話言葉でリクエストしても、コンテキストからCodexがいい感じに理解して指示を出してくれる
  • ショートカットにも音声入力を設定できる
外部の音声入力サービスを使ってない人は
ぜひCodexの標準機能を試してみて！
便利さに驚くはず

⑤ ステアリング: 実行中に割り込み指示

  • Codexが動いている途中に次の指示を割り込ませることができる
  • 指示をどんどん蓄積させ、次から次へと連続で指示を送ることが可能
  • デスクトップアプリを使えば手を止めずに次のタスクを詰められるのが特徴
  • 操作性が非常に高く、スムーズな作業フローを実現できる
動いてる最中に「あ、これもやって」
と追加できるのはめちゃくちゃ便利！
手を止めずに詰めるのがCodex流。

⑥ メモリ: 保管庫ディレクトリの構造

  • 公式で推奨する保管庫ディレクトリの切り方を共有している
  • agents.md — エージェントの運用方法を定義するファイル
  • todo.md — プロジェクト横断の優先度とフォローアップを管理
  • プロジェクトごとにディレクトリが存在し、アクティブプロジェクトのインデックスも貼れる
  • エージェント自体の作業の好みなどをマークダウン形式で保管できる
📁保管庫/
├── agents.md（運用方法）
├── todo.md（優先度管理）
├── プロジェクトA/
├── プロジェクトB/
└── notes/（メモ・好み）

⑦ メモリ活用: GitHub連携とレビュー

  • 保管庫をGitHubに置くことで、Codex自身がメモリのレビュー対象にできる
  • Codexが書き留めるに値すると判断したものをレビューで確認できる仕組み
  • 話題が出たら関連するメモを更新
  • プロジェクトが前に進んだらプロジェクトページを更新
  • 案件が閉じたら「クローズ済み」の印をつける
  • これらの運用ルールをagents.mdに定義しておく形式

⑧ Computer Use vs Browser Use vs コネクター

  • Browser Use — ローカルアプリを作るだけならブラウザーユースで十分
  • ログイン情報を使いたい場合はChrome拡張を使うのが推奨。ブラウザーユースでは権限が限定的
  • Computer Use — デスクトップアプリの操作など幅広いタスクに対応
  • コネクター — Slackなどの外部サービスを連携可能
  • 複数のツールを組み合わせたタスクはスキルとして再利用させるのがおすすめ
権限の使い分けが大事！
目的に応じて使い分けましょう。
スキル化で毎回の手間を省く。

⑨ Codex Mobile連携: 寝ながら仕事も可能

  • リモート操作 — Codex Mobileを使えばスマホからCodexを操作できる
  • デスクトップアプリとスマホを連携させて寝ながら仕事も可能（？）
  • ただしCodexにロックインされてしまう懸念点もある
  • 使い勝手は非常に良く、リモート環境でもスムーズに操作ができる
「寝ながら仕事」はジョークっぽいけど
モバイル連携の使い勝手は本物！

⑩ スレッド自動化: ハートビートで定期監視

  • スレッドに「自動化追加」でハートビートを設定できる
  • 何分おきにチェックするかなどタイミング設定が可能
  • 監視し続けて問題があったらSlackに転送など条件付きアクションを実行
  • これは外ループ的な機能。Claude Codeの/loopに近いもの
  • デスクトップアプリ内で完結するため、難しい概念も直感的に使える
ハートビートはすごく便利！
これがCodexの強みの一つ。
外ループをアプリ内で完結させるのがポイント。

⑪ ループの概念: 内ループと外ループ

  • Codexのループ構造: コンテキスト → ツールズ → メモリー → リファレンス → レビュー
  • 内ループ — ジェネレーターとエバリュエーターの分割などミクロなレベル
  • 外ループ — タイマーやイベントトリガーをフックにしたエージェント的な判断
  • 公式の説明はちょっと分かりにくいという印象もある
  • どっちかと言うと今回は外ループ的な話が中心
内ループと外ループの話が混ざると
混乱しがち。マクロかミクロかで
意識を分けると理解しやすいです。

⑫ ループの定義例: 3つのユースケース

  • アシスタント — 30分ごとにSlackやGmailの未読をチェックし、優先付け・回答下書きまで自動化
  • フィードバック監視 — 毎日特定のSlackチャンネルを確認し、上位フィードバックのテーマをまとめて下書きを作成
  • カスタマーサポート — 5分単位で返金リクエストを処理。ユーザー反応に応じて柔軟にモードを変更
返金処理のループが1分ごとの
チェックに切り替わるなど
リアルタイムで柔軟に対応できるのが強み！

⑬ レビュー: 人間がやるべきこと

  • Codexは未読メッセージや関連文脈の確認、返信の下書きまでは準備する
  • しかし承認は人間が行うという明確な役割分担
  • トーンや方向付けも人間の仕事。エージェントが勝手に送信しない仕組み
  • 最終判断とタイミングの決定は人間がコントロール
  • 人間が適切にレビューすることで安全性を担保する設計
エージェントは下書きまで。
送信ボタンは人間が押す。
この役割分担は大事ですね。

⑭ Goalコマンド: 目標設定の重要性

  • /goalコマンドはClaude Codeにもあるが、Codexでも同様の概念が存在する
  • マークダウンのプランファイルをアバウトに渡すと、フェーズ別に実行してくれる
  • Opus 4.8やGPT-5.5環境ではドリフトを起こしやすいので注意が必要
  • ゴールをフレームワークで綺麗に定義することが推奨される
  • 制約、レビュー条件、受け入れ条件を明記した上でゴールを定期するのが重要
/goalは本当にいっぱい使うべき！
条件を明記しないとドリフトする。
フレームワーク使って定義しよう。

⑮ サイドパネル: 作業の場が広がる

  • デスクトップアプリでサイドパネルを簡単に増やせる
  • サイドチャットがタブで追加でき、複数の会話を並行管理可能
  • ブラウザータブも追加できて、Web操作やプレビューをサイドで確認できる
  • レビューでファイルも確認できる
  • サイドパネル上でプレビュー・インタラクション・コメントなど多彩な操作が可能
  • 公式は「サイドパネルがCodexの作業が起こる場所になり始めている」と表現

⑯ スキル化: 再利用性を高める

  • コネクター（Slackなど）やComputer Use、Browser Useを複数組み合わせて1つのタスクを実行
  • 毎回Codexに教えるのではなく、スキルとして保存して再利用させるのが推奨
  • Chrome拡張もComputer Useの一種として利用可能
  • コーディングエージェントを日常的に使っている人にとっては当たり前の話かもしれない
スキル化は時短の基本！
一度作れば何度でも使える。
Codex使いなら必須の知識ですね。
