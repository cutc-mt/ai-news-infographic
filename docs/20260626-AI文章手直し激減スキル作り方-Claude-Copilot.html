AI文章の手直しが激減する「スキル」の作り方

Claude / Copilot で一発で良い文章を出力する秘訣を徹底解説
2026年6月26日

① AI文章の悩み: 出力のたびに手直しが必要

  • AIに文章を書かせても、毎回トーンや文体がブレる問題に悩まされる人が多い
  • 「もう少しカジュアルに」「専門用語を減らして」— 指示を繰り返すたびに修正コストが膨らむ
  • 手直しの時間が生成時間より長くなることすらある。本末転倒の状態
  • このループを断ち切る鍵が「スキル」の作成にある
AIに書かせてから直すのではなく、
最初から正しい仕様を教え込む！

② 「スキル」とは: AIに覚えさせる手順書

  • スキルとは、AIに特定のタスクの手順やルールをまとめた指示ファイルのこと
  • ClaudeならSKILL.mdというファイルに記述。Copilotなら.mdcipファイルを使用
  • 一度設定すれば、毎回同じ品質で出力が得られる。プロンプトを毎回書く手間がゼロに
  • チームメンバーとも共有できる。組織全体の文章品質を統一する強力な仕組み
プロンプトは「毎回渡すもの」。
スキルは「AIに覚えさせるもの」。
この違いが大きい！

③ 手直しが激減する3つの理由

  • 文体の一貫性 — です・ます調、だ・である調など、トーンを固定できる
  • 禁止表現の排除 — 「值得注意的是」「と言えるでしょう」などAIくささの原因を事前にブロック
  • 出力形式の指定 — 見出しの構造、段落の長さ、箇条書きのルールなどを正確に指定
この3つをスキルに書き込むだけで、
手直し回数が激減します。
体感で半分以下に！

④ スキルの基本構造: 4つの要素

  • 役割（Role） — AIに「シニア編集者」「技術ライター」など具体的なペルソナを与える
  • 手順（Procedure） — 作業のステップを順序立てて記述。AIが迷わず実行できるように
  • 制約（Constraint） — 禁止ワード、文字数上限、使用してはいけない表現を明示
  • チェックリスト — 出力前に確認する品質基準。自分でセルフレビューさせる仕組み
この4つを漏れなく書くのが
品質の秘訣！

⑤ Claude Skills: SKILL.mdの書き方

  • Claudeの~/.claude/skills/ディレクトリにSKILL.mdを配置する
  • ファイルの先頭にスキルの名前と説明を2行で書く。これがAIの認識基準になる
  • 手順は番号付きリストで記述。AIが順番に実行できるフォーマットが理想
  • 作成後は対話の中でスキルを呼び出し、実行結果を見て微調整していく
ファイル名は内容が
一目で分かる名前に！
例: blog-writing-style.md

⑥ Copilot Skills: .mdcipファイルの活用

  • GitHub Copilotは.mdcip（Markdown Copilot Instruction Package）形式を採用
  • プロジェクトの.github/copilot-instructions/に配置してチーム共有が可能
  • Lazy Loading機構あり。マニフェストだけ先に読み込み、必要な時に詳細を読み込む設計
  • Gitで管理できるため、バージョン管理とレビューがしやすいのが大きなメリット
Claudeは個人設定、Copilotは
チーム設定に向いている。
使い分けが重要です。

⑦ 役割設定の重要性: ペルソナで品質が変わる

  • 単に「文章を書いて」と言うのではなく、「10年経験のシニア編集者」として指定する
  • 読者層を明示する。「初中級プログラマー向け」で内容の粒度が決まる
  • 目的も明確に。「読者が5分で実践を始められる」など具体的なゴールを設定
  • ペルソナが曖昧だと、AIが一般的で無難な文章しか出力しなくなる
AIは「誰に向けて書くか」を
教えないと何でも屋の文章
になってしまいます。

⑧ AIくささを消す: 禁止表現リストの作成

  • AIが頻出する定番フレーズを禁止リストに登録するのが最も効果的
  • 代表的な禁止表現：「值得注意的是」「〜と言えるでしょう」「まずは〜について」「〜という点が挙げられます」
  • これらの表現はAIの出力に特有のパターン。読者が「AIが書いた」と感じる最大の要因
  • スキルに「出力前に禁止表現をチェックし、置き換える」というセルフレビュー手順を組み込む
禁止リストだけで
文章の「AI感」が劇的に
下がります！

⑨ 出力形式の指定: 構造で品質を担保する

  • 見出しのレベルをH2→H3→H4の順序で統一。階層構造を明確に定義する
  • 段落は3〜5文以内に制限。長い段落は読者の離脱原因になる
  • 箇条書きは3〜7項目を推奨。項目ごとにアイコンや絵文字を付けるルールも設定
  • 全体の文字数や導入部の長さも指定可能。「最初の段落は3文以内」など
PREP法（Point→Reason→Example→Point）
など構造の型を指定すると
論理的な文章が安定して出ます。

⑩ XMLタグで指示を構造化する

  • 元のテキストを<original>...</original>で囲む。AIが対象と指示を区別できる
  • 修正ルールを<rules>...</rules>でグループ化。XMLタグはClaudeが高い精度で認識
  • 意図しない修正を防止できる。対象テキストと指示が混ざる事故を防ぐ仕組み
  • ClaudeはXMLタグの構造を学習済み。マークダウンよりも確実に指示を伝えられる
XMLタグはAIにとっての
「仕切り」として機能。
指示の精度が跳ね上がります。

⑪ 文体の固定: 一貫したトーンを出力させる

  • です・ます調かだ・である調かを明示。混在を許可しないルールを設定
  • 語り口のトーンも指定。「親しみやすい口語体」か「厳密な専門誌調」か
  • 敬語のレベルも統一。「Desu/Masu調で丁寧語を使用」など細かく指定
  • 実際の良い文章のサンプルを2〜3個スキルに添えるとAIがトーンを正確に模倣する
サンプル添えるだけで
出力の質が別物に。
例示は最強の武器です。

⑫ テストと反復: スキルを磨き上げる

  • 作成したスキルを実際のタスクでテストする。期待通りの出力か検証するステップが必須
  • 初回は完璧でなくてOK。出力を見て修正を加えることを2〜3回繰り返して精度を上げる
  • A/Bテストも有効。スキルあり/なしで出力を比較し、品質差を数値化する
  • 良い出力例を「良い例」としてスキルに蓄積していく。AI自身の学習材料にする
スキルは生き物です。
使い続ける中で
どんどん賢くなっていきます。

⑬ 実践例: ブログ記事執筆スキル

  • 役割: SEOを意識したテックブロガー。読者は初中級のエンジニア
  • 手順: ①タイトル案3つ提示 ②構成案を作成 ③本文執筆 ④セルフチェック
  • 禁止: 「〜という点が」「〜と言えるでしょう」「值得注意的是」等の定型句
  • 形式: 導入3文、段落は3〜5文、H2見出しの間に箇条書き1つ以上
このスキル1つで、
ブログ記事の手直しが
80%以上激減します。

⑭ 実践例: ビジネスメール作成スキル

  • 役割: 丁寧で簡潔なビジネスパーソン。社内向けと社外向けでトーンを切り替える
  • 手順: ①目的を1行で要約 ②背景を2文で補足 ③要件を箇条書きで提示 ④結びの言葉
  • 禁止: 感情的な表現、曖昧な期限（「なるはや」等）、長い前置き
  • 文字数: 300字以内を原則。長くなる場合は添付ファイルで詳細を補完
ビジネスメールは
短さが正義。
AIに制約を設けましょう。

⑮ チーム共有と標準化

  • Claude SkillsはGitリポジトリで管理。チーム全体で同じ品質基準を共有できる
  • Pull Requestでスキルの変更をレビュー。プロンプトもコードと同じように管理する文化
  • Copilotは.github/に配置するだけでチーム全員に自動適用されるのが便利
  • 導入前後で文章の修正回数やレビュー時間を比較。定量的に効果を測る
1人のスキルを共有すれば
チーム全体の文章品質が
一気に底上げされます。

⑯ Claude vs Copilot: スキル機能の比較

  • Claude: 個人の裁量が大きく、細かいニュアンスの調整に優れる。手動で有効・無効を切り替え可能
  • Copilot: チームスケールでの一括適用が強み。Lazy Loadingでパフォーマンスを確保
  • Claudeは「深さ」、Copilotは「広さ」。用途に応じて使い分けるのが正解
  • 両方を併用するユーザーも増加中。個人作業はClaude、チーム作業はCopilotという分け方が主流
対立ではなく補完関係。
両方使える人は
確実に生産性が高いです。

⑰ 効果の測定: 手直し削減を数値で確認

  • スキル導入前後で平均修正回数を比較。1回の生成で完成する割合が目安
  • かかった時間を測定。「生成→確認→修正」のサイクルタイムが短縮するか確認
  • チームメンバーへのアンケートも有効。「AI出力の使いやすさ」を5段階で評価
  • スキルを継続的に改善することで、数週間で品質が安定するサイクルに入る
数値で効果を示せば
組織導入の説得力が
格段に上がります！
