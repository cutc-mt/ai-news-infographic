① Googleが50ページの白書を公開

  • 著者はAddy Osmani、Shubham Saboo、Sokratis Kartakisの3名。KaggleのAIエージェントコースの一環として無料公開
  • タイトルは「The New SDLC with Vibe Coding」。約50ページでAI時代のソフトウェア開発ライフサイクルを体系的に解説
  • 核心のメッセージは1行で要約できる：「構造がスケールする、気分（バイブ）はしない」
  • エンジニアリングリーダーとアーキテクトが必読の文書として話題になっている
この白書は単なる解説ではなく、
AI時代の設計思想そのもの。
1午後で読めるのが嬉しいポイント。

② 2026年の開発者AI利用の現状

  • プロの開発者の85%がAIコーディングツールを日常的に使用（JetBrains調査）
  • 51%が毎日AIツールを使っている。もう「AIを使うかどうか」ではなく「どう使うか」の時代
  • 新規コードの約41%がAI生成と推定。標準的な10ファイルの機能なら4ファイルは人間が書いていない
  • コードをレビューする時間が書く時間を超える開発者が、もう「平均的」な存在に
「AIを使っている」と言う開発者が
少数派ではなくなった。議論の軸は
使う品質に移っている。

③ バイブコーディングとは何か

  • 2025年2月にAndrej Karpathyが命名。「気分に任せて」自然言語で指示し、出力を受け入れるスタイル
  • 壊れたらエラーメッセージをプロンプトに貼り付け直す。これの繰り返し
  • プロトタイプ、ハッカソン、個人のスクリプトには適している。思考スピードで試せるのが強み
  • ただし検証が軽視されがちで、コードベースへの理解も浅いまま。本番環境には不向き
「バイブコーディング」はウイルスのように
拡散したが、2026年には過去の遺物
のようになりつつある。

④ Agentic Engineeringの登場

  • Karpathy自身がバイブコーディングの枠が狭すぎると認め、Agentic Engineeringを導入
  • 2025年12月にはKarpathyのコードの約80%がAIエージェントによるものだったと報告
  • AIを制約・テスト・フィードバックループの枠組みの中に配置する規律あるアプローチ
  • 生産システムやチームスケールの開発に適した「真剣モード」と言える
問いはもはや「AIに書かせるか」
ではなく「どうガバナンスするか」。
ここが全てを分ける。

⑤ スペクトラムで考える3つのモード

次元 バイブコーディング 構造化AI支援 Agentic Eng.
意図 カジュアルな自然言語 詳細な制約付きプロンプト 正式な仕様書
検証 「動いてる？」 手動テスト 自動テスト+CI
リスク 高 中 低
適用範囲 プロトタイプ 機能開発 本番システム
白書のCTOテスト：
「決済システムをバイブコーディングで
作っている」→警報が鳴るべき。

⑥ 検証こそが最大の分かれ道

  • テストは決定論的な部分をカバー。入力Aに対して出力Bが返ってくるか
  • 評価（Evals）は非決定論的な部分をカバー。エージェントが適切な経路を選んだか
  • テストとEvalsの両方が揃って初めてAgentic Engineeringと呼べる
  • どれだけ洗練されたプロンプトでも、検証がなければそれはバイブコーディングに過ぎない
「テストとEvalsがエージェントへの
意図伝達の手段になる」
この発想の転換が重要です。

⑦ Agent = Model + Harness（ハーネス）

  • 白書の核心公式：エージェント ＝ モデル ＋ ハーネス。モデル単体ではエージェントにならない
  • ハーネスにはルールファイル（AGENTS.md等）、ツール・MCP、サンドボックス、ガードレール、監視ツールが含まれる
  • ハーネスはあなたのチームが所有する部分。モデルプロバイダーが管理するものではない
  • 生のモデルは車のエンジン。ハーネスは車体と道路と交通法規。エンジンだけでは走れない
「ほとんどのエージェントの失敗は、
正直に調べると構成（ハーネス）の失敗」
モデルのせいにしないこと。

⑧ ハーネスがスコアを劇的に変える実証

  • Terminal Bench 2.0で、あるチームがコーディングエージェントをトップ30圏外からトップ5へ
  • 方法は驚くほどシンプル。モデルの変更は一切なし。ハーネスだけを改善しただけ
  • LangChainの別の研究では、固定モデルのままでシステムプロンプトとミドルウェアを調整し、13.7ポイントのスコア向上
  • 教訓：モデル選びに時間をかけすぎるのではなく、ハーネスの構築に投資すべき
モデルは同じなのにスコアが
大幅に上がる。これは決定的な
エビデンスですね。

⑨ コンテキスト工学とは

  • プロンプト工学からコンテキスト工学への移行。キラワード1回の最適化からセッション全体の最適化へ
  • 発想の転換：「AIを騙して良いコードを書かせる」ではなく「新しいチームメンバーに何を伝えるか」
  • コンテキストの質が出力の質を決定づける。プロンプトの巧みさは二の次
  • AGENTS.md、システムプロンプト、Evalsスイート、スキルライブラリをコードとして扱うべき
「プロンプトの囁き」から「工学」へ。
AGENTS.mdはPRでレビューし、
バージョン管理する。

⑩ 6種類のコンテキストと静的/動的の分類

  • 命令（役割と境界）、知識（ドキュメント、図解）、記憶（セッションログ＋長期状態）
  • 例（参照パターン）、ツール（API・スクリプト）、ガードレール（安全ルール）の6種類
  • 静的コンテキストは常に読み込まれる。信頼性が高いがトークンコストが毎回かかる
  • 動的コンテキストはオンデマンドで読み込まれる。必要な時だけコストを払う効率的な設計
静的が多すぎると信号が薄まり、
少なすぎると重要なルールを忘れる。
この境界線が設計判断。

⑪ Agent Skillsがゲームチェンジャー

  • Agent Skillsは手続き知識のポータブルなパッケージ。タスクに応じてオンデマンドでロードされる
  • 段階的開示（Progressive Disclosure）で、軽量な汎用エージェントが多数の専門能力を持てる
  • 現在使っている1つのスキルに対してのみトークンを支払う。他の数十のスキルはコストゼロ
  • プロジェクト間で再利用可能。一度構築すれば何度でも使える。チーム全体の資産になる
スキルライブラリは一度作って
何度も洗練する。複利のように
成果が蓄積していきます。

⑫ AIがSDLCの各フェーズをどう変えるか

  • 要件・計画：手渡しのドキュメントから対話へ。仕様とプロトタイプを同時に生み出す
  • 設計・アーキテクチャ：最も人間に依存するフェーズ。AIは実装は得意だが設計判断は不得意
  • 実装：数週間が数時間に圧縮。ただしMETR研究ではベテランが時々遅くなることも判明
  • テスト・QA：出力評価＋経路評価の両方が必要。テストはエージェントへの意図伝達手段になる
古いSDLCの加速版ではない。
全く違う形の作業になる。
実装から仕様書きへ重心が移る。

⑬ コードレビュー・保守・メンテナンス

  • AIによるファーストパスレビューがバグ、スタイル、セキュリティを自動チェックし、レビュー負荷を軽減
  • ただし設計と保守性の人間の判断はAIでは代替できない
  • レガシーコードの保守が最大の盲点勝利。著者以外に理解されなかったコードをAIがナビゲート可能に
  • リファクタリングと移行作業がこれまで「リスクが高すぎて手を出せなかった」ものから実行可能に
レガシーコードの保守こそ
AIの最も過小評価された
利点かもしれない。

⑭ ファクトリーモデル — 工場の設計者になれ

  • 開発者の主要な出力はコードではなくコードを生み出すシステムに変化した
  • 仕様とコンテキストが何を作るかを定義し、エージェントが実装に翻訳する
  • 工場長は1つ1つの製品を組み立てない。ラインを設計し、品質管理を担う
  • 成功基準を与えてステップバイステップの指示は与えない。エージェントに反復させるのが正解
自分がコードを書く人ではなく
コードを書く仕組みを作る人。
この意識の変革が全てです。

⑮ 指揮者モードとオーケストレータモード

  • 指揮者モード：リアルタイム・インIDEでの操作。複雑なロジックやデバッグに適している
  • オーケストレータモード：非同期・高抽象度・マルチエージェント。タスクを委譲して結果をレビュー
  • 両モードのスキルは異なる。仕様作成、分解、評価、システム設計が求められる
  • 指揮者モードですべての変更を通すと自分がボトルネックになるリスクに注意
両モードを自在に切り替える。
複雑な作業は指揮者、
明確な作業はオーケストレータ。

⑯ 80%問題 — 残り20%の壁

  • エージェントは約80%の機能を確実に生成できる。しかし残り20%が深い文脈を必要とする
  • 残り20%とは：エッジケース、エラーハンドリング、統合ポイント、微妙な正確性
  • AIのエラーの性質が変わった。構文エラーから概念エラーへ。コードは見た目が正しくテストも通るがロジックが間違っている
  • 成功する開発者はAIが得意なことに任せ、自分の注意力をAIが不得意な部分に予約する
概念エラーは見た目が正しい。
だからこそ恐ろしい。
人間の判断がここで決定的に重要。

⑰ 経済学 — 速度ではなくTCO

  • 重要な指標は「どれだけ速く出荷できるか」ではなく総所有コスト（TCO）
  • バイブコーディングはCapExは低いがOpExが複利で増加：高トークン消費、保守税、セキュリティ修正費用
  • Agentic EngineeringはCapExが高いがOpExが低い。事前投資のあとは機能ごとの限界コストが急減
  • OpExを下げる2つのレバー：コンテキスト工学で初回成功率を上げ、インテリジェントモデルルーティングでコスト適正化
難しいタスクはフロンティアモデル、
決定論的な作業は小型モデル。
この使い分けでコスト劇的下落。

⑱ 3つの永続的原則と出発点

  • 原則1：構造がスケールする、気分はしない。正式なプロセスが生産性を加速させる
  • 原則2：AIは既存のエンジニアリング文化を増幅する。強さも弱さも拡大される
  • 原則3：人間の役割は進化する、縮小しない。仕様、評価、アーキテクチャ判断が新しい職人技
  • 出発点：コンテキスト工学を第一級の実務にし、Evalsでゲートし、ハーネスを共有資産として投資する
「生成はほぼ解決された。
検証・判断・方向付けが
新しい職人技である。」
