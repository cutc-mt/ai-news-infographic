Sakana AI「Fugu」の仕組みを徹底解説

Mythos級の国産LLM？「束ねる」AIが世界を変える
2026年6月26日

① Sakana AIとは: 日本発のフロンティアAI企業

  • 東京拠点 2023年に設立された日本発のAI企業。「Building Frontier AI in Japan」を掲げる
  • 社名は「魚」を意味する日本語に由来。群れの群知性（スワーム・インテリジェンス）で複数のAIが協調して動く世界観を体現
  • Fuguを国際商用AIプロダクトとして位置づけ、日本発でありながら世界市場を狙う戦略
  • 進化的アルゴリズムと深層学習を融合させる独自の研究路線で注目を集めてきた
OpenAI・Anthropic・Googleのような米国一極集中の体制に、日本から「第三の路線」を提示しているのがポイントです！

② Fuguの誕生: 日程とリリースの概要

  • 2026年4月24日 ベータ版を公開。約500名のテスターが参加
  • 2026年6月22日 一般提供（GA）を開始。FuguとFugu Ultraの2種類
  • タイミングが絶妙：Claude Fable 5が輸出規制で停止した直後という空白を突いたリリース
  • ベータ期間中のユーザーから「同一コードベースのレビューでFugu Ultraが20件以上の問題を指摘した」等の声が寄せられた
Fable 5停止 → わずか10日後にFuguがGA開始。このタイミングの良さが話題を呼びました！

③ Fuguとは何か: 「束ねる」AIの概念

  • ユーザーから見れば1つのモデルAPIに見えるが、内部では複数のLLMをエージェントとして呼び分ける設計
  • Claude系・GPT系・Gemini系を含む複数のフロンティアモデルを難易度や用途に応じてチーム編成
  • モデルプールが完全に差し替え可能。特定ベンダーへの依存リスクを低減できる
  • 「マルチエージェント・オーケストレーション」を1つのAPIに収めた「集合知モデル」
魚の群れのように個々のAIが協調して動く ― Sakana AIのブランド哲学そのものです！

④ Conductor: 7Bパラメータの「指揮者」

  • Fuguの中心にあるのは「Conductor（指揮者）」と呼ばれるオーケストレーションモデル
  • わずか7Bパラメータの小型モデルで、強化学習（RL）によってコーディネーション戦略を学習
  • どのモデルに・どの順序で・どんなプロンプトでタスクを投げるかを自動で判断
  • タスクの中間結果を見ながら戦略を動的に変更する能力も持つ
大型モデルを丸ごと訓練するのではなく、既存モデルへの「委任」を学習させた小型モデルが司令塔になる ― 新しいアーキテクチャです！

⑤ TRINITY: Thinker・Worker・Verifierの3役割

  • Thinker タスク全体の方針を立て、問題を構造化する。難問では複数モデルを並列起用
  • Worker Thinkerが定義したサブタスクを実際に解く。コード生成・計算・データ抽出を担当
  • Verifier Workerの出力を確認し、誤りや欠落を指摘。別モデルが担当してバイアスを抑制
  • 「人が設計したワークフロー」ではなく学習したコーディネーターが動的に決めるのが肝
Thinker・Worker・Verifierの役割割り当てをICLR 2026で発表したTRINITYフレームワークがベースです。

⑥ 再帰呼び出し: Fuguは自分自身も呼び出せる

  • Fugu自身がエージェントプールの中に再帰的に含まれるのが最大の特徴
  • Conductorは必要に応じて自分自身を「Worker」として呼び出せる
  • 実行結果を読み取り、失敗していれば別の戦略で再実行するループを持つ
  • 大きなタスクを段階的に分解し、各段階で別のサブチームを動的に編成
「推論時の計算量を増やすことで品質を上げる」という新しい軸 ― recursive test-time scaling！

⑦ モデルプールの構成: どのAIが動いているか

  • GPT-5.5 OpenAIの最新フロンティアモデル
  • Gemini 3.1 Pro Googleの長文脈対応モデル
  • Claude Opus 4.8 Anthropicの最新高性能モデル
  • Fable 5とMythos Previewは輸出規制で公開停止中のためプールに含まれていない
  • 今後はオープンモデルやSakana AI独自のモデルも組み込んでいく方針
Fable 5・Mythosがプールにないにも関わらず、それらと肩を並べるスコアを記録したのが驚きです！

⑧ Fugu vs Fugu Ultra: 2種類のモデル

  • Fugu（標準版） 日常的なコーディング・チャットボット向け。低レイテンシで素早い応答
  • Fugu Ultra 最難問のタスクに対応。データ分析・論文再現・セキュリティ分析向けの深い協調
  • 標準版はプロバイダー・モデルの除外設定が可能（例: OpenAI製を使わない）
  • Fugu Ultraは品質維持のためエージェントプールが固定で、個別モデルの除外はできない
データ主権やコンプライアンス要件に応じてモデルを除外できるのが標準版の強みです！

⑩ 正確な評価: 「全領域最強」ではない

  • 勝った領域 Terminal-Bench・CharXiv・LiveCodeBench・GPQA-D など推論とコーディング系
  • 負けた領域 SWE-Bench Pro（73.7 vs 80.0）・HLE（50.0 vs 53.3）・SciCode
  • 公式の控えめな表現：「Fable 5やMythos Previewと肩を並べる（shoulder-to-shoulder）」
  • 「全領域で最強」ではなく「束ねるだけでフロンティア級と渡り合える領域が広がった」が正しい解釈
「最強」という言葉に飛びつく前に、勝つ領域と負ける領域を冷静に見極めることが大事です！

⑪ 実務パフォーマンス: 方針維持力が光る

  • 50週株式取引パイプラインで平均リターン+19.43%。他モデル単体は+15%未満に留まった
  • コードレビューでFugu Ultraが20件以上の問題を指摘、他モデル単体は3件のみという事例も
  • 「複数日の連続タスクで方針を維持する力」に単一モデルとの差が明確に出る
  • 従来のMoAは1クエリ約11,203トークン消費するが、Conductorは約1,820トークンで6倍効率
ベンチマーク数値では現れない「多段ワークフローでの差」こそがFuguの真価です！

⑭ OpenAI互換API: 既存コードがそのまま動く

  • エンドポイントURLとAPIキーを差し替えるだけで既存アプリからの移行が最小限
  • GitHub Copilotのカスタムエンドポイントに設定可能
  • OpenAI Codex CLIなどのカスタムモデルプロバイダーにも直接設定できる
  • Claude CodeはAnthropic形式前提のため、LLM Gatewayのプロキシが必要
  • LangChainやLlamaIndexの「openai」クライアントをそのままSakana Fuguに切り替え可能
「複数モデル契約を1つに統合できる」というのが公式の最大の訴求ポイントです！

⑮ 輸出規制への「保険」: モデルプールが差し替え可能

  • 2026年6月12日に米国商務省の輸出管理命令でFable 5・Mythos Previewがグローバルで停止
  • Fuguはエージェントプールを柔軟に差し替えられる設計。特定ベンダーのモデルを除外可能
  • データ主権・プライバシー要件に応じたコンプライアンス制御が組み込める
  • ただし「輸出規制リスクゼロ」ではなく、分散によるリスク低減として捉えるべき
3ヶ月かけてFable 5のフローを整備した直後に停止した企業は多数。「いつモデルが使えなくなるか分からない」時代に！

⑰ 使いどころ: 効く場面と向かない場面

  • 効く 大規模コードレビュー・脆弱性検出。複数モデルの観点で総合的に問題点を抽出
  • 効く 論文・特許のランドスケープリサーチ。数十本の文献を横断して比較・要約
  • 効く 論文の再現・実験自動化。環境構築から実行まで自律的に完遂
  • 向かない 応答速度が最優先のチャットボット。Conductorのオーバーヘッドが発生
  • 向かない EU/EEAでのデータ処理。GDPR対応が未完了のため現時点では利用不可
  • 向かない コンテキストが小さい単発タスク。マルチエージェントの恩恵が薄くコスト効率が悪化
「判断の複雑さが高くてアウトプットの重要度が低い」タスクから試すのがおすすめ！

⑱ 失敗パターンと今後の展望

  • 失敗① 「束ねれば最強」と思い込んで精度検証を省く。自社タスクで20〜50件比較評価が必要
  • 失敗② コストを月額料金だけで判断。内部で複数モデルを呼ぶためトークン消費が増える
  • 失敗③ データ経路を確認せず機密情報を投げる。裏で複数ベンダーが動いていることを意識すべき
  • 失敗④ レイテンシを無視してリアルタイム用途に適用。非同期・バッチ処理から始める
  • 今後の展望: AIを選ぶ意思決定の層と使う業務処理の層を分離する「束ねる時代」が到来
  • Sakana AIは米中AI競争の外で戦う第三の路線として注目を集めている
「三国時代」から「束ねる時代」へ。自社モデルを最強にする競争ではなく、既存の最強モデルをうまく束ねる競争が始まりました！
