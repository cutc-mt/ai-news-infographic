AIにどこまで仕事を任せる？

「Can Trust Own」の3つの評価軸で仕事設計を徹底解説！
📅 2026年6月27日 | 🎬 いけともch

①みんなのモヤモヤ: AIにどこまで任せていい？

  • 会議やクライアントとの打ち合わせで最もよく聞かれる質問
  • 企業や業務内容でケースバイケースで変わる
  • しかし共通の不安の根底には3つの懸念がある
  • 結論は「決めて育てる」ことが重要
「AIに仕事を任せていいんですか？」
この質問、みんな抱えてる共通の悩みなんですね！

②3つの不安: ミス・情報漏洩・責任

  • 取り返しのつかないミスをAIがやってしまったら？
  • 情報漏洩: 本来見てはいけないデータに触れてしまう
  • 問題が起きたら誰の責任になるのか決まっていない
  • 会社・担当者・AI提供会社の責任の所在が混在
ミスしたら誰が責任取るの？
この曖昧さが
みんなを不安にしてる！

③独自フレームワーク: Can Trust Own

  • 単なるヒューマン・イン・ザ・ループではアバウトすぎる
  • Can: 技術的にその仕事ができるか
  • Trust: 結果が安定していて信用できるか
  • Own: 問題が起きた時の責任の分担はどうするか
「人間が確認しましょう」だけじゃ参考にならない
具体的な3つの軸で考えよう！

④3つの軸は「決めの問題」

  • Can: できるかできないかはある程度決定可能
  • Trust: どこまで安定性を求めるかは組織で決める
  • Own: 組織をどう設計するかという決めの問題
  • 考えれば解ける問題であり、魔法ではない
難しく考える必要はない！
組織で方針を決めるだけ！
後は回して検証するだけ

⑤Can: 技術的にできるか？

  • Claude Codeやエージェントで外部ツールを操作可能
  • ブラウザ操作やパソコンの操作を代替できる
  • デジタル上の仕事は大部分が技術的に可能
  • 詳細はクロードコード特集を参照してほしい
結論: デジタル仕事は大抵できる
問題は「信用できるか」と
「責任をどうするか」の方！

⑥Trust: 新人アルバイトの例え

  • 入社初日の様子で「ダメだ」と決めるのは早すぎる
  • トレーニング・やり方の伝達・チェック・フィードバックが必要
  • 3ヶ月後の成長を見て初めて判断できる
  • AIも同じ: 育ててから判断すべき
初日に「この人使えない」と
決めちゃダメ！教育もしてないのに
判断するのはおかしいよね

⑦信頼は時間軸で育てるもの

  • 信頼は始めた瞬間には決まっていない
  • 業務を任せて帰ってくるクオリティで判断する
  • 今の技術だけで決めるのはもったいない
  • 育て上げた後にどれぐらいできるかが論点
今の瞬間でAIを判断するのは
もったいない！
サイクルを回して育てよう！

⑧エンジニアリングの4世代の進化

  • プロンプトEngineering: オーナーがシェフに注文
  • コンテキストEngineering: レシピ・材料・環境を整える
  • ハーネスEngineering: 安全装置を仕込む
  • ループEngineering: シェフ自身で品質を管理
料理のオーナーとシェフの
関係性で考えていくと
わかりやすいですね！

⑨ループエンジニアリング: 最新概念

  • Peter Steinberger氏が提唱（OpenClau作成者）
  • AnthropicのClaude Code責任者も推奨
  • Googleも「これってループエンジニアリングですね」と言及
  • 2026年5〜6月に急激に概念が盛り上がった
ビッグテック全社が
一斉に注目している
最新の考え方！

⑩ループの具体例: YouTube運用

  • ニュースの選定 → 深掘り → 考える → 資料化
  • 資料を話す → サムネイル作成 → 概要欄作成
  • 結果を分析してSlackに通知 → 次の企画へ
  • 全自動ではなく人間が選択しながらループを回す
完全自動じゃなくていい！
人間の判断を挟みながら
サイクルを回すのが大事

⑪ループの具体例: X運用とバックオフィス

  • 記事タイプ投稿を作成 → 投稿 → リポストで拡散
  • 人気の記事を数ヶ月後に再利用して成果を積む
  • バックオフィス: クレカ明細の経費登録
  • 最初は項目間違いや金額ミスがあっても指摘→修正のループで向上
3月から始めてまだ3ヶ月目！
新人の教育と同じ。
1年2年かけて育てる感覚で！

⑫フロンティアエコシステム: Microsoft

  • Microsoftの先田（Sana）氏が提唱・数千万ビュー
  • AIモデルはビッグテックが作るので差別化にはならない
  • 学習ループを企業独自に回すことが競争優位性
  • プロンプト・コンテキスト・ハーネスは副産物に過ぎない
タスクは渡せるが、
学習は渡せない
自分たちのループを回すのが
本質的な価値！

⑬Own: 責任設計の歴史

  • 産業革命時代: 機械の責任は現場の作業者にあった
  • テイラー・フォード時代: 雇用主の責任へ移行
  • ファクトリーオートメーション: 3者モデルに分化
  • AI時代: モデル作成会社・システム会社・企業の3層
機械の責任問題は
200年前から議論されてる！
AIも同じ道を辿っている

⑭日本のAI責任ガイドライン

  • 2026年4月に国が民事責任の手引を公表
  • 補助型: AIを支援ツールとして使う → 利用者の責任
  • 代替型: AIが人の判断を代替する → 大きな論点
  • 提供者がリスクを説明・軽減すれば、企業側に責任が移る
日本もようやくガイドライン出た！
「AIが全部やってくれる」
代替型の責任が今後の焦点

⑮3層の責任分担モデル

  • 経営者: 期待値とミス許容を設定。AI導入を後押し
  • AIエージェント・マネージャー: 大きなループの責任を担当
  • 現場: 自分の業務範囲でチェックと確認の責任
  • マネージャーはHBR×Salesforceの提唱概念
AIエージェント前提の組織では
マネージャーが新職種に！
ロボットの安全管理責任者みたいな

⑯現実的な期待値: ミスゼロは無理

  • ミスゼロを求めると導入できない
  • 人間の正解率をベンチマークに設定する
  • 例: カスタマーセンターで人間100件中5件問題 → AIなら2件なら改善！
  • 人間より上回るレベルであればOKとする設計
人間だってミスしてる！
人間のミス率より下回れば
十分に導入の価値がある

⑰AIモデルの進化も追いかける

  • AIモデルはどんどん進化している
  • 今できないことが次のモデルではできるかも
  • 育てたループの流れを新しいモデルでも試す
  • サイクルを回すことで副産物（プロンプト等）も蓄積
今のモデルでできなくても
次のモデルでできるかも！
ループを回し続けることが鍵

⑱まとめ: 決めて育てるが全て

  • Can: デジタル仕事は大部分が技術的に可能
  • Trust: 育てるサイクルを回してから判断する
  • Own: 経営・監督・現場の3層で責任分担を設計
  • AIに任せる人ではなくAIを育てる人に価値がある
今日から1つの業務で
学習ループを回してみよう！
作って終わりじゃなくて
育てていくのが本物の価値！
