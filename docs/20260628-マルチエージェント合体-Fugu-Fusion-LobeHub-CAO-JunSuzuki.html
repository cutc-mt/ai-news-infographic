Sakana Fugu の概要：マルチエージェントを1つのモデルとして

①Fugu（河豚）とは何か

Sakana AIが開発したマルチエージェント・オーケストレーションシステム。単一のAPIエンドポイントから複数の最先端モデルを動的に組み合わせて協調させる仕組みです。

  • ユーザーは1つのモデルとして扱える
  • 内部では複数モデルがルーティングされる
  • GPT-5.5、Opus 4.8、Gemini 3.1 Pro等を統合
「河豚」のように小さいが、中身は複数の強力なモデルを統率するオーケストレーターです！

②新しいスケーリングの軸

Fuguはゼロから学習した独自の重みを持たない。あくまで「指揮者」として機能し、既存の強力なモデル群を組み合わせて Collective Intelligence（集合知能）を実現します。

  • 従来：巨大な単一モデルに投資
  • Fugu：複数モデルの協調で性能を追求
  • サードパーティ製モデルも利用可能
「もっと大きく」ではなく「もっと賢く組み合わせる」という新しいアプローチ！

③OpenAI互換APIで利用可能

FuguはOpenAI互換のAPIを提供。既存のツールやフレームワークにそのまま組み込めます。OpenRouter、Vercel AI Gateway、Merge API等でも利用可能です。

  • OpenRouter Vercel Merge API
  • ローカル実行は不可（ホスト型サービス）
  • どの基盤モデルが応答したかは非公開

TRINITY：進化型LLMコーディネーター

④TRINITYの基本構造

TRINITYは約0.6Bの小型言語モデル（バックボーン）に、約1万パラメータのヘッドを組み合わせた軽量コーディネーターです。バックボーンはクエリを読み取り、ヘッドが各ワーカーのスコアを出力します。

  • バックボーン：~0.6Bパラメータ
  • ヘッド：約10Kパラメータ（学習対象）
  • テキストを生成せず「選択」だけを出力
通常のLLMはトークンを一つずつ生成するが、TRINITYは「誰に任せるか」だけを瞬時に決める設計！

⑤Thinker・Worker・Verifierの3役割

TRINITYの研究では、モデルに3つの役割を割り当てて複数ターンにわたって統括します。コーディング・数学・推論・知識タスクに応じて適応的に振り分けます。

  • Thinker：問題を分析・計画する
  • Worker：実際のタスクを実行する
  • Verifier：結果を検証する
チーム開発と同じ：思考役・実行役・検証役で品質を担保する仕組み！

⑥2段階の学習プロセス

TRINITYの学習は監督付き微調整（SFT）と進化戦略（sep-CMA-ES）の2段階で行われます。勾配を使わず、ブラックボックス最適化でパラメータを改善します。

  • Stage 1: SFT — 各タスクで誰が得意かを学習
  • Stage 2: 進化戦略 — セッション全体の成功を最大化
  • 報酬は単純な0/1（成功/失敗）
勾配ベースのREINFORCEではノイズに埋もれて失敗。進化戦略ならパラメータ全体を一括で最適化できる！

⑦ドメインごとのルーティング

TRINITYの優秀な点はドメインに応じてルーティング分布が変化することです。数学ならGPT、コーディングならOpusなど、各分野の最強モデルに自動で振り分けます。

  • 数学的推論 → GPT-5.5（60%）
  • ターミナルコーディング → GPT-5.5
  • 競技プログラミング → Gemini 3.1 Pro
  • セキュリティ → Claude Opus 4.8
「誰が得意か」をデータから学習。手作りのif-elseルールではなく、賢い自動振り分け！

Conductor：オーケストラの指揮者 & Fugu Ultra

⑧Conductorの仕組み

Fugu Ultraの心臓部は約7BパラメータのConductor。TRINITYと異なり、フルの言語モデルとしてテキストでワークフロー全体を記述し、各エージェントへの割り当てと通信トポロジーを決定します。

  • サブタスクのリストを出力
  • 各タスクにワーカーを割り当て
  • アクセスリストで通信経路を定義
  • 再帰的に自分を呼び出し可能

⑨通信トポロジーの柔軟性

Conductorが生成するアクセスリストによって、チェーン、多数決、ディベートツリー、再帰呼び出しなど多様な協調パターンを実現します。これらは手書きではなく学習から自然に現れます。

  • チェーン：順次処理
  • 多数決（Best-of-N）：独立に実行して投票
  • ディベート：意見をぶつけ合う
  • 再帰：自分自身を再度呼び出し
「triviaにはディベートツリーを使え」なんて手作業で書いていない。報酬最大化から自然に浮かび上がる戦略！

⑩オープンソース限定でも強い

驚くべきことに、Conductorはオープンソースモデルのみで構成した場合でも、Claude Sonnet 4を約10%上回る性能を発揮しました。これは協調の力が、単に強いモデルに乗るだけではない証拠です。

  • オープンソース限定のConductor構成
  • Claude Sonnet 4を約10%上回る
  • 「協調」そのものが価値を生む
コストやコンプライアンスで制約があっても、賢い組み合わせでフロンティアモデルを超えられる！

Model Fusion & LobeHub CAO：合体の別のアプローチ

⑮Model Fusion（モデル融合）の世界

Model Merging / Fusionは、別々に学習された複数モデルの重みを数学的に統合する技術。Fuguの「APIレベルの協調」と異なり、重みレベルで1つのモデルに合体させます。

  • TIES、DARE、SLERP等の手法
  • 学習不要で統合が可能
  • 2026年ACM Computing Surveysに包括的サーベイ論文
Fuguは「APIでつなぐ」アプローチ。Fusionは「重みを混ぜる」アプローチ。両者が並行して進化中！

⑯LobeHub CAO：チーフ・エージェント・オペレーター

LobeHub CAOはオープンソースのマルチエージェントプラットフォーム。20万以上のスキルのマーケットプレイスからエージェントを組み合わせ、チームとしてタスクを遂行させます。

  • 20万+スキル のマーケットプレイス
  • Agent Groupsでチーム編成
  • Claude Code、Codex、OpenClaw等を統合
  • 逐次・並列・反復・ディベートの協調モード
Fuguが「プロプライエタリな指揮者」なら、LobeHub CAOは「オープンでユーザー主導のチーム管理」！

⑰LobeHubのアーキテクチャ

LobeHubはAgent RuntimeとModel Runtimeの2つのエンジンを搭載。Plan-Execute LoopでマルチステップAI動作を管理し、30以上のAIプロバイダーのAPI差異を吸収します。

  • Plan-Execute Loop：計画→実行→結果処理のループ
  • コンテキスト圧縮で長文脈に対応
  • Human-in-the-Loopのセキュリティチェック
  • macOSアプリ＆Web UIを提供
Fuguは非公開のルーティングだが、LobeHubはユーザーがモデル・プロバイダーを自由に選べる！

⑱3つのアプローチを比較

今回紹介した3つの手法はそれぞれ異なる戦略でマルチモデル合体を実現しています。用途と制約に応じて最適な選択が変わります。

  • Sakana Fugu：学習済みオーケストレーター、API完結
  • Model Fusion：重みを統合、1モデルに合体
  • LobeHub CAO：オープンソース、ユーザー主導のチーム編成
