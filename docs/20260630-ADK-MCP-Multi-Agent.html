ADK & MCPでマルチエージェントを構築

GoogleのAgent Development KitとModel Context Protocolで複数AIが協調する仕組みを作る
2026年6月30日

① マルチエージェントシステムとは

  • 複数のAIエージェントが協調して複雑なタスクを完遂する仕組み。1つのモデルに全部任せるのではなく、役割を分担させるアプローチ
  • 例えば「リサーチ担当」「コーディング担当」「レビュー担当」のように専門性ごとに分割して並行処理させる
  • 単一エージェントだと時間がかかる作業も、並行実行で大幅に高速化できる
  • タスクを細かく分割して各エージェントに割り当て、結果を統合するというMapReduce的な考え方
人間のチームワークみたいなもの。
各自の得意分野で協力すれば、
1人でやるより速くて正確！

② ADK（Agent Development Kit）とは

  • Googleが提供するエージェント開発キット。GeminiモデルをベースにしたAIエージェントを簡単に構築できるフレームワーク
  • マルチエージェントの調整から、ツール統合、状態管理まで一貫してサポートする包括的なSDK
  • PythonとTypeScriptの両方に対応。既存のプロジェクトにも組み込みやすい設計
  • MCPサーバーへの接続やツールの定義が宣言的に書けるため、ボイラープレートが少ない
ADKを使えばエージェントの
定義から起動まで数行で書ける。
Google公式のエージェントフレームワークです。

③ MCP（Model Context Protocol）とは

  • エージェント間やエージェントと外部ツール間の通信を統一するオープンプロトコル。いわば「AIのためのUSB端子」
  • MCPサーバーとしてツールを公開すれば、どのエージェントからでも統一したインターフェースで利用できる
  • ローカル・リモートを問わず接続可能。stdioとSSEの2つのトランスポート方式をサポート
  • 各ツールの認証やアクセス制御もプロトコルレベルで管理できるため、セキュリティも安心
MCPはエージェント界の共通規格。
対応していればどんなエージェントでも
同じツールを使えます。

④ ADK×MCPの強力な組み合わせ

  • ADKがエージェントの枠組みを提供し、MCPが外部ツールへの接続を統一管理するという役割分担
  • MCPサーバーさえ用意すれば、ADKのエージェントは設定だけでツールを使えるようになる
  • ツールごとに独自の実装を書く必要がなくなり、プラグインのように組み合わせ可能に
  • 「エージェントを定義する」「ツールを繋ぐ」「動かす」という3ステップだけでシステム構築が完了
ADKとMCPは最高の相棒。
Googleが推し進めるエージェント
エコシステムの中核です。

⑤ エージェントの定義と役割分担

  • ADKではAgentクラスを使ってエージェントを定義。名前、役割説明、利用可能なツールを指定するだけ
  • 各エージェントに明確な役割と人格を与えることで、タスクの精度が向上する
  • 親エージェントが子エージェントを管理する階層構造が基本パターン
  • 「ルーター」エージェントが入力を分類し、適切な専門エージェントに振り分ける設計が推奨される
agent = Agent(
  name="research_agent",
  model="gemini-2.5-pro",
  instruction="調査を行い、事実を整理して報告",
  tools=[search_tool]
)
エージェントの名前と指示文を
うまく書くのが成功の鍵。
役割が曖昧だと動きがブレます。

⑥ MCPを使ったツール統合

  • ADKはMcpToolを使ってMCPサーバーをエージェントのツールとして直接接続できる
  • ファイルシステム、データベース、API、検索エンジンなど、MCPサーバーが存在するものなら何でも統合可能
  • ツールの追加・削除が設定変更だけで完了。エージェント本体のコードをいじる必要がない
  • 複数のMCPサーバーを同時に接続でき、エージェントが必要に応じて適切なツールを自律的に選択する
mcp_tool = McpTool(
  server="filesystem",
  params={"path": "/data"}
)
agent = Agent(tools=[mcp_tool])
MCPのエコシステムは急速に拡大中。
コミュニティ製サーバーも充実しています。

⑦ エージェント間のコミュニケーション

  • ADKのLlmAgentは他のエージェントをツールとして呼び出せる。自然的な連携が実現する
  • 親エージェントがタスクを子エージェントに委任（delegation）するパターンが基本
  • 結果の受け渡しは自動で処理される。エージェント間のコンテキスト共有もシームレス
  • ループ構造も設定可能。「結果をレビューして改善する」ような反復処理も自然に記述できる
エージェント間の通信は
ADKが裏側で全部やってくれます。
開発者は役割の設計に集中すればOK。

⑧ 実装デモ — プロジェクトの全体構成

  • 動画のデモでは旅行計画エージェントを構築。目的地のリサーチから日程作成までを自動化する
  • ルーターエージェントがユーザーの要求を解釈し、適切な専門エージェントにタスクを振り分ける
  • 「フライト検索」「ホテル検索」「観光スポット提案」など、役割ごとにエージェントを分割
  • 最終的に全エージェントの結果を1つの旅行プランとして統合して出力する流れをデモ
旅行プランは分かりやすい例。
実務ではコードレビューや
データ分析パイプラインにも応用できます。

⑨ デリゲーション — 任せて見守る

  • 親エージェントがタスク全体を管理し、各子エージェントに具体的な作業を委任する設計パターン
  • 親エージェントは子の出力結果を評価し、品質が不十分ならやり直しを指示できる
  • 委任の深さを制御可能。浅い委任（1階層）から深い委任（多階層）まで柔軟に設定
  • 無限ループを防ぐための最大反復回数の設定も推奨される
マネージャーと専門家の
関係に似ています。
マネージャーが全体を見て専門家に振る。

⑩ ストリーミング — リアルタイムで結果を見る

  • ADKはストリーミング対応。エージェントの思考プロセスやツール呼び出しをリアルタイムで確認できる
  • RunContextを使って、各エージェントのステータスやツール実行結果をストリームとして出力
  • チャットUIなどに組み込めば、ユーザーはエージェントが何をしているかをリアルタイムに把握できる
  • デバッグ時もストリーミングが役立つ。どのエージェントがどこでつまずいているかがすぐに分かる
黒い箱の中身が見えるのは
大きな安心感。
透明性がエージェント運用の要です。

⑪ エラーハンドリングとフォールバック

  • ツール呼び出しが失敗した場合、ADKは自動的にリトライや代替手段を検討する仕組みを持っている
  • エージェントが誤った出力をした場合、親エージェントが修正指示を出して再実行させる設計が可能
  • 各エージェントにガードレールを設定して、許可されていない操作や危険な出力をブロックできる
  • エラーログや実行トレースを記録する機能も内蔵。問題発生時の原因追跡が容易
エージェントを本番環境で
動かすならエラーハンドリングは必須。
ADKはそれを最初から考えてます。

⑫ スケーラビリティと拡張性

  • 新しいエージェントやツールを追加するだけでシステムを拡張できるモジュラーな設計
  • Google Cloud上で動かせば、負荷に応じて自動スケーリングも利用可能
  • エージェントの設定ファイルを別プロジェクトに再利用しやすい構造
  • ローカル開発から本番環境への移行もスムーズ。設定値を切り替えるだけで対応できる
最初は2〜3個のエージェントで
始めて、必要に応じて増やしていく。
段階的にスケールできるのが理想です。

⑬ ユースケース — タスク分割と並行処理

  • 複雑なタスクを小さな単位に分割して、複数のエージェントで同時に実行する
  • 例えば「レポート作成」を「データ収集」「分析」「文章執筆」「フォーマット」に分けて並行処理
  • 直列実行なら10分かかる作業が、並行処理で3〜4分に短縮されることも
  • 最後にレビューエージェントが品質を確認し、問題があれば修正ループに入る
並行処理の効果はエージェントの数
とタスクの独立性に比例。
依存関係の少ないタスクほど効果的。

⑭ ユースケース — コード開発パイプライン

  • 「要件定義」→「コード生成」→「テスト作成」→「コードレビュー」→「デプロイ」というソフトウェア開発の全工程をエージェント化
  • テストエージェントがテストを書き、レビューエージェントがコードの品質をチェックする自動化が可能
  • バグを見つけたら自動で修正エージェントに振り、修正→再テスト→再レビューのループを回す
  • MCP経由でGit操作やCI/CDツールと連携すれば、完全自動化も視野に入る

⑰ 学びとベストプラクティス

  • エージェントの役割を明確に定義する。曖昧な役割分担はパフォーマンス低下の原因になる
  • MCPを活用してツール統合を標準化する。独自実装は最小限に
  • 小さく始めて段階的に拡張。最初は2〜3エージェントから始めて十分
  • ストリーミングとログを活用して、エージェントの動作を常に可視化する
  • エラーハンドリングとガードレールを最初から設計に組み込む
