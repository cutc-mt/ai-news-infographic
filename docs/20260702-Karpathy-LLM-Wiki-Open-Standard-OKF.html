① Open Knowledge Format (OKF) v0.1

  • 2026年6月 GoogleがOKF v0.1を公開。Karpathyの「LLM Wiki」アイデアをオープン標準に形式化
  • ベンダー中立・エージェントフレンドリー・人間可読・組織間ポータブルな知識フォーマット
  • Apache 2.0ライセンス。誰でも自由に使える本格的なオープン標準
  • 「パターンは常に良かった。欠けていたのは合意」— Googleの表現が全てを物語る
YAMLフロントマター付きMarkdownを

② KarpathyのLLM Wikiとは

  • 元々はKarpathyが公開したgist。AIエージェントに共有のMarkdownライブラリを与えるアイデア
  • コンセプト: 時間とともに有用性が増す知識ベースをエージェントに持たせる
  • 「LLMは退屈しない、相互参照の更新を忘れない、1パスで15のファイルに触れる」
  • エージェントが面倒な部分（相互参照更新・インデックスメンテナンス）を担当。人間がコンテンツをキュレーション
  • コードのように知識を管理 — git repoでホストできるのが肝
「退屈しない」が重要！
人間はメンテをサボるけど
AIは24時間365日忠実に更新してくれる。

③ パターンの広まり — みんなが似たことを始めた

  • Obsidian vaults × コーディングエージェントの組み合わせが広く普及
  • AGENTS.md / CLAUDE.mdの規約ファイル群がリポジトリに標準装備に
  • index.md / log.mdアーティファクトがリポジトリ内で自律的に生成されるようになった
  • データチーム内のメタデータasコードリポジトリも同様のパターン
  • 問題: すべてが独自仕様。互いに協調するよう設計されていなかった
同じ考えが至る所で同時に
芽吹いていた。でも共通言語が
なかったからバラバラだった。

④ OKFの仕様 — 1ページに収まる最小設計

  • プレーンMarkdown: どんなエディタでも読める、GitHubでレンダリング可能、検索ツールでインデックス可能
  • ただのファイル: tarballで配布可能、git repoでホスト可能、任意のファイルシステムでマウント可能
  • YAMLフロントマター: type, title, description, resource, tags, timestampの構造化フィールド
  • v0.1仕様は1ページに収まる — 最小限の設計が美学
---
type: BigQuery Table
title: Customer Orders
description: 1行=1完了注文
resource: https://console.cloud.google.com/...
tags: [sales, orders, revenue]
timestamp: 2026-06-15T10:30:00Z
---

# Customer Orders
YAMLフロントマターは
Obsidian/Hugoユーザーには
お馴染みのフォーマット。

⑤ コンセプトドキュメントの例

  • type: BigQuery Table — データセットの種類を宣言
  • title: Customer Orders — 人間が読める名前
  • description: 1行=1完了注文 — 簡潔な説明
  • resource: BigQueryコンソールURL — 実データへのリンク
  • tags: [sales, orders, revenue] — 検索・フィルタ用
  • timestamp: ISO 8601形式 — 最終更新日時
ファイルパスがコンセプトのアイデンティティ。
Markdownリンクでグラフ化できる！

⑥ OKFが解決する問題 — 知識の断片化

  • 組織の内部知識が断片化: メタデータカタログ・Wiki・コードコメント・シニアエンジニアの頭
  • AIエージェントが回答するため、散在する情報を組み立てる必要がある
  • エージェントビルダーが毎回同じコンテキスト統合問題をゼロから解決
  • OKFがこれらの表面間の相互運用フォーマットになる
「〇〇さんに聞いてください」
がデータカタログのエントリに
なっている組織、多いよね…
OKFで終わり。

⑦ 設計原則① 最小限の意見

唯一必須フィールドは「type」のみ

それ以外はプロデューサーに委ねる。OKFは「どう書くべきか」を決めない。

  • typeだけが必須。title, description, tags等は全部オプショナル
  • 「最小限の意見」＝ フォーマットが使い方を縛らない
  • BigQueryテーブルにも、社内Wikiページにも、APIエンドポイントにも使える
  • 小さく始めて大きく育てる — OKFの哲学そのもの
1ページに収まる仕様書。
複雑さは敵という姿勢が
ここにも表れている。

⑧ 設計原則② 生産者/消費者の独立

誰が書いて誰が読むかを固定しない

生産者と消費者が独立しているからこそ、多様な組み合わせが可能。

  • 人間手書き → AI消費: 人が書いてエージェントが読む
  • メタデータエクスポート → ビジュアライザー: システム出力を人間が確認
  • LLM合成 → 別LLMクエリ: AIが書いてAIが読む
  • 3つのパターンがすべてOKF上で成立する
AI→AIのルートが面白い。
エージェント間で知識バンドルを
交換できる世界が見える。

⑨ 設計原則③ フォーマットでなくプラットフォーム

特定のベンダーに縛られない

クラウド、DB、モデルプロバイダー問わず使えるからこそのApache 2.0。

  • 特定のクラウド（AWS/GCP/Azure）に縛られない
  • 特定のDB（BigQuery/Postgres/MongoDB）に縛られない
  • 特定のモデルプロバイダー（OpenAI/Anthropic/Google）に縛られない
  • 組織変更・ツール変更・ベンダー変更を生き延びる知識層を実現
「知識がツール変更を
生き延びる」って素晴らしい。
これができなかったから
知識はいつも消えてた。

⑩ リファレンス実装: Enrichment Agent

  • Enrichment Agent: BigQueryデータセットを巡回 → 全テーブルのOKFドキュメントを自動作成
  • エージェントが自動的にテーブルのスキーマ・説明・タグを抽出してYAMLフロントマターを生成
  • スケジュール実行で常に最新の状態を維持。人の手でメンテ不要
  • 「エージェントが面倒な部分を担当」— Karpathyのビジョンをそのまま実装
