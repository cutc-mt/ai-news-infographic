KarpathyのLLM Wikiがオープン標準に

Open Knowledge Format (OKF) — AIエージェントが知識を共有する共通言語が誕生
2026年7月2日

① Open Knowledge Format (OKF) — 概要

  • 2026年6月 GoogleがOKF v0.1を公開。オープン標準としての知識フォーマット
  • Andrej Karpathyの「LLM Wiki」アイデアをオープン標準として形式化
  • ベンダー中立・エージェントフレンドリー・人間可読・組織間ポータブル
  • Apache 2.0ライセンス — 誰でも自由に利用・拡張可能
  • v0.1仕様は1ページに収まる最小限の設計。過剰設計を避けたアプローチ
KarpathyのアイデアがGoogleによって
オープン標準として具現化！
パターンに合意ができた瞬間。

② KarpathyのLLM Wikiとは

  • 元々はKarpathyが公開したgist（断片的なメモ集）
  • コンセプト: AIエージェントに共有のMarkdownライブラリを与え、時間とともに有用性が増すようにする
  • 「LLMは退屈しない、相互参照の更新を忘れない、1パスで15のファイルに触れる」
  • エージェントが面倒な部分を担当：相互参照更新・インデックスメンテナンス
  • 人間がコンテンツをキュレーションし、コードのように管理
人間が内容を書き、AIが保守する —
最強の分業がここから始まった。

③ パターンの広まり — 誰もが似たことを始めた

  • Obsidian vaults × コーディングエージェント — Markdown知識ベースをAIが活用
  • AGENTS.md / CLAUDE.mdの規約ファイル群 — エージェントにコンテキストを渡す標準的アプローチ
  • index.md / log.mdアーティファクトがリポジトリ内で自律的に生成
  • データチーム内のメタデータasコードリポジトリの普及
  • 問題: すべてが独自仕様。互いに協調するよう設計されていなかった
誰もが同じアイデアに到達したのに、
言葉がバラバラだった。
OKFがその共通言語になります。

④ OKFの仕様 — シンプルさが武器

  • プレーンMarkdown: どんなエディタでも読める、GitHubでレンダリング可能、検索ツールでインデックス可能
  • ただのファイル: tarballで配布可能、git repoでホスト可能、任意のファイルシステムでマウント可能
  • YAMLフロントマター: type, title, description, resource, tags, timestampの構造化フィールド
  • v0.1仕様は1ページに収まる — 最小限の設計で最大の効果を狙う
---
type: BigQuery Table
title: Customer Orders
description: 1行=1完了注文
resource: https://console.cloud.google.com/...
tags: [sales, orders, revenue]
timestamp: 2026-06-15T10:30:00Z
---

# Customer Orders
Markdown+YAMLフロントマターを
書いているなら90%既にOKF対応！

⑤ コンセプトドキュメントの例

  • type: BigQuery Table — コンセプトの種類を識別
  • title: Customer Orders — 人間可読の名前
  • description: 1行=1完了注文 — 簡潔な説明
  • resource: BigQueryコンソールURL — 実際のリソースへのリンク
  • tags: [sales, orders, revenue] — 検索・フィルタリング用
  • timestamp: ISO 8601形式 — タイムスタンプ
  • ファイルパスがコンセプトのアイデンティティ。Markdownリンクでグラフ化
ファイルパス＝ID、リンク＝関係。
知識グラフが自然に構築されます。

⑥ OKFが解決する問題 — 断片化された組織知識

  • 組織の内部知識が断片化: メタデータカタログ・Wiki・コードコメント・シニアエンジニアの頭
  • AIエージェントが「週間アクティブユーザーをどう計算するか？」に答えるため、散在する情報を組み立てる必要がある
  • エージェントビルダーが毎回同じコンテキスト統合問題をゼロから解決
  • OKFがこれらの表面間の相互運用フォーマットになる
「情報はあるのに見つからない」
この最大の課題をOKFが解決。
知識が蓄積するのではなく断片化するのを防ぐ。

⑦ 3つの設計原則

1. 最小限の意見

唯一必須フィールドは「type」のみ。それ以外はプロデューサーに委ねる。過剰な制約を避ける。

2. 生産者/消費者の独立

人間手書き→AI消費、メタデータエクスポート→ビジュアライザー、LLM合成→別LLMクエリ

3. フォーマットでなくプラットフォーム

特定のクラウド・DB・モデルプロバイダー・エージェントフレームワークに縛られない

最小限のルールで
最大の自由を保証。
ベンダーロックインなし！

⑧ Googleのリファレンス実装

  • Enrichment Agent: BigQueryデータセットを巡回→全テーブルのOKFドキュメントを自動作成→文献から引用・スキーマ・結合パスを強化
  • Static HTML Visualizer: OKFバンドルを対話的グラフビューに変換。バックエンド不要・インストール不要
  • Sample Bundles: GA4 eコマース・Stack Overflow・Bitcoinデータセット
  • すべてApache 2.0でオープンソース。コードを読んで学べる
「仕様だけ出して終わり」じゃない。
Googleが実装まで用意して
すぐに試せる状態に。

⑨ Enrichment Agent — AIが知識を自律構築

  • BigQueryデータセットを巡回スキャンし、全テーブルのメタデータを自動検出
  • 各テーブルに対してOKF形式のコンセプトドキュメントを自動生成
  • 関連ドキュメントからの引用・スキーマ情報・結合パスを自動で強化
  • データが更新されればドキュメントも自動更新。エージェントがメンテナンスを担当
  • Karpathyの「LLMは退屈しない」をまさに体現する実装
AIが自分で知識を書いて
自分で保守する。
人間はキュレーションだけ！

⑩ Static HTML Visualizer — 知識グラフを可視化

  • OKFバンドル（Markdownファイルの集まり）を対話的グラフビューに変換
  • バックエンド不要・インストール不要。HTMLファイルを開くだけで動作
  • Markdownリンクをたどってコンセプト間の関係性をグラフで可視化
  • Obsidianのグラフビューと同じ体験を、OKFバンドルに対して提供
  • チーム間で知識バンドルを共有→視覚的に確認→即座に理解可能
Obsidian好きには馴染み深い
グラフビュー。でもOKFネイティブ。

⑯ OKFのYAMLフロントマター構造

  • type（必須）: コンセプトの種類。唯一の必須フィールド
  • title: 人間可読のタイトル。任意だが推奨
  • description: 簡潔な説明。エージェントがコンテキストを理解
  • resource: 実際のリソースへのURL。外部システムとの接続点
  • tags: 検索・分類用のキーワード配列
  • timestamp: ISO 8601形式。作成日・更新日の追跡
typeだけ必須。
残りは自由に拡張できる
最小限の合意が最大の効果を。

⑰ 知識グラフ化 — リンクで繋がる知識

  • ファイルパス＝コンセプトのアイデンティティ。ファイルシステムそのものが知識の構造
  • Markdownの[[リンク]]でコンセプト間の関係性を定義。Obsidianのグラフビューと同じアプローチ
  • Static HTML Visualizerがリンクをたどってインタラクティブなグラフを自動生成
  • エージェントはグラフを横断して関連コンセプトを自動発見。人間には見えないつながりを見つける
  • Enrichment Agentが新しいテーブルを発見すると自動的にリンクを張り、グラフが自律的に成長
書くたびに知識ネットワークが
自律拡張。
Obsidianの体験をエンタープライズ級に。
