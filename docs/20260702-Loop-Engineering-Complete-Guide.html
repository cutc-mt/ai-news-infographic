③ ループの基本構造

Act実行する
Observe結果を観察する
Decide次を判断する
Repeat繰り返す
  • act（実行） — タスクを実行する。コードを書く、テストを走らせる等
  • observe（観察） — 結果を観察する。エラーを見る、出力を確認する
  • decide（判断） — 次に何をすべきか判断する。修正するか進むか
  • repeat（繰り返し） — ゴール条件を満たすまで繰り返す
この4ステップのサイクルこそが
ループエンジニアリングの心臓部！

④ 状態を外部に持たせる（超重要）

  • 進捗をAIの会話コンテキスト内に溜め込んではいけない
  • 外部の永続成果物に状態を逃がす:
  •   → git履歴 — コミットで進捗を刻む
  •   → 進捗ファイル（progress.txt） — 現在の状態を書き出す
  •   → タスク定義（prd.json） — やるべきことを外部定義
  • 毎回フレッシュな状態で再起動→外部記録から「どこまで終わったか」を読み直して続行
  • コンテキストが膨らんで迷子になる前に毎回リセット
コンテキストが膨らむとAIは
「迷子」になります。
外部ファイルが命綱！

⑤ 代表的なループ実装

  • Ralph loop — 1つのプロンプトファイルを完了まで繰り返し食わせる手法。bashの while true 一行から始まったパターン。1イテレーション = 1タスク
  • Claude Code /loop — Claude Code内蔵のループ実行。タスクを渡して自走
  • Codex /goal — OpenAI Codex CLIの組み込みRalph loop。目標を渡して永続ワークフロー化
  • Ralphify等の専用ランタイム — ralph loops標準に沿った実行基盤
while true; do
  claude -p "$(cat prompt.md)" \
    --allowedTools "Edit,Write,Bash"
done
Ralph loopの原点は
たった1行のbashスクリプト！
シンプルさが強さ。

⑥ ループに必須の4要素

  • 明確なタスク — 何を達成すれば「完了」かが一文で言える状態にする
  • 安全な作業場 — 失敗してもダメージが閉じる環境（ブランチ・サンドボックス）で回す
  • 本物の成功テスト — テスト・ビルド・チェック等、機械的に合否を判定できる基準を置く
  • ハードストップ — 回数・時間・コストの上限を必ず設ける（無限ループ・トークン暴走防止）
この4つが欠けたループは暴走する！
設計の前提条件です。
特にハードストップは絶対に。

⑧ Claude Codeでの具体的実践

  • CLAUDE.mdにタスクの回し方を書く — 完了を勝手に名乗らせないルールを定義
  • ループ協議 — 各タスクは「直線」ではなく「ループ」として走らせる設計
  • /loopコマンド — Claude Code内蔵のループ実行。タスクを渡して自走
  • /goal（Codex） — 目標を渡して永続ワークフロー化、TUIで制御
  • 進捗管理 — progress.txt等の外部ファイルに進捗を書き出す
# CLAUDE.md
## ループルール
- 完了したら progress.txt を更新
- テストが全パスするまで繰り返す
- 最大10イテレーションで停止
CLAUDE.mdにルールを書けば
AIが自律的に守る！
設計のコツは「外部化」。

⑨ 企業が向き合うべき3つの指針

  • いきなり全自走を目指さない — 失敗しても損失が小さい1業務（定型レポート・テスト付き小改修等）から始める
  • 成功テストと停止条件を先に決める — 「何をもって完了か」「何回・何分・何円で止めるか」を先に書く
  • ツール選びより業務設計 — Claude Codeの/loopでもCodexの/goalでも、回す中身の設計が9割
ツールの差より設計の差。
Claude CodeでもCodexでも、
良いループ＝良い設計。

⑩ ループ化に向かないタスク

  • 成功を機械的に判定できないタスク — 「良いデザインか」等の主観的判断はループできない
  • 最終的な経営判断 — 企業の方向性を決めるのは人間の役割
  • 正解が一つでない創作の方向性決定 — ブランディングやUXの方向性等
これらは人間が握るべき。
全てを自動化するのではなく、
適材適所が重要です。

⑫ 実践的なループ設定例

  • タスク定義（task.json） — 何を・いつまで・どうやって完了するか
{
  "task": "ユーザー認証APIの実装",
  "done_when": "全テストパス + 型チェックOK",
  "max_iterations": 15,
  "max_cost_usd": 5.00,
  "sandbox": "feature/auth-v2",
  "check_cmd": "npm test && npx tsc --noEmit"
}
  • 4要素を全て定義：明確なタスク ✓ 安全な作業場 ✓ 成功テスト ✓ ハードストップ ✓
これが1つのループ定義。
このjsonを渡すだけで
AIが自走します！

⑬ 外部状態管理の実践パターン

  • progress.txt — ループのたびに進捗を追記。次イテレーションで最初に読む
# progress.txt
[1/5] ✅ ユーザーモデル作成
[2/5] ✅ 認証ミドルウェア実装
[3/5] 🔄 JWT検証ロジック（テスト失敗中）
[4/5] ⬜ パスワードリセットAPI
[5/5] ⬜ ドキュメント更新
  • gitコミット — 各イテレーション終了時にコミット。失敗時は git reset で戻れる
  • 人間もprogress.txtをチェックすればリアルタイム監視が可能
progress.txtが人間とAIの共通言語。
いつでも途中経過を確認できる。

⑭ 安全な作業場の設計

  • Gitブランチ分離 — ループは常にfeatureブランチで回す。mainには直接触らせない
  • Dockerサンドボックス — コンテナ内で実行。ホスト環境を汚さない
  • 権限制限 — 読み取り専用のリポジトリや制限されたAPIキーを使用
  • スナップショット — 開始前に状態を保存。失敗したら即座に復元
  • コスト上限 — APIのmax_tokensや予算上限を設定
「失敗しても良い環境」を
先に作るのがループ設計の第一歩。

⑮ 成功テストの設計コツ

  • 機械的に判定可能 — 「テストが通る」「ビルドが成功する」等の明確基準
  • 段階的なチェック — lint → type check → unit test → integration test の順
  • 高速フィードバック — テストは数秒〜数分で終わるもの。長時間のE2Eはループに不向き
  • リグレッション検知 — 既存テストが落ちたら即座に停止・ rollback
# チェックコマンド例
npm run lint && \
npx tsc --noEmit && \
npm test -- --coverage && \
echo "✅ ALL CHECKS PASSED"
「パスした/失敗した」の二値判定。
曖昧さがなくてこそループが回る！

⑯ ハードストップの実装

  • イテレーション回数 — 例: 最大15回。1回あたりのコストを見積もって上限を設定
  • 時間上限 — 例: 30分。深夜に回す場合は必須
  • コスト上限 — 例: $5.00。API使用量を監視して超過したら停止
  • デッドマンスイッチ — 一定時間レスポンスがない場合の自動停止
# ハードストップ設定例
MAX_ITERATIONS=15
MAX_TIME_MINUTES=30
MAX_COST_USD=5.00
TIMEOUT_SECONDS=300
ハードストップなし＝
無限ループの危険。
トークン暴走で請求が膨らむ！

⑰ ループ設計の全体像まとめ

  • Step 1 — タスクを一文で完結させる。「〇〇を実装してテストを全パスさせる」
  • Step 2 — 安全な作業場を用意。featureブランチ or Docker
  • Step 3 — 成功テストを定義。テストコード or ビルドコマンド
  • Step 4 — ハードストップを設定。回数・時間・コストの上限
  • Step 5 — 外部ファイルで状態管理。progress.txt + task.json
  • Step 6 — ループを起動。人間は監視・介入だけ
設計6ステップで完了。
あとはAIに任せるだけ！
人間は監視者に。
