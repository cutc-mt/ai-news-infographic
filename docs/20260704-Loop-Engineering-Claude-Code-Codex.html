① AIチャットからAIエージェントへ: パラダイムの大転換

  • 昔は1問1答のチャットが基本だった
  • ChatGPT、Claude、Gemini、Grokなどが爆発的に普及
  • 今はAIが自律的に動いて目的に向かってアクションする時代
  • 調べて、ツールを使って、実行してくれるエージェントが登場
チャットで質問して終わり…ではなく、AIが自ら考えて動く時代に！

② AIモデルの進化: 4つの要因がエージェントを可能に

  • 指示の追従性が大幅に向上 — より正確に指示を遂行
  • コンテキストウィンドウの拡大 — 数万→100万〜200万トークンへ
  • 推論能力の向上 — 段階を踏んで答えを導き出せるように
  • ツール利用能力 — いつどのツールを使うかをAI自身で判断
Claude登場時は10万トークンがすごかった…今はその5〜10倍！

③ コーディング能力の飛躍: Claude Code & Codex の台頭

  • Claude Code — AnthropicのCLI型コーディングエージェント
  • Codex — OpenAIのコーディング特化型エージェント
  • ファイルの読み書き、編集ツールを使いながら自律的に開発
  • これらを使いこなす人がめちゃくちゃ増えている
AIエージェントを使いこなす人が、これからの時代の勝者になる！

④ ループエンジニアリングとは: AIエージェント時代の核心

  • AIが調査→実行→検証→修正を繰り返せるようにする設計
  • このループをどう設計するかが仕事のスピードと質に直結
  • 誰が使っても同じようになるわけではない — 設計力が差になる
  • プロンプトエンジニアリングに代わる次の重要スキル
「どう質問するか」から「どうループさせるか」へ！時代が変わった

⑤ オーケストレーターとサブエージェント: 企業の組織と同じ

  • オーケストレーター = 中間管理職 — 計画を立てて作業を割り振るAI
  • サブエージェント = 実務担当 — 実際にコーディング・調査するAI
  • Claude CodeでもCodexでもサブエージェントを立てられる
  • 企業の「ディレクター→各担当者」と同じ構造でAIが動く
オーケストレーターが全体を管理して、サブエージェントに仕事を振るイメージ！

⑥ 人間の役割: ディレクターとしての4つの責務

  • 目的の決定 — 何を達成したいのかを明確にする
  • 制約の設定 — やってはいけないこと・守るべきルール
  • 優先順位の決定 — 何を優先すべきかの方向性
  • 承認と最終判断 — 危険な操作や最終結果を人間がチェック
作業そのものはAIがやる！人間は方向決めと承認に徹する時代

⑦ ループの具体例: バグ修正のワークフロー

  • ① 原因を調査 — ログやコードを分析して根本原因を特定
  • ② 修正を実行 — 原因に基づいてコードを修正
  • ③ テストを実行 — 修正が正しく機能するかを検証
  • ④ 結果をレビュー — 問題があれば再修正へループバック
調査→修正→テスト→レビュー→（問題あれば）修正…のサイクル！

⑧ ループの具体例: 資料作成のワークフロー

  • ① 情報のリサーチ — 必要な情報をWeb等から収集
  • ② 構成案の作成 — 目次や章立てを設計
  • ③ ドラフトの作成 — スライドや文書を生成
  • ④ レビューと修正 — クオリティチェックして必要なら再生成
100ページでも200ページでもAIのループで自動生成できる時代！

⑨ 画像生成もループ: 品質チェックの自動化

  • スライド資料の画像生成もAIが自動で実行
  • 生成した画像が適切かどうかもAIが自動チェック
  • クオリティが低ければ自動で再生成して内容を作り替え
  • 最終的にPDFとして出力されるまで全自動
ゴースト文字やクオリティ不足もAIが検知して再生成！

⑩ CLAUDE.md / CODex.mdの活用: 指示書の書き方が命

  • CLAUDE.md — Claude Codeに読み込ませる指示書ファイル
  • Codex.md — Codex用の同様の設定ファイル
  • ここに目的・制約・優先順位・作業手順を書き込む
  • 指示書の質で成果が9割決まると言っても過言ではない
プロンプトの上手い下手じゃなくて、指示書（設定ファイル）の質が全て！

⑪ 目的とゴールの明確化: ループの出口を決める

  • ゴールを明確に設定しないとAIは止まらない
  • 「どうなったら終わりか」を明示的に定義する
  • 成功基準や完了条件を定量化して指定
  • 出口がないと無限ループに陥るリスクがある
ゴール未設定 = 無限ループ！必ず「いつ終わるか」を決めよう

⑫ タスクの適切な分割: 大きすぎず小さすぎず

  • 大きすぎるタスクはサブタスクに分割して割り振る
  • 各タスクに明確な担当者（サブエージェント）を割り当て
  • 小さすぎてもオーバーヘッドが増えるのでバランスが重要
  • タスク間の依存関係を整理して順序よく実行
ひとつのエージェントに詰め込みすぎない、逆に細かすぎてもダメ！

⑬ コストの爆増: API利用時の最大リスク

  • API利用時は従量課金で、使えば使うほどコストが増加
  • サブスクでもレート制限があるため実質コスト増と同じ
  • 無意味なループが無限に課金し続ける危険性
  • コスト上限や止める仕組みを必ず設計に組み込む
「ループしすぎて請求書が怖い」を防ぐ仕組みが必須！

⑭ ファイル衝突の危険: 複数エージェントの競合

  • 複数のエージェントが同じファイルを触ると衝突が発生
  • 一方の変更が先祖帰り（revert）して消える事故が起きる
  • Gitのブランチを分けて作業場所を分離するのが正解
  • 最後にマージして統合 — コンフリクトはその時解決
ブランチを3つ作って→それぞれ作業→最後にマージ！

⑮ セキュリティ対策: 作業環境の分離が必須

  • ローカルで動かすと全ファイルにアクセスできるリスク
  • 機密情報をAIが勝手に読み取る可能性がある
  • プロンプトインジェクション攻撃で情報が外部に漏洩する危険
  • クラウド環境やDockerコンテナで隔離して動かす
機密ファイルがある環境では絶対にそのまま動かさないこと！

⑯ 人間とAIの役割分担: オーケストレーション型の仕事術

  • 人間 = 目的決定・制約設定・優先順位・承認
  • AI = 計画・実行・検証・修正のループ処理
  • 人間が上に立ち、AIがオーケストレーターとして中間管理
  • 人間の仕事はより大きな意思決定にシフトしていく
作業しない！方向性を決めるのが人間の新しい仕事の形

⑰ スキル駆動型の自動化: スライド資料もループで生成

  • 超実務的なスキルが作られており、ループに組み込める
  • 例えばスライド作成スキルを回せば自動で資料生成
  • 内容を与えるだけで100〜200ページでも自動生成可能
  • 画像生成→品質チェック→修正のループも全自動
今回のスライド自体もAIのループで作られているかも！？

  • プロンプトエンジニアリング → ループエンジニアリング へ
