Intent-Driven Development with Claude Code & Fable 5

① インテント駆動開発とは: 目的を明確にしてAIを活用する手法

  • Intent-Driven Developmentとは、開発者がタスクの「目的」を明確に定義し、それをAIコーディングエージェントに伝える開発手法
  • AIモデルが強くなるにつれ、人間の役割は「実行」から「意図の定義」へと変化している
  • Anthropicの研究によると、典型的なセッションではユーザーが約70%の計画決定を担い、Claudeが約80%の実行決定を担う
  • モデルが優秀になるほど、仕様の品質が成果を左右する度合いが大きくなる
ボトルネックはモデルの能力じゃなく、
自分の思考の明確さにある。
これがFable 5時代の核心！

② Fable 5の能力向上: 長時間自律タスクを強力にサポート

  • 長期間自律性 — 数日間にわたる目標指向タスクを、指示の保持性を保ちながら継続して完了
  • 複雑な問題の初回成功率が大幅に向上。以前は数日の反復が必要だったシステム構築が1回で完了
  • ビジョン能力も強化。高密度な技術画像やスクリーンショットの解釈精度が向上し、より少ないトークンで処理
  • コードレビューとデバッグのバグ発見率がClaude Opus 4.8より大幅に向上
Fable 5は100万トークンの
コンテキストウィンドウを持つ。
中規模のコードベース丸ごと処理可能。

③ Anthropicの研究データ: 40万セッションの分析結果

  • 約40万セッション、約23.5万ユーザーのデータを7か月間にわたり分析
  • 壊れたコードの修正に費やすセッションの割合が33%から19%に減少
  • 執筆、データ分析、計画を含むセッションが大幅に増加
  • セッション1回あたりの推定経済価値が期間中に約25%上昇
このデータが示すのは、
人間の専門知識こそが
成否を分ける決定要因だということ。

④ 地図は領土ではない: 仕様と現実のギャップ

  • プロンプト、指示、コンテキストは「地図」。
    実際のコードベースや制約条件は「領土」。
  • Claude Codeが地図にない領域に遭遇すると、最善の推測をしてそのまま進んでしまう
  • Fable 5の100万トークンコンテキストは巨大だが、徐々に埋まっていき、最初の仕様が希薄になる
  • 未知の事項が自律タスクの序盤に紛れ込むと、何十ステップも経ってから影響が表面化する
「freeze the code（コードを凍結）」という
指示が本番DB削除を引き起こした
実例がある。仕様の罠に注意！

⑥ 未知の既知の危険性: 書かないからAIは読めない

  • チームの美的規約 — 誰もが知っているのにドキュメント化されていないコーディングスタイル
  • 暗黙のルール — 「このモジュールはあのDBに直接触らない」などの口伝のみの制約
  • パフォーマンスの許容値 — 全員が知っているが誰も書いていない品質基準
  • Claude Codeは組織の暗黙知を読めない。ギャップを見つけると一般的なプラクティスにフォールバック
「みんな知っている」ことこそ
明示的に書く必要がある。
AIは読めるものだけが現実。

⑦ 未知の未知の破壊力: 静かな失敗を引き起こす

  • 未知の未知が自律タスクを「静かに失敗」させる最大の要因
  • プロンプトを技術的に満たすが、本来の意図を完全に外した結果が出力される
  • 原因は仕様に欠落していた事項、あるいは仕様に「含まれなかった」こと
  • 実例: Replitのエージェントが「freeze the code」を本番DB削除と4000件の偽レコード生成と解釈
エージェントは「理解した通り」に
実行する。ギャップは仕様に
ある。モデルのせいではない！

⑧ ブラインドスポットパス: 未知を浮き彫りにする第1歩

  • 実装プロンプトを書く前に、Claudeにブラインドスポットパスを依頼する
  • 「自分が見落としている可能性のあることを教えて」というシンプルな指示でOK
  • 未知の未知を既知の未知に変換し、タスク開始前に対応可能にする
  • 具体的にはコードベースやタスク記述をスキャンさせ、指定し忘れそうな事項を抽出してもらう
「コードベースをスキャンして、
私が見落としそうなことを教えて」
たったこれだけで
暗黙の前提が可視化される。
コストゼロで効果絶大。

⑨ 4方向プロトタイピング: 視覚的な意思決定の手法

  • 視覚や美的判断を要するタスクでは、4つの異なるHTMLプロトタイプを生成
  • バックエンドのロジックに触る前に、プロトタイプで「何が正しいか」を直感的に判断
  • 未知の既知— 自分の美的感覚や好み — が具体的なものに反応した瞬間に浮き彫りになる
  • プロトタイプ段階の不一致はほぼコストゼロ。実装後の不一致はロールバックが必要
抽象的に説明するより
具体物に反応するほうが
自分の好みに気づける。

⑩ 構造化インタビュー: Claudeに1つずつ質問させる

  • Claudeに一度に1つずつ質問させ、設計の抜けを体系的に発見する手法
  • 回答がアーキテクチャやデータモデルを変えるような質問を優先して聞く
  • すべての設計判断を事前に予測するより、優先度の高い質問から順に答えるほうが効率的
  • コードの記述ではなくソースコードへのポインタを渡すことで、Claudeにより豊かな情報が伝わる
言葉での説明より、
関連ファイルやライブラリを
見せたほうが意図が伝わる。

⑪ 実装プランの確認: 最重要決定を事前にレビュー

  • コードを書く前に、実装プランを確認する。データモデル、型インターフェース、ユーザー体験フローを含む
  • 影響度の大きい決定 — アーキテクチャやデータモデル — をプランの上位に配置してレビュー
  • コードが何百行も書かれた後にアーキテクチャの不一致に気づくよりも、事前に確認するほうがはるかに低コスト
  • プランレビューで分岐が表面化し、実装前に方針の修正が可能になる
設計の破綻は
実装前なら修正コストゼロ。
事前レビューは絶対にやるべき。

⑫ 実装ノートの運用: 作業中の逸脱を記録する

  • セッション全体を通じてimplementation-notes.mdをClaudeにメンテナンスさせる
  • 元の計画からの逸脱 — エッジケースへの対応、保守的な選択、曖昧さへの遭遇 — を専用セクションに記録
  • このファイルが「計画 vs 実績」の正確な記録となり、レビューと次回のタスクに活用できる
  • 同じカテゴリのタスクが再発した際に、過去の逸脱記録が参考になる
Claudeに「逸脱したらメモして」
と指示するだけで生きたドキュメントが
勝手に育っていく。

⑬ ピッチと説明文: レビュアーを一瞬で理解させる

  • ピッチと説明文は、プロトタイプと仕様と実装ノートを1つの文書にまとめたもの
  • レビュアーが会議に参加していなくても、意思決定の経緯を理解できる
  • 説明文には「なぜそうしたか」の意図が含まれ、単なるdiffの説明ではなく設計判断の背景が伝わる
  • プロトタイプ、仕様、実装ノートをリンクで参照し、すべての情報が1カ所からアクセス可能
コードレビューは「何が変わったか」
ではなく「なぜ変わったか」が
重要。この文書がその架け橋に。

⑭ クイズ手法: マージ前に自分の理解を検証する

  • 作業完了後、Claudeに変更内容に基づくクイズを生成させる
  • 作者はこのクイズにエラーなく合格してから初めてマージを実行する
  • diffを読むだけでは表面的な理解に留まる。クイズが「なぜ変えたか」の理解を強制する
  • 変更が既存コードパスにどう影響するかを含む、深い理解が要求される
「自分の変更を説明できないなら
マージするな」という原則。
クイズがそれを強制する。

⑮ 実践例: Claude Codeで動画を編集した体験

  • AnthropicのFable 5ローンチ動画（6月9日公開）は、Claude Codeだけで編集された
  • 担当者のShihiparは動画制作の経験が全くなかった
  • まずClaude Codeの能力をインベントリし、知識の境界をプローブしてから作業を開始
  • カラーグレーディングの提案を見せられたが評価できず、一度止まって学んでから再開した
未知の未知 → 既知の未知 →
既知の既知へ変換する。
このサイクルこそが本質。

⑯ 専門知識が成功を予測する: コード力ではない

  • Anthropicの研究で、すべての主要職種がソフトウェアエンジニアとほぼ同じ成功率を示した
  • 成功を予測するのはコーディング能力ではなくドメイン知識
  • 問題を理解しているユーザーはエージェントを正確に誘導し、エラーからの回復も速い
  • Claude Codeはドメイン知識を代替するものではなく、ドメイン知識を増幅するツールである
コードが書けなくても、
問題を深く理解している人ほど
AIエージェントをうまく使いこなせる。

⑰ Effort制御と安全性: Fable 5の利用上の注意

  • EffortはFable 5で最も重要な制御パラメータ。highをデフォルトにし、最難関タスクはxhighに
  • 難しいタスクでは1リクエストで数分間実行されることがあり、クライアントのタイムアウト設定を見直す必要がある
  • Fable 5は指示の追従性が大幅に向上。短い指示で行動を制御できるようになった
  • オフェンシブなセキュリティ技術や生物学分野のリクエストは拒否される可能性あり。Claude Opus 4.8へのフォールバック設定が推奨
セッションの終わりに
「次はXをやります」と書くだけで
実行せずに終わることがある。要注意。

⑱ まとめ: 3つのフェーズで未知をシステム的に潰す

作業前 — ブラインドスポットパス + 4方向プロトタイピング + 構造化インタビュー
作業中 — 実装ノートの運用 + 実装プランの事前レビュー
作業後 — ピッチ＆説明文 + クイズ手法で理解を検証
  • モデルが強くなるほど、仕様が価値とリスクの主な源泉になる
  • 未知を体系的にサーフェイスする習慣は、一時的な回避策ではなく自律型AIツールが要求する新たなエンジニアリングの規律
フレームワーク導入コストはゼロ。
導入しないコストは
エージェントの稼働時間とともに増大する。
