⚖️ KarpathyのLLM Wikiは愛されているが
スケールしない。では何がスケールするのか

📝 チャンネル: Cole Medin
①

AIエージェントの2つの種類: 個人型と本番環境型

  • パーソナルエージェント — Claude CodeやHermesなど、個人のローカルで動くもの
  • 本番環境エージェント — プラットフォームの一部として複数ユーザーに提供されるもの
  • 現在人人は個人型に過度に注目していると指摘
  • 真のビジネス価値の大半は本番環境に出荷される側にある
誰もがパーソナルAIに夢中だけど、企業価値は本番環境の方にあるんだね
②

パーソナルエージェントの仕組み: マークダウンWiki方式

  • Karpathy LLM Wiki的なナレッジベースを構築するのが主流
  • タグ付け・分類されたインデックス文書とエンティティで構成
  • エージェントと対話しながら時間をかけて知識を蓄積していく
  • Claude CodeやHermesなどの既製品もよく使われる
自分専用のセカンドブレインは最高に便利！でも…
③

パーソナル方式の強み: シンプルさと柔軟性

  • すべてを自社システム上に保持し、高速なアクセスを実現
  • 自作でも既製品でも、最もシンプルで柔軟な方法
  • 第二の脳と対話して自然に知識ベースを構築できる
  • ガバナンスやアクセス制御は気にしなくてよい
個人用途ならこれで十分！でも他の人が使うと話が変わる…
④

スケールの壁: マークダウン方式は限界にぶつかる

  • 他のユーザーが同時にエージェントを使い始めると破綻する
  • アクセス制御や大規模データ取得の問題が発生
  • すべてをマークダウンで管理すると検索や整理が不可能に
  • 結論：データベースが存在する理由はここにある
「そもそもデータベースがあるのには理由があるんです」 — Cole
⑤

コスト問題: セカンドブレインは意外と高い

  • ClaudeやCodexのようなコーディングエージェントSDKはサブスクリプション連携
  • そのサブスクは個人利用専用なので本番環境では使えない
  • システムプロンプトやマークダウン文書を丸ごと読み込ませる方法は非効率
  • データ取得規模が増えるとコスト最適化が必須になる
個人サブスクを本番で使えないのが最大の壁！
⑥

本番環境エージェントのアーキテクチャ: データベース方式

  • マークダウンではなくデータベースを使ってスケーリング
  • HermesやClaude CodeのWikiを用意するだけでは不十分
  • データベースが保存・処理すべきものが2つある
  • この仕組みは個人エージェントよりもはるかに一般的
本番環境では根本的にアプローチを変える必要がある！
⑦

2つの核となる要素: コンテキスト取得器とエージェントメモリ

  • コンテキスト取得器 — エージェントにビジネスデータへのアクセス権を付与
  • エージェントメモリ — ユーザーの短期記憶と長期記憶に相当
  • データ形式を伝え、エージェントが何を照会できるかを理解可能にする
  • 時間をかけてユーザーに関する知識を蓄積していく仕組み
この2つが本番環境エージェントの肝！後半で詳しく解説します
⑧

Redis Irisの紹介: エージェント向けデータベース

  • Redis Irisは現在プレビュー版の強力な新機能
  • ビジネスコンテキストとユーザーメモリを1箇所で実現可能
  • データベースとしてRedisをラップし、エージェントのアクセス権限を向上
  • アイデアは任意の本番システムに適用可能（DBラッパー）
Redisを使わなくても概念はすべてのDBプラットフォームに通用するよ
⑨

デモ環境の構成: eコマースストアの模擬データ

  • ECサイト向けの模擬データをRedisのキーバリューストアに格納
  • 顧客情報・注文・製品・出荷など非構造化データとして保存
  • 分析担当者と顧客対応担当者の2つのユースケース
  • Pydantic AIでエージェントを構築
ECサイトなら分析エージェントとサポートエージェントが使える！
⑩

Pydantic AIの推し: コーディングSDKとの違い

  • Coleはすべての本番環境エージェントにPydantic AIを推奨
  • Claude Agent SDKやCodex SDKは長いコーディングタスク向け
  • 処理速度が遅く、トークン消費も多い
  • 本番環境にデプロイするものにはPydantic AIの方が適している
コーディング補助と本番エージェントは別物という主張！
⑪

デモの結果: 多段階のデータ検索が1リクエストで完結

  • 「ジョーダン・リベラ、注文が遅れているのはなぜ？」と尋ねる
  • エージェントが記憶→顧客情報→注文→出荷と自動で追跡
  • 1000トークン未満で全情報を収集して回答を生成
  • 「返金より再発送を希望」という過去の記憶も自動参照
何万ものトークンを使わずに完璧な回答！すごい…
⑫

コンテキスト取得器の仕組み: 構造の提供

  • Redisデータはすべてキーと値のペア（非構造化）
  • エージェントは何を検索できるかも把握していない状態
  • コンテキスト取得器が構造とデータインテリジェンスを提供
  • KarpathyのWikiのメタデータのような役割を果たす
WikiのメタデータをDBレイヤーで自動実現するイメージ
⑬

MCPツールの自動生成: エンティティからツールが生まれる

  • エンティティを定義すると、自動的にMCPツールが生成される
  • 都市で顧客フィルタリング、メールで検索など属性ごとにツール
  • テキスト型には全文検索機能も自動付与
  • 必要なツールがなければエンティティを調整して即追加可能
これがこの動画で一番気に入ってる機能！自動生成最高
⑭

実践例: 1回のMCP呼び出しで全データ取得

  • 「遅延している注文を表示」→ 1つのMCP呼び出しでステータスフィルタ
  • 「返金についてのチケット」→ テキスト検索で即座に該当件を表示
  • インデックスドキュメントの読み込みや巨大な全文検索は不要
  • 超高速・超効率なデータアクセスを実現
マークダウンWikiでやっていたことがDBなら1発で終わる！
⑮

エージェントメモリ: 短期記憶と長期記憶の管理

  • 短期記憶 — すべての会話セッションを保存
  • 長期記憶 — 重要な情報を抽出して永続的に保持
  • 短期から長期への自動的な記憶の昇格プロセスを実行
  • ベクトルを使ったセマンティック検索で記憶を検索
セカンドブレインと同じ手法をDBでスケールさせる！
⑯

記憶の昇格: 会話から長期記憶への自動抽出

  • Redis Irisがバックグラウンドプロセスで自動的に実行
  • 個人のセカンドブレインではmemory.mdに手動抽出する手法が一般的
  • ここでは自動で重要な会話内容をRedisに保存
  • 各ユーザーが数百万件のメモリを持っても対応可能
memory.mdの手作業が不要になるのが革命的！
⑰

メモリの実証: 新しいセッションでも記憶が継続

  • 完全に新しいセッションを開始して好みを尋ねる
  • エージェントが即座に長期記憶を引き出して回答
  • 過去の注文・サポートチケット・ユーザーの好みを要約して提示
  • 「返金より再発送を希望」が別セッションでも正確に反映
明示的に記憶を指示しなくても自動抽出される！すごすぎ…
⑱

まとめ: 個人と本番でアプローチを分けよ

  • 個人エージェント → LLM Wiki・Hermes・Claude Codeで十分
  • 本番環境エージェント → DB＋コンテキスト取得器＋メモリが必須
  • 記憶抽出・検索ツール構築のプロセスを維持する必要がある
  • すべてがDBで管理されることで完全な制御と可視性を確保
  • どちらの分野も今後コンテンツを継続していくと宣言
正しいツールを正しい場所で使うことが、スケールの鍵だ！
