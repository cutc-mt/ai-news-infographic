🚀 Anti-GravityでSDLC全体を自動化
AIエージェント開発ライフサイクルの実践

📝 チャンネル: Google Cloud Tech
①

AIコーディングエージェントの
よくある不満とは

  • 多くの機能を開発する一方で、多くのバグも生み出してしまうという課題
  • コード量が増えると、必然的に技術的負債も増える傾向
  • コード生成には過度に依存しているが、保守・レビュー・テストがおろそか
  • これが今回の講演の出発点
機能を作るだけじゃ
ダメなんだよね…
バグも増える
②

SDLC（ソフトウェア開発
ライフサイクル）の基本

  • 問題を特定し理解することから始まる
  • 次に解決策を設計する
  • それを実装する
  • 最後にレビューしてテストを行う
  • これはSDLCのすべてではないが、議論には十分
問題→設計→実装→
レビュー・テスト！
このサイクルが基本📝
③

GitFit：GitHubの
トレーディングカード

  • git.foots.comで貢献度スコアやコミット数などを表示するサイト
  • まるでサッカーのトレーディングカードみたいなUI
  • オープンソースで公開されている
  • 講演者はこのプロジェクトのファン（関係者ではない）
自分のカードを
見るのが楽しい！⚽卡片
④

GitFitで発見した
名前表示のバグ

  • ユーザーの姓が正しく表示されていないという問題
  • 3つの名前から構成されるフルネームの姓の処理が間違っている
  • アプリが姓の最後の部分だけを姓と認識してしまう
  • ミドルネームがある場合に誤判定が発生
ミドルネームがあると
姓の判定がズレる！🐛
⑤

Daftの複雑な問題を
Anti-Gravityで要約

  • Daft：オープンソースのデータ処理ライブラリ
  • 問題と添付の議論がかなり複雑で読むのが大変
  • Anti-Gravityに渡して「何が起きているのか要約して」と依頼
  • 問題の特定と理解のプロセスが大幅に迅速化
複雑なIssueも
AIで一瞬で理解！✨
⑥

解決策の設計：
GitFitに分布機能を追加

  • 自分のスコアがGitHub全体でどの位置にいるかを知りたい
  • 分布図で上位X%に入っているか確認できる機能
  • 視覚的なデザインとシステム面の設計が必要
  • 実際にプルリクエストを送信した
自分の位置が
わかるのは面白い！📊
⑦

プライバシーに配慮した
データ収集の設計

  • 約20,000件のアカウントを取得して分布を作成
  • プライバシーに配慮した匿名化された方法で保存
  • 過去1年間に活動のあった人も調査
  • メンテナーと「どうやってデータを取得するか」を議論
プライバシー配慮は
設計の重要な一部分！🔒
⑧

実装フェーズ：
エージェントとの対話プロセス

  • まずリポジトリのURLをエージェントに伝える
  • 「このフォークで作業して、最新版をプルして」
  • 問題の内容を説明し、実装したい機能を具体的に指示
  • エージェントとの繰り返しのやり取りで完成度を高める
エージェントとの
対話がカギ！💬
⑨

名前バグの解決策：
カスタマイズ可能なアプローチ

  • 既存の「最後の2つを姓とする」手法はミドルネームで破綻
  • ユーザーが自分で名前の表示をカスタマイズできるようにする
  • 国選択機能と同じ仕組みを名前にも適用
  • 最も柔軟で包括的な解決策
ユーザーが自由に
設定できるのがベスト！💡
⑩

URLパラメータによる
状態の保存と復元

  • 国選択をURLパラメータに保存（例：?country=CA）
  • ページ遷移しても情報が保持される
  • パラメータを削除すればリセットも可能
  • 名前のカスタマイズも同じ方式で実装する方針
URLパラメータは
シンプルで便利！🔗
⑪

フォークを使った
安全な実装プロセス

  • 元のプロジェクトに影響を与えずにフォークで実験
  • Anti-Gravity関連のプロジェクトを専用フォルダに整理
  • エージェントがプロジェクトの場所を把握できるようにする
  • クローン、プル、ブランチ切替をエージェントに指示
フォークで安全に
試せるのが良い！🌿
⑫

ローカルでの
動作確認とデモ

  • 「npm run dev でしたっけ？」とエージェントに実行方法を質問
  • package.jsonを確認してローカルサーバーを起動
  • 実際にデモブランチで動作を確認
  • 名前の編集機能がURLパラメータに保存されることを検証
動かして確認するのが
一番大切！▶️
⑬

レビューとテスト：
プルリクエストの確認

  • タイトルと説明はAnti-Gravityで自動生成されたもの
  • 多数の変更ファイルを効率的にレビューする必要がある
  • チームメンバーが作ったPRを自分で見直すケースも
  • レビュー前に品質を担保したい
PRのレビューも
AIで効率化！📋
⑭

各ファイルの要約を
AIで1〜2文にまとめる

  • 「変更されたファイルをそれぞれ1〜2文で要約して」と指示
  • GitHubから情報を取得して各ファイルの要約を作成
  • 例：「画像レンダリングAPIルートがカスタム名パラメータを読むように更新」
  • 確認済みのファイルは既読マークで管理
1〜2文なら
サクサク確認！⚡
⑮

気になる部分を
対話的に深掘りレビュー

  • 共有ユーティリティの変更など理解が難しい部分を発見
  • 「この部分がよくわかりません」と質問
  • 「一行ずつ説明してもらえますか？」で詳細な解説を取得
  • 必要な深さまでいつでも掘り下げられる
わからないところは
とことん聞ける！🔍
⑯

AIを使ってもコードの
制御は失わない

  • 「AIに任せるとコードの制御を失うのでは？」という懸念
  • 実際は逆 — より多くの制御を得られる
  • コードベースのあらゆる部分を簡単に掘り下げられるから
  • プロセスを特定する手助けにより迅速かつ正確な判断が可能
AIのおかげで
むしろ制御力UP！💪
