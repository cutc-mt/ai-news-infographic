① なぜCLAUDE.mdが必要か

  • CLAUDE.mdはClaude Codeにプロジェクト固有の指示を与える設定ファイル。
  • 同じプロンプトでも、CLAUDE.mdの有無で出力品質が劇的に変わる。
  • Claude Codeを自分のチームメイトのようにカスタマイズできるのが最大の価値。
  • Anthropic公式も推奨する設定ファイルで、コミュニティでは10万スターを獲得。
同じ指示なのに結果が全然違う！
これがCLAUDE.mdの魔法 ✨

② CLAUDE.mdを機能させる4原則

  • 具体的に書く — 曖昧な指示ではなく、具体的なルールと制約を明記する。
  • プロジェクトに合わせる — 汎用的な指示よりプロジェクト固有の文脈が重要。
  • 維持し続ける — プロジェクトが進化したらファイルも更新し続ける。
  • 過剰にならない — 必要な情報だけを書き、冗長な指示は避ける。

⑩ 書くと逆効果 — 失敗する7パターン

  • 過剰な指示 — あらゆるケースを網羅しようとして逆にClaudeを混乱させる
  • 矛盾するルール — 「短く書け」と「詳細に書け」が共存している
  • 過度な制約 — ファイルサイズや行数の厳しすぎる制限
  • 抽象すぎる指示 — 「きれいなコードを書け」などの具体性のない指示
  • 古い情報 — プロジェクトが進化したのにCLAUDE.mdを放置
  • 丸写しの禁止リスト — すべての「やらないこと」を羅列しすぎ
  • 重複する設定 — .eslintrcやtsconfigと重複する内容を二重管理

⑪ 過剰・矛盾・制約 — 何が起きるか

  • 過剰な指示を書くと、Claude Codeはどのルールを優先すべきか判断できず品質が低下する。
  • 矛盾するルールがあると、毎回ランダムに異なる出力が生成される。
  • 厳しすぎる制約は、本来の能力を発揮できずに「安全な」出力ばかりに。
  • 解決策は「最小構成」から始めて、必要なルールだけを少しずつ追加していくこと。
少ないほど良い！
1つのルールが100%守られる方が
10のルールが半分守られるより価値がある ⚖️

⑮ ループ制御セクションの書き方

  • ループ制御はClaude Codeが自律開発中にいつ止まるべきかを定義するセクション。
  • 「テストが全てパスしたら止める」「エラーが3回連続したら人間に確認」といったルール。
  • 無限ループを防ぐため、最大イテレーション数やタイムアウトも設定可能。
  • grill-meと組み合わせると、計画フェーズと実行フェーズを明確に分離できる。
# ループ制御の例 max_iterations: 10 stop_conditions: - all_tests_pass - error_count >= 3 human_review: complex_logic_changes
