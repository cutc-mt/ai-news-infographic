Claude Code: ModelとEffortの使い分け

公式直伝の結論 — 「賢さ」と「粘り強さ」は別のツマミ
📅 2026年7月18日 | まさおAIじっくり解説ch

①ModelとEffortの根本的な違い

  • Model = どれだけ賢いか（何を知っているか）
  • Effort = どれだけ丁寧に作業するか（どれだけ頑張るか）
  • 片方は能力、もう片方は労力。完全に別物
  • この2つを混ぜないことが全ての出発点
結論：
Model = 知力
Effort = 労力
この軸を握れ！

②最大の誤解: 「考える時間」ではない

  • Effort ≠ 思考時間だけ（これは半分しか合っていない）
  • どれだけファイルを読むか：関連コードを自力でどこまで確認するか
  • どれだけ検証するか：テスト実行・ダブルチェックの回数
  • どこまで自走するか：確認を返す前に多段タスクをどこまで進めるか
Effortは「Claudeがどれだけ
一人で頑張り切るか」のツマミ。
思考時間だけじゃない！

③Modelが決めること: 重み（パラメータ）

  • Enterを押すと、全てが1つのAPIリクエストにまとまる
  • テキストはトークン化され、整数の配列に変換される
  • モデルの重み（数十億パラメータ）が確率を計算
  • 重みは訓練時固定。プロンプトでは1ビットも変わらない
Modelを変える＝
凍結された重みのセットを
丸ごと入れ替えること。

④コンテキストは「操縦」であって「教育」ではない

  • コンテキストで与えた知識はそのターン限り
  • 「一度教えたから次も分かる」は成り立たない
  • CLAUDE.mdやスキルは毎回コンテキストに供給し続ける設計が必須
  • 比喩: Model＝地図（書き換え不可）、コンテキスト＝行き先指示
地図は変えられない。
でもどこへ向かうかは
あなたが決められる！

⑤1トークンずつ生成される仕組み

  • 入力全体を重みに通して次トークンを1個予測
  • それを配列末尾に追加し、全体をもう一度計算
  • 200トークンの応答 = 重みを通る200回のパス
  • 待ち時間とコストの大半がこのループから生まれる
「const x = await」→
「fetch」→「(」→「url」→...
1文字ずつ確率計算！

⑥出力トークンの3つの顔

  • 思考（Thinking）: アクション前後の推論。ストリームされる
  • ツール呼び出し: Read/Edit等の構造化ブロック
  • あなたへのテキスト: 計画・進捗報告・最終まとめ
  • 3つは同じループから生まれる地続きのトークン。同じ単価で課金
思考もツール呼び出しも
テキストも同じトークン。
区別して課金されるわけじゃない！

⑦Effortの正体: 確信に達するまでの粘り

  • 低Effort: 少ないトークンで手早く着地。確かならあなたに尋ねる
  • 高Effort: 約7倍のトークンでより高い確信に到達
  • 計画→ファイル読込→検証→確認。順次プランを更新
  • 見つけたら残りをスキップ: 「最初のチェックで見つかったので不要」
「考えすぎ」は意図的に回避。
Anthropicがモデル訓練中に
特に警戒している挙動！

⑧Effortレベルの選び方

  • low: 最小限。分類・抽出・フォーマット変換
  • medium: バランス型。要約・一般的なコード解析
  • high（既定）: 高品質。詳細分析・品質重視
  • xhigh: コーディング・エージェント作業の推奨
  • max: 最難関のみ。複雑設計判断・大規模分析
迷ったら既定のhigh。
コード書くならxhigh。
maxはセッション限定！

⑨専門家・エキスパート・ジェネラリスト

  • Fable: ほぼ誰も解けない問題を捌く専門家（スペシャリスト）
  • Opus: 広い知識と深い洞察を持つエキスパート
  • Sonnet: 本当に優秀なジェネラリスト
  • Effortは、彼らがタスクにどれだけ時間を注ぐかのツマミ
Fableは「皆が詰まった時」に取っておく。
低Effortでも誰も気づかないものを
見抜く識別力！

⑩Model×Effortの組み合わせ例

  • Opus × 低Effort: エキスパート5分間。知識は豊富だがコードは速読
  • Sonnet × 高Effort: 午後使えるジェネラリスト。全て読み・実行・ダブルチェック
  • Fable（低Effortでも）: 皆が詰まった時の専門家。低Effortでも最高の識別力
Fableは取っておく戦略。
その識別力に
一番高い対価を払っているから！

⑪間違えた時: どちらを変えるか

  • まずコンテキストを疑え: プロンプト・CLAUDE.md・ツール・スキル
  • それでも間違えたら自問: 知識不足か？労力不足か？
  • 知識不足 → Modelを大きく
  • 労力不足 → Effortを上げる
  • ルーチン → Modelを小さく（速度↑・コスト↓）
設定を回す前に上流を見ろ！
多くの不具合はコンテキストで直る。
8割はここで解決！

⑫コストとパフォーマンスの相互作用

  • ルーチン: 大きいModelも小さいModelも正解するが、大きいModelは余分な検証で高コスト
  • 難しい多段タスク: 式が逆転。小さいModelは力尽き、大きいModelは少ないステップで到達
  • Fableは他Modelが到達できないタスクを完了させることがある
  • Effort設定を間違えるとModel選びよりコスト差が大きい
Effortの設定ミスが
Model選びミスよりコストに効く。
ここに気をつけよう！

⑬Effortの設定方法

  • CLI: --effort low|medium|high|xhigh|max
  • 対話モード: /effortコマンドで動的切替
  • 環境変数: CLAUDE_CODE_EFFORT_LEVEL
  • settings.json: effortLevelキー（maxは保存不可）
  • 優先順位: CLI > 環境変数 > local > project > user
maxはセッション限定。
設定ファイルに
保存できないから注意！

⑭パイプライン別Effort最適化

  • ファイル一覧取得 → low（単純作業）
  • コード解析 → medium（推論が必要）
  • README生成 → high（品質重視）
  • フェーズ別配分で40〜60%のコスト削減が可能
  • --session-idと--resumeでコンテキストを引き継ぎ
全部highで回すのではなく
タスクに合わせた配分で
コストを半分以下に！

⑮--max-budget-usdでコスト管理

  • --max-budget-usdで1回あたりの上限を設定
  • 月$10・1日20回実行なら1回あたり$0.016が目安
  • 上限に達するとis_error: trueで実行停止
  • --fallback-modelで混雑時の自動切り替えも可能
予算超過＝実行停止。
安全装置なしで
エージェントを回さないこと！

⑯--modelと--fallback-model

  • --modelでメインモデルを指定
  • --fallback-modelでレート制限時の代替を設定
  • 例: Sonnet指定 → 混雑時はHaikuに自動フォールバック
  • 注意: fallbackはprintモード専用。対話モードでは使えない
パイプラインの耐障害性を
フォールバックで担保。
本番運用では必須の設定！

⑰Opus 4.8の実務メモ

  • Opus 4.8のデフォルトはOpus 4.7のデフォルトとほぼ同じトークン量でより良い結果
  • デフォルトのままでアップデートの恩恵を受けられる
  • ultracode設定: xhigh送信＋マルチエージェント自動計画を有効化
  • ultracodeはセッション限定。/effort ultracodeで有効化
アップデートしたらデフォルトで。
余計な設定変更不要で
性能改善が得られる！

⑱まとめ: 正しいダイヤルを回す

  • 2つは別の問いに答える。混同するな
  • まず上流を疑う。8割はコンテキストで直る
  • 切り分け: 知識不足→Model、労力不足→Effort
  • ルーチンが続くならModelを下げて速度とコストを回復
  • Fableは最後の切り札。その識別力は最高に高い対価
次に「答えがイマイチ」と思ったら
正しいダイヤルへ
迷わず手を伸ばせる。
この地図が手元にあれば！
