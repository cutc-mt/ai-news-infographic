Geminiが勝手にPCを操作!?

新機能「Computer Use」がヤバすぎる — AIが画面を見て自分で動く時代が到来
2026年7月18日

① ついにAIが「見て・動く」段階へ

  • 2026年6月24日 GoogleがGemini 3.5 Flashに「Computer Use」を統合して発表
  • これまでのAIは「質問に答える」だけだったが、今回は画面を見て自分でクリックや入力を実行できるように
  • 以前は「Gemini 2.5 Computer Use」という単独モデルだったが、主力のFlashモデルに標準搭載されたのが大きな変化
  • 軽量で速いFlash系に組み込まれたことで、より実用的で使いやすくなった
AIが「答える」だけでなく「操作する」段階に入ったのは、業務自動化における見逃せない変化です！

② Computer Useの仕組み

1
アプリが画面のスクリーンショットをAIに渡す
2
AIが「次にやるべき操作」を判断する
3
アプリが実際に画面で操作を実行
4
操作後の画面をまたAIに渡す
5
目的を達成するまで繰り返す
人が画面を見ながら作業するのとほぼ同じ流れ。AIが「見て・判断して・動かす」の往復で複数手順をこなします。

③ Gemini 3.5 Flashの独自の特徴

  • 「なぜその操作をしようとしているのか」という意図も一緒に出力されるのがポイント
  • 開発者はAIの意図を把握しやすくなり、動きの確認や改善がしやすい
  • ブラウザ・スマホ・パソコンの3つのプラットフォームで動作可能
  • Chrome DevTools Protocol経由でブラウザを直接操作する実装も可能
単独モデルから組み込みツールに
進化したことで、開発者は
より簡単に導入できるようになりました！

④ 何ができるのか? 具体的な使い道

  • 業務の自動化 複数の画面をまたぐ定型作業をAIに任せられる
  • 情報収集 複数のサイトを横断して必要なデータを集める
  • テスト自動化 Webアプリの動作確認を人が毎回クリックしなくて済む
  • アクセシビリティ サイトが使いやすいかを確認する作業を支援
  • 例：「複数のサイトを順番に開いて決まった項目を転記する」という地味だけど時間のかかる作業が対象
現時点では主にWebブラウザの操作に最適化。スマホやPCのファイル操作は今後の拡張に期待。

⑥ 競合モデルとの比較

  • Claude Opus 4.8 がOSWorldで83.4とトップ。AnthropicのComputer Useは先行実績が豊富
  • GPT-5.5 が78.7で2位。OpenAIもOperator等でブラウザ操作に注力
  • Gemini 3.5 Flashは78.4でSonnet 4.6と並びながら、コスト面で有利なのが強み
  • Flashモデルは軽量・高速なので、大量の操作タスクを回す用途に最適
精度は上位モデルに迫りながら、
コストと速度で実用性を勝負。
バランス型としての強み。

⑦ 安全上の懸念 — 間接プロンプトインジェクション

  • 画面を読むAIには従来のAIにはなかった「だまされるリスク」がある
  • 悪意のあるページに「ここをクリックして」と仕込まれると、AIがそれを正しい指示と勘違いする危険
  • 間接プロンプトインジェクション：Webページを通じてこっそりAIを誘導する攻撃手法
  • 人間なら「怪しい」と気づけることでも、AIは騙されてしまうのが注意点
「便利だから全部おまかせ」ではなく
重要な操作は人間が確認する。
この姿勢が絶対に欠かせません。

⑧ Googleの安全対策 — 多層防御アプローチ

  • 確認フロー 取り消しにくい操作や機密性の高い操作ではユーザーに確認を求める仕組み
  • 自動停止 間接プロンプトインジェクションを検知したらタスクを自動的に止める
  • サンドボックス 安全に隔離された環境での実行を推奨
  • アクセス権限 の厳格な管理と人による確認を組み合わせる「多層防御」を推奨
Googleはdefense-in-depth（多層防御）の考え方を採用。一枚の盾ではなく、何重もの防御線を敷いています。

⑨ 料金体系 — コストパフォーマンス

  • Computer Useの追加により、入力トークン当たりの料金がわずかに高い
  • しかし単独モデル「Gemini 2.5 Computer Use」と比べると統合型の方が効率的
  • Flashモデルなので処理速度が速く、大量の操作タスクでもコストを抑えられる
  • 長時間のタスク実行においては、安価なFlashモデルの連続実行がコスト面で有利
単独モデル時代から統合型への移行で、
導入コストと運用の手間が両方とも削減！

⑩ 開発者向けの統合方法

  • Google AI Studio から直接Computer Useツールを利用可能
  • Vertex AI APIを通じてプログラムから操作を自動化できる
  • 組み込みツールとして統合されたことで、API呼び出し1回で画面操作まで完結
  • Chrome DevTools Protocolと連携して、ブラウザの直接操作も実装可能
// 例: Gemini APIでComputer Useを使う
const response = await model.generateContent({
  tools: [{ computerUse: {} }],
  contents: [{ role: "user", parts: [{
    text: "検索サイトでAIニュースを検索して"
  }]}]
});
以前は別モデルを呼ぶ必要があったが、
今は1つのAPI呼び出しで完了！
開発のハードルが大幅に下がりました。

⑪ ブラウザ操作 vs デスクトップ操作

  • ブラウザ操作 が現在最も最適化されている。Webサイト上のクリックや入力が得意
  • スマホ操作 も視野に入っているが、UIの複雑さで精度に課題
  • デスクトップ操作 ファイル管理やアプリ操作も徐々に対応予定
  • まずは「ブラウザ上の作業」から始めるのが現実的
ブラウザベースのタスクが圧倒的に多い現代において、まずはWebという戦略は正解です。

⑫ 従来のAIとの決定的な違い

  • 従来のAI 「質問 → 答え」のチャット形式。最後に「実際にボタンを押す」のは人間
  • Computer Use 「スクリーンショットを見て → 操作を指示 → 実行」のループで自律的に動作
  • Google AI StudioのStream機能はリアルタイムの視覚情報を捉えるだけだったが、Computer Useはさらに一歩進んで実際に操作を実行
  • Gemini 2.5 Proの視覚理解・推論機能を基盤にFlashの軽量性を融合
「アドバイスをくれる」から「代わりにやってくれる」へ。
AIの役割が根本的に変わったと言えます。

⑬ 企業での活用シーン

  • データ入力 複数システム間の手動転記作業を自動化。人間がやっていた定型業務をAIに代替
  • QAテスト Webアプリの regression test を自動化。毎回のクリック確認が不要に
  • レポート作成 複数のダッシュボードからデータを集めてレポートにまとめる
  • カスタマーサポート FAQページやマニュアルを参照して対応案を自動生成
特に地味な定型作業こそ
AIによる自動化の最大の効果を発揮。
目立たないけれど一番効果が大きい。

⑭ 今後の展望と進化方向

  • モバイル拡張 スマートフォンアプリの操作への対応が今後の大きな課題
  • デスクトップ対応 PC上のファイル操作やネイティブアプリの制御への拡張
  • エージェント化 複数のアプリをまたいだ自律的なタスク実行へ
  • MCP等の外部プロトコルとの連携で、より広範なツール操作が可能に
今は「ブラウザ操作の自動化」段階ですが、
最終的にはAIエージェントがPC全体を管理する
未来に向かっています。

⑮ 導入前に確認すべきこと

  • タスクの適合性 ブラウザ上のクリックと入力で完了する作業かどうかを確認
  • セキュリティ 機密情報を扱う操作は、サンドボックス環境で実行する
  • 監視体制 重要な操作は人間が確認するフローを必ず組み込む
  • スケール 大量のタスクを回す場合のコストとレイテンシを見積もり
まずはリスクの低い定型作業から試して、
