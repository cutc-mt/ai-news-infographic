Loop Engineering: AIエージェントが「考える」ループを設計する

プロンプトエンジニアリングは終わった。次は「ループ」を設計しよう。
📅 2026年7月18日 | AI Anytime

①Loop Engineeringとは何か

  • Loop Engineeringとは、AIエージェントが実行するループそのものを設計する新しいパラダイム
  • プロンプトを頑張るのではなく、エージェントが自律的に動く仕組みを構築する
  • エージェントは「LLM + ツール + whileループ」の構造で動く
  • Peter Steinberger（OpenClaw開発者）とBoris Cherny（Claude Code責任者）がほぼ同時に提唱
「もうプロンプトを書かない。
僕の仕事はループを書くことだ」
— Boris Cherny, Anthropic

②進化の歴史: プロンプト→ループ

  • Prompt Engineering：1回の指示を最適化する（2023年頃）
  • Context Engineering：モデルが見る全情報をカリキュレーション（2024年）
  • Harness：モデルを「作業エンジン」にする枠組み（LangChain）
  • Loop Engineering：エージェントの実行ループ全体を設計（2026年）
進化の軸は「1回の対話」→「状態管理」→「自律実行ループ」
すべてモデルを操縦するという同じ目的！

③現在のエージェント開発の問題

  • プロンプトを書く→権限を許可→テスト実行→エラー発生→修正依頼...
  • 同じサイクルを何度も繰り返してベビーシッター状態に
  • 20分経つと、自分がOffloadしたかった作業を自分でやっていることに気づく
  • 本来なら「考える作業」に集中すべきなのに、単純作業をしている
自分でOffloadしたはずの作業を、
結局自分が監視していて本末転倒...
ベビーシッター問題

④ループの2つの必須条件

  • トリガー（起動条件）：PRが開いた、CIが失敗した、スケジュール、Slackメッセージなど
  • 検証可能なゴール（停止条件）：全テスト通過、レビューOK、CIグリーンなど
  • 停止条件がないと、自信過剰なトークン炉になるだけ
  • cron（固定スクリプト）とは違う：エージェントが状態を見て判断する
cron = 「毎時このスクリプトを実行」
ループ = 「状態を見て次の行動を自分で決める」
この違いが超重要！

⑤5つのレバー: ツール設計

  • ACI（Agent-Computer Interface）：エージェント版のUI
  • AnthropicのSWE-benchではツール設計にプロンプト以上の時間を費やした
  • Simon Willisonはシェルコマンドを推奨：コーディングエージェントはシェルが得意
  • エージェントが「語れる」だけでなく「行動できる」ようにする設計
UIは人間向けインターフェース。
ACIはエージェント向けインターフェース。
同じくらい慎重に設計すべき！

⑥5つのレバー: 停止条件

  • 成功時・最大イテレーション・予算上限・進捗なし検出
  • 停止条件がないループは早期に勝利宣言するか永遠に止まらない
  • Softなゴール：レビューモデルがUIと仕様を照合する
  • Deterministicなゴール：全テスト通過 + CIグリーン
止まらないエージェント =
費用無限大のトークン炉💥
硬いブレーキが必須！

⑦5つのレバー: コンテキスト管理

  • Compaction：ウィンドウ限界で要約して再スタート
  • NOTES.md：エージェントがコンテキスト外にメモを残す
  • サブエージェント：数万トークン消費後、1,000〜2,000トークンの要約を返す
  • トークン数が増えるとリコール（思い出し）性能が劣化する問題
モデルは忘れるけど、リポジトリは忘れない。
外部アーティファクトに
判断と証拠を残すのが正解！

⑧5つのレバー: 検証（Verification）

  • ループが「本当にやった」ことを証明する仕組み
  • Anthropic：200以上の機能リストを全て passes: false で開始
  • テストスイートがエージェントの価値を劇的に amplifyする
  • レビューエージェント + diffと仕様の照合も有効
エージェントの「終わったよ！」
を鵜呑みにしない。
テストが通るまで終わらない！

⑨5つのレバー: ガードレール

  • サンドボックス：危険な操作を隔離する環境
  • スコープ限定の認証情報：テスト/ステージング環境限定
  • Simon WillisonはClaude CodeにFly.ioの$5予算を設定
  • 暴走ループが本番の課金に触れないようにする設計
Solomon Hykes（Docker創業者）:
「AIエージェントとは、
ループの中で環境を破壊するLLM」

⑩実践: Addy Osmaniの6要素

  • Automations：ループが自動で起動する仕組み
  • Worktrees：並列エージェントの衝突を防ぐ
  • Skills：プロジェクトルールを再利用可能に
  • Plugins：GitHub・Linear・Slack等と接続
  • Sub-agents：コードを書くのとレビューするのを分離
  • Memory：モデルは忘れるがリポジトリは忘れない
Skillsがないループは毎回ゼロから
プロジェクトを再発見する。
トークンを無駄に燃やすだけ！

⑪スキル（Skills）の重要性

  • マークダウンのファイル群：規約・例・テストコマンド・禁忌事項
  • 1スキル = 1タスク。できるだけ密度高く書く
  • 良いスキルは複利効果を生む：毎回の実行が改善を蓄積
  • エージェントにインデックスを作らせる：どのスキルを開くべきか自動判断
スキルは「プロジェクトの記憶」。
コンテキストに毎回手動で入れるのではなく、
自動で参照されるように！

⑫具体的なループの例

  • 毎朝自動起動：昨日のCI失敗・オープンISSUE・最新コミットを読む
  • 状態ファイルに優先順位付きで取り組むべき事項を書く
  • ISSUEごとに別ワークツリーを開き、エージェント1体を派遣
  • もう1体がレビュー＆テスト。テスト通過ならPR作成
  • 失敗時：エラーをフィードバック。行き詰まり：人間の受信箱へ
あなたは7回も指示を出さない。
7ステップのシステムを
一度設計するだけ！

⑬自動化 vs ループの違い

  • 自動化：「ステップ1→2→3」と固定シナリオ
  • ループ：「状態を見る→判断→実行→確認→次の判断」
  • ループは小さなエンジニアリングプロセスに近い
  • ループは作業の単位（Unit of Work）になる：スケジュール実行・サブエージェント・永続化
古いシステム：LLMをもう一度実行
新しいループ：作業の単位として
スケジュール・ワークツリー対応

⑭2つの危険な落とし穴

  • 問題1：ゴールの定義が難しい
  • ソフトウェア開発は探求的なことが多い：最終形が最初は分からない
  • 問題2：コスト爆発
  • 自己レビュー→サブエージェント→リトライで数百万トークンを瞬時に消費
ループ設計は大企業のラボなら
推奨されやすい。個人開発者は
手動起動＋監視が安全！

⑮カスタマーサポートでの応用

  • サポートチケットはループ設計の最適なユースケース
  • 着信→意図推定→情報検索→アクション→確認→解決 or 担当者へ
  • 従来チャットボット：10〜25%解決率
  • エージェント型プラットフォーム：70〜85%のエンドツーエンド解決率
CXリードの言葉:
「AIには自信があるものだけ
処理させ、それ以外は手を出すな」

⑯診断ガイド: 症状からレバーを特定

  • 自信満々に間違える → 検証＋ガードレール不足
  • 永遠にループ → 停止条件がない
  • 長いタスクで忘れる → コンテキスト管理不足
  • 語れるが動けない → ツール不足
  • 破壊的なことをした → ガードレール不足
修正はプロンプトの書き換えではなく
レバーの調整で！

⑰どのタスクにループを適用すべきか

  • ✅ ループが適している：繰り返し作業・明確な成功/失敗信号
  • ❌ ループが不適切：曖昧なタスク・創造的作業・「良いもの」が定義できない
  • 一度きりの作業 → そのままプロンプトで
  • 反復＋明確な信号 → ループを構築
  • 「自分がやっているのは単純作業か？」→ YESならループ化を検討
「より速く進む」ために使うか、
「理解を回避する」ために使うか。
正しい選択をしよう！

⑱まとめ: エンジニアであり続けよう

  • 5年前：自分でコードを書く
  • 2年前：モデルにプロンプトでコードを書かせる
  • 昨年：Claudeがコードを書くのを監視する
  • 今：適切なタスクにはループを設計する
  • 出荷したものを読む、品質を所有する、停止条件を定義する
プロンプトエンジニアリングは死んだわけじゃない。
レバレッジポイントが移動しただけ。
エンジニアであり続けよう！
