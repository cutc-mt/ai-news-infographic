Agent Frameworksの誤解 — A2Aプロトコルでマルチエージェントを正しく理解する

「どのフレームワークがいい？」と聞く前に知るべき2つのレベル

📅 2026年7月20日

①「どのエージェントフレームワークがおすすめ？」: それは間違った質問

  • コメント欄でよく聞かれる質問だが、前提が抜けている。
  • フレームワークを選ぶ前に、マルチエージェントの種類を理解する必要がある。
  • 二人が「マルチエージェント」と言っても、まったく違うものを指していることが多い。
質問を聞く前に、まず前提を整理しよう！

②マルチエージェントの2つのレベル: 根本的に異なる世界

  • マルチエージェントには2つのレベルが存在する。
  • レベル1: システム内部のマルチエージェント（単一システム内）
  • レベル2: システム間のマルチエージェント（ネットワーク経由で通信）
  • この違いを理解しないと、まったく話が噛み合わない。
同じ言葉でも意味が違う、ここが混乱の原因！

③レベル1: システム内部のマルチエージェント

  • 一つのシステムの中に複数のエージェントが存在するパターン。
  • 例：LangGraphやDeepAgentなどのフレームワーク内部で動作。
  • 内部でプランナー・リサーチャー・コーダーの3エージェントが協調。
  • メモリもコードベースも共有、直接関数呼び出しで通信。
  • HTTP通信もネットワーク呼び出しもなし。

④内部エージェントは「実装の詳細」: 外部からは見えない

  • HTTPで公開すると、外部からは単一のエージェントに見える。
  • APIは1つ、エンドポイントも1つ。
  • 外部の利用者は、内部に3つのエージェントが動いていることを知らない。
  • つまり、内部のマルチエージェント構造は実装の詳細に過ぎない。
外から見れば「1つのAPI」ただそれだけ

⑤レベル2: システム間のマルチエージェント通信

  • 独立したエージェント同士がネットワークプロトコルで通信する世界。
  • 各エージェントは独自のエンドポイントとアイデンティティを持つ。
  • 異なるフレームワークで構築されたエージェント同士でも通信可能。
  • 例: LangGraphのエージェントとHaystackのエージェントが連携。
  • レベル1とは完全に別の世界。

⑥A2Aプロトコル: エージェント間通信の土台

  • A2A（Agent-to-Agent）はシステム間通信に最適なプロトコル。
  • 講師が推奨する、エージェント通信の完璧な基盤。
  • A2Aはプロトコルであり、複数のライブラリが存在する。
  • 重要なのは「どのフレームワークを使うか」ではなく「どう通信するか」。
A2Aこそがエージェント通信の正解！

⑦エージェントカード: A2Aで最も重要な概念

  • エージェントカードはエージェントのプロフィール情報。
  • 何ができるのか、どうアクセスするのか、どんな能力を持つのかを記載。
  • この仕組みによってA2Aのエージェントは分散型で柔軟でスケーラブルになる。
  • カードを参照するだけで、他のエージェントの能力がわかる。
名刺のようなもの、見れば相手がわかる！

⑧デモプロジェクトの構造: 5つのサーバーファイル

  • VS Codeでバケーション予約システムを構築。
  • 左側に5つのサーバーファイルがあり、各ファイルに独立したエージェント。
  • 複数のMCPサーバーとclient.pyも配置。
  • 各エージェントはA2Aプロトコルで通信しながら独立して動作。
実践的なコードで仕組みを理解しよう

⑨ワークフロー全体: オーケストレーターが全体を統括

  • オーケストレーターがサブエージェントのエージェントカードを取得。
  • 各エージェントの能力とスキルをメモリに保存。
  • ユーザーが質問すると、オーケストレーターがタスクを振り分け。
  • サブエージェントがMCPサーバーを使って回答を生成。
  • オーケストレーターが最終回答をまとめてユーザーに返す。

⑩起動方法: サブエージェント→オーケストレーターの順

  • 起動の順番が重要: まずサブエージェントを立ち上げる。
  • その後オーケストレーターを起動すると、全サブエージェントのカードを自動取得。
  • Windows: start.ps1 をPowerShellで実行。
  • Linux/Mac: start.sh を実行。
  • その後 localhost:8000 にアクセス。
順番を間違えると動かないので注意！

⑪能力の提示: エージェントカードが教える情報

  • localhost:8000 にアクセスすると「バケーションオーケストレーター」が登場。
  • 各サブエージェントのエージェントカードに基づいて能力一覧が表示される。
  • ユーザーは「自分の質問にどのエージェントが対応できるか」を把握可能。
  • 天気の質問には対応できない旨も自動で判定して回答。

⑫フライト検索のデモ: ミュンヘン→マヨルカ

  • 「ミュンヘンからマヨルカへのフライトを探して」と入力。
  • オーケストレーターが正しいサブエージェントにタスクをルーティング。
  • フライト検索エージェントが結果を返す。
  • 利用可能なフライトのオファーが表示される。
適切なエージェントに自動振り分け！

⑬予約から請求書まで: 複数エージェントの連携

  • 「このフライトを確定予約して」と入力すると予約エージェントが起動。
  • 予約エージェントが予約ツール（MCPサーバー経由）を使用。
  • 続いてインボイスエージェントにもルーティングされる。
  • PDF形式の請求書が自動生成されてプレビュー可能。
検索→予約→請求まで自動でつながる

⑭オーケストレーターの役割: 指揮者としての責任

  • オーケストレーターは全体の指揮者として機能。
  • ユーザーの質問を分析し、最適なサブエージェントにタスクを振り分ける。
  • 複数エージェントの回答を統合して最終回答を構成。
  • ユーザーとの対話インターフェースを担当。
  • エージェントカードの情報をメモリに保持してルーティングに活用。

⑮MCPサーバー: エージェントの「裏側のツール」

  • 各サブエージェントはMCPサーバーを裏で利用している。
  • MCP（Model Context Protocol）が外部ツールへのアクセスを提供。
  • フライト検索、予約、PDF生成など各機能に専用のMCPサーバー。
  • MCPによってエージェントの能力が拡張される仕組み。
MCPがエージェントの「手」を伸ばす！

⑯フレームワーク選びの正しいアプローチ

  • ❌ 「どのフレームワークが一番いい？」と聞くのは間違ったアプローチ。
  • ✅ まず「どのレベルのマルチエージェントが必要か」を明確にする。
  • 内部の実装ならLangGraphやDeepAgentが適している。
  • 外部通信ならA2Aプロトコルが正解。
  • 両方を組み合わせることも可能。
目的に応じて使い分けるのが正解！

⑰異なるフレームワークの混在が可能: A2Aの真価

  • A2Aの最大の強みはフレームワークに依存しないこと。
  • LangGraphで作ったエージェントとも通信できる。
  • DeepAgentやHaystackのエージェントとも連携可能。
  • ベンダーロックインを避けられるのが大きな利点。
  • 将来の技術進化にも柔軟に対応できる。
特定のフレームワークに縛られない自由！

⑱まとめ: エージェントフレームワークを正しく捉えよう

  • マルチエージェントにはシステム内部とシステム間の2レベルがある。
  • システム間通信にはA2Aプロトコルが最適な選択。
  • エージェントカードが分散型で柔軟な通信を実現する。
  • デモコードはリポジトリからクローンして独自のビジネスロジックに改造可能。
  • 「どのフレームワーク？」ではなく「どのレベルの通信が必要？」が最初の問い。
正しい理解が良いシステム設計の第一歩！
