Claude Codeのセキュリティ対策: settings.jsonの書き方

📅 2026年7月20日

①Claude Code × 1Password連携: セキュリティの本質

  • APIキーを1Passwordに保存し、Claude Codeから参照させる運用。
  • アクセス時に指紋認証が必要で、AIが勝手に取得することは不可能。
  • 情報セキュリティ担当者から見れば「ありえない」が、実務では使う人が多い。
認証さえあれば、パスワードをAIに渡しても安全に使える仕組み

②settings.jsonとは: Claude Codeの設定ファイル

  • settings.jsonはClaude Codeの振る舞いを制御する公式の設定ファイル。
  • 「読ませたくない」「実行させたくない」操作を明示的に拒否できる。
  • CLAUDE.mdのお願い書きとは違い、settings.jsonには強制力がある。
お願いではなく、システムレベルで強制できるのがポイント！

③.clignoreの罠: ハルシネーションに注意

  • .clignore（ドットクロードイグノア）は公式機能ではない。
  • AIが勝手に作った情報が拡散され、今でも信じている人がいる。
  • Claude Codeがうまく解釈して動くこともあるが、保証なし。
.clignoreに頼るのは危険！settings.jsonを使うのが正解

④settings.jsonの書き方: 読み取り拒否設定

  • .envや機密情報ファイルの読み取りを拒否する設定を書く。
  • APIキー、認証情報、秘密鍵、AWS関連ファイルなどを保護。
  • permission.denyでreadツールからのアクセスをブロック。
{
  "permissions": {
    "deny": [
      "Read(.env*)",
      "Read(*secret*)",
      "Read(*credential*)"
    ]
  }
}

⑤settings.jsonの書き方: 編集・削除の拒否

  • settings.json自体の編集を拒否する設定も必須。
  • Claude Codeが「settings.jsonを書き換えて制限を解除」するのを防ぐ。
  • rmやRemoveなど削除系コマンドも拒否しておくと安全。
{
  "permissions": {
    "deny": [
      "Edit(settings.json)",
      "Edit(.claude/*)",
      "Bash(rm *)",
      "Bash(Remove*)"
    ]
  }
}

⑥サンドボックスの限界: ファイル読み取り問題

  • サンドボックスはコマンドの実行を制限するが、ファイル読み取りは制限できない。
  • テストフォルダ内でサンドボックスを有効にしても、外のファイルが読める。
  • だからこそsettings.jsonでのpermission.denyが重要。
サンドボックスだけで安心してはいけない

⑦フック設定: 実行制限の追加

  • フック機能で特定の操作を事前にブロックできる。
  • 外部と通信するコマンド（curlやwgetなど）の実行も制限可能。
  • Git関連コマンドやBashの外部通信もpermission.denyで制御。
外部通信系のコマンドは特に注意して制限しよう

⑧自動運転のリスク: 長時間ループの危険性

  • 長時間自動でループエンジニアリングを回す時、ずっと見ているわけにはいかない。
  • 人間が不在でも危険な操作をしないよう設定しておくことが必須。
  • settings.jsonを書いていない状態で自動運転するのは非常に危険。
settings.jsonなしで自動運転？絶対やめよう！

⑨企業のAI導入: トップダウンが不可欠

  • トップダウンでAIの使用を決められる組織でないと導入が進まない。
  • 現場の担当者が頑張っても上層部が反対すると止められてしまう。
  • 「リスクが大きい」「情報漏洩が怖い」という反対派を社長が抑え込む必要がある。
AI導入は社長の決断力がすべて

⑩コパイロット縛りのリスク: 大企業の課題

  • 大企業ではCopilotしか使えない会社が多い（Microsoftの影響）。
  • 1日8〜10時間をCopilotだけに触る時間はもったいない。
  • Copilot使いのスキルでは市場価値が高くならない可能性が高い。
Claude CodeやCodexも触れる環境を作ろう

⑪リスクとリターン: AI活用のジレンマ

  • 車の運転と同じ: メリット＞デメリットと判断すれば使う。
  • 「10年使っても何もない」かもしれないが「1日で情報漏洩」の可能性も。
  • これは人それぞれの判断で、正解はない。
運転しない人もいるように、AIを使わない選択もあり

⑫タイミングの重要性: 今やるか5年後か

  • 同じ努力量でも今やるか5年後にやるかでリターンが全然違う。
  • 今なら発信すれば人が見てくれるが、5年後なら誰も見ない。
  • Claude Codeも他のAIも、今やる方が圧倒的に有利。
今のうちに触っておくのが一番の投資

⑬ベンチャー企業の生存戦略: スピードが全て

  • カ部氏の言葉: 「ベンチャーの生き残り方はスピード」。
  • スピードを上げるにはAIを回すことが1番有効な手段。
  • 品質を高くしたいなら高いお金で外部に頼めばいい。
小さい会社ほどAIを使い倒さないと生き残れない

⑭ITリテラシーの必須性: AIを使う前に

  • ITリテラシーがないままAIを使うのは振り回されるだけ。
  • ターミナル操作やOSの基礎を知らないと危険なコマンドを実行してしまう。
  • AI＋ITリテラシーの両輪が初めて安全な運用を可能にする。
AIが使える ≠ 安全に使える。リテラシーが必須！

⑮AI活用の本丸: 単なる自動化ではない

  • リサーチや資料作成の自動化はサブに過ぎない。
  • 1人では絶対にできなかったことがAIでできるようになるのが本丸。
  • 業務基盤そのものをAIに乗せることが次のステップ。
「今までの延長」じゃない、新しいことができるようになる

⑯柔軟な組織作り: AI時代の生存条件

  • 1つのビジネスモデルに最適化しすぎるとAIの変化に追いつけない。
  • 変化に対応できる柔軟な組織を作ることが重要。
  • 能力が高くなくても、AIをうまく使える人材が強い。
何が来ても対応できる組織がAI時代の勝者

⑰マイナスから+10か、0から+10か

  • 周りがAIを導入してから追従するのでは0に戻すだけ。
  • 周りより早くAIを導入すればプラスに持っていける。
  • その差は20にもなる。今のうちに動くことが最大のアドバンテージ。
マイナスから+10と、0から+10じゃ全然違う

⑱まとめ: Claude Codeを安全に使い倒そう

  • settings.jsonでしっかりセキュリティを設定することが第一歩。
  • .clignoreやCLAUDE.mdのお願い書きに頼らず、公式機能を使おう。
  • セキュリティを整えた上で、AIを思い切り使い倒すのが正解。
安全を担保してこそ、AIの本領発揮！今がチャンス
