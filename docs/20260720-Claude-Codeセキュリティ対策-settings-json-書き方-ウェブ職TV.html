Claude Codeのセキュリティ対策 — settings.jsonの書き方

①settings.jsonとは: Claude Codeの設定ファイル

  • settings.jsonはClaude Codeの挙動を制御する設定ファイルです。
  • 「この操作はやらないで」という禁止事項を書いておける。
  • 逆に「これだけはやってもいい」という許可も設定可能です。
Claude Codeを安全に使うための
第一歩がこのファイル！

②.envファイルの読み取りを拒否する方法

  • .envファイルにはAPIキーや機密情報が含まれる。
  • Claude Codeが勝手に読み取って外部に送信するリスクがある。
  • ReadツールとEditツールを対象に拒否ルールを書くのが正解。
{
  "permissions": {
    "allow": [],
    "deny": [
      "Read(.env*)",
      "Edit(.env*)"
    ]
  }
}

③.claudeignoreの罠: ハルシネーションに注意

  • .claudeignoreでファイルを読ませない方法が広く流布中。
  • これは公式機能ではないため保証が一切ない。
  • AIが意図を解釈して動くが、穴があって機密情報が漏れる危険も。
AIが作った情報が
拡散して信じられてる罠！

④権限拒否リスト: permissions.denyの書き方

  • Read・Editツールにdenyルールを設定するのが基本。
  • APIキー、認証情報、秘密鍵、.env系を全部拒否。
  • AWS認証ファイルや秘密会議用フォルダも対象に。
denyリストに書いておけば
機密情報へのアクセスを防げる

⑤settings.json自体の書き換え防止: Edit拒否

  • Claude Codeは賢いので「読めないなら設定を消そう」とする。
  • settings.jsonのEditを拒否しておけば自分で書き換えられない。
  • Hooks（フック）の編集も拒否設定しておくと安全。
AIが自分のルールを
書き換えるのを防ぐ！

⑥削除コマンドの拒否: Remove権限を止める

  • rmコマンドなど削除系の操作は意図しない消去のリスク大。
  • 自分の意図しないファイルが消される事故を防げる。
  • 安全装置はあるが、denyリストで二重に止めておくのが安牌。
二重の安全装置で
ファイル消失を防止！

⑦サンドボックスの罠: デフォルトでは機能しない

  • サンドボックスは中のコマンド実行を制限するだけ。
  • 外のファイルの読み取りは普通にできてしまう。
  • test/フォルダの中で実行しても、外の.envは読めてしまう。
サンドボックスだけでは
機密ファイルの保護は不十分

⑧Bash・Gitコマンドの制限: 外部通信を止める

  • Bashコマンドで外部と通信するものも拒否推奨。
  • git pushなど意図しない操作を防ぐためにGitコマンドも制限。
  • curlやwgetなどでデータを外部に送るリスクも回避。

⑨1Password連携: 安全なパスワード管理

  • 1PasswordとClaude Codeの連携が公式に可能に。
  • 認証突破できないので、AIが勝手にアクセスできない。
  • 指紋認証が必要で、自分がいないと使えない仕組み。
生体認証なら
遠隔からの不正利用を防げる

⑩情報共有のジレンマ: メリットとリスクの天秤

  • AIに個人情報やパスワードを渡せば作業効率が上がる。
  • しかし情報漏洩時のダメージは甚大になる可能性。
  • 車の運転と同じ：メリットよりリスクが大きいなら使わない判断も。
渡す情報を増やすほど
リスクとリターンが変わる
