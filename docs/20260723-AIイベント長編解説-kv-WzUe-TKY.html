①開発フロー改革の3段階: FindingのAI活用レベル

  • レベル1 バイブコーディング ― 早く作る段階
  • レベル2 エージェンティックワークフロー ― 正しく作る段階
  • レベル3 必要なものを作る段階
  • AI導入でコードは早く書けるが質が落ちる問題を解消するため改革を実施
「AIに使われている」状態から脱却するには、質とスピードを両立する仕組み化が必要！

②PR自動生成スキル: FindingのAI活用レベル1

  • 変更内容をもとにタイプチェック、テスト、品質チェック、ビルドを自動実行
  • ブランチ戦略・コミット戦略を自動検知してPRを生成
  • セルフレビュースキルと組み合わせてコミット→PRオープンの平均時間を30%削減
  • 社内トップ5の使用率を誇る人気スキル
誰でも一定品質以上のPRを簡単に作れるようになった！

③セルフレビュー: Findingで1番使われるスキル

  • コード変更に対して規約順守、コード品質、セキュリティを専門エージェントがフルボッコに
  • 出てきた問題点をAIがさらにコード修正として反映
  • 全スキルの中でFinding全体で1番の使用率を記録
「コード生成の質を一定以上担保」ができればAIの信頼度は跳ね上がる！

④AIフレンドリーの仕様書: レベル2の実現

  • 要件の明確化 → 現状コード探索 → 設計方針決定 → タスク分解
  • 親子構造で仕様書を作成。親タスク→子タスクの順にAIが実装
  • 例: 通知テーブル追加 → API作成 → フロント実装の順序で自動実行
  • 結果として1人当たりのプロダクト作成数が1.5倍に
タスク分解の精度と質が成果物の質に直結する。ここは何度でもトライ＆エラー！

⑤Finding Library公開: AIエージェントも読めるドキュメント

  • 会員登録不要・完全無料の開発ナレッジ公開サイト
  • WebだけでなくAIエージェントからの読み込みも前提として設計
  • MCP（Model Context Protocol）にも対応。エージェントに知識を注入可能
  • 開発の基礎基本＋AI活用コンテンツを随時追加更新中
MCP経由で自社AIに組み込めば、今日の内容をすぐに真似できる！業界の底上げを目指す意欲作！

  • 自己満足で終わり、現場では「めんどくさいからChatGPTでやってます」と使われない
いいものでも使いやすくなければダメ。「バカでも使える」レベルが目標！

⑧Multicaで30個のOSS運用: AA114の挑戦

  • PI（Claude Code拡張）向けの約30個の小さなOSSを複数エージェントで並列運用
  • 総計39パッケージ公開、総合ダウンロード数3万に到達
  • Multicaの看板画面で全エージェントの状態を一覧把握可能
  • 安価なモデルでも迷わず・止まりにくく回すための設計が鍵
上位モデルを1人に渡すのは費用爆弾。安価モデルでも止まらない仕組みが命！

⑨AIを迷わせない4つの設計原則

  • ① エージェントの道具を最小限に絞る（PIのツールは7つだけ）
  • ② 作業を小さく分割 ― 1タスク＝1件の変更。触らない範囲を明示
  • ③ 定型判断をスクリプトに移す ― 重複割り当て防止もスクリプトで判定
  • ④ 状態と証拠を会話の外に残す ― Multica＝実行状況、Obsidian＝長期記憶
スクリプトが決める、AIは曖昧判断をする。役割分担が安価運用のコツ！

⑩質問の宛先設計: 佐藤のC3ハーネス

  • 自装で1番怖いのは止まるべきところで止まらない・止まった後に誰も拾わないこと
  • タスク寿命の質問 → 起案者に聞けばOK（再現手順など）
  • プロジェクト寿命の質問 → メンテナやリードに届ける（判断原則・設計方針）
  • 原則昇格の認証は人の善意ではなく権限で機械検証（GitHubの権限モデルを活用）
セキュリティの権限昇格が悪意0でも起きる構造！チケットシステムでも共通の課題！

⑪「餌箱」ハーネス: ファイバベースの抽象化

  • Gitハブの抽象化「餌箱」＝ステータス推移と質問履歴がファイルに残る仕組み
  • 要件が空の餌はLLM起動前にスクリプトが門前払い
  • タスク品質スタンプはループの入口で実行
  • 1タスク＝ヘッドレスのクロードが隔離ボックスで数十円〜数十秒で実行
C3はOSSとして公開中！同じ場所で思考錯誤している人はぜひ渾身会で！

⑬プラットフォームエンジニアリング: 守り＆加速するAI開発

  • ゴールデンパス（舗装された道）で開発環境を即座に提供
  • ガードレールで意識せずとも安全な状態を維持。加速と守りは両立可能
  • ツールチェーン＋Model as a Serviceを繋ぎ、ポチッとでAI駆動開発が開始できる状態に
  • ハーネスを個人・プロダクト・組織の3層で整備。プラットフォームは下層を提供
「舗装された道があるから早く走れる。ガードレールがあるから安心してアクセルを踏める」

⑭AIに任せるための評価設計: 砂田のフレームワーク

  • AIは自分の間違いに気づかないまま完了と報告する。これを直すのがハーネス
  • 評価を作る3要素: 理由（なぜやる）制約（何ができない）成果物（何が欲しい）
  • 言語化が最も重要 ― AIが正しく意図を汲み取れるまで言語化・仕組み化する
  • 自動化できる範囲には限界がある。メンテコストがかかることを前提に設計
評価観点がずれてると、改善しても「良くなった」自体が間違ってる！なんとなく運用は避ける！

⑮Figma MCPの活用: デザインと実装の差分検出

  • URLを渡すだけではFigmaのデザインがうまく再現されない問題
  • Figma MCPでデザインを画像として取得させてから実装させる
  • 画像と実際の画面をサブエージェントで比較させ、差分を言語化して修正
  • AIは自分の作ったもののレビューは甘い。専用の評価サブエージェントを別途用意する
AIの特性を知って、「評価する人」と「実装する人」を分けるのが正解！
