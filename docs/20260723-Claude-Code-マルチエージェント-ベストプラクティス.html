マルチエージェント活用 — 1体で抱えない、編成術の極意
2026年7月23日

① ソリストに全部任せていませんか？

  • 大きなタスクを一体のClaude Codeに丸ごと投げるのは危険
  • コンテキストはどんどん埋まり、途中で迷子になる。調査、実装、レビューを1体に順番に抱えさせると、1つの作業が終わるまで次が始まらない
  • 役割を一体に集中させるほど待ち時間も見落としも増えていく
  • これはソリストにオーケストラの仕事をさせているのと同じ。指揮者はあなた、奏者は複数のClaudeに分けるべき
1体に何もかもやらせず、
役割を分担して動かす発想！
これがマルチエージェントの原点

  • この編成には3つの階層がある：サブエージェント、エージェントチームズ、外部オーケストレーター
  • 今日覚えて帰って欲しい言葉は1つだけ：「1体で抱えない」

③ 3つの階層 — 名前も仕組みも別物

  • サブエージェント — 1つのセッションの中で完結する仕組み。安定していてローカルと本番運用にも向いている
  • エージェントチームズ — 2026年2月にOpus 4.6と同時に登場した実験的な機能。複数のClaude Codeがメールボックスと共有タスクリストで直接連携
  • 外部オーケストレーター — 複数のリポジトリ、複数のチームを跨いで動かす外部ツール群。ダッシュボードやワークフロー管理機能を持つ
  • 名前もできることもバラバラなのに、なんとなく1つの話として語られがち。3つを明確に区別することが重要
この3つを1枚の地図に整理して
使い分けの判断基準を明確に！

④ 使い分けはシンプル — 3層の早見表

  • 1つのセッションで完結する仕事はサブエージェント
  • 複数のエージェントが長時間共同で進める仕事はエージェントチームズ
  • 組織全体・複数リポジトリで動かすなら外部オーケストレーター
  • まずは一番出番の多いサブエージェントから始めるのが正解。調査役とレビュー役を同時に走らせるだけで体感速度が変わる

⑤ サブエージェントの仕組み — 独立コンテキスト

  • 1つのメインセッションから専門特化したサブエージェントを呼び出す仕組み
  • それぞれが独立したコンテキストウィンドウを持っているのが重要ポイント
  • メインの文脈を汚さずに、調査やレビューだけを任せられる。メインの作業領域にノイズが入らない
  • 安定して動作し、ローカル開発から本番運用まで幅広く使える。今日からすぐ始められる
独立したコンテキスト =
メインの邪魔をしない
これが安定動作の秘密

⑥ 調査役とレビュー役を同時に走らせる

  • 調査役 — コードの調査・情報収集を担当するエージェント
  • レビュー役 — コードの品質チェックを担当するエージェント
  • 同じメインセッションから2体を同時に呼び出して並列に動かす
  • それぞれの結果だけがメインセッションに戻ってくる。手戻りが減り、体感速度がはっきり変わる
調査とレビューを同時に走らせるだけで
体感速度が劇的に変わる。
まずはここから試してみよう！

⑦ 設定ファイルに書いてルール化できる

  • この使い分けは設定ファイルに1行加えるだけでルール化できる
  • Claude Codeの設定（CLAUDE.mdなど）に記載しておけば、毎回のセッションで自動的に適用される
  • 一度設定すれば繰り返し使える。運用ルールとしてチームに共有も可能
  • シリーズの概要欄でサブエージェント運用ルールのテンプレートを配布している

  • サブエージェントはメインに結果を報告するだけだったが、エージェントチームズは違う
  • メールボックスの仕組みでチームメイト同士が直接メッセージを送り合う
  • 共有のタスクリストからそれぞれ自分の仕事を取っていく仕組み

⑨ チームズの有効化 — 環境変数1つ

  • 有効にするには環境変数を1つ設定するだけ
  • デフォルトではオフになっている。まだ実験的な機能だということが分かる
  • 向いているのは複数の視点で並列に調査やレビューをしてお互いの発見を付き合わせる仕事
  • 新しい機能を分担して同時に作ることにも向いている

⑪ 失敗パターンの分析 — 1642件の実行ログ

  • MaST研究 — 1642件の実行ログを調べた結果が報告されている
  • 失敗率は41%から86.7%に跳ね上がるというデータ
  • そのうちの36.9%は協調の破綻が原因だった
  • トークン消費も単体のおよそ15倍というデータもある。マルチエージェントはコストとリスクが跳ね上がる

⑫ やってはいけない — 3つの落とし穴

  • 単体で十分な仕事にマルチ化 — 簡単なタスクにまでマルチエージェントを導入してしまう
  • 堂々巡りの受け渡し — AがBに、BがCに、CがAに戻す。受け渡しの度に文脈が失われていく
  • 4体を超えて増やしすぎ — 協調のオーバーヘッドが増設の効果を食いつぶす。折り返し地点がある
投入する前に自分に問いかけて：
「並列に調べる価値は本当にあるか？」
「役割は本当に違うか？」
この2つに自信を持てないなら一体で十分

⑬ ルートエンジニアリングの4層構造

  • 感知 — 状況を捉える。何が起きているかを把握する
  • 判断 — どう動くかを決める。最適なアプローチを選択する
  • 実行 — 手を動かす。実際のコード変更や作業を行う
  • 強化 — 結果を振り返る。レビューと改善を行う
  • この4つをそれぞれ複数のエージェントで分担できる
感知→判断→実行→強化
この4層のどこを分担させるかが
設計の肝です。

⑭ CLAUDE.mdエンジニアリングで層を制御

  • CLAUDE.mdエンジニアリングのルーク制御セクションで、どの層を誰に任せるかを書き下す
  • 例えば実行の層だけを複数のエージェントに分担させたりする
  • 強化の層を別のエージェントに任せて、独立したレビュー役にすることもできる
  • 前回の教訓を忘れずに：何でもかんでもマルチ化せず、実行層だけ分担して判断層は単体で担う。抑え目な設計がちょうどいい

⑮ 外部オーケストレーター — 組織全体を見渡す

  • サブエージェントとエージェントチームズには共通の天井がある。どちらも1つのセッションの中で動く仕組み
  • 複数のリポジトリを同時に触ったり、チームを跨いで進捗を見渡したりするのはこの2つだけでは手が届かない
  • オーケストレーターはツリー単位で複数の作業を平行管理する
  • 進捗をダッシュボードで横断的に見せてくれる。何体ものエージェントが今どのリポジトリで何をしているか一目で分かる
CLAUDE.mdは人間のチームの憲法。

⑯ 導入前のチェックリスト

  • 並列に調べる価値は本当にあるか？ — 単体で十分なら無理にマルチ化しない
  • 役割は本当に違うか？ — 同じ仕事を複数に割っても意味がない
  • 受け渡しの文脈損失を許容できるか？ — A→B→Cと渡すたびに情報が薄れる
  • エージェントは4体以下か？ — 協調のオーバーヘッドが効果を食いつぶさないか確認
  • 本番コードへの影響は？ — 実験的機能で無人の変更は危険
