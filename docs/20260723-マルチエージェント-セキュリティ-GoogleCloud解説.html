マルチエージェントシステムのセキュリティ

Google Cloud セキュリティアドボケートによる実践解説
2026年7月23日 | Aaron Adelmann & Leonid Yonolan

① 間接プロンプトインジェクション: ユーザーは無実

  • ユーザーは「この300ページの文書を読んで」と普通の指示を出しただけ。ユーザーは完全に無実
  • しかし、文書の中に悪意のある指示が仕込まれていた場合、エージェントがそれを別のエージェントやツールに渡してしまう
  • プロンプトがどこから来るかに関係なく — ユーザー入力でも、インターネット上のテキストでも — 攻撃ベクトルになり得る
  • ユーザー自身が気づいていない間に、エージェントがツール呼び出しを実行してしまうのが最も危険
「薬剤の急速注射」 —
ユーザーが入力したつもりもない命令が
エージェントの連鎖を通じて実行される。

② エージェント時代の攻撃: SQLインジェクションとの対比

  • SQLインジェクションは攻撃者がAPIを直接叩く古典的な手法。システム侵入のパターンはよく知られている
  • エージェントの場合、すべての指示がエージェント自身によって無意識にやり取りされる。攻撃者は直接APIに触れない
  • 悪意のあるテキストがエージェントA → エージェントB → ツール呼び出しという連鎖を通じて拡散する
  • 表面上は完全に正当に見えるリクエストでも、高度な攻撃を仕込めるのが新たな脅威
攻撃者が直接システムに入るのではなく、
エージェント同士の会話の中に
悪意が潜り込むのが新しいパターン。

③ LLMのガードレールだけでは不十分な理由

  • 基礎となるLLMには何らかの安全柵が設けられている。「武器の作り方」はブロックされる
  • しかし、表面上は完全に正当に見える攻撃に対しては、LLMガードレールは検知できない
  • LLMの役割は豊富なコンテンツを提供することであって、セキュリティ対策専用ではない。だからセキュリティに特化した別ステップが必要
  • 悪意のあるURLの混入やフィッシングサイトへの誘導など、静的なリストで対応すべき脅威も存在する
LLMにセキュリティを任せるのは
セールスマンに警備を任せるようなもの。
専門家を別に立てるのが正解です。

④ Model Armor: セキュリティ専用の防御層

  • Googleが提供するセキュリティと安全性に特化したツール。LLMとは完全に分離された独立した防御システム
  • 常に最新の攻撃パターンを研究・追加。進化する脅威に追従し続ける
  • 悪意のあるURLの検出 — フィッシングサイトなどへの誘導を、Googleのスレッドインテリジェンスで最新状態に保つ
  • LLMに賄賂を渡す必要はなく、標準的なモデルでさえツール呼び出しだけで機密情報を漏洩させる可能性がある
Model Armorはモデル本体から分離 —
同じバイアスを持たない独立した監視者。
Googleへの攻撃未遂の全ケースを調査。

⑤ 機密データ保護（DLP）: PII・認証情報・秘密情報

  • ユーザーが入力欄に薬の種類、クレジットカード番号、生年月日などを誤って入力してしまう問題
  • コーディングエージェントにハードコードされたAPIキーやパスワードが含まれるリスク
  • 200種類以上の機密データを識別可能。個人名、クレジットカード、社会保障番号など
  • クレジットカードはLuhnアルゴリズムで真正性を検証。単なる16桁の数字は弾かない
プロンプトインジェクションガードレールでは
ハードコードされたAPIキーを
検出できない。専用の検知が必要です。

⑥ ブロックではなく「編集」: データの部分削除

  • 悪いデータをブロックするだけでは不十分。多くの場合、ユーザーはブロックではなく一部だけを削除したい
  • 電話番号やクレジットカードの例：下4桁だけ表示して残りをマスク。実用性を保つ
  • 社会保障番号を固有識別子として使うアプリでは、可逆ハッシュに置き換えてログから隠す
  • Model Armorではクレジットカードの何桁を隠すかなど、マスク方法を柔軟に設定可能
店でカードの下4桁を見るのと同じ。
情報の一部を保持しつつ、
機密部分だけを削除するスマートなアプローチ。

⑦ 方向性は問わない: 入力も出力も検査

  • プロンプトがどこから来るか — ユーザー、インターネット、エージェント — 関係なく検査する
  • モデルへの入力も検査する。ユーザーが入力したプロンプトに悪意がないかを確認
  • モデルからの出力も検査する。応答に機密データや有害な指示が含まれていないか確認
  • 発生源に関係なく、同じ問題検知の仕組みを適用。統一的なセキュリティポリシーを実現
入力→検査→エージェント→検査→出力
両方向のゲートで守るのが
エージェントセキュリティの基本。

⑧ 可能な限り早期に防御: 効率性の観点

  • 悪意のあるプロンプトが最初から分かるなら、後々リソースを無駄にしたくない
  • エージェントがプロンプトを受け取ると、一連のプロセスが起動して複数のエージェントがプロビジョニングされる
  • たとえ自動スケーリングがあっても、悪意のあるリクエストでリソースを消費するのはセキュリティとコストの両面で非効率
  • 最初のエントリーポイントでブロックすれば、広範なエージェントシステム全体を停止できる
入り口で弾けば、後続のエージェント4つも
起動する必要なし。コールドスタートも
無駄にしないのが賢い設計。

⑨ セキュリティ境界の設計: 全エージェントに必要？

  • ケースごとに分析が必要。すべてのエージェントを保護する必要はない
  • 研究者と審査員の間に境界は不要。審査員は研究者の結果をそのまま評価すべきだから
  • セッションでコンテキストを共有するエージェントのグループの外で保護すれば十分なケースも多い
  • 各エージェントごとに設定が必要な場合もあるが、過剰なセキュリティはコストと複雑さを増やすだけ
「鎧を着すぎて動けない」を避ける。
本当に必要な境界だけに
Model Armorを配置するのが正解。

⑩ ADKで構築されたコース作成エージェント群

  • 教師向けのカリキュラム作成ツール。4つのエージェントが連携して動作
  • オーケストレーター — 現在のステップを管理し、次のエージェントへの移行を決定
  • 調査員 — インターネットで資料を探しに行く
  • 審査員 — Gemini Proを使用して調査結果を評価。矛盾やハルシネーションをチェック
コンテンツビルダーが最後に
構造化されたコースにまとめる。
各エージェントは個別のCloud Runにデプロイ。

⑪ ADKへのModel Armor統合: 数行のコードで実装

  • 169行目の`chat_stream`メソッド内で、エージェントに送信する前に数行でプロンプトのサニタイズを実行
  • 問題が検出された場合は即座に停止して報告。エージェントを起動するのを待つ必要なし
  • フィードバックが非常に迅速。「これはできない」と最初から教えてくれる
# agent.py - chat_stream メソッド
async def chat_stream(request):
    # Model Armorでプロンプトをサニタイズ
    sanitized = model_armor.sanitize(
        request.prompt  # ← 数行で実装
    )
    if sanitized.blocked:
        return sanitize_result  # 即座に停止
    # 問題なければオーケストレーターへ
    return await orchestrator.run(sanitized)
最も単純で直接的な使い方。
エージェント起動前に弾くから
コールドスタートの無駄もゼロ。

⑫ デモ結果: レモンケーキ vs 病原菌

  • 「レモンパウンドケーキの作り方」 → Model Armorなしでパス。全エージェントが起動してコースを作成
  • 「花火の作り方」 → 責任あるAIフィルタリングでブロック。脱獄ではないが危険と判定
  • 「浴槽で病原菌を発生させる方法」 → 脱獄未遂として検出。モデルの安全対策を回避する試みと判定
  • 脱獄検出と安全コンテンツ検出は異なるカテゴリ。ビジネスロジックが結果的に異なる処理を行う
「レモンパウンドケーキは
危険な食べ物かもしれない」
セキュリティ感度の調整次第で変わる！

⑬ Model Armorテンプレート: 感度の調整

  • 悪意のあるURLの検出とプロンプトインジェクションを有効化するテンプレート
  • 信頼度の閾値を「低」「以上」に設定すると、無害なプロンプトまで検出する誤検知リスクがある
  • 消費者向けアプリでは全セキュリティを最低レベルに。大学・研究機関では安全性設定を緩める必要がある
  • 病院などの内部利用では安全性の問題は深刻ではなく、機密データ保護に重点を置く
階層的なアプローチが鍵。
使い方に応じて感度を調整し、
過剰ブロックと漏れのバランスを取る。

⑭ DLPテンプレート: 検査・匿名化・構造化

  • 基本モード — クレジットカード、APIキー、パスワード、社会保障番号などの限定的セット
  • 詳細設定 — 地域に応じた識別。タイの納税者番号やジョニーの納税者番号など200種類以上
  • 人名の検出は注意が必要。「エイブラハム・リンカーン」のコースを作りたい場合に誤って削除される
  • 画像ファイルに含まれる名前も墨消し（リダクション）可能。構造化・非構造化どちらも対応
検査テンプレートで「何を見つけるか」を定義。
匿名化テンプレートで「どう変えるか」を定義。
二段階でデータを保護します。

⑮ ロギング: プロンプトの記録とガバナンス

  • Model Armorにはプロンプトと応答のログ記録を有効にするオプションがある
  • ログはGoogle Cloud Loggingに送信。アプリケーション側でログコードを書く必要なし
  • 開発者が書かずにプロンプトを記録できる。「レモンパウンドケーキ」のリクエストも確認可能
  • トラブルシューティング完了後はログを無効化してプロンプトを完全に非表示にできる
テスト中はログON、運用中はログOFF。
ガバナンスとプライバシーの
バランスを柔軟にコントロール。

⑯ 統合モード: Model Armorの2つの導入方法

  • Model Armorサービス統合 — プロジェクトレベルでGeminiとGoogle MCPサーバーへの全呼び出しに適用
  • 開発チームがModel Armorと直接やり取りするコードを書く必要なし
  • テンプレートと呼び出しの制御をセキュリティチームやプラットフォームチームに移管できる
  • Agent Platform — 外部のカスタムモデルやMCPサーバーも対応。コードゼロで保護を実現
サービス統合はGemini限定。
Agent Platformなら
あらゆるモデルやMCPサーバーで
Model Armorを使える。

⑰ Q&Aハイライト: 強化学習攻撃とSRE自動化

  • 強化学習攻撃への対策 — Model Armorはモデル本体から分離されているため、同じバイアスを持たない。攻撃パターンを継続的に研究
  • オーケストレーターが監視エージェントの役割を果たす。ADKフレームワークで決定論的なwhileループとして実行
  • NOC/SREの自動化 — AIは非構造化ログの解析や新しい問題のデバッグに強力。ただし決定論的なフローが必要
  • インシデント対応の自動化には「わからない場合は人間をプロセスに組み込む」別のアプローチが重要
SREのベストプラクティスを
しっかり実践している組織ほど
AI活用が最も容易になる。

⑱ まとめ: マルチエージェントセキュリティの基本原則

  • Model Armorでモデル本体とは別にセキュリティ層を設ける。同じバイアスを持たない独立した防御
  • 可能な限り早期にプロンプトを検査。入り口で弾けばリソースの無駄を防げる
  • 入力も出力も両方向を検査。方向性は問わない
  • ブロックだけでなく編集・マスクでデータを保護。ユースケースに応じて感度を調整する
エージェントの力は強大。
だからこそセキュリティ設計が
エージェントの信頼性を決める。
