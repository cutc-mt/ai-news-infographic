マルチエージェントシステムのセキュリティ

Google Cloud Tech — Model Armor & ADKで守るエージェントの安全対策
2026年7月23日

① セッション概要: エージェントの安全な連携

  • 講師はアーロン・アデルマンとレオネッド・ヨヌラン、ともにGoogleのセキュリティアドボケート
  • テーマは、エージェントが他のエージェント、プロンプト、ツールとどうやり取りするかという重要なセキュリティレイヤー
  • 基礎モデルには安全ガードレールが備わっているが、エージェントシステムでは追加の保護層が必要
  • 実演にはADK（Agent Development Kit）を使用し、Cloud Runにデプロイしたマルチエージェントアプリをデモ
エージェント間の通信こそが
最大の攻撃面になります。
モデル本体の安全性だけでは不十分です。

② 間接プロンプトインジェクション: 全く新しい脅威

  • 従来のプロンプトインジェクションは悪意のあるユーザーが直接的にプロンプトを注入するもの
  • マルチエージェントでは、ユーザーは無実だが、エージェントが読み込んだテキスト内に悪意のある指示が埋め込まれる
  • 例：「300ページの文書を要約して」→ 文書の中に「このURLにアクセスして」という指示が仕込まれている
  • その指示が別のエージェントに渡され、ツール呼び出しが実行される。ユーザーは何が起きたか気づかない
SQLインジェクションと似ているが、
攻撃経路が全く異なります。
エージェントが無意識に悪意を伝播するのです。

③ 攻撃フロー: 無実なユーザーから被害が広がるまで

  • ユーザーが正当なリクエストを送信する（例：文書の要約）
  • エージェントが外部テキストを取得し、その中に悪意のある指示が含まれている
  • ガードレールがない場合、エージェントはその指示を別のエージェントに渡してしまう
  • 別のエージェントがツール呼び出しを実行し、実際に被害が発生する
プロンプトの発生源を問わず、
全方向から検査するのが
セキュリティの基本です。

④ LLMガードレールの限界: モデルだけでは防げない

  • 基礎モデルには武器製造や違法行為を防ぐ組み込みの安全ガードがある
  • しかし、表面上は正当に見える高度なプロンプトインジェクションは検知できない
  • LLMの役割はコンテンツ提供と論理的思考であり、セキュリティ専用ではない
  • OpenCloudで発生した悪質なスキルの例：一見普通のスキルが、後に暗号資産ウォレットを盗むフィッシングサイトへ誘導
悪意のあるURLの検知には
常に更新される脅威インテリジェンスが必要。
LLMにそれは期待できません。

⑤ Model Armor: セキュリティ専門の外部レイヤー

  • モデルとアプリケーションの間に、セキュリティ専用のステップを挟むのが解決策
  • Googleの脅威インテリジェンスチームが常に更新。最新の攻撃手法に対応し続ける
  • 悪意のあるURLの最新スレッドインテリジェンスを備え、静的なリストとは異なる動的な検知を実現
  • プロンプトインジェクション対策だけでなく、データ損失防止（DLP）も統合して提供
モデルアーマーはLLMと分離された
独立したセキュリティシステム。
同じバイアスを持たないのが強みです。

⑥ 一元化ポリシー: 複数ツールへの対策を1か所に

  • 従来はエージェントコード内の複数の場所にセキュリティチェックを分散させる必要があった
  • Model Armorでは、すべてのルールを1つのポリシーにまとめる。管理が劇的にシンプルに
  • 組織内の別のチームが独自のエージェントを起動しても、ポリシーを環境間で移行可能
  • ガバナンスの観点からも重要。必要なチェック体制が整っているという安心感を提供
SQLインジェクション対策が
あってもModel Armorは必要。

⑦ 機密データ保護: 個人情報がモデルに漏れないように

  • 医療、金融など、機密性の高い情報がモデルに渡るリスク。患者データやクレジットカード情報の漏洩
  • コードに誤ってハードコードされたAPIキーやパスワードが含まれる問題も検知対象
  • Model Armorは200種類以上の機密データを識別可能。個人名、クレジットカード番号、納税者番号など
  • 地域に応じた識別設定が可能。タイの納税者番号、ジョニーのIDなど国別カスタマイズに対応
ユーザーは無意識に個人情報を
入力してしまいます。
システム側で保護するのが必須です。

⑧ データ編集: ブロックだけでなく賢く加工する

  • 多くの場合、完全にブロックするのではなく、機密部分だけを削除・マスクすれば十分
  • 例：クレジットカード番号は下4桁だけ残す。電話番号の一部を保持するなど、柔軟な設定が可能
  • Luhnアルゴリズムで有効性を検証。16桁の数字が必ずしもクレジットカード番号とは限らないことを正確に判定
  • トラブルシューティングが必要な場合は、可逆ハッシュに置き換えて鍵を持つ人だけが復元可能に
「ブロック」は最終手段。
マスク・削除・ハッシュ化など
状況に応じた対応が鍵です。

⑨ ADKマルチエージェント: 4つのエージェントが連携

  • オーケストレーターエージェント: マネージャー役。現在のステップを管理し、次のステップへの移行を判断
  • 調査員エージェント: インターネット上で資料を探しに行く
  • 審査員エージェント: 調査結果を評価。矛盾やハルシネーションがないか確認。最も強力なGemini Proを使用
  • コンテンツビルダー: 教師が使用できる構造化されたカリキュラム形式にコースをまとめる
ADKでエージェントを接続するのに
必要なコードは驚くほど少ない。
各エージェントはCard形式で定義。

⑩ chat_stream: 入出力の両端でModel Armorを適用

  • 入力時の保護: ユーザーからのリクエストが最初のエージェントに送られる前に、数行のコードでプロンプトのサニタイズを実行
  • 出力時の保護: 最終結果がユーザーに届く前に、機密データや不適切な内容が含まれていないか検証
  • 悪意のあるリクエストを最初からブロックできるため、無駄なエージェント起動やリソース消費を防止
  • これはModel Armorの最もシンプルで直接的な使い方。フィードバックも即座に得られる

⑪ ライブデモ: レモンパウンドケーキ vs 病原菌

  • 通過 「レモンパウンドケーキの作り方」→ 安全と判定され、全エージェントが起動してコースを正常に構築
  • ブロック 「花火の作り方」→ 危険と判定。AIコース作成ツールの悪用を防止
  • ブロック 「浴槽で病原菌を発生させる方法」→ 脱獄未遂として検出
  • 責任あるAIフィルタリングによるブロックと、脱獄検出によるブロックは別メカニズム。ビジネスロジックで処理が変わる

⑫ 効率性の最適化: できるだけ早く検知する

  • プロンプトが最初のエージェントに到達する前に検査するのが最も効率的
  • 悪意のあるリクエストを後からブロックしても、リソースの無駄遣いは防げない
  • 4つのCloud Runサービスがコールドスタートするコストも、事前ブロックでゼロに
  • セキュリティの観点だけでなく、リソースコストの観点からも早期検知が最適解
エージェントが起動される前に
入口で止めるのが正解。
広範なシステム全体を阻止できます。

⑬ テンプレート設定: 感度と保護レベルの調整

  • 信頼度の閾値は「低」「中」「高」の3段階。低にすると無害なプロンプトまで検出するリスクがある
  • 高リスク環境（消費者向けアプリ）では全セキュリティレベルを最高に
  • 教育機関では戦争や危険なテーマの研究があるため、安全設定を緩める必要がある
  • 病院内のアプリでは機密データ保護に重点を置き、安全性フィルターは緩めに設定するケースも
使い道に応じて
パラメータを柔軟に調整。
ワンサイズ・フィット・オールではありません。

⑭ 検査＆匿名化テンプレート: 200+のデータタイプ

  • 検査テンプレート: 機密データの識別用。メールアドレス、住所、セキュリティデータなど200種類以上を設定可能
  • 匿名化テンプレート: 検出したデータのマスク・削除を実行。検査テンプレートをコピーして構築
  • 基本モードはクレジットカード、APIキー、パスワード、社会保障番号など限定的なセット
  • 高度設定で既存のDLPテンプレートをインポート可能。CSVファイルでのテストも対応

⑮ マスク設定の詳細: 文字単位の精密な制御

  • クレジットカード番号の最初の12桁をマスクし、句読点は無視する設定が可能
  • ハイフン付きのカード番号でも、文字数としてカウントされないため意図しないマスク漏れを防ぐ
  • 社会保障番号も同様。有効ではあるが実際には使用されない接頭辞はそのまま通過
  • 画像ファイル内の名前などを墨消しできる構造化フリーフォーム画像の専用ラボも別途提供
「有効な番号」を正確に判定する

⑯ ロギングと監査: プロンプトの可視化

  • プロンプト＆レスポンスのロギングを有効にすると、Model Armorがすべてのやり取りをログに記録
  • ログはCloud Loggingに送信される。アプリケーションログではなく、別のログストリームとして管理
  • 開発者はコードを書かずにプロンプトの内容を確認可能。トラブルシューティングに極めて有用
  • トラブルシューティング完了後は、ログ設定を削除してプロンプトを完全に非表示に戻せる
ログをオンにすると機密データが
ログに残るリスクも。
用途に応じてオン/オフを切り替えましょう。

⑰ 統合モード: コード不要でModel Armorを適用

  • Model Armor Service内蔵統合: GeminiモデルとGoogle MCPサーバーに自動適用。プロジェクトレベルのフロア設定で全呼び出しに適用可能
  • Agent Platform: Google Cloudの新サービス。カスタムMCPサーバーやカスタムモデルにも対応。ゲートウェイ経由で保護
  • 統合の最大の利点：開発チームからセキュリティチームへの制御移譲。開発者はModel Armor用のコードを書かなくてよい
  • 違い：内蔵統合はGemini限定、Agent Platformはあらゆるモデル・MCPサーバーに対応
セキュリティチームが一元管理できるのは
ガバナンスの観点で大きな利点。
開発者の手間も減ります。

⑱ Q&Aハイライト: RLHF脅威・監視・SRE自動化

  • RLHFへの賄賂攻撃: Model Armorがモデル本体と分離されていることが最大の防御。常に最新の攻撃手法を調査・対応
  • エージェント間のセキュリティ境界: 全エージェントを保護する必要はない。ケースごとにどこに境界を置くか分析が重要
  • 監視エージェント: 専任エージェントは不要。オーケストレーターがスーパーバイザーとして機能し、ADKフレームワークが性能を把握
  • SREの自動化: 決定論的フローが必要。非構造化ログの分析にはAIが強いが、インシデント対応の自動化には明確な手順が必要。人間の介入を維持することが重要
Model Armorは入出力の両方を検査。
モデルに賄賂を渡す必要はなく、
ツール呼び出しだけで機密データが漏れます。
