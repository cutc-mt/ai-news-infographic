マルチエージェントに潜む脅威の全体像

①マルチエージェントとは: AI同士が連携する新しい仕組み

  • 複数のAIエージェントが役割を分担して連携するシステム
  • ユーザーは通常の指示を与えるだけでタスクが実行される
  • 例：「300ページの文書を要約して」という無害なリクエスト
ユーザーは悪意がないのに、システムの裏側で危険なことが起きるのが間接プロンプトインジェクションの怖さ！

②間接プロンプトインジェクション: 知らぬ間に仕込まれる罠

  • エージェントが読み込むテキスト内に悪意のある指示を紛れ込ませる攻撃手法
  • ユーザーはこの挿入に全く気づかない
  • 注入された指示が別のエージェントに渡り、ツール呼び出しで実行される
ユーザー「要約して」と思ってるだけ。でも裏で攻撃コマンドが実行される！ 危険

③SQLインジェクションとの類似性: 古典的脅威の復活

  • SQLインジェクションと同様に、不正な指示がシステム内部に侵入する
  • ただし侵入経路が全く異なる — API呼び出しではなく自然言語で侵入
  • 攻撃者が直接操作するのではなく、エージェント同士が無意識に伝搬
SQLインジェクションの「言語版」がプロンプトインジェクション。登場人物がユーザーではなくAIなのが新しさ！

④外部コンテンツの読み込み: インターネットからのリスク

  • エージェントがインターネット上の資料を自動で検索・読み込む
  • その資料の中に悪意のあるプロンプトが仕込まれている可能性
  • プロンプトの発生源を問わず、すべてを検査できる仕組みが必要
「どこから来たか」じゃなくて「中身が安全か」が重要。方向性を問わず検査すべし！
GoogleのModel Armorで守る

⑤Model Armorとは: エージェント向けの専用防御層

  • LLMの基本安全柵とは別に、セキュリティ専用の独立した防御レイヤーを設ける
  • 常に最新の脅威インテリジェンスで更新される
  • LLMモデル自体にセキュリティ対策を組み込む必要がない
モデルはコンテンツ生成に集中。セキュリティは別のシステムが担当する分離アプローチ！ ベストプラクティス

⑥設定可能な感度: 使い方で柔軟に調整

  • 信頼度の閾値を「低」「中」「高」で設定可能
  • 消費者向けアプリ: すべて最低レベルで厳しく保護
  • 医療・研究用途: セーフティ制限を緩和しつつピンポイントで検知
「低」すぎると無害なプロンプトまで弾く。「高」だと確実な脅威だけ。用途に合わせて調整が大事！

⑦悪意のあるURL検出: フィッシング攻撃も防ぐ

  • 表面上は普通に見えるスキルが暗号資産ウォレットを盗むフィッシングサイトへ誘導する例
  • LLMはURLの悪質性を静的に判断できない
  • Googleの脅威インテリジェンスが最新リストでリアルタイムに対応
「このサイトにアクセスして」の一言が大惨事に。静的リストじゃ追いつけないからGoogleが毎日更新！ 実例あり

⑧秘密情報の漏洩防止: APIキー・パスワードも検知

  • コーディングエージェントに誤ってハードコードされたAPIキーを渡してしまう事故
  • クレジットカード番号、パスワード、認証情報などを200種類以上で識別
  • 個人名や電話番号も検知可能（ただし文脈による調整が必要）
プロンプトインジェクション対策じゃ検出できない！秘密情報探知は専用のDLPが必要！
機密データ保護の工夫

⑨マスキングと削除: ブロックより賢いデータ処理

  • 機密データを見つけたらすべてブロックするわけではない
  • 下4桁だけ残すなど、一部を保持したまま機密部分だけを削除可能
  • クレジットカード番号のハイフンは文字数に含めないなど細かな設定が可能
全ブロックはユーザー体験を損なう。スマートにマスクして使い勝手を維持！ 実用的

⑩可逆ハッシュ化: トラブルシューティングにも配慮

  • 社会保障番号などを可逆ハッシュに置き換える手法
  • キーへのアクセス権がある人が元の情報を復元可能
  • ログに直接書かずに済むので、情報漏洩リスクを大幅に低減
「見えないけど必要なときに戻せる」。運用とセキュリティの両立！

⑪高度な検出: 200種類以上のデータタイプ

  • 基本モード: クレジットカード、APIキー、パスワード、社保番号など主要なもの
  • 詳細モード: DLPテンプレートを取り込んで地域固有の識別子にも対応
  • タイの納税者番号やジョージアの納税者番号など、地域に応じたカスタマイズ
16桁の数字がクレジットカード番号とは限らない。Luhnアルゴリズムで正確に判定！ 精密判定

⑫早期検知の重要性: リソースを無駄にしない

  • プロンプトができるだけ早い段階で検査することが最も効率的
  • 悪意のあるリクエストを最初に弾けば、エージェントの起動コストもゼロ
  • セキュリティ観点とリソースコスト観点の両方で最適化される
4つのエージェントを全部起動してからブロックしたらリソースの無駄！入口で止めるのが正解！ コスト注意
ADKで構築する安全なマルチエージェント

⑬ADKのオーケストレーション: 4つのエージェントの連携

  • オーケストレーター: 次のステップを決定し、実行順序を管理するマネージャー
  • リサーチャー: インターネットで資料を探しに行く調査員
  • ジャッジ: Gemini Proを使用して調査結果を評価・矛盾や幻覚をチェック
  • コンテンツビルダー: 構造化された形式にコースをまとめる
オーケストレーターはモデルを使わない完全決定論的ループ！シンプルなwhileループで制御。

⑭コード実装: 数行でサニタイズ

  • Cloud Runにデプロイしてサーバーレスで実行
  • chat_streamメソッド内でModel Armorを数行のコードで呼び出し
  • 問題が検出されたら即座に報告して実行を停止。エージェント起動不要
# chat_stream内でのModel Armor呼び出し例（169行目付近）
response = model_armor.sanitize(prompt)
if response.blocked:
    return {"error": "検出されました"}
# 問題なければオーケストレーターへ転送
エージェントを起動するのを待たずに「これはできない」を即判定！高速フィードバック！

⑮入出力の二重チェック: 入口と出口の両方で検証

  • 入口（入力時）: ユーザーのプロンプトが安全か検査
  • 出口（出力時）: 最終結果に機密データや不正な内容が含まれていないか検証
  • 業務状況に応じて、エージェントごとに保護レベルを変えることも可能
入り口で悪意を弾き、出口で情報漏洩を防ぐ。二重防御で安全度アップ！ 多層防御

⑯統合モード: セキュリティチームが制御を握る

  • Model Armor統合: GeminiモデルとGoogle MCPサーバーに自動適用
  • Agent Platform統合: カスタムモデル・外部MCPサーバーも対応可能
  • 開発者がセキュリティコードを書かなくても、プラットフォーム側で一元管理
開発チームはアプリ開発に集中。セキュリティ設定はセキュリティチームが一元管理できる！ チーム分離
実演結果とQ&Aのハイライト

⑰ライブデモ結果: レモンケーキ✅ 病原菌❌

  • 「レモンパウンドケーキの作り方」→安全と判定、コース正常生成
  • 「花火の作り方」→ 責任あるAIフィルタリングで停止（脱獄ではない）
  • 「浴槽で病原菌を発生させる方法」→脱獄未遂として検出・ブロック
「責任あるAIフィルタリング」と「脱獄検出」は別物！どちらもブロックできるのがModel Armorの強み！ 検出成功

⑱Q&Aハイライト: SRE・強化学習・ロギング

  • SRE自動化: AIは非構造化ログの解析に強いが、決定論的フローが必要
  • 強化学習攻撃: モデルとガードレールを分離すればバイアスの影響を受けない
  • プロンプトロギング: モデル装甲が全ログをCloud Loggingに送信。開発コード不要
「人間をプロセスに組み込む」ことが最終的な安全網。AIに100%任せない！ 重要
