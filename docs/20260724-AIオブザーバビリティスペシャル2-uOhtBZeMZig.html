費用可視化 & サブエージェント最適化

③Claude CodeのCGU機能でトークン消費を可視化

  • CGU（Claude GPU Usage）でタスクごとのトークン消費量を確認可能
  • 「各タスクがどのくらいトークンを消費しているか」を意外と見ていない
  • フェイブル単体 vs サブエージェント分割の2パターンで比較
  • モデルの料金表を計算するアプリを実験的に構築
CGUでタスクごとのトークン消費を見える化。
同じアプリを2パターンで作って比較検証！

④コスト削減効果: フェイブル単体18ドル → 分割15ドル

  • フェイブル単体: 18ドル / サブエージェント分割: 15ドル
  • コスト16%削減を達成
  • 要件はどちらもすべて達成、品質も十分満たすレベル
  • 細かいUIのディテールはフェイブル単体の方がやや上
  • 「機能要件さえ満たせばOK」なら最適化する価値あり
年間換算で44万円〜530万円の差に
1日8時間×5回×250営業日 = 年間1,250回分の差が顕著に

1人1日1億トークン時代のやりくり

⑥1人1日1億トークンの消費実態とプロダクティビティ

  • ファインディの西村（村長）が1日約1億トークンを消費
  • クロードコードだけでデイリー50件のプルリクを作成
  • 枠を作りながら並列処理で効率化
  • トークン消費増加とプルリクの成功数の増加が相関
  • 「より少ないトークンでより多くのプルリクを」が課題
1日1億トークンって...実態が見える化されて初めて
「減らそう」ではなく「効率よく使おう」という方向に

⑦伸び代を勘で探さない: セッションログ分析

  • フェイブルに自分のセッションログを分析させる
  • 「何日から何日までの無駄なトークン消費を探して」のプロンプトのみ
  • フェイブルが勝手にプランを立てて分析してくれる
  • 実装タスクはフェイブルよりソネットでOK、オーパスのサブエージェントも検討
発見例:
毎回モデル・エフォートを適当に選んでいた → 役割ごとの最適化が可能
タスク難易度に応じたモデル選択の知見が得られた

⑧モデルとエフォートの最適な割り当て戦略

  • フェイブル: 設計・影響範囲調査・週次作成
  • ソネット: コード実装・テスト（設計が固まっている状態）
  • Haiku / 中速: gitコマンド・プルリク作成・ワークツリー作成
  • スキルも工程ごとに分割して徹底管理
  • チケット作成 → 実装 → テスト → セルフレビューを各スキルに全部分割
スキル内でサブエージェントを呼ぶなら

トークンマキシマム → トークンミニマム

⑪トークンミニマムの考え方: 量から質への転換

  • 「不要なトークンは企業の生産性の直接課税だ」（Aaron Chia）
  • 転換1 たくさん使う → 最小限で価値を出す

「水道の蛇口を出しっぱなしにしないのと同じ」

AI共同力の定義とフレームワーク

⑫AnthropicのAI Fluidity Index: 4つのDで定義

  • Delegation: 人とAIの役割分担
  • Description: 目的を完璧に言語化してAIを導く
  • Discernment: 出力の妥当性を見抜く
  • Diligence: 利用と結果に責任を持つ
  • CoderBiteの5スキルとも大枠で一致、海外で収束し始める方向性
海外では「質の定義」が同じ方向に収束し始めている。
CoderBite: 明確に統制・分解・トレードオフ理解・疑う・統合

⑬AI共同力の5つの観点と評価結果

  • 意図の言語化: 制約と完了条件を明示、問題フレーミング
  • 環境構築: コンテキスト整備・ガードレール・ツール整備
  • 共同提案: 提案の評価・選択・引き出し
  • 実行の統制: 修正の収束・異常の調整・起動中の修正
  • 品質保障: 批判的評価・品質除去・検証の設計
300社・100万件の評価データから判明:
「共同提案」と「品質保障」のスコアが特に低い
→ AIに使われるんではなく、AIを使うための指標

⑭提案の評価・採択と引き出しの重要性

  • 提案の評価と採択: AIの方針を根拠を持って採択・却下
  • 丸投げせず確認もせず採択するのはNG
  • トレードオフを踏まえた判断が必須
  • 代替案の引き出し: 設計機能でAIにアプローチの選択肢を出させる
  • 探索の範囲・比較視点を課題に適切に処理
「これからはAIを使っていますではなくて、
AIとうまく共同できてますと証明する時代」
可视化 → 育成 → ギャップを埋める

⑯GitHub CopilotをOpenTelemetryで観測する挑戦

  • GitHubがエンタープライズレベルのOTel設定に対応（7月更新）
  • VSC側ドキュメントで設定方法が整理されている
  • コレクターはDockerが必要（サーバーレスではコンテナアップス必須）
  • GitのマネージドセッティングJSONで組織内一括設定が可能に
  • バックエンド: Grafana / Datadog / New Relic / Jaeger等に対応
「正直Dockerなしでやろうとしたら全然違った」
環境構築のハードルが意外と高い ⚙️

⑰スペックキットでコストが見える家計が残る

  • スペックキットでタスクをユニットに分割して管理
  • 実言アナライズ: 仕様→計画→タスクの整合性チェック
  • Executor-Advisorパターン: 実行を安いモデル・計画を強いモデル
  • Director-Workerパターン: 強いモデルが指示、安いモデルが実行
  • 両パターンとも約半分のコストで同等のスコアを維持（Anthropic公式）
受け皿の重要性:
ClickHouse / LangFuse / Claude Outpost Gateway（OTLP対応）

Q&Aからの知見

⑱Q&Aから見えた3つの重要知見

  • コンテキスト維持の工夫: HandOffスキルでセッション間のドキュメントを自動生成
  • 受け入れ条件の明示: サブエージェントにはエンドポイント・テスト要件を明記して渡す
  • ソネット意外と使える: ハイクでは無理でもソネットならギリできるケースが発見
「エージェントの原理原則を把握し、適切なコンテキストを渡す。
出力は必ずレビューする」
AIにまるっと任せて丸呑みするのは一番危険！
