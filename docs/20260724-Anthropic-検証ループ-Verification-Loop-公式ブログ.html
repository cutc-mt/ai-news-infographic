Anthropic公式ブログ - 検証ループ（Verification Loop）解説

Claude Codeが自分自身で検査して直していく「検証ループ」の考え方と実践方法を解説

📅 2026年7月24日

①ブログの結論: 人間の手作業確認をスキル化すれば、Claudeが自律的に検証してくれる

  • Anthropic公式ブログが検証ループの考え方を公開した。
  • 人間が手でやっている確認作業は、スキル化すればClaudeが自律的に実行できる。
  • めちゃくちゃ直感的な話だが、実践する価値が非常に高い考え方。
  • まだ検証ループをやっていない人は、ぜひ今日から始めてほしい。
シンプルだけど重要な考え方！

②検証ループとは: Claudeが自分の仕事を確かめる工程のこと

  • プロンプトを入力 → エージェントがコンテキストを収集 → 行動する → 検証する、という一連の流れ。
  • 検証結果がNGならコンテキスト収集に戻ってループし、OKならレスポンスとして返す。
  • このループ全体を「エージェンティック・ループ」と呼んでいる。
  • 人間が次にプロンプトを入力して、またループが始まる。
検証→修正→再検証のサイクル

③検証の具体例: スクリプト・チェックリスト・リンター・型チェッカー

  • スクリプトを使って動作確認をする方法。
  • エージェントにJSON形式のチェックリストを渡して検査させる方法。
  • リンター（Linter）や型チェッカー（Type Checker）を活用する方法。
  • 検証ツールの質が上がると、より高い品質の仕事をしてくれると公式も実感している。
検証の質を上げるほどClaudeの品質も上がる

④残る課題: 機械が拾えない部分をどうするか？

  • スクリプトやテストで機械的に検証できる部分はうまく自動化できる。
  • しかし、実装のたびに人間が手で確認している部分は依然として課題。
  • この手作業をループに置き換えれば、人間は手放しで仕事ができる。
  • つまり、手作業のスキル化こそが最大のポイント。
手作業をどう自動化するかが鍵！

⑤解決策: 検証ツール・評価基準・ルーブリックを作成する

  • Anthropic公式はエバリュエーター（評価器）の作成を推奨している。
  • 評価の基準やルーブリック（採点基準表）をしっかり作り込むことが重要。
  • 基準を明確にすれば、Claudeがそれに沿って自律的に品質チェックできる。
  • 人間が毎回手でやっていたことを、スキルとして永続化する。
品質基準を形にしてClaudeに任せる

⑥スラッシュverifyコマンド: Claude Codeに組み込みの検証機能

  • Claude Codeにはスラッシュverifyコマンドが最初から用意されている。
  • ビルドして動かして観察する、という一連の検証を自動で実行してくれる。
  • まずはこれを使えば、最小限の労力で検証ループが構築できる。
  • 一度試してみることを公式も強く推奨している。
/verify から始めるのが第一歩！

⑦ツールチェーンによる検証: リンター・エラー検出・警告の自動拾い上げ

  • リンターを使って、コードに残ったエラーを自動的に拾い上げて対処する王道の手法。
  • 警告やエラーを検出し、Claudeが自律的に修正するループを構築。
  • すでに設定している人も多い、基礎的な検証ループ。
  • ツールチェーンのエラー出力をClaudeが読んで修正する仕組み。
リンターのエラーをClaudeに直させる

⑧コードレビュー機能: /code-reviewコマンドと@Claudeコメント機能

  • スラッシュcode-reviewというコマンドが存在する。
  • 現在はチームプラン・エンタープライズユーザー限定で利用可能。
  • コードレビューの指摘に@Claudeでコメントして、直させることができる。
  • 昔から「便利だ」と言われているClaudeの定番機能の一つ。
チームプランなら@Claudeで自動修正！

⑨GitHub Actionsとの連携: PRごとの自動検証ジョブを定義する

  • GitHub Actionsで検証スキルを呼ぶジョブを定義しておく。
  • プルリクエストやプッシュ時に自動でローカルと同じ検証が走る仕組み。
  • CI/CDパイプラインの中に検証ループを組み込むイメージ。
  • マージ前に必ず検証が通ることを保証できる。
PRを出すだけで自動検証が走る！

⑩スペックバリデーション: 設計ドキュメントに沿ってUIを検証する

  • 自分で作成したデザインMD（設計ドキュメント）に沿って実装を検証するスキル。
  • UIの実装が設計通りに仕上がっているかをClaudeに確認させる方法。
  • デザインと実装の乖離を早期に発見できる。
  • 設計ドキュメントを信頼できる唯一の情報源（SSOT）として扱う。
設計と実装のズレを自動で検出

⑪ルーブリック検証: 採点基準表を磨いてサブエージェントに評価させる

  • 採点基準表（ルーブリック）を自分の中でしっかり磨き込んでおく。
  • コンテキストがクリアなサブエージェントを別途立ち上げて評価させる。
  • 作成者と評価者を分けることで、客観的な品質チェックが実現。
  • ハーネスエンジニアリングを深くやっている人なら一度は見た手法。
作成者と評価者を分けるのがコツ

⑫独自検証ツールの作成: 自分なりの「良い成果物」を定義して検証する

  • 独自のスクリプト検証ツールを作って、柔軟な検証ループを構築する方法。
  • 例えば「より良い画像とは何か」を自分で定義して検証する。
  • Claude Designとの照らし合わせスキルを作って検証するやり方もある。
  • 公式のツールだけでなく、自作ツールも検証ループに組み込める。
自分なりの品質定義が最強の武器

⑬手作業を文章化する: 毎回の確認手順を言葉にしてエージェントに渡す

  • 毎回手でやっている動作確認を一度言葉に起こす。
  • Claudeに渡すぐらいの丁寧さで書いて、それを元に実装させる。
  • 客観的なルールを書けばOK。書けば書くほど Claudeが頼れる相棒になる。
  • 迷ったら基本的に聞けばいい。すり合わせて認識を明確にしてスキルを作る。
手作業を言葉にするだけで自動化の第一歩

⑭スキルクリエイターの活用: 検証ループをスキルとして実装する

  • スキルクリエイターを使って検証ループをスキルにする。
  • YAML部分をしっかりこだわって作るのが最初の一歩。
  • 拡張としてハーネスを作ったり、スキルクリエイター自体を自作する上級者もいる。
  • 最初は公式配布のスキルクリエイターを使うのがミニマムなスタート。
小さく始めて徐々に拡張していく

⑮4つの実行段階: 単発 → 埋め込み → 連鎖 → PR乗説

  • 単発: 毎回手動で「検証ループを走らせて」と指示する。
  • 埋め込み: スキルのステップの中に検証を組み込み、自動発火させる。
  • 連鎖: スキルから別のスキルを呼び出し、検証込みの開発サイクルを丸ごと自動化。
  • PR乗説: チームの標準仕組みとして、PRごとに全員に発火する検証ループ。
小さく始めて徐々にステップアップ！

⑯連鎖の実例: Anthropicチームの検証パイプライン

  • コードから差分が生まれたら、コードレビューを実行。
  • 次にSimplifyを行い、コードを整形・最適化。
  • その後Verifyで検証し、Designと照合する。
  • 最終的にプルリクエストとしてチーム全員で品質を守り抜く。
ベルトコンベアのように検証が流れていく

⑰連鎖のトレードオフ: 自動化と引き換えにトークン消費が増える

  • 連鎖は柔軟性と引き換えに自動化を得る仕組み。
  • 段階が増えるほどトークン消費も増えるので注意が必要。
  • 頻度が高まったものだけを上位のステージに格上げするのが賢いやり方。
  • すべてを連鎖にするのではなく、状況に応じて適切な形を選ぶ。
効率とコストのバランスを見極めよう

⑱まとめ: 手作業をスキル化して、回せば回すほど強くなる仕組みを作ろう

  • 手作業をどんどんスキル化して検証ループを作る。
  • スキルクリエイターなどを使って、タスクごとに再現性を確保する。
  • チームでも共有して改善し、手戻りを減らす仕組みを回す。
  • 回せば回すほどどんどん強くなる、そんな形を目指そう。
今日から/verifyを試してみよう！
