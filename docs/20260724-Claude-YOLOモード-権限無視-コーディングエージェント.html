Claude YOLOモード

危険なほど権限を無視するコーディングエージェントの真実と、
Dockerサンドボックスによる安全対策

📅 2026年7月24日

YOLOモードとは？

① YOLOモードとは: 全許可の危険な世界

  • YOLO = You Only Live Once — コーディングエージェントがユーザーの許可を一切求めず、どんなコマンドでも自由に実行できるモード
  • Claude Code、Cursor、Codexなど全プログラミングエージェントに何らかの形で搭載されている
  • 能力は非常に高まるが、同時に致命的なリスクも伴う
毎日使っているけど、実際に何が起きているか考えたことある？

② 実際の被害事例: データベース消去やディレクトリ削除

  • Claude Code、Cursor、Codexがデータベース全体を消去したりディレクトリを削除した事故が実際に発生している
  • コーディングエージェントは賢いので可能性は低いと考えがちだが...
  • たった一度でも起これば、非常に深刻な結果を招く
  • 自分には起こらないと思っていても、実際にはそうではない可能性が高い
リスクを「知っているつもり」でも、知らないことの方が多いかも...

③ 解決策の方向性: YOLOモードは止めない、安全に使う

  • 解決策はYOLOモードの使用を避けることではない — 全アクションを承認するのは時間の無駄で、エージェントが自律性を失う
  • YOLOモードで安全に実行する必要がある
  • Dockerサンドボックスのようなツールで隔離環境を提供すれば、エージェントに完全な自律性を与えつつ安全を確保できる
  • Dockerサンドボックスは無料でセットアップも非常に簡単
自律性 ＋ 安全性 ＝ サンドボックス！

リスクの全貌

④ ファイルシステムへのフルアクセス

  • 隔離されていない場合、エージェントはコンピュータ上のファイルやフォルダすべてにアクセス可能
  • 読み取り、編集、削除までできてしまう
  • OSのパーミッション設定で一部は制限できるが、基本はエージェントに丸出しの状態
ホームディレクトリ内のファイルすら保護できない...！

⑤ プロセスと環境全体へのアクセス

  • コンピュータ上で実行中のすべてのアプリケーションに干渉可能
  • プロセスの強制終了や環境変数の書き換えなど、何でもできる
  • ポート競合の解決時に、動作中のアプリを勝手に終了してしまうことがある
  • 独自のDockerエンジンすら操作できてしまうため、他のコンテナの削除リスクも

⑥ ネットワーク経由の情報漏洩リスク

  • プロンプトインジェクション攻撃を受けると、APIキーが取得される危険性
  • 攻撃者が待ち構えるURLに機密情報が送信される可能性
  • 対策として、アクセス可能なURLの許可リストを設定する必要がある
  • 調査やAPIテストの際に外部通信を制限することが極めて重要
ネットワーク分離はプロンプトインジェクション対策の要！

実際に起きた事故

⑦ node_modulesの自動削除: エージェントの暴走

  • node_modulesとロックファイルを削除して依存関係を全再インストールする操作が、エージェントの自主判断で実行される
  • デバッグで行き詰まると、「よし、依存関係の問題か」と自動的に判断して削除に走る
  • 時間がかかり、トークン消費量も多く、悪影響も大きい
  • ユーザーが明示的に指示しなくても勝手に実行されるので本当にイライラする
一回で「了解！」して全消去って怖すぎる...

⑧ 秘密鍵と環境変数の不正読み取り

  • 「読み取らないで」と指示していても、デバッグ目的で秘密鍵や環境変数を読み取ってしまう
  • コードベース内のキーだけでなく、.sshフォルダなど作業環境外の場所にまでアクセスしてしまう
  • SSH接続のデバッグ中にこの問題が頻発する
  • エージェントは一切の抵抗なく機密ファイルを読み込んでしまう

⑨ データベーススキーマの破壊: ロールバック地獄

  • コードのバグを特定できないと、データベースのスキーマに問題があると勝手に決めつける
  • 移行をロールバックして新しい移行を実行 → データベースが完全に混乱
  • アプリケーションが完全に壊れる可能性がある
  • 運が良ければバックアップを作ってくれるが、常にそうとは限らない
バックアップがあればいいけど、いつもあるとは限らない...

LLMの愚かな領域

⑩ 文脈の腐敗: LLMが愚かになる瞬間

  • 全LLMには「愚かな領域」が存在する — 最初の数十万トークンまでは最高のパフォーマンス
  • 会話が長くなると、特に20万〜30万トークンを超えるとLLMが完全に過負荷状態に
  • システムプロンプトを含む初期の指示を忘れてしまい、愚かな判断を下すようになる
  • ひどい事故はコンテキストウィンドウの終わり付近で起こりやすい
長時間のトラブルシューティングが一番危険！

⑪ プロンプトガードレールは頼りにならない

  • 「この操作はしないで」と設定しても、文脈が腐敗すると全て忘れる
  • エージェントが「何をしてほしくないか」を理解しているように見えても、常にそうとは限らない
  • 最初は「これは危険な操作だ」と抗議するが、再度指示すると簡単に実行に切り替わる
  • プロンプトではなくサンドボックスで強制的に保護すべき
「リスクが高すぎる」→「よし、やってみよう」の遷移が早すぎ！

⑫ rm -rfの恐怖: 実際のGitHub Issues

  • Claude CodeのGitHubで複数人が同じ被害を報告
  • 悪名高いrm -rfコマンドでホームディレクトリ全体を削除
# これを実行させたくない...
rm -rf /
# R = 再帰的（全フォルダを削除）
# F = 強制（確認なしで実行）
  • わずか1ヶ月の間に別のユーザーも同じ被害に遭った
  • イシューは「予定外」としてクローズ — LLM側の問題でClaude Code自体にバグはない
Git stash全消しや本番DB削除も報告されてる...

Dockerサンドボックスによる解決

⑬ Docker Sandbox: ワンコマンドで隔離環境

  • 1つのコマンドでインストール完了、MacでもWindowsでも対応
  • 完全無料でローカル実行 — Docker Desktopと同様にマシン上で動作
  • Claude Codeなどのコーディングエージェントと即座に連携可能
  • インストール自体もコーディングエージェントに任せられる
コーディングエージェントにリンクを渡すだけ！

⑭ sbxコマンド: エージェントをサンドボックスで起動

  • 起動はたった1つのコマンド:
sbx claude
# sbx = sandbox の略
# claude = コーディングエージェントの名前
  • 見た目は普通のClaude Codeターミナルと変わらない
  • 内部的には隔離されたサンドボックスVMで動作している
  • 依存関係のインストール、開発サーバーの起動、Dockerイメージのビルドもすべて可能
通常通り作業できるのに、完全に保護されている！

⑮ VM内での検証: ホストマシンにアクセスできない

  • コードベースはVMにマウントされているが、.sshフォルダにはアクセスできない
  • 許可リストにないURLへのアクセスは403エラーでブロックされる
  • 検証結果:ワークスペース外のホストファイルは非表示
  • ホストサービスへのアクセス不可、Dockerソケットもホストではなくネストされたデーモン
  • 独立したDockerエンジンで完全に分離されている
やろうとしてもアクセスできない → それが安全！

4つの隔離レイヤー

⑯ ハイパーバイザー分離: ファイルシステムとプロセス

  • VMレベルの隔離で、ホストマシン上のプロセスの誤終了やファイルの誤削除を防止
  • VM内ではエージェントが疑似権限ユーザーとして実行 — 何でもできるがVM外には影響しない
  • 完全なクリーンアップ機能付き — sbx rm でVMと中身を一括削除
  • 複数のサンドボックスインスタンスを作っても、ディスク容量を圧迫しない
VM内なら何やっても安全！それが分離の目的。

⑰ ネットワーク分離: URL許可リストで通信を制御

  • 特定のサイトへのリクエストのみ許可するポリシーと許可リストを管理
  • プロンプトインジェクション対策として最も重要なレイヤー
  • 新規サイトへのアクセスは許可プロセスを経由 — 一度許可すれば次回以降もアクセス可能
# ポリシー設定例
sbx policy allow https://api.example.com
# → このサイトだけ通信OK
APIキーが外部に漏れない = ネットワーク分離の力！

⑱ Dockerエンジン分離 ＋ ワークスペース分離

  • 独立したDockerエンジン — ホストのDocker Desktopとは完全に別管理
  • アプリごとに隔離されたコンテナ環境を構築・テストできる
  • ホストのDocker Desktopは自分用のアプリ実行用に予約し、開発はサンドボックスで
  • 直接マウントでコードベースをそのまま編集するか、クローンモードで完全なコピーを作成して隔離度を最大化
  • クローンモードならGit履歴をいじられるリスクもゼロ
必要な分離レベルを自由に選べる — これぞDocker Sandbox！
