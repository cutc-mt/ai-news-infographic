マルチエージェントが上手く動く3つのパターン

  • モデルが自然に「エージェント的」になったことで実用的なパターンが可能に
  • 鍵は知能を追加しつつ書き込みはシングルスレッドという狭いクラス
  • 「セクシーなアイデアの多くは採用されていない。でも動くパターンを見つけた」
「作るな」から「動く3パターン」へ。
10ヶ月で現実が追いついたエージェント進化の証拠。

② エージェント利用が8倍に爆増

  • Devinの大企業向け利用が過去6ヶ月で約8倍に増加
  • Push要因：性能向上でユーザーが自然とマルチエージェントを試し始めた
  • Pull要因：利用爆増でコストも爆増。低コスト高性能化が急務に
  • ユーザー自ら「DevinにDevinを管理させる」スクリプトを作り始める
8倍！モデルが自然にツールを使い

③ コンテキストエンジニアリングの原則

  • プロンプトエンジニアリングはgimmick。「あなたはシニアです」は無意味
  • コンテキストエンジニアリングは「正しいコンテキストを与える」技術
  • 原則1：エージェント間で可能な限りコンテキストを共有
  • 原則2：アクションは暗黙の決定を伴う。並行書き込みで決定が分裂
大部分は「読み取り専用サブエージェント」
に留まっている。
真の協調は難しい。

④ センセーショナルなデモ vs 現実

  • Cursor：200エージェントでブラウザ構築（20万行）
  • Anthropic：エージェントでCコンパイラ（10万行）
  • Karpathy：LLM学習スクリプト最適化（1万回反復）
  • 全てに共通：検証可能な成功基準がある。現実は人間の判断力をスケールする必要あり
デモは凄いが現実の開発では
人間の判断力をどうスケールするかが真の課題。

⑤ パターン1：コードレビューループ

  • Devin Reviewは1PRあたり平均2バグ検出。58%が重症
  • 複数サイクルで毎回新バグ発見。人間がPR開く時にはほぼ修正済み
  • DevinとDevin Reviewがネイティブに反復する仕組み
  • 「自分のコードをレビューしても無意味」→ 大間違い。劇的に機能する
「バカすぎて動くはずがない」
と言われたパターンが
最も実用的なものに。

⑥ 直感に反する — コンテキストを共有するな

  • 重要発見：事前にコンテキストを共有しない方がレビュー精度が高い
  • AIには「自尊心」がない。コンテキストが違えば独立した判断をする
  • クリーンなコンテキストで見落としを深掘りできる
  • スペックなしで実装から逆算推論。ユーザーの指示エラーも検出
人間なら自分のレビューは甘くなる。
でもAIはコンテキストが違えば別物。

⑦ Context Rot — 長いコンテキストで劣化

  • Context Rot：コンテキストが長いと判断品質が低下する現象
  • アテンションヘッド数に上限。重要な詳細が漏れる
  • 実装エージェントは何時間も作業し巨大なコンテキストを蓄積
  • レビュー側はdiffだけを見てゼロから再読み込み。短いコンテキストで高知性を発揮
クリーンなコンテキスト＝
高い知性。
これがレビューループの核心。

⑧ 通信ブリッジ — 3者間インタラクション

  • バグ報告をユーザー指示でフィルタリング
  • これがないと無限ループや指示無視が発生
  • 人間・実装・レビューの3者間インタラクションが実現
  • Takeaway：クリーンコンテキスト＋明確な通信
バグ発見だけじゃない。
フィルタリングで
本当に重要なものだけを修正。

⑨ パターン2：スマートフレンド

  • 人気モデルがSonnet級からOpus級にシフト。スケーリングが復活
  • ジレンマ：フロンティアは高コスト vs 小型モデルは能力不足
  • 解決策：小型モデルを主エージェント、大型モデルを相談相手に
  • SWE-1.5（950tok/sec）+Sonnet 4.5で低コスト+高性能を両立する実験
小さいモデルが自分の限界を知り、
賢いモデルにいつ相談するかを判断。

⑩ スマートフレンドの2つの課題

  • 課題1：小型モデルが自分の限界をどう知るか？「弱いモデルがいつ相談すべきか判断」が難しい
  • 解決策：最低1回は相談、マージコンフリクトでは必ず相談などのドメイン特化ルール
  • 課題2：賢いモデルからの逆方向の通信。知らないファイルがあれば「調査してからまた聞いて」と返す
  • 質問に答えるだけでなくオーバースコープで追加指導する方が効果的
弱い→強いの一方向じゃない。
双方向通信の設計が鍵。

⑪ SWE-1.5 vs SWE-1.6の教訓

  • SWE-1.5は主モデルとして不十分。Sonnet 4.5とのギャップが大きすぎた
  • SWE-1.6はOpus-4.5レベルを達成しギャップを埋め始めている
  • フロンティア同士（Claude+GPT）の組み合わせは本番で効果的
  • クロスフロンティア通信は「難易度エスカレーター」ではなく「ケイパビリティルーター」
モデルによって得意不得意が違う。
デバッグが得意、テストが得意を
適材適所に配置。

⑫ パターン3：マネージャー+子Devin

  • マネージャーDevinが大きなタスクを分割し、子Devinに並列で作業させる
  • 進行管理は内部MCPで調整。10PRまたぐ機能や12サービス渡りの移行も
  • Map-Reduce-Manage：マネージャー分割→子実行→マネージャー統合
  • Devinはすでに「Devin Manage Devins」機能として本番稼働中
非構造化スウォームは邪道。
実践的な形は
Map-Reduce-Manage。

⑬ パターン3の落とし穴

  • 小スコープの委任で訓練されたマネージャーは過剰に指示的になる傾向。深いコードベースコンテキストがないと逆効果
  • エージェントが子と状態を共有していると錯覚する問題。実際には共有されていない
  • サブエージェントからマネージャーへのクロスエージェント通信がデフォルトでは発生しない
  • これら全てに個別の専用修正が必要。まだ改善途上
「動く」と言っても完璧ではない。
コンテキストの同期が
最大の難所。

⑭ 非構造化スウォームは邪道

  • 任意のエージェント同士が交渉する非構造化スウォームアプローチはほぼ気晴らし
  • 実践的な形はMap-Reduce-Manage：マネージャーが分割→子が実行→マネージャーが統合して報告
  • このシステムを単一エージェントと同じように一貫性のある体験にすることが2026年の中心課題
  • 人間の味と判断力をどうスケールするかがCognitionの着眼点
エージェントを大量に放つのは
楽しいけど実用性は低い。
構造化された協調が正解。

⑮ 3パターンに共通する核心

  • 全パターンに共通：書き込みはシングルスレッド。追加エージェントは知能を提供しアクションは取らない
  • クリーンコンテキストのレビューアーがコーダーの見えないバグを発見
  • フロンティア級のスマートフレンドが弱い主エージェントの見落としを補完
  • マネージャーが子エージェント間のスコープを調整し決定の分裂を防止
知能の注入は複数でも
書き込みは1箇所。
これが成功の鉄則。

⑯ 未解決は全て「通信問題」

  • 弱いモデルがいつエスカレートするかをどう学ぶか？
  • 子エージェントが発見したことをどうやって兄弟の作業に反映するか？
  • エージェント間でコンテキストを転送して受信者を溺れさせないにはどうするか？
  • プロンプトでそこそこの成果は出るが、次世代モデルのトレーニングで真に解決するとCognitionは考えている
全ての未解決問題は
コミュニケーションにある。
技術ではない。

  • スマートフレンドパターンの価値がさらに高まる。高コストモデルを「相談相手」として効率的に活用する需要増
  • Anthropicも「Advisor Strategy」ベータを開始。小型モデルから大型モデルへの相談機能を実装
  • Cognitionは自社でもこの往復通信を前提としたモデルのトレーニングを計画中
次世代は「1つの巨大モデル」ではなく
複数モデルの協調が
当たり前になる。

⑱ まとめ — 人間の判断力をスケールする

  • レビューループ：クリーンコンテキストが鍵。通信ブリッジで一貫性担保
  • スマートフレンド：双方向通信が鍵。フロンティア同士の協調が現時点で最も効果的
